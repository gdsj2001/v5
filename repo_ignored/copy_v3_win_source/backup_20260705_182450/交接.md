# 8ax.WinRemote 交接

更新时间：2026-06-18

## 当前定位

`8ax.WinRemote` 是板端 UI 的远程显示和远程触摸终端，不是第二套控制系统。

正式操作链路必须保持为：

```text
Windows WinRemote
  -> board remote_ui_relay HTTP/WebSocket
  -> 板端 UI 原输入事件
  -> Command Broker UDS
  -> LinuxCNC / HAL / EtherCAT
```

WinRemote 只允许做两件事：

- 显示板端 relay 提供的 full frame / dirty rect 画面。
- 通过 relay WebSocket 发送 pointer event，等价于人在板端屏幕触摸。

WinRemote 不允许：

- 直接实现 Start、Home、Jog、E-stop、WCS、倍率、设零等业务命令。
- 直接写 LinuxCNC、HAL、EtherCAT。
- 直接写 `/run` 下 JSON、FIFO、`tap`、`tap_cmd` 等临时控制文件。
- 把直接 `/dev/fb0` 抓屏路径当远程显示、诊断截图或闭环证据路径。

## 板端需要怎么改才能对接 WinRemote

本轮没有修改板端。下面是后续板端要补齐的正式对接要求。

### 1. 板端必须启动正式 relay

当前只读检查显示：

- 板端 UI 主进程在跑。
- UI loop/刷新状态必须通过 remote relay、截图/帧证据或 SHM 读数确认，不再依赖周期性 perf JSON。
- `http://192.168.1.221:18080/remote/info` 当前连不上。
- 未看到 `v3_remote_ui_relay.py` 进程。
- 直接原因是 `enable_remote_relay` marker 没启用或 relay 未启动。

要让 WinRemote 正式接入板端，需要板端后续完成：

```sh
touch /run/8ax_v3_product_ui/enable_remote_relay
touch /run/8ax_v3_product_ui/enable_remote_input
/opt/8ax/v3_product_ui/re-v3-lvgl-ui.init restart
```

注意：这会重启板端 UI，必须先确认看板部署锁和当前板端验证任务状态。

这只是启动层。最终产品化时，不应靠人工临时 touch；应由板端启动配置或维护开关决定是否创建 marker，并由 init 脚本统一启动/停止 relay。

涉及板端文件：

- `lvgl_app/scripts/re-v3-lvgl-ui.init`
- `lvgl_app/scripts/v3_remote_ui_relay.py`
- `lvgl_app/src/v3_product_platform.c`

### 2. 板端画面链要走 LVGL flush dirty rect

WinRemote 期望板端提供：

```text
GET /remote/info
GET /remote/frame/full
WS  /remote/stream
```

板端实现边界：

- `v3_product_platform.c` 在 LVGL flush 时只做轻量复制。
- UI 主进程不做 zlib、PNG、WebSocket、TLS、磁盘日志或阻塞网络 I/O。
- `v3_remote_ui_relay.py` 或后续常驻 relay 服务负责 HTTP/WebSocket 会话、full frame、dirty rect、心跳和客户端连接。
- 直接 `/dev/fb0` dump 抓屏已退役，不能作为诊断截图、远程显示链或闭环证据路径。

当前已有运行文件方向：

```text
/run/8ax_v3_product_ui/remote_info.json
/run/8ax_v3_product_ui/remote_framebuffer.bgra
/run/8ax_v3_product_ui/remote_dirty
/run/8ax_v3_product_ui/remote_ui_relay_status.json
```

这些文件/缓冲只能作为板端 relay 内部实现细节；WinRemote 不能直接读写它们。

### 3. 板端输入链必须注入原 UI 事件

WinRemote 正式输入只会走：

```text
WS /remote/input
  -> control_request
  -> pointer_event down / move / up
  -> pointer_ack
```

板端 relay 收到 pointer event 后，必须通过板端本地 UDS 输入口注入 UI 原输入事件队列：

```text
remote pointer event
  -> /run/8ax_v3_product_ui/remote_input.sock UDS framed JSON
  -> LVGL input event
  -> 原页面/按钮 handler
  -> Command Broker UDS
  -> LinuxCNC / HAL / EtherCAT
```

禁止做成：

```text
WinRemote
  -> relay
  -> 直接发 Start/Home/Jog/E-stop/WCS/倍率/设零命令
```

也禁止做成：

```text
WinRemote 或 relay
  -> /run/.../tap 或 tap_cmd
  -> 测试 tap 路径
```

也禁止做成：

```text
WinRemote 或 relay
  -> /run/.../remote_input FIFO
  -> UI 读取全局临时控制 FIFO
```

正式输入只能是连接式 WebSocket 会话加板端本地 UDS framed input。`tap`、`tap_cmd`、SSH tap 和 `/run/.../remote_input` FIFO 只能作为历史/测试痕迹清理对象，不能作为 WinRemote 正式接入链路。

### 4. relay input 的最小协议要求

板端 `/remote/input` 至少要做到：

- 每个客户端独立 session。
- `control_request` 成功后才允许 pointer event。
- pointer event 必须包含 session、seq、phase、x、y、button、client_time。
- phase 只允许 `down`、`move`、`up`。
- x/y 必须校验在 1024 x 600 画面范围内。
- 每个事件返回 `pointer_ack`，包含 accepted、seq、phase、server_time。
- relay 或 UI 不可用时 fail-closed，不吞掉错误。
- 禁止把 pointer event 翻译成业务命令；只能翻译成 UI 输入。

### 5. 板端验收顺序

先做只读显示，再做输入。

显示验收：

```powershell
Invoke-WebRequest http://192.168.1.221:18080/remote/info
Invoke-WebRequest http://192.168.1.221:18080/remote/frame/full
D:\re\v3\8ax-win\publish\win-x64\8ax.WinRemote.exe --view-only true
```

通过标准：

- `/remote/info` 返回宽高、pixel_format、view_only 等字段。
- `/remote/frame/full` 能返回完整画面。
- `WS /remote/stream` 有 dirty rect 更新。
- WinRemote 能稳定显示板端画面，不依赖 `/dev/fb0` 轮询。
- 板端 UI loop 仍接近 30 Hz，relay 不明显拖慢 UI。

输入验收：

```powershell
D:\re\v3\8ax-win\publish\win-x64\8ax.WinRemote.exe
```

通过标准：

- Win 端点击或拖动产生 `pointer_event`。
- 板端 relay 返回 `pointer_ack`。
- 板端 UI 收到和本地触摸一致的事件。
- 若点击按钮，后续仍走原按钮 handler 和 Broker。
- Win 端首个 dirty rect 反馈应在目标时间内返回。
- 失败时 Win 端和板端都记录明确错误，不伪造成功。

### 6. 不需要改 WinRemote 的前提

如果板端按上面协议提供 relay，当前 WinRemote 不需要再增加控制命令。

WinRemote 的默认启动路径已经是：

```text
http://192.168.1.221:18080/
relay stream + relay input
```

后续主要工作应在板端：

- 启动 relay。
- 保证 dirty rect 输出。
- 保证 `/remote/input` 注入 LVGL 原事件。
- 保证 relay ACK 和视觉反馈延迟达标。

## 后续闭环建议

1. 先处理板端 relay 启用和启动链，不要再改 WinRemote 走 SSH tap。
2. 板端 `/remote/info`、`/remote/frame/full`、`WS /remote/stream` 正常后，再用 WinRemote 默认启动验证画面刷新。
3. relay input 验收要记录：
   - Win 端 click / drag。
   - board relay `pointer_ack`。
   - 板端 UI 收到原输入事件。
   - Broker 命令结果。
   - 首个 dirty rect 反馈时间。
4. WinRemote 操作必须按“板端原始触发链”验收，不能只用直接 UDS 调用替代。
