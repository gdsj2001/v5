## 3. 原始 Operator Path

- Direct UDS/Broker/product handler 只适合诊断或验证 UDS 合同本身，不能替代 UI 按钮、页面、弹窗或触控缺陷的通过证据。
- `remote_input.sock` 是产品远程输入能力；常规启动应默认可用。`disable_remote_input`、`disable_remote_relay` 只允许作为显式临时诊断，交付前必须移除并恢复默认 enable marker。
- 退休路径 `enable_test_tap`、`enable_test_tap_session`、`tap_cmd` 不得恢复为产品或测试输入路径。
- 截图或 remote frame 只作为 operator path 的可视证据；如果截图与现场屏幕不一致，修抓屏工具链，不把截图细节写成独立产品规则。

最小证据通常包括：点击坐标或 selector、截图或 remote frame、`events.jsonl`、需要时的 `touch_events.jsonl`、SHM/UI 状态、Broker 结果或 action 结构化结果。

## 4. `cc.ngc` Golden Motion

运动闭环的验证输入是板端 `/opt/8ax/phase0_bus5/nc/cc.ngc`，源码样例是 `多格式g代码/cc.ngc`。不得为了通过测试修改 `cc.ngc` 或任何用户 G-code。

主页面 golden loop 的默认范围是：

```text
Home -> Open Program -> select cc.ngc -> Start -> E-stop -> reset -> Start -> final safe stop
```

判定重点：

- 必须证明 `screen_path_used=true`。
- 程序路径或 hash 必须匹配 `cc.ngc`。
- Start 必须看到速度或 joint/encoder 位移，不只看点击响应。
- E-stop 或 safe stop 后必须取得 microkernel/native 最终安全状态；V3 自动化不得用 SHM 速度、JSON 快照或私有容差替代底层安全状态。
- 涉及 Home 的 BUS/rotary 场景必须等待 microkernel/native Home gate 和原生状态返回完成条件，例如 `all_homed`、`enabled`、`!estop` 以及底层 owner 的安全窗口；不能用固定 15 秒窗口或上层速度容差替代。
- 若任务要求 Home 前设零，必须走设置页轴设零路径；不能用主页面 Work Zero 冒充 settings axis zero。


## 5. 硬件回归矩阵

`tools/v3_hardware_regression_matrix.py` 只验证证据结构和覆盖范围，不连接板端、不部署、不操作运动，也不能单独支撑 `board_verified`。

矩阵至少应覆盖这些场景的证据形状：

- settings drive chain
- settings axis zero
- Home BUS / Home Pulse
- Jog
- Start `cc.ngc`
- MDI/G-code
- E-stop/reset/HAL safety
- SDO/checkpoint
- SHM freshness

真实通过仍必须来自完整当前 worktree 的 VM 构建、板端部署、原 operator path 和必要 motion probe。

## 6. 失败处理

失败不是交付终点，是下一步诊断输入：

5. 先重跑能覆盖该失败的最小本地门禁，再升级到 required mode。
6. 板端忙、硬件不安全、网络/串口/relay 不可达或证据无法取得时，按 `AGENTS.md` 记录 blocker，不得要求用户人工代替闭环。



`failure_triage` 必须至少表达：需要先分析、候选失败域包含 `project_source` 和 `automation_tool`、当前判定状态、以及继续前的下一步动作。未判定前的状态必须是 `unclassified`，代表还不能把失败归咎于源码或工具任一侧。



## 8. 快速判断

| 问题 | 答案 |
| --- | --- |
| dry-run 能否证明功能通过 | 不能，只能分类和计划 |
| operator path 能否替代 motion golden loop | 不能 |

| board_deployed 是否等于功能通过 | 不是 |
