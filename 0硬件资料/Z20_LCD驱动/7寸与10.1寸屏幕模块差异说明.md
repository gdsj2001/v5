# 7 寸与 10.1 寸屏幕模块差异说明

## 1. 对比对象

本文对比的是以下两套正点原子下载资料：

- 7 寸：`ATK-MD0700R`，资料覆盖 800×480、1024×600，本项目重点参考 1024×600 型号。
- 10.1 寸：`ATK-MD1018R`，分辨率为 1280×800。

注意：V5 当前登记的实际目标屏幕是 **10.1 寸、1024×600**，实际触摸控制器
按 **FT5x26 系列、I2C 地址 0x38** 登记。新下载的
`ATK-MD1018R` 是 **10.1 寸、1280×800**，不是当前目标屏幕的完全匹配型号。

## 2. V5 当前实际组合

当前设备不是完整采用某一套下载模块，而是以下组合：

- **LCD 驱动板：采用 7 寸屏配套的驱动板。**
- **实际 LCD 屏幕：10.1 寸、1024×600。**
- **实际触摸控制器：FT5x26 系列，I2C 地址 0x38。**
- **当前显示时序：1024×600 `timing2`。**

因此，“7 寸”只表示当前 LCD 驱动板的来源/版本，不能据此把实际屏幕登记为
7 寸；“10.1 寸”表示当前物理屏幕尺寸，也不能据此套用新下载的 10.1 寸
1280×800 `ATK-MD1018R` 参数。当前正式合同仍是 **7 寸版驱动板 + 10.1 寸
1024×600 屏幕 + FT5x26 触摸**。

## 3. 主要硬件差异

| 项目 | 7 寸 ATK-MD0700R（1024×600） | 10.1 寸 ATK-MD1018R（1280×800） |
| --- | --- | --- |
| 物理尺寸 | 7 寸 | 10.1 寸 |
| 分辨率 | 1024×600 | 1280×800 |
| LCD 接口 | 24 位并行 RGB | 24 位并行 RGB，并额外提供 4 通道 LVDS |
| RGB 数据格式 | RGB888，可兼容 RGB565 接法 | RGB888，可兼容 RGB565 接法 |
| 供电/IO | 5 V，3.3 V IO | 5 V，3.3 V IO |
| 手册典型像素时钟 | 约 51.2 MHz | 约 71.1 MHz |
| 典型有效区 | 1024×600 | 1280×800 |
| 典型水平总周期 | 1344 pixel clocks | 1440 pixel clocks |
| 典型垂直总周期 | 635 lines | 823 lines |
| 电容触摸控制器 | GT911 或 FT5446 | GT9271 |
| 常见触摸 I2C 地址 | GT911：0x14/0x5D；FT5446：0x38 | GT9271：0x14/0x5D |
| 触摸点数 | 5 点 | 控制器资料标称最多 10 点 |
| 5 V 工作电流 | 1024×600 约 100～340 mA | 约 160～720 mA |

两者虽然都可使用 40 针 RGB FPC 信号，但不能因为接口外形相似就直接互换。
分辨率、像素时钟、行场时序、触摸芯片、背光功耗和可能使用的 LVDS/RGB
链路均需要与实际屏幕型号一致。

## 4. 参考驱动源码的关系

分别核对两套资料中阿波罗 STM32H743 示例的
`Drivers/BSP/ATK_RGBLCD` 目录后，下列 10 个文件逐文件 SHA-256 一致：

- `atk_rgblcd.c`
- `atk_rgblcd.h`
- `atk_rgblcd_font.h`
- `atk_rgblcd_ltdc.c`
- `atk_rgblcd_ltdc.h`
- `atk_rgblcd_touch.h`
- `atk_rgblcd_touch_ftxx.c`
- `atk_rgblcd_touch_gtxx.c`
- `atk_rgblcd_touch_iic.c`
- `atk_rgblcd_touch_iic.h`

因此，两套资料并不是两份不同实现的 STM32 LCD 驱动，而是同一套多屏兼容
驱动。驱动读取屏幕硬件 ID 后，从参数表选择对应分辨率、时序和触摸类型：

- 7 寸 1024×600：示例参数表 PID 为 `7016`，触摸类型为 GTxx。
- 10.1 寸 1280×800：示例参数表 PID 为 `1018`，触摸类型为 GTxx。

参考代码参数表给这两个型号填写的像素时钟均为 45 MHz，但对应模块手册的
典型值分别约为 51.2 MHz 和 71.1 MHz。移植时不得只照抄示例代码中的
45 MHz，应由实际屏幕规格、设备树时序、时钟锁定状态和物理画面验证共同裁决。

## 5. 与 Z20/V5 驱动的边界

上述 `ATK_RGBLCD` 文件属于 STM32 BSP，只能作为屏幕模块接口、时序和触摸
协议的参考；它不是 Z20 Linux DRM 驱动，不能直接覆盖本目录 `drivers/` 中的
Z20 Linux 驱动。

V5 当前目标为 10.1 寸 1024×600，因此：

- 7 寸资料与当前目标分辨率相同，但物理面板、背光和触摸硬件合同不一定相同。
- 10.1 寸资料与当前目标物理尺寸相同，但 1280×800 分辨率、显示时序及
  GT9271 触摸配置不匹配当前 1024×600、FT5x26 合同。
- 在取得当前 10.1 寸 1024×600 屏幕的准确型号或规格书前，不能把 `7016`
  或 `1018` 的整套参数直接认定为 V5 正式参数。

正式接入必须确认：pixel clock、水平/垂直 front porch、back porch、sync
pulse、HSYNC/VSYNC/DE 极性、RGB/LVDS 链路、数据位宽、触摸控制器及 I2C
地址、复位/中断脚、背光 PWM 极性和上电时序，并经过 Z20 current 镜像与
物理屏观察验证。

## 6. 基于当前硬件的驱动与代码建议

> 状态：本节是后续实施建议，不表示这些源码修改已经构建、部署或经过板端验证。

### 6.1 总体建议

建议把 V5 做成**固定单屏产品配置**，不要继续把它当作可以根据 LCD ID 自动
切换 7 寸/10.1 寸参数的通用开发板：

1. 7 寸版 LCD 驱动板只代表当前电气转接板来源；它读出的 LCD ID 不能作为
   10.1 寸物理屏型号真值。
2. 当前显示真值固定为 10.1 寸 1024×600、`timing2`、51.2 MHz。
3. 显示时序只由当前产品设备树保存，LCD 驱动只读取设备树 `native-mode`，
   不在 C 代码中再复制一张时序表，也不允许失败后退回 800×480/1280×800。
4. FT5x26 继续使用 Linux 5.4 已有的 `edt_ft5x06` 驱动，不另写第二套触摸
   驱动；具体子型号未确认前不得发明内核不支持的 compatible。
5. 屏幕方向只由 `raw-evdev-cal-v2` 校准矩阵处理，设备树不要再叠加
   `touchscreen-swapped-x-y` 或 inverted 属性，避免双重变换。

### 6.2 必须修改的生效 owner

| 作用 | 当前应修改的唯一文件 | 说明 |
| --- | --- | --- |
| 产品 LCD 合同 | `功能/LCD显示硬件链路.md` | 先登记屏、驱动板、时序、时钟和验收要求 |
| 实际 DTB 输入 | `board/petalinux/project-spec/meta-user/recipes-bsp/device-tree/files/system-top.dts` | 当前 `device-tree.bbappend` 只打包这份静态 DTS |
| LCD DRM bridge | `board/petalinux/project-spec/meta-user/recipes-kernel/linux/linux-xlnx/xlnx_atk_lcd.c` | 读取唯一 native mode、设置像素时钟及处理 ID/RGB 复用脚 |
| Clock Wizard | `linux/kernel/drivers/clk/clk-xlnx-clock-wizard.c` | Windows canonical Linux 时钟驱动 owner |
| 内核配置 | `board/petalinux/project-spec/meta-user/recipes-kernel/linux/linux-xlnx/display.cfg` | LCD、PWM、FT5x26 输入链必需 Kconfig |
| 触摸设备链接 | `board/petalinux/project-spec/meta-user/recipes-apps/v5-base-overlay/files/udev/99-touchscreen.rules` | 只建立 `z20-touchscreen` 唯一链接 |
| 现场校准 owner | `board/config/safe_ui/re_touch_calibration.json` | 真实手指五点校准后回灌 Windows owner |

当前 `device-tree.bbappend` 设置了静态 `system-top.dts` 和
`do_configure[noexec] = "1"`。因此只修改 `system-user.dtsi` 不会进入 current
DTB，不能作为修复路径。

### 6.3 LCD 设备树建议

当前 `system-top.dts` 中的 `timing2` 参数与 V5 合同一致，应作为唯一产品模式：

| 字段 | 建议值 |
| --- | ---: |
| `clock-frequency` | 51200000 Hz |
| `hactive` / `vactive` | 1024 / 600 |
| `hback-porch` / `hfront-porch` / `hsync-len` | 140 / 160 / 20 |
| `vback-porch` / `vfront-porch` / `vsync-len` | 20 / 12 / 3 |
| `hsync-active` / `vsync-active` | 0 / 0 |
| `de-active` / `pixelclk-active` | 1 / 0 |

水平总周期为 1344，垂直总周期为 635；51.2 MHz 下刷新率约为 59.9925 Hz。
建议最终只在产品 DTS 中保留一个具名模式并令 `native-mode` 指向它，例如：

```dts
display-timings {
    native-mode = <&v5_lcd_1024x600>;

    v5_lcd_1024x600: timing {
        clock-frequency = <51200000>;
        hactive = <1024>;
        vactive = <600>;
        hback-porch = <140>;
        hfront-porch = <160>;
        hsync-len = <20>;
        vback-porch = <20>;
        vfront-porch = <12>;
        vsync-len = <3>;
        hsync-active = <0>;
        vsync-active = <0>;
        de-active = <1>;
        pixelclk-active = <0>;
    };
};
```

800×480 和 1280×800 只保存在硬件参考资料中，不应继续留作产品运行 fallback。
特别禁止因为屏幕是 10.1 寸就选择 `timing3`/71.1 MHz。

### 6.4 LCD 驱动代码建议

当前 `xlnx_atk_lcd.c` 先按 LCD ID 选择时序，随后又强制
`display_timing = 2`。建议改成下列单一路径：

```c
ret = of_get_videomode(dev->of_node, vm, OF_USE_NATIVE_MODE);
if (ret)
    return ret;
```

具体要求：

- 增加 `#include <video/of_display_timing.h>`，使用该内核版本定义的
  `OF_USE_NATIVE_MODE`，不再依赖“第 2 个 timing 节点”这一顺序魔数。
- 删除 LCD ID 到 `timing0/1/2/3` 的选择 `switch` 以及默认时序 fallback。
- LCD ID 三个 GPIO 与 R7/G7/B7 最高位复用；Vivado `rgb2lcd.v` 会在读 ID
  时把对应 lane 切为高阻，读完后仍必须把三个 GPIO 切为 output-low 以恢复
  RGB 数据通道，不能简单删除 GPIO 初始化。ID 值只允许记录诊断，不再控制
  显示模式。
- 任一 ID GPIO 申请失败应 fail-closed，不能 `break` 后继续操作未初始化 GPIO。
- `of_get_videomode()`、`clk_round_rate()`、`clk_set_rate()`、
  `clk_prepare_enable()` 的返回值全部检查；设置后用 `clk_get_rate()` 输出实际
  像素时钟，禁止设置失败后继续点亮背光。
- 时序数值只保存在设备树，驱动不得再硬编码第二份 porch/sync 参数；可在
  构建检查中断言当前 native mode 是 1024×600，但不要在运行代码复制整表。
- `lcd_id` 当前没有其他产品消费者；完成 GPIO/RGB 复用整理后不应继续
  `EXPORT_SYMBOL(lcd_id)` 制造外部控制入口。

在上述代码改造完成前，不得删除或重排 DTS 中的 `timing0/1/2/3`，因为现有
驱动仍按顺序索引 `display_timing = 2`；应在同一个实现切片中先切换到
`OF_USE_NATIVE_MODE`，再删除非产品 timing。

### 6.5 像素时钟代码建议

51.2 MHz 必须由 Clock Wizard 正确产生。32 位 ARM 上不仅设置分频的
`parent_rate * 1000` 会溢出，时钟回读和 VCO fractional 路径也必须统一审计：

- `clk_wzrd_dynamic_reconfig_f()`：乘法前转 `u64`，使用 64 位除法/取整。
- `clk_wzrd_recalc_ratef()`：回读计算中的 `parent_rate * 1000` 同样先转 `u64`。
- `clk_wzrd_vco_mul_recalc_rate_f()`：先把 `parent_rate` 提升为 `u64` 再乘。
- `clk_wzrd_vco_mul_dynamic_reconfig_f()`：`rate * 8` 必须先提升为 `u64`。

当前工作区已出现前两处动态分频设置计算的 64 位修改，但不能只修设置、不修
回读/VCO 路径。最终应同时验证 `clk_set_rate()` 返回、Clock Wizard lock、
`clk_get_rate()`/`clk_summary` 和实际分频寄存器，不能只看画面暂时亮起。

建议加入固定算术测试向量：100 MHz 输入、1.05 GHz VCO、51.2 MHz 目标时，
输出分频约为 20.508，实际可达频率约 51,199,532 Hz。软件门禁可先要求实际
频率相对目标偏差不超过 0.1%，最终阈值由当前屏幕规格书裁决。

### 6.6 背光建议

当前设备树 `pwms` 周期是 5,000,000 ns（200 Hz），而 LCD 驱动又硬编码为
50,000 ns（20 kHz）、80% 占空比，形成两个配置真源。建议：

1. 以功能 owner 已裁决的设备树合同为准：周期 5,000,000 ns（200 Hz）、
   100% 占空比；删除驱动中的 20 kHz/80% 硬编码。
2. 最终改用标准 `pwm-backlight` 节点保存周期、极性、亮度级和默认最大亮度，
   或由 LCD 驱动严格读取设备树 PWM args/占空比；两种方式只能保留一个，
   不得在 C 文件内提供另一套默认参数。
3. 若暂时保留现有写法，必须检查 `devm_pwm_get()`、`pwm_config()`、
   `pwm_enable()` 的返回值；当前取 PWM 失败分支还可能使用旧 `ret` 值。
4. 只有像素时钟设置成功、显示链准备完成后才开背光。背光亮度不能用来掩盖
   像素时钟导致的闪烁。

### 6.7 Vivado 默认时钟建议

当前产品 FPGA 唯一 owner 是 `vivado_hw_project`；其 LCD Clock Wizard output0
默认仍为约 33.333 MHz，Linux 启动后才动态切换到 51.2 MHz。

- **最小修复**：先完整修复 Linux Clock Wizard 动态设置/回读路径，由内核在
  LCD 驱动 probe 时切到 51.2 MHz。这一方案不需要重新生成 bitstream/XSA。
- **发布级加固**：把 `vivado_hw_project` 中 output0 默认值也改为 51.2 MHz，
  保持第二路 mclk 合同不变，再完整执行 synthesis、implementation、bitstream
  和 XSA 交接。这样复位后默认就是正确像素时钟，但成本和影响范围更大。

不得为了本 LCD 修改 `new-vivado/z20_v1_5_hw_project`；它不是当前产品镜像的
硬件输入，也不能作为 `vivado_hw_project` 的 fallback。

### 6.8 FT5x26 触摸建议

不建议另写触摸驱动。当前可保留：

```dts
touchscreen@38 {
    compatible = "focaltech,ft5426", "edt,edt-ft5406";
    reg = <0x38>;
    /* GPIO58：下降沿中断；GPIO59：低有效复位 */
    touchscreen-size-x = <1024>;
    touchscreen-size-y = <600>;
};
```

其中当前 Linux 5.4 匹配表没有 `focaltech,ft5426` 或通用
`focaltech,ft5x26`，实际由第二个 `edt,edt-ft5406` 绑定并按最多 5 个触点
处理。现阶段应保持这条已知可绑定路径，直到通过芯片丝印、I2C 寄存器 ID 和
内核识别日志确认具体 FT5x26 子型号；不得添加内核不存在的
`focaltech,ft5x26`/`edt,edt-ft5x26`。

同时建议：

- 保留 `CONFIG_INPUT_EVDEV=y`、`CONFIG_INPUT_TOUCHSCREEN=y`、
  `CONFIG_TOUCHSCREEN_EDT_FT5X06=y`。
- 当前硬件已确认为 FT5x26 后，删除 disabled Goodix 设备树节点和
  `CONFIG_TOUCHSCREEN_GOODIX`，避免保留第二触摸 fallback。
- 删除未被当前 `linux-xlnx_%.bbappend` 选用的上级重复 `linux/display.cfg`，
  只保留 `linux-xlnx/display.cfg` 这一份内核 fragment owner。
- 核对 GPIO58/GPIO59 确实连接 7 寸版驱动板上的 INT/RST；仅凭 DTS 数值不能
  证明实物接线正确。
- udev 只建立 `/dev/input/by-path/z20-touchscreen`；`eventN` 必须动态解析，
  产品代码不得写死 `/dev/input/event0`。
- 产品 UI、诊断和校准入口应移除任意设备环境变量/CLI 覆盖，只消费唯一链接；
  测试注入必须留在不进入产品运行路径的测试工具中。
- `ui_ready` 前验证链接真实指向字符设备、驱动为 `edt_ft5x06`，且设备具有
  `EV_ABS`、`ABS_MT_POSITION_X/Y` 和有效轴范围；仅 `open()` 成功不够。
- 不在 DTS 添加坐标交换/反向；更换屏或驱动板组合后执行真实手指五点校准，
  再把矩阵回灌 `board/config/safe_ui/re_touch_calibration.json`。

### 6.9 推荐实施顺序

1. 先确认当前 10.1 寸 1024×600 屏的型号/规格书，以及 FT5x26 丝印或寄存器
   ID；无法取得时，以当前已点亮参数为基线，但保留“待实物确认”状态。
2. 更新 LCD/触摸 owner 文档和静态 `system-top.dts`，将产品显示模式收敛为
   唯一 1024×600 native mode。
3. 完整修复 Clock Wizard 的 ARM32 fractional 设置与回读计算。
4. 整理 `xlnx_atk_lcd.c`：保留 ID/RGB 复用脚切换，但显示模式只读 DT；补齐
   时钟、PWM和错误回滚。
5. 收敛 FT5x26 节点、内核配置、udev、UI/校准消费者，删除 Goodix 和任意
   event fallback。
6. 增加 focused 静态门禁，检查生效 DTS 的唯一模式、51.2 MHz、porch/sync、
   FT5x26 0x38、GPIO、udev 链接以及禁止的 1280×800/Goodix fallback。
7. 再进入最小 ARM kernel/device-tree 构建、current 镜像、部署和板端验收。

### 6.10 验收条件

本地与构建阶段至少需要：

- 编译并反编译最终 DTB，证明生效节点确实来自 `system-top.dts`，native mode
  是上述 1024×600 参数，触摸是 PS I2C1/0x38/正确 GPIO。
- 受影响内核、device-tree、rootfs 和 current 镜像构建通过。
- 检查 ARM ELF/ABI、Linux source identity、PetaLinux source identity 和
  部署清单，不能使用 VM 手改源码或旧镜像。

板端必须进一步证明：

- DRM 唯一 preferred mode 与 framebuffer 均为 1024×600。
- Clock Wizard 已锁定，实际 pixel clock/readback 与 51.2 MHz 合同一致。
- 背光回读为 200 Hz、100% 占空比，冷启动和长时间静止/动态画面均无闪烁。
- `/sys/bus/i2c/devices/*-0038/driver` 指向 `edt_ft5x06`，唯一 udev 链接指向
  动态 `eventN`，evdev 能产生真实多点坐标。
- 真实手指完成五点、四角、边缘和产品按钮验证，校准矩阵源位置回读一致。
- clean SD 冷启动后无需手工补文件即可出现稳定画面和可用触摸。

完成源码或本地检查时最多只能声明 `local_verified_only`；只有 current 镜像
部署、冷启动和物理屏/真实手指验收全部完成后，才能声明 `board_verified`。
