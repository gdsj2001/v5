## 3. 控制链和状态链

控制链只能是：

```text
UI C -> Command Broker UDS framed socket -> product action -> LinuxCNC/HAL/EtherCAT
```

状态链只能是：

```text
state publisher -> /dev/shm/v3_status_shm typed SHM -> UI SHM reader
```

LinuxCNC/HAL/EtherCAT/native helper 已拥有或可拥有的控制语义、运动规则、状态、参数、坐标、工具、RTCP、WCS、G92、IO 和安全联锁，V3 只能做请求、显示、门禁、适配、诊断和证据，不得新建第二套真源。

禁止把以下内容作为产品控制或状态真源：

- 旧 FIFO/JSON 控制 IPC。
- `v3_status_snapshot.json` 热路径轮询。
- UI 直接读写磁盘参数。
- Broker 结果 JSON、`/run` 快照、诊断快照、UI dirty 值或默认值。
- V3 私有 JSON/cache 持久化微内核已有状态。
- 短生命周期 Python fallback 和产品热路径 popen/fork/exec。

## 4. 主要工程分层

| 层 | 当前 owner/入口 | 说明 |
| --- | --- | --- |
| LVGL UI C | `lvgl_app/src/` | 显示、输入、UI 状态和命令请求构造 |
| Command Broker | `lvgl_app/scripts/v3_command_broker.py` | UDS framing、动作注册、状态 epoch、忙域和结构化结果 |
| Product actions | `lvgl_app/scripts/product_actions/` | Home/Jog/Start/E-stop/Program/Axis 等产品动作 |
| State Publisher | `lvgl_app/scripts/v3_state_publisher.py` | LinuxCNC stat owner，生成 typed SHM |
| SHM ABI | `lvgl_app/include/v3_state_shm.h`、`lvgl_app/scripts/v3_state_shm.py` | C/Python 固定布局、seq/CRC/version |
| Settings apply | `lvgl_app/scripts/v3_settings_apply.py` 及 `settings_apply_*` owner | 设置写入、runtime INI、drive-only payload、读回验证 |
| Drive profile | `功能/1驱动命令映射表命令需求.md`、`config/drive-profiles/`、`v3_drive_*` 脚本 | 标准 `drive.*` 命令和 profile 分发/选择 |
| 工具 | `tools/` | preflight、pipeline、VM sync、板端 power cycle、motion probe、诊断包等 |

超过 500 行的运行时文件不能继续承接新业务 owner。新功能优先放到已有合适 owner，或新建 owner/sub-owner；大文件只允许 bug fix 或薄转发。

## 6. 开发板访问和失联处理

常规访问顺序：

2. 测 `ping 192.168.1.221`、SSH、`18080` relay 和 `/remote/info`。
3. 若网络失联，使用 `tools/v5_board_power_cycle.py` 通过继电器重启开发板。
4. 重启后用 COM console 诊断 boot、网卡、服务和 LinuxCNC/backend 状态。
5. 不要把历史 lock 文件当成当前物理占用；但真实运行进程、资源占用、板端忙和运动安全必须实测。

典型端口记录在 `tools/z20_port_map.json`，执行前仍要复核。失联时不要直接重复全量 pipeline；先单点确认电源、boot、网络、relay、SSH、UI 服务和状态链。

## 7. VM 访问和同步边界

首选 VM SSH alias 是 `z20-vm`。同步或构建前先探测：

```powershell
ssh z20-vm "whoami; test -d /home/sj/Desktop/deliverables/re_plnx_rebuild/work/8ax-native-linuxcnc-current/v3 && readlink -f /home/sj/Desktop/deliverables/re_plnx_rebuild/work/8ax-native-linuxcnc-current/v3"
```

边界：

- 不安装、不启用、不依赖 WSL。
- VM 只是构建/同步目标，不是源码真相。
- 同步前用本地 git 状态和 diff 明确当前改动；不要把旧 VM 产物当作当前源码。
- 板端可见或运动相关改动最终需要 full current sync/build/deploy，再走原始 UI/operator path。


当前产品显示/输入链路：

```text
板端 LVGL UI
  -> v3_remote_ui_relay.py / 0.0.0.0:18080
  -> remote_input.sock framed input
```


- 板端没有留下 `disable_remote_relay` 或 `disable_remote_input`。
- `remote_ui_relay_status.json` 和 `remote_input_status.json` 处于 ready。
- `Test-NetConnection 192.168.1.221 -Port 18080` 和 `/remote/info` 可用。
- UI 问题按截图/remote-frame 单步确认后再点击，不先跑整条自动化。

## 9. 参数、设置和驱动 profile

参数表规则以 `功能/0-1开机参数入内存.md` 为最高功能 owner。稳定显示路径是 disk/native owner -> runtime memory/State Publisher -> typed SHM/config projection -> UI；保存请求的 `status_epoch` 必须在提交/构造瞬间从当前 State Publisher/SHM provider 取，不用进入设置页时的旧 epoch。

`settings_runtime.json` 只允许持久化 Settings Drive 私域和最小 drive/count-domain 证据，不得持久化微内核已有状态。驱动 profile 的规则看 `功能/1驱动命令映射表命令需求.md`：新增品牌只改 VPS/profile，不改板端业务代码；板端只用下载后的 public/private 缓存表，并按真实从站唯一选择。

## 10. 旋转、RTCP、WCS 和工具状态

| 主题 | owner/原则 |
| --- | --- |
| 旋转等效角、count-domain、Rebase | `功能/3-3旋转轴等效角目标与解决方案.md` 和索引中的 rotary REQ |

| WCS/G92/tool/TLO | LinuxCNC/native truth，V3 只能请求、显示、读回和诊断 |

缺 native gate、缺单周期原子 proof、缺 SHM readback 或缺板端 motion evidence 时，不能宣称 rotary/rebase/RTCP/WCS 相关行为已闭环。

## 11. 修改流程

默认顺序：

```text
AGENTS.md
  -> 功能/需求真源索引.md
  -> 对应 REQ owner 文档
  -> git status / git diff
  -> active 改进文档或 遗留.md
  -> 备份要改的文件到 repo_ignored/<short_task>/backup/
  -> 本地针对性修改和测试
  -> 必要时 VM full-current build
  -> 必要时 board deploy + 原始 UI/operator path + cc.ngc
```

规则冲突时，最新用户明确反馈优先于项目旧文档；先修冲突的 owner 文档，再修源码或配置。

## 12. 验证分级

| 改动类型 | 最低证据 |
| --- | --- |
| 文档/规则 | `git diff --check`、人工 owner/index diff review |
| 本地代码 | touched-file compile/py_compile、focused pytest/contract、preflight |
| 板端可见功能 | 本地门禁 -> VM full-current build -> deploy -> 原始 UI/operator path 截图/日志/SHM/Broker evidence |
| 运动相关 | 板端可见功能证据 + `nc/cc.ngc` golden motion loop |
| Git push | 只表示备份/同步，不表示代码正确、板端已验或 release ready |

没有跑到对应证据时，最终状态只能写 `source_only`、`local_verified_only`、`not_board_verified` 或 `blocked`，不能写 fixed/done/verified/board_verified/release_ready。

## 13. 快速判断表

| 看到的情况 | 判断 |
| --- | --- |
| 需要新业务逻辑但目标文件超过 500 行 | 新建 owner/sub-owner 或落到已有小 owner；大文件只做薄转发 |
| UI 想直接读磁盘配置显示参数 | 不允许；走 State Publisher/SHM-backed projection |
| Python product action 想 `import linuxcnc` | 不允许；LinuxCNC stat owner 是 State Publisher，控制走 native gate/product action 边界 |
| 产品热路径想读 HAL pin 或跑 `halcmd` | 不允许；必须注册 native helper 或受控 apply/restart 路径 |
| 新驱动需要支持 | 改 VPS/profile 和 `drive.*` 映射，不改板端厂家分支 |
| 板端网络断了 | 先 relay/SSH/18080 探测；再继电器重启和 COM 诊断 |
| UI 自动化想直接跑全流程 | 先截图确认单个点击/功能，单个闭合后再全量 |
| 有本地测试全绿但没上板 | 只能 `local_verified_only`，不能称板端已修好 |
