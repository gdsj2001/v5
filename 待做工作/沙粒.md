# 沙粒感推测原因

状态：推测记录，尚未完成 A/B 板端验证。

## 现象

2026-07-01 最近一轮 UI/刀路相关修改上板后，操作员反馈电机运行时的“沙粒感”明显消失或减轻。当前不能把它直接归因于伺服参数变好；更像是 UI/State Publisher/刀路刷新对 CPU 和总线时序的干扰尖峰下降后，运动侧抖动被削弱。

## 当前最高概率原因

1. UI 刀路图层不再遇到无效帧就清空重画。
   - 之前 A/C 轴线、WCS/MCS 坐标系、红点、刀柄/刀尖等图层在状态短暂无效、投影失败或坐标微抖时会被清空，下一帧再重建。
   - 这种“清空 -> 重投影 -> 重画”的循环会制造 LVGL 主线程 CPU 尖峰，运行 G-code 时可能打扰运动侧调度。
   - 当前改成保留上一帧有效图形，下一帧有效后再更新，减少了高频 UI 抖动。

2. 刀路结构签名对高频微抖做了 0.01 量化。
   - 编码器和坐标值存在偶发跳动，微小变化不应触发静态刀路、坐标系、几何包络全量重建。
   - 当前签名量化后，低于 0.01 的微抖更多被当作显示噪声处理，只更新动态显示，不扩大成结构刷新。

3. RTCP OFF 时黄色轨迹不再跟随 A/C 姿态重算。
   - 非 RTCP 模式下，黄色轨迹保持程序/WCS 预览，不因 A/C 机械值每帧变化而重投影。
   - 这减少了运行中无必要的黄线投影计算。

4. 成功弹窗改为顶部 2 秒文字提示，减少全屏 Modal 干扰。
   - 成功动作不再弹全屏遮罩，有助于减少 UI 事件层、重绘和遮挡处理开销。
   - 该项更偏 UI 体验优化，可能不是电机沙粒感消失的主因。

## 暂不认为是主因的项

1. `mmap.flush()`。
   - 当前只是增加了 `RE_V3_STATE_PUBLISHER_FLUSH_MMAP_EACH_FRAME` 可逆开关，默认仍为开启。
   - 除非板端环境已经显式设置为 `0/off`，否则不能把本次沙粒感改善归因于关闭 flush。

2. Start 运动门禁 latch 缓存修复。
   - 该修复主要减少按下 Start 到运行开始之间的重复门禁成本。
   - 它改善启动响应，不太直接影响 G-code 已经运行后的持续电机手感。

3. 伺服参数本身。
   - 本轮没有以“调伺服参数”为目标闭环验证。
   - 未经驱动参数读回、运行曲线和重复 A/B 证据前，不应把沙粒感改善归因于驱动参数改变。

## 后续验证方法

1. 板端 A/B：保留当前 UI 刀路逻辑，与临时恢复“无效帧清空图层”的版本对比。
2. 板端 A/B：保持其它逻辑不变，只切换 0.01 签名量化，观察 UI CPU 尖峰、刷新丢帧和电机手感。
3. 板端 A/B：确认 `RE_V3_STATE_PUBLISHER_FLUSH_MMAP_EACH_FRAME=1/0` 对 state publisher timing、SHM seqlock retry、UI stale/degraded、急停响应是否有影响。
4. 运行同一段黄金 G-code，采集：
   - State Publisher 返回样本 timing；
   - typed SHM probe 与 UI 截图/帧率；
   - CPU per-core 占用；
   - LinuxCNC/RTAPI overrun；
   - Broker 急停响应；
   - 操作员主观手感记录。

## 判断口径

只有在相同 G-code、相同使能/回零/RTCP 状态、相同板端部署条件下，重复 A/B 能稳定复现“旧逻辑有沙粒感、新逻辑无沙粒感”，才可把根因从“推测”升级为“已验证”。
