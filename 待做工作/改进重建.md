# 改进重建作业指导书

## 1. 作业目的

按 `功能/` owner 文档重建一条干净主干，不再把 `D:/v3` 作为继续修补的源码主线。源码放置遵守“哪边编译、编辑、验证方便就放哪边”的总原则，不人为规定所有源码必须在 Windows 或 VM 单侧闭包；但同一个源码域只能有一份真源，严禁 Windows/VM 双边镜像、互相同步副本或保留备用副本。当前分工是：Linux/native/LVGL/运行服务源码真源放在 VM `/root/Desktop/v5`；Markdown 文档和必须用 Windows/Vivado 编辑的 FPGA 工程真源放在 Windows `D:/v5`。任何新版本运行、编译、部署、测试或脚本引用都不得依赖 `D:/v3` 的源码路径。

现有代码只作为素材库，允许搬运经过筛选的 UI 显示、VPS/auth/download、drive profile 和少量 native/linuxcncrsh 封装；旧 Broker/Python 运动中介、JSON 状态、SHM 泛状态、自造门禁、rotary 本地算法、program SHA sidecar 和测试假入口一律不搬。

最高需求来源：

1. `AGENTS.md`
2. `功能/需求真源索引.md`
3. `功能/0开机参数入内存.md`
4. `功能/共享内存.md`
5. `功能/7主页面按钮功能.md`
6. 其它 `功能/` owner 文档

本文是重建执行指导，不替代 `功能/` owner。实现中如果发现需求不清，先查 `功能/需求真源索引.md`，再读对应 owner。

## 2. 重建总原则

1. 源码按编译/编辑/验证便利性归属真源：哪边是主要工具链和验收环境，源码就放哪边；不得为了目录整齐强行跨环境放置源码。
2. 同一个源码域只能有一份真源：Linux/native/LVGL/运行服务源码不得在 Windows `D:/v5` 镜像；Vivado/FPGA 工程不得在 VM `/root/Desktop/v5` 镜像；临时迁移包、备份副本、同步副本和双边可编辑副本必须删除。
3. 新主干优先，不在旧主干上继续包兼容层。
4. 先建立最小可编译骨架，再逐步搬入允许复用块。
5. 搬旧代码必须先过白名单；不在白名单内的旧代码默认删除或不迁移。
6. 新主干只有一条控制路径：`UI C -> command gate -> native/linuxcncrsh/microkernel`。
7. 新主干只有一条显示投影路径：`microkernel/native memory -> State Publisher -> /dev/shm/v3_status_shm -> UI display`。
8. 运行热路径不得读盘、扫目录、懒导入、写 result JSON、等待 proof 文件、计算 program SHA 或通过 Broker/Python 运动中介。
9. VM/Linux 是 native、LinuxCNC、HAL、`/dev/shm`、socket、ARM/板端相关代码的主闭合环境；后续 Linux/native 编译以 `/root/Desktop/v5` 为准。
10. VM `/root/Desktop/v5` 是 Linux/native/LVGL/运行服务源码和编译运行真源，不是文档副本；VM 内禁止 Markdown 文档，禁止 Vivado/FPGA 工程镜像，`*.md`、`功能/`、`待做工作/`、`*.xpr`、`*.srcs/` 不进入 VM v5。
11. Windows `D:/v5` 当前只保留两类内容：Markdown 文档，以及必须在 Windows/Vivado 环境中编辑和验证的 FPGA 工程源码。后续若某类源码的主要工具链在 Windows，也按同一原则放 Windows；不得放 Linux/native/LVGL/app/services/config/linuxcnc/tools/third_party/build/artifacts 副本。
12. VM 缺少编译/验证工具时允许安装所需软件；优先使用不破坏系统包的用户级安装路径，安装后必须在本文记录工具名、版本、安装路径和后续命令使用方式。缺工具不是停止推进的理由。
13. Codex 重启后应优先使用 `v5_vm` MCP server 访问 VM；直接 SSH 仍保留为 fallback。`v5_vm` MCP 配置位于 `C:/Users/Administrator/.codex/config.toml`，server 脚本位于 `C:/Users/Administrator/.codex/mcp_servers/v5_vm_server.py`，目标为 `z20-vm:/root/Desktop/v5`。MCP 不得再从 Windows `D:/v5` 覆盖 VM 真源。
14. Codex 重启后应优先使用 `v5_board` MCP server 访问开发板；直接 SSH 仍保留为 fallback。`v5_board` MCP 配置位于 `C:/Users/Administrator/.codex/config.toml`，server 脚本位于 `C:/Users/Administrator/.codex/mcp_servers/v5_board_server.py`，目标为 Windows SSH config 中的 `z20-board`，当前解析到 `root@192.168.1.221:22`。可用工具预期为 `board_exec(command, timeout_sec)` 和 `board_diagnose(timeout_sec)`；新 MCP 需要重启 Codex/会话运行环境后才会出现在工具列表。
15. 本文后续历史状态中出现的 Windows `D:/v5` 上 Linux/native/LVGL 源码/编译证据只表示迁移早期的 Windows 临时副本证据，不再是当前执行路径。当前 Linux/native/LVGL 代码、编译、扫描和源码验收以 VM `/root/Desktop/v5` 为准；当前 Markdown 文档与 Vivado/FPGA 工程修改以 Windows `D:/v5` 为准，并按需要同步回 `D:/v3` 过渡文档。

## 3. VM `/root/Desktop/v5` 独立目录闭包

目标：VM `/root/Desktop/v5` 能在 `D:/v3` 被删除后继续作为 Linux/native/LVGL/运行服务源码和编译根目录使用；Windows `D:/v5` 能在 `D:/v3` 被删除后继续作为 Markdown 文档和 Vivado/FPGA 工程根目录使用。

### 3.0 Windows `D:/v5` 文档与 Vivado 工程目录

目标：Windows `D:/v5` 只作为 v5 文档目录和 Windows 工具链源码目录，方便阅读修改需求/作业指导，并编辑、综合、打开 Vivado/FPGA 工程；它不是 Linux/native/LVGL 源码副本目录。

允许：

1. `*.md`
2. 只包含 Markdown 的目录，例如 `功能/`、`待做工作/`。
3. Vivado/FPGA 工程源码目录，例如 `new-vivado/z20_v1_5_hw_project/` 与 `vivado_hw_project/`，允许 `.xpr`、`.srcs/`、`rtl/`、`ip_repo/`、`hls_manual_src/`、`constraints/`、`scripts/`、`sim/`、板端输入和工程说明。

禁止：

1. Linux/native/LVGL/app/services/config/linuxcnc/tools/third_party 源码、配置、脚本和构建入口副本。
2. `app/`、`services/`、`config/`、`linuxcnc/`、`third_party/`、`tools/`、`build/`、`artifacts/` 等源码或产物目录。
3. `.git`、`repo_ignored`、缓存、日志、压缩包、传输包和 sidecar。
4. Vivado 运行产物、缓存和导出产物，例如 `.Xil/`、`*.runs/`、`*.cache/`、`*.gen/`、`*.hw/`、`*.sim/`、`*.jou`、`*.log`、`*.str`、`*.bit`、`*.ltx`、`*.dcp`、`*.rpt`、`*.wdb`。
5. VM `/root/Desktop/v5` 已经持有真源的任何 Linux/native/LVGL/运行服务源码镜像。

检查：

1. `Get-ChildItem -Recurse D:/v5 -Directory | Where-Object { $_.Name -in 'app','services','config','linuxcnc','third_party','tools','build','artifacts','.git','repo_ignored' }` 必须无结果。
2. `Get-ChildItem -Recurse D:/v5 -File -Filter *.xpr` 必须只指向当前保留的 Vivado 工程。
3. `Get-ChildItem -Recurse D:/v5 -Directory | Where-Object { $_.Name -match '/.(runs|cache|gen|hw|sim)$' -or $_.Name -eq '.Xil' }` 必须无结果，除非用户明确要求保留某个可复现产物作现场证据。
4. VM `/root/Desktop/v5` 中 `find . -name '*.md' -o -name '*.xpr' -o -name '*.srcs'` 必须无结果。
5. `D:/v5/待做工作/改进重建.md` 与 `D:/v3/待做工作/改进重建.md` 在过渡期必须保持一致。

### 3.1 目录结构硬规则

VM `/root/Desktop/v5` 根目录必须清爽，只允许放项目入口文件和一级职责目录。源码、脚本、配置、构建、工具、第三方库、运行产物不得散落在根目录；Markdown 文档不得进入该目录树。

VM 根目录只允许：

1. `CMakeLists.txt`
2. `.gitignore`
3. 一级目录。

根目录禁止：

1. 业务 `.c/.h/.py/.sh/.json/.ngc/.bmp` 文件。
2. 临时截图、编译产物、日志、result、ready、heartbeat、sidecar。
3. `__pycache__`、`.pytest_cache`、`repo_ignored`、旧备份目录。
4. 从 v3 复制来的散文件。
5. 任何 `*.md` 文档。

标准目录树：

```text

|-- CMakeLists.txt
|-- app/
|   |-- include/
|   |-- src/
|   |-- cmake/
```

目录职责：

| 目录 | 职责 | 禁止 |
| --- | --- | --- |
| `app/include` | UI 与共享 ABI 头文件 | Python、脚本、临时生成头。 |
| `app/src` | LVGL UI、显示、command client、只读 SHM reader | Broker/Python 中介、旧测试入口。 |
| `app/assets` | 字体、图片、图标 | 源码、状态文件、运行数据。 |
| `app/platform` | framebuffer、touch、remote input 等平台 I/O | 产品动作、运动门禁。 |
| `services/command_gate` | native/linuxcncrsh 唯一控制 gate | JSON Broker、fallback、旧 helper wrapper。 |
| `services/state_publisher` | native/microkernel -> SHM 显示投影 | 泛状态、动作结果、诊断总线。 |
| `services/auth_download` | VPS、DNA、授权、下载身份链 | 与授权无关的产品门禁。 |
| `services/microkernel` | native 状态/API、运行闭包 manifest、native gate registry | 上层重算运动真值。 |
| `services/drive_profile` | profile schema、下载后内存快照、命令映射 | 运行热路径读文件。 |
| `config/` | 静态配置素材和 profile map | result、ready、heartbeat、运行状态。 |
| `linuxcnc/` | INI/HAL/component 原始素材 | V3 上层门禁和脚本中介。 |
| `tools/` | 本地/VM 构建、复制、扫描工具 | 产品运行依赖。 |
| `third_party/` | 第三方源码或头文件 | 自写业务代码。 |
| `build/` | 本地构建输出 | 源码和文档。 |
| `artifacts/` | 打包、VM 编译、交付产物 | 产品运行时真值。 |

目录执行规则：

1. 新文件必须先归类到上表目录；找不到目录时先修改本文，不得先丢根目录。
2. `build/` 和 `artifacts/` 可删除重建，不能被源码或文档引用为真源。
3. `tools/` 只能辅助构建/扫描/迁移，产品运行不得 import 或 exec `tools/` 里的逻辑。
4. `services/*` 之间不得互相复制状态文件；跨服务共享只通过登记 ABI、UDS/API 或启动 RAM manifest。
5. 从 v3 搬来的文件必须在搬入时改到 v5 目录结构内，不能保持 v3 的散落根目录形态。

## 4. 新主干必须包含

### 4.1 启动 RAM 闭包

目标：开机一次性把运行所需内容读入 RAM，运行期热路径只消费内存。

必须包含：

1. LinuxCNC/微内核运行闭包：executable、共享库、RTAPI、HAL component、kinematics component、INI/HAL、`PARAMETER_FILE`、`tool.tbl`、G53/RTCP 几何、WCS/G92 初始 readback、native gate 表。
2. UI 运行闭包：页面、字体、图片、布局、按钮表、命令表、资源索引。
3. 产品自写运行闭包：Command Broker 最小 gate、Actions/provider 登记表、drive profile 内存快照、VPS/auth/download 所需 schema。
4. 参数 owner 映射闭包：字段、唯一 owner、唯一 writer、readback 来源、generation/readback token。

禁止：

1. 运行期临时从硬盘发现脚本。
2. 运行期动态导入未登记 action。
3. 运行期读 `settings_runtime.json`、INI、`linuxcnc.var`、`tool.tbl`、result JSON、ready/heartbeat 文件补当前状态。
4. Start/Resume 时重读 G-code 文件或重建 runtime。

### 4.2 SHM 显示投影

目标：SHM 只服务 UI 显示。

允许字段：

1. 编码器反馈机械坐标。
2. 运动学理论坐标。
3. 轨迹动态显示点。
4. `runtime_modal_text`
5. `spindle_speed_rpm`
6. `linear_velocity_mm_per_min`
7. `feedrate_override`
8. `spindle_override`
9. frame envelope、epoch、CRC、valid bits。

禁止进入 SHM：

1. WCS/G92/tool/TLO。
2. RTCP actual。
3. active driver mode。
4. safety/stillness/busy/idle。
5. 动作结果、保存证明、诊断 JSON。
6. CPU/PS IP/network 状态。
7. rotary/wcheckpoint/rebase。
8. 参数表稳定显示。

### 4.3 UI command gate

目标：UI 只发请求，不拥有 native 真值。

必须支持：

1. Start/Pause/Resume：C UI 直达 native/linuxcncrsh gate。
2. Program Open：读取程序文件，生成运行内存副本、静态刀路模型、runtime metadata；Start 不再读文件。
3. WCS/Work Zero/G92：走登记 native/linuxcncrsh gate，并由 native/interpreter readback 更新 UI。
4. RTCP：只通过 native actual provider 判定状态，通过 native/linuxcncrsh gate 下发开关。
5. Emergency stop：直达微内核/native safety owner。

禁止：

1. `program_open_cc`
2. `start_run`
3. `pause_resume_program`
4. `first_point` blocked 占位。
5. `wcs_switch` Broker/Python 中介。
6. UI 手写 JSON 给旧 Broker。

### 4.4 State Publisher

目标：常驻进程从 native/microkernel 内存采样，30ms 单向写 SHM 显示投影。

必须：

1. 启动阶段加载到 RAM。
2. 不在 steady-state fork helper。
3. 不读 result JSON、状态快照、硬盘配置补帧。
4. 状态源缺失时发布 degraded/unavailable，不用默认值或旧缓存伪造 fresh。

### 4.5 Program Open/Start

目标：程序打开和程序启动分工清楚。

Program Open 负责：

1. 文件选择。
2. 一次性读取 G-code。
3. 生成运行内存副本。
4. 生成静态刀路模型。
5. 记录当前 runtime metadata。

Start/Resume 负责：

1. 消费 Program Open 生成的内存副本。
2. 直达 native/linuxcncrsh gate。
3. 不读程序文件。
4. 不计算 SHA。
5. 不过滤 G-code。
6. 不等待 proof/result 文件。

5. rename-only wrapper。

## 6. 旧代码一律不搬清单

以下旧路径不允许进入新主干：

1. 旧 Broker/Python 运动中介。
