## 1. 完成目标

最终链路固定为：

```text
Position native sample
  -> Position display projection唯一完成等效角、三位坐标和一脉冲抗抖
  -> C Scene Producer复用稳定坐标和resident program model
  -> 一份immutable final display scene
  -> State Publisher一次组帧、CRC、SeqLock commit
  -> UI单周期一次读取，custom-draw直接绘制
```

AI负责修改Windows canonical source、运行focused测试、构建受影响target；只有用户明确要求本轮上板时才部署。AI不执行Home、Start、运动和最终CPU/视觉验收。

## 3. 禁止边界

- 不改变native、LinuxCNC、HAL、EtherCAT中的raw/count、多圈和运动真值。

- 不修改原始G-code，不生成过滤文件、sidecar或执行副本。
- SHM只承载允许的显示投影，不作为动作、运动、安全、保存或WCS/RTCP证明。
- 原始WCS、G92、RTCP、G53、tool、safety和rotary proof不得进入SHM。
- Producer、Publisher、UI和relay留在CPU1；不得进入CPU0。
- 不改变页面布局、颜色、线宽、三位显示和30Hz动态显示目标。
- 不用降到10Hz、提高nice、关闭CRC/SeqLock或断开remote冒充CPU优化。
- 不增加第二producer、第二IPC路径、第二scene ABI、第二scene cache或兼容fallback。

发现违反项时先停止当前错误步骤并修正。只有真实外部阻塞，或同一问题经过至少三次有效尝试仍不能推进，才写`blocked`。

## 4. 作业步骤

### 步骤1：固定等效角和三位坐标的唯一owner

只修改现有Position显示投影链：

1. `NativeRotaryDisplayProjection`从同一count-domain/native sample生成active model旋转轴等效相位。
2. `PositionDisplayStabilizer`统一完成：
   - 所有轴第四位四舍五入为固定三位。
   - `360.000`、`720.000`、`-360.000`、`-720.000`和`-0.000`归一为`0.000`。
   - `0/360`按环形最短距离判定相邻显示桶。
   - 按每轴`unit_per_count`完成一次一脉冲边界抗抖。
3. 同一稳定snapshot同时生成：
   - `mcs[5]`
   - `cmd_mcs[5]`
   - `unit_per_count[5]`
   - `display_digits[5]`，固定为3
   - `following_error[5] = mcs[i] - cmd_mcs[i]`
4. C Scene Producer只消费上述稳定结果，不得再做取模、四舍五入或抗抖。
5. 删除UI、renderer、label和relay中的第二次量化、取模、抗抖和跟随误差计算。

完成判据：

- raw/count、多圈、unwrap、wcheckpoint逐样本不变。
- 同一source generation只有一份稳定显示值。
- 坐标、标签、红绿点、刀路姿态和跟随误差使用同一稳定snapshot。

### 步骤2：拆分Scene缓存键

保留一份最终scene，但缓存至少分为以下三层：

| 缓存层 | 允许失效的输入 | 不得触发 |
| --- | --- | --- |
| resident program model | Program Open、`program_generation` | Position 30Hz sample、手势 |
| 静态世界/姿态几何 | program、active model、RTCP、WCS、有效旋转显示相位 | 仅X/Y/Z或速度/倍率变化 |
| 屏幕投影与动态marker | view/fit、动态MCS/CMD点、刀柄/标签dirty | G-code重解析、自动fit |

规则：

1. 完整scene envelope可以携带最新`source_generation`，但不得把它直接作为“重算全部512个程序点”的理由。
2. 只改变X/Y/Z、速度、倍率或freshness时：
   - 更新动态marker、坐标和dirty bounds。
   - 不重建resident program model。
   - 不重新扫描fit bounds。
   - 不重新变换静态黄色程序点。
3. active model旋转显示相位、RTCP、WCS或model变化时，只使真正依赖它们的姿态/黄色轨迹层失效一次。
4. view变化只project已有世界几何一次；不得重新解析G-code、重新采样native或重建fit。
5. 输入在生成过程中继续前进时只保留一个latest-wins请求，不补算旧scene。

完成判据：

- 相同完整key连续请求100次，build/transform/project各不超过一次。
- `source_generation`连续递增100次但显示有效值完全相同，静态黄色轨迹的build/transform/project计数不增加。
- 连续X/Y/Z变化时只增加动态marker/project计数，不增加静态黄色轨迹transform计数。
- RTCP OFF时`rtcp_transform_count=0`。
- RTCP ON时同一相关generation最多执行一次RTCP变换。

### 步骤3：独立view_generation和fit_generation

generation推进规则固定如下：

| 事件 | view_generation | fit_generation |
| --- | --- | --- |
| Program Open/首次完整scene | 按需推进 | 推进一次 |
| 点击XY/XZ/YZ/3D | 推进一次并恢复默认视角 | 推进一次 |
| 手势旋转、缩放、平移 | 推进一次 | 不变 |
| 机械坐标、姿态、RTCP、WCS、模态变化 | 按依赖推进scene | 未真实越界时不变；真实越界按下一行 |
| 新的可见候选真实越出当前冻结边界 | 不变 | 只推进一次并立即冻结 |
| 页面隐藏、timer tick | 不变 | 不变 |
| 页面由隐藏恢复可见 | 不变 | 显隐事件本身不推进；首个可见候选未越界时不变，若隐藏期间已真实越界则只推进一次并立即冻结 |

修改要求：

1. 删除`request.fit_generation = request.view_generation`。
2. 保存独立、非零、单调的`fit_generation`。
3. fit候选包含黄色轨迹、WCS/MCS、active model几何、红绿点、刀柄和标签。
4. 每次允许的fit完成后立即冻结参数；运行、暂停、坐标变化和手势本身不得触发fit。
5. 只有唯一Producer可以使用已经生成的本帧可见候选，对当前冻结边界做一次有界越界判定；UI、renderer、relay和timer不得重复判断。
6. 新的可见候选真实越出当前冻结边界时，只为该越界事件重算一次全部当前可见元素的fit、推进一次`fit_generation`并立即冻结。
7. 同一候选、同一scene generation或持续越界状态不得连续触发；只有后续fresh scene的新候选再次越出新的冻结边界，才允许形成下一次越界事件。
8. 页面隐藏时不得执行越界判断或fit；手势和显隐变化本身不得伪造越界事件。恢复可见后的首个候选必须由Producer对冻结边界判断一次：未越界不fit，隐藏期间已真实越界则只fit一次并立即冻结。
9. 手势只对冻结fit后的统一scene施加view transform。

### 步骤4：收紧Program Scene IPC

继续使用现有`v5_program_scene_ipc.*`，不得新增第二socket。

1. 512点program payload只在`program_generation`变化时复制和发送。
2. view/fit变化只发送小型view metadata，Producer引用已缓存的program model。
3. scene尚未匹配时不得每33ms重复`open/send/close`同一份大请求。
4. 重试必须有界退避并latest-wins；同一request generation只发送一次，失败后按明确重试generation处理。
5. request validation继续检查magic/version/size、点数、WCS index、break信息和有限数值。

若必须扩展协议，只能在同一IPC中增加明确的`program_model`与`view_update`消息类型；不得复制一套并行协议。

### 步骤5：收口typed SHM ABI version 3

当前代码与部署链已经统一为version 3/7128；本作业只闭合该ABI的生产、校验和消费合同，不改变布局。因此：

1. 固定`V5_STATUS_SHM_VERSION=3`、frame size `7128`和scene size `6168`，本作业不得改变字段、size或offset。
2. 只有以后真实改变布局时才允许升version 4，并原子更新全部消费者；不得用升version掩盖reader或smoke缺陷。
3. 保持当前frame/scene size与所有`_Static_assert`一致。
4. frame必须明确包含：
   - `unit_per_count[5]`
   - `display_digits[5]`
   - `following_error[5]`
   - bounded final display scene
5. 从`V5StatusDisplayScene`删除`active_wcs_offset[5]`及对应writer/UI消费。
6. raw WCS offset继续由UI文字/provider和Scene Producer直连native owner读取；只允许派生后的最终screen geometry进入scene。
7. State Publisher只在Scene Producer成功时设置`V5_STATUS_VALID_DISPLAY_SCENE`并推进真实`scene_generation`。
8. scene生成失败时：
   - 清除display scene valid。
   - `scene_generation=0`，不得写`source_generation`冒充。
   - 不得清除独立有效的Position坐标valid，避免scene失败让坐标整列变成`--`。
9. C reader、Python reader和boot-ready必须同时检查：
   - version、size、SeqLock、CRC
   - point/segment/marker/trajectory数量上限
   - generation和必要有限数值
10. writer identity/inode变化后reader必须重连；普通帧不得每周期`stat/open/close`。

### 步骤6：把State Publisher native读取改为常驻

当前State Publisher每33ms分别读取RTCP、WCS、G53和modal/tool状态块，其底层仍使用`fopen/fread/fclose`。

修改要求：

1. 为四个native状态块建立进程生命周期内的常驻mmap/reader。
2. 启动或inode/writer identity变化时打开或重连。
3. 稳态33ms周期只做有界内存读取和freshness校验。
4. 单块暂时无效只标记对应输入degraded，不得阻塞Position坐标发布。
5. 不增加文件、JSON、helper、fork或命令执行fallback。

### 步骤7：UI只读并统一custom-draw

UI每个动态周期只能：

1. 有界读取一次完整SHM frame。
2. 建立同一份const status/scene view。
3. 坐标面板直接使用`mcs/cmd_mcs/following_error/display_digits`。
4. `scene_generation`变化时把最终screen scene交给一个custom-draw renderer。
5. renderer只遍历point、segment、marker和dirty bounds。
6. 只invalidate producer给出的dirty区域。

必须物理删除：

- `v5_ui_status_view_rotary_phase_deg`及其调用。
- UI侧`fmod(..., 360)`、再次四舍五入和抗抖。
- UI侧`mcs-cmd_mcs`跟随误差计算。
- UI侧AC/BC projector、RTCP/WCS/world-to-screen和fit计算。
- 迁移后的多`lv_line`数组、`lv_line_set_points()`更新入口和旧toolpath cache。
- 第二scene producer、第二scene cache、旧ABI fallback和退役测试入口。

512点程序预览和16点动态trajectory必须继续使用独立容量，不能互相截断。
