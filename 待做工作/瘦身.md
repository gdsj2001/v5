# Zynq 平台 Linux 系统深度瘦身与固化方案审查版

更新时间：2026-06-22

本文用于指导 V3 发布镜像的 Linux rootfs 瘦身、只读固化、启动优化和 A/B 回滚设计。本文不是直接删除命令，也不是板端已执行记录；任何裁剪必须先生成 inventory、dry-run manifest、staging rootfs 变更和回滚证据，再进入真实固件构建和板端验证。

## 1. 审查结论

总体方向合理：发布镜像应减少开发期暴露面、降低断电写损坏风险、保留可回滚升级路径，并把运行期写入集中到 tmpfs 或明确的数据分区。

但原方案不能按字面直接执行，主要问题如下：

1. `20MB ~ 30MB` 只能作为挑战目标，不能作为当前承诺。LinuxCNC、Python runtime、EtherCAT、HAL 组件、LVGL UI、诊断工具和许可证文件都有硬保留项，最终大小必须以 staging rootfs inventory 和 release manifest 为准。
2. 不能在运行板端热删 rootfs、LinuxCNC、HAL、Python 包、Tk/Tcl、systemd/init 或调试工具。清理只允许在 staging rootfs 或固件打包流程中执行。
3. 不能把“删除 `.py` 源码，只留 `.pyc`”写成通用策略。`.pyc` 不是安全边界；只能用于自研私有脚本的发布副本，并且必须保留仓库源码真源、source manifest、源码 SHA、pyc SHA 和 GPL Source Offer 边界。
4. 不能为了白标化删除 GPL 许可证、版权声明、Source Offer 或第三方组件清单。LinuxCNC 等开源组件的合规交付必须独立满足。
5. 不能把当前 EtherCAT 运行事实写成 EtherCAT-only 裁剪策略。后续 `stepdir/pulse/hybrid` profile、FPGA/DMA、bitstream、device-tree overlay 和诊断资产必须按 profile allowlist 保留，除非有单独废弃签核和验证。
7. 只读 rootfs 不等于全系统不可写。`/run`、`/tmp`、`/dev/shm`、日志、参数持久化、升级状态、A/B boot flag 必须有明确写入位置、容量上限和断电一致性方案。

因此，本方案的正确落地方式是：先做只读 inventory 和 dry-run 计划，再做 staging rootfs 裁剪，最后按硬件 profile 和板端闭环验证发布镜像。

## 2. 与当前项目架构的硬边界

瘦身固化不得破坏现有产品架构：

- 控制路径仍是 `UI C -> Command Broker UDS framed socket -> product action -> LinuxCNC/HAL/EtherCAT`。
- 状态路径仍是 `state publisher -> /dev/shm/v3_status_shm typed SHM -> UI SHM reader`。
- LinuxCNC/HAL/EtherCAT 仍是运动真源；发布镜像不能通过删除底层工具来掩盖运动状态或错误。
- remote relay/input 是默认开启的产品能力；涉及 init、网络、UI runtime、relay/input 的改动，交付前必须验证 `v3_remote_ui_relay.py` 仍监听 `0.0.0.0:18080`。
- 不能恢复旧 IPC、JSON hot path、native FIFO、短生命周期 Python fallback 或 UI `/run` scratch JSON。

## 3. 现有工具和测试应作为落地入口

当前仓库已有一组偏安全的发布硬化工具，后续优先复用或扩展，不另起危险脚本：

- `tools/v3_hardening_manifest.py`：生成 dry-run hardening manifest，按 `ethercat/stepdir/hybrid` profile 区分 keep 和 candidate。
- `tools/v3_rootfs_hardening_plan.py`：生成只读 rootfs hardening plan；当前策略是不删除、不 chmod、不重启服务、不触发运动。
- `tools/v3_pyc_release_bundle.py`：生成离线 pyc-only staging bundle；不删除源码、不部署到运行 rootfs。
- `tools/v3_live_rootfs_cutover_runner.py`：带备份和强显式开关的 live cutover 工具；只能在明确窗口和完整备份后使用，不作为默认瘦身入口。
- `tools/v3_support_diagnostics_collect.py`、`tools/v3_support_diagnostics_package.py`：Support Mode 只读诊断采集与打包。
- `tools/v3_release_cutover_plan.py`、`tools/v3_gpl_delivery_bundle.py`、`tools/v3_script_alias_bundle.py`：发布切换、GPL 交付和脚本别名相关计划。

## 4. 目标定义

### 4.1 体积目标

- 第一阶段目标：用 inventory 量化当前 rootfs、kernel、initramfs、应用、LinuxCNC、Python、日志、文档、GUI-only 包和示例资源的真实体积。
- 第二阶段目标：先删除低风险 GUI-only、示例、缓存、locale、开发工具链和包管理缓存，记录每项收益。
- 第三阶段目标：再评估内核和 init 栈优化。
- `20MB ~ 30MB` 只作为长期挑战目标；如果保留 Python runtime、LinuxCNC、HAL、EtherCAT 和诊断能力后无法达到，不得为追数字删除运行真源或诊断退路。

### 4.2 启动目标

- 冷启动目标可以设为 `<= 12s`，但必须拆分计时：
  - 上电到 U-Boot。
  - U-Boot 到 kernel start。
  - kernel 到 rootfs mount。
  - backend/LinuxCNC/EtherCAT ready。
  - Broker ready。
  - state publisher typed SHM ready。
  - C UI 首帧和坐标刷新 ready。
  - remote relay/input ready。
- 未完成分段计时前，不能把 `<= 12s` 写成已达成。

### 4.3 只读和断电目标

- rootfs 只读挂载是方向，但必须保留明确可写区：
  - `/run`
  - `/tmp`
  - `/dev/shm`
  - 有容量上限的日志或诊断临时区
  - 参数持久化分区或 overlay
  - A/B 升级状态和 boot flag
- 只读 rootfs 不能阻止 settings apply、驱动 profile、授权信息、Support Mode 输出和升级元数据的合法写入；这些写入必须迁移到受控持久化位置。

### 4.4 A/B 回滚目标

- A/B 分区是正式板建议方向，不应要求当前开发板立即具备。
- A/B 设计必须包括镜像 hash、boot attempt counter、health check、回滚条件、断电中断恢复和手动恢复入口。
- 升级成功的判定不能只看 Linux 能启动；必须至少包含 Broker、SHM、UI、remote relay/input、backend 状态和必要 motion/profile smoke。

## 5. 禁止直接执行的高风险动作

以下动作不得直接对运行板端执行：

- `rm -rf /usr/share/linuxcnc` 或删除 LinuxCNC 运行目录。
- 删除 `/usr/bin/linuxcnc`、`halcmd`、`halrun`、`ethercat`、LinuxCNC Python 扩展、HAL 模块、NML 文件。
- 删除 GPL 许可证、Source Offer、版权声明或第三方组件 manifest。
- 删除 `.py` 源码后只保留 `.pyc`，且没有 source manifest、仓库源码真源、pyc ABI 证明和启动 smoke。
- 删除 Tk/Tcl、`halshow`、`halmeter`，但没有 support-only 替代诊断路径。
- 删除 FAT/VFAT、USB mass storage 或外部存储支持，除非确认程序导入、维护导入、升级介质和产线烧录都不依赖它。
- 删除 NFS/TFTP/网络启动支持，除非产线烧录、恢复演练和维护升级路径已经替代。
- 删除 Pulse/STEP-DIR、stepgen、encoder、GPIO、FPGA/DMA、bitstream、device-tree overlay 或 hybrid profile 资产，除非对应硬件路线正式废弃。

## 6. 推荐实施批次

### 6.1 批次 0：恢复防线和基线采集

目标：先证明可以恢复，再谈裁剪。

必须完成：

1. 备份当前可启动固件、rootfs、BOOT.BIN、image.ub、bitstream、device-tree、LinuxCNC 配置、产品应用和参数。
2. 备份和本轮证据放入 `repo_ignored/linux_slimming/<date>/`；大文件如果必须外置保存，文档中记录 sha256、大小、来源路径和恢复路径。
3. 在板端演练至少一次恢复路径：TF/SD、TFTP、JTAG、QSPI/eMMC 工具链择一实际跑通。
4. 采集当前 rootfs inventory：目录大小、文件 hash、已加载 HAL 组件、Python import、运行进程、监听端口、服务启动耗时、许可证文件。

禁止：本批次不删除、不 chmod、不替换 init、不改内核配置。

### 6.2 批次 1：dry-run manifest

目标：生成候选清单而不是执行删除。

建议入口：

```powershell
python tools\v3_hardening_manifest.py --root <staging_rootfs> --repo . --profile hybrid --out repo_ignored\linux_slimming\<date>\hardening_manifest.json
python tools\v3_rootfs_hardening_plan.py --root <staging_rootfs> --repo . --profile hybrid --out repo_ignored\linux_slimming\<date>\rootfs_hardening_plan.json
```

要求：

- `--root` 必须是 staging rootfs，不允许是开发机 `/` 或运行板端 `/`。
- 默认 profile 用 `hybrid`，除非明确只做 EtherCAT 或 stepdir 发布镜像。
- manifest 必须区分 `keep_required`、`remove_candidate`、`permission_candidate`、`license_required`、`support_required`。
- 每个候选必须有 rollback hint、释放空间估计、风险等级和验证项。

### 6.3 批次 2：低风险 rootfs 裁剪

优先候选：

- LinuxCNC GUI-only 入口：`axis`、`gmoccapy`、`touchy`、`qtvcp`、`gladevcp`。
- 示例、教学、仿真配置和不参与 V3 发布镜像的 GUI 资源。
- 包管理缓存、编译缓存、临时文件、无用 locale。
- 开发工具链：`gcc`、`g++`、`make`、`git`、`gdb` 等，但必须确认 Support Mode 不依赖它们。

保留或隔离：

- `linuxcnc`、`halcmd`、`halrun`、`ethercat`、LinuxCNC Python 扩展和 HAL 运行模块。
- `halshow`、`halmeter` 等如果售后仍需要，应从普通操作员环境移入 support-only wrapper，而不是直接物理删除。
- GPL、Source Offer、版权清单和第三方组件 manifest。

执行规则：

- 只在 staging rootfs 应用。
- 应用后生成 apply manifest、删除前后 SHA、大小变化和回滚 manifest。
- 运行本地工具测试和最小 rootfs smoke。

### 6.4 批次 3：Python 发布形态收敛

合理目标：

- 仓库源码仍是唯一真源。
- 发布副本可以对自研私有 handler 做 pyc-only staging，但不能删除仓库源码、测试和构建脚本。
- pyc-only 必须记录 Python ABI、源码 SHA、pyc SHA、入口 wrapper、异常日志和启动 smoke。
- GPL 覆盖组件、第三方开源组件和需要源码交付的内容不得因为 pyc-only 包装规避合规义务。

建议入口：

```powershell
python tools\v3_pyc_release_bundle.py --repo . --root <staging_rootfs> --profile hybrid --out-dir repo_ignored\linux_slimming\<date>\pyc_bundle
```

### 6.5 批次 4：只读 rootfs 和写入路径迁移

必须先列出所有运行期写入：

- `/run/8ax_v3_product_ui/*`
- `/dev/shm/v3_status_shm`
- UI/Broker/state publisher 日志
- LinuxCNC 日志
- settings runtime、INI 生成、drive profile、授权 token
- Support Mode 输出
- A/B 升级状态

固化策略：

- rootfs 挂载 `ro`。
- `/run`、`/tmp`、`/var/tmp`、`/dev/shm` 使用 tmpfs，并设置容量上限。
- `/var/log` 可用 tmpfs、环形日志或受控持久化分区，不能无限写 eMMC。
- 参数和授权数据使用单独小分区、overlay 或事务写入机制；必须有掉电一致性证明。
- `/dev/shm/v3_status_shm` 容量和权限要固定，不能被 rootfs 只读改动破坏。

验收：

- 普通 rootfs 路径写入失败。
- 合法 runtime 路径写入成功。
- Broker、SHM、UI、remote relay/input 正常。
- 断电重启后参数一致，不出现 stale 状态或 JSON hot-path 回退。

### 6.6 批次 5：启动优化

启动优化先做测量，再做替换。

允许低风险项：

- 减少无用 service。
- 去掉无意义串口 verbose，但保留维护可见的错误日志。
- 并行化不互相依赖的准备步骤。
- 记录每个服务 ready 时间。

高风险项：

- 用 BusyBox init 替换 systemd/init。
- 改 U-Boot bootcmd、rootfs mount、kernel compression。
- 裁剪内核驱动。

高风险项必须单独成批次，并验证：

- LinuxCNC backend。
- EtherCAT 或 stepdir/hybrid profile。
- Broker。
- state publisher typed SHM。
- UI 首帧和坐标刷新。
- Support diagnostics。
- 断电恢复。

### 6.7 批次 6：Kernel 和驱动裁剪

只能在明确硬件 profile 后裁剪。

可评估删除：

- 无线、蓝牙、音频、无关显卡、无关网卡、无关文件系统。

需要谨慎保留：

- 板载网卡 PHY。
- EtherCAT 相关驱动。
- stepdir/pulse/hybrid 相关 GPIO、FPGA/DMA、device-tree、bitstream、encoder/stepgen 组件。
- USB/FAT/VFAT，如果程序导入、维护升级或产线恢复依赖外部介质。
- NFS/TFTP，如果产线烧录或恢复演练仍依赖网络启动。

任何 kernel 改动都必须重新生成 boot artifact，并记录 XSA/bitstream/device-tree/kernel/rootfs 的对应关系。

### 6.8 批次 7：A/B 和 OTA 回滚

A/B 不只是分区布局，必须有状态机：

- 当前 slot。
- 目标 slot。
- 镜像 hash。
- boot attempt counter。
- health check。
- confirm boot。
- rollback reason。
- 手动恢复入口。

最小 health check：

- kernel 和 rootfs 启动。
- backend ready。
- Broker ready。
- typed SHM fresh。
- UI 首帧和坐标。
- remote relay/input ready。
- Support diagnostics 可采集。

运动相关发布镜像还必须在安全条件下跑 `nc/cc.ngc` golden motion loop，不能只靠系统启动判定成功。

## 7. 验收矩阵

| 项目 | 最小证据 | 状态口径 |
| --- | --- | --- |
| rootfs 体积 | inventory、目录体积、manifest、压缩前后大小 | 只可称 `measured`，不能用估算当结果 |
| dry-run 候选 | hardening manifest、rootfs plan、候选风险标签 | `local_verified_only` |
| staging 裁剪 | staging apply manifest、回滚 manifest、hash、smoke | `local_verified_only` |
| 只读 rootfs | 写入失败/合法 tmpfs 成功、服务正常、日志策略 | 未上板前仍是 `local_verified_only` |
| 启动优化 | 分段计时、服务 ready 证据、relay/input 证据 | 板端执行后才可 `board_verified` |
| A/B 回滚 | 损坏镜像回滚、boot flag、health check、日志 | 板端执行后才可 `board_verified` |
| 运动闭环 | 原 UI/operator 路径触发和 `nc/cc.ngc` golden loop | 通过后才可称 motion 相关 `board_verified` |

## 8. 当前不建议立即推进的事项

- 不建议现在直接追求 `20MB ~ 30MB`，先得到真实 inventory 和可回滚 staging 裁剪收益。
- 不建议现在替换 systemd/init，除非先完成完整启动依赖图和回滚镜像。
- 不建议现在删除 Tcl/Tk/halmeter/halshow，除非 Support Mode 已有等价诊断能力。
- 不建议现在删除 `.py` 源码，只能做发布副本 pyc-only 试验。
- 不建议现在裁剪 USB/FAT/NFS/TFTP，除非产线烧录、程序导入、维护恢复路径已明确替代。
- 不建议现在做 live rootfs cutover；默认只做 staging rootfs。

## 9. 最小执行顺序

1. 读取 `AGENTS.md`、本文、`待做工作/V3产品系统加固与安全诊断方案.md` 和 `待做工作/板载存储产品化与量产烧录方案.md`。
2. 建立 `repo_ignored/linux_slimming/<date>/` 作为证据目录。
3. 采集当前 rootfs、kernel、服务、端口、HAL、Python、许可证 inventory。
4. 运行 hardening manifest 和 rootfs plan。
5. 审查候选清单，删除明显不安全项。
6. 只在 staging rootfs 应用第一批低风险裁剪。
7. 运行本地工具测试和 staging smoke。
8. 构建完整镜像。
9. 上板部署完整镜像，不做单文件替换。
11. 如果涉及 motion，运行 `nc/cc.ngc` golden motion loop。
12. 记录结果为 `local_verified_only`、`board_verified` 或 `blocked`。
