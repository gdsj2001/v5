# 板端系统在线升级（OTA）架构边界

引用需求真源：`REQ-DOC-SINGLE-SOURCE`、`REQ-AUTO-STATE-DRIVEN`、`REQ-NATIVE-OWNER-FIRST`、`REQ-LINUXCNC-COMMAND-GATE`、`REQ-PARAM-MEMORY-LIGHTWEIGHT-SAVE`、`REQ-SETTINGS-RUNTIME-DRIVE-ONLY`、`REQ-WCS-NATIVE-OWNER`、`REQ-RTCP-G53-NATIVE-ACTUAL`、`REQ-NO-PERIODIC-DISK-WRITES`、`REQ-DRIVE-PROFILE-AUTH-CHAIN`。

本文是板端 OTA 的设计边界和后续实现卡，不是产品真源，也不是当前已实现能力说明。通用执行规则以 [AGENTS.md](../AGENTS.md)、[CMV自动化体系.md](../CMV自动化体系.md)、[功能/需求真源索引.md](../功能/需求真源索引.md) 为准。

范围限定：本文不处理硬件物理损坏，例如 eMMC/Flash 坏块、供电损坏、I2C/SPI 总线物理故障、加密芯片损坏、PL/PS 芯片损坏、PCB 损坏或外部设备损坏。本文只处理软件升级过程中的坏包、断电、网络中断、写入失败、启动失败、health check 失败、回滚和 Recovery 恢复。

## 1. 当前代码结论

当前仓库没有完成正式 OTA 产品链。现有代码可分为三类：

| 范围 | 当前代码锚点 | 结论 |
| :--- | :--- | :--- |
| 开发/验证部署 | `tools/v3_vm_full_current_sync.py`、`tools/v3_board_deploy_product_ui.py` | 可从完整当前 worktree 同步到 VM、ARM 构建、部署 LVGL 产品 UI、检查 runtime identity 和 relay；这不是 OTA，不写 inactive rootfs，不做 A/B commit/rollback |
| rootfs/release 计划 | `tools/v3_rootfs_hardening_plan.py`、`tools/v3_release_cutover_plan.py` | 生成非破坏性计划和 manifest；不能单独支撑发布升级 |
| 显式 live cutover 工具 | `tools/v3_live_rootfs_cutover_runner.py` | 带 `--apply` 和 `--i-understand-live-rootfs-cutover` 强显式开关，测试覆盖 fake rootfs 的备份、GPL/legacy 阻断和 cutover；不是默认 OTA，也不是 A/B 安全升级 |
| 授权/VPS | `v3_device_authorization_download.py`、`v3_drive_profile_download.py`、`device_dna_register_auth.py` | 当前只覆盖设备授权、驱动 profile 下载、DNA/签名/权限链；没有 OTA package 下载、OTA manifest 验签或升级授权动作 |
| 板端闭环 | `tools/v3_board_deploy_product_ui.py`、`tools/v3_cc_screen_motion_probe.py` | 可以证明 deploy/runtime/operator/motion 的验证状态；`board_deployed`、`board_runtime_ready` 不等于 OTA 成功 |

当前缺口：

- 没有 DNA private/public OTA scope 解析和私有包优先选择器。
- 没有 signed OTA manifest schema、package verify/install state machine。
- 没有 inactive 分区写入器、分区 hash readback 或 A/B boot target 切换。
- 没有 U-Boot `bootcount`/`bootlimit`/`fw_setenv`/`fw_printenv` 产品集成证明。
- 没有 OTA health-check service、commit/rollback 状态机和 watchdog 复位闭环。
- 没有 Data owner 迁移工具和迁移失败回滚证据。
- 没有 Secure Boot/eFUSE/BBRAM/BootROM 启用和可信启动链实证。
- 没有 PL bitstream + device tree + driver + HAL/profile 冷启动 bundle OTA 证明。

因此，本文当前状态只能是 `source_only` 设计卡。任何“OTA 已完成、可发布、board_verified”的表述都不成立。

## 2. 当前可用的板端部署链

当前能安全使用的是开发/验证部署链：

```text
PROJECT_ROOT 当前完整 worktree
  -> tools/v3_vm_full_current_sync.py
  -> VM/ARM LVGL build
  -> tools/v3_board_deploy_product_ui.py
  -> 当前 active rootfs 的 /opt/8ax/v3_product_ui
  -> board runtime identity / relay check / operator probe / motion probe
```

这条链的边界：

- 只部署产品 UI 源包、ARM binary、frame pacer 和 init script；`re_hal_select_helper` 已退役删除，不得作为可选部署产物恢复。
- `v3_board_deploy_product_ui.py` 会生成 source manifest、排除 build/cache、使用安全 tar 解包、写 install stage/history、恢复 remote relay/input gate、写 installed hashes。
- 它修改当前 active rootfs 的应用目录，不写 inactive rootfs，不切 boot env，不承诺断电回滚。
- `tools/v3_board_deploy_product_ui.py` 最多到部署和 runtime identity；功能通过仍需要 operator probe，运动通过仍需要 `ui_direct_native_start` / `nc/cc.ngc` golden motion。

这条链可以作为 OTA 实现前的构建和部署证据来源，但不能当成 OTA 本身。



目标链路：

```text
  -> /remote/ota/upgrade
  -> Command Broker ota_upgrade action
  -> 板端 OTA client
  -> VPS 当前 DNA private/public scope manifest
  -> package staging / SHA256 / signature / version / profile / anti-rollback verify
  -> inactive rootfs 写入和 hash readback
  -> 操作员确认后切换 boot target / reboot
  -> health check commit 或 rollback
```

私有/公共 OTA 包选择规则：

- 升级请求必须解析当前 DNA private 区和 public 区；manifest/metadata 可以都解析，package 下载互斥。
- 当前 DNA private 区存在 OTA 包记录时，只能下载 private OTA 包，不得下载 public OTA 包。
- 当前 DNA private 区明确 no_package/absent/skipped 并记录 reason 后，才允许下载 public OTA 包。
- private OTA 包存在但授权、版本、hash、签名、profile 或防降级校验失败时必须 fail-closed，不得改用 public 包替代。
- 一次升级只能有一个 `selected_scope`、一个 manifest 和一个 package hash；manifest/URL/目录名/状态摘要不得暴露 PL DNA 或 PL DNA hash，private 绑定用 `vps_distribution_id` 与 `dna_binding=server_verified` 表达；UI/VPS 状态不能把 public 成功显示为 private 成功。

按钮边界：

- 板端 OTA client 负责下载、验签、staging、写 inactive、切换、commit/rollback。
- 后台下载、验签和 staging job 必须可取消；进入写 inactive 分区、boot target 切换或重启前的不可取消阶段前，必须有操作员显式确认。
- 当前按钮和 relay endpoint 只能返回 `not_implemented`/fail-closed，不能调用现有 `v3_board_deploy_product_ui.py` 或 `v3_live_rootfs_cutover_runner.py` 冒充 OTA。

VPS 发布入口当前已实现：

- `8ax-factory-client-source` 新增 `OTA发布` 页，通过厂家账号调用 `POST /api/v1/admin/ota/packages`。
- 发布内容包含 package、detached signature、package/signature SHA256、scope、`pl_dna_hash` 校验字段、版本、profile、key id、签名算法、防降级最低版本和发布原因。
- private 发布路径为 `/opt/8ax-auth/storage/private/<vpsDistributionId>/ota/<product>/<channel>/`，public 发布路径为 `/opt/8ax-auth/storage/ota/public/<product>/<channel>/`；`pl_dna_hash` 只作为 VPS 侧设备绑定校验字段，不进入 private 路径、manifest 或可见摘要。
- private 发布前 VPS 必须用 `vpsDistributionId` 和 `pl_dna_hash` 查询 `devices` 表确认同一设备绑定；设备缺失、ID/hash 不匹配或数据库校验不可用时 fail-closed，不写 private 目录或静态镜像。
- VPS admin API 源码位于 `8ax-factory-client-source/vps-admin-api/vps_ota_admin_api.py`，负责校验 package/signature SHA256 和 size、落盘、生成/保存 manifest 和 SHA256 sidecar，并镜像到 `/var/www/html/updates/ota/...`。
- 该入口只做 VPS package 发布和 hash 校验，不生成签名、不选择 public/private、不启动板端升级。

## 3. 安全信任链目标

正式 OTA 推荐链路：

```text
离线签名/HSM 签名
  -> VPS 分发签名包和 manifest
  -> 板端 OTA client 下载到 staging
  -> SHA256 / 版本 / profile / 签名校验
  -> SoC Secure Boot 链确认 Bootloader / Kernel / Rootfs
  -> 可选安全芯片做设备身份、授权或挑战验签
  -> 写入 inactive 分区并读回 hash
  -> 操作员确认切换
  -> 重启进入候选分区
  -> health check 通过后 commit
  -> 失败则 Bootloader/watchdog 自动回滚
```

硬规则：

- 工厂主私钥不进 VPS、不进板端、不进仓库。
- VPS manifest 必须包含版本、包 SHA256、签名、签名算法、key id、目标硬件/产品 profile、最低兼容版本、防降级、key rotation/revocation 和回滚策略。
- 板端写 Flash 前必须完成内存中 SHA256 和签名校验。
- 外接 I2C/SPI 安全芯片不能单独阻止 CPU 写 Flash；只有被 Secure Boot 保护的 Bootloader/升级客户端强制执行策略时，它的验签结果才有产品安全意义。
- H0 若确认 Secure Boot 未启用或不可用，OTA 只能声明完整性校验和升级可靠性，不能声明可信启动安全闭环。
- I2C 地址、总线号、芯片型号必须以硬件资料和板端扫描为准；不能用推荐型号或旧资料替代 H0 实测。

## 4. 分区模型目标

推荐分区：

| 分区 | 用途 | 升级规则 |
| :--- | :--- | :--- |
| A rootfs | 当前运行系统 | 运行时不擦写 |
| B rootfs | inactive 目标系统 | OTA 写入、验签、切换 |
| Recovery | 极简救援系统 | 只读或写保护；普通 OTA 禁止修改 |
| Data | 本机数据和授权 | OTA 不覆盖；按 owner 迁移 |
| Staging | 下载临时区 | 临时名写入、fsync、验签通过后原子 rename；写 Flash 前再次验签 |

H0 必须先确认 Bootloader 类型、bootcount backend、环境变量区、redundant env、`/etc/fw_env.config`、watchdog 设备、实际 rootfs 分区、Recovery 启动路径和现有启动日志。未完成 H0 前，本文里的变量名和超时都只是设计占位。

## 5. A/B 自动状态机前置

正式 A/B + Recovery 至少需要四层组件：

1. **Bootloader 启动计数机制**
   - U-Boot 需启用启动计数能力，例如 `CONFIG_BOOTCOUNT_LIMIT`。
   - 变量名如 `upgrade_pending`、`bootlimit`、`bootcount` 只能作为候选；实际名和语义以 H0 板端核查为准。
   - 候选系统启动失败并超过 bootlimit 时必须自动回旧 active 分区。

2. **硬件 watchdog**
   - 必须通过 device tree 加载并启用板端物理 `/dev/watchdog`。
   - 新系统 kernel/rootfs/关键守护进程失败时，不能继续喂狗；watchdog reset 应推动 bootcount 累加。

3. **用户态 health check service**
   - 检查 Command Broker、State Publisher、typed SHM frame、UI/runtime identity、remote relay/input、必要参数回读。
   - OTA health check 只能读状态和执行提交/回滚状态机，不得接管产品 SHM 写路径。
   - 全部检测通过后才清除候选状态并 commit；失败或超时必须触发 reboot/watchdog 回滚。

4. **用户态 boot env 工具**
   - 必须配置正确 `/etc/fw_env.config`。
   - 每次 `fw_setenv` 后必须 `fw_printenv` 回读一致；失败不得进入 pending reboot。

## 6. Data owner 规则

OTA 不得把现场数据笼统称为 `/data` 文件。必须按 owner 分类：

| 数据 | owner | OTA 规则 |
| :--- | :--- | :--- |
| 授权信封、设备授权、DNA 缓存 | 授权/安全 owner | 保存在 Data；升级前后 hash 和可用性验证 |
| Settings Drive 私域和驱动证据 | Settings Drive owner | 可随 Data 保留；不得扩展成 native 参数真源 |
| WCS / G54-G59.3 | LinuxCNC `PARAMETER_FILE` / native owner | 按 `REQ-WCS-NATIVE-OWNER` 保留和回读，不由 OTA 私有 JSON 接管 |
| SCALE、软限位、HOME、运动约束 | runtime INI/native/settings apply owner | 通过受控 apply 和 State Publisher/SHM 回读证明 |
| RTCP/G53 几何 | RTCP/G53 native owner | 不由 OTA 覆盖或迁移成私有缓存 |
| tool table / TLO / modal | LinuxCNC/native owner | 保留 owner 文件和运行态回读路径 |
| G-code 程序和 sidecar | 程序目录 / program owner | 保留或按 manifest 迁移；不得修改用户 G-code |
| 日志、诊断、临时缓存 | 诊断/cache | 可清理，不能作为升级成功真源 |

升级包不得直接覆盖 Data 分区中的 owner 文件。需要 schema 迁移时必须提供版本化迁移工具、dry-run、幂等执行、备份、读回证明和回滚策略。迁移失败不得 commit 新系统，也不得留下半迁移数据作为成功状态。

## 7. 操作员确认边界

OTA 可以后台下载、校验和写 inactive 分区，但不能后台自动重启、自动切换 active boot target、自动重启 UI/backend、修改当前 active rootfs，或覆盖 Data owner 文件。

进入重启/切换前必须满足：

- 升级包已下载到 staging。
- staging 包已通过临时名写入、fsync、原子 rename 和读回 hash。
- SHA256、签名、版本、目标 profile、anti-rollback 和空间检查均通过。
- inactive 分区写入完成并读回 hash。
- 当前 active 分区、boot env 和 Data owner 文件已备份或记录 hash。
- 操作员显式确认，不允许定时自动确认。

## 8. FPGA / PL OTA

默认产品策略：FPGA/PL 更新必须和 Bootloader、device tree、kernel driver、HAL/profile 作为同一个 cold boot bundle 绑定发布。

禁止：

- 单独分发 `.bit` 并让当前系统直接加载。
- 新 bit 与旧驱动、旧 DTS、旧 HAL 混用。
- 运行态动态重载作为产品默认 OTA。
- 动态重载期间仍允许 CPU/HAL/driver 访问 AXI/PL 寄存器。

运行态动态重载仅允许实验模式，且必须先证明 driver stop/unbind、AXI quiesce、ABI 一致、失败恢复和 Recovery 入口。

PL bitstream 加密应优先使用 SoC 支持的 eFUSE/BBRAM/Secure Boot 机制；不能只靠应用层鉴权保护 IP。

## 9. 实施阶段

| 阶段 | 目标 | 关闭条件 |
| :--- | :--- | :--- |
| H0 事实核查 | I2C、存储布局、Bootloader、Recovery、Secure Boot、watchdog、bootcount backend、fw env | 有原理图、板端扫描、分区、启动日志、`fw_printenv`、watchdog 和启动计数证据 |
| H1 信任链设计 | 签名、manifest、key 管理、Secure Boot、安全芯片角色、防降级 | 私钥不进 VPS；验签链路、key rotation/revocation、anti-rollback 和失败码明确 |
| H2 OTA package/client 原型 | VPS public/private package 发布、DNA private/public manifest 解析、私有包优先选择、下载、staging、验签、空间检查、失败码 | 厂家客户端可上传 package/signature/hash/metadata 到 VPS admin API；板端 private 包存在时不下载 public；坏包、断网、空间不足、签名失败均 fail-closed |
| H3 inactive 写入 | 写 B 分区、读回 hash、不中断 active | 写入中断后当前 active 仍可启动 |
| H4 A/B boot env | pending boot、try counter、commit/rollback | 启动失败、bootcount 溢出、env 写入失败、Recovery 均可恢复 |
| H5 health check | Broker、State Publisher、typed SHM、remote input/relay、参数 readback | health 通过才 commit；失败/超时自动回滚 |
| H6 Data owner 迁移 | 列出 owner 文件和迁移策略 | WCS/SCALE/RTCP/授权/G-code 均有保留和读回证明 |
| H8 FPGA bundle | bit/DTS/driver/HAL/profile 冷启动绑定 | 新旧 bundle 启动、回滚和 ABI 证据齐全 |
| H9 全流程回归 | 下载到 commit/rollback 的完整矩阵 | 板端证据包完整，最终启动/回滚状态明确 |

任一阶段未过，不得进入下一阶段的产品化实现。

## 10. 验证矩阵

最低测试与自动状态机验证：

- 签名错误包拒绝。
- SHA256 不匹配拒绝。
- manifest 产品 profile、硬件 profile、最低版本和 key id 不匹配拒绝。
- 低于允许版本的降级包拒绝或进入明确的受控回滚策略。
- 网络中断可恢复。
- staging 空间不足 fail-closed。
- staging 下载中断不会生成可安装包。
- 写 inactive 分区中断后当前 active 仍可启动。
- 切换 boot env 失败不得重启进入未知分区。
- env 损坏或读取失败进入 fail-closed 或 Recovery。
- U-Boot `bootcount` 溢出自动回滚。
- 新系统启动失败自动回滚。
- health check 服务未启动时自动回滚。
- 用户态 health check 异常或超时后停止喂狗并回滚。
- `fw_setenv` / `fw_printenv` 写读一致后才能 commit。
- commit 前断电必须回到旧系统或继续候选检测，不得声明升级成功。
- commit 后断电不得误回旧分区。
- Recovery 可启动并能恢复网络/诊断。
- Data 分区 owner 文件未被覆盖。
- Data owner 迁移失败能恢复旧数据，且不会 commit。
- FPGA bundle ABI 不匹配拒绝。

板端通过必须包含：manifest、包 hash、签名结果、分区 hash、boot env、启动日志、health result、rollback/commit 结果和 UI 截图。

## 11. 禁止事项

- 将主私钥放到 VPS 或仓库。
- 写“覆盖硬件损坏”“0 风险”或“绝对恢复”。
- 把安全芯片写成能单独阻止 Bootloader/CPU 写 Flash。
- OTA 覆盖 Data 分区 owner 文件。
- 用私有 JSON 接管 WCS、SCALE、RTCP/G53、tool table 或运行模态。
- 未经操作员确认就后台自动重启或自动切换 active 启动目标。
- OTA 升级客户端修改机床运动控制语义。
- 有当前 DNA private OTA 包时继续下载 public OTA 包，或用 public 包替代 private 包失败。
- 单独热更新 `.bit` 进入产品默认路径。
- 在新系统通过全部 health check 指标前，提前重置 `bootcount` 或清除 `upgrade_pending`。
- 在硬件 watchdog 未启用或未证明生效前发布 A/B 升级产品包。
- 把当前 `v3_board_deploy_product_ui.py` 或 `v3_live_rootfs_cutover_runner.py` 结果写成 OTA 完成。
- 把推荐变量名、推荐超时或实验板端结果写成所有板端都已验证的事实。
- 在 Secure Boot 未启用时宣称可信启动安全闭环。
- 未做断电/坏包/回滚/Recovery 测试就声明软件 OTA 可发布。

## 12. 推荐状态机

实现时建议使用显式状态机，状态和持久化记录至少覆盖：

```text
idle
  -> downloaded
  -> verified
  -> written_inactive
  -> operator_confirmed
  -> pending_reboot
  -> candidate_boot
  -> health_ok
  -> committed
```

失败路径必须明确进入 `rejected`、`rollback_pending`、`rolled_back` 或 `recovery_required`，不得用日志文字替代机器可判断状态。只有 `committed` 才能向 UI/VPS 报告升级成功。
