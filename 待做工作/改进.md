# V5 坐标比例链、CPU1 降载与上电提速实施作业指导书

## 1. 本指导书解决什么

AI 必须围绕以下三个实际结果修改代码，不得把写文档、补门禁或证明流程当作独立目标：

1. C 轴目标精度为 0.0001 deg/count 时，raw 一 count 在 INI、HAL、native、SHM 和 UI 中始终只对应 0.0001 deg；同时排查其他轴是否有相同问题。
2. 保证原始 G 代码、运动质量、安全链、坐标和远程功能不退化，降低 CPU1 空闲占用及运行 G 代码时的持续满载。
3. 缩短开发板从上电到主页面可见、触摸可用、控制链可以正常操作的时间。

需求冲突时按 功能/需求真源索引.md 路由到唯一 owner。owner 已正确而代码没有兑现时，直接修改 canonical source，不重复写需求摘要。

## 2. 必须遵守的实现原则

### 2.1 同一数据只生产一次

同一个 source generation 内，任何数据都只能由一个 producer 完成一次读取、计算、组装和发布：

- 微内核/native resident 数据只采样一次。
- 坐标、RTCP actual、active model、G53、WCS 和必要 UI 投影只组装一次。
- 同一帧 SHM payload 只生成一次。
- 同一帧远程画面 payload 只捕获、压缩和组装一次。
- 同一黄色轨迹 scene snapshot 只计算一次。
- 多个本地页面、远程客户端或 UI 部件只能引用同一份不可变结果，各取所需。

不同 TCP 客户端仍需要各自发送字节，但不得为每个客户端重新读取 framebuffer、重新投影、重新编码、重新压缩或重新组装 payload。

禁止：

- 同一份数据按消费者数量重复计算；
- 同一个 SHM block 被多个 publisher 重复生成；
- UI、relay、provider 分别再算一遍同一坐标；
- 为了复用而复制出第二份 JSON、第二 SHM、第二 parser 或第二后台服务；
- 用缓存中的旧 generation 混合当前 generation。

### 2.2 SHM 只承载一次计算后的显示结果

SHM 白名单是人为制定的工程边界，可以根据真实消费者修改，不能把旧白名单当作永远正确。

修改白名单必须同时满足：

1. 字段确实有两个或以上真实消费者，或跨进程传递能消除重复计算。
2. 字段由 microkernel/native 或唯一 display-projection producer 一次生成。
3. 字段是 display-only 结果，不得反向参与运动控制、安全判断、保存成功、动作完成或参数 owner readback。
4. ABI 有 version、size、generation、timestamp、valid mask、SeqLock 和 CRC。
5. producer 数量恒为一；消费者增加不增加计算次数。
6. 字段没有真实消费者时立即删除，不留兼容占位。

共享内存中的 UI 坐标、姿态和必要派生显示值，应根据同一帧 microkernel/native resident 数据一次性计算完成。UI、主页面、设置页、远程显示和诊断只读取自己需要的字段，不得再次执行同一变换。

### 2.3 黄色轨迹和 RTCP

已实测黄色轨迹在 10Hz 和 30Hz 时 CPU1 都会拉爆，因此刷新率不是根因。停止继续用 5Hz、10Hz、30Hz互换来试错。

正确实现：

- Program Open 时只解析一次 G 代码，形成 resident program model。
- 同一帧唯一 scene producer 消费 active model、RTCP actual、WCS/G53、当前姿态、view 和冻结 fit。
- RTCP ON：RTCP 黄色轨迹变换、active model 几何、WCS 和屏幕投影在同一帧一次完成，输出一个不可变 scene snapshot。
- RTCP OFF：直接跳过 RTCP 专属计算，只使用不需要 RTCP 的基础场景结果。
- UI renderer 只消费 scene snapshot，不再计算 RTCP、模型变换、fit 或完整点集投影。
- relay 只消费最终 framebuffer/dirty payload，不参与黄色轨迹计算。

### 2.4 UI 和板端操作边界

- V5 产品代码和 native owner 是实现真源；界面和交互视觉参考 D:\v3。
- 不得从 D:\v3 覆盖 V5 的 owner、协议或产品源码。
- 已退役的嵌入浏览器查看功能必须连同源码、启动入口、监听端口、依赖、测试和文档引用彻底删除。
- 后续 AI 查看开发板只能使用 MCP 工具提供的板端 screen capture。
- AI 点击前先用 MCP 抓 fresh frame，确认页面、目标、遮挡和急停状态。
- AI 每次只用 MCP pointer/mouse 做一个小动作，动作后重新抓屏并读取 actual。
- 8ax.WinRemote 只供人查看和操作，AI 不得启动、查看、截图、聚焦、点击、键入或控制该程序。
- 正式 WinRemote 产品能力必须保留；删除的是 AI 嵌入浏览器/测试桥，不是正式客户端。

### 2.5 退役代码立即物理删除

确认属于 retired、duplicate、fallback、shadow 或 bypass 的实现后，先迁移真实消费者，随后在同一切片物理删除：

- 源文件、头文件和声明；
- CMake、recipe、install、package 入口；
- init/service/CLI/env 入口；
- socket、端口和运行进程；
- 测试、fixture、文档示例和正向引用；
- ignored 目录中的源码副本和运行入口。

禁止保留 wrapper、alias、disabled、环境开关、备份、TODO 或测试入口，避免后期死灰复燃和错误引用。

## 4. AI 每次执行的固定流程

每次只关闭一个具体缺陷、一个 owner 链和一个最小 target：

1. 读取本工序的进入条件和唯一 owner。
2. 运行 git status 和 focused diff，保留已有修改。
3. 获取将编辑文件的精确锁；已有活跃锁时不覆盖、不复制第二实现。
4. 冻结本次允许文件、最小 build target 和测量场景。
5. 先取得修改前基线。
6. 修改 Windows D:\v5 的 canonical source。
7. 运行最近的 focused compile/unit/contract。
8. 只构建最小受影响 target；失败先修该层。
9. 只有当前工序需要板端结果时才部署。
10. 使用原始 operator path，同场景只改变一个变量。
11. 满足关闭条件后立即交付，不自动扩展镜像、全仓审计或无关安全问题。
12. 真实依赖迁移完成后，同切片删除退役实现和全部入口。

阶段状态只允许：

- source_only
- local_verified_only
- board_verified
- blocked

文档、计划、dry-run、旧截图、direct backend 请求或设置文件写入都不能替代 live runtime actual。

## 5. 工序 P01：修复 C 轴比例链并审计全部轴

P01 是当前第一工序。P01 未关闭前，不用坐标跳动评价 CPU 优化结果。

### 5.1 进入条件

- 机器保持安全静止，不执行运动。
- 明确实际 active mode、active INI 和部署 identity。
- 当前 BUS 的 C 固定为 JOINT_4、cia402.4、status_slot 4。
- 只有实际 Pulse active 时才查 JOINT_5 和 zynq_stepgen_hw.5。
- 同一采样窗口 active model 和 mapping generation 不变。

### 5.2 全轴矩阵

| 运行链/轴 | 当前 source SCALE | 当前单位/count | 本工序动作 |
| --- | ---: | ---: | --- |
| BUS X / JOINT_0 | 10000 | 0.0001 mm | 核对 INI、HAL、native、SHM、显示 |
| BUS Y / JOINT_1 | 10000 | 0.0001 mm | 核对 INI、HAL、native、SHM、显示 |
| BUS Z / JOINT_2 | 10000 | 0.0001 mm | 核对 INI、HAL、native、SHM、显示 |
| BUS A / JOINT_3 | 1000 | 0.001 deg | 核对实际设置；不得因 C 批量改成 10000 |
| BUS C / JOINT_4 | 1000 | 0.001 deg | 当前目标明确改为 10000 / 0.0001 deg |
| Pulse X/Y/Z | 10000 | 0.0001 mm | 启用前消除错误 INI 入口和 HAL 硬编码 |
| Pulse A/B/C | 1000 | 0.001 deg | 按每轴实际 precision 生成，不保留硬编码 |

其他轴确有同类问题：

- X/Y/Z 已为 0.0001，但三位小数显示同样丢失一位。
- 所有轴共用“文件保存成功不等于 live runtime actual”的缺口。
- A 当前 0.001 与三位小数相符，不能把 C 的参数套给 A。

### 5.3 修改前同帧采样

连续采样必须覆盖一次用户看到的跳动，每个样本绑定同一 monotonic 时间和 generation：

1. active INI 路径、identity、JOINT_4 SCALE。
2. cia402.4.pos-scale。
3. C 物理 slave raw actual。
4. v5-bus-axis-router.joint-actual-position-04。
5. cia402.4.drv-actual-position、cia402.4.pos-fb、joint.4.motor-pos-fb。
6. home-counts-per-unit-04 和 mapping generation。
7. wcheckpoint C logical/base/runtime counts 和 generation。
8. Position、最终 SHM 的 C double。
9. UI 格式化前 C double 和画面文本。

不得把不同时间的值拼成一条链。

### 5.4 根据第一处异常选修复切片

| 结果 | 根因 | 只修这一层 |
| --- | --- | --- |
| INI=10000，HAL=1000 | 已落盘但 LinuxCNC 未正确重启/装载 | restart、active identity、live readback |
| HAL=1000，raw delta=1，pos_fb delta=0.001 | 旧 SCALE 正在运行 | 唯一 SCALE 生成与 HAL 装载 |
| HAL=10000，pos_fb delta=0.0001，native/SHM 更大 | native SCALE、wcheckpoint generation 或槽位不同源 | native 同 generation 提交 |
| SHM double delta=0.0001，画面跳 0.001 | 固定三位小数量化 | 唯一 per-axis formatter |
| raw 自身大幅突变、符号翻转或异常回绕 | PDO/router/driver signed/unwrap | raw owner，禁止 UI 平滑 |
| 坐标只变 0.0001，但黄色轨迹/模型大跳 | scene/RTCP 重复投影 | 转工序 C03 |

### 5.5 P01-A：BUS precision 到 runtime actual

允许文件：

- board/linuxcnc/ini/v5_bus.ini
- board/services/command_gate/v5_settings_apply*.c
- board/services/command_gate/v5_native_motion_parameters*.c
- board/linuxcnc/hal/v5_bus_2ms.hal
- 对应 focused tests

实施步骤：

1. precision 保持唯一用户输入，SCALE=1/precision。
2. 当前 C=0.0001 必须生成 JOINT_4 SCALE=10000。
3. BUS HAL 只从 active JOINT_N SCALE 设置 cia402.N.pos-scale。
4. 保存结果拆分为 persisted_readback、restart_pending、runtime_actual。
5. clean restart 前 runtime_actual 必须为 false。
6. restart 后同时核对 active INI identity、cia402.4.pos-scale=10000、native counts-per-unit=10000。
7. 任一不一致时 fail-closed，不允许显示“已生效”。
8. 删除迁移后的默认 SCALE、历史比例字段和 fallback。

最小验证：

- C precision=0.0001 只产生 SCALE=10000。
- raw N 到 N+1 时 cia402 pos_fb 只增加 0.0001。
- 未重启时 persisted=1、restart_pending=1、runtime_actual=0。
- 重启同源回读后才 runtime_actual=1。
- X/Y/Z/A/C 表驱动测试证明轴、JOINT、status slot 映射正确。

### 5.6 P01-B：native 和 wcheckpoint 同 generation

允许文件：

- board/services/command_gate/v5_native_motion_parameters*.c
- board/services/state_publisher/v5_machine_status_projection.py
- 相关 native/wcheckpoint focused tests

实施步骤：

1. counts-per-unit 只读 active runtime INI 同一 SCALE generation。
2. mapping、counts-per-unit、base/logical/runtime count 原子绑定同一 generation。
3. generation 变化期间丢弃样本，不混用旧比例或旧 base。
4. status slot 只来自 active model descriptor；AC 的 C 必须为 slot 4。
5. 唯一 display-projection producer 一次生成旋转显示坐标并进入同帧 SHM。
6. UI 不再根据 raw、wcheckpoint 或 SCALE 重算。
7. 删除旧 counts-per-unit、旧 mapping 和历史 fallback。

### 5.7 P01-C：唯一按轴分辨率格式化入口

允许文件：

- board/app/src/v5_main_page_status_text.c
- board/app/src/v5_coordinate_panel.c
- board/app/src/v5_v3_offset_page.c
- 一个唯一共享 UI formatter owner 及 focused tests

实施步骤：

1. 开机或 canonical reload 时把每轴 unit_per_count 和 generation 放入 resident display metadata。
2. 小数位至少为 max(3, ceil(-log10(unit_per_count)))，并设置合理上限。
3. 当前 0.0001 轴至少显示四位；0.001 轴至少三位。
4. 主页面、设置页、偏置页和 following error 共用一个 formatter。
5. 运行期格式化不读 INI、JSON 或磁盘。
6. 迁移后删除各页面私有三位小数实现。
7. 不增加 deadband、平均、低通或 UI 去抖掩盖一 count。

### 5.8 P01-D：Pulse 全轴第二比例 owner

Pulse 仍为 cold-staged 时只做 source 和 focused gate，不启动 Pulse 板端运行。

1. runtime INI 路径由 fresh active mode/SelectLink owner 选择，禁止固定写 BUS INI。
2. Pulse 每轴 SCALE 由该轴 precision 生成。
3. zynq_stepgen_hw.N.scale 只消费 Pulse JOINT_N SCALE。
4. 消费者迁移后物理删除 v5_pulse.hal 六个硬编码 scale。
5. active mode unknown、contract 不可运行或 INI 不一致时 fail-closed。

### 5.9 P01 关闭条件

- C 的 INI、cia402.4.pos-scale、native counts-per-unit 都为 10000 且 generation 一致。
- raw |delta|<=1 时，格式化前 C |delta|<=0.0001 deg。
- 画面至少四位小数，不再出现三位量化制造的 0.001 跳变。
- X/Y/Z/A/C 都有 precision、SCALE、HAL/native actual 和显示位数结果。
- 合法 359.9999 到 0.0000 相位回绕不会被误判为异常大跳。

## 6. 工序 C01：删除嵌入浏览器/测试桥并重取 CPU 基线

1. 搜索嵌入浏览器/测试桥源码、端口、启动、install、package、测试和文档引用。
2. 保留正式 board relay 和正式 WinRemote 协议。
3. 迁移真实消费者后物理删除嵌入浏览器/测试桥全部实现。
4. 删除按客户端 IP 判重和 same-peer replacement 试错代码。
5. 确认退役进程、socket、listener 为 0。
6. 无正式客户端时 stream_active_sessions 必须为 0。
7. 只用 MCP 抓屏/鼠标进行 AI 板端操作，不增加正式 stream 会话。

## 8. 工序 C03：同帧一次计算 RTCP 与黄色轨迹

### 8.1 进入条件

- C01 新基线仍证明 UI/scene 是显著热点。
- C02 单 custom-draw scene 已稳定。
- 当前真实消费者已清点完成。

### 8.2 唯一 producer 的输入

- resident program model；
- 同帧 microkernel/native 坐标与姿态；
- RTCP actual；
- active model 与 generation；
- G53/WCS；
- view generation；
- 冻结 fit generation。

### 8.3 唯一输出

一份不可变 V5ToolpathSceneSnapshot，至少绑定：

source_generation + program_generation + native_generation + active_model_generation + rtcp_generation + view_generation + fit_generation

### 8.4 实施步骤

1. Program Open 只解析一次 G 代码。
2. 同一 generation 只读取一次 native/SHM input snapshot。
3. RTCP ON 时，在唯一 producer 内一次完成 RTCP 黄色轨迹变换、active-model 几何、WCS 和屏幕投影。
4. RTCP OFF 时直接跳过 RTCP 专属变换，不调用空计算，不生成 RTCP 中间缓存。
5. scene snapshot 一次生成后，主页面 renderer、dirty 计算和需要的其他显示消费者引用同一结果。
6. UI renderer 只画结果，禁止再次执行 RTCP、模型变换、fit 或完整点集投影。
7. relay 只读取最终 framebuffer/dirty payload。
8. generation key 未变化时直接复用，不遍历黄色点。
9. 真实跨进程消费者需要 scene 数据时，可修改 SHM display whitelist，建立独立有界 scene block；没有跨进程消费者时不新造 scene SHM。
10. 迁移后删除所有重复 projector、RTCP UI 计算、旧点缓存和测试入口。

### 8.5 关闭条件

- 10Hz 和 30Hz 场景下，一个 generation 的完整 transform/project 计数都最多一次。
- 消费者数量增加不增加 transform/project/encode 次数。
- RTCP OFF 的 RTCP 专属计算计数为 0。
- RTCP ON 的黄色轨迹、WCS、模型几何 generation 相同。
- UI 只消费 snapshot。
- 同场景 UI/CPU1 有可重复净改善；无改善则停止扩建并找下一实际热点。

## 9. 工序 C04：共享内存一次组装、多方复用

### 9.1 State SHM 常驻映射

1. State startup 创建唯一 writable handle 并 mmap 一次。
2. 每帧只从同一 native input snapshot 组装一次最终 frame。
3. 每帧只执行一次 SeqLock/CRC commit。
4. UI、诊断和其他消费者 startup 各自 read-only mmap 一次。
5. 周期内只读取现有映射，不重复 open/mmap/munmap/close。
6. 不为每个消费者重新组装 frame。
7. 每帧不调用 msync。
8. 迁移后删除 stateless publish/read 入口。

### 9.2 Position resident ABI

1. block 包含 magic、version、size、sequence、generation、timestamp、valid mask、writer identity 和 CRC。
2. Position startup 创建并 mmap 一次。
3. 每帧从 microkernel/native resident 数据一次生成 position snapshot。
4. State 只引用该 snapshot，不再 fopen 或重算坐标。
5. reader 有限重试；generation、CRC 或 writer identity 不一致时丢弃。
6. 迁移后删除临时文件、rename 和每帧 fopen。
7. 不增加第二 Position writer、第二 SHM 或 JSON fallback。

### 9.3 远程 payload

1. dirty/full 判断一次。
2. framebuffer 捕获一次。
3. 压缩、编码和 payload 组装一次。
4. 多个连接引用同一不可变 PayloadViews。
5. 慢客户端只影响自己的发送队列，不触发上游重新生产。
6. 断开的旧会话立即退出，不能继续占用上游 payload。

### 9.4 关闭条件

- 一个 Position writer、一个 State writer、一个 relay payload producer。
- 消费者数量增加不增加上游计算、组装和压缩次数。
- steady-state 不再每帧执行 VFS/mmap 生命周期。
- reader 不使用撕裂帧、旧 generation 或错误 writer。
- Position、State、UI、relay 和 CPU1 均记录同窗改善。

## 11. LinuxCNC 热点的进入条件

只有 P01、C01、C03、C04 完成后，同一 cc-ac 窗口证明 milltask/usrmot 是新增负载第一名，才允许修改 LinuxCNC。

进入前必须有：

- 线程 CPU ticks；
- commands/s；
- queue depth；
- underflow；
- servo overrun；
- acknowledgment/wait 热点。

没有这些证据不得修改 eager、ack、lookahead、插补或 servo period。一次只改一个点；Pause/E-stop 变慢、underflow、停顿、overrun 或 BUS DC/WKC 异常立即判失败。

## 15. 总关闭条件

- C 轴 0.0001 deg/count 在 INI、HAL、native、SHM 和 UI 同 generation 闭环。
- raw 一 count 不被软件层放大；其他轴完成同类矩阵审计。
- 同一 native 数据只采样、计算和组装一次，多消费者各取所需。
- RTCP ON 的黄色轨迹在同帧唯一 producer 内完成，UI 直接消费。
- RTCP OFF 的 RTCP 专属计算次数为 0。
- 10Hz/30Hz 不再被当作根因试错，真实重复计算链已被删除。
- 嵌入浏览器/测试桥源码、入口、端口、进程、测试和引用为 0。
- AI 板端操作只使用 MCP 抓屏和鼠标；8ax.WinRemote 未被 AI 操作。
- 正式多客户端远程功能保留，上游 payload 只生产一次。
- steady-state SHM 没有每帧 VFS/mmap 生命周期。
- CPU1 在固定 G 代码场景不再持续 100%，p99 达到阶段目标。
- 启动由真实完成条件连续推进，没有正常路径固定 sleep 和重复入口。
- 冷启动三轮中位数低于 45 秒，功能、安全和实时链不退化。
- 最终回读 estop_active=1、machine_enabled=0。
- 所有退役、重复、fallback、shadow、bypass 路径均已物理删除，没有 wrapper、disabled、env gate、备份或测试入口。
