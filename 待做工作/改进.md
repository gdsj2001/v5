# V5 审查问题 AI 修改与上板清单

## 1. 固定分工

本文件只用于让 AI 快速完成代码整改和上板，不让 AI 扩展成自动验收工程。

AI 负责：

1. 读取本卡登记的唯一 `REQ-*` owner。
2. 只修改本卡列出的 canonical source。
3. 运行语法检查和最小受影响 target 编译。
4. 构建当前受影响的板端产物并部署。
5. 确认部署 artifact/hash 正确、相关服务能够启动。
6. 交接修改内容和人工测试重点。

用户负责：

1. 页面、按钮和显示是否符合预期。
2. Home、Start、Pause、E-stop 等真实操作。
3. BUS/Pulse、AC/BC、RTCP 和 G-code 运动测试。
4. Golden Motion、CPU1 压力、触摸手感和最终功能验收。

AI 禁止：

- 自动执行 UI 点击、Golden Motion、CPU1 压力或运动测试。
- 为取得 `board_verified` 反复补测试、截图、runner 或证据工具。
- 顺手重构无关大文件、清理全仓、升级 Linux/PetaLinux 或做离线全量认证。
- 创建第二实现、fallback、wrapper、备份文件或 VM/板端源码副本。
- 当前修改卡完成后擅自进入下一卡。

AI 完成上板后的统一口径：

```text
状态：local_verified_only
部署状态：已部署待人工验收
功能结论：等待用户最终测试，不声明 board_verified
```

文中“最小检查/编译”只用于避免语法错误、构建失败或错误 artifact 上板，不属于用户保留的最终功能测试。

## 2. AI 固定执行流程

每次只执行一张修改卡。

1. 从 `功能/需求真源索引.md` 找到本卡 owner，只读快读卡和命中章节。
2. 查看本卡文件的 `git status` 和 focused diff，保护已有改动。
3. 获取准备编辑文件的精确锁。
4. 按“必须修改”逐项改代码；“不要修改”中的内容一律不碰。
5. 只运行本卡列出的最小检查/编译。
6. focused gate 通过后释放文件锁。
7. 用户要求本轮上板时，按第 9 节一次性构建和部署所选修改卡。
8. 部署后停止，把第 10 节交接清单交给用户。

停止条件：

- owner 与本卡冲突：以 owner 为准，不自行改需求。
- 文件被其他活跃任务锁定：不覆盖。
- 最小 target 编译失败：只修当前失败层。
- 板端不可达或部署失败：报告真实 blocker，不转去补功能测试。
- 需要 BUS/Pulse 产品范围决定：停止并等用户明确选择。

## 3. 修改顺序

| 卡号 | 内容 | 默认处理 |
| --- | --- | --- |
| M01 | 删除生产 TCF | 执行 |
| M02 | Position → State Publisher → typed SHM 单链 | 执行 |
| M03 | 唯一 scene producer，删除 UI 重复计算 | M02 完成后执行 |
| M04 | Program Open 准备完成，Start 只走热路径 | M03 完成后执行 |
| M05 | BUS/Pulse active-mode 闭环 | 默认暂停，用户单独指定后执行 |

本轮明确不做：

- 为 500 行规则做无关全仓拆分。
- PetaLinux 2020.2、Linux 5.4 版本升级。
- 完整离线灾难恢复认证。
- 自动 operator、Golden Motion、CPU1 压力或触摸验收。

## 4. M01：删除生产 TCF

Owner：

- `REQ-SERVICE-RUNTIME-SANDBOX`
- `功能/系统代码架构硬边界守则.md`

只允许修改：

- `board/petalinux/v5_bitbake_source_inventory.json`
- `board/third_party/petalinux-source-packages/v5_source_packages.json`
- `board/third_party/petalinux-source-packages/git2_git.eclipse.org.gitroot.tcf.org.eclipse.tcf.agent.tar.gz`
- `board/petalinux/project-spec/configs/rootfs_config`
- `board/tools/petalinux/verify_v5_petalinux_source.py`
- `board/tools/petalinux/test_verify_v5_petalinux_tcf_retirement.py`
- `board/tools/deploy/check_v5_runtime_policy.py`
- `board/tools/deploy/check_v5_board_runtime_policy.py`
- 与上述合同直接对应的 source identity

必须修改：

1. 从生产 `rootfs_config` 删除 `tcf-agent` 选择。
2. 从 BitBake inventory 删除 TCF recipe、package、URL 和 checksum。
3. 从 source package manifest 删除 TCF 登记。
4. 物理删除 TCF source archive。
5. 统一静态门禁与板端检查使用的函数/token 名。
6. 保留板端对 TCF 文件、init、进程和 TCP 1534 listener 的不存在检查。
7. 更新直接受影响的 identity 和 focused test。

不要修改：

- 不保留 disabled recipe、空 archive、环境开关或诊断 fallback。
- 不删除板端 TCF 不存在检查。
- 不顺手整理其它 PetaLinux package。

AI 最小检查：

```powershell
python -m py_compile `
    board/tools/petalinux/verify_v5_petalinux_source.py `
    board/tools/petalinux/test_verify_v5_petalinux_tcf_retirement.py `
    board/tools/deploy/check_v5_runtime_policy.py `
    board/tools/deploy/check_v5_board_runtime_policy.py

python board/tools/petalinux/test_verify_v5_petalinux_tcf_retirement.py
python board/tools/petalinux/verify_v5_petalinux_source.py
python board/tools/deploy/check_v5_runtime_policy.py
```

上板要求：

- 该卡改变 rootfs，不能只推送 runtime 文件。
- 最终批次必须生成一套包含新 rootfs 的 current image，并按 `功能/SD卡重建.md` 的 canonical 路径更新开发板。
- 更新后 AI 只确认新 artifact/hash 和服务启动；TCF 实际不存在由用户最终验收。

用户测试重点：

- 系统正常启动。
- 原产品服务正常。
- 无 TCF 进程和 1534 listener。

## 5. M02：Position → typed SHM 单链

Owner：

- `REQ-SHM-DISPLAY-PROJECTION`
- `功能/0-4共享内存.md`

只允许修改：

- `board/services/state_publisher/v5_position_status_publisher.py`
- `board/services/state_publisher/v5_native_sample.c`
- `board/services/state_publisher/v5_state_publisher_service.c`
- `board/services/state_publisher/v5_status_shm_writer.c`
- `board/services/state_publisher/v5_status_shm_mmap.c`
- `board/app/include/v5_status_shm.h`
- `board/app/src/v5_ui_model.c`
- 直接对应的 ABI/writer/reader tests

必须修改：

1. Position writer 只保留一个 singleton 和一个长期 writable mapping。
2. `v5_native_sample.c` 改为 startup 建立长期 read-only mapping。
3. 删除 steady-state 每周期 `fopen/fread/fclose` 和反复 `mmap/munmap`。
4. Position 的 writer identity、source time 和 source generation 原样传到 State Publisher 和最终 SHM。
5. State publish epoch 不得把旧 source 伪装为 fresh。
6. 按 owner 同时更新 ABI version/size、writer、reader、mask 和必要 display metadata。
7. writer identity、inode、version、size 或 generation 改变时，reader 自动重连新对象。
8. 删除第二 writer、JSON fallback 和旧兼容入口。

不要修改：

- SHM 不加入控制、保存、安全或动作结果真值。
- 不改变 owner 已规定的显示位数、显示过滤和 30Hz 合同。
- 不创建第二 Position block。

AI 最小编译：

```powershell
cmake --build repo_ignored/build/windows-mainline `
    --config Release `
    --parallel 2 `
    --target v5_state_publisher v5_state_publisher_smoke v5_ui_shm_refresh_smoke
```

上板目标：

- `v5_state_publisher`
- `v5_lvgl_shell`
- 对应 init/runtime 文件和 SHM ABI consumer

用户测试重点：

- 坐标连续刷新。
- 重启 State Publisher 后页面能够恢复。
- 坐标、跟随误差和轨迹没有跳回旧数据。

只允许修改：

## 9. AI 构建和上板

### 9.1 普通 runtime 修改卡

M02–M04 可在同一批次完成 Windows 最小编译后，只做一次 VM 构建和上板。

使用唯一 VM 投影和 `/root/v5-build/board`，构建实际受影响 target；不得建立第二 build/source tree。

部署使用：

```sh
cd /mnt/v5-source/board
V5_REPO_ROOT=/mnt/v5-source/board \
V5_BOARD_BUILD_DIR=/root/v5-build/board \
V5_BOARD_SSH="$V5_BOARD_SSH" \
V5_BOARD_SSH_PORT="${V5_BOARD_SSH_PORT:-22}" \
tools/deploy/run_v5_board_acceptance.sh --apply
```

该命令在本模式下只用于：

- 构建当前 runtime targets；
- 部署 canonical artifact；
- 核对 hash；
- 重启并确认服务能够启动。

AI 在命令完成后不得继续 UI 点击、Home、Start、motion 或 Golden Motion。

### 9.2 M01 rootfs 修改

M01 必须单独走当前 PetaLinux image 闭包：

```sh
VM_BUILD_ROOT=/root/v5-build \
VM_SOURCE_PROJECTION_ROOT=/root/v5-build/temp_source/current \
sh /mnt/v5-source/board/tools/linuxcnc/build_v5_linuxcnc_petalinux.sh --full
```

随后严格按 `功能/SD卡重建.md` 使用新 current image 更新开发板。AI 只确认 image/hash、启动和基础服务；最终功能由用户测试。
