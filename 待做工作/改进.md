## 1. 唯一目标、裁决顺序和读取包

本文唯一目标是在不降低功能、不迁移普通工作到 CPU0、不牺牲 native actual、安全、30Hz 显示和远程功能的前提下，降低 CPU1 固定负载与 G 代码运行放大负载，并对真正的 dense-G1 LinuxCNC 热点设置证据门。

1. [AGENTS.md](../AGENTS.md) 的入口核、锁、构建梯级、退役和状态词规则。
2. [功能/需求真源索引.md](../功能/需求真源索引.md) 的 AI 任务阅读包，定位本卡唯一 REQ owner。
3. owner 的 AI_FAST_READ_BEGIN 至 AI_FAST_READ_END，再读本卡命中的 detail_sections。
4. 当前 canonical source、当前 git status、focused diff、相关构建/安装/服务入口和运行证据。

本问题常用 owner：

- [功能/0-1开机参数入内存.md](../功能/0-1开机参数入内存.md)：REQ-GCODE-RUN-HOT-PATH、REQ-CPU0-MOTION-REALTIME-DOMAIN。

裁决规则：

- 用户改变产品行为或 contract：先改唯一 owner 正文和同文件快读卡，再改 canonical source/config/tests。

### 1.1 固定分工和整份实施授权

本次实施已经获得“按本文全部作业卡持续推进”的明确授权。单卡独立闭环用于限制影响面，不代表每完成一张卡就停止：当前卡按第11节交付后，只要仍有未关闭作业卡且没有真实 blocker，就必须重新检查下一卡进入谓词并继续；禁止跨卡混改，也禁止以 `local_verified_only` 为由结束整份实施。

固定角色如下：

- 监控 AI：只读监督轻量 AI 的当前卡、锁、进程、diff、构建范围和 focused 证据；给出具体纠偏建议；独占 `D:\v5` 下 Markdown 的修改职责。除文档及其文档路由/文本门禁外，不修改产品源码、配置、脚本或测试，不启动实现构建、部署或板端动作。
- 轻量 AI：读取本文和唯一 owner，只修改本卡允许的非 Markdown 实际实现，包括源码、配置、构建/安装脚本和 focused tests；执行本卡最小 gate。不得新增、修改、删除或改名任何 `.md`，也不得持有 Markdown 文件锁。
- 轻量 AI 发现 owner、快读卡、本文或其它 Markdown 不正确/缺失时，只报告 `doc_path + owner_req + 冲突章节/行 + 当前源码证据 + 所需语义变化`，暂停受冲突语义影响的编辑；由监控 AI 锁定、修正文档并运行文档门禁。监控 AI 通知完成后，轻量 AI 必须从磁盘重读再继续。
- 监控 AI 发现代码、测试、安装、服务或退役残留问题时，只把精确文件、错误谓词、允许范围和最小 gate 发给轻量 AI，不代替其修改实际代码。
- R02 等纯 Markdown 作业由监控 AI 执行；若文档审计发现非 Markdown 的退役实现或正向入口复活，立即交给轻量 AI 作为当前优先代码退役切片，监控 AI 不直接删除代码。

## 2. 不可破坏合同

所有卡必须同时满足：

1. 坐标、轨迹动态点、红绿点、刀柄刀尖、速度和倍率显示保持 30Hz 目标；按钮状态 10Hz；急停显示和 actual 回读 10Hz；普通诊断 5Hz。
2. 只允许跳过未变化对象、合并尚未发布的同窗 dirty、丢弃过期显示工作；不得降低 native/State freshness，不得用缓存、默认值或重写时间冒充 actual。
3. 运行期禁止周期整屏刷新、整页重建和 full-frame polling；full frame 只允许 initial 或明确 repair。
4. UI 只写 remote BGRA mmap 和 dirty FIFO；18080、HTTP、WebSocket、编码和发送由独立 relay 进程承担。
5. UI、relay、Publisher、网络、日志、诊断和普通 LinuxCNC task/interpreter 不得迁入 CPU0，不得提升为 SCHED_FIFO。
6. CPU0 只属于有界 deadline 的 servo、motmod、kinematics、HAL realtime、EtherCAT、safety 和必要的最小有界 queue-feed 闭集。
7. 即使 CPU1 满载或 UI 卡死，CPU0 仍须无 queue starvation、servo overrun、DC/WKC 异常、编码器不连续和安全失效。
8. 不得通过停 relay、减少客户端、停 Publisher、降低 servo/安全周期、修改产品 G 代码或关闭证据来制造低 CPU。
9. UI/刀路不得改原始 G 代码；dense 诊断样本只进 repo_ignored/cpu1_optimization/scratch，不得替代 cc-ac.ngc、cc-bc.ngc 或原始 MPF。
10. 急停 edge 必须直通 native safety owner，不能被性能节流、debounce、日志或普通命令队列延迟。

当前非暂停 Start 的唯一序列必须保持：

~~~text
Set Task_Plan_Init
Set Mode Auto
Set Open <原始 runtime_program_path>
Set Run 0
~~~

Program Open 只登记原始路径、identity 和显示 metadata，不提前执行 native Set Open。不得删除 Start 中唯一 Set Open，不得改成 sidecar、过滤文件、预览产物或运行副本。

## 3. AI 唯一执行入口和状态机

AI 只能沿下面状态机前进；任一步失败都留在当前状态修正，不得跳到镜像、部署或下一卡。

~~~text
START
  -> P0_AUTHORITY       读取 AGENTS、需求索引、唯一 owner
  -> P0_DOC_HANDOFF     文档有误时轻量只报告，监控 AI 修文档并通知重读
  -> P1_IDENTITY        记录 HEAD、git status、focused diff、相关构建 identity
  -> P2_SELECT          按第6节选择第一个满足进入条件的作业卡
  -> P3_IMPACT_AUDIT    按第4.6节建立精确词元并核对定义、消费者和运行入口
  -> P4_PROVE           用当前源码检测谓词证明缺陷仍存在
  -> P5_LOCK            获取所有待编辑精确文件锁
  -> P6_EDIT_DELETE     编辑；确认退役项在同卡物理删除
  -> P7_LOCAL_GATE      Windows 最近 focused gate；POSIX/ABI 才进 VM 最小 target
  -> P8_SURVIVOR_SWEEP  扫源码、构建、安装、服务、测试、文档正向残留
  -> P9_ANTI_RESURRECT  证明唯一替代 owner；运行旧入口重引入即失败的 focused 门禁
  -> P10_OPTIONAL_BOARD 仅在当前目标要求时部署并走原始 operator 路径
  -> P11_CLEANUP        清本线程锁、探针、临时进程和 task-only scratch
  -> P12_CARD_HANDOFF   按第11节报告本卡唯一结果和下一卡进入条件
      -> CARDS_REMAIN   回到 P2_SELECT；不得因本卡 local_verified_only 停止
      -> ALL_CLOSED     END
      -> REAL_BLOCKER   BLOCKED
~~~

硬停止条件：

- 需求 owner 冲突或没有唯一替代 owner。
- focused 文件存在无法归属的用户改动，且本卡会覆盖同一行或同一语义。
- 精确文件锁或 vm_board.lock 被活跃任务持有。
- 当前 target identity 与待验证源码不一致，且无法完成最小合法投影/构建。
- 定义、直接调用、动态注册、构建安装或运行入口中仍有无法归类的影响面命中。
- 退役路径仍有未迁移的真实消费者。
- 安全运动条件不成立，或板端验收结束无法回到 ESTOP/Machine Off。

### 4.6 直接从当前源码建立关系和影响面

任何卡都不得依赖外部图文件、历史关系缓存或已删除的分析工具。关系证据只来自当前 checkout 的需求 owner、canonical source、构建/部署入口、focused 编译测试和当前运行证据。

先建立本卡精确审计词元，至少覆盖适用项：

- 函数、类型、全局变量、宏、头文件和源文件 basename。
- Python module/import、脚本入口、callback/factory/route/函数指针注册。
- IPC command、opcode、ABI 字段、JSON key、协议字符串。
- HAL component/pin/signal、INI section/参数、EtherCAT PDO/SDO、FPGA 寄存器和 bit。
- CMake target、recipe、package、manifest、installer、service、init/rc link、CLI。
- PIDFILE、SHM、socket、运行文件路径、板端进程名和现役/退役 owner 名称。

不得只查一个函数名便宣称影响面完整。把词元和本卡直接相关目录填入下面模板：

~~~powershell
$AuditTokens = @(
    '<EXACT_SYMBOL_OR_PATH_1>',
    '<EXACT_SYMBOL_OR_PATH_2>'
)

$AuditRoots = @(
    '<CURRENT_OWNER_SOURCE_ROOT>',
    '<DIRECT_BUILD_OR_MANIFEST_ROOT>',
    '<DIRECT_INSTALL_OR_SERVICE_ROOT>',
    '<DIRECT_TEST_ROOT>',
    '<OWNER_DOCUMENT>'
)

foreach ($Root in $AuditRoots) {
    if (-not (Test-Path -LiteralPath $Root)) {
        throw "audit root missing: $Root"
    }
}

foreach ($Token in $AuditTokens) {
    $Hits = & rg -n --encoding utf-8 -F -- $Token $AuditRoots
    $Rc = $LASTEXITCODE
    if ($Rc -gt 1) {
        throw "rg audit failed: $Token"
    }
    if ($Rc -eq 0) {
        $Hits
    }
}
~~~

每个命中必须归入 impact.tsv 的一种类型：

- declaration：声明。
- definition：实现。
- direct_consumer：直接调用或 include 消费者。
- registration：callback、函数指针、factory、route、宏表或字符串动态注册。
- protocol：IPC、ABI、HAL、EtherCAT、FPGA 或 SHM 消费者。
- build：CMake、Make、recipe、package。
- deploy：manifest、installer、service、init、SD。
- test：unit、smoke、fixture、mock、verifier、policy。
- positive_doc：仍把路径或 owner 描述为现役。
- negative_gate：只清除旧板残留或阻止退役实现复活。

无法归类的命中视为影响面未闭合，不得编辑或删除。rg 返回 2 及以上也不得当作零命中；应修正路径/权限/断链问题，或缩小到本卡精确目录后重新执行并记录。

跨模块链必须从真实入口逐跳扩展；每发现一个协议字段、注册名或运行入口，就加入下一轮审计词元：

~~~text
UI event callback
  -> UDS/native command gate
  -> LinuxCNC/HAL owner
  -> EtherCAT PDO/SDO
  -> FPGA register
  -> native actual/readback
  -> allowed SHM display projection
  -> UI display consumer
~~~

每一跳记录 from_file、from_symbol、boundary、token_or_field、to_file、to_symbol、owner_req 和 focused_gate。不得用函数名相似、注释或旧聊天推断关系；字符串加载、回调注册、函数指针、宏展开、service 启动、manifest 安装和运行时 import 必须单独核对。

修改前后固定检查：

1. 精确符号的声明、定义、调用和 include 消费者。
2. callback/函数指针/route/factory/宏表和字符串注册。
3. IPC、ABI、SHM、HAL、EtherCAT、FPGA 字段消费者。
4. CMake target、recipe、manifest、installer 和 package。
5. service、init、CLI、PIDFILE、SHM、socket 和运行文件路径。
6. unit/smoke/mock/verifier/policy 与 owner 文档现役示例。
7. 受影响 target 的真实 focused compile/test。
8. 当前任务要求的 package 或板端 process/file/link/actual survivor。

需要比较 BUS、Pulse、Home、Jog、急停等跨层链路时，直接生成 impact.tsv，不生成第二份关系文档：

~~~text
feature	ui_entry	ipc_command	native_owner	hal_component_or_pin	ethercat_pdo	fpga_register	actual_readback	safety_gate	build_target	runtime_service
~~~

每个单元格必须指向当前文件、符号或唯一 owner；缺证据留空并报告，不得猜测补齐。文本零命中只能证明没有找到该精确字符串，不能替代编译、测试、部署闭包、native actual/readback 或板端存活证明。

## 5. 退役路径物理删除协议

### 5.1 一句话硬规则

退役不是改名、禁用、停止调用或留兼容壳。只有唯一替代 owner 已覆盖输入、输出、readback、安全、构建、安装和部署依赖后，才可确认退役；一旦确认，必须在迁移真实依赖的同一作业卡内物理删除并清理干净，不能留到下一卡。

### 5.2 退役确认门

删除前逐项满足：

1. 唯一替代 owner、ABI、readback 和启动闭包已由需求 owner 与源码证明。
2. 按第4.6节建立旧符号、文件、service、CLI、安装路径、SHM/PIDFILE 等精确词元，逐层核对当前源码、动态注册、构建、部署、测试、owner 文档和运行入口。
3. 旧符号、文件名、service 名、CLI 名、SHM/PIDFILE 名、安装路径的真实消费者已全部迁移。
4. 替代路径 focused compile/unit/contract gate 通过。
5. build/CMake、manifest、installer、init/service、SD、package、verifier、policy、tests 和正向文档引用已列入删除清单。
6. 若需要板端关闭，当前进程、打开文件、SHM、PIDFILE、init/rc 链接和安装文件均已纳入清残证明。

任一项不满足：不得删，也不得加 fallback；报告 blocked 及缺失的唯一接受条件。

### 5.3 同卡必须物理删除的范围

- 旧源码文件、声明、定义、调用、include 和只为旧路存在的常量/状态。
- CMake/Make/recipe/manifest/package/install 条目。
- init/service/rc 链接、daemon 启动、CLI 参数、环境开关和测试入口。
- wrapper、alias、兼容转发、disabled 分支、编译开关、空函数和 TODO。
- _old、legacy、bak、backup、archive、repo_ignored 副本、VM 副本、板端手工副本。
- 把旧路径描述为现役 owner 的文档正向引用和示例。

删除只能用 apply_patch 的 Delete File 或精确行删除；禁止宽泛递归删除，禁止先备份，禁止恢复旧 fallback 让 canonical 暂时“能跑”。

### 5.4 正向残留和负向清残必须分开

零残留要求针对能安装、导入、执行、调用、注册或宣传旧实现的正向路径，它们必须为 0。

为清除已部署旧板而保留的单向 rm -f、test ! -e 和 source-absence policy 是负向迁移门禁，不是旧实现。只有 owner 明确要求兼容已发布旧镜像原位升级时才可逐项 allowlist；它们不得携带旧实现 payload，不得安装、启动、import 或调用旧代码，并必须绑定具体旧版本、移除条件和唯一 installer owner。owner 判定升级支持窗口结束时，由轻量 AI 在“结束该窗口”的代码退役卡内同步物理删除这些 tombstone；不得无限期保留或复制到新入口。

不得为了追求文本零命中而提前删除唯一旧板清残门，导致现场继续残留旧进程或旧文件。

### 5.5 板端删除边界

禁止 SSH/SCP/sed/tee/vi/nano 直接改板端产品文件。板端清残必须来自 Windows canonical source 构建的 installer/package/rootfs；验证只读检查 process、file、service、link、SHM 和 readback。

### 5.6 每卡固定零残留检查

精确旧符号检查：

~~~powershell
rg -n --encoding utf-8 -F '<OLD_SYMBOL_OR_PATH>' -- <本卡精确目录>
if ($LASTEXITCODE -eq 0) { throw 'retired positive survivor remains' }
if ($LASTEXITCODE -gt 1) { throw 'rg survivor scan failed' }
~~~

文件存在性、manifest 和 service 检查必须逐项列出，不能只搜索一个函数名。对每个 retired token，`retirement.tsv` 必须把下列层级逐层记录，不能聚合成一个总数：source、config、build、recipe、manifest、package、installer、service/init/rc、CLI、test/verifier/policy、现役 doc。每层都要记录精确 roots、实际命令、退出码和数值命中数；没有对应层也必须写 `N/A` 和证明命令。任一适用层未执行，本卡不得关闭。

最终 retirement.tsv 还必须记录：

- 删除的文件和符号。
- 删除的 build/install/service/test/doc 正向入口。
- 仍保留的每个负向项、kind、owner 理由和移除条件。`cleanup_tombstone` 只指产品 installer 中兼容已发布旧版本的单向清残动作；`anti_resurrection_assertion` 只指 test/policy 的禁用字面量或 source-absence 断言，二者不得混称，也不得携带旧实现 payload。
- focused gate 与源码、package、板端 survivor 计数。

### 5.7 防复活和错误引用门禁

物理删除只关闭当前残留，不能单独证明以后不会恢复旧路。每个退役项在本卡结束前必须同时建立以下最小门禁：

1. 为旧符号、文件、module、service、CLI、manifest/package entry、安装路径、SHM/PIDFILE 名分别登记精确 token，不得只登记其中一个 basename。
2. 明确唯一替代 owner、唯一构建/安装入口和唯一运行入口；focused test 或 runtime policy 必须断言它们各自只有一个现役定义/登记，不能只断言旧文件不存在。
3. 对 source、config、build、recipe、manifest、package、installer、init/rc、CLI、tests、verifier 和现役文档建立正向扫描根；旧 token 在这些根中的可执行、可导入、可安装、可启动、可调用或现役描述命中必须为 0。
4. 旧 token 只允许出现在逐文件登记的负向 allowlist：`rm -f`、`test ! -e`、source-absence 断言或 policy forbidden token。allowlist 必须记录文件、语义、owner 理由和移除条件；不得用目录、通配符或“测试需要”笼统放行。
5. 最近的 focused policy/test 必须先在当前 canonical checkout 返回 0，再用同一 detector、同一 retired token 集和同类扫描根对仅存在于 task-only 临时 fixture 的旧入口返回非零；必须核对包含旧 token 与命中层级的预期 failure marker。失败探针不得修改 canonical source/config/test，不得以无关异常、缺文件或语法错误冒充命中；结束后必须证明 fixture 和临时进程已清理。禁止通过把旧 token 从审计词表删除、缩小扫描根或把正向命中改称注释来获得通过。
6. test、mock、example 和文档不得继续 import、执行或示范旧实现。测试中保留旧 token 只能用于负向断言，且不能附带可恢复旧实现的源码、wrapper、fixture 副本或命令行。

`retirement.tsv` 对每个旧 token 至少写一行：

~~~text
retired_token	layer	positive_roots	detector_command	exit_code	positive_hit_count	negative_kind	negative_file_and_semantics	negative_owner_reason	negative_removal_condition	canonical_replacement	unique_entry_assertion	anti_resurrection_gate	canonical_result	injected_failure_result_and_marker	fixture_cleanup_result
~~~

任一旧 token 的正向命中不为 0、替代 owner 不唯一、负向命中未逐项 allowlist，或门禁没有证明“重新引入会失败”，本卡都不得报告关闭。

10. 防复活：断言唯一替代 owner，运行旧入口重新引入即失败的 focused policy/test，并逐项复核负向 allowlist。
11. 可选板端：只在授权条件成立时执行原始 operator/readback。
12. 清理与交付：释放本线程锁和探针，按第11节报告。

## 9. 建议性能预算

以下只用于优化判断，不是 owner 需求真源：

| 场景/指标 | 建议目标 |
| --- | --- |
| 两个远程客户端、Machine Off 空闲 | CPU1 低于约 35%至40% |
| cc-ac 连续运动 | CPU1 p99 低于约 80%至85%，不得持续 100% |
| display frame work | p99 小于 33ms，无历史补帧积压 |
| relay shared producer | 建议低于约 10% |
| runtime full frame | initial/真实 repair 外为 0 |
| LVGL invalidation | frame 内不超过 32 项，无 overflow 退整屏 |
| Publisher | 每个 owner 恰好一个实例、锁和 writer identity |
| source freshness | 正常不超过约 2至3 个显示周期，超限明确 stale |
| CPU0 运动域 | servo overrun=0、DC/WKC 稳定、queue 无 underflow |

CPU 数字未达预算但功能、安全和 CPU0 隔离全部满足：报告真实数据并继续定位，不得降功能。CPU 下降但 actual、remote、30Hz 或安全回退：优化失败。

## 10. 全局一票否决

- 确认退役后不物理删除，改成 wrapper、alias、disabled、环境开关、编译开关、旧文件改名、TODO、test-only 入口或 backup。
- 退役项只做当前零命中扫描，却没有唯一替代 owner 断言、分层正向零引用证明和“旧入口重引入即失败”的防复活门禁。
- 把旧实现放入 test/mock/example/repo_ignored，或把可执行、可导入、可安装、可启动的正向残留伪装成负向清残 allowlist。
- 恢复退役 Python safety/RTCP/G53 owner、第二 publisher 或 canonical 暂坏时的旧 fallback。
- 保留 V5 NML 5005 remote TCP、linuxcncrsh 5007 wildcard/external listener、部署 probe 的 direct `Set`/`Machine On` 模式，或用固定口令/防火墙代替唯一 UDS gate 闭合。
- 让 `/dev/uio*`、`/dev/EtherCAT*` 保持或回退为 world-writable，使用不稳定 `/dev/uio0` 序号，或在权限失败时 `|| true` 继续启动。

- 把 J15-J18 的 policy 只写成 audit/warn 后仍返回 0，或失败探针因无关异常而“通过”。
- 把 UI、relay、Publisher、网络、日志、诊断或整个 milltask 迁入 CPU0。
- 降低 servo、安全、EtherCAT、坐标/刀路 30Hz、按钮/急停 10Hz、诊断 5Hz。
- 停 relay、断远程、减少正式验收客户端或改手动刷新。

- sampler、临时 daemon、本线程锁或测试进程在交付后残留。

## 附录 B：根因与场景边界

空闲固定负载：

1. relay 的 Python shared producer、行拷贝、payload 固化和两个客户端发送。
2. Position→State→UI 快链的临时文件/rename、fopen/fread、open/ftruncate/mmap/msync/munmap/close。
3. Publisher 超期后的零等待追赶反馈。
4. following-error/坐标最终字符串变化持续产生 dirty；这不是旧 G 代码脏数据。

cc-ac 是运动放大器：当前 cc-ac_siemens.mpf 为 20 个连续长 G3，按 5 度预览约 361 点。任意 DYNAMIC 被升级 POSE 会重复投影；64 个全窗 line 对象与最多 12 个 dirty rect 的源码上界约 4320 次 draw/帧。随后像素继续经过 LVGL→g_frame→physical/remote mmap→relay payload→socket。

dense-G1 是条件分支：只有同窗证明 milltask/usrmot/emcTaskEager 是相对 cc-ac 的主要新增负载，才允许修改 LinuxCNC。不能用 20 个长 G3 的 cc-ac 证明 dense-G1 ack polling。

“脏数据”判断必须分层：

- live duplicate/writer：必须有进程、锁、writer identity、generation 证据。
- stale source 被洗 fresh：由 source timestamp 链证明。
- 已确认退役代码：不会仅因文件存在自动耗 CPU，但必须按第5节清除，避免错误入口、双 owner 和未来复活。
- 无 caller 死函数：不产生现役 CPU，但确认退役后仍须物理删除，保持唯一实现。
