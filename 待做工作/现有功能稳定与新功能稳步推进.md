# 现有功能稳定与新功能稳步推进

## 0. 文档定位

本文记录当前项目推进目标和执行方法，属于 `待做工作/` 下的活动输入，不是产品需求真源、第二套架构规则、任务看板或完成证明。

执行时必须按以下入口回到唯一真源：

- 需求路由和唯一 owner：[`功能/需求真源索引.md`](../功能/需求真源索引.md)。
- 全局配置、日常增量和构建边界：[`功能/全局通用配置需求.md`](../功能/全局通用配置需求.md)。
- 控制链、状态链和 native owner：[`功能/系统代码架构硬边界守则.md`](../功能/系统代码架构硬边界守则.md)。
- 验证状态、板端和 operator 闭环：[`功能/自动闭环测试方式.md`](../功能/自动闭环测试方式.md)。
- 每次任务自动执行核：[`AGENTS.md`](../AGENTS.md)。

本文与上述 owner 冲突时，先按需求索引找到唯一 owner；只有用户需求确实变化时才修改 owner，不能用本文覆盖 owner。

## 1. 当前总目标

项目仍有较多功能尚未加入。后续开发必须同时满足两个目标：

1. 现有功能、架构和板端链路不能因新增功能持续退化或形成第二实现。
2. 新功能以可独立验证、可部署、可回读的最小纵向切片稳步进入 canonical mainline。

主干在任意时刻都必须能明确回答：

- 当前实现读取哪个唯一 owner；
- 当前控制和状态经过哪条 canonical 链；
- 哪些功能已有本地或板端证据；
- 当前最高状态是 `source_only`、`local_verified_only`、`board_verified` 还是 `blocked`；
- 尚缺的验证是什么，满足什么条件才能继续提高状态。

## 2. 必须保护的现有底座

普通新功能默认只能消费下面的底座，不得私自复制、旁路或重新拥有它们。

### 2.1 控制链

```text
LVGL/C UI
  -> framed UDS native Command Gate
  -> LinuxCNC/native owner
  -> HAL/EtherCAT/drive/FPGA
```

- UI、Python业务脚本、临时工具和SHM不得形成第二控制入口。
- 高频实时控制、安全和物理保护继续留在LinuxCNC realtime、HAL、EtherCAT、驱动或FPGA/native owner。
- 新功能需要新的控制语义时，先登记唯一owner、opcode/ABI、readback和失败边界，再修改消费者。

### 2.2 状态链

```text
native resident owner/API
  -> 登记的显示投影 producer
  -> typed SHM
  -> 本地UI和远程显示消费者
```

- SHM只承担owner允许的显示投影，不承担控制、保存、安全、门禁或动作成功真值。
- generation、writer identity、SeqLock、CRC和freshness必须随ABI保持闭合。
- 新增字段先判断能否由native owner直接提供；不能为了UI方便把JSON、缓存或SHM变成第二真源。

### 2.3 实时和生命周期

- CPU0继续只承载运动实时闭集；UI、Publisher、Command Gate用户态传输、网络、日志、文件和诊断留在CPU1。
- startup DAG只使用真实owner事件和最终barrier，不加入固定sleep、轮询旁路、第二ready文件或重复完整合同。
- UI或CPU1失效只能让显示/普通交互降级，不能让正在执行的运动依赖CPU1继续获得调度。

### 2.4 源码、构建和部署

- Windows `D:\v5` 是唯一canonical owner；VM投影、板端、staging和旧SD都不是源码真值。
- 日常修改先完成Windows focused gate，再构建最小受影响target/package。
- runtime文件通过部署manifest进入板端，不用SSH/SCP直接修改产品文件实现修复。
- 未改变部署内容时不自动生成rootfs/image；发布和离线认证继续作为独立里程碑。

## 3. 当前推进要求

### 3.1 同时进行的工作上限

- 同一时间只允许一个P0/P1功能处于主动实现状态。
- 一次只关闭一个具体缺陷、一个owner链和一个最小target。
- 当前切片未达到`local_verified_only`前，不开始另一个同层功能。
- 新发现的独立问题保留为下一切片输入，不顺手扩成架构整理或多功能重构。
- 文档、测试、source、build和板端状态必须始终区分，不能把计划写成已经实现。

### 3.2 未开始或未完成的功能

- 未正式进入实现的功能只保留需求输入，不提前注册runtime、init、CMake、manifest、IPC或SHM字段。
- 探索性实现只放在`repo_ignored/<work-id>/scratch/`，验证结束后要么进入正式纵向切片，要么删除。
- 不用disabled分支、环境开关、wrapper、fallback、shadow path、备份文件或旧入口保存半成品。
- 正式实现一旦接入mainline，必须在同一切片删除已确认退役的重复路径。

### 3.3 功能优先级

按以下顺序选择下一项工作：

1. 会破坏运动、安全、owner真值或现有可用链路的缺陷。
2. 已经导致当前focused gate、板端启动或核心操作路径失败的问题。
3. 多个后续功能共同依赖的唯一公共能力。
4. 能独立形成用户价值和验收闭环的新功能。
5. 不影响正确性的性能、体验和维护性优化。
6. 只在发布、灾难恢复或法律边界变化时触发的离线全量认证。

每个新功能开始前必须先填写下面内容。产品语义写回索引登记的唯一owner；本节模板只用于当前切片执行，不形成第二真源。

```text
功能名称：
用户最终结果：
唯一REQ/owner：
当前实现事实：
控制链入口：
状态owner/readback：
持久化owner/writer：
涉及安全或运动：是/否
允许修改文件：
禁止修改文件：
最小build target：
需要删除的旧路径：
本地focused gate：
板端operator路径：
最低关闭状态：source_only | local_verified_only | board_verified
缺失证据和下一条件：
```

准入卡无法明确唯一owner、控制链、readback或安全边界时，只允许继续读owner和源码，不得开始写路径或创建新的状态文件。

## 5. 单功能落地方法

### 5.1 开始前

1. 读取最新用户需求和本文件。
2. 从需求索引找到唯一REQ和owner，只读owner快读卡及命中正文。
3. 读取当前canonical source、`git status`和focused diff。
4. 明确已有用户/其它操作者改动，禁止覆盖。
5. 冻结允许修改文件、最小target和最低关闭状态。
6. 获取精确文件锁；目标文件已有活跃锁时只做其它不冲突工作。

### 5.2 需求与contract

- 用户需求或产品行为变化：先修改唯一owner正文和同文件快读卡，再修改source/config/test。
- owner已经正确而实现未兑现：引用owner，直接修canonical source/config/test，不重复改写文档。
- 先补最小contract、smoke或失败复现，使当前缺口可以稳定重现。
- contract必须验证行为、owner和readback，不能只检查字符串或文件存在。

### 5.3 实现

- 下一步直接编辑canonical source，不持续扩大全仓分析。
- 优先复用现有owner、协议、helper和构建target，不新增平行链路。
- ABI、schema、部署项变化时，同步writer、reader、manifest和focused test。
- 发现退役路径时，真实依赖迁移完成后在同一切片物理删除。
- 不做与当前用户结果无关的顺手重构。

### 5.4 本地关闭

按受影响范围运行最近的验证：

- 文档：严格文档路由校验和`git diff --check`。
- Python：touched `py_compile`及focused pytest/contract/self-test。
- C/CMake/LVGL：Windows syntax/static/CMake最小target及对应smoke。
- native/SHM/parameter：现有focused audit、ABI/owner/readback smoke。
- Windows客户端：对应`.NET` build和协议/更新/显示smoke。
- FPGA：改动命中时运行当前owner的约束、仿真、synth/impl/timing/DRC gate；未修改不重复运行。

focused gate通过且没有要求立即上板时，当前切片可以交付`local_verified_only`并进入下一项工作。

### 5.5 板端关闭

以下情况进入板端闭环：

- 用户明确要求上板或取得`board_verified`；
- 当前功能本身只能由板端ABI、硬件actual或operator路径证明；
- 已到板端集成批次或发布里程碑；
- UI可见改动要求现场确认。

板端顺序：

```text
Windows focused gate
  -> 最小受影响ARM/package
  -> 必要manifest/current artifact
  -> 原子部署
  -> runtime/ui_ready/readback
  -> 原始operator路径
  -> fresh截图或native/motion证据
  -> 最终ESTOP安全状态
```

板端UI变化必须使用fresh board frame和真实operator路径；本地截图、旧截图、代码diff和后台接口不能替代板端可见结果。

## 6. 四层回归体系

### L0：编辑级

每次编辑立即执行，只覆盖当前文件及直接依赖：

- 文本/语法检查；
- 最近compile或contract；
- focused diff和锁释放。

### L1：切片级

每个功能达到`local_verified_only`前执行：

- 受影响模块smoke；
- 直接上下游协议/ABI；
- 当前target构建；
- 旧路径退役检查；
- runtime policy中与当前模块直接相关的门禁。

### L2：板端集成级

多个已经本地关闭、依赖一致的切片组成一次板端批次：

- source/artifact/deploy identity；
- runtime manifest和服务启动；
- SHM/Publisher/Command Gate readiness；
- 受影响operator路径；
- UI改动fresh截图；
- 安全回读和ESTOP终态。

### L3：发布级

只在发布或重大里程碑执行：

- 三次断电冷启动和阶段计时；
- 设置页A-G及核心主页黄金链；
- AC/BC匹配golden motion；
- CPU1满载/UI冻结下运动连续性；
- servo latency、overrun、WKC/DC和编码器连续性；
- FPGA timing/DRC及硬件identity；
- 完整SD生成、写入、全量回读和冷启动；
- 必要时空缓存离线恢复认证、安全和CVE审查。

## 7. 现有功能生命线

每次普通切片只回归受影响部分；每次板端集成或发布按风险扩展到以下生命线：

1. BUS和Pulse模式启动及active model/readiness。
2. ESTOP、取消急停、Machine On/Off和最终安全状态。
3. Home及对应native actual。
4. Jog按下、持续、松开和watchdog。
5. Program Open、Start、Pause、Resume和Abort。
6. WCS、Work Zero、G92和G53。
7. AC/BC模型、RTCP和旋转轴连续角。
8. 轴映射、驱动参数设置及fresh readback。
9. 保存并彻底clean restart后的owner actual。
10. 本地UI、远程显示/输入、程序上传和必要售后入口。

当前仓库没有一个自动完成全部设置、真实点击、真实触摸和motion golden loop的单一runner。近期先复用现有focused工具组合，不为了追求一个“全能入口”阻塞功能；当重复组合已经稳定后，再补只负责调度现有工具、输出机器可读结果的最小入口。

### 7.1 当前回归入口映射

本地C smoke统一使用CMake/CTest登记，不另造测试实现：

```text
cmake -S board -B repo_ignored/build/v5_local -G Ninja -DCMAKE_BUILD_TYPE=Debug
cmake --build repo_ignored/build/v5_local --target v5_local_regression
ctest --test-dir repo_ignored/build/v5_local --show-only=json-v1 -L v5_local
```

`v5_local_regression`只构建并调度当前原生主机可编译、可执行的已有smoke。Windows当前登记15项跨平台smoke；原生UNIX主机构建额外登记21项依赖POSIX/Linux ABI的smoke。ARM交叉构建不登记可执行CTest入口，避免在x86 VM误跑ARM产物。它是L1本地回归入口，不证明板端服务、真实触摸、Home/Jog、运动、安全actual、clean restart或golden motion已经通过。

| 生命线 | 已有L1/focused入口 | L2/L3必须补齐 |
| --- | --- | --- |
| 1. BUS/Pulse启动、模型与readiness | `v5_microkernel_manifest_smoke`、`v5_position_status_publisher_smoke`、`check_v5_active_model_branches.py` | 两模式冷启动、runtime readiness、active model/native actual |
| 2. ESTOP和Machine状态 | `v5_estop_clean_state_smoke`、`test_v5_estop_abort_rtcp_output_policy.py` | 原始operator路径、安全actual、最终ESTOP/Machine Off |
| 3. Home | UNIX `v5_main_page_actions_smoke`、`v5_home_router_bridge_smoke.py` | 真实Home运动、motion complete、in-position和native坐标回读 |
| 4. Jog与watchdog | UNIX `v5_main_page_actions_smoke`、`test_v5_linuxcnc_control_surface_policy.py` | 真实按下/持续/松开、watchdog和编码器位移 |
| 5. Program Open/Run控制 | `v5_program_command_smoke`、UNIX `v5_program_runtime_smoke`、`v5_program_page_double_click_smoke` | UI打开原始程序、program identity、Start/Pause/Resume/Abort actual |
| 6. WCS/Work Zero/G92/G53 | UNIX `v5_native_wcs_status_smoke`、`v5_native_g53_geometry_status_smoke`、`v5_native_work_zero_smoke` | UI操作、唯一native owner readback和坐标实际结果 |
| 7. AC/BC、RTCP与旋转轴 | `v5_program_scene_producer_smoke`、UNIX `v5_native_rtcp_status_smoke`、`v5_wcheckpoint_router_handoff_smoke.py` | active model匹配的`cc-ac.ngc`/`cc-bc.ngc`及旋转轴actual |
| 8. 轴映射与驱动参数 | `v5_drive_write_window_smoke`、UNIX `v5_settings_axis_table_smoke`、`v5_bus_module_owner_smoke.py` | 设置页owner write/readback、重启后HAL/native/drive实际映射 |

| 10. 本地UI、远程和程序入口 | `v5_coordinate_digits_smoke`、`v5_lvgl_remote_display_delta_smoke`、`v5_ui_shell_refresh_smoke`、UNIX UI smoke | fresh board frame、逐次输入后新frame、上传/删除/诊断真实入口 |

新增功能先选择受影响生命线并运行对应focused入口；本地统一入口失败时停止合入新功能并修复当前切片。板端批次只扩展到受影响生命线，发布里程碑才跑完整L3。

## 8. AI执行模板

每次交给AI的任务至少包含：

```text
当前功能/缺陷：
用户可见结果：
唯一REQ/owner：
允许修改文件：
禁止修改文件：
最小target：
当前已知基线：
最低关闭状态：
是否要求立即上板：
```

AI交付必须包含：

- 实际修改文件；
- 实际运行的验证及结果；
- 当前状态词；
- 未执行的板端、触摸或运动测试；
- blocker及下一接受条件；
- 是否改变控制链、状态owner、ABI、生命周期或部署闭包。

建议把实现、独立审查、验证和发布检查拆成不同任务上下文，避免同一上下文把自己的假设当成验收事实。

## 9. 集成节奏

推荐循环：

```text
选择一个功能
  -> owner/准入卡
  -> contract和实现
  -> focused gate
  -> local_verified_only
  -> 原子提交当前关闭切片
  -> 选择下一功能

积累依赖一致的本地关闭切片
  -> 一次板端集成
  -> 受影响生命线回归
  -> 形成新的板端稳定基线

达到发布里程碑
  -> 完整L3验证
  -> 固定source/artifact/manifest identity
```

板端批次不是长期拖延验证：涉及真实硬件语义、ABI或用户明确要求的改动必须按对应owner及时上板；普通局部功能不因追求最高状态反复重建整张镜像。

## 10. 停步条件

遇到以下情况停止扩大实现范围：

- 唯一owner无法裁决当前需求；
- 目标文件被其它活跃操作者锁定；
- 当前控制/状态链存在两个真实writer或两个运行入口；
- 当前target直接输入缺失、hash不符或ABI不闭合；
- 当前生命线因本轮修改变红；
- 板端运动、安全或readiness actual失效；
- 需要新的用户授权、外部协调或破坏性操作。

如果当前基线本来就有已知缺口，不要求先修完整个项目；必须明确它是既有缺口、当前切片是否使其恶化，以及本切片自己的关闭条件。

## 11. 当前执行顺序

1. 确认一个工作区干净、文档路由和runtime policy通过的当前source基线。
2. 只登记仍真实存在的未闭合问题，删除过期或重复遗留项。
3. 固定上述十条生命线与现有测试入口的对应关系。
4. 从待加入功能中选一个最高优先级、可独立关闭的功能填写准入卡。
5. 按单功能落地方法做到`local_verified_only`。
6. 达到板端批次条件时统一构建、部署和回归。
7. 每个发布里程碑执行完整L3验证并固定identity。

## 12. 本工作方法的完成条件

当后续功能持续满足以下条件时，可认为项目已经进入稳定推进状态：

- mainline没有半接入runtime的功能和第二链路；
- 每个进行中的功能都有唯一owner、readback和最小target；
- 任意时刻最多一个P0/P1切片在实现；
- focused gate可以快速暴露当前切片回归；
- 板端批次能重复从Windows source生成、部署并回读；
- 生命线失败会立即停止新增功能，而不是继续累积；
- 状态词始终与实际证据一致；
- 规则和门禁只为真实重复风险存在，不继续无限增长。
