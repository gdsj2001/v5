# 遗留问题记录规则与模板

上下文压缩的时候重读AGENTS.md 如果规则和本文档有冲突修改AGENTS.md 有解决不了的问题记录到遗留.md

本文只保留遗留问题记录规则与可复制模板；不要在本文保留具体的遗留问题条目、过程日志、命令流水、历史聊天或 AI 推理。

## 记录规则
上下文压缩的时候重读AGENTS.md 如果规则和本文档有冲突修改AGENTS.md 有解决不了的问题记录到遗留.md
- 本文只记录未闭合、被阻塞、延期处理、未完成板端/运动验证，或必须留给后续验收的遗留项。
- 已修复且拿到对应验收证据的条目必须删除；被功能 owner 文档、改进文档或代码验收记录取代的旧条目也必须删除。
- 同一控制链、同一门禁、同一部署/板端验收路径的问题合并为一条，不按聊天轮次、执行轮次或失败次数重复追加。
- 每条只写问题、影响范围、当前状态、已有证据、阻塞/缺口和下一验收条件；不要写过程日志、命令流水、长证据、历史聊天或 AI 推理。
- 证据只保留最短摘要和必要的本地相对路径；大段输出、截图原始文件、临时脚本日志应放在对应 `repo_ignored/<task>/` 下。
- 状态必须使用明确结果词：`blocked`、`source_only`、`local_verified_only`、`not_board_verified`。已经 `board_verified` 的条目不应留在本文。
- 板端、UI、operator、motion 验证必须按 `AGENTS.md` 的增量 UI 自动化门禁执行；本文只记录尚缺哪一步验收，不重复粘贴门禁全文。
- 本文不是任务看板、锁文件、发布说明或过程文档；不要在这里规划日常任务、登记锁、记录 push 状态或沉淀临时诊断入口。

## 清理规则

- 发现旧条目已经被证据关闭：直接删除该条。
- 发现旧条目描述已经过期但问题仍存在：改写成当前最小问题、当前状态和下一验收条件。
- 发现多个条目指向同一关闭条件：合并为一条，并删除重复条目。
- 发现条目只有代码细节、过程流水或聊天背景，没有明确缺口和验收条件：删除或改写为模板格式。
- 每次结束时，本文应只留下仍然需要人或后续 agent 接手的真实未闭合事项。

```markdown
相关入口：
```
