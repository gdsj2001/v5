## 拆分停线规则

- 最高规则以 `AGENTS.md` 的 `TOUCH_OWNER_DOWNSHIFT` 为准；执行方法以 `待做工作/拆分.md` 为准。
- 1000 行以上且混职责的产品运行时文件必须继续下沉；700-1000 行且混状态机、协议、业务流程或跨层决策的文件优先下沉。
- 500-700 行文件不是机械必拆项；只有内部存在多个职责，或本轮需要修改并会继续承接业务逻辑时，才按 owner 原子化拆分。
- 纯表、纯数据、纯 UI 样式可以超过 500 行，不作为机械拆分目标；兼容壳、旧入口薄转发、fallback wrapper 和 rename-only survivor 不得作为拆分停线理由，发现后必须迁到真实 owner 并删除壳。
- 拆分目标是让单个 owner 可读、可测、可接回，而不是清零所有 500 行以上文件；拆到主入口只保留 routing、dispatch、glue、生命周期组合或兼容薄转发即可停线。

## 工具和编译原则

- `tools/v3_c_owner_lift.py`：按函数名机械提取 C/INC direct route，生成 owner `.c/.h`、原文件薄转发、include/CMake 接入；遇到不支持依赖 fail-closed，支持 dry-run 后 apply。
- `tools/v3_py_top_level_lift.py`：按顶层函数/类名机械提取 Python 顶层 owner，生成 owner module 并给原文件补 import；用于 websocket/input message 等非 class method 职责下沉。
- `tools/v3_py_compat_bind.py` 已退役并删除；不得再生成 `bind_exports()` 兼容壳或 `globals().update(...)` 旧 API 幸存者。旧 API 名不再作为保留理由，仍被支持的调用方必须迁到真实 owner。
- 拆分工作优先使用自动化工具；遇到重复统计、接入检查、合同扫描或本地门禁缺工具时，先补最小可复用工具再继续拆分，目的是提高效率并减少人工漏项。
- 取消每刀和单项 VM/ARM 编译；VM/ARM 不能作为首轮编译器或调试工具。拆分1范围内的拆分目标在本地行数/owner 检查、本地编译或语法检查、focused contract 和 `git diff --check` 全部闭合后，才允许全量同步当前 PROJECT_ROOT 到 VM 并一次性 ARM 编译。
- `拆分1.md` 只记录当前对象、规则、状态、下一关闭条件、复核项和最终验证证据；行数表和建议测试尽量由工具生成，避免人工写错；不要堆历史流水或已完成文件清单。
- 每拆一个 owner，必须同步配一个小 focused contract；不要把新测试继续堆进已有超大测试文件。
- C/LVGL 拆分优先级保持：先收完当前 `v3_product_platform.c` 到 500 行以下；旧 `v3_product_command_routes.inc` 已物理删除，不得作为后续拆分对象恢复；再按 P0 顺序处理 `v3_command_broker.py`、`v3_remote_ui_relay.py`、`v3_drive_profile_download.py`、`v3_ui_app.c`、`product_actions_common_tail.py`。

## 执行程序

1. 重读 `AGENTS.md`、`待做工作/拆分.md` 和本文档，确认当前目标文件、下一关闭条件和最新人类反馈。
2. 运行 `git status --short`，查看目标文件、候选 owner、测试和工具的 dirty/untracked 状态；不得覆盖已有改动。
3. 重新统计当前目标文件和候选 owner 行数，不沿用表格旧数字。
4. 如果当前目标文件超过 700 行且混合职责，继续拆当前目标；如果是 501-700 行，先确认存在多个职责或本轮会继续承接业务逻辑后再拆；如果已经低于 500 行或属于纯表、纯数据、纯 UI 样式、纯兼容壳，先完成本地门禁和状态记录，再进入下一项。
5. 每刀只选一个职责族，例如 route table、job routes、drive routes、program routes、touch event0、remote relay、parser、projection。
6. 新建职责清楚的 owner/sub-owner 文件；名称必须表达职责，不使用无边界的 `utils`、`common`、`helper`。
7. 原大文件只保留 routing、dispatch、glue 或生命周期组合；不得保留兼容薄转发、旧入口转发、fallback wrapper，也不得新增业务 helper、状态机、解析器、算法、表驱动块或跨层决策。
8. 同步 import/include、registry、CMake/source list、runtime manifest、init/deploy、包清单、测试读取路径和必要文档引用。
9. 为新 owner 配套 focused contract；已有测试过大时，新建小测试，不继续堆进大测试文件。
10. 先跑本地检查：touched Python 的 `py_compile`、focused pytest/contract、`git diff --check`。互相独立的 focused pytest 可用 `python -m pytest -n auto ...` 加速。
11. 本地门禁失败时，先修当前刀或补最小本地工具；不得跳到 VM/ARM、全流程流水线或板端碰运气。
12. 拆分1范围内目标全部本地闭合后，才全量同步当前 PROJECT_ROOT 到 VM 并一次性 ARM 编译。
13. VM/ARM 全量编译通过后，才进入上板验证；没有真实 board/operator/motion 验证，不得写板端通过，普通拆分刀口不主动上板。
14. 有实质进展后，更新本文档对应条目的状态、行数、owner、已跑本地证据和下一关闭条件。
