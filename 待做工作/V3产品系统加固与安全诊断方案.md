# V3 Product Hardening And Diagnostics

本文只约束 V3 产品硬化、诊断分级、配置保护、通信隔离、SHM/NML、发布镜像裁剪、许可证合规和 Support Mode。A/C 旋转轴 anti-wrapping、drive rebase、homing 和 MPG 安全见 `V3_ROTARY_ANTI_WRAPPING_STRATEGY.md`。

## 0. 发布硬化最终基线

本文件的最终口径是“发布镜像收敛暴露面，但保留可重建、可诊断、可合规交付、可切换硬件 profile”。当前量产运行事实是 EtherCAT；后期 PL Ethernet DMA、Pulse/STEP-DIR 和 hybrid profile 仍是必须保留的工程边界，不能把清理策略写成 EtherCAT-only。

发布硬化统一门禁：

| 主题 | 最终策略 | 必须证据 |
| --- | --- | --- |
| 发布镜像清理 | 只在 staging rootfs 和固件打包流程中执行；运行板端不得热删 LinuxCNC、HAL、rootfs 或产品 runtime 文件 | inventory、dry-run manifest、apply manifest、回滚来源、清理前后 SHA256 |
| GPL 两层交付 | 第一层 rootfs 留存 `COPYING` 和 `Source_Offer.txt`；第二层随设备或交付包提供可见开源告知 | rootfs 路径/权限/hash、交付介质或文档 hash、出货 checklist |
| UI 白标 | LVGL 操作员 UI 和 About 页可以不出现 LinuxCNC/GPL/调试入口；合规资料通过 rootfs 和随附资料满足 | UI 截图或资源清单、许可证/Source Offer 交付证明 |
| rootfs 去特征 | `/etc/issue`、`/etc/motd`、hostname、shell prompt 可产品化；不能隐藏安全告警或许可证事实 | 登录文件 hash、support diagnose 导出的真实版本/rootfs/kernel/product manifest |
| INI/HAL minify | 仅生成发布副本，仓库真源和源码交付包保留可读原文；必须用解析器或保守规则处理注释 | minify 前后 SHA、LinuxCNC parse、HAL load、profile smoke、失败回退记录 |
| Python GUI 依赖裁剪 | 只裁剪 Axis/Gmoccapy/Touchy/QtVCP/GladeVCP 等不参与 V3 运行的 GUI-only 包 | import graph、`import linuxcnc, hal`、产品脚本 py_compile、broker/SHM/support smoke |
| 私有脚本 pyc-only | 只适用于自研私有脚本的发布副本；`.pyc` 不是安全边界，也不是唯一真源 | Python ABI、源码 SHA、pyc SHA、入口 wrapper、干净 rootfs 启动和异常日志验证 |
| HAL module allowlist | 使用 `ethercat`、`pulse`、`hybrid` profile 参数；hybrid 是两者并集；未签核前不得因“当前未加载”删除 Pulse 资产 | profile keep.list、HAL 引用扫描、加载组件、board/profile smoke |
| Support Mode | 调试能力从操作员环境隔离到 root/维护组和授权诊断包；不能物理销毁排障退路 | 权限、PATH、诊断包内容、只读采集证明、授权失败 fail-closed |

任何后续实现任务都必须按上述门禁落地：先生成候选和证据，再在 staging rootfs 应用，最后按硬件 profile 分别验证。文档里出现的路径、大小和候选项都不是直接删除命令。

## 0.1 状态摘要

当前真实状态：

- rootfs pruning、Support Mode 权限、Source Offer、INI/HAL minify、Python/Tcl/Tk cleanup、pyc-only packaging、HAL allowlist、login banner de-branding 都仍是待实现策略。

## 0.3 当前代码对齐口径

本节用于把本文和当前源码状态对齐，防止后续任务按旧路径实现。

- 状态热路径当前是 `v3_state_publisher.py -> v3_state_shm.py -> /dev/shm/v3_status_shm -> v3_product_main.c`。`v3_state_shm_writer.py --input-json` 是退役产品路径，不得在本文作为实现落点或 fallback 重新引入。
- 当前 SHM typed ABI 在源码中已推进到 `RE_V3_STATE_SHM_VERSION=7`、`RE_V3_STATE_SHM_HEADER_SIZE=1712`，并新增旋转 count-domain 字段。所有文档、测试和 smoke 必须按源码 ABI 同步；如果测试仍断言 v6/1600 或 64-byte header，只能视为待修测试债，不能作为发布验证。
- JSON 只允许作为诊断镜像、归档证据或低频人工排查材料；UI 热路径和控制正确性不得回退到 `v3_status_snapshot.json`、临时响应 JSON、FIFO 或短生命周期 Python fallback。
- `BROKER_ACTION_DOMAIN_BUSY`、action-domain busy 提示和部分 Broker 互斥能力已经存在；但 `rotary_rebase`、断电 checkpoint 恢复、`0xE1F009` 操作员跳窗和单周期 cmd/fb 原子提交证明仍属于未闭环目标，不能写成已验证功能。
- Support diagnostics 已有仓库工具 `tools/v3_support_diagnostics_collect.py`、`tools/v3_support_diagnostics_package.py` 和 release/cutover 规划工具；运行时 Support Mode 的授权入口、Broker action 和板端集成仍需要单独验收。

## 1. 错误显示和诊断分级

目标不是“隐藏 LinuxCNC 错误”，而是把错误分为操作员视图和工程诊断视图。

操作员视图：

- UI 显示商业错误码、中文原因和安全动作建议。
- 示例：`0xE1F001 伺服轴跟踪误差超限，请检查伺服使能、机械阻力或负载。`
- 示例：`0xE1F009 旋转轴绝对位置基准失效，请执行【轴设 0】重新建立可信机械原点。`（当开机 checkpoint 校验失败、大位移超窗口或 unwrap 无效导致 `rotary_unwrap_valid = false` 时必须显示该结果码并跳窗引导设 0）。
- 不在主界面直接暴露长英文堆栈、底层变量或内部路径。

工程诊断视图：

- 保留 LinuxCNC 原始错误文本，例如 `Joint N following error`。
- 记录来源通道、原始文本、映射后的商业码、`status_epoch`、触发命令 `request_id`、程序 hash 和当前 WCS。
- 针对开机投影恢复失败，必须在诊断日志中详细记录具体失效根源：如 `checkpoint_crc_error` (CRC损坏)、`axis_mismatch` (轴错配)、`low32_divergence_exceeded` (位移超限) 或 `encoder_profile_changed` (参数变动)。
- 日志权限限制为 root 或维护组可读，纳入 evidence/handoff。

落地位置：

- 错误采集优先放在现有 broker、状态采样或产品主循环链路中，不新建 `/opt/8ax/v3_product_ui/error_filter.py` 作为第二套真源。
- 若需要映射表，使用仓库内明确文件并配测试，避免只在板端保存。
- broker 内部不能因为错误轮询阻塞命令响应；耗时采集必须独立、可超时、可降级。
- 当前实现状态：通用错误映射和部分 busy/动作域提示已有代码支撑；`0xE1F009`、checkpoint 失败原因和 `rotary_unwrap_valid=false` 的操作员跳窗仍是旋转 count-domain 闭环的一部分，落地前必须补 UI、SHM、Broker/product action 和测试证据。

## 2. 配置文件和参数保护

INI、HAL、NML、settings 是运行真源，不能因为产品化包装而不可重建、不可诊断。

允许：

- 生产镜像中把配置文件权限收紧到 root 或维护组。
- 生成只读配置摘要、SHA256、构建版本和机器型号绑定信息。
- 对外发布时只给操作员 UI，维护账号通过授权方式读取完整诊断包。
- 如果将来做配置加密，必须保留仓库明文真源、构建脚本、解密审计 hash 和稳定运行路径。

禁止：

- LinuxCNC 启动后删除临时 INI/HAL。
- 把加密后的板端文件当作唯一真源。
- 让 UI 或 broker 在运行时改写关键运动参数来绕过 LinuxCNC/HAL 预检。

## 3. 服务命名和启动画面

产品化服务名和启动画面可以改善交付体验，但不能成为诊断伪装。

允许：

- systemd/init 脚本使用产品服务名，同时保留内部日志能追踪到实际可执行文件和版本 hash。
- 开机显示品牌 Logo，内核日志保留到串口、journal 或维护日志。
- 正常启动隐藏对操作员无意义的滚屏文本。

禁止：

- 伪装 Python、LinuxCNC 或 HAL 进程来阻止维护人员识别问题。
- 屏蔽 kernel panic、实时线程错误、EtherCAT master、Pulse/FPGA-DMA 通道错误等安全相关输出。
- 删除版权、许可证或第三方组件声明。

## 4. UDS Broker 隔离

目标链路是面向连接的隔离通道：AF_UNIX + `SOCK_STREAM` + 每客户端独立连接和独立应答。

当前路径：

- Broker socket：`/run/8ax_v3_product_ui/command_broker.sock`
- Broker 源码：`lvgl_app/scripts/v3_command_broker.py`
- 旧 C 端直连：`lvgl_app/src/v3_product_command.c` 已物理删除；不得作为 UDS Broker 控制入口恢复，后续控制入口只能按当前 owner 重新实现

必须满足：

- socket 文件权限默认不超过 `0660`，父目录由启动脚本创建并限制写权限。
- 请求必须携带 `request_id`、`status_epoch`、明确 `action` 和必要参数。
- broker 对过期 epoch、缺参、路径不匹配、hash 不一致、状态不允许必须 fail-closed。
- C 端 connect/send/recv 必须有硬超时；阻塞 connect、`SIGPIPE`、半包和超大包都要有测试覆盖。
- legacy JSON/request 文件只能作为低频诊断镜像或归档证据，不能作为控制正确性真源，也不能作为产品运行 fallback。
- **Rebase 运动域独占互锁**：在执行 A/C 旋转轴运行时 Rebase 或设 0 动作期间，Broker 必须强制获取并锁定 `motion-domain`（运动域）独占锁。在此锁占用期间，Broker 入口必须无条件拦截并拒绝任何其他下发的点动（Jog）、MDI、G-code 启动或轴定位命令，统一返回结构化错误码 `BROKER_ACTION_DOMAIN_BUSY`，保障逻辑重置时的安全。
- **单周期原子同步提交红线**：修改 LinuxCNC 内部轴的指令位置（`cmd`）与反馈位置（`fb`）偏置必须在**同一个实时 HAL 线程周期（单周期）内原子化同步完成**，严禁分帧生效以防产生一帧巨大的跟随误差尖峰导致伺服报警。如果 native gate/实时组件不具备单周期同步提交偏置的能力，则该 `rotary_rebase` 动作在 Broker 侧必须保持未注册或 fail-closed 状态。
- 当前实现状态：`BROKER_ACTION_DOMAIN_BUSY` 已是现有错误码，普通动作域互斥可以复用；`motion-domain` 的旋转 rebase 专用锁、`rotary_rebase` action 注册和单周期 native gate 证明尚未形成可发布闭环。未完成前只能保持 fail-closed，不得用 Python/HAL 分帧写入替代。

慢连接防挂死要求：

- Broker `accept()` 后必须对每个 client socket 设置最大读写超时，推荐上限 1.5s；也可以使用非阻塞 socket 加 selector/event loop，但行为必须等价。
- `_read_exact()` 或等价 frame reader 必须捕获 `socket.timeout`，返回明确 `BROKER_FRAME_TIMEOUT` 或同类错误，并关闭该客户端连接。
- 半包、只发 header 不发 body、body 长度超过 `MAX_FRAME_BYTES`、连接后完全不发数据，都不能阻塞其他客户端。
- 每个连接只能占用一个有界线程、任务或 fd；如果继续使用 per-client thread，必须有超时退出、最大 frame、异常关闭和压力测试。
- 测试必须覆盖一个慢客户端占线时，另一个正常 `health` 或 `jog_continuous_stop` 请求仍能得到响应。

## 5. SHM 状态热路径

目标是把高频状态从 JSON 轮询迁移到二进制共享内存，同时保留 JSON 诊断镜像。JSON 不能作为 UI 热路径 fallback，也不能作为控制正确性真源。

必须满足：

- C 结构体使用显式 packed/layout，并用 `_Static_assert` 校验 size 和关键 offset。
- Python 端使用 `struct` 或 `ctypes` 测试同一 size/offset。
- 写端使用 odd/even `seq`，读端前后读取 `seq` 并校验一致。
- 读写端必须有必要内存屏障或等价 acquire/release 语义。
- CRC、epoch、field gate、degraded/fail-closed reason 必须进入诊断字段。
- 不能把 `volatile` 和 `mmap.flush()` 当成完整同步方案。

现有实现和测试应继续围绕：

- `lvgl_app/include/v3_state_shm.h`
- `lvgl_app/src/v3_state_shm.c`
- `lvgl_app/src/v3_product_main.c`
- `lvgl_app/scripts/v3_state_publisher.py`
- `lvgl_app/scripts/v3_state_shm.py`
- `lvgl_app/tests/test_state_shm_*.py`
- `lvgl_app/tests/v3_state_shm_*_smoke.c`

Cache-line 对齐要求：

- 当前真实 ABI 使用 `ReV3StateShmHeader`，不是简化示例里的 `v3_shm_payload_t`；当前源码 ABI 为 `RE_V3_STATE_SHM_VERSION=7`、`RE_V3_STATE_SHM_HEADER_SIZE=1712`，并由 `_Static_assert(sizeof(ReV3StateShmHeader) == RE_V3_STATE_SHM_HEADER_SIZE)` 固定。
- `seq`、时间戳、epoch、CRC、typed mask/flags 等固定前缀字段必须保持 offset contract；扩展字段必须 bump version/header size，并同步 C/Python layout、mmap reader、writer、frame round-trip 和 smoke 测试，不能只改一端。
- Python 端 `lvgl_app/scripts/v3_state_shm.py` 的 header pack/unpack 格式必须与 C 端 size/offset 测试同步更新。
- 如果后续新增固定二进制 payload 结构，必须按 64 字节 cache line 对齐，并同时更新 C `_Static_assert`、Python `struct.calcsize()`、C smoke 和 Python layout 测试。
- 不得仅把 padding 从 6 改到 14 后宣称完成；任何 ABI 改动都必须证明 C/Python 双端 size、offset、seq/CRC、mmap reader 和 writer 全部兼容。
- 当前旋转 count-domain 相关字段已经进入 v7 header；对应测试如果仍写死 v6/1600，必须先更新测试，不能用旧测试结论证明新 ABI。

三步屏障拷贝规约：

- C 读端从 mmap/SHM 拷贝到本地缓冲时，必须按 `seq_before -> acquire fence -> memcpy -> acquire fence -> seq_after` 的顺序执行。
- `seq_before` 和 `seq_after` 必须来自共享内存头部中的 `seq`，并传入 `re_v3_state_shm_validate_copied_frame_with_seq()` 统一校验。
- 编译器或 CPU 不得把 `seq_after` 读取提前到 `memcpy` 前；C11 `atomic_thread_fence(memory_order_acquire)` 或 GCC/Clang `__sync_synchronize()` 是当前可接受实现。
- 当前 `v3_product_main.c` 已有 `state_shm_read_seq_acquire()` 和 `state_shm_acquire_fence()` 形态，后续重构不得退化为无屏障普通 `memcpy`。

示例顺序：

```c
const uint8_t *mapped = (const uint8_t *)shm_ptr;
uint8_t *local_buf = frame;

uint64_t seq_before = state_shm_read_seq_acquire(mapped);
state_shm_acquire_fence();

memcpy(local_buf, mapped, total_size);

state_shm_acquire_fence();
uint64_t seq_after = state_shm_read_seq_acquire(mapped);

status = re_v3_state_shm_validate_copied_frame_with_seq(
    local_buf,
    total_size,
    seq_before,
    seq_after,
    &header,
    &payload,
    &payload_size);
```

## 6. NML 和网络暴露面

关闭不必要 TCP 监听是安全硬化，不是隐藏真源。

允许：

- 将 NML 通道限制为本地 SHMEM/LOCAL，前提是 `emcCommand`、`emcStatus`、`emcError` 全部验证。
- 用防火墙、服务账号权限和网络分区限制外部访问。
- 在启动预检中记录实际监听端口、NML 配置 hash 和 LinuxCNC 状态。

禁止：

- 只改单个 NML 通道后宣称 TCP 已关闭。
- 为了隐藏端口而破坏 LinuxCNC 工具链、状态读取或维护调试。

NML/RTAPI SysV SHM 残留清理：

- 后端启动脚本 `lvgl_app/scripts/re-v3-8ax-backend.sh` 在拉起 LinuxCNC 前，必须先确认旧 LinuxCNC/HAL/RTAPI 进程已经停止，再清理死残留 IPC SHM。
- 当前脚本已有 `cleanup_ipc_segments_after_stop()`，按 owner/perms 清理 `petalinux` 用户遗留的 600/700 权限小 SHM 段；后续如要改成按 key 清理，必须保留“确认进程已停”这个前置条件。
- 可以增加针对 NML 默认 key `0x00100000` 的清理探测，但必须先确认该 key 属于本机 LinuxCNC/NML 残留，不得删除其他进程仍在使用的共享内存。
- 清理动作必须记录到启动日志：key、shmid、owner、perms、bytes、清理结果和执行时刻。
- `ipcrm` 失败不能静默吞掉；必须进入 backend start log，并在 LinuxCNC 后续 bind 失败时关联到同一故障码。

实现只需保留上述前置条件和日志字段；具体命令由启动脚本 owner 和测试维护，不在本文复制。

## 7. 调试工具和发布镜像

发布镜像可以减少攻击面，但不能“物理销毁”必要维护能力。

允许：

- 移除生产镜像中不需要的头文件、静态库和编译器。
- strip 产品二进制，同时保留符号文件到受控归档。
- 将 `halcmd`、`halrun`、`lsmod`、`modinfo` 限制为 root 或维护组可执行。
- 提供受控 support mode，开启后可采集 HAL、LinuxCNC、当前硬件通道（EtherCAT 或 Pulse/STEP-DIR）、broker、SHM 和 UI 证据。
- 发布硬件可以不引出 UART/JTAG 调试接口，生产网络可以默认关闭开发机 SSH 通道；维护访问必须走受控账号、授权工装或离线诊断包。
- 发布镜像可以清空或产品化 `/etc/issue`、`/etc/motd`、hostname 等登录标识，消除开发系统出厂特征；维护诊断包仍必须记录真实构建版本、内核、rootfs hash 和产品版本。
- 自研 Python 脚本可以在发布镜像中采用 `.pyc` only 形式降低明文暴露和现场篡改风险；仓库真源、构建脚本、测试和符号/源码归档必须完整保留。

禁止：

- 删除所有诊断工具后导致现场无法排障。
- 删除工具作为许可证、第三方组件或架构事实的替代说明。
- 把 stripped 二进制当作可重建真源。
- 把 `.pyc` 当作强加密或不可逆保护；Python bytecode 只能算低强度源码暴露控制，必须配合文件权限、hash 校验、只读 rootfs 和签名发布。
- 删除运行端 `.py` 后没有验证对应入口能从 `.pyc` 启动、异常回溯不泄露开发路径、日志仍可定位版本。

## 8. LinuxCNC 发布镜像清理策略

这部分只针对发布镜像瘦身和攻击面收敛，不是运行板端热删清单。清理必须先做 inventory 和 dry-run，再在 staging rootfs 中验证，最后才允许进入固件打包流程。

### 8.1 裁剪分类

| 分类 | 候选方向 | 保留/验证边界 |
| --- | --- | --- |
| 原生桌面 GUI | Axis、Gmoccapy、Touchy、GladeVCP、QtVCP、配置向导、示例 GUI 资源 | 只在产品 LVGL UI、support mode 和诊断脚本均不依赖后裁剪；路径必须来自 staging inventory |
| Python GUI/Tcl/Tk | `gladevcp`、`qtvcp`、PyQt5、Tkinter、Tcl/Tk GUI-only 运行库 | `linuxcnc*.so`、`hal*.so`、产品脚本依赖和 support diagnose 依赖默认保留；必须跑 `import linuxcnc, hal`、py_compile、broker/SHM/support smoke |
| HAL/RT 模块 | 未加载、未引用、未被 profile keep 的仿真/测试模块 | 实际目录是 `/usr/lib/linuxcnc/modules`；`lcec`、`cia402`、`bitslice`、`motmod`、`xyzac-trt-kins`、`homemod`、`tpmod`、`stepgen`、`encoder` 和 profile manifest 声明资产按 profile 保留 |
| INI/HAL minify | 发布副本去注释、去空白、去开发描述 | 只处理 staging rootfs 副本；仓库真源、源码交付包、Source Offer、许可证和版权清单保留完整可读版本；必须 LinuxCNC parse/HAL load/profile smoke |
| 示例/文档/locale | 未被产品引用的示例配置、示例程序、非许可证文档、低收益 locale | 许可证、版权、Source Offer、发布 manifest 不得裁剪；收益由工具实测，不写死 |
| bytecode-only | 自研私有 Python 脚本发布副本 | 仓库真源、构建源、归档源码、测试和 manifest 必须保留；`.pyc` 不能成为唯一真源，也不能规避 GPL 源码义务 |
| 登录标识 | `/etc/issue`、`/etc/motd`、hostname、shell prompt 产品化 | 不隐藏安全告警、kernel panic、实时线程错误、EtherCAT/Pulse/FPGA-DMA 通道错误、许可证和第三方组件声明 |

### 8.2 不可删除或必须隔离的能力

| 能力 | 规则 |
| --- | --- |
| support 工具 | `halcmd`、`halrun`、`halshow/halmeter`、`ethercat`、`lsmod`、`modinfo` 不能从所有镜像物理删除；可从普通操作员 PATH 移除并迁入 root/support-only wrapper |
| Support Mode | 优先复用 `tools/v3_support_diagnostics_collect.py` 和 `tools/v3_support_diagnostics_package.py`；诊断 action 必须走 Broker，带 `request_id/status_epoch/action/授权`，授权失败或 epoch 过期 fail-closed |
| 诊断包 | 可包含 HAL、dmesg、EtherCAT、LinuxCNC、Broker、State Publisher、SHM reader、UI 版本和配置 hash；必须只读，不得 Home/Jog/Start/MDI、写 HAL pin、重启 backend 或清理运行缓存 |
| 硬件 profile | 清理工具必须支持 `ethercat`、`pulse`、`hybrid`；`pulse/hybrid` 下不得删除 stepgen、encoder、GPIO/FPGA/DMA、脉冲总线、device-tree/bitstream 关联运行和诊断依赖 |
| NML | 禁止把 NML 文件直接覆盖成极简配置并跳过 LinuxCNC command/status/error、Broker、SHM/state publisher 和 UI 验证 |

### 8.3 GPL 和两层交付

白标 UI 可以不显示 LinuxCNC/GPL/第三方列表，但二进制分发必须满足 GPLv2 和第三方许可证义务。两层交付必须同时成立：

| 层级 | 最小内容 | 证据 |
| --- | --- | --- |
| rootfs 只读留存 | `/usr/share/doc/linuxcnc/COPYING`、`/usr/share/doc/linuxcnc/Source_Offer.txt` 或等价路径，权限 `root:root 0444` 或等价只读策略 | 目标路径、owner、权限、SHA256、构建脚本版本 |
| 随附资料或交付包 | 保修卡/合格证、参数备份 U 盘/SD 卡、技术手册 PDF 或等价交付渠道中的开源告知和 Source Offer | 批次号、文件 hash、印刷/电子版本号、出货 checklist |

禁止通过删除许可证、删除文档、改名进程、隐藏组件来源、只把 Source Offer 藏在拆机后才能看到的位置，来规避 GPLv2 和第三方许可证义务。Source Offer 响应包只覆盖 GPLv2 组件对应源码、修改补丁和构建/安装脚本；独立商业 UI 源码是否需要提供由法务结论决定。

### 8.4 实施顺序和工具要求

顺序固定为 `inventory -> candidate -> dry-run -> staging rootfs -> boot validation -> support validation -> license validation -> delivery validation -> release gate`。

工具优先复用或扩展 `tools/v3_hardening_manifest.py`、`tools/v3_gui_prune_manifest.py`、`tools/v3_rootfs_hardening_plan.py`、`tools/v3_live_rootfs_cutover_runner.py`，或接入真实固件打包入口。工具必须满足：

- 默认 `--dry-run`，没有显式 `--apply` 不删除。
- 只接受 staging rootfs 路径，不接受 `/`；`--apply` 前要求 rootfs marker，例如 `.v3_staging_rootfs`。
- 删除前生成 manifest、SHA256、allowlist、denylist、license keep-list 和 support keep-list。
- 每个候选输出风险级别：`safe_candidate`、`needs_runtime_probe`、`keep_required`、`license_required`。
- 记录软链接真实目标，不能只按文件名删除 `linuxcnc`、`emc`、`halshow`、`halmeter`。
- 校验普通 UI 用户 PATH 与维护账号 PATH，确认调试工具对操作员不可见、对维护账号可用。

## 9. 验证清单

文档中任何策略进入实现前，必须有对应测试和证据。

最低要求：

- Markdown 检查和 `git diff --check`。
- 对新增或修改的源码运行本地单元测试。
- 涉及 C/SHM 的改动必须跑 C smoke、Python layout/offset 测试。
- 涉及命令链路的改动必须模拟屏幕按钮或 broker 请求，验证 fail-closed。
- 涉及 UI 显示的改动必须板端截图。
- 涉及 LinuxCNC/HAL/EtherCAT/Pulse/STEP-DIR 的改动必须板端最小验证和 SHA256 记录。
- 纯文档改动不部署板端，但 handoff 要说明“不适用板端部署”的原因。
