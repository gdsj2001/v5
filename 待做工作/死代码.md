# V3 系统双链路深度审计死代码与弃用分支清单

根据项目 `AGENTS.md` 中的 `HIGHEST_DEPRECATED_CODE_RETIREMENT_RULE`（退役代码无幸存原则），按照**控制链**与**状态链**的流向，对整个系统进行了模块与节点的深度代码审查，整理出以下物理存留的死代码、死函数、死参数、影子回退分支、违反工业安全的隐式容错分支以及自动写盘动作。

## 一、 控制链 (Control Chain) 审计

**控制链流向**：`UI C -> Command Broker UDS framed socket -> product action -> LinuxCNC/HAL/EtherCAT`

### 1. 核心调度层 (Command Broker) 的“死参数”与“死属性”残留
随着 Broker 动作调度 100% 切换为 in-process 内存注册载入模式（`inprocess_module`），旧的物理脚本调用机制已失效。但在各处理器中仍残留大量指向已废弃脚本路径的无用变量传参：
* **模块**：[broker_action_handlers_settings.py](file:///d:/v3/lvgl_app/scripts/broker_action_handlers_settings.py)
  * **死参数**：`script=self.settings_apply_script`、`script=self.rtcp_toggle_script`、`script=self.settings_axis_zero_script`、`script=self.wcs_switch_script`
  * **死属性**：`self.settings_apply_script` 等由配置类生成的路径指针，在底层实际执行中已被 `del missing_code` 抛弃，属于无任何业务作用 of 死参数。
* **模块**：[broker_action_handlers_home.py](file:///d:/v3/lvgl_app/scripts/broker_action_handlers_home.py) 与 [broker_action_handlers_program.py](file:///d:/v3/lvgl_app/scripts/broker_action_handlers_program.py) 等亦存在完全相同的 `script=self.xxx_script` 死参数残留。
* **初始化无谓路径计算**：在 [broker_config.py](file:///d:/v3/lvgl_app/scripts/broker_config.py#L65-L101) 中，为了给这套已退役的 `self.*_script` 参数喂数据，初始化阶段无谓地调用了 10 余次 `_release_handler_path` 计算文件物理路径，增加了不必要的初始化耗时。

### 2. 故障复位时的历史诊断落盘分支
* **文件**：[broker_action_handlers_fault_estop.py](file:///d:/v3/lvgl_app/scripts/broker_action_handlers_fault_estop.py#L78-L82)
  * **弃用分支**：在执行 `fault_reset_check` 动作后，Broker 仍在尝试以同步方式往磁盘写入 `fault_reset_result.json`、`pre_status.json`、`post_status.json` 以及 `hal_status.txt`、`drive_status.txt` 五个大文本文件。在强类型 UDS 报文接管后，这套写盘分支无任何后级消费，纯属性能累赘，应彻底关闭。

### 3. C UI 发送侧已抽空的写文件响应空壳分支（`write_response_file` 影子残留）
* **文件**：旧 `lvgl_app/src/v3_product_command.c`
  * **弃用分支**：在旧时代，UI 在向 Broker 发送命令后，需调用 `write_response_file` 将结果写入本地 JSON 诊断文件。该通道停用后，前人将此函数体**强行抽空为无实质操作的空壳**：
    ```c
    static void write_response_file(const ReV3CommandResult *result, const char *script_output)
    {
        (void)result;
        (void)script_output;
    }
    ```
  * **收口状态**：该文件已物理删除；后续不得以“兼容旧命令结果/响应文件”为理由恢复 `write_response_file` 或 `accept_broker_job_for_ui_async` 路线。

### 4. 已退役控制通道常量的测试残留
控制链已全面禁止 FIFO 及临时文件通信，但在测试桩中仍有以下冗余模拟：
* **文件**：[test_v3_microkernel_param_audit.py](file:///d:/v3/lvgl_app/tests/test_v3_microkernel_param_audit.py#L39-L40)
  * **死代码**：模拟已废弃的控制流通道常量 `DAEMON_FIFO = "native_command.fifo"` 与 `RESPONSE_PATH = "native_command_response.json"`。
* **文件**：[product_command_runner_smoke.c](file:///d:/v3/lvgl_app/tests/product_command_runner_smoke.c#L298)
  * **死代码**：C 语言冒烟测试中仍对作废 of `native_command_response.json` 执行匹配与校验。

## 二、 状态链 (State Chain) 审计

**状态链流向**：`v3_state_publisher -> /dev/shm/v3_status_shm typed SHM -> UI/Broker read-only`

### 1. 单/双页 SHM 兼容备用分支（`single-page-shm` 影子回退）
状态链已 100% 切换为 `dual_page_shm`（双页无锁 odd/even 奇偶锁），但为了兼容所谓的“诊断降级”，代码里仍大费周章保留了对单页旧布局的冗余分支逻辑：
* **发布器参数**：在 `state_publisher_cli.py` 中，仍然提供并支持 `--single-page-shm` 备用标志以允许退回发布单页 layout。
* **C 语言解析层分支**：在 [v3_shm_reader.c](file:///d:/v3/lvgl_app/src/v3_shm_reader.c#L119-L142) 中，仍保留了判断单页 `RE_V3_STATE_SHM_MAGIC` 与双页 `RE_V3_STATE_SHM_DUAL_PAGE_MAGIC` 的双分支解析代码。在实际运行中，单页分支为完全停用的死分支。

### 2. 状态采集层 (Publisher) 的“死函数”残留
在 `v3_state_publisher.py` 的发布和重连封装中，存在仅定义、但无任何调用入口的死函数：
* **文件**：[v3_state_publisher.py](file:///d:/v3/lvgl_app/scripts/v3_state_publisher.py#L49-L52)
  * **死函数**：`_refresh_linuxcnc_import_cache()` 模块导入缓存刷新函数，在重构后已被边缘化，没有任何业务逻辑对其发起调用。

### 3. 共享内存 (SHM) 数据结构中的“死字段”残留
`status_epoch` 已确立为状态链的唯一时间戳，原有的 `generated_at_ns` 字段已被拍扁废弃，但在接口层依然残留：
* **文件**：[v3_state_shm.h](file:///d:/v3/lvgl_app/include/v3_state_shm.h#L143)
  * **死字段**：`uint64_t generated_at_ns;`
* **文件**：[v3_state_shm.h](file:///d:/v3/lvgl_app/include/v3_state_shm.h#L257)
  * **静态断言死代码**：校验已废弃字段结构体偏移量的静态 assert。
  ```c
  _Static_assert(offsetof(ReV3StateShmHeader, generated_at_ns) == 36U,
      "state shm generated_at_ns offset changed");
  ```
* **文件**：[state_publisher_publish.py](file:///d:/v3/lvgl_app/scripts/state_publisher_publish.py#L176)
  * **发布器写操作死代码**：在每一帧状态封包时，仍然往内存中强行多余刷写一次 `generated_at_ns`。
* **业务动作层（Product Actions）的兼容旁路死代码**：
  在 `actions_home.py`、`actions_axis_zero_*.py` 等数十个业务脚本中，为了兼容这个已退役字段，依然普遍残留 `status.get("generated_at_ns")` 的冗余取值容错逻辑。

### 4. UI 状态应用层 (UI State Apply) 的历史硬编码残留
* **文件**：[v3_ui_state_apply.c](file:///d:/v3/lvgl_app/src/v3_ui_state_apply.c#L68-L69)

## 三、 违反工业安全规范的“胡乱兜底/静默容错”分支

数控系统核心设计原则是“非对即错、不允许隐式容错”。但代码库中仍残留以下静默吞错误或返回编造假状态的危险分支：

### 1. 网络及系统性能采样的静默 `pass` 分支
* **网卡IP采样**：[state_publisher_snapshot_exports.py](file:///d:/v3/lvgl_app/scripts/state_publisher_snapshot_exports.py#L54-L56)
  当系统发生网络接口物理崩溃或 fcntl 超时错误时，直接静默捕获 `OSError` 错误并悄悄返回 `""` 空字符串。
* **系统CPU采样**：[state_publisher_system_metrics.py](file:///d:/v3/lvgl_app/scripts/state_publisher_system_metrics.py#L21-L27)
  读取 `/proc/stat` 时如遇任何异常，直接静默捕获，并伪造出 `[None, None]` 的空闲假数据让 UI 继续画图，掩盖了底层文件系统故障的严重风险。

### 2. 回零校准与零位投影中的异常静默覆盖
* **回零映射**：[actions_home_memory.py](file:///d:/v3/lvgl_app/scripts/product_actions/actions_home_memory.py#L226-L232)
  在极度依赖数学严密性的回零坐标映射中，当公式计算发生除零等任何 `Exception` 时，程序静默吞掉，并在内存中强行伪造出 `invalid_reason` 的覆盖写入，而不是立即 fail-closed (安全关闭) 并中止整个动作。

### 3. C UI 端的“假坐标”兜底覆盖分支
* **文件**：[v3_ui_state_apply.c](file:///d:/v3/lvgl_app/src/v3_ui_state_apply.c#L10-L16)
  * **问题**：当检测到 SHM 状态过期或获取失败时，UI 并未发起联动急停，而是通过 `re_v3_ui_state_clear_memory_projected_display_fields()`，强制向屏幕全轴物理坐标（mcs/wcs）灌入假数值 **`9999.999`** 进行隐式画面兜底，极易在工业加工调试中引发严重误判。

## 四、 缺乏物理适应性的“死限值校验”（过度严苛阻断）

数控系统对于物理传感器抖动必须具备合理容差，否则极易由于噪声产生控制链死锁：

### 1. 回零指令零点极严校验造成的死锁隐患
* **文件**：[actions_home_zero_plan.py](file:///d:/v3/lvgl_app/scripts/product_actions/actions_home_zero_plan.py#L22)
* **硬编码死门限**：`BUS_HOME_COMMAND_ZERO_EPSILON = 5.0e-6`（0.000005 毫米/度）以及 `BUS_HOME_ZERO_TOLERANCE = 0.01`。
* **审计分析**：在机床回零完成后的指令偏差判断中，校验精度门限被写死为 5 纳米（`5.0e-6`）。由于真实的工业现场中强电磁和电机运行存在物理噪声，编码器受到细微干扰即会产生数微米的微量跳动（Jitter）。此极严的死校验由于缺少物理抖动适应性，会导致机床在实际回零时频繁抛出 `HOME_BUS_COMMAND_POSITION_NOT_ZERO` 报错阻断，需要在后续维护中放宽（如线性轴放宽至 1e-3，物理容差放宽至 0.05）。

## 五、 系统自动运行中的“自动写盘动作”大审计（高频/间歇性磁盘操作）

通过对全量 C 与 Python 产品代码的文件写入操作（`write`、`fopen("a"/"w")`、`fsync`）进行深度追踪，核实系统中存在以下处于激活状态的自动写盘操作，均会在运行期对磁盘产生持续的 I/O 压力：

### 1. 指令网关下的自动追加审计日志 (`NATIVE_GATE_AUDIT_PATH`)
* **文件位置**：[product_actions_native_gate_policy.py](file:///d:/v3/lvgl_app/scripts/product_actions_native_gate_policy.py#L69-L70)
* **自动机制**：每当系统通过 native_gate（如发送 MDI、下发控制指令、使能轴动作等）执行动作时，都会以 `"a"` 追加模式同步将审计记录写入 `/var/log/8ax_v3_native_gate_audit.log` 类似的物理路径。在 G 代码连续运行或高频前台交互中，会产生高频密集的追加写盘行为。

### 2. 物理触摸事件产生的高频调试日志追加 (`TOUCH_LOG`)
* **文件位置**：[v3_product_platform_touch_runtime.c](file:///d:/v3/lvgl_app/src/v3_product_platform_touch_runtime.c#L62) 和 [L86](file:///d:/v3/lvgl_app/src/v3_product_platform_touch_runtime.c#L86)
* **自动机制**：在触摸屏驱动层，只要操作员在物理屏幕上进行任何长按、滑动或点击，C 语言底端均会自动使用 `"a"` 追加模式向 `TOUCH_LOG` 物理磁盘文件高频刷写日志，这在日常操作时构成了极高频的写盘底噪。

### 3. UI 操作事件日志自动追加 (`g_ui.event_log_path`)
* **文件位置**：[v3_ui_events.c](file:///d:/v3/lvgl_app/src/v3_ui_events.c#L12)
* **自动机制**：用户在前台 UI 进行任何按钮点击、输入或页面切换，事件记录系统均会自动以 `"a"` 追加方式，将此次交互记录写入 `g_ui.event_log_path` 指定的物理文件。

### 4. 旋转轴检查点的间歇自动写盘与 fsync (`_write_json_atomic_fsync`)
* **文件位置**：[state_publisher_rotary_checkpoint.py](file:///d:/v3/lvgl_app/scripts/state_publisher_rotary_checkpoint.py#L441)
* **自动机制**：为防止断电丢失旋转轴多圈数据，只要 A/C 轴累计运动量超过设定的步长门限（`step_counts`），就会自动触发原子写盘，生成新的 checkpoint 记录，并强制调用 **`os.fsync`** 对物理磁盘进行一次强同步。若加工程序中旋转轴高频往复运动，会对慢速 Flash 带来严重的局部 I/O 卡顿和擦写负荷。
