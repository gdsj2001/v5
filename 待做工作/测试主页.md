# 主页模拟测试流程

## 失忆/接手必读

若上下文丢失、失忆、换人接手或从压缩摘要恢复，先读 PROJECT_ROOT 下的 `AGENTS.md` 和 `功能/系统代码架构硬边界守则.md`。读完后再继续主页模拟测试。

## 单点迭代与最终上板规则

- 单点诊断阶段可以优先跑 focused gate 和相关模块测试；必要时可用本轮修改文件和直接相关依赖做临时同步/部署来定位问题，但这只能作为诊断或截图前置证据。
- 任何 runtime 源码修复必须先落在 `D:\v5\board` canonical source；Markdown/截图/Windows/Vivado 文件仍落在 `PROJECT_ROOT` 对应 owner。不得只改 VM、板端、staging、deploy_tmp、repo_ignored 或其它临时副本。
- 最终板端闭合必须使用完整当前 PROJECT_ROOT 统一 VM/ARM 编译、打包、部署的新产物，再按原始 UI/operator 路径验证；旧产物、单文件同步或局部部署不能声明黄金链路通过。
- 只有修改跨共享 ABI、CMake/构建系统、公共 UI 框架、SHM/Broker 协议或轻量诊断无法证明时，才提前升级到更大范围编译；但这不降低最终完整构建/部署要求。
- 不要回退、覆盖或清理其他 AI/用户已经落在工作树里的修改；发现冲突时先保留现状，按当前 diff 合并或记录阻塞。

## 目的

AI 统一按 [`功能/自动闭环测试方式.md`](../功能/自动闭环测试方式.md) 第 5 节使用 MCP 板端抓屏与 pointer/mouse，按真实主页 operator 路径逐步闭合主页黄金链路。主页测试不是一口气跑自动化；每一步都必须先在 fresh frame 看见目标、只点一次、再重新抓屏，有问题就停下修当前问题。

## 开始前置

- 本轮现场确认：电机为空载，且当前没有安全问题；该确认放行本轮 model-matched `cc-ac.ngc` / `cc-bc.ngc` 主页运动段单步测试，但不取消逐步截图、单次点击、SHM/Broker 回读、急停可见和异常即停规则。
- 正式开始前，先用继电器重启一次开发板。
- 记录继电器重启证据：断电/上电动作、板端重新连通、MCP 抓屏/鼠标能力可用、Broker 和 SHM 就绪。
- 重启后先用 MCP 板端抓屏观察并保存第一张 fresh frame，确认当前在主页面。
- 第一张屏必须直观看到：急停按钮、启动按钮、回零按钮、打开程序入口、坐标区域、刀路区域。
- 开始前读取 fresh native/SHM，确认 `stale=false`、`estop_active=true`、`machine_enabled=false`、`current_vel=0`、机器空闲；clean boot 不得自动取消急停或上使能。
- 若有弹窗遮挡，必须先按“MCP 抓屏确认当前 fresh frame -> 单次点击 -> 重新抓屏 -> 状态读取”的规则处理该单点。取消急停和上使能只能作为黄金顺序中的第一个真实 UI 动作执行。

## 黄金顺序

每个 active model 先完成一次前置，再在同一 program identity 和有效 `all_homed` 上连续三轮运行：

1. 点击 `取消急停`，以 fresh native actual 确认 `estop_active=false && machine_enabled=true`。
2. 选择 `机械全轴` 并点击 `回零`：除 UI/Broker 成功外，必须观察编码器反馈，确认等效角/关节角确实运动并最终稳定；Home owner 必须 force-off RTCP。
3. 单独完成一次 RTCP ON/OFF 按钮测试并回到 OFF；程序运行期 RTCP 只由 golden 内 `M64 P0/M65 P0` 控制。
4. 点击 `打开程序`，按 fresh native active model 第一次选中、第二次打开唯一匹配程序：`XYZAC_TRT` 打开 `cc-ac.ngc`，`XYZBC_TRT` 打开 `cc-bc.ngc`；禁止使用退役泛名 `cc.ngc` 或跨模型程序。
5. 连续三轮点击 `启动`，采集真实速度、行号和编码器位移；约 1 秒后点击 `急停` 并确认运动停止。轮间只点击一次 `取消急停` 并确认 fresh machine enable actual，不重新 Home、不重新打开程序。
6. 最后一轮急停后确认 `estop_active=true && machine_enabled=false`，同时 program identity、刀路和 homed 仍保留。

## 每步执行规则

- 每个模拟点击前必须用 MCP 板端抓屏取得 fresh frame，并核对页面、按钮位置、遮挡状态和急停可见。
- 每次只允许执行一个点击动作；不能把多个点击合并成脚本连续打。
- 点击后必须立刻用 MCP 板端抓屏取得并按需保存新 frame，并读取 native readback、Broker 结果或事件日志确认动作结果。
- 观察 live frame 时必须检查直观问题：刀路区是否有应显示的图形、程序名/MDI 显示是否正确、坐标区域是否闪空/乱值、弹窗是否遮挡急停、顶部提示是否误导。
- 回零步骤必须记录回零前、回零中、回零后的编码器反馈或 SHM 等效角/关节角摘要；不能只看按钮变色、弹窗或 Broker 成功。若等效角未运动、方向异常、跳变异常或回零后不稳定，立即停止后续步骤。
- 打开程序前，刀路区已有内存数据的 MCS/WCS/A-C/红绿点等应可见；黄色轨迹只要求在打开 G-code 后出现。
- 打开 active-model golden 程序后，主页面必须显示精确程序名 `cc-ac.ngc` 或 `cc-bc.ngc`，并显示黄色刀路或明确 degraded 原因；不能出现后台成功但 UI 无图形、MDI 无程序显示的状态。
- 启动 active-model golden 程序后，必须采集运行中状态，确认速度/行号/运动状态变化；运行中至少用 MCP 板端抓屏保存两张间隔 fresh frame，直观看黄色轨迹或当前运动标记是否随 G-code 运行推进，不能只看 native/Broker 成功。
- G-code 开头和结尾若出现 `A0`、`C0`、`A0 C0`、`AC0` 这类 A/C 归零目标，必须通过微内核/native 运行链和编码器反馈确认旋转轴 unwrap/连续角闭环；V3 不再检查本地运行副本、filter manifest 或等效角 Python 结果。不能只看源 G-code 文本里的 `0` 就判定已经等效角运行；必须证明显示相位回到 0 附近、raw/编码器实际按规划运动并最终稳定。
- 涉及 RTCP 的 G-code 运行测试还要确认运行到程序内 `M64 P0` 后 native RTCP/switchkins actual 为 ON，运行到 `M65 P0` 后回到 OFF；RTCP actual 不以 SHM 字段作为动作证明，不得恢复历史 `M128/M129`。

## 本轮单功能验收清单

- 程序打开：从 fresh native active model 选择唯一 model-matched `cc-ac.ngc` 或 `cc-bc.ngc`，第一击只选中，第二击打开；主页程序名、runtime program identity 和原文件 hash 必须一致，不得使用退役 `cc.ngc`、跨模型程序或后端直开。
- 刀路静态/动态层：Program Open 后必须看到黄色程序线，Start 后至少保存两张间隔 fresh frame，确认行号、速度、坐标和当前运动标记推进；普通动态刷新不得重建静态黄线或反复 fit。
- 坐标与模态：主页同时显示当前模态、机械坐标和 active WCS 加工坐标；active model 的 X/Y/Z/A/C 或 X/Y/Z/B/C 行必须连续、有限且不乱跳，CPU 和速度不得异常留在 `--`。
- Home/RTCP/Golden Motion：每个 active model 只做一次 fresh 机械全轴 Home 和一次 Program Open，先单独验证 UI RTCP ON→OFF，再在同一 program identity 和 `all_homed` 上连续三轮 `Start -> 约1秒 -> E-stop`，轮间只取消急停。另需至少一次让 model-matched 程序经过完整 `M64 P0 -> M65 P0 -> 末端 G53 清理`，证明 native RTCP actual ON→OFF、旋转轴连续相位和最终稳定，不得用 SHM 或源文本代替动作证明。
- 急停与恢复：每次急停都必须确认 native safety actual、速度 0、队列/执行清理、RTCP OFF，轮间取消急停后必须是 fresh `estop_active=false && machine_enabled=true`；最终保持 `estop_active=true && machine_enabled=false`。
- 坐标显示 fail-hold/脏绘：先用 MCP 板端抓屏保存一张 `mcs_valid=true`、`wcs_pos_valid=true` 的正常 fresh frame 和 SHM 显示投影摘要；再在安全的只读/失联场景下保存 stale、epoch 0 或无效坐标 frame，确认上一帧有效坐标不被空白、0 值或占位值覆盖，坐标 canvas 不整块闪白；恢复 fresh SHM 后再保存一张正常 30Hz 更新 frame。
- RTCP Toggle：用 MCP 板端抓屏确认“刀尖”按钮可见且无遮挡；点击 RTCP ON 后读取 fresh native RTCP actual，必须看到 `rtcp_active=true`，按钮视觉稳定为“刀尖开”；再点击 RTCP OFF 后读取同源 native actual，必须看到 `rtcp_active=false`，按钮视觉稳定为“刀尖关”。
- RTCP 失败边界：若 Broker 返回 `RTCP_TOGGLE_NOT_CONFIRMED`、`RTCP_TOGGLE_STATUS_UNAVAILABLE` 或 native actual 未 fresh，必须停在该单点记录按钮前/后 fresh frame、Broker 响应和 native readback，不得继续进入打开程序或启动步骤。
- 证据最小集合：每个单功能至少保留 MCP 板端抓屏取得的点击/状态变化前后 fresh frame、对应 native/显示投影摘要、Broker 响应或事件日志摘要；只有这些单点证据齐全后，才能进入完整主页黄金顺序。

## 停步和修复

- 任何一步 MCP 抓屏/鼠标不可用、画面看不清目标、页面不对、按钮被遮挡或急停不可见，禁止点击。
- 任何一步点击后没有对应 UI 变化、Broker 结果、SHM 变化或事件日志，立即停止后续步骤。
- 中途发现问题，先修本地源码或配置，上板部署后从“回零”重新开始，不从失败中间点续跑。
- 未闭合单步前不能进入全量自动化；全量自动化只能作为所有单点通过后的最终回归证据。
- 不需要为了每个小问题全量编译；改 C UI 时可先构建并部署 UI 做单点诊断，改 Python/Broker/脚本/配置时可先同步对应产品路径并重启相关服务定位问题。但最终运动相关结论仍需要完整当前 PROJECT_ROOT 产物部署后的 model-matched `cc-ac.ngc` / `cc-bc.ngc` 主页路径证据。
