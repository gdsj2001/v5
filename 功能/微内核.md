## P0 速度前瞻

- G-code/MDI 进入运行态前，必须使用 LinuxCNC/微内核 trajectory planner 的原生前瞻能力；目标预读深度为 100 行，等价实现必须证明运行时能提前看到后续小线段并做连续速度规划。
- 密集小线段必须按连续路径运行，不能因为每一小段都被 V3、UI、Broker、product action 或诊断层拆成独立启停动作而频繁加速、减速或停顿。
- 默认运行模态必须包含连续路径策略；如果程序显式给出更严格模态，以 LinuxCNC 解释器当前 runtime modal actual 为准，但 V3 不得在每行之间插入等待、重新门禁、重新读盘或重新规划。
- 速度前瞻属于微内核/LinuxCNC runtime owner。V3 只能提交程序/MDI 文本、显示微内核/native 内存事实的同帧投影、记录诊断和处理安全/fail-closed 结果，不得实现第二套轨迹规划器、前瞻缓存或小线段合并器。
- Start、Pause/Resume、Program Open/Load 和运行期状态发布都不得破坏前瞻队列。Start/Pause/Resume 必须由 UI C 端直接向 LinuxCNC/微内核 native command gate 下发，不能再经过 Broker/Python product action 中间层。非暂停 Start 从 runtime 程序或 MDI 第 1 行提交后，应让微内核连续消化后续行；暂停后再启动只恢复暂停处之后的微内核队列，不得回到 UI 层逐行节拍。

## P0 实现边界

- 100 行预读的实现位置只能在 LinuxCNC/微内核配置、解释器/trajectory planner 能力或已登记 native gate/helper 中；不能落到 UI 状态、JSON 文件、`/run` 结果、Broker 快照或 Python product action 私有队列。
- V3 运行热路径只允许 O(1) 消费已前置到 RAM 的程序身份、运行内存副本、模态和安全 actual。运行中不得为了确认前瞻而重新解析完整 G-code、扫描文件、计算 SHA、读设置表、读驱动 SDO、懒导入脚本或写诊断 JSON。
- 小线段连续进给的验收以板端实际运动证据为准：同一程序在密集短线段段落中应保持连续进给，不出现每段速度掉零、每段重新启动或 UI/Broker 逐行等待造成的节拍。
- 若板端 LinuxCNC/微内核暂时无法提供可证明的 100 行预读能力，必须 fail closed 到未闭环状态并记录 blocker；不得用 UI 层缓存、模拟行号或成功弹窗冒充前瞻已启用。

## P0 常驻内存状态源

- 微内核/native actual 必须优先由微内核/native 常驻内存/API、HAL pin、native component 或登记 native memory block 持有；V3 SHM 不是给 UI/Broker 的通用状态投影，只保留 UI 坐标值、UI 轨迹变化、模态文字、转速、进给速度和主轴/进给倍率显示所需的 30ms UI 显示投影。State Publisher 不得把 typed SHM 或受控小 SHM/status block 当 actual 主真源，也不得让 UI/Broker/product action 通过 SHM 访问来决定可下沉语义能否执行。
- SHM 同步链路必须是常驻内存链路：常驻 State Publisher 或登记 native/helper daemon 从微内核/native 内存采样并单向写入 UI 显示 SHM。不得用 cron/loop 短脚本、启动前一次性注入、`/run` 文件、硬盘 JSON、短生命周期 helper 或诊断输出作为同步通道。
- 硬盘 owner 只允许在开机加载、设置保存事务、程序文件选择/打开并生成运行内存副本、backend restart/reload、backend generation 或 owner 文件签名变化、显式诊断时读取。UI、Broker、product action、运动门禁、State Publisher steady-state 和 Start/Resume 热路径不得用硬盘 JSON/INI/result 文件、程序文件、脚本目录或 UI 资源文件作为 actual、fallback 或同步通道。
- native 状态源缺失、过期、结构错误或回读失败时，UI/provider 必须返回 invalid/unknown/degraded 和同源失败原因；不得在空闲、运行周期、fast overlay 或 cold snapshot 中循环 fork helper、调用 `halcmd`、读取旧 `/run` result 或硬盘状态文件兜底。
- 允许的过渡形态是受控常驻 daemon 订阅或维护 native 状态，再写登记 native memory block 或极小诊断状态块；状态块只是临时 native gap adapter，不是控制入口、不是新的参数 owner，也不是可下沉语义的长期访问路径，且不得并入 `/dev/shm/v3_status_shm` 通用帧。

## P0 RTCP native 状态源

- RTCP/switchkins actual 的 owner 在微内核、LinuxCNC/HAL、switchkins 或登记 native component/helper；UI/Broker/product action 必须直连 owner 已发布的微内核/native 内存/API 状态，RTCP/switchkins actual 不得写入 `/dev/shm/v3_status_shm`。
- 首选形态是 HAL pin/native component 输出：微内核或 switchkins owner 直接提供 `rtcp_active`、`switchkins_type` 等状态 pin。UI/provider 通过已有 LinuxCNC/HAL/native 状态源读入，不得为该状态再启动 `halcmd` 子进程查询。
- 允许的过渡形态是 native/helper 常驻 daemon + 登记 native memory block/极小诊断状态块：受控 native/helper 进程订阅或维护 RTCP 状态，优先写微内核/native 可直接读取的内存块；确有 native gap 时才可临时写极小诊断状态块。State Publisher、UI、Broker 和 product action 只能读登记 native 内存/状态块，不能在 steady-state、fast overlay 或 cold snapshot 中 fork；同一语义一旦可下沉，必须删除小 SHM/status block 依赖。
- RTCP 状态源必须采用登记的 native RTCP 状态块或等价 native provider：常驻 owner 在同一次 `v3_native_status` / `linuxcnc.stat()` 采样中，把 `mcodes` 派生的 `rtcp_active` 写入 `v3_native_rtcp_status.bin`；C UI 通过 `v3_native_rtcp_status.c` 读取 fresh `valid/active` 后才允许 RTCP 按钮按 actual 反向下发 `M128/M129`。该块不是 `/dev/shm/v3_status_shm` 字段，不使用 JSON、`halcmd`、短生命周期 helper 或 SHM modal 作为动作证明；板端必须证明该状态块由真实 LinuxCNC/native 运行态持续刷新。
- 允许的集成形态是登记 native gate readback：RTCP 读取必须从登记 native 内存/API/gate readback 取得，不能写入 `/dev/shm/v3_status_shm`，也不能再用 `subprocess.run(helper status)`、`halcmd getp motion.switchkins-type` 或短生命周期 helper 查询。
- 已删除的 RTCP SHM 状态块路径不得恢复：`re_hal_select_helper status-daemon`、`/dev/shm/v3_rtcp_status.json`、`RE_V3_RTCP_STATUS_PATH`、`state_publisher_rotary_rtcp_actual.py` 以及 State Publisher 对该小状态块的读取都属于退役路径。后续补 RTCP actual 只能补 HAL pin/native API/登记 native memory block 直连 provider，不能把该状态重新写入 SHM。
- native 状态源缺失、过期、结构错误或回读失败时，UI/provider 必须返回 invalid/unknown/degraded 和同源失败原因；不得在空闲或运行周期内循环 fork 兜底，也不得把 request、dirty、设置文件、旧快照或非同源 cache 当作 actual。
- 一次性 `re_hal_select_helper status`、`halcmd getp motion.switchkins-type` 等只能作为人工显式诊断或非产品路径排障；不得作为受控降频 fallback，不得由 `RTCP_STATUS_HELPER_FALLBACK`、`RE_V3_RTCP_STATUS_HELPER_FALLBACK` 或等价环境变量重新打开 State Publisher helper 查询。

## P0 周期性落盘删除

- 设置保存是唯一允许为了持久配置而写盘的运行期主路径；程序文件选择/打开只允许生成运行内存副本，Start/Resume 热路径不得再读程序文件。UI 主线程和 State Publisher 不得用 JSON 写盘表达心跳、ready、性能或状态。
- UI 主线程不得写性能指标文件；State Publisher 不得写 status JSON 充当心跳。ready/heartbeat 必须直接读取微内核/native 内存事实或 UDS health；SHM header、epoch、CRC、stale/degraded 状态只能说明 30ms 坐标投影健康度，不能替代可下沉语义的下层事实。
- 已判定无用的周期性落盘必须从产品路径删除，不允许保留“默认关闭、环境变量再打开、调试开关再启用”的临时停用入口。将来确有需要时，必须重新按本文 owner 建立新的明确需求，再补当前 canonical 实现。
- 正常空闲、稳定显示、无用户动作、无错误时，不得每秒或按固定周期写 `/run` 诊断 JSON、性能 JSON、心跳 JSON 或等价状态文件；State Publisher 错误/退出由 supervisor、stderr/stdout 和坐标 SHM degraded/fail-closed 事实诊断，不能恢复旧 status JSON ready 路径，也不能用硬盘文件表示 steady-state actual。`command_broker.ready`、pid、stdout/stderr、rc 等文件只能作为 supervisor 启动诊断，不得作为运行态 actual、freshness、heartbeat 或运动准入。SHM frame 内不得保留通用 JSON payload。

## P1 验收证据

- 本地证据：配置或源码中存在微内核/trajectory planner 前瞻启用路径，且静态测试证明 Start/Resume 不会逐行等待或插入小线段门禁。
- 常驻内存状态源本地证据：源码中可下沉语义由微内核/native 常驻内存/API 或登记 native memory block 持有，未列入 UI 显示投影例外的状态不进入 SHM；State Publisher 普通帧不以硬盘 JSON/INI/result 文件作为 actual/fallback；generation 或签名变化才触发必要 reload。
- RTCP native 状态源本地证据：源码中 UI/provider、诊断或必要 publisher 路径 steady-state 只读 native 内存/API/登记 native memory block；`v3_native_rtcp_status.bin` 这类登记块必须是二进制、带 timestamp/CRC/freshness，不得并入 `/dev/shm/v3_status_shm` 或 JSON；不存在 `RTCP_STATUS_HELPER_FALLBACK`、`RE_V3_RTCP_STATUS_HELPER_FALLBACK` 或等价环境变量重新打开 helper 查询；契约测试覆盖缺状态源 fail-degraded、helper fallback 不存在、`status-daemon`/RTCP SHM 状态块入口不存在。
- 板端证据：运行包含密集小线段的 `nc/cc.ngc` 或等价 golden 程序，采集速度、行号、状态和 UI 截图；结果应证明不是每个小线段都减速到零再启动。
- RTCP native 状态源板端证据：backend 启动后存在登记 native 状态生产者、native memory block 或 HAL/native 状态 pin；UI/provider 直接读取 native 状态源，RTCP actual 不进入 SHM；空闲和运行期未观察到 State Publisher 周期性 fork helper/halcmd 查询 RTCP actual。
- 诊断输出只能说明微内核前瞻配置、runtime modal actual、速度采样和运动结果；不能把诊断输出本身当成前瞻 owner。
- 周期性落盘删除证据：产品源码和启动脚本中不存在 UI perf JSON 写入、perf 写盘环境变量、State Publisher 正常周期诊断写盘参数或以 status JSON 判定 ready 的路径；ready/heartbeat 检查必须读微内核/native 内存事实或 UDS health，不得进入 SHM；一次性 supervisor ready 文件只作为启动诊断。
