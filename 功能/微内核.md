## P0 速度前瞻

- G-code/MDI 进入运行态前，必须使用 LinuxCNC/微内核 trajectory planner 的原生前瞻能力；目标预读深度为 100 行，等价实现必须证明运行时能提前看到后续小线段并做连续速度规划。
- 密集小线段必须按连续路径运行，不能因为每一小段都被 V3、UI、Broker、product action 或诊断层拆成独立启停动作而频繁加速、减速或停顿。
- BUS当前2ms的`[TRAJ] ARC_BLEND_GAP_CYCLES=4`表达约8ms短段吞并阈值；`linuxcnc/src/emc/tp/blendmath.c`直接按`gap_cycles * cycle_time`比较。因此初始1ms迁移必须改为`ARC_BLEND_GAP_CYCLES=8`，保持约8ms墙钟语义，不能机械保留4而把阈值减半。以后若要改变8ms语义，必须由本owner作为独立trajectory变量裁决，并重新完成密集短段、圆弧和模型匹配golden motion证据，不能夹入servo周期迁移。
- 默认运行模态必须包含连续路径策略；如果程序显式给出更严格模态，以 LinuxCNC 解释器当前 runtime modal actual 为准，但 V3 不得在每行之间插入等待、重新门禁、重新读盘或重新规划。
- 速度前瞻属于微内核/LinuxCNC runtime owner。V3 只能提交程序/MDI 文本、显示微内核/native 内存事实的同帧投影、记录诊断和处理安全/fail-closed 结果，不得实现第二套轨迹规划器、前瞻缓存或小线段合并器。
- 高频轨迹流、实时插补、脉冲输出、伺服闭环、硬实时安全互锁和伺服周期级 override 不得通过 linuxcncrsh TCP 文本命令逐点或高频下发。linuxcncrsh/native command gate 只负责离散控制入口；连续运动、低抖动和实时控制必须由 LinuxCNC realtime、HAL realtime component、EtherCAT、驱动、FPGA 或微内核/native resident owner 在常驻运行域内完成。
- Start、Pause/Resume、Program Open/Load 和运行期状态发布都不得破坏前瞻队列。Start/Pause/Resume 必须由 UI C 端直接向 LinuxCNC/微内核 native command gate 下发，不能再经过 Broker/Python product action 中间层。非暂停 Start 从 runtime 程序或 MDI 第 1 行提交后，应让微内核连续消化后续行；暂停后再启动只恢复暂停处之后的微内核队列，不得回到 UI 层逐行节拍。

## P0 实现边界

- 100 行预读的实现位置只能在 LinuxCNC/微内核配置、解释器/trajectory planner 能力或已登记 native gate/helper 中；不能落到 UI 状态、JSON 文件、`/run` 结果、Broker 快照或 Python product action 私有队列。
- V3 运行热路径只允许 O(1) 消费已前置到 RAM、且仍指向操作员原始程序文件的只读 resident identity、内存索引、显示模型、模态和安全 actual；不得生成或消费执行用运行副本。运行中不得为了确认前瞻而重新解析完整 G-code、扫描文件、计算 SHA、读设置表、读驱动 SDO、懒导入脚本或写诊断 JSON。
- 小线段连续进给的验收以板端实际运动证据为准：同一程序在密集短线段段落中应保持连续进给，不出现每段速度掉零、每段重新启动或 UI/Broker 逐行等待造成的节拍。
- 若板端 LinuxCNC/微内核暂时无法提供可证明的 100 行预读能力，必须 fail closed 到未闭环状态并记录 blocker；不得用 UI 层缓存、模拟行号或成功弹窗冒充前瞻已启用。

## P0 常驻内存状态源

- 微内核/native actual 必须优先由微内核/native 常驻内存/API、HAL pin、native component 或登记 native memory block 持有；V3 SHM 不是给 UI/Broker 的通用状态投影，只保留 UI 坐标值、UI 轨迹变化、转速、进给速度和主轴/进给倍率显示所需的 30Hz UI 显示投影。当前模态文字必须通过 native modal/tool 状态块或等价原生 API 读取，不进入 SHM。State Publisher 不得把 typed SHM 或受控小 SHM/status block 当 actual 主真源，也不得让 UI/Broker/product action 通过 SHM 访问来决定可下沉语义能否执行。
- SHM 同步链路必须是常驻内存链路：常驻 State Publisher 或登记 native/helper daemon 从微内核/native 内存采样并单向写入 UI 显示 SHM。不得用 cron/loop 短脚本、启动前一次性注入、`/run` 文件、硬盘 JSON、短生命周期 helper 或诊断输出作为同步通道。
- 硬盘 owner 只允许在开机加载、设置保存事务、Program Open/Load 读取操作员原始程序并生成只读 resident identity/内存索引/显示模型、backend restart/reload、backend generation 或 owner 文件签名变化、显式诊断时读取。Program Open 不生成执行用运行副本。UI、Broker、product action、运动门禁、State Publisher steady-state 和 Start/Resume 热路径不得用硬盘 JSON/INI/result 文件、程序文件、脚本目录或 UI 资源文件作为 actual、fallback 或同步通道。
- native 状态源缺失、过期、结构错误或回读失败时，UI/provider 必须返回 invalid/unknown/degraded 和同源失败原因；不得在空闲、运行周期、fast overlay 或 cold snapshot 中循环 fork helper、调用 `halcmd`、读取旧 `/run` result 或硬盘状态文件兜底。
- 允许的过渡形态是受控常驻 daemon 订阅或维护 native 状态，再写登记 native memory block 或极小诊断状态块；状态块只是临时 native gap adapter，不是控制入口、不是新的参数 owner，也不是可下沉语义的长期访问路径，且不得并入 `/dev/shm/v3_status_shm` 通用帧。

## P0 G53 几何进入微内核运动学

- 运动模型是微内核运动学和 UI 轨迹显示共同消费的同一个参数。设置页“运动模型”选择什么，就必须提交到微内核/native `motion_model` owner；微内核运动学、UI 黄色轨迹、active model 轴线、当前加工坐标系 RTCP ON 投影都必须消费同一个 fresh active model readback，不得各自维护第二套模型参数。Active Model 支持必须由一个登记 model descriptor 驱动，descriptor 至少给出稳定 model id、旋转轴字母/槽位、对应 G53 字段组、kinematics 模块/branch key 和 UI projector key；唯一 dispatcher 只根据 descriptor 恰好选择一个 kinematics branch 和一个 UI branch。控制、解析、坐标面板、刀路和测试只能查询 descriptor；消费者不得散落 `if AC else BC`、固定 A/C 轴位或文件名推断模型。AC、BC 是首批 descriptor，不是封闭枚举；未知/未登记 model、descriptor/branch 不一致或 branch 缺失必须 fail-closed，而不是回退 AC/BC/上一个模型。
- 设置页 G53 机床坐标目标位置表保存的 A/B/C 几何字段是微内核运动学真源输入。这些字段由 `board/services/command_gate/v5_settings_apply_geometry.c` 写入唯一 runtime INI `[RTCP]` owner，并在开机或受控 clean restart/reload 时由 LinuxCNC HAL 启动的 `/usr/bin/v5_native_hal_owner` 一次装载进常驻 native geometry 内存；运行期不得再由独立 Python 进程轮询 INI/HAL 或周期重装。
- 模型选择参数是实际运动学几何组和实际运动学模块的唯一选择依据。受控 clean restart 必须按 descriptor 只装载一个 active kinematics 模块：AC/`XYZAC_TRT` 只装载 `xyzac-trt-kins` 并只链接 A/C 正逆解，使用 `G53_A_Y/G53_A_Z` 与 `G53_C_X/G53_C_Y`；BC/`XYZBC_TRT` 只装载 `xyzbc-trt-kins` 并只链接 B/C 正逆解，使用 `G53_B_X/G53_B_Z` 与 `G53_C_X/G53_C_Y`。公共 TRT 层只允许坐标到 joint 映射、HAL pin ABI 和无模型数学原语，不能同时保存 AC/BC 公式或父子轴关系。后续增加模型时必须新增独立 kinematics branch/module、descriptor 条目、独立测试和对应 UI branch；不得修改既有 AC/BC 公式来塞入新模型，也不得让一个模块携带另一模型实现。
- LinuxCNC/HAL kinematics、switchkins 和任何 native 运动学组件必须消费 active model 选中的 native geometry readback 作为实际旋转中心；不得继续使用旧 `X_ROT_POINT/Y_ROT_POINT/Z_ROT_POINT/X_OFFSET/Y_OFFSET/Z_OFFSET` 常量、UI 组合显示值、设置页表格文本、默认 0、`settings_runtime.json`、SHM 或旧 cache 作为运动学几何真源。
- LinuxCNC TRT kinematics 需要 `rot-point/offset` 引脚时，共同输入 ABI 固定为 `x_rot_point/y_rot_point/z_rot_point/x_offset/y_offset/z_offset`，但模型公式必须落在各自独立 branch：共同 C 轴基准为 `x_rot_point=G53_C_X`、`y_rot_point=G53_C_Y`、`z_rot_point=0`；AC branch 为 `x_offset=0`、`y_offset=G53_A_Y-G53_C_Y`、`z_offset=G53_A_Z`；BC branch 为 `x_offset=G53_B_X-G53_C_X`、`y_offset=0`、`z_offset=G53_B_Z`。C HAL/native geometry owner 的唯一 dispatcher 在启动事务中选择一个 branch，直接驱动已选 kinematics pins并发布原始 G53 readback；共享 owner 不得用 `if AC else BC` 承载模型公式，也不得保留 `X_ROT_POINT/Y_ROT_POINT/Z_ROT_POINT/X_OFFSET/Y_OFFSET/Z_OFFSET` 第二套可编辑参数。
- `motion.switchkins-type` 只允许在已经由 active model/clean restart 选中的同一个 kinematics 模块内切换 identity 与该模型的 RTCP 正逆解；它不是 AC/BC 模型选择器，运行期不得借 switchkins type 在模型之间跳转。改变 active model、坐标字母、kinematics module/prefix 或 G53 branch 必须走设置保存后的彻底 clean restart，并以 runtime INI、已加载模块、native active-model/G53 readback 三者一致为成功。
- 主页面底部 A/C 或 B/C G53 组合显示值是 UI 显示层口径：AC 模型可显示 A/C 组合中心，BC 模型可显示 B/C 组合中心；这些组合显示分量不得反向喂给微内核运动学。微内核运动学只使用设置页保存并经 native geometry owner 回读、再由 active model 选中的 G53 几何字段。

### Active Model 分支隔离与扩展合同

`active-model` 必须在 clean restart 边界只解析一次，并作为 UI、运动学和运行生效轴集合的共同总开关。唯一 registry/descriptor 只保存稳定身份、active axes/槽位、G53 输入组、kinematics module key 和 UI projector key；唯一 dispatcher 只做 `model id -> descriptor/branch` 选择，不得承载任何模型公式。

每个模型必须形成两个相互对应但物理隔离的编译支路：

```text
active-model registry entry
  -> one LinuxCNC kinematics module/branch
  -> one UI projector branch
```

- `XYZAC_TRT` 的 LinuxCNC module 只能链接 AC forward/inverse 公式，UI branch 只能包含 A 携带 C 的显示公式。
- `XYZBC_TRT` 的 LinuxCNC module 只能链接 BC forward/inverse 公式，UI branch 只能包含 B 携带 C 的显示公式。
- `XYZAC_TRT` 的 active axes 固定为 `X/Y/Z/A/C`，slot 为 `X:0/Y:1/Z:2/A:3/C:4`，B 的运行绑定必须为 `NAT`；`XYZBC_TRT` 的 active axes 固定为 `X/Y/Z/B/C`，slot 为 `X:0/Y:1/Z:2/B:3/C:4`，A 的运行绑定必须为 `NAT`。这组 axis/slot 必须来自同一个 descriptor，不能在设置、Home、drive 或 UI 中各自重建。
- 设置表可以保存未激活轴的 `AXIS_*` 参数，供未来切换模型使用；未激活轴不得写入当前 `JOINT_n`，不得进入 kinematics、Home run mask、drive target set 或 native active mapping。模型切换必须在同一事务把目标逻辑 AXIS 参数迁入对应 active JOINT/status slot，并按 descriptor 自动生成默认 axis→slave：active axis 绑定同号 status slot/slave，离开 active set 的转换轴写 `NAT`。轴参数 `从站` 下拉仍可由操作员覆盖默认绑定；clean restart 后 native/HAL 路由必须读取最终表，校验完整性/唯一性并以同一 mapping generation 一次性 commit，使普通运动、Home 和 drive target 都跟随最终物理从站。物理从站零点/count-domain/drive actual 随从站保持，模型事务和人工覆盖均不得复制零点到逻辑轴。
- 公共 TRT/G53 层只允许 joint 映射、HAL pin/context ABI、descriptor、dispatcher 和无模型数学原语；不能同时保存两套正逆解、旋转父子关系或 `if AC else BC` 模型公式。
- native G53 owner 必须同时校验 active-model 与实际 `KINS_MODULE`，两者不匹配、未知模型、branch 缺失或重复注册时启动失败；不得加载一个模型再把另一模型的 G53 offset 喂进去。
- 新增模型只能新增一个 registry/descriptor 条目、一个独立 kinematics module/branch、一个独立 UI branch 和针对该模型的公式/隔离/golden tests。已有 AC/BC branch 不允许为新模型增加条件、共享可变状态、fallback 或跨 branch 调用。
- `switchkins-type` 只在当前已选 module 内切 identity/RTCP，不参与 active-model 选择。运行期切模型禁止；设置保存后必须彻底 clean restart，并用 runtime INI、已加载 module、native model/G53 actual 和 UI readback 四方一致证明切换完成。

静态门禁必须审计每个 module 的 object/link closure、每个 UI branch 的 ops/scene closure 和 native G53 branch 输入，证明没有跨模型符号引用；板端验收必须按 fresh active-model 选择匹配的 `cc-ac.ngc` 或 `cc-bc.ngc`，分别证明 UI 图形与实际运动学，不得用一个模型通过推断另一个模型也通过。

## P0 RTCP native 状态源与补偿归属

- RTCP/switchkins actual 的 owner 在微内核、LinuxCNC/HAL、switchkins 或登记 native component；UI/Broker/后台 action/native dispatcher 必须直连 owner 已发布的微内核/native 内存/API 状态，RTCP/switchkins actual 不得写入 `/dev/shm/v3_status_shm`。当前 BUS 主线由 LinuxCNC HAL 启动的 C `v5_native_hal_owner` 暴露窄 UDS 控制接口和 native status block，直接连接同一个 HAL actual；Python HAL polling adapter 不是允许的产品 owner。
- RTCP 运动学补偿只属于微内核/LinuxCNC/HAL/switchkins native 运行域。
  - RTCP OFF 时，active model 旋转轴运动不得由系统自动补偿 X/Y/Z；X/Y/Z 只能按程序、MDI、Jog 或 native owner 明确下发的线性轴命令运动。
  - RTCP ON 时，active model 旋转轴运动必须由微内核/LinuxCNC/switchkins 按 native 运动学自动计算并执行所需 X/Y/Z 补偿，使刀尖/RTCP 点满足底层运动学约束。
- UI 里的当前加工坐标系显示投影按 `REQ-RTCP-G53-NATIVE-ACTUAL` 分支消费同一帧 native RTCP actual、active model 姿态和 geometry。
  - RTCP OFF 时，当前加工坐标系不跟随 active model 旋转轴旋转。
  - RTCP ON 时，当前加工坐标系必须跟随 active model 旋转轴旋转。
- UI、State Publisher、Broker、product action 和刀路显示不得计算、注入、抑制或伪造底层 X/Y/Z 补偿，也不得把黄线旋转、WCS 显示、SHM 投影或弹窗成功当作 RTCP 补偿证明；补偿证明只能来自微内核/native actual、commanded/readback 或登记 gate 的同源回读。
- 首选形态是 HAL pin/native component 输出：微内核或 switchkins owner 直接提供 `rtcp_active`、`switchkins_type` 等状态 pin。UI/provider 通过常驻 C native owner 的同源 readback 读入，不得为该状态启动 Python HAL 轮询、`halcmd` 子进程或短生命周期 helper 查询。
- 允许的过渡形态是 native/helper 常驻 daemon + 登记 native memory block/极小诊断状态块：受控 native/helper 进程订阅或维护 RTCP 状态，优先写微内核/native 可直接读取的内存块；确有 native gap 时才可临时写极小诊断状态块。State Publisher、UI、Broker 和 product action 只能读登记 native 内存/状态块，不能在 steady-state、fast overlay 或 cold snapshot 中 fork；同一语义一旦可下沉，必须删除小 SHM/status block 依赖。
- RTCP 状态源和控制源必须采用登记的 native RTCP 状态块、一个持久启动默认、两条 steady-state 直驱微内核运行输入、一个 Home 事务性 force-off 入口和同一个 HAL/switchkins actual：active runtime INI `[RTCP].DEFAULT_ACTIVE` 是开机初值的唯一持久 owner，只允许 `0/1`，由 `v5_native_hal_owner` 在 LinuxCNC HAL 启动时装载一次到 UI latch 初值；当前 seed 为 `0`。启动、Machine On、Program Open 和 Start 不得命令式强制 OFF，也不得把持久默认值直接发布成 actual。第一条 steady-state 输入是 UI 按钮链路：C UI 读取 fresh `valid/active` 后，通过 command gate 的 framed UDS 请求 `v5_native_hal_owner` 设置 UI latch，并只接受同一个 `motion.switchkins-type` actual 的 ack/readback。第二条 steady-state 输入是 G-code 程序链路：model-specific `cc-ac.ngc`/`cc-bc.ngc` 这类直通 LinuxCNC/微内核执行的 G-code 程序必须用标准 `M64 P0` / `M65 P0` 控制已登记的 RTCP 程序输出输入。Home/positioning 是唯一事务性强制例外：机械全轴 Home、MCS 单轴回机械零和 WCS 单轴回加工零在运动前必须向同一个 native RTCP owner 提交 `force_off` 意图，由该 owner 原子清除/压低当前 UI latch 与残留程序输出请求，等待同一个 `motion.switchkins-type` actual 回读 OFF 后才允许运动；不得用 `M65 P0`、MDI、linuxcncrsh 普通文本、UI状态或第三套actual实现。Home结束后保持OFF，不恢复旧ON，后续只由新的UI按钮请求或新程序输出显式开启；该事务不修改 `[RTCP].DEFAULT_ACTIVE`。RTCP 按钮不得把点击转义、转换或降级为G-code程序输出、历史M-code文件、MDI文本、linuxcncrsh普通文本命令、`halcmd`子进程、Python HAL adapter或Python product action；历史M128/M129文件、部署项、测试断言和推导函数必须删除。native RTCP状态块不是`/dev/shm/v3_status_shm`字段，不使用JSON、短生命周期helper、`linuxcnc.stat().mcodes`、按钮请求值或SHM modal作为动作证明；板端必须证明该块由C native HAL owner对真实LinuxCNC/HAL actual持续刷新。
- 允许的集成形态是登记 native gate readback：RTCP 读取必须从登记 native 内存/API/gate readback 取得，不能写入 `/dev/shm/v3_status_shm`，也不能再用 `subprocess.run(helper status)`、`halcmd getp motion.switchkins-type` 或短生命周期 helper 查询。
- 已删除的 RTCP SHM/轮询路径不得恢复：`re_hal_select_helper status-daemon`、`/dev/shm/v3_rtcp_status.json`、`RE_V3_RTCP_STATUS_PATH`、`state_publisher_rotary_rtcp_actual.py`、`v5_rtcp_status_publisher.py` 以及 State Publisher 对 RTCP 状态块的写入都属于退役路径。后续 RTCP actual 只能由 HAL pin/native API/C native owner 的登记 memory block 直连 provider，不能把该状态写入通用 SHM 或恢复 Python writer。
- native 状态源缺失、过期、结构错误或回读失败时，UI/provider 必须返回 invalid/unknown/degraded 和同源失败原因；不得在空闲或运行周期内循环 fork 兜底，也不得把 request、dirty、设置文件、旧快照或非同源 cache 当作 actual。
- 一次性 `re_hal_select_helper status`、`halcmd getp motion.switchkins-type` 等只能作为人工显式诊断或非产品路径排障；不得作为受控降频 fallback，不得由 `RTCP_STATUS_HELPER_FALLBACK`、`RE_V3_RTCP_STATUS_HELPER_FALLBACK` 或等价环境变量重新打开 State Publisher helper 查询。

## P0 周期性落盘删除

- 设置保存是唯一允许为了持久配置而写盘的运行期主路径；Program Open/Load 只允许从操作员原始程序生成只读 resident identity、内存索引和显示模型，不生成执行用运行副本，Start/Resume 热路径不得再读程序文件。UI 主线程和 State Publisher 不得用 JSON 写盘表达心跳、ready、性能或状态。
- UI 主线程不得写性能指标文件；State Publisher 不得写 status JSON 充当心跳。ready/heartbeat 必须直接读取微内核/native 内存事实或 UDS health；SHM header、epoch、CRC、stale/degraded 状态只能说明 30Hz 坐标投影健康度，不能替代可下沉语义的下层事实。
- 已判定无用的周期性落盘必须从产品路径删除，不允许保留“默认关闭、环境变量再打开、调试开关再启用”的临时停用入口。将来确有需要时，必须重新按本文 owner 建立新的明确需求，再补当前 canonical 实现。
- 正常空闲、稳定显示、无用户动作、无错误时，不得每秒或按固定周期写 `/run` 诊断 JSON、性能 JSON、心跳 JSON 或等价状态文件；State Publisher 错误/退出由 supervisor、stderr/stdout 和坐标 SHM degraded/fail-closed 事实诊断，不能恢复旧 status JSON ready 路径，也不能用硬盘文件表示 steady-state actual。`command_broker.ready`、pid、stdout/stderr、rc 等文件只能作为 supervisor 启动诊断，不得作为运行态 actual、freshness、heartbeat 或运动准入。SHM frame 内不得保留通用 JSON payload。

## P0 LinuxCNC 最小板端运行闭包

- Windows `linuxcnc/` 必须继续保存固定版本的完整 LinuxCNC 源码、V5 native 修改、source identity、构建输入以及依法必须随产品保留或交付的许可/归属材料；“完整源码可重建”和“板端最小安装”是两个不同边界，禁止为了缩小 rootfs 删除 Windows canonical 源码或许可材料。
- 正式产品图形界面唯一为 `board/app/` 自建 LVGL UI。这里的“最小微内核”是 V5 产品必需运行闭包的最小化，绝不等于字面裁到只剩实时内核模块；解释器、任务层、motion、RTAPI/HAL、当前产品使用的 kinematics/switchkins 与 HAL components、linuxcncrsh 控制接口、已登记 native gate/readback、State Publisher 实际使用的 LinuxCNC 状态 API/绑定、产品 INI/HAL/NC/runtime 文件，以及这些组件不可替代的动态库和运行依赖都必须保留。某个组件只有在 canonical manifest 中登记实际消费者和 operator path 后才能进入 rootfs。
- Axis、Gmoccapy、QtVCP、Touchy、TkLinuxCNC、PyVCP、GladeVCP 和其它 LinuxCNC 上游 GUI 不得进入产品 rootfs；LinuxCNC 闭包不得以启动脚本、配置检查或备用界面为由保留 Qt、GTK、Tk、桌面环境、窗口管理器、X11/Wayland、图标、主题、翻译和多媒体运行依赖。核心启动或 HAL 配置若仍需要 Tcl，只允许保留不链接 Tk/图形栈的 headless Tcl 闭包；若同名框架被非 LinuxCNC 的已登记产品功能真实依赖，必须由该功能 owner 单独登记，不能借 LinuxCNC 完整包间接带入。
- LinuxCNC 示例配置、演示 G-code、测试套件、开发头文件、编译器/调试器、上游帮助文档、手册生成工具、桌面启动项和未被产品运行闭包消费的脚本/资源不得进入产品 rootfs。人工维护和板端诊断确实需要的命令行工具必须逐项进入白名单并说明消费者，不能用“可能调试要用”保留整套开发环境。
- canonical LinuxCNC recipe、PetaLinux image recipe 和 runtime deploy manifest 必须以显式安装白名单生成 rootfs；构建过程可以从完整 Windows 源码编译依赖目标，但最终 package、sysroot-to-rootfs 安装和制卡阶段只能收录白名单产物。禁止先把完整 LinuxCNC、完整 GUI 或完整开发包安装进 rootfs，再通过不启动、隐藏菜单、删除桌面图标、systemd disable、运行期开关或制卡后手工删除伪装成“最小化”。
- V5 自写 UI、常驻服务、脚本、模块和配置的产品运行闭包统一服从 `REQ-PRODUCT-RUNTIME-CLOSURE`：部署 manifest 决定唯一产品集合，CMake 产品目标从 manifest 派生，未分类生产文件必须在构建前失败，正式制卡后按完整 rootfs 文件清单逐项回读；本文不另建第二套自写目标清单。
- 依赖裁剪必须删除实际 recipe/package dependency，不得只删除文件表象；一个被禁止 GUI 或框架仍被构建并打入 package feed 但未安装到 rootfs，可以作为可重建构建产物存在，不能成为板端运行依赖。产品 rootfs 的 package manifest、文件 manifest 和动态库依赖闭包必须能反向证明每个 LinuxCNC 产物由上述白名单中的真实消费者需要。
- 最小化迭代必须先单独完成 `linuxcnc-prebuilt` configure/compile/install/package、rootfs package 白名单和 PetaLinux overlay/BitBake 解析聚焦验证；未修改 kernel 时不得反复执行 kernel clean，Yocto sstate 可保留为可删除编译加速缓存，但 VM `downloads` 源码缓存不得持久保留。BitBake 若强制要求 `DL_DIR`，只能只读指向 Windows `v5` 内登记的 canonical 源码包目录，或使用本轮 `${VM_BUILD_ROOT}/temp_source/<run_id>` 并在结束后删除。只有聚焦 package/rootfs 验证全部通过后，才允许执行一次最终干净全量构建、制卡和板端闭环，不能把每轮全量构建当作 recipe 试错入口。
- 最小化不能破坏原生 owner：G-code 解释与运行、Home、Jog、Work Zero/WCS、G92、RTCP、运动学、HAL/EtherCAT、参数持久化、native 状态回读、原生错误映射、急停/使能和 clean restart 仍必须走现有唯一链路；不得因裁剪上游 GUI 而在 LVGL、Broker、Python product action 或脚本中重写第二套解释器、任务层、轨迹规划、HAL 控制或状态真源。

## P0 微内核源码可重建

- Windows `linux/kernel/` 是 Xilinx Linux kernel 完整源码 owner，identity 是 `linux/kernel/v5_linux_source_identity.json`；`linux/realtime/` 是 PREEMPT_RT/Yocto realtime metadata owner，identity 是 `linux/realtime/v5_realtime_source_identity.json`；`linuxcnc/` 是 LinuxCNC 与 V5 native 修改 owner，identity 是 `linuxcnc/v5_linuxcnc_source_identity.json`。完整 PetaLinux project-spec、硬件输入和 meta-user recipes 已迁入 `board/petalinux/`，identity 是 `board/petalinux/v5_petalinux_source_identity.json`。`board/linuxcnc/` 只保留 components/HAL/INI/NC/runtime 和 LinuxCNC image recipe，不得保留完整 LinuxCNC 源码、活动 native patch stack、source lock 或源码缓存作为第二实现。
- Linux/PREEMPT_RT 补丁必须按同一固定 kernel commit 的完整依赖链落地；不得在 PetaLinux recipe 中删除建立锁、IRQ、内存分配或调度语义的前置补丁/hunk，再靠 `sed`、构建目录手改或等价后处理拼出部分实现。确需适配 Xilinx 源码上下文时，适配结果必须成为 Windows `linux/realtime/` 的 canonical patch/metadata，登记上游来源、目标 kernel commit 和 SHA-256，并由无 reject 的全新构建及实体板启动证明整条补丁链一致。
- 当 Windows canonical Linux/PREEMPT_RT 源码出现成片缺失、污染或版本不兼容，且聚焦修复已无法以完整补丁依赖、identity 和实体板证据证明可靠时，只允许由 Windows 主机从官方上游重新下载登记的固定版本，直接写入 `D:\v5` 的唯一 canonical owner 并整体替换；VM 不得下载。替换时必须先固定官方 URL/commit/tree/hash，再逐项补回板卡驱动、设备树、kernel config、实时补丁和 V5 自写功能，更新全部 source identity，完成干净全量构建和正常 SD 启动验收，最后物理删除旧源码及其 VM/recipe shadow 路径。不得把旧 SD、参考 SD、VM fetched tree 或板端文件复制回来冒充源码修复。
- 受控构建只读消费 Windows owner 派生的唯一 `${VM_BUILD_ROOT}/temp_source/current` 投影，禁止持久同步、复制或解压第二套 Linux/LinuxCNC 源码到 VM，也禁止直接修改 VM/板端文件后只执行 `make` 或只替换 `motmod.so`、`milltask`、kinematics 模块。LinuxCNC 2.9.7 上游构建系统要求 in-tree 布局时，只允许在 canonical LinuxCNC build 目录建立短生命周期 copy-on-write build view：lowerdir 绑定唯一只读 `current` 投影，upper/work/merged 全部位于固定外部 build 目录，recipe 的 `S` 仍指只读投影、`B` 只指该 merged build view；任务结束必须卸载并清除本轮 upper/work，不得把 merged view 登记为源码投影、持久 owner、下载落点或反向同步源。所有 generated source、配置输出、中间物和产物只能进入该外部 build 目录；固定版本不匹配、源码含未登记 diff、仍存在旧活动 patch 实现或产物身份文件缺失时必须失败。
- SD 卡丢失后的重建入口必须直接使用 `board/petalinux/`，由 `board/tools/petalinux/v5_petalinux_overlay.sh` 生成挂载外 overlay，并由 `board/tools/petalinux/verify_v5_petalinux_source.py` 校验 source identity；LinuxCNC 构建与 identity 校验入口在 `board/tools/linuxcnc/`，image recipe 是 `board/linuxcnc/yocto/linuxcnc-prebuilt.bb`。构建产物必须登记上游提交、补丁 SHA-256、目标架构和各安装产物 SHA-256；板端验收必须回读同一身份，不能以文件时间或“进程能启动”代替。
- 可测试 SD 卡的唯一产品布局是 MBR、2 GiB FAT32 `V5_BOOT` 启动分区和占用其余空间的 ext4 `rootfs` 分区；`BOOT.BIN` 从当前 FSBL、当前 `system.bit` 和当前 U-Boot 生成，`image.ub` 只封装当前 kernel 与 device tree。当前启动命令源码只在 `board/petalinux/project-spec/meta-user/recipes-bsp/u-boot/u-boot-zynq-scr/boot.cmd.default.ext4`，必须显式使用 `root=/dev/mmcblk0p2 rw rootwait`；已删除的 `boot.cmd.default.initrd` 不是 fallback。唯一制卡入口是 `board/tools/petalinux/write_v5_sd_card.sh`：它只能直接读取 VM 上只读挂载的 Windows owner、挂载外 PetaLinux 构建产物和 `board/config/deploy/v5_runtime_deploy_manifest.tsv`，必须交叉编译 ARM 产品二进制、把 manifest 的完整运行闭包安装进新 rootfs、校验目标为非系统可移动盘、写后回读哈希和文件系统，再安全卸载。旧卡内容、板端文件、VM source copy、initrd fallback 和人工补文件都不能作为新卡输入。
- 不再建立 VM 到 NAS 的备份、源码归档或普通目录镜像。离线恢复必需的工具链、board inputs、SD 制卡输入和不可再生介质必须进入 `D:\v5` 的唯一 canonical 路径，并随 `origin/main`/Git LFS 形成非活动恢复历史；构建产物和 sstate 是可删除输出，不是恢复真源。
- VM 构建目录只能保存由只读挂载/唯一 `temp_source/current` 投影产生的 CMake/Ninja/Yocto 状态、中间物、产物和上述 LinuxCNC 短生命周期 copy-on-write build view。build view 不是第二源码投影，不得人工编辑、复用为输入、反向覆盖 Windows 或在构建结束后保持挂载；除此之外不得出现 `source/`、sync staging、task source tree 或可编辑源码副本。发现持久副本必须核对其内容已由 Windows owner 承载后立即删除，不得改名、归档或保留为 fallback。
- VM 清理必须先按内容和 owner 分成三类，禁止按目录名、文件名或“看起来是下载文件”批量删除：

| VM 条目分类 | 删除判定 |
| --- | --- |
| 可重建生成物 | 旧 CMake/Ninja/Yocto/PetaLinux build 输出、日志、临时目录和已过期部署产物，在确认没有活动构建、不是当前验收证据且能从 Windows 真源重新生成后可以直接删除。 |
| 源码或唯一构建输入 | PetaLinux `project-spec`、Linux/PREEMPT_RT/LinuxCNC/Yocto 源码、patch、recipe、device tree、XSA/bit/RTL/IP、集成脚本以及任何 VM-only 唯一副本，必须先分别迁入 Windows `linux/`、`linuxcnc/`、`board/` 或 `vivado_hw_project/` canonical owner，登记上游 URL/commit、补丁和内容 hash，并由干净重建证明覆盖完整后才能删除 VM 副本。 |
| 恢复输入与缓存 | 离线重建必需的工具链安装包、固定源码包、board inputs、SD 制卡输入和不可再生介质必须唯一落入 Windows `v5` 并随 Git LFS 闭合；不得依赖以后重新下载、Windows 外目录、NAS 镜像或 VM `downloads`。sstate、历史 workdir 和构建制品是可删除输出，不得替代 canonical build input。 |

- “Windows 已有同名目录或文件”不等于迁移完成。删除 VM 唯一输入前必须同时证明 owner 落点正确、内容/身份完整、构建链只读消费 Windows 真源、全新 SD 卡可从零重建；任一项未证明时状态必须是 `blocked`，不得先压缩、改名或复制到另一个 VM 目录后删除原路径。
- VM 上 `$MOD`、`$ORIG`、`$TMP`、EtherCAT 实时源树或同类外部源码若尚未固定成 Windows 上游+补丁+recipe 闭包，必须先迁入 Windows owner、登记身份并删除 VM 副本；迁移完成前相关 native 构建和上板关闭状态必须为 `blocked`，不再允许 `vm_only_external_sources` 例外分类。
- 旋转轴等效角只允许由登记的微内核 motion owner 实现。G-code 文件按打开原文投喂，task/public status 不得把 A/B/C 坐标二次包装到 `0..360`，UI 显示解析也不能反向改写 native actual。历史 `taskintf.cc` 的 `z20_wrap_public_*`/`WRAPPED_ROTARY` 包装、旧 planner-idle 归零和同类坐标改写属于退役分支，固定提交重建和板端产物都必须证明不存在。
- 当前 LinuxCNC 登记基线为 2.9.7 上游提交 `472ef181410c0c0baac1d7005e9a6287d586db68`。旧 rotary patch 文件和 `board/linuxcnc/patches/` 已删除；其历史路径与 SHA-256 只记录在 `linuxcnc/v5_linuxcnc_source_identity.json` 的 `migration_provenance`，当前实现只存在于 `linuxcnc/src/emc/motion/` 登记文件中，构建不得再寻找或应用旧 patch。升级版本或修改 native owner 时，必须先更新本文、`linuxcnc/` source identity 和错误映射，再运行完整 native/board 验收。

## P1 验收证据

- 本地证据：配置或源码中存在微内核/trajectory planner 前瞻启用路径，且静态测试证明 Start/Resume 不会逐行等待或插入小线段门禁。
- 常驻内存状态源本地证据：源码中可下沉语义由微内核/native 常驻内存/API 或登记 native memory block 持有，未列入 UI 显示投影例外的状态不进入 SHM；State Publisher 普通帧不以硬盘 JSON/INI/result 文件作为 actual/fallback；generation 或签名变化才触发必要 reload。
- RTCP native 状态源与补偿归属本地证据：源码中 UI/provider、诊断或必要 publisher 路径 steady-state 只读 native 内存/API/登记 native memory block；`v3_native_rtcp_status.bin` 这类登记块必须是二进制、带 timestamp/CRC/freshness，不得并入 `/dev/shm/v3_status_shm` 或 JSON；不存在 `RTCP_STATUS_HELPER_FALLBACK`、`RE_V3_RTCP_STATUS_HELPER_FALLBACK` 或等价环境变量重新打开 helper 查询；契约测试覆盖缺状态源 fail-degraded、helper fallback 不存在、`status-daemon`/RTCP SHM 状态块入口不存在；源码静态检查应证明 UI、State Publisher、Broker、product action 和刀路显示没有自算、注入或屏蔽 RTCP X/Y/Z 补偿的路径。
- 板端证据：运行 native active model 对应且包含密集小线段的 `board/gcode/golden/cc-ac.ngc` 或 `board/gcode/golden/cc-bc.ngc`，采集速度、行号、状态和 UI 截图；结果应证明不是每个小线段都减速到零再启动。
- RTCP native 状态源与补偿归属板端证据：backend 启动后存在登记 native 状态生产者、native memory block 或 HAL/native 状态 pin；UI/provider 直接读取 native 状态源，RTCP actual 不进入 SHM；空闲和运行期未观察到 State Publisher 周期性 fork helper/halcmd 查询 RTCP actual。需要证明补偿行为时，必须在安全工况下按 active model descriptor 选择旋转轴，用 native actual/commanded/readback 证明 RTCP OFF 下当前模型旋转轴运动不会自动补偿 X/Y/Z，RTCP ON 下同一旋转轴运动由微内核/LinuxCNC/switchkins 自动产生对应 X/Y/Z 补偿；至少覆盖 AC 的 A/C 和 BC 的 B/C，后续登记模型复用同一测试模板。UI 黄线、加工坐标系显示或弹窗不能作为补偿行为证明。
- 诊断输出只能说明微内核前瞻配置、runtime modal actual、速度采样和运动结果；不能把诊断输出本身当成前瞻 owner。
- 周期性落盘删除证据：产品源码和启动脚本中不存在 UI perf JSON 写入、perf 写盘环境变量、State Publisher 正常周期诊断写盘参数或以 status JSON 判定 ready 的路径；ready/heartbeat 检查必须读微内核/native 内存事实或 UDS health，不得进入 SHM；一次性 supervisor ready 文件只作为启动诊断。
- LinuxCNC 最小运行闭包本地证据：canonical recipe/manifest 只有显式运行白名单；依赖图中不存在由 LinuxCNC 上游 GUI 引入的 Tk/GTK/Qt/桌面/窗口系统依赖；rootfs package/file manifest 不含 Axis、Gmoccapy、QtVCP、Touchy、TkLinuxCNC、PyVCP、GladeVCP、示例、测试、开发工具或桌面入口。每个保留的 LinuxCNC 二进制、模块、绑定和动态库都必须能映射到 task/interpreter、motion、RTAPI/HAL、kinematics/switchkins、linuxcncrsh、登记 native gate/readback、State Publisher 或产品配置中的一个真实消费者。
- LinuxCNC 最小运行闭包板端证据：从 canonical Windows 源码和白名单 recipe 全新生成 rootfs 并制卡部署；开机不得启动或发现任何 LinuxCNC 上游 GUI，正式画面只由 LVGL 接管；通过原始 LVGL operator path 完成 Machine On、Home、Jog、程序打开、Start/Pause/Resume、Work Zero/WCS、G92、RTCP、急停/恢复、native 状态与错误回读，运行 active model 对应 golden G-code，并核对 rootfs package/file manifest 和 `ldd`/等价依赖结果未把禁止组件重新带回，最后回到 ESTOP。
- 源码可重建证据：先运行 `board/tools/petalinux/verify_v5_petalinux_source.py`、`board/tools/linuxcnc/verify_v5_linuxcnc_source.py` 并核对 `linux/kernel/`、`linux/realtime/` identity，再让 VM 默认直接只读挂载 `linux/`、`linuxcnc/`、`board/` 和 `vivado_hw_project/`，在挂载外建立全新干净构建目录并由 `board/petalinux/` 与 canonical recipe 完整构建；工具强制的单次临时源码展开必须证明身份一致且在任务结束后删除。生成身份必须与四份 Windows source identity 和板端安装产物逐项一致。VM 不得存在持久源码副本，`board/linuxcnc/` 不得存在活动 native patch/source lock。`milltask` 不得包含 `z20_wrap_public_*`/`WRAPPED_ROTARY`，`motmod.so` 必须包含登记的 `v5_wrapped_rotary_*` owner，且重建后必须按原 UI 打开程序、Home、Start 路径完成板端运动验证并最终回到急停状态。
