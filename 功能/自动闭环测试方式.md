# 自动闭环测试方式

引用需求真源：`REQ-BUS-PULSE-MODE-DIFFERENCE`、`REQ-DOC-SINGLE-SOURCE`、`REQ-PARAM-MEMORY-LIGHTWEIGHT-SAVE`、`REQ-SHM-DISPLAY-PROJECTION`、`REQ-NATIVE-OWNER-FIRST`、`REQ-LINUXCNC-COMMAND-GATE`、`REQ-WCS-NATIVE-OWNER`、`REQ-AUTO-STATE-DRIVEN`、`REQ-BOARD-PROGRAM-FULL-BUILD-CLOSURE`、`REQ-MOTION-STILLNESS-TOLERANCE`。

本文是 `REQ-AUTO-STATE-DRIVEN` 和 `REQ-BOARD-PROGRAM-FULL-BUILD-CLOSURE` 的 owner。启动内存、参数 owner、SHM 投影、native owner 和 LinuxCNC command gate 的通用规则分别以对应 `REQ-*` owner 为准；本文只记录自动闭环、operator path、板端程序修改关闭口径和当前代码可执行的工具链。

## AI 阅读入口

- 本文只裁决验证状态、operator path、板端程序修改闭环和当前可执行工具链；参数、SHM、native owner、LinuxCNC command gate 和运动语义仍跳回对应 `REQ-*` owner。
- 判断“能不能说好了”时先看 `可声明状态` 和 `快速判断`；`relay_ready`、`board_deployed`、VM build、dry-run、direct UDS 和旧 result JSON 都不能替代真实闭环。
- 当前测试台架硬件状态只在本文维护；若现场机械结构、刀具、工件或负载改变，先更新本文，再调整运动验证门禁。

## 1. 当前代码事实

- Windows 真源在 `D:\v5\board`；VM `/root/Desktop/v5/board` 是同步产物，不得直接改。
- Windows 到 VM 的当前同步入口是 `D:\v5\board\tools\sync_win_source_to_vm.py --project-root D:\v5`。
- 板端部署和验收工具在 `board/tools/deploy/`，不是项目根目录 `tools/`。
- 当前没有一个能自动完成所有设置页 A-F、真实 UI 点击、真实触摸和 motion golden loop 的单一 runner。闭环必须由当前工具分段组合，并按每段证据判断。
- `tools/v3_hardware_regression_matrix.py` 当前不在仓库内，不能在本文或验收中作为现有工具引用。
- 当前测试台架硬件状态：开发板连接的电机为空载，未连接机床机械结构、丝杠、转台、刀具或工件；因此本阶段自动闭环测试默认允许直接执行真实电机运动，包括 Home、Jog、MDI、Start、`cc.ngc` golden motion 和回零相关运动，不再因为机械结构碰撞、行程干涉或负载风险把真实运动验证降级为 direct UDS、dry-run、只截图或只构建。当前台架状态下，机械碰撞/负载安全风险不是默认 blocker；凡验收运动链路，必须跑真实运动证据。运动命令仍必须走对应 native/LinuxCNC/HAL/EtherCAT owner，急停、驱动故障、通信异常、owner fail-closed 和最终停机状态仍然是真实证据，不得伪造或用旧 result 替代。


默认 `V5_BOARD_BUILD_TARGETS` 是当前部署需要的目标集合；如果修改进入了其它板端产物，必须显式扩展目标或跑能覆盖该产物的构建门禁，不能用默认目标掩盖未构建产物。

## 2. 可声明状态

| 范围 | 最低验证 | 可声明状态 |
| --- | --- | --- |
| 只改文档/规则 | `git diff --check`，必要时做单源 owner 审计 | `doc_updated` |
| 非板端本地代码 | focused compile、`py_compile`、focused pytest 或静态扫描 | `local_verified_only` |
| 进入板端产物或影响板端运行 | Windows 真源同步 VM、当前 board source 构建、部署、运行闭包重启、原始 control/operator 路径证据 | 只能按证据说明；未走 operator path 不得写 `board_verified` |
| UI/operator 缺陷 | 当前页面截图或 remote frame、逐步输入、UI/action/touch 事件、结果和 readback | `operator_path_verified`，但不等于 motion 通过 |
| Start/Jog/Home/Stop/E-stop/MDI/G-code/运动门禁 | operator path 加 motion owner 要求；默认还需要 `cc.ngc` golden motion 和最终安全状态 | `board_verified` |
| 硬件不安全、板端不可达、证据拿不到 | 记录阻塞、已尝试动作、缺失证据和下一验收条件 | `blocked`、`source_only` 或 `local_verified_only` |

`relay_ready`、`board_deployed`、`board_runtime_ready`、VM build、dry-run、matrix pass、direct UDS 和旧 result JSON 都不是功能通过。旧 result 文件早于当前相关源码、配置或文档修改时，只能作为历史诊断，不能证明当前工作树。

## 3. 板端程序修改闭环

任何会进入板端产物或影响板端运行的程序、配置、脚本、服务、部署项修改，都属于板端程序修改，包括 `D:\v5\board` 下的 UI C/C++、native/helper、Broker/action、State Publisher、LinuxCNC/HAL/EtherCAT 配置、参数 provider、启动/部署脚本、init 脚本、部署 manifest 和板端资源文件。

板端程序修改关闭条件：

- 先跑最小本地门禁，暴露语法、合同和局部行为错误。
- 用 `board/tools/sync_win_source_to_vm.py` 从 Windows 真源同步到 VM。
- 在 VM 当前 board source 上构建本次会进入部署产物的目标集合。
- 用 `run_v5_board_acceptance.sh --apply` 或等价 deploy 脚本部署 canonical artifact。
- 检查实际运行服务、socket、SHM/native block、runtime INI、drive provider、UI relay 和需要的 result/readback。
- 回到用户报告的原始 UI/operator/control 路径验证；UI 问题不能只用 direct UDS 关闭。
- 涉及运动或安全的，再执行对应 motion/safety owner 要求。

若部署脚本返回非零，不能直接宣称通过。必须先按 `failure_triage` 分类：如果独立 init status、进程、socket、SHM/readback 证明服务实际已恢复，可把它记录为 `automation_tool` 或 deploy check 缺口；如果不能证明，状态必须保持 `blocked` 或未完成。

## 4. 参数文件门禁

修改 `self_parameter_table.tsv`、`drive_parameter_table.tsv`、drive profile map、LinuxCNC INI、`tool.tbl`、`linuxcnc.var` 等板端持久 owner 时，必须先单独上传到板端 owner 位置并做 hash/readback，再进入完整构建、部署、重启和功能复测。

原因是当前 `push_v5_runtime_to_board.sh` 会处理 board-owned 文件：会 merge self parameter table、保留或刷新 drive parameter table，并可能从板端 owner 回拉部分文件。参数变更若未先单独写入真实 owner，完整部署阶段可能把本地期望覆盖回板端旧 owner。

参数文件单独上传闭合至少包括：

- 本地 owner 文件 hash。
- 板端目标路径 hash。
- 字段级 readback，例如 INI section/key、TSV 行或 JSON profile sha256。
- 后续完整部署后再次确认该字段仍在板端 owner 中。

## 5. Operator Path 和输入工具

当前代码有三类输入/证据工具，不能混用证据含义。

`capture_v5_board_ui.sh --apply` 只抓 framebuffer，不发送任何输入。它可作为点击前页面确认或最终画面证据。

`collect_v5_remote_input_evidence.py` 通过 relay WebSocket 发送 layout-only pointer 事件，并保存 before/after frame、`ui_events.jsonl` delta 和 `remote_input_events.jsonl` delta。它适合非运动、非真实触摸校准的 UI 布局点击诊断。使用规则：

- 每次 click/drag 前必须先确认 before frame 的页面、按钮状态和目标位置。
- 每次只做一个小动作，动作后读取 after frame 和事件 delta。
- 如果点击命中与预期不一致、页面不对、坐标漂移、事件日志显示触发了其它按钮，立即停止并把失败先归为 `automation_tool` 或 `unclassified`，不得继续批量点击。
- 它不能证明真实手指触摸、触摸校准、Start/MDI/运动安全或真实 operator motion。

`collect_v5_board_touch_evidence.sh` 和 `collect_v5_board_ui_action_evidence.sh` 等待真实手指输入，不发送 synthetic input。它们用于真实触摸和本地按钮证据，要求先有截图证据，并保存 `touch_events.jsonl`、`ui_events.jsonl` delta。

最小 operator evidence 通常包括：点击前 frame、目标位置、输入方式、点击后 frame、`ui_events.jsonl`、必要时 `touch_events.jsonl` 或 `remote_input_events.jsonl`、actiond/Broker result、owner readback。

## 7. `cc.ngc` Golden Motion

后续自动闭环测试的默认执行入口是 `../待做工作/跑cc.md`。完整闭环必须按该执行卡跑：继电器断电重启开发板，等待 LinuxCNC/EtherCAT/command gate/UI relay/State Publisher/native 状态块恢复后，系统必须已经自动 `Machine On`；自动上使能只能发生在微内核/native 运行闭包完全起来之后，这至少包含 LinuxCNC/EtherCAT OP、command gate 常驻连接、RTCP/WCS/G53/native safety 状态块、State Publisher 和 UI relay 可读，不允许只在 SSH 或 LinuxCNC backend 刚起来时提前 `Machine On`。若此时 `linuxcncrsh Get Machine`、native safety readback 或同源 machine enable actual 仍为 `MACHINE OFF`/`enabled=false`，测试失败并修复开机自动上使能链路，不得通过手工点 `取消急停`、direct UDS、linuxcncrsh 或测试脚本补发 `Set Machine On` 掩盖。随后每次模拟键盘/远程输入前先截图确认位置，打开程序并双击打开 `cc`，执行回零、启动、1 秒后急停、取消急停、再次回零，并在全过程采样编码器反馈。编码器反馈用于证明 `cc.ngc` 中 `A0 C0` 或一圈内 A/C 绝对相位目标实际走最近 `k*360` 等效角，两次回零实际走 360 度最短角差等效回零；不得只用坐标显示为 0、按钮提交成功、`Set Run` 成功或 direct UDS 结果替代。

当前 acceptance 脚本的 golden 源文件默认是：

```text
board/gcode/golden/cc.ngc
```

`cc.ngc` 必须是可直接交给 LinuxCNC/微内核解释的原始 G-code 文件；验证时打开的内容、复制到板端的内容和 Start/`Set Open` 投喂的内容必须一致，不得用 UI 预览解析结果、旋转等效角改写结果、临时 sidecar、运行副本、短程序或“等效处理后”的 G-code 代替。等效角、连续多圈、A/C normalization 和 fail-closed 判定属于微内核/native owner，UI 显示/移动链路的解析结果不参与 golden 执行语义。

`cc.ngc` 是直通 LinuxCNC/微内核执行的 golden 程序，程序运行所需的 RTCP 开/关必须由 `cc.ngc` 自己控制微内核运行态：非切削预定位、A/C 归零/归位和机床坐标清理段默认保持 RTCP OFF；到达弹簧起点后、第一条弹簧 `G1` 进给前，程序文本内必须用 LinuxCNC 原生 `M64 P0` 打开已登记的 RTCP/switchkins 程序输出，并等待 native actual ON；最后一条弹簧 `G1` 到达弹簧终点后、进入 `Machine-coordinate return` 清理段前，必须用 `M65 P0` 释放该程序输出。G-code 程序链路与 UI 按钮链路相互独立：程序内 `M64/M65` 直驱微内核程序输出，UI 按钮只通过 `native_rtcp_control` latch 直驱微内核；两条链路不互相代发、不互相预置，最后只在同一个 HAL/native switchkins actual 上合流。`M65 P0` 之后只允许执行末尾 G53/G49 机床坐标清理动作，例如 Z 回零、A/C 回零和 X/Y 回零；不得再进入加工段或继续以 RTCP ON 语义解释 A/C 轨迹。golden runner 只能负责 Machine On、真实 Home 前置和提交 `Set Open`/`Set Run`，不得在 Start 前通过 `native_rtcp_control`、MDI、linuxcncrsh 普通文本、临时 sidecar、UI 预处理或 Python product action 替 `cc.ngc` 外部预置 RTCP。历史 M128/M129 不得恢复。

若自动闭环在 `M64 P0` 之后、`M65 P0` 之前按下急停，后续 `回零` 的 Home owner 必须先中止 Auto，并允许在同一 FIFO 控制事务内执行一次 `M65 P0` 来清除被中断 G-code 程序链路遗留的 `motion.digital-out-00`/`v5-rtcp-gcode-request` 锁存；随后仍必须通过 `native_rtcp_control` 请求 OFF 并以 fresh native RTCP actual OFF 作为证明，才能进入真实 Home。这个清理动作只属于中断后机器回零域准备，不算 Start 前外部预置 RTCP，也不能替代 `cc.ngc` 自己在正常程序路径里的 `M64 P0`/`M65 P0` 控制。

主页面 UI 的 RTCP 按钮不是 G-code 程序路径，仍必须直通 `native_rtcp_control` 微内核 latch，并跟随同一个 native RTCP actual 做 ON 高亮和刀长下方 `RTCP ON/OFF/--` 显示。UI 按钮不得把点击转义成 G-code 程序输出、`M64/M65`、M128/M129、MDI 文本或 linuxcncrsh 普通命令；G-code 程序内的 `M64/M65` 只属于程序链路直驱微内核。

`run_v5_board_acceptance.sh --apply --motion` 默认把它复制到板端：

```text
/tmp/v5_golden/cc.ngc
```

并调用：

```sh
V5_ALLOW_MOTION=1 /usr/libexec/8ax/v5_linuxcncrsh_golden_run --program /tmp/v5_golden/cc.ngc --start
```

`v5_linuxcncrsh_golden_run --start` 必须在 `V5_ALLOW_MOTION=1` 下先确认 Machine On，清理当前 Auto/程序运行状态并回到 Manual，再通过已登记 BUS Home owner 执行真实 Home 前置和 native all_homed/readback 确认，然后才允许提交 `Set Open`、`Set Mode Auto`、`Set Run 0`、`Set Resume`。这个脚本能证明 golden 文件已复制、linuxcncrsh probe 可用、Machine On/Home 前置已按当前工具链闭合并提交了 start；它只能作为 `../待做工作/跑cc.md` 完整自动闭环的前置/辅助证据，本身不是完整自动闭环通过证据。运动相关 `board_verified` 仍必须补足：

BUS Home 前置必须在未 homed 的 joint jog 域和已 homed/coordinated 的轴字母 jog 域之间按 native all_homed 状态选择；两种状态都必须产生真实运动和到位/readback 证明，不得因为已经 homed、坐标显示为 0 或刚跑过程序就跳过 Home、只 `Set Home` 或只清坐标。

A/C 旋转轴的 Home 到位判断必须使用 360 度等效最短角差；`359.999`、`-0.001` 这类等效零只能在编码器/微内核实际到位且角差在容差内时判定为 0，不能按线性差值误判失败，也不能只改 UI 显示。

- 原始 UI/operator path 或 owner 指定的真实 control path。
- 程序路径或 hash 匹配 `cc.ngc`。
- Start 后可证明的速度、joint 或 encoder 位移。
- Stop/E-stop/reset 或最终 safe stop 后的 microkernel/native 安全状态。
- Home 或设0 前置条件必须走对应 owner：要求设置页轴设0时，不能用主页 Work Zero 冒充。

在当前空载电机测试台架上，凡是用户要求验证 Home/Jog/Start/MDI/G-code/回零等运动路径，默认应跑真实运动证据；`--motion` 路径应显式设置 `V5_ALLOW_MOTION=1` 并执行，不能再把“未确认机械安全”作为本项目当前台架的默认 blocker。若后续接上机械结构、刀具、工件或负载，必须先更新本文硬件状态，再恢复相应的现场安全门禁。

不得用短程序、空程序、示例折线或为了测试改写后的用户 G-code 代替 golden。

## 8. 失败处理

失败是下一步诊断输入，不是交付终点。失败初始状态必须是 `unclassified`，候选域至少包括：

- `project_source`：源码、配置、owner 文档或部署产物问题。
- `automation_tool`：截图、relay、remote input、坐标、证据采集脚本或部署检查误报。
- `board_runtime`：服务生命周期、socket、SHM、LinuxCNC/HAL/EtherCAT、drive provider 状态。
- `hardware_safety`：运动条件、急停、machine on/off、驱动使能、真实触摸或现场安全。

每次失败后先重跑能覆盖该失败的最小门禁，再升级到 required mode。板端忙、硬件不安全、网络/串口/relay 不可达或证据无法取得时，按 AGENTS.md 报告 blocker；不得要求用户人工代替闭环，也不得把新增弹窗、禁用按钮或错误提示当作修复完成。

## 9. 快速判断

| 问题 | 答案 |
