## 1. 当前代码事实

- V5 可维护源码只存在于 Windows `D:\v5` 这一棵 Git 工作树：完整 Linux kernel 在 `linux/kernel/`，PREEMPT_RT/Yocto metadata 在 `linux/realtime/`，完整 LinuxCNC/native 在 `linuxcnc/`，完整 PetaLinux project-spec 在 `board/petalinux/`，板端集成/LVGL/services/tools 在 `board/` 其余 owner，硬件工程在 `vivado_hw_project/`。
- VM 必须把 Windows 真源实时只读挂载到 `VM_SOURCE_MOUNT_ROOT=/mnt/v5-source`；generated files、中间物和产物只写入 `VM_BUILD_ROOT=/root/v5-build`。为避开共享目录小文件性能问题，只允许 `${VM_BUILD_ROOT}/temp_source/current` 一套由该挂载 identity 派生、不可编辑的加速投影：identity 一致时复用，不一致时只刷新这一处；`/root/Desktop/v5`、第二投影、源码同步 manifest、VM checkout 和其它持久 VM 源码副本均为退役路径。
- 板端部署和验收工具在 `board/tools/deploy/`，不是项目根目录 `tools/`。
- 当前没有一个能自动完成所有设置页 A-G、真实 UI 点击、真实触摸和 motion golden loop 的单一 runner。闭环必须由当前工具分段组合，并按每段证据判断。
- 历史 `v3_hardware_regression_matrix.py` runner 当前不在仓库内，不能在本文或验收中作为现有工具引用。
- 当前测试台架硬件状态：开发板连接的电机为空载，未连接机床机械结构、丝杠、转台、刀具或工件；因此本阶段自动闭环测试默认允许直接执行真实电机运动，包括 Home、Jog、MDI、Start、`cc-ac.ngc`/`cc-bc.ngc` golden motion 和回零相关运动，不再因为机械结构碰撞、行程干涉或负载风险把真实运动验证降级为 direct UDS、dry-run、只截图或只构建。当前台架状态下，机械碰撞/负载安全风险不是默认 blocker；凡验收运动链路，必须跑真实运动证据。运动命令仍必须走对应 native/LinuxCNC/HAL/EtherCAT owner，急停、驱动故障、通信异常、owner fail-closed 和最终停机状态仍然是真实证据，不得伪造或用旧 result 替代。

当前板端 acceptance 入口：

```sh
cd /mnt/v5-source/board
V5_REPO_ROOT=/mnt/v5-source/board \
V5_BOARD_BUILD_DIR=/root/v5-build/board \
V5_BOARD_SSH=root@192.168.1.221 \
V5_BOARD_SSH_PORT=22 \
tools/deploy/run_v5_board_acceptance.sh --apply
```

`run_v5_board_acceptance.sh --apply` 当前执行顺序是：

1. `cmake --build "$V5_BOARD_BUILD_DIR" --target $V5_BOARD_BUILD_TARGETS`
2. `tools/deploy/precheck_v5_board.sh`
3. `tools/deploy/push_v5_runtime_to_board.sh --apply`
4. `tools/deploy/verify_v5_board_runtime.sh`
5. `tools/deploy/capture_v5_board_ui.sh --apply`

默认 `V5_BOARD_BUILD_TARGETS` 是当前部署需要的目标集合；如果修改进入了其它板端产物，必须显式扩展目标或跑能覆盖该产物的构建门禁，不能用默认目标掩盖未构建产物。

部署入口对 canonical CMake 目录做 ARM 身份校验时，必须同时接受 `CMakeCache.txt` 中的显式 `CMAKE_C_COMPILER` 与 CMake 生成的 `CMakeFiles/<version>/CMakeCCompiler.cmake` 真值；两处都缺失或解析出的编译器不是 `arm-xilinx-linux-gnueabi-gcc` 才可 fail-closed。不得把 CMake 正常省略 cache 项误判为非 ARM 构建，也不得因此绕过 canonical 部署入口。

### v5 MCP 工具边界

Codex 当前用于本项目的 v5 远端工具只保留两个执行入口：VM 只使用 `vm_exec`，开发板只使用 `board_exec`。`vm_exec` 负责验证 `/mnt/v5-source` 是 Windows 真源的实时只读共享挂载、核对/刷新唯一 `/root/v5-build/temp_source/current` 投影，并在 `/root/v5-build` 执行受影响 target/package、必要 rootfs/manifest、一次 current 镜像和部署命令；`board_exec` 负责在 `z20_plnx` 执行板端只读状态探测、产物核对和 operator-path 所需命令。部署使用 canonical `board/tools/deploy/run_v5_board_acceptance.sh --apply` 或 owner 登记的 focused deploy entry。不新增 MCP 专用旁路，不允许通过 MCP 创建第二源码投影或直接编辑板端产品文件。

除 `vm_exec` 和 `board_exec` 外，v5 远端 MCP 不再暴露其它命令执行、构建、扫描、源码复制或诊断入口；测试、构建、部署和验收文档不得恢复、别名化或新增 v5 远端 MCP 旁路。Codex 自身的 `node_repl` 和其它通用插件 MCP 不属于 v5 远端工具边界。

## 3. 板端程序修改的 `board_verified` 闭环

任何会进入板端产物或影响板端运行的程序、配置、脚本、服务、recipe、部署项修改，都属于板端程序修改，包括 `linux/kernel/` 的 Linux kernel、`linux/realtime/` 的 PREEMPT_RT/Yocto metadata、`linuxcnc/` 的 LinuxCNC/native、`board/petalinux/` 的 PetaLinux project-spec，以及 `board/app/`、`board/linuxcnc/`、`board/services/microkernel/`、`board/services/command_gate/`、`board/services/state_publisher/`、其它 `board/services/` 和 `board/tools/` 的 UI、HAL/INI/runtime packaging、native integration、action、参数 provider、启动/部署脚本、init 脚本、部署 manifest 和板端资源文件。微内核合同与常驻 owner 已有独立目录，不得继续笼统写成 Command Gate 内部文件。

任何板端程序修改默认先完成最小受影响闭环：Windows focused gate、当前 target/package、必要 artifact、部署、runtime readiness、受影响原始 operator/control 路径最小测试和 fresh owner readback。只有完整设置页 A-G、Golden Motion、CPU1 压力、发布级生命线等超出当前受影响闭环的专项验收，才需要用户目标或 owner 明确命中；不得为了小修追加无关 rootfs/image 或全量回归。用户明确保留最终人工测试时，AI 仍按当前受影响闭包构建并部署，但在 hash/服务启动健康检查后停止，不进入 operator/golden/motion；交付状态保持 `local_verified_only` 并注明“已部署待人工验收”。板端不可达、安全条件不满足或外部资源真实阻塞时，必须报告当前实际状态、已完成证据和下一接受条件，不得把未上板的 `local_verified_only` 当作任务关闭。

需要 `board_verified` 时，板端程序修改关闭条件：

- 先跑最小本地门禁，暴露语法、合同和局部行为错误。
- 用 `findmnt` 或等价证据确认 `VM_SOURCE_MOUNT_ROOT` 是当前 Windows 真源的实时只读共享挂载；核对 `${VM_BUILD_ROOT}/temp_source/current` 的 machine-readable identity 与挂载一致，并确认 `/root/Desktop/v5`、第二投影、VM checkout 等其它源码副本不存在。current 投影允许跨任务复用，但不得人工编辑、提交或反向覆盖 Windows。
- 在 `VM_LINUX_BUILD_DIR` / `VM_BOARD_BUILD_DIR` 构建本次会进入部署产物的目标集合；源码挂载内不得产生 generated files、中间物或产物。
- 用 `run_v5_board_acceptance.sh --apply` 或等价 deploy 脚本部署 canonical artifact。
- 任何manifest destination改名或源码入口退役都必须同时修改canonical installer的retired-file清理清单；manifest缺少旧行不会自动删除板端旧文件。focused installer测试必须先构造旧目的文件、执行安装、再证明旧文件不存在且新文件hash正确，部署readiness前还要从板端fresh readback absence；旧文件仍存在即为survivor，不能声明部署闭合。
- 检查实际运行服务、socket、SHM/native block、runtime INI、drive provider、UI relay 和需要的 result/readback。
- 回到用户报告的原始 UI/operator/control 路径验证；UI 问题不能只用 direct UDS 关闭。
- 涉及运动或安全的，再执行对应 motion/safety owner 要求。

若部署脚本返回非零，不能直接宣称通过。必须先按 `failure_triage` 分类：如果独立 init status、进程、socket、SHM/readback 证明服务实际已恢复，可把它记录为 `automation_tool` 或 deploy check 缺口；如果不能证明，状态必须保持 `blocked` 或未完成。

## 4. 参数文件门禁

修改 `self_parameter_table.tsv`、`drive_parameter_table.tsv`、drive profile map、LinuxCNC INI、`tool.tbl`、`linuxcnc.var` 等板端持久 owner 时，必须先从 Windows 真源生成独立参数部署产物，通过 canonical deploy/apply 事务写入板端 owner 并做 hash/readback，再执行受影响 target/package、必要 downstream current artifact、部署、重启和功能复测；禁止用 SSH/SCP/临时命令直接改板。

原因是当前 `push_v5_runtime_to_board.sh` 会处理 board-owned 文件：会 merge self parameter table，并保留或刷新 drive parameter table。板端当前值只允许只读导出到 `${VM_BUILD_ROOT}/temp_parameter_snapshot/<run_id>` 的单次事务临时快照，并在 `VM_BUILD_ROOT` 下的部署产物中合并；不得回拉覆盖 Windows 跟踪源码。事务成功或失败恢复后必须删除临时快照，不得建立 `bak/` 或持久备份目录。参数变更若未先通过独立 canonical 事务写入真实 owner，完整部署阶段可能继续保留板端旧值。

参数文件单独上传闭合至少包括：

- 本地 owner 文件 hash。
- 板端目标路径 hash。
- 字段级 readback，例如 INI section/key、TSV 行或 JSON profile sha256。
- 后续完整部署后再次确认该字段仍在板端 owner 中。

## 5. Operator Path 和输入工具

Codex的板端operator执行面固定为`board/tools/deploy/capture_v5_board_ui.sh`或`collect_v5_remote_input_evidence.py capture`抓取fresh frame，以及`board/tools/deploy/collect_v5_remote_input_evidence.py`逐次执行pointer/mouse模拟并保存动作前后frame；SSH证据通道可用时同时保存事件delta，不可用时必须由`board_exec`补取同一动作后的事件与native actual。`8ax-win/publish/win-x64/8ax.WinRemote.exe`只供人查看和操作，Codex不得用它自动输入；不得启动第二relay或使用V3产物。

Codex/AI查看开发板当前屏幕、进行页面导航以及模拟click/drag时，每次输入前先用上述抓屏工具取得fresh frame并确认页面、目标和遮挡；再用上述pointer工具只执行一个小动作；动作后读取该工具保存的after frame、UI/input事件delta与native actual。用户已明确授权自动执行时不再逐步请求确认。禁止裸坐标连点、自动操作WinRemote、恢复已退役的本地查看桥、第二relay或直接FIFO写入替代该交互面。

上述pointer/mouse工具的模拟click/drag可以证明板端UI operator路径命中，但不能证明真实手指触摸、触摸校准、real-finger ergonomics或运动/安全闭环；这些仍按下文native actual、golden motion和最终ESTOP证据补齐。

`board/tools/deploy/capture_v5_board_ui.sh --apply` 只抓 framebuffer，不发送任何输入。它只用于补机器可读 framebuffer 证据或 MCP 抓屏链路故障诊断；日常看屏以 MCP 板端抓屏为准。

`board/tools/deploy/collect_v5_remote_input_evidence.py` 通过 relay WebSocket 发送 layout-only pointer 事件，并保存 before/after frame、`ui_events.jsonl` delta 和 `remote_input_events.jsonl` delta。它只用于补机器可读证据或诊断 MCP 输入链路，不是日常查看/模拟操作的交互面。使用规则：

- 只有明确需要补机器可读证据时才运行；运行前仍须用 MCP 板端抓屏确认当前页面和目标。
- 每次只采集一个已明确的小动作，随后读取脚本生成的 before/after frame 和事件 delta；不得用它替代 MCP 抓屏与鼠标单次操作。
- 如果点击命中与预期不一致、页面不对、坐标漂移、事件日志显示触发了其它按钮，立即停止并把失败先归为 `automation_tool` 或 `unclassified`，不得继续批量点击。
- 它不能证明真实手指触摸、触摸校准、Start/MDI/运动安全或真实 operator motion。

`board/tools/deploy/collect_v5_board_touch_evidence.sh` 和 `board/tools/deploy/collect_v5_board_ui_action_evidence.sh` 等待真实手指输入，不发送 synthetic input。它们用于真实触摸和本地按钮证据，要求先有截图证据，并保存 `touch_events.jsonl`、`ui_events.jsonl` delta。

最小 operator evidence 通常包括：MCP 点击前板端抓屏、目标位置、输入方式、点击后板端抓屏、`ui_events.jsonl`、必要时 `touch_events.jsonl` 或 `remote_input_events.jsonl`、actiond/Broker result、owner readback。

## 6.1 Golden Motion 前置：设置页 A-G 完整闭环

每轮完整自动闭环在进入第 7 节 Golden Motion 前，必须先执行 `../待做工作/测试设置.md` 的设置页测试。该执行卡提供逐步操作细节，本文固定其在总流程中的强制顺序和最低通过条件；不得因为后端动作、旧结果文件、一次 VM 构建或后续运动成功而跳过设置页。

前置环境必须先经继电器断电重启或等价 clean restart，确认 UI、State Publisher、Command Gate、settings actiond、LinuxCNC/HAL/EtherCAT 和 drive provider 已自动就位。AI 必须用 MCP 板端抓屏与 pointer/mouse 完成普通设置功能的 canonical remote input，每次点击前后保留对应 fresh frame、UI/actiond 事件、fresh result 和实际 owner readback；若声明真实触摸闭环则必须补真实手指的 `touch_events.jsonl` 证据。Direct UDS/socket 仅用于诊断。

| 顺序 | 设置页阶段 | 必须执行 | 进入下一阶段前的最低条件 |
| --- | --- | --- | --- |
| A | 身份与下载链 | 必要时登记本机码，再依次执行下载授权、服务器下载并确认 resident profile snapshot | 本机码为合法 6 位 ID；授权结果为 `DEVICE_AUTH_DOWNLOAD_OK` 且签名通过；profile 的 public/private 状态、授权与 hash 回读真实，不能把 absent/skipped 写成成功 |
| B | 驱动只读链 | 扫描从站、读取驱动 | `DRIVE_SCAN_OK`、`slaves` 非空、候选与轴绑定规则正确；最终必须是 `DRIVE_READ_OK`，任何 `DRIVE_READ_PARTIAL` 立即停步 |
| C | 驱动维护链 | 必要时复位驱动、清除故障、再次读取驱动 | 依次取得 `DRIVE_RESET_OK`、`DRIVE_FAULT_RESET_OK`、`DRIVE_READ_OK`；复位后的1/1和invalidated状态不是最终加工状态，禁止在旧进程用设置驱动按钮立即覆盖，后续必须由G阶段新启动链恢复最终egear/mode并fresh readback闭合 |
| D | 字段编辑链 | 至少分别测试普通输入、下拉、`bit`、G53 可编辑字段，并核对提交后的 UI 单元格 | `settings_axis_param_commit ok=true`；self、drive-only、runtime INI、kinematics native owner 各回到自己的真实 readback；缺失 `bit` 只能显示 `--`，不得提交伪值；G53 写入需在规定 restart/reload 后由 native/API 证明生效 |
| E | 运动模型选择链 | 先读取 fresh native current model，再在设置页选择与当前不同的登记模型并提交；读取轴参数表确认默认映射，再任选两条 active 轴通过 `从站` 下拉做一次唯一的人工交换后回读 | 提交必须带非零 `owner_generation`/`readback_token`，`settings_axis_param_commit motion_model ok=true`；同一事务必须回读目标 MODEL/KINS/TRAJ、AXIS→JOINT 和 descriptor 默认 axis→slave，人工交换必须回读同一 self owner。保存前只证明持久目标配置，不得冒充 active或写物理驱动 |
| F | 设置驱动预检链 | 通过真实 `设置驱动` 按钮冻结最终active axes/status slots/mapping/INI/profile/final bit | 返回`DRIVE_SET_RESTART_REQUIRED`或等价结构化结果，`restart_required=true`且`restart_deferred=true`；按钮期间零HAL写入、零SDO写入、零即时驱动readback，不得出现`DRIVE_SET_OK`或“已写入” |
| G | 返回主页/保存并重启 | 无有效参数变更且无restart pending时验证`返回主页`；形成有效字段变更后验证原按钮切换为`保存并重启`并提交完整clean reboot handoff | 无变更时不得触发apply/restart；有变更时旧进程不得重做设置驱动。重启后的新启动链必须一次读取最终model/mapping/INI/profile，HAL消费同一INI SCALE，驱动actual不一致时才写入并整批fresh readback；任一失败保持Machine Off。若模型切换后还需对新active axes执行BUS设0，则在第一次重启激活新映射后完成设0；只有该动作形成新的restart pending时再执行第二次clean restart |

上述每个异步动作必须用同一个 `run_id` 对齐 actiond `started/finished`，result `mtime` 必须晚于本轮动作开始，结果码和 provider/native readback 必须匹配。运动模型测试在单轮至少选择一个与当前 native model 不同的目标；完整双模型回归应分别证明 AC→BC 与 BC→AC。每次先保存所有最终字段并完成设置驱动预检，再以第一次 clean restart 激活目标model/mapping并由启动链闭合最终驱动写入/readback；随后才对新active axes执行BUS设0。设0若仅走已登记live事务且不形成重启pending，可直接进入对应原始程序Golden Motion；若较重字段明确待重启，则第二次clean restart后再进入。旧runtime拒绝新模型独有轴操作是应有的fail-closed，不得为了减少重启而放宽。任何`*_PARTIAL`、precheck failure、启动写入失败、`write_unverified_readback_failed`、motion model token/generation缺失、native active model未切换、readback mismatch、页面/按钮命中错误、弹窗无阶段推进或clean restart不完整，都必须停止总流程；修复并重跑当前失败阶段后才能继续，不能先进入Golden Motion。

## 6.2 Home / Start / E-stop 上板 focused 回归

Home、Start、E-stop 或驱动设置链的板端改动部署后，先关闭本节 focused 回归，再进入完整设置页 A-G 和 Golden Motion。该回归只用于尽快暴露当前受影响链的首错，不能替代第 6.1 节或第 7 节的完整闭环。

1. 在设置页通过真实 `设置驱动` 按钮执行一次最终持久输入预检，确认同一 `run_id` 返回`DRIVE_SET_RESTART_REQUIRED`（或等价结果）和restart pending；同时证明按钮期间没有HAL写入、SDO写入、Machine切换或“已生效”readback，按钮松手后退出黄色。
2. 通过右上角 `保存并重启` 执行 canonical clean restart；旧进程只提交handoff。重启后的新启动链重新核对全部当前目标的轴—从站绑定、INI SCALE/MOTOR_REV/LOAD_REV/PITCH、profile/final bit，HAL只消费同一INI SCALE，驱动actual不一致时才写egear/mode，随后整批fresh readback mode/egear/statusword/error_code与PDO/axis binding；缺项或失败保持Machine Off并不得进入运动。
3. 从 fresh native `estop_active=true`、`machine_enabled=false` 且本次 `estop_clean_generation` 已终态的干净执行态开始，先通过真实 UI 完成一次 `取消急停 -> 机械全轴回零 -> 打开程序 -> 选择并打开 active model 对应程序`，取得本次 fresh Home transaction、真实运动/到位、`all_homed`、RTCP force-off actual，以及 active-model 匹配的 program identity/hash。Home 未成功不得打开程序，程序未正确载入不得 Start。
4. 在同一 program identity 和有效 `all_homed` 上至少连续三轮执行 `启动 -> 1 秒后急停`；轮间只通过真实 UI 点击一次 `取消急停`，并在 fresh native `estop_active=false && machine_enabled=true` 后进入下一轮，不重新 Home、不重新打开程序。每次 Start 必须取得新的真实运动；每次急停必须先取得 native 安全 actual，再以新的 clean generation 证明 task/interpreter idle、队列为空、RTCP OFF、旧执行 active/pending 已终结，同时保留 program identity、刀路和 homed。最后一轮急停后保持 ESTOP / Machine Off。
5. 任一步失败只修本轮第一条真实失败层，部署内容未变不重复部署，Home 未成功不继续打开程序，程序未正确载入不继续 Start，急停未确认不执行恢复。不得通过重复点击、direct UDS/linuxcncrsh、延时兜底或旧 result 把该轮写成通过。

## 6.3 CPU0 运动实时域抗 UI 卡死验收

本节验证的最终产品目标只有一个：UI 即使完全卡死，运动也不能卡。它不是普通 UI 性能测试，而是 `REQ-CPU0-MOTION-REALTIME-DOMAIN` 的 P0 板端放行条件；只有板端运动相关改动、CPU/IRQ/调度改动、UI/Publisher/relay 性能改动或发布验收需要执行，本地文档和普通 source slice 不因此自动进入上板。

1. 在空载测试台架完成设置页 A-G、fresh Home、active-model 匹配程序打开和正常 golden motion 基线。开始压力前记录同一部署identity、program identity/hash、native active model、servo周期/latency/overrun、EtherCAT DC/WKC和各轴编码器连续性。冷/热启动窗口保留从servo-thread创建到readiness后30秒的未清零`tmax`；其它稳态窗口开始前先保存上一窗口读数，再清零`servo-thread.tmax`及同thread各function的RW `*.tmax`并记录monotonic起点，窗口结束必须先读回再允许下一次清零。
2. 读取并保存 LinuxCNC/RTAPI realtime servo、motmod/trajectory realtime、realtime kinematics、servo-period HAL、EtherCAT 周期线程、运动总线 IRQ，以及仍为防止运行期轨迹队列耗尽所必需的最小有界 lookahead/queue-feed 线程的 affinity/cpuset/scheduler；同时读取 UI/LVGL、Broker、Command Gate 用户态 transport、State/WCS Publisher、provider、remote relay/镜像、日志、网络/文件系统任务及其 IRQ/workqueue。CPU0 只允许 `REQ-CPU0-MOTION-REALTIME-DOMAIN` 登记的运动实时闭集，非运动任务或 IRQ 命中 CPU0 即失败；若运动仍依赖 CPU1 上的普通 interpreter/file parser 持续补队列，也直接失败。
3. 本项实施前必须新增、登记并通过`board/tools/linuxcnc/v5_cpu1_motion_stress.py`及其focused self-test；canonical工具不存在时本验收为`blocked`，不得用临时SSH busy loop、`yes`、全树扫描或未登记`stress-ng`替代。板端只允许把同hash工具临时放到`/tmp/v5_test_tools/`，不得进入runtime manifest或产品服务。
4. 工具默认压力60秒，只允许显式选择30～60秒。启动后必须先用`sched_setaffinity`把自身和全部worker限定到CPU1，再用`sched_getaffinity`与`/proc/<pid>/status`双重回读；allowed mask包含CPU0、CPU1不可用、worker身份不匹配或采样器未固定CPU1时立即退出且不得开始压力。UI冻结只允许工具针对精确PID加完整cmdline identity执行有界30秒`SIGSTOP`/`SIGCONT`；禁止按进程名批量停止。
5. 压力和UI冻结窗口均按100ms节拍采样，工具必须记录monotonic时间、样本序号、CPU0/CPU1 ticks、servo period/Time/Max-Time/overrun、trajectory queue/requested-vel/current-vel、5轴OP、Domain WKC/complete、`dc-phased`、`dc-time-valid/age/ok-seq/error-count`、phase-jitter、各轴encoder/following error和相关错误计数。每个有效样本必须为5轴全OP、WKC 10/10 complete、DC valid/age=0/seq推进/error-count无增量；1ms验收还要求`abs(phase-jitter)<=100000ns`。漏样本、错误计数增量或无法绑定同一generation都使窗口无效，不能当成稳定。
6. 压力期间允许UI、镜像和非安全日志降帧、延迟、丢弃陈旧帧或完全卡住，但同一G-code运动必须持续推进，轨迹队列不得耗尽；不得出现位置停住后再跳、速度突变、可见抖动、servo overrun、EtherCAT DC/WKC异常、编码器反馈中断或运动总线掉周期。硬件急停、限位和realtime safety owner仍必须保持原生有效，不能为了压力测试关闭安全链。
7. 工具必须在正常结束、异常、超时、SIGINT和SIGTERM路径恢复被停止的精确UI PID、停止并回收全部worker、写出终态后删除板端临时工具；随后证明无残留测试PID、UI/Publisher/relay从fresh native actual恢复、CPU1在有界窗口回到基线，且没有重放旧控制请求或陈旧帧改变运动。恢复前后运动轨迹和编码器时间序列必须连续；UI恢复本身不能造成运动停顿。
8. 任一“UI卡死导致运动也卡”、CPU1任务/IRQ溢出到CPU0、servo掉周期、DC/WKC异常、压力工具清理不完整或UI未恢复，均为P0失败，不得声明`board_verified`。测试结束必须执行真实急停并证明`estop_active=1`、`machine_enabled=0`。

## 7. `cc-ac.ngc` / `cc-bc.ngc` Golden Motion

只有第 6.1 节设置页 A-G 在同一轮全部闭合后，才允许进入 Golden Motion。后续自动闭环测试的默认执行入口是 `../待做工作/跑cc.md`，原始 golden 程序只在 `board/gcode/golden/cc-ac.ngc`、`board/gcode/golden/cc-bc.ngc`。完整闭环必须按真实 UI/operator 路径跑：继电器断电重启开发板，等待 LinuxCNC/EtherCAT/command gate/UI relay/State Publisher/native 状态块恢复后，系统必须保持 realtime latch 安全态和 `Machine Off`，服务启动/重启不得自动 reset 或 `Machine On`。完成一次设置驱动和 canonical clean restart 后，先完成一次真实 `取消急停 -> 机械全轴回零 -> 打开程序 -> 选择并打开模型匹配程序`，随后在同一 program identity 和有效 `all_homed` 上至少连续三轮 UI `启动 -> 1 秒后急停`；轮间只取消急停，不重新打开程序、不再次 Home，最后一轮急停后保持 ESTOP / Machine Off。AC/`XYZAC_TRT` 只允许通过程序页先选中、再双击或第二次点击同一行打开 `cc-ac.ngc`；BC/`XYZBC_TRT` 同样只允许打开 `cc-bc.ngc`。UI 程序名、runtime program identity 和原始文件 hash 必须共同证明实际载入的是模型匹配文件；未知模型、回读不新鲜、程序不匹配或 UI 没有完成打开都必须 fail-closed，不得回退到另一模型或旧 `cc.ngc`。每次 Start 均须取得新运动；每次急停均须取得安全 actual 与干净执行态 terminal，同时保留 program identity、刀路和 homed。编码器反馈用于证明程序开头的机床坐标快速归一化 `A0 C0` 或 `B0 C0`、随后的模型专属首点相位以及一圈内旋转轴绝对相位目标实际走最近 `k*360` 等效角；不得只用坐标显示为 0、按钮提交成功、`Set Run` 成功、direct UDS 或后端 runner 结果替代。

golden 源文件固定为两个独立的 LinuxCNC 原始程序：

```text
board/gcode/golden/cc-ac.ngc
board/gcode/golden/cc-bc.ngc
```

`cc-ac.ngc` 只使用 XYZAC 轴字，`cc-bc.ngc` 只使用 XYZBC 轴字；两者表达各自运动模型下的同一类 golden 运动意图。两份程序的五圈弹簧都必须是同一个不中断的 RTCP 加工段：XYZ 使用 G17、增量 I/J 圆心和同步 Z 的 `G3` 螺旋圆弧，不得用密集 `G1` 小线段逼近；为了保留 A/B 轴每圈 `45°→55°→45°→35°→45°` 的姿态摆动，每圈固定为四个相切的90°圆弧、五圈共20块，块间不得插入暂停、G0、exact-stop、抬刀或RTCP切换。BC 模型的首个弹簧锚点固定为 `X0 Y10 Z0 B45 C90`：按当前 `xyzbc-trt-kins` 逆运动学，该点相对旋转中心的机床 Z 分量为 `10×sin(45°)`，处于该半径圆周的最高点，保证刀具能从上方接触首点；整条 BC 加工段的 C 目标必须在原连续五圈相位上统一加 `+90°`，即首点 `C90`，随后每个 90° 圆弧依次为 `C0、C-90 … C-1710`，仍保持每段 `-90°` 和总计五圈连续旋转。禁止只改首点而使第一段产生 `-180°` 跳变。它们都必须是可直接交给 LinuxCNC/微内核解释的原始 G-code 文件；验证时打开的内容、复制到板端的内容和 Start/`Set Open` 投喂的内容必须一致，不得由 UI、runner、脚本或中间层做 A/B 轴替换、预览解析、旋转等效角改写、临时 sidecar、运行副本、短程序或任何“等效处理”。等效角、连续多圈、旋转轴 normalization 和 fail-closed 判定属于微内核/native owner，UI 显示/移动链路的解析结果不参与 golden 执行语义。旧 `cc.ngc` 不再是支持入口，不得保留为 fallback、别名或模型不明时的默认程序。

两个 model-specific golden 程序都由 UI Program Open 路径把已打开的原始文件直通 LinuxCNC/微内核执行，程序运行所需的 RTCP 开/关必须由当前程序文本自己控制微内核运行态：非切削预定位、旋转轴归零/归位和机床坐标清理段默认保持 RTCP OFF；到达弹簧起点后、第一条弹簧 `G3` 螺旋圆弧进给前，程序文本内必须用 LinuxCNC 原生 `M64 P0` 打开已登记的 RTCP/switchkins 程序输出，并等待 native actual ON；最后一条弹簧 `G3` 螺旋圆弧到达弹簧终点后、进入 `Machine-coordinate return` 清理段前，必须用 `M65 P0` 释放该程序输出。G-code 程序链路与 UI 按钮链路相互独立：程序内 `M64/M65` 直驱微内核程序输出，UI 按钮只通过 `native_rtcp_control` latch 直驱微内核；两条链路不互相代发、不互相预置，最后只在同一个 HAL/native switchkins actual 上合流。`M65 P0` 之后只允许执行末尾 G53/G49 机床坐标清理动作，例如 Z 回零、当前模型旋转轴回零和 X/Y 回零；不得再进入加工段或继续以 RTCP ON 语义解释轨迹。后端 runner、direct UDS、linuxcncrsh 文本、MDI、临时 sidecar、UI 预处理或 Python product action 均不得替代 `取消急停 -> UI机械全轴回零 -> 打开程序 -> 选中 -> 双击/第二次点击打开 -> 回主页面 -> UI启动` 的黄金测试入口，也不得替程序外部预置 RTCP。历史 M128/M129 不得恢复。

若自动闭环在 `M64 P0` 之后、`M65 P0` 之前按下急停，后续机械全轴 Home 必须先由 Home owner 中止残留 Auto，并通过同一个 `native_rtcp_control` owner 事务性 force-off：不得伪造或替程序发送 `M65 P0`，但必须原子清除当前 UI latch 与残留程序输出请求，并等 fresh `rtcp_valid=true && rtcp_active=false` 后才切入不依赖 world/switchkins 姿态的 native joint/Home 域执行真实 Home。force-off 失败必须零轴运动并返回本次结构化失败；Home 完成、失败或取消后保持 RTCP OFF，不恢复旧 ON，也不修改持久 `[RTCP].DEFAULT_ACTIVE`。G-code 正常路径中的 `M64 P0`/`M65 P0` 仍只由当前 model-specific 程序自己控制；测试必须记录 Home 前后 UI/program request 与同源 RTCP actual，证明 Home 使用 native force-off 而非 G-code/文本旁路。

主页面 UI 的 RTCP 按钮不是 G-code 程序路径，仍必须直通 `native_rtcp_control` 微内核 latch，并跟随同一个 native RTCP actual 做 ON 高亮和刀长下方 `RTCP ON/OFF/--` 显示。UI 按钮不得把点击转义成 G-code 程序输出、`M64/M65`、M128/M129、MDI 文本或 linuxcncrsh 普通命令；G-code 程序内的 `M64/M65` 只属于程序链路直驱微内核。

`run_v5_board_acceptance.sh --apply` 只负责构建、部署、runtime verify 和截图，不得直接启动 golden 程序。`--motion` 若保留，只能调度真实 UI/operator 自动化：每一步输入前抓取当前 frame，先通过 UI 取消急停并完成机械全轴 Home，再点击主页面 `打开程序`，在程序页第一次点击模型匹配行完成选中，第二次点击同一行完成打开，回主页面核对程序名/runtime identity/hash，再通过 UI 点击启动；它不得调用 `v5_linuxcncrsh_golden_run --start`、direct UDS、`Set Open`、`Set Run` 或其它后端直跑入口。`v5_linuxcncrsh_golden_run` 如保留，只能做无运动的 active model/程序匹配诊断，不能作为黄金测试执行器、前置动作或通过证据。运动相关 `board_verified` 仍必须补足：

BUS Home 前置必须在未 homed 的 joint jog 域和已 homed/coordinated 的轴字母 jog 域之间按 native all_homed 状态选择；两种状态都必须产生真实运动和到位/readback 证明，不得因为已经 homed、坐标显示为 0 或刚跑过程序就跳过 Home、只 `Set Home` 或只清坐标。

A/C 旋转轴 Home 必须证明本次完整运动、native motion-complete、in-position 和连续静止，再由本次 native Home 坐标同步使 MCS/等效相位回读为 0；自动测试不得再以 target-error/最短角差容差决定 Home 成败，也不能只改 UI 显示。

- 原始 UI/operator path，且程序必须通过 `打开程序 -> 选中 -> 双击/第二次点击打开` 进入运行态；不接受 owner 指定的后端替代路径。
- 程序路径和 hash 匹配 native active model 对应的 `cc-ac.ngc` 或 `cc-bc.ngc`。
- Start 后可证明的速度、joint 或 encoder 位移。
- Stop/E-stop/reset 或最终 safe stop 后的 microkernel/native 安全状态。
- Home 或设0 前置条件必须走对应 owner：要求设置页轴设0时，不能用主页 Work Zero 冒充。

在当前空载电机测试台架上，凡是用户要求验证 Home/Jog/Start/MDI/G-code/回零等运动路径，默认应跑真实运动证据；`--motion` 路径应显式设置 `V5_ALLOW_MOTION=1` 并执行，不能再把“未确认机械安全”作为本项目当前台架的默认 blocker。若后续接上机械结构、刀具、工件或负载，必须先更新本文硬件状态，再恢复相应的现场安全门禁。

不得用短程序、空程序、示例折线或为了测试改写后的用户 G-code 代替 golden。

## 8. 失败处理

失败是下一步诊断输入，不是交付终点。失败初始状态必须是 `unclassified`，候选域至少包括：

- `project_source`：源码、配置、owner 文档或部署产物问题。
- `automation_tool`：截图、relay、remote input、坐标、证据采集脚本或部署检查误报。
- `board_runtime`：服务生命周期、socket、SHM、LinuxCNC/HAL/EtherCAT、drive provider 状态。
- `hardware_safety`：运动条件、急停、machine on/off、驱动使能、真实触摸或现场安全。

每次失败后先重跑能覆盖该失败的最小门禁，再升级到 required mode。默认 AI 被要求取得 `board_verified` 时，不得擅自把应由 AI 完成的闭环转交用户；但用户已明确保留最终人工测试时，缺少 operator/golden/motion 证据不是 blocker，AI 在完成构建、部署和启动健康检查后按人工验收模式交接。板端忙、构建或部署失败、网络不可达仍按 AGENTS.md 报告真实 blocker；不得把新增弹窗、禁用按钮或错误提示当作修复完成。

## 9. 快速判断

| 问题 | 答案 |
| --- | --- |
| dry-run 能否证明功能通过 | 不能，只能分类和计划 |
| `run_v5_board_acceptance.sh --apply` 是否等于功能通过 | 不是，它只完成构建、部署、runtime verify 和截图 |
| direct UDS 能否替代 UI 按钮验证 | 不能，除非缺陷本身就是 UDS 合同 |
| MCP pointer/mouse synthetic click 能否证明真实触摸 | 不能 |
| MCP pointer/mouse synthetic click 能否单独证明 Start/MDI/运动 | 不能；还必须取得 native actual、真实运动和最终安全态 |
| `--motion` 是否等于完整 golden loop | 不能，只是提交 golden start 的当前工具入口 |
| 只跑 VM build 能否说上板好了 | 不能 |
| board_deployed 是否等于功能通过 | 不是 |
| 用户指定最终人工测试时 AI 做到哪里 | 修改 canonical source，完成最小编译、受影响产物构建、部署和启动健康检查；随后以 `local_verified_only` 交接“已部署待人工验收” |
| 运动相关通过的核心证据 | 原 UI/operator/control path、active model 匹配的 `cc-ac.ngc` 或 `cc-bc.ngc`、真实位移、最终 native 安全状态 |
