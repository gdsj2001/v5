## 1. 当前代码事实

- V5 可维护源码只存在于 Windows `D:\v5` 这一棵 Git 工作树：完整 Linux kernel 在 `linux/kernel/`，PREEMPT_RT/Yocto metadata 在 `linux/realtime/`，完整 LinuxCNC/native 在 `linuxcnc/`，完整 PetaLinux project-spec 在 `board/petalinux/`，板端集成/LVGL/services/tools 在 `board/` 其余 owner，硬件工程在 `vivado_hw_project/`。
- VM 必须把 Windows 真源实时只读挂载到 `VM_SOURCE_MOUNT_ROOT=/mnt/v5-source`；generated files、中间物和产物只写入 `VM_BUILD_ROOT=/root/v5-build`。为避开共享目录小文件性能问题，只允许 `${VM_BUILD_ROOT}/temp_source/current` 一套由该挂载 identity 派生、不可编辑的加速投影：identity 一致时复用，不一致时只刷新这一处；`/root/Desktop/v5`、第二投影、源码同步 manifest、VM checkout 和其它持久 VM 源码副本均为退役路径。
- 板端部署和验收工具在 `board/tools/deploy/`，不是项目根目录 `tools/`。
- 当前没有一个能自动完成所有设置页 A-G、真实 UI 点击、真实触摸和 motion golden loop 的单一 runner。闭环必须由当前工具分段组合，并按每段证据判断。
- 历史 `v3_hardware_regression_matrix.py` runner 当前不在仓库内，不能在本文或验收中作为现有工具引用。
- 当前测试台架硬件状态：开发板连接的电机为空载，未连接机床机械结构、丝杠、转台、刀具或工件；因此本阶段自动闭环测试默认允许直接执行真实电机运动，包括 Home、Jog、MDI、Start、`cc-ac.ngc`/`cc-bc.ngc` golden motion 和回零相关运动，不再因为机械结构碰撞、行程干涉或负载风险把真实运动验证降级为 direct UDS、dry-run、只截图或只构建。当前台架状态下，机械碰撞/负载安全风险不是默认 blocker；凡验收运动链路，必须跑真实运动证据。运动命令仍必须走对应 native/LinuxCNC/HAL/EtherCAT owner，急停、驱动故障、通信异常、owner fail-closed 和最终停机状态仍然是真实证据，不得伪造或用旧 result 替代。

当前板端 acceptance 入口：

```sh
cd /mnt/v5-source/board
V5_REPO_ROOT=/mnt/v5-source/board \
V5_BOARD_BUILD_DIR=/root/v5-build/board \
V5_BOARD_SSH=root@192.168.1.221 \
V5_BOARD_SSH_PORT=22 \
tools/deploy/run_v5_board_acceptance.sh --apply
```

`run_v5_board_acceptance.sh --apply` 当前执行顺序是：

1. `cmake --build "$V5_BOARD_BUILD_DIR" --target $V5_BOARD_BUILD_TARGETS`
2. `tools/deploy/precheck_v5_board.sh`
3. `tools/deploy/push_v5_runtime_to_board.sh --apply`
4. `tools/deploy/verify_v5_board_runtime.sh`
5. `tools/deploy/capture_v5_board_ui.sh --apply`

默认 `V5_BOARD_BUILD_TARGETS` 是当前部署需要的目标集合；如果修改进入了其它板端产物，必须显式扩展目标或跑能覆盖该产物的构建门禁，不能用默认目标掩盖未构建产物。

### v5 MCP 工具边界

Codex 当前用于本项目的 v5 远端工具只保留两个执行入口：VM 只使用 `vm_exec`，开发板只使用 `board_exec`。`vm_exec` 负责验证 `/mnt/v5-source` 是 Windows 真源的实时只读共享挂载、核对/刷新唯一 `/root/v5-build/temp_source/current` 投影，并在 `/root/v5-build` 执行受影响 target/package、必要 rootfs/manifest、一次 current 镜像和部署命令；`board_exec` 负责在 `z20_plnx` 执行板端只读状态探测、产物核对和 operator-path 所需命令。部署使用 canonical `board/tools/deploy/run_v5_board_acceptance.sh --apply` 或 owner 登记的 focused deploy entry。不新增 MCP 专用旁路，不允许通过 MCP 创建第二源码投影或直接编辑板端产品文件。

除 `vm_exec` 和 `board_exec` 外，v5 远端 MCP 不再暴露其它命令执行、构建、扫描、源码复制或诊断入口；测试、构建、部署和验收文档不得恢复、别名化或新增 v5 远端 MCP 旁路。Codex 自身的 `node_repl` 和其它通用插件 MCP 不属于 v5 远端工具边界。

## 2. 可声明状态

| 范围 | 最低验证 | 可声明状态 |
| --- | --- | --- |
| 只改文档/规则 | `git diff --check`，必要时做单源 owner 人工 review | `source_only` |
| 非板端本地代码 | focused compile、`py_compile`、focused pytest 或静态扫描 | `local_verified_only` |
| 进入板端产物或影响板端运行 | VM 只读挂载身份、挂载外 canonical build、部署、运行闭包重启、原始 control/operator 路径证据 | 只能按证据说明；未走 operator path 不得写 `board_verified` |
| 非运动 UI/operator 缺陷 | 当前页面截图或 remote frame、逐步输入、UI/action/touch 事件、结果和 readback | 该非运动功能满足 owner 验收后可写 `board_verified`；不代表 motion 通过 |
| Start/Jog/Home/Stop/E-stop/MDI/G-code/运动门禁 | operator path 加 motion owner 要求；默认还需要 active model 对应的 `cc-ac.ngc` 或 `cc-bc.ngc` golden motion 和最终安全状态 | `board_verified` |
| 硬件不安全、板端不可达、证据拿不到 | 记录阻塞、已尝试动作、缺失证据和下一验收条件 | `blocked`、`source_only` 或 `local_verified_only` |

`relay_ready`、`board_deployed`、`board_runtime_ready`、VM build、dry-run、matrix pass、direct UDS 和旧 result JSON 都不是功能通过。旧 result 文件早于当前相关源码、配置或文档修改时，只能作为历史诊断，不能证明当前工作树。

## 3. 板端程序修改闭环

任何会进入板端产物或影响板端运行的程序、配置、脚本、服务、recipe、部署项修改，都属于板端程序修改，包括 `linux/kernel/` 的 Linux kernel、`linux/realtime/` 的 PREEMPT_RT/Yocto metadata、`linuxcnc/` 的 LinuxCNC/native、`board/petalinux/` 的 PetaLinux project-spec，以及 `board/app/`、`board/linuxcnc/`、`board/services/microkernel/`、`board/services/command_gate/`、`board/services/state_publisher/`、其它 `board/services/` 和 `board/tools/` 的 UI、HAL/INI/runtime packaging、native integration、action、参数 provider、启动/部署脚本、init 脚本、部署 manifest 和板端资源文件。微内核合同与常驻 owner 已有独立目录，不得继续笼统写成 Command Gate 内部文件。

板端程序修改关闭条件：

- 先跑最小本地门禁，暴露语法、合同和局部行为错误。
- 用 `findmnt` 或等价证据确认 `VM_SOURCE_MOUNT_ROOT` 是当前 Windows 真源的实时只读共享挂载；核对 `${VM_BUILD_ROOT}/temp_source/current` 的 machine-readable identity 与挂载一致，并确认 `/root/Desktop/v5`、第二投影、VM checkout 等其它源码副本不存在。current 投影允许跨任务复用，但不得人工编辑、提交或反向覆盖 Windows。
- 在 `VM_LINUX_BUILD_DIR` / `VM_BOARD_BUILD_DIR` 构建本次会进入部署产物的目标集合；源码挂载内不得产生 generated files、中间物或产物。
- 用 `run_v5_board_acceptance.sh --apply` 或等价 deploy 脚本部署 canonical artifact。
- 检查实际运行服务、socket、SHM/native block、runtime INI、drive provider、UI relay 和需要的 result/readback。
- 回到用户报告的原始 UI/operator/control 路径验证；UI 问题不能只用 direct UDS 关闭。
- 涉及运动或安全的，再执行对应 motion/safety owner 要求。

若部署脚本返回非零，不能直接宣称通过。必须先按 `failure_triage` 分类：如果独立 init status、进程、socket、SHM/readback 证明服务实际已恢复，可把它记录为 `automation_tool` 或 deploy check 缺口；如果不能证明，状态必须保持 `blocked` 或未完成。

## 4. 参数文件门禁

修改 `self_parameter_table.tsv`、`drive_parameter_table.tsv`、drive profile map、LinuxCNC INI、`tool.tbl`、`linuxcnc.var` 等板端持久 owner 时，必须先从 Windows 真源生成独立参数部署产物，通过 canonical deploy/apply 事务写入板端 owner 并做 hash/readback，再执行受影响 target/package、必要 downstream current artifact、部署、重启和功能复测；禁止用 SSH/SCP/临时命令直接改板。

原因是当前 `push_v5_runtime_to_board.sh` 会处理 board-owned 文件：会 merge self parameter table，并保留或刷新 drive parameter table。板端当前值只允许只读导出到 `${VM_BUILD_ROOT}/temp_parameter_snapshot/<run_id>` 的单次事务临时快照，并在 `VM_BUILD_ROOT` 下的部署产物中合并；不得回拉覆盖 Windows 跟踪源码。事务成功或失败恢复后必须删除临时快照，不得建立 `bak/` 或持久备份目录。参数变更若未先通过独立 canonical 事务写入真实 owner，完整部署阶段可能继续保留板端旧值。

参数文件单独上传闭合至少包括：

- 本地 owner 文件 hash。
- 板端目标路径 hash。
- 字段级 readback，例如 INI section/key、TSV 行或 JSON profile sha256。
- 后续完整部署后再次确认该字段仍在板端 owner 中。

## 5. Operator Path 和输入工具

当前代码有三类输入/证据工具，不能混用证据含义。

`board/tools/deploy/capture_v5_board_ui.sh --apply` 只抓 framebuffer，不发送任何输入。它可作为点击前页面确认或最终画面证据。

`board/tools/deploy/collect_v5_remote_input_evidence.py` 通过 relay WebSocket 发送 layout-only pointer 事件，并保存 before/after frame、`ui_events.jsonl` delta 和 `remote_input_events.jsonl` delta。它适合非运动、非真实触摸校准的 UI 布局点击诊断。使用规则：

- 每次 click/drag 前必须先确认 before frame 的页面、按钮状态和目标位置。
- 每次只做一个小动作，动作后读取 after frame 和事件 delta。
- 如果点击命中与预期不一致、页面不对、坐标漂移、事件日志显示触发了其它按钮，立即停止并把失败先归为 `automation_tool` 或 `unclassified`，不得继续批量点击。
- 它不能证明真实手指触摸、触摸校准、Start/MDI/运动安全或真实 operator motion。

`board/tools/deploy/collect_v5_board_touch_evidence.sh` 和 `board/tools/deploy/collect_v5_board_ui_action_evidence.sh` 等待真实手指输入，不发送 synthetic input。它们用于真实触摸和本地按钮证据，要求先有截图证据，并保存 `touch_events.jsonl`、`ui_events.jsonl` delta。

最小 operator evidence 通常包括：点击前 frame、目标位置、输入方式、点击后 frame、`ui_events.jsonl`、必要时 `touch_events.jsonl` 或 `remote_input_events.jsonl`、actiond/Broker result、owner readback。

## 6. 设置页闭环的当前代码入口

设置页闭环证据必须同时看：

- actiond `started` 和 `finished` 事件中的同一个 `run_id`。

## 6.1 Golden Motion 前置：设置页 A-G 完整闭环

每轮完整自动闭环在进入第 7 节 Golden Motion 前，必须先执行 `../待做工作/测试设置.md` 的设置页测试。该执行卡提供逐步操作细节，本文固定其在总流程中的强制顺序和最低通过条件；不得因为后端动作、旧结果文件、一次 VM 构建或后续运动成功而跳过设置页。

前置环境必须先经继电器断电重启或等价 clean restart，确认 UI、State Publisher、Command Gate、settings actiond、LinuxCNC/HAL/EtherCAT 和 drive provider 已自动就位。每次点击前后都要保留当前 frame、UI/actiond 事件、fresh result 和实际 owner readback；普通设置功能可使用 canonical remote input，若声明真实触摸闭环则必须补真实手指的 `touch_events.jsonl` 证据。Direct UDS/socket 仅用于诊断。

| 顺序 | 设置页阶段 | 必须执行 | 进入下一阶段前的最低条件 |
| --- | --- | --- | --- |
| A | 身份与下载链 | 必要时登记本机码，再依次执行下载授权、服务器下载并确认 resident profile snapshot | 本机码为合法 6 位 ID；授权结果为 `DEVICE_AUTH_DOWNLOAD_OK` 且签名通过；profile 的 public/private 状态、授权与 hash 回读真实，不能把 absent/skipped 写成成功 |
| B | 驱动只读链 | 扫描从站、读取驱动 | `DRIVE_SCAN_OK`、`slaves` 非空、候选与轴绑定规则正确；最终必须是 `DRIVE_READ_OK`，任何 `DRIVE_READ_PARTIAL` 立即停步 |
| C | 驱动写入链 | 复位驱动、清除故障、设置驱动、再次读取驱动 | 依次取得 `DRIVE_RESET_OK`、`DRIVE_FAULT_RESET_OK`、`DRIVE_SET_OK`、`DRIVE_READ_OK`；复位后的 1/1 和 invalidated 状态不是最终加工状态，必须由设置驱动后的 `write_verified_readback`、目标 egear/mode/statusword/error_code 和最终读取闭合 |
| D | 字段编辑链 | 至少分别测试普通输入、下拉、`bit`、G53 可编辑字段，并核对提交后的 UI 单元格 | `settings_axis_param_commit ok=true`；self、drive-only、runtime INI、kinematics native owner 各回到自己的真实 readback；缺失 `bit` 只能显示 `--`，不得提交伪值；G53 写入需在规定 restart/reload 后由 native/API 证明生效 |
| E | 运动模型选择链 | 先读取 fresh native current model，再在设置页选择与当前不同的登记模型并提交 | 提交必须带非零 `owner_generation`/`readback_token`，`settings_axis_param_commit motion_model ok=true`；保存前只证明目标配置写入/readback，不得冒充 active。G 阶段 clean restart 后，native `motion_model` 必须 fresh 且等于目标 canonical model，LinuxCNC kinematics/coordinates、主页面轴组、G53几何和黄色轨迹必须消费同一 readback |
| F | 轴设0链 | BUS 模式通过设置页目标轴 `设0` | UI 发出 `settings_axis_zero_request`，actiond 走 `settings_axis_zero -> axis-zero`，编码器/count-domain、raw soft limits 和 active scale 同事务回读，结果为 `SETTINGS_AXIS_ZERO_ENCODER_MATCH` 或当前 owner 定义的等价成功；Pulse HOME_OFFSET 另按其 owner 验证 |
| G | 返回主页面/保存并重启 | 无 dirty 时验证 `返回主页面`；形成有效字段变更后验证 `保存并重启` | 无变更时不得触发 apply/restart；有变更时必须执行等价断电重启的 clean restart，安全去使能并重建 UI/Broker/actiond/State Publisher/LinuxCNC/HAL/EtherCAT/drive provider、owner、运动模型、轴绑定和比例链，重启后由 resident/native/drive readback 证明新值仍有效 |

上述每个异步动作必须用同一个 `run_id` 对齐 actiond `started/finished`，result `mtime` 必须晚于本轮动作开始，结果码和 provider/native readback 必须匹配。运动模型测试在单轮至少选择一个与当前 native model 不同的目标；完整双模型回归应分别证明 AC→BC 与 BC→AC，且每次后续 Golden Motion 都只能使用重启后 fresh active model 对应的原始程序。任何 `*_PARTIAL`、precheck failure、写入失败、`write_unverified_readback_failed`、motion model token/generation 缺失、native active model 未切换、readback mismatch、页面/按钮命中错误、弹窗无阶段推进或 clean restart 不完整，都必须停止总流程；修复并重跑当前失败阶段后才能继续，不能先进入 Golden Motion。

## 7. `cc-ac.ngc` / `cc-bc.ngc` Golden Motion

只有第 6.1 节设置页 A-G 在同一轮全部闭合后，才允许进入 Golden Motion。后续自动闭环测试的默认执行入口是 `../待做工作/跑cc.md`，原始 golden 程序只在 `board/gcode/golden/cc-ac.ngc`、`board/gcode/golden/cc-bc.ngc`。完整闭环必须按真实 UI/operator 路径跑：继电器断电重启开发板，等待 LinuxCNC/EtherCAT/command gate/UI relay/State Publisher/native 状态块恢复后，系统必须已经自动 `Machine On`；自动上使能只能发生在 `board/services/microkernel/v5_microkernel_manifest.*` 登记的微内核/native 运行闭包完全起来之后，这至少包含 LinuxCNC/EtherCAT OP、command gate 常驻连接、RTCP/WCS/G53/native safety 状态块、State Publisher 和 UI relay 可读，不允许只在 SSH 或 LinuxCNC backend 刚起来时提前 `Machine On`。若此时 `linuxcncrsh Get Machine`、native safety readback 或同源 machine enable actual 仍为 `MACHINE OFF`/`enabled=false`，测试失败并修复开机自动上使能链路，不得通过 direct UDS、linuxcncrsh 或测试脚本补发 `Set Machine On` 掩盖。随后先读取微内核/native fresh active model，再在主页面点击 `打开程序` 进入程序页：AC/`XYZAC_TRT` 只允许在列表中先选中、再双击或第二次点击同一行打开 `cc-ac.ngc`；BC/`XYZBC_TRT` 同样只允许通过该 UI 路径打开 `cc-bc.ngc`。打开后必须回到主页面，并由 UI 程序名、runtime program identity 和原始文件 hash 共同证明实际载入的是模型匹配文件；未知模型、回读不新鲜、程序不匹配或 UI 没有完成打开都必须 fail-closed，不得回退到另一模型或旧 `cc.ngc`。之后只能通过主页面 UI 依次执行回零、启动、1 秒后急停、取消急停、再次回零，并在全过程采样编码器反馈。编码器反馈用于证明对应程序中的 `A0 C0` 或 `B0 C0` 以及一圈内旋转轴绝对相位目标实际走最近 `k*360` 等效角，两次回零实际走 360 度最短角差等效回零；不得只用坐标显示为 0、按钮提交成功、`Set Run` 成功、direct UDS 或后端 runner 结果替代。

golden 源文件固定为两个独立的 LinuxCNC 原始程序：

```text
board/gcode/golden/cc-ac.ngc
board/gcode/golden/cc-bc.ngc
```

`cc-ac.ngc` 只使用 XYZAC 轴字，`cc-bc.ngc` 只使用 XYZBC 轴字；两者表达各自运动模型下的同一类 golden 运动意图。它们都必须是可直接交给 LinuxCNC/微内核解释的原始 G-code 文件；验证时打开的内容、复制到板端的内容和 Start/`Set Open` 投喂的内容必须一致，不得由 UI、runner、脚本或中间层做 A/B 轴替换、预览解析、旋转等效角改写、临时 sidecar、运行副本、短程序或任何“等效处理”。等效角、连续多圈、旋转轴 normalization 和 fail-closed 判定属于微内核/native owner，UI 显示/移动链路的解析结果不参与 golden 执行语义。旧 `cc.ngc` 不再是支持入口，不得保留为 fallback、别名或模型不明时的默认程序。

两个 model-specific golden 程序都由 UI Program Open 路径把已打开的原始文件直通 LinuxCNC/微内核执行，程序运行所需的 RTCP 开/关必须由当前程序文本自己控制微内核运行态：非切削预定位、旋转轴归零/归位和机床坐标清理段默认保持 RTCP OFF；到达弹簧起点后、第一条弹簧 `G1` 进给前，程序文本内必须用 LinuxCNC 原生 `M64 P0` 打开已登记的 RTCP/switchkins 程序输出，并等待 native actual ON；最后一条弹簧 `G1` 到达弹簧终点后、进入 `Machine-coordinate return` 清理段前，必须用 `M65 P0` 释放该程序输出。G-code 程序链路与 UI 按钮链路相互独立：程序内 `M64/M65` 直驱微内核程序输出，UI 按钮只通过 `native_rtcp_control` latch 直驱微内核；两条链路不互相代发、不互相预置，最后只在同一个 HAL/native switchkins actual 上合流。`M65 P0` 之后只允许执行末尾 G53/G49 机床坐标清理动作，例如 Z 回零、当前模型旋转轴回零和 X/Y 回零；不得再进入加工段或继续以 RTCP ON 语义解释轨迹。后端 runner、direct UDS、linuxcncrsh 文本、MDI、临时 sidecar、UI 预处理或 Python product action 均不得替代 `打开程序 -> 选中 -> 双击/第二次点击打开 -> 回主页面 -> UI 回零 -> UI 启动` 的黄金测试入口，也不得替程序外部预置 RTCP。历史 M128/M129 不得恢复。

若自动闭环在 `M64 P0` 之后、`M65 P0` 之前按下急停，后续 `回零` 的 Home owner 只允许中止 Auto 并切入不依赖 world/switchkins 姿态的 native joint/Home 域执行真实 Home；不得替程序发送 `M65 P0`、不得清 `motion.digital-out-00`/`v5-rtcp-gcode-request`、不得通过 `native_rtcp_control` 请求 OFF，也不得把 RTCP actual OFF 作为 Home 前置。测试必须同时记录 Home 前后 RTCP native actual，证明 Home 没有改写 UI latch、程序输出或持久 `[RTCP].DEFAULT_ACTIVE`；G-code 正常路径的 `M64 P0`/`M65 P0` 仍只由当前 model-specific 程序自己控制。

主页面 UI 的 RTCP 按钮不是 G-code 程序路径，仍必须直通 `native_rtcp_control` 微内核 latch，并跟随同一个 native RTCP actual 做 ON 高亮和刀长下方 `RTCP ON/OFF/--` 显示。UI 按钮不得把点击转义成 G-code 程序输出、`M64/M65`、M128/M129、MDI 文本或 linuxcncrsh 普通命令；G-code 程序内的 `M64/M65` 只属于程序链路直驱微内核。

`run_v5_board_acceptance.sh --apply` 只负责构建、部署、runtime verify 和截图，不得直接启动 golden 程序。`--motion` 若保留，只能调度真实 UI/operator 自动化：每一步输入前抓取当前 frame，点击主页面 `打开程序`，在程序页第一次点击模型匹配行完成选中，第二次点击同一行完成打开，回主页面核对程序名/runtime identity/hash，再通过 UI 点击回零和启动；它不得调用 `v5_linuxcncrsh_golden_run --start`、direct UDS、`Set Open`、`Set Run` 或其它后端直跑入口。`v5_linuxcncrsh_golden_run` 如保留，只能做无运动的 active model/程序匹配诊断，不能作为黄金测试执行器、前置动作或通过证据。运动相关 `board_verified` 仍必须补足：

BUS Home 前置必须在未 homed 的 joint jog 域和已 homed/coordinated 的轴字母 jog 域之间按 native all_homed 状态选择；两种状态都必须产生真实运动和到位/readback 证明，不得因为已经 homed、坐标显示为 0 或刚跑过程序就跳过 Home、只 `Set Home` 或只清坐标。

A/C 旋转轴的 Home 到位判断必须使用 360 度等效最短角差；`359.999`、`-0.001` 这类等效零只能在编码器/微内核实际到位且角差在容差内时判定为 0，不能按线性差值误判失败，也不能只改 UI 显示。

- 原始 UI/operator path，且程序必须通过 `打开程序 -> 选中 -> 双击/第二次点击打开` 进入运行态；不接受 owner 指定的后端替代路径。
- 程序路径和 hash 匹配 native active model 对应的 `cc-ac.ngc` 或 `cc-bc.ngc`。
- Start 后可证明的速度、joint 或 encoder 位移。
- Stop/E-stop/reset 或最终 safe stop 后的 microkernel/native 安全状态。
- Home 或设0 前置条件必须走对应 owner：要求设置页轴设0时，不能用主页 Work Zero 冒充。

在当前空载电机测试台架上，凡是用户要求验证 Home/Jog/Start/MDI/G-code/回零等运动路径，默认应跑真实运动证据；`--motion` 路径应显式设置 `V5_ALLOW_MOTION=1` 并执行，不能再把“未确认机械安全”作为本项目当前台架的默认 blocker。若后续接上机械结构、刀具、工件或负载，必须先更新本文硬件状态，再恢复相应的现场安全门禁。

不得用短程序、空程序、示例折线或为了测试改写后的用户 G-code 代替 golden。

## 8. 失败处理

失败是下一步诊断输入，不是交付终点。失败初始状态必须是 `unclassified`，候选域至少包括：

- `project_source`：源码、配置、owner 文档或部署产物问题。
- `automation_tool`：截图、relay、remote input、坐标、证据采集脚本或部署检查误报。
- `board_runtime`：服务生命周期、socket、SHM、LinuxCNC/HAL/EtherCAT、drive provider 状态。
- `hardware_safety`：运动条件、急停、machine on/off、驱动使能、真实触摸或现场安全。

每次失败后先重跑能覆盖该失败的最小门禁，再升级到 required mode。板端忙、硬件不安全、网络/串口/relay 不可达或证据无法取得时，按 AGENTS.md 报告 blocker；不得要求用户人工代替闭环，也不得把新增弹窗、禁用按钮或错误提示当作修复完成。

## 9. 快速判断

| 问题 | 答案 |
| --- | --- |
| dry-run 能否证明功能通过 | 不能，只能分类和计划 |
| `run_v5_board_acceptance.sh --apply` 是否等于功能通过 | 不是，它只完成构建、部署、runtime verify 和截图 |
| direct UDS 能否替代 UI 按钮验证 | 不能，除非缺陷本身就是 UDS 合同 |
| remote input click 能否证明真实触摸 | 不能 |
| remote input click 能否证明 Start/MDI/运动 | 不能 |

| 只跑 VM build 能否说上板好了 | 不能 |
| board_deployed 是否等于功能通过 | 不是 |
| 运动相关通过的核心证据 | 原 UI/operator/control path、active model 匹配的 `cc-ac.ngc` 或 `cc-bc.ngc`、真实位移、最终 native 安全状态 |
