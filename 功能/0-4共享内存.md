## P0 唯一允许的 SHM 内容

/dev/shm/v3_status_shm 只允许承载 UI 显示投影。微内核/native 到 SHM 的稳态同步目标固定为 30Hz（约 33ms）坐标/轨迹动态显示；实现可用不低于 30Hz 的 30ms tick，但不得把全 UI 都提升到 30Hz，也不得用周期性整屏刷新或整页重绘消费 SHM。CPU0/CPU1 占用属于 5Hz 以下诊断显示，唯一 State Publisher sampler 只按 2 秒 monotonic 墙钟窗口动态汇总每核全部 `/sys/devices/system/cpu/cpuN/cpuidle/state*/time`，计算一次并在后续 33ms 帧中原样复用最近 immutable snapshot；仅 cpuidle 不可用时才允许 `/proc/stat` fallback。板屏不得再独立采样，relay `/remote/info` 和 Windows 顶部不得再次计算。普通按钮轮询、急停按键显示和其它普通 UI 状态刷新按 `REQ-UI-RUNTIME-REFRESH-RATE` 分层处理；UI 必须按坐标、轨迹动态图层、按钮、急停、诊断等 dirty 对象/dirty 区域分块按需刷新。本地板屏/producer 黄色轨迹在 10Hz 和 30Hz 都拉满 CPU1 时，降低本地频率只可作定位变量，不能作为关闭条件；必须消除同一 generation 的重复采样、RTCP/姿态/投影、对象重写和按 subscriber 重复组包。WinRemote 是独立的非运动镜像消费端，允许把 30Hz 本地 dirty latest-wins 合并为 10Hz 网络展示。100ms~300ms 只允许作为 SeqLock/CRC/源读取短暂失败且已经存在 last-good 帧时的显式 `stale/degraded` 视觉过渡，不能作为正常 native 采样、SHM 发布、板屏 UI model 应用或本地轨迹刷新周期；同步进程必须属于开机已加载的产品自写运行闭包，从微内核/native 常驻内存/API、OS resident counters 或登记 native memory block 采样后单向发布，不得在 steady-state 读取硬盘文件、扫描脚本目录、动态导入模块、执行一次性 helper 或读取 `/run` result 补帧。

同一 `scene_generation` 的显示数据只允许唯一 display projection producer 完成一次完整 native sample 校验、每轴显示量子/位数、坐标、跟随误差、RTCP gate、active-model 姿态和 bounded 黄色轨迹/scene 派生，再由唯一 Publisher 完成一次组帧、CRC 和 SeqLock commit；RTCP OFF 不得调用 RTCP 变换，RTCP ON 同一 scene generation 最多调用一次。不得因坐标面板、刀路、设置页、跟随误差、标签、本地缓存、远程画面或其它显示消费者数量增加而重复采样 native owner、重复投影、重复组帧或重复写 SHM。UI shell 在每个动态刷新周期只允许向 inactive 固定帧槽执行一次有界 SHM reader，校验成功后原子切换 active 槽；`V5UiStatusView`、`V5MainPage`、custom-draw renderer 和诊断只能持有该 active immutable scene 的 const 引用并各取所需。禁止把 6168 字节 scene 再赋值到状态 view 或页面对象，也禁止把 512 点再次逐点复制到 `trajectory_points`、`lv_point_t` 或等价 UI 数组；下一次读必须写另一固定槽，读失败不得覆盖仍被消费的 last-good 槽。UI widget、toolpath renderer 和 relay 不得再次读取源数据、计算坐标/跟随误差/RTCP/黄色轨迹、重打包同一字段或建立第二份 display projection。远程 relay 只消费 UI 已提交的 framebuffer dirty 投影，不得为了远程终端再次读取或派生 SHM 数据。

### P0 单帧一次计算与多方直接消费合同

“同一帧”不是某个 widget 的刷新调用，也不是任意一次 33ms timer tick，而是由不可变 generation 组合唯一标识的数据版本。实现必须把源采样、场景派生、SHM 提交和显示消费分开计数，任何一层都不得因为下游消费者数量增加而重复上游工作：

| 阶段 | 唯一键 | 本阶段只允许执行一次 | 唯一输出 | 明确禁止 |
| --- | --- | --- | --- | --- |
| 源采样 | `source_generation` | 从微内核/native 常驻内存取得一组完整 actual/theory/速度/倍率和 source metadata | immutable source view | 坐标控件、刀路、标签、UI 页面或 relay 分别读取 native owner |
| 场景派生 | `source + program + native + active_model + rtcp + wcs + view + fit generation` | 量子/位数、坐标、跟随误差、RTCP gate、active-model 姿态、黄色轨迹、WCS-follow、`world -> screen` 和 dirty bounds | 一份 immutable display scene snapshot，并分配唯一 `scene_generation` | 按图层、widget、页面或 subscriber 重算、重投影、复制为等价 scene |
| SHM 发布 | `scene_generation` | typed ABI 组帧、CRC、SeqLock commit | 一份可原子读取的 SHM frame | 同一 scene 被多个 Publisher、多个 writer 或 timer tick 重复 commit |
| UI 消费 | `scene_generation` | UI shell 完成一次有界 SeqLock/CRC 读取并建立只读 view | 同一个 `V5UiStatusView/V5DisplaySceneView` | 每个 widget 各读一次 SHM、修改 snapshot、重新计算坐标/RTCP/黄色轨迹 |
| 远程输出 | `frame_id/base_frame_id` | 对同一 dirty frame 捕获并准备一份 immutable payload | 所有有效远程连接引用的 prepared payload | 每个连接重新捕获 framebuffer、重新编码、重新组 metadata 或反向读取 SHM |

跟随误差必须作为 SHM 的强类型必备显示字段与坐标一次性同步生成：realtime motion/native owner 对每个 `source_generation` 只允许用同一伺服周期的已受理 commanded 与 actual 计算一次 `actual - commanded`，并把五轴结果锁存在同一 immutable source snapshot；Position display producer 只读取一次、只做一次 count-to-unit/等效角/三位显示投影，并与同代 `mcs/cmd_mcs/unit_per_count` 一起写入 `following_error[5]`。State Publisher 只透传并在同一 CRC/SeqLock 事务提交，板屏、设置页、刀路、relay 和 WinRemote 只能直接读取该字段，不得各自相减或再次换算。只要任一轴的 commanded/actual 不属于同一 source generation、单位比例无效或结果非有限值，本帧跟随误差 valid 必须清除并显示 unavailable，禁止用异步 `mcs-cmd_mcs`、上一帧、PDO 诊断值、UI 文本或默认 0 补成有效。

30Hz 调度只负责请求和消费“当前最新完整帧”。相同 scene key 的 tick 必须直接复用已发布 snapshot，不得为满足固定 30Hz 而重复 transform/project/payload-build；输入 generation 在生产过程中再次前进时，只保留单槽 latest-wins 请求，允许丢弃尚未发布的陈旧显示结果并直接生成最新 key，禁止排队积压多份过期 scene 后补算。只有 `view_generation` 或 `fit_generation` 改变而 native `source_generation` 未变时，必须复用已有 immutable source view，只生成一次新 scene，不能重新采样 native owner。

snapshot 一经发布即不可修改，直到所有本周期消费者释放引用。双页、双缓冲、引用计数、arena 或固定 pool 只能用于生命周期和原子切换，不能保存第二套语义相同的可变 scene。不同 TCP 连接仍需要各自发送网络字节，但只能发送同一 prepared payload，不能据此在板端重新做坐标、轨迹、像素捕获或编码计算。

SHM 读端必须采用有限重试的 SeqLock/CRC 校验。UI reader 在检测到 seq 分裂、CRC 不一致、size/version 不匹配或帧复制失败时，只允许在有限次数内重读；达到上限仍失败时必须返回 stale/degraded/unavailable 显示状态并退出读取循环，不能无限自旋卡住 UI 主线程。C reader、Python reader 和 pre-UI boot-ready 必须在接受 CRC 正确帧前执行同一组合同校验：非零 `writer_identity/source_acquired_mono_ns/source_generation`；display scene 有效时非零 `scene_generation`；`trajectory_count <= 16`、scene `point_count <= 512`、`segment_count <= 48`、`marker_count <= 16`；所有被 valid mask 声明有效的坐标、`unit_per_count`、`following_error`、速度、倍率、CPU 和 scene 数值 finite，且 `unit_per_count > 0`。任一失败整帧拒绝，不能依赖后层 UI 再检查、renderer 截断、valid bit 未设置或生产者通常正确。短暂使用上一帧只允许作为明确标记的视觉 stale 过渡，最多用于坐标、轨迹、转速、进给速度和倍率显示；不得作为动作证明、保存证明、控制准入、参数真源、安全证明、Work Zero/WCS/RTCP/G92 证明或 native gate readback。

reader 打开成功时必须记录当前映射的 device/inode、size/version、非零 `writer_identity` 和最近 generation。普通帧不得每周期 `stat/open/close`；但路径 inode 变化、writer identity 变化、generation 在 writer 已重启后长期不前进、连续读失败或块 size/version 变化时，必须关闭旧 mmap，并在有界退避内重新打开 canonical 路径。writer 若采用 unlink/recreate，必须保证该重连条件可观测；若改为稳定 inode 原地重建，也必须发布新的 writer identity/generation。任何一种方案都必须保证 publisher/init 重启后 UI 无需人工 `reader_close`、页面重建或整机重启即可切换到新 writer，禁止永久读取已被 unlink 的旧映射。

UI 启动首屏不同于运行期 stale 过渡：`v5-ui-relay` 或等价 supervisor 可以在 gated/offscreen 状态提前拉起，但在清屏、发布本地/远程首帧和开放输入前必须等待 `/dev/shm/v3_status_shm` 已存在、SeqLock/CRC/size/version 校验通过、`status_epoch` fresh，且坐标/速度/倍率显示白名单至少有一组 dynamic valid mask。State 中记录的 Position `writer_identity` 必须等于当前 Position writer，且满足 `0 < state.source_generation <= position.current_source_generation`、`0 < state.source_time <= position.current_source_time`；generation/time 必须属于该 writer 并持续有效推进。Position和State独立按30Hz发布，禁止用两块SHM的瞬时坐标数组逐元素完全相等作为同源或首屏准入条件。未达到 lineage 条件时不得先显示空坐标界面；只能保持 UI 未出屏并 fail-closed 记录启动阻塞。运行期短暂 stale 只用于已经有 last-good 帧后的视觉过渡。

Position block、最终 typed SHM 和 UI reader 属于同一个不可拆 ABI 发布域。任何变更 `v5_status_shm.h`、Position block version/size/layout、State writer/native sample、C/Python reader 或 boot-ready 校验的部署，只要 manifest 缺少 Position Publisher 可执行文件及 init、共享 projection/codec、State Publisher 二进制及 init、UI shell/C reader、Python reader/boot-ready、UI/relay 及 init 任一域，canonical installer 必须在复制文件前 fail-closed；禁止 UI-only、State-only、先换 reader 后换 writer或先换 writer 后换 reader。部署时先停止 UI/relay、State Publisher、WCS Publisher 和 Position Publisher，再一次性安装完整 bundle；安装完成后的进程启动和正式画面交接分开处理：

1. canonical `backend_data_ready` 成立后立即把 Position/WCS/State Publisher 变为 runnable；State进程允许与Position同批拉起，但在当前Position块未valid前只能等待或发布明确invalid/degraded状态，不得发布伪valid或旧值。UI/relay supervisor可以更早进入gated/offscreen准备态，不得接管framebuffer、发布远程帧或开放输入。
2. 读回 Position `magic/version/size` 与当前 ABI 一致、`writer_identity != 0`、SeqLock 为非零偶数、`source_acquired_mono_ns/source_generation != 0`、CRC 正确且 source fresh；再读回 State `magic/version/size`、typed valid bits、scene generation、SeqLock/CRC/freshness，并以Position writer identity与source generation lineage证明State消费的是当前writer。
3. 唯一最终barrier同时取得上述lineage、`motion_backend_ready`、全部页面`cache_ready`和relay生命周期actual后，才允许同一事务发布本地/远程正式首帧并开放输入；任一级失败都保持输出和输入gated，不得使用旧SHM、默认值或旧进程继续显示。

完整 SD/rootfs 冷启动也必须保持相同的依赖关系和最终barrier，不要求用init脚本编号强制全链串行。运行核验必须确认不存在旧 version/size 的 Position 或 State writer 进程，并从 `/remote/info` 与 fresh framebuffer 证明坐标和 CPU 已消费 fresh 同代投影而不是 `--`；该显示核验不授权 Home、Start 或任何运动。

后台 Action Dispatcher、Broker 服务、提权 gate、native gate wrapper 和参数/动作 owner 的源程序不得直接或间接读取 `/dev/shm/v3_status_shm` 或其历史 typed SHM 结构；不得 include、mmap、包装 helper、读取 UI stale frame，或通过 UI status view、缓存对象、Broker result 把 SHM 投影转成动作输入。SHM 只允许前台 UI 显示 reader 单向消费，且消费结果只进入显示模型。

凡是需要动作结果证明或安全/参数同源事实的路径，必须通过 UDS、登记 native memory block 或等价强类型 native readback 直连微内核/native owner。例如 Work Zero、WCS 写入、G92/RTCP 切换、坐标设零成功确认和参数运行态生效证明，命令下发后必须通过 `EMC_STAT_GET` 或等价强类型查询从微内核/native 内存/API 回读 offsets、modal、state、generation 和 readback token；不得用 SHM fresh/stale 帧、UI 显示值、`status_epoch`、`/run` result 或旧缓存补证明。

硬实时安全主权属于 LinuxCNC realtime、HAL、EtherCAT 主站、驱动和微内核 native owner。上层 UDS readback 与 UI 显示只作为离散请求反馈和状态展示，不作为实时安全互锁依据；使能、限位、插补、急停、servo 周期和硬件安全互锁必须在底层周期控制内独立闭合并 fail-closed。

State Publisher 和 WCS position publisher 只承担显示投影发布，不是硬实时控制 owner。Position publisher 必须在采样和首次写 canonical Position 块前取得全生命周期内核独占锁；Position 块头部保留字段承载本进程非零 `writer_identity` 并纳入 CRC，使读端和诊断能够核对连续帧是否来自同一唯一 writer。PIDFILE 只作诊断，不得裁决 singleton；PIDFILE 丢失、陈旧、PID 复用或 orphan 状态下也不得产生第二 writer。它们不得使用 `SCHED_FIFO`，其 affinity/cpuset 必须明确排除 CPU0，所触发的普通 IRQ、workqueue 和 housekeeping 也不得落入 CPU0；不能依赖调度器“通常会放在 CPU1”。即使 CPU1 达到 100% 或 UI 完全卡死，也只能通过常驻内存、限时采样、poll/reconnect、合并或丢弃陈旧显示帧、degraded/unavailable/stale 发布和 UI 读端有限重试来退化显示，不得迁入 CPU0、改变运动 IRQ、降低 servo 周期或阻塞运动链。

当前 Position publisher 的运行实现必须是独立 native C 可执行文件，保持本节规定的独立 30Hz 生命周期、256 字节 Position v3 ABI、100ms heartbeat、200ms bus status 节奏和现有旋转投影/无状态三位四舍五入语义；不得把 Position writer 合入 State 进程，也不得并存 Python writer、Python fallback、双启动分支或同一 canonical SHM 的第二 writer。C 实现完成等价 smoke、ARM 构建、原子 bundle 部署和板端 lineage/readback 后，旧 Python publisher 源文件、安装项、启动 argv、部署识别分支及仅服务于旧实现的测试必须物理删除；仍被 WCS、boot-ready 或诊断读取链实际消费的 Python codec/reader 不是弃用代码，不得连带删除。

Position C 实现内部固定为三层且保持单进程：`v5_position_status_sampler.*` 只负责常驻 HAL library/pin 绑定、同代原始 source/checkpoint 与 BUS actual 采样；`v5_position_status_publisher_core.*` 只负责旋转投影、无状态三位显示桶、Position v3 ABI block 构建和显示相等判定；`v5_position_status_publisher.c` 只负责生命周期独占锁、采样节拍、heartbeat、Position/BUS mmap、SeqLock/CRC commit 和进程退出。拆分不允许新增进程、第二 writer、线程、fallback或改变任何 ABI/version/size、pin名、采样顺序、generation、CRC、节拍和降级语义。

当前 SHM 源码位置：ABI 在 `board/app/include/v5_status_shm.h`；UI reader/view 在 `board/app/src/v5_status_shm_reader.c`、`v5_ui_status_view.*`；native 采样、publisher 生命周期和 mmap/write 分别在 `board/services/state_publisher/v5_native_sample.*`、`v5_state_publisher*.c`、`v5_status_shm_mmap.*`、`v5_status_shm_writer.c`。非 SHM native 状态清单在 `board/services/microkernel/v5_native_status_api.*`，其中登记项必须保持 `export_to_shm=0`；不得把该合同误写成 SHM writer 所在目录的第二份状态白名单。

### 当前坐标显示方法（与现行源码一致）

当前实现只有 State/Position display projection producer 能生成坐标显示值，处理顺序固定如下：

1. 从同一 `source_generation` 取得 native/HAL/驱动实际机械坐标和 commanded/theory 坐标；原始值继续保持 `0.0001 unit/count` 运动量子，不在这里修改运动值。
2. 旋转轴先由 native count-domain 投影得到等效角；随后所有轴都以 `DISPLAY_COORDINATE_SCALE=1000` 转成三位显示桶。正数按 `floor(value*1000 + 0.5)`、负数按 `ceil(value*1000 - 0.5)`，即第四位四舍五入，恰好位于半格时取远离零的方向，而不是向零截断；结果再除以 `1000`，`-0.000` 统一显示为 `0.000`。
3. 当前 active model 的两个旋转显示 slot（AC 为 A/C，BC 为 B/C）按 `360000` 个千分之一度取模；因此四舍五入得到 `360.000`、`-0.000` 或其它整圈等效值时统一显示 `0.000`，`0/360` 的桶距离按环形最短距离计算。
4. 每个有效 generation 都只根据该 generation 的 source 值独立生成结果并立即提交；不得读取上一帧坐标来决定本帧结果，不得因 generation 重复、跳号或倒退而复用、延迟或修正坐标。
5. producer 对机械坐标和 commanded/theory 坐标各做一次无状态投影，并从同一 immutable snapshot 派生加工坐标、跟随误差、模型姿态和刀路所需显示结果。State Publisher 只提交一次 SHM；主页面、设置页、刀路、标签、relay 和 Windows 远程端只读这一份结果，不得再次四舍五入、取模、抗抖、平滑、保持或从文本反算。

坐标显示链禁止保留任何 `stable/candidate/hysteresis/debounce/hold` 等状态或同义实现。source invalid/degraded 时仍按 SHM freshness 合同显示 stale/degraded/unavailable，但 last-good 帧容错只能表达整帧通信状态，不能修改 fresh 帧内的坐标数值，也不能充当坐标抗抖。

`source_acquired_mono_ns` 必须在 Position Publisher 成功取得一组完整 native actual/commanded/速度/倍率样本时产生；`source_generation` 只在取得新的完整 source sample 时前进。Position block、State 投影、SHM writer 和 UI reader 必须原样透传这两个值。State Publisher 的 `status_epoch`、重打包时间、heartbeat 或重写同一 payload 都不得修改 source metadata；旧 Position sample 被新 State 帧反复发布时，UI 必须按 source age/generation 进入 stale/degraded，不能被新的 `status_epoch` 洗成 fresh。
State Publisher/WCS position publisher 必须以常驻进程进入 poll/reconnect 循环；不得在启动后用 `linuxcnc.stat()`、坐标 ready、WCS ready、CMD_MCS ready 或其它 preflight 稳定探测做无限等待而不发布。LinuxCNC/HAL/native 暂不可用时只允许发布 invalid/degraded 帧并继续重连；一旦 native actual 可读，必须立即写入 fresh position block 和 SHM 坐标投影。

## P0 远程投屏解耦边界

产品节拍决策（2026-07-21 已确认）：为降低 CPU1 占用，WinRemote 网络镜像正式采用 `100ms/10Hz latest-wins` 展示，属于可接受的产品取舍，不按 30Hz 回退或缺陷处理。该降频只发生在 `remote framebuffer/FIFO -> relay -> Windows` 非运动出口；板端物理 LCD、State/Position Publisher、SHM freshness、控制命令、动作证明、安全门禁和急停直通链仍保持各自既定节拍，其中板端坐标/轨迹动态显示目标仍为 30Hz。后续审查不得要求把 WinRemote 恢复为 33ms/30Hz，也不得把 10Hz 扩散到板端本地显示或控制链；只有用户再次明确变更需求时才允许修改这一产品节拍。

当前源码 owner 已按本地投影、输入 IPC、网络 relay 和 Windows 客户端分开：UI 侧 mmap/FIFO 写入在 `board/app/src/v5_lvgl_remote_display.*`，LVGL 指针设备在 `v5_lvgl_remote_input.*`，`remote_input` FIFO 解析在 `v5_remote_input_ipc.*`，板端 CPU/显示指标在 `v5_remote_metrics.*`；独立网络进程在 `board/services/ui/v5_remote_ui_relay.py`，协议、合同、状态和公共支持分别在同目录的 `v5_remote_ui_protocol.py`、`v5_remote_ui_contract.py`、`v5_remote_ui_state.py`、`v5_remote_ui_support.py`。Windows 端界面在 `8ax-win/src/8ax.WinRemote/MainWindow.xaml`，会话、dirty/full frame、指针、程序和 OTA 已分别拆到同目录 `MainWindow.RelaySession.cs`、`MainWindow.RelayFrame.cs`、`MainWindow.Pointer.cs`、`MainWindow.Programs.cs`、`MainWindow.OtaUpgrade.cs`；`MainWindow.xaml.cs` 只保留窗口总入口。后续修改不得把这些职责重新并回 UI 主页面、relay 单文件或 Windows 主窗口总入口，也不得创建第二套 relay。

UI 本地显示内部继续只有一份 canonical BGRA frame：`v5_lvgl_remote_display_capture.*` 只拥有物理 framebuffer mode/stride/bpp/mmap、正式首帧 claim、BGRA 到16/24/32-bit物理像素转换，以及不接触frame内容的dirty矩形合并算法；`v5_lvgl_remote_display_delta.*` 只负责在候选 dirty 范围内比较 canonical frame 与上一已发布 remote framebuffer、复制真实变化 tile 并生成有界 rect；`v5_lvgl_remote_display.c` 保留 LVGL flush、canonical frame/page cache、dirty队列状态、remote framebuffer mmap与FIFO通知。capture/delta 不得创建 remote mmap/FIFO、维护 frame id 或复制第三份完整画面；IPC桥不得重新实现物理像素格式转换、像素差分或dirty合并算法。拆分不得改变颜色通道、frame/base identity、cache、首帧交接、blackout和远程协议。

UI、remote relay、镜像编码/发送和输入适配全部属于 CPU1 非运动域。CPU1 过载时，relay 必须优先合并 dirty 区域、丢弃陈旧帧并对网络施加背压；允许远程画面降帧、延迟甚至暂时卡住，但不得把 relay/UI/Publisher 或其非运动 IRQ/workqueue 迁到 CPU0，也不得让帧锁、socket、日志或输入队列反向等待运动实时链。最终边界不是“UI 尽量少占 CPU”，而是“UI 即使卡死，运动仍连续”。

远程投屏属于 UI 显示投影的网络出口，不是 UI 主循环的职责。UI 进程不得监听 `18080` 或任何 TCP 远程投屏端口，不得解析 HTTP/WebSocket，不得在 UI 主线程里执行 `accept`、`recv`、网络发送、全屏网络帧拷贝或 peer allow-list 判断。远程连接数量、网络缓冲区就绪状态、上位机重连频率和 WebSocket 协议开销不得影响 UI 主循环调度；UI 主循环底部必须保留 10ms sleep/yield 保底，防止远程投屏或其它非运动辅助路径把 CPU1 拖入 busy loop。

UI 进程只允许在 `/run/8ax_v5_product_ui/` 创建远程显示本地 IPC：`remote_framebuffer.bgra` 是 BGRA32 mmap 帧缓冲，大小为当前 UI 宽度乘高度乘 4；`remote_dirty` 是 FIFO dirty rect 通知。LVGL invalidated rect 只代表“可能变化”的候选区域，不能直接当成远程有效像素量；物理 LCD 仍对每次本地 flush 的 invalidated rect 立即刷新并保持 30Hz，不得被远程节流或像素筛选跳过。远程侧须跨本地帧累计候选区域，每到 100ms 单调时钟窗口才在同一个 framebuffer 写锁内，以固定 `16x16` tile 比较当前 canonical frame 与上一已发布 remote framebuffer，只复制并通知最终真实发生像素变化的 tile；完全相同的 tile 不得进入 mmap copy、dirty FIFO 或网络 payload，中间 30Hz 画面不得逐帧复制到 remote mmap。真实变化 tile 必须有界合并到协议已登记的最多 64 个 rect，禁止因 16 rect 容量不足把稀疏变化重新膨胀成数个大包围框。只有至少一个像素变化时才推进 `frame_id`，不得用阈值、降色深、压缩失真、降采样或冻结轨迹换取 CPU。页面 cache blit 与首帧仍可明确发布完整帧。FIFO 通知只表达显示投影帧序和 dirty 区域，不承载控制、动作证明、状态真源、诊断 JSON 或 fallback 数据。没有 relay reader 时，UI 只能非阻塞丢弃通知或等待下一次 dirty，不得阻塞 UI 主循环。

独立 Python relay 是唯一允许监听 `18080` 的远程显示/输入网络进程。它读取 `remote_dirty`，从 `remote_framebuffer.bgra` 切出 dirty rect，对外提供 `/remote/info`、`/remote/frame/full`、`WS /remote/stream` 和 `WS /remote/input`；bind 地址和 peer CIDR allow-list 仍按 `REQ-SERVICE-RUNTIME-SANDBOX` 执行，但CIDR只能作为第二层收口，不能替代TLS身份和应用认证。18080只允许TLS 1.2及以上的HTTPS/WSS，relay启动时必须从`/etc/6x-cnc/remote-relay/server-cert.pem`与`server-key.pem`加载当前设备独立证书身份，并从`clients.json`一次性装入RAM中的客户端ID、256位随机secret和允许scope；证书、私钥或授权文件缺失、权限过宽、格式错误、设备ID不一致时服务fail-closed，不得启动明文listener。板端私钥、客户端secret和Windows授权配置不得进入Git、VPS、URL、日志、截图或通用镜像；Windows授权配置只从`D:\授权私钥`中的设备专用文件显式选择，WinRemote必须同时严格固定证书SHA256 fingerprint和证书中的6位设备ID，不得使用跳过证书校验、自动接受新证书或HTTP/WS fallback。

应用认证固定为`v5.remote.auth.v1`两步协议：客户端先经TLS请求一次性32字节随机challenge，再用客户端secret对协议版本、client_id、challenge_id、nonce、device_id和排序后的requested scopes做HMAC-SHA256；challenge只在RAM保存30秒、绑定client_id并且仅可消费一次。验证成功后relay签发32字节随机session token，session只在RAM保存、最长10分钟、绑定client_id/peer/scopes；每个HTTP请求和WebSocket握手都必须通过`Authorization: V5Session <token>`提交，URL/query/body中的token一律拒绝。scope固定映射：`viewer`只允许`/remote/info`、`/remote/frame/full`和`/remote/stream`；`diagnostics`只允许`/remote/diagnostics`；`program_manager`只允许程序list/file/upload/delete；`operator`只允许`/remote/input`；`ota_admin`只允许OTA入口。缺认证返回401，认证有效但缺scope返回403；不能因input启用、非空`session_id`、peer IP、UI ready或已有frame自动授予控制。

relay启动阶段只允许loopback客户端访问独立的`/local/health`最小健康接口；该接口仍使用同一TLS设备证书，只返回进程/dirty-reader是否就绪，不返回frame、诊断、程序、输入或OTA内容，也不能从非loopback访问。正式`/remote/*`无匿名health例外。客户端认证和scope在WebSocket升级前完成；input socket升级后把已验证session/scopes驻留连接内，急停等每个已认证输入事件不得再访问VPS、磁盘、clients.json、挑战服务或普通业务门禁。session过期或身份变化必须断开并由WinRemote重新认证，不能在既有连接上静默续期。

为降低运动中 CPU1 和网络占用，relay 必须把不超过 100ms 窗口内连续的 30Hz 本地 dirty 通知 latest-wins 合并为一个 10Hz coalesced stream frame 发送；该 frame 优先保留多个原始 dirty rect，避免远隔小区域被单个外接矩形放大，只有 rect 数异常多或超出协议/CPU 边界时才允许退化为 bounded union rect。合并后的 stream frame 以客户端已持有帧为 `base_frame_id`，以窗口内最新 UI 投影帧为 `frame_id`，不得因为合并导致 full-frame repair。远程输入经 relay 解析 WebSocket 后只能写入本地输入 FIFO，由 UI 在自己的 LVGL 输入驱动里消费；不得把 TCP fd、HTTP request 或 WebSocket parser 传回 UI 进程。

relay 对每个合并后的 `frame_id` 只允许从 framebuffer 捕获并组装一份不可变 prepared payload；该捕获必须在同一共享帧锁内由板端 C native packer 对只读 mmap 执行一次有界 rect/row 遍历，输出到一块 Python owned 缓冲，Python 只负责 rect 归一化、调度、metadata 与网络发送，不得在 10Hz 网络热路径按行创建 `memoryview`、执行 `join` 或保留解锁后仍引用 framebuffer 的 view。native packer 缺失、ABI 不匹配、越界或输出长度不符时 relay 必须 fail-closed，不得保留 Python 像素打包 fallback。所有有效 stream handler 只能引用这一份 payload，不得各自重复 mmap 切片、像素拷贝、编码或 JSON metadata 组装。stream handler 必须同时消费客户端 close/ping/pong 与 TCP 断线，连接关闭后立即 unsubscribe，旧 handler 不得继续发送；禁止用 peer IP 粗暴互斥，因为同一 PC/NAT 后可能存在多个真实终端并会形成抢占重连风暴。不同独立 TCP 终端仍需分别传输网络字节，但不得因此在板端重做画面计算和 payload 捕获；同一 PC/网关后的多个本地观察者优先由 PC/网关使用一个上游流扇出。

运动态远程验收必须同时记录 UI/relay 进程 CPU、`dirty_events`、`dirty_payload_native_frames`、`dirty_payload_bytes`、`dirty_payload_shared_build_ms`、relay 进程 ticks 和 CPU1 墙钟占用；禁止只报静止态。必须证明物理 LCD 仍按本地 30Hz 更新，而 remote mmap/FIFO 与 relay payload build 都约束在 10Hz；若 LVGL 候选 dirty 面积大但实际像素变化稀疏，发布后的平均 payload 必须以真实变化 tile 为准显著小于候选矩形原始 BGRA 字节量，且实际输出 rect 数可使用完整 64 项协议容量，不得长期卡在 16 项并膨胀为大包围框。像素筛选前后必须逐字节还原成同一 BGRA frame，并覆盖 RTCP ON/OFF、AC/BC、512 点黄色轨迹、坐标数字、红绿点和无残影；30Hz 本地连续帧经 UI 100ms canonical snapshot 与 relay latest-wins 后，远程必须稳定输出 10Hz 最新完整视觉状态且不得损坏 base/frame 链。任何像素不一致都必须 fail closed，不能退回未筛选的大矩形常态发送。

Windows 端远程画面是产品常驻观察/操作链路，默认必须保持可用并允许长连接一直存在。正常会话初始化只消费 `WS /remote/stream` 内的一次初始 full-frame；Windows 客户端不得先请求 `/remote/frame/full` 再连接 stream，HTTP full-frame 只保留给明确 repair 或独立诊断。CPU 占用整改不得把断开 Windows 远程端、停止 `v5-ui-relay`、关闭 `/remote/stream`、降级为手动刷新或删除远程输入作为解决方案；只能通过 UI dirty 区域收敛、relay mmap/低拷贝、100ms latest-wins 合并窗口、大 dirty rect 限频、异常客户端 fail-closed 清理、publisher 事件化和 CPU affinity 重新分配来降低负载。验收必须在 Windows 远程画面保持连接、10Hz 远程展示、远程输入可用、`/remote/info` 和 `WS /remote/stream` 正常的状态下完成；稳态不得靠高频 full-frame polling 维持画面。

安全验收必须覆盖同一板端实例的正负路径：TLS握手后正确fingerprint、正确6位device_id、正确challenge HMAC和`viewer+operator` scope可由WinRemote取得画面并完成一个非运动点击；`program_manager`可在同一canonical G-code目录完成list/upload/read/delete；明文HTTP/WS、未知client_id、错误secret/MAC、challenge重放、challenge/session过期、token放URL、viewer访问input/program/diagnostics/OTA、证书fingerprint或device_id不匹配均必须失败且不写FIFO/程序目录。验收结束必须关闭远程输入连接并保持ESTOP/Machine Off；负向测试不得用真实运动证明。

CPU 归因必须区分“远程长连接存在”和“远程显示有实际负载”：`stream_sessions > 0` 只证明远程端打开，不是高 CPU 根因；只有在同一采样窗口内 `dirty_events`、`dirty_rect_frames`、`dirty_payload_bytes`、`full_frame_requests`、`stream_repair_full_frames` 或异常 polling 计数持续增长，才能把远程显示链路列为当前 CPU 负载来源。产品显示百分比必须使用上表同代 cpuidle 墙钟快照；`/proc/stat` 在未启用 IRQ time accounting 的 PREEMPT_RT 内核上可能出现 CPU 栏位时间与 idle 重叠，只能作为 cpuidle 缺失时的降级值或结合线程 ticks/IRQ/softirq 的诊断明细，不能继续作为正常 CPU0 显示真值。后续远程显示代码必须保留这些 diagnostics counter，并用同一墙钟窗口的 cpuidle、线程 CPU ticks 和 relay metrics 增量一起判断；不得用单次截图、单次连接状态、`ps %CPU` 长周期均值或 SSH 诊断扰动替代因果证据。

- 扫描证据：产品路径不存在 `v3_state_shm_writer.py --input-json`、RTCP SHM 状态块、settings projection/config cache SHM payload、status/ready/perf JSON 周期发布。
- 读端证据：C SHM reader 对 seq/CRC/size/version 错误必须有限重试并返回 stale/degraded/unavailable；测试或静态检查必须证明不会无限自旋，且 stale 帧不被控制、保存、参数、WCS/G92/RTCP、native gate 或安全路径消费。测试还必须主动 unlink/recreate SHM 或重启 publisher，证明 reader 按 device/inode/writer identity/generation 自动关闭旧 mmap 并连接新 writer，且普通帧没有每周期 `stat/open/close`。
