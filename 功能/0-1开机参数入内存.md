最高口径：LinuxCNC/微内核/native 到 UI 的轻量数据，以及 LinuxCNC/微内核自身运行所需闭包，必须在开机阶段一次性进入常驻 RAM。运行期 UI、Broker、动作脚本和显示 provider 不得反复读取硬盘、JSON、INI、var、tool 文件、脚本快照或 result 文件来补当前状态。

这里的“全读进内存”指 LinuxCNC/微内核运行闭包、UI 运行闭包和所有产品自写运行代码闭包，不是把 SHM 扩展成通用状态池，也不是把 UI 变成控制真源。

```text
硬盘/闪存持久 owner
-> 开机一次性读取 LinuxCNC/微内核运行闭包、UI 运行闭包、产品自写运行闭包和参数 owner 映射到 RAM
-> LinuxCNC/微内核/native 常驻内存与原生 readback
-> display projection producer 按同一 source generation 一次性生成坐标、每轴显示量子/位数、跟随误差和 bounded toolpath scene
-> State Publisher 只组帧并 commit 一份不可变 typed SHM 显示快照
-> 坐标面板、刀路、标签、跟随误差、本地画面和远程 relay 只读同一快照并各取所需
```

硬盘/闪存只允许作为开机装载、设置保存落盘、重启/重载前的持久 owner；运行期控制、运动准入、状态 readback、UI 显示 provider 和动作证明必须来自 LinuxCNC/微内核/native 常驻内存或原生 API。

开机常驻内存闭包必须至少包含四类：

1. LinuxCNC/微内核运行闭包：executable、共享库、RTAPI/实时模块、HAL component、kinematics component、INI/HAL 配置、`PARAMETER_FILE`、`tool.tbl`、G53/RTCP 几何、WCS/G92 初始 readback、驱动映射和已登记 native gate 表。
2. 参数 owner 映射闭包：参数表字段、唯一 owner、唯一 writer、源位置回读字段、generation/readback token、drive-only 例外字段和删除旧路径条件。
3. UI 显示投影闭包：只允许同一 native source generation 派生的坐标、每轴显示量子/位数、跟随误差、bounded 轨迹/scene、主轴转速、进给速度、主轴倍率、进给倍率进入 `/dev/shm/v3_status_shm`；原始 RTCP/WCS/G53/tool/safety actual 仍由 native owner 裁决，不得因最终显示 scene 入 SHM 而变成 SHM 真值。当前模态文字直连微内核/native modal/tool 状态块或等价原生 API；历史 `runtime_modal_text` 和 `V5_STATUS_VALID_MODAL` 必须从 SHM ABI 删除，不得作为 padding、诊断文本、兼容字段或 fallback 保留。
4. 产品自写运行闭包：C UI、Command Broker、非控制/诊断/设置域 action、provider、native gate wrapper、linuxcncrsh 常驻客户端状态、单 FIFO 命令队列、连接健康状态、运行期需要的资源索引、Python 字节码和动作登记表必须在启动阶段加载到 RAM。Home、Work Zero、Jog、Start、RTCP/WCS 等控制动作不得经 Python product action 中转，按 `功能/0-3native_helper白名单与验收说明.md` 直达 native gate。运行期 UDS 调用不得临时从硬盘发现脚本、动态导入 action、扫描目录补入口、读取 UI 资源文件补状态、按命令重建 linuxcncrsh 短连接或靠 fallback wrapper 恢复旧路径。

设置页、设置驱动、从站扫描和参数编辑属于受控设置阶段，不属于运动控制、状态显示或 Start/Resume 热路径。受控设置阶段允许按字段唯一 owner 读写硬盘/闪存持久文件，例如字段保存写回、扫描结果落地、候选列表刷新、drive-only 证据保存和受控重启/重载准备；写入后必须刷新或重载对应 RAM/native/provider 快照，并以同 owner 源位置回读或受控重载后的 native/provider readback 作为结果依据。凡修改 LinuxCNC runtime INI、`PARAMETER_FILE`、`tool.tbl`、自建/驱动参数表、drive profile map 或等价参数文件，必须先从 Windows source truth 生成独立参数部署产物，并通过 canonical deploy/apply 事务写入板端对应持久 owner，再用板端 hash/源位置回读确认一致后，才允许继续整包部署、受控重启、UI/operator 验证或运动闭环；禁止用 SSH/SCP/临时命令直接编辑板端文件，也不得把参数文件夹带在后续整包部署里当作已完成参数部署。

BUS 启动必须把 active model/active slot 与最终 axis→slave 表形成的 committed mapping actual，和每个物理从站的 Home 零点 owner ready 分开裁决。映射表完整、唯一、同代且 router 已原子接收时，mapping generation/active mask/axis/slave table 必须立即发布给 Position 的 BUS 语义状态；不得因为自定义 `zero_model` 缺失、失配或尚未装载而撤销这张映射或持续报告 `native_bus_status_mapping_not_committed`。Home 零点 owner 仍逐轴裁决：某个已映射生效物理从站从未保存过自定义 `zero_model` 时，使用显式缺省 `zero_counts=0` 和该逻辑轴 active `SCALE`，该轴 Home config ready 为 true；只有已经存在但字段不完整、数值不一致、从站失配或同从站重复的零位证据才让对应 Home config ready 为 false 并禁止 Home。开机 `all_homed=false` 仍继续阻断 Start/Jog 等运动；不得用 mapping committed actual 冒充已经完成 Home。反之，映射缺失、重复、越界、模型不符或 table/router generation 不一致时，BUS 状态必须发布由同一 writer/sequence/CRC/freshness 证明的结构化 unavailable 帧，Position/State/UI/relay 仍启动到可修复界面，native 运动入口继续拒绝使能、Home、Start 和 drive target。

运行期允许触发硬盘/闪存的场景只有：受控设置阶段读写唯一持久 owner、Program Open/Load 读取操作员选择的原始程序并生成只读 resident program identity/内存索引/显示模型、事件触发且限频的日志/诊断写入、受控重启或重载。Program Open 不生成执行用运行副本。日志/诊断不得进入 Start/Resume、State Publisher steady-state、UI 稳定显示或正常空闲循环的周期落盘路径。除此之外，UI 稳定显示、控制动作、运动准入、保存成功、参数稳定显示和安全判断都不得读硬盘兜底；Start/Resume 等热路径不得再读程序文件、脚本、配置或资源。

G-code 打开和投喂边界：用户在 Program Open/Load 打开的原始 G-code 文件是什么内容，Start/Resume 就必须把同一个原始文件身份投喂给 LinuxCNC/微内核；`runtime_program_path`、`Set Open` 路径、resident `source_path` 和 `source_sha256` 必须全部对应操作员打开的原始 G-code 文件，不得改成过滤后的临时文件、sidecar、运行副本、等效角改写文件或 UI 预览解析产物。Program Open/Load 允许建立只读 resident runtime identity、hash、line count、预览点和 first-point metadata，但这些 metadata 只能服务显示、身份校验和 native gate 请求，不能改变投喂给微内核的 G-code 文本、行顺序、A/B/C 等旋转轴 word、M-code 或模态。Program Open 可在最小身份登记后返回主页面，但只有 resident program model、全部可见元素组成的首个完整 scene 和 screen fit 都完成，并由唯一 preparation owner 原子发布同一 program generation 的 `program_scene_ready=true` 与非零 `scene_generation/fit_generation` 后，程序才算“可 Start”；Start/Resume 只读该 resident token，不得点击后补建。

G-code active model 对应旋转轴的一圈内等效角分类属于 `功能/3-3旋转轴等效角目标与解决方案.md` 的微内核/native rotary owner；AC 模型下的 `A0`/`C0` 只是示例，BC 模型和后续模型按 active model 对应轴由同一个 native owner 判定。Program Open/Load 可以在 resident runtime metadata 中记录 native parser 需要的只读身份和上下文；Start/Resume 热路径不得由 UI、Broker、Python action、刀路显示层或运行副本重写器读取/重写源 G-code 来实现等效零。`G90/G53 A0 C0` 这类已由 native 判定为非连续累计段的一圈内相位/回零目标，必须由 native/motmod/microkernel 走最近等效角或最近 `k*360` 等效零；连续加工段、多圈累计目标和语义不明目标仍交由 native `continuous_delta` / `ambiguous_block` 判定。

设置页右上角默认状态是 `返回主页`：本次进入设置页后尚未产生任何已写入唯一 owner 并完成源位置回读的有效参数变更，也没有动作 owner 回读出的 `restart_required/restart_deferred` 时，点击只执行本地页面导航，不触发 apply 或 restart。任一字段提交完成 owner 写入/readback，或设置动作成功并由 owner 明确返回需要重启后，同一按钮立即、单向切换为 `保存并重启`，本次设置会话内不得再退回 `返回主页`。

`保存并重启` 是“确认最终持久 owner 已保存 -> 开发板断电后重新上电等价 clean restart -> 新启动链统一设置驱动并回读”的单一事务，不是普通页面返回、局部服务 restart、provider reload 或只重启 UI/Broker。字段提交和 `设置驱动` 预检在按钮切换前只完成持久 owner 写入、源位置回读和 restart pending；旧进程不得发送电子齿轮/mode SDO，也不得读取当前 HAL 参数宣称新比例已生效。handoff 停止 UI、Broker、Action Dispatcher、State Publisher、LinuxCNC/微内核运行服务、HAL/EtherCAT/drive provider 等运行闭包，清理或重建 `/run` 运行态、UDS socket、SHM 显示投影、native 状态块、resident snapshot、命令队列和进程内 cache，然后按开机链路从唯一持久 owner 重新装载到 RAM。新启动链在 Machine Off 下按最终 active model/status slots/axis→slave 映射一次读取 `[JOINT_n] SCALE`、`[AXIS_*] MOTOR_REV/LOAD_REV/PITCH` 和 profile/final bit；先读驱动 actual，一致则跳过写入，不一致才写电子齿轮/CSP并整批 fresh readback。任一失败保持 Machine Off并不得交接运动输入。后台已经返回 `SETTINGS_SAVE_RESTART_BOARD_REBOOT_SCHEDULED` 或等价“系统级重启已受理”终态后，操作员点击结果弹窗 `关闭` 必须在该次 `LV_EVENT_RELEASED` 内把物理 framebuffer 和远程 canonical frame 原子提交为纯黑，并立即停止旧 UI 的后续帧发布，直到新开机链路接管。

最终统一读取时点固定在完整重启后的新启动链，不在 `保存并重启` 的旧进程最后准备阶段。旧进程只提交 reboot handoff；新进程丢弃旧 resident 快照，重新从最终持久 owner 一次读取 active model、active axes/status slots、axis→slave 表、每轴 INI 比例、profile 和 final bit，再按该身份装载 HAL、比较/必要时写物理驱动并整批 fresh readback。fresh batch开始前必须记录monotonic起点；完成后由启动owner把本代native lifecycle generation、mapping generation、五轴status-slot mask、readback count、起点和transaction SHA256提交给唯一native readiness owner。只有native原样回读并发布`drive_verified=1`后，才允许同代`motion_backend_ready=1`；generation失效、mapping/数量/掩码/时间/digest不一致或Machine Off未确认时必须保持`safety_force`、拒绝取消急停并禁止运动。普通 `设置驱动` 只可生成待重启预检结果，同时使当前backend lifecycle generation内的drive attestation失效，不能留下可跳过本次读取的 actual。LinuxCNC/HAL/native/drive provider 不一致时保持 Machine Off，不能自动选取另一套比例或映射。

`SETTINGS_SAVE_RESTART_BOARD_REBOOT_SCHEDULED` 的准确含义是 clean-restart 脚本已准备完成、action owner 正等待该结果窗绑定的 exact `run_id` 提交确认；它不是“已经开始停服务”。在 `关闭` 提交前不得停止 UI、远程画面 relay 或 Action Dispatcher，也不得调用 reboot。操作员释放 `关闭` 时，UI 必须先向 action owner 提交同一 `run_id`；只有收到 `SETTINGS_SAVE_RESTART_COMMIT_ACK` 后，才在同一次释放回调内纯黑并抑制旧帧。action owner 必须给黑帧发布保留至少 0.5 秒交接窗口，随后才停止运行闭包并系统级重启；提交未受理、run_id 不一致或重复到其它 run 时不得黑屏或重启，弹窗必须保留并显示准确原因。

## P0 CPU 热路径要求：慢投影只能缓存后发布

CPU 调度最高要求：CPU0 是运动实时专用域，不是“普通任务尽量少用”的优先核。凡是正在运行时一旦延迟就会造成 servo 掉周期、轨迹队列耗尽、速度突变、总线失步或安全链失效的任务，都是运动必需项，必须进入 CPU0 的显式闭集：LinuxCNC/RTAPI realtime servo、motmod/trajectory realtime、实时 kinematics、随 servo 周期执行的 HAL realtime function、EtherCAT cyclic/master kernel thread及其现场总线实时 IRQ、必须在同一截止期内完成的 realtime safety/limit/I/O latch，以及不能通过 Start 前预装消除、且有明确有界 deadline 的最小 lookahead/trajectory queue feed 线程。CPU0 准入按线程和实时职责裁决，不能按整个进程名称笼统放行；进入 CPU0 的线程不得夹带文件 I/O、日志、UI、网络或普通诊断。

CPU1 是全部非运动任务和系统 housekeeping 域：Program Open、兼容翻译、Start 前 G-code 文件读取/解析/索引/预装、UI/LVGL、Broker、Command Gate用户态传输、State Publisher、WCS publisher、provider/action脚本、远程镜像/relay、framebuffer辅助、日志、文件I/O、网络协议、SSH、诊断、自动化、部署/构建辅助、普通timer、RCU callback、非运动softirq/workqueue和管理网口/display等非运动IRQ都必须排除CPU0。Start 前能够完成的工作必须提前完成并驻留/预缓冲；Start 后持续运动不得以 CPU1 上的 UI、文件解析、普通 LinuxCNC task/interpreter 或其它用户态服务继续获得调度为条件。若现有实现仍需 LinuxCNC task/interpreter 在运行中持续补给轨迹，必须把该依赖改为足量预缓冲，或把不含文件/日志/控制杂务的最小有界 queue-feed 职责拆入 CPU0 运动闭集。必须通过启动期 affinity/cpuset、调度策略、IRQ `smp_affinity_list`、RCU/workqueue offload和运行期readback形成硬隔离，不允许依赖调度器“通常会放在CPU1”的偶然结果，也不允许在CPU1繁忙时把普通任务回退或漂移到CPU0。

当前双核 ARM 产品禁止用 `isolcpus=0` 冒充 CPU0 隔离：该内核解析启动参数时只有 boot CPU0 处于 present 状态，会打印 `Housekeeping: must include one present CPU, using boot CPU:0` 并把 CPU0 放回 housekeeping；此时 `/proc/cmdline` 虽保留参数，`/sys/devices/system/cpu/isolated` 仍为空。产品与 QSPI recovery bootargs 都不得携带 `isolcpus=`。CPU0 运动闭集必须继续由启动期 affinity/cpuset、线程调度策略、IRQ affinity 及其运行期 readback落实。以后只有内核 owner 实现并证明 ARM 次核上线前后的 housekeeping 迁移语义后，才允许把内核级 CPU isolation 作为新的单变量 A/B；仅看到 cmdline 字符串不得判定生效。

最终故障隔离目标：即使UI进程死循环、画面冻结、远程镜像积压或CPU1持续100%，正在执行的CPU0运动链仍必须保持轨迹连续、servo周期稳定、轨迹队列不枯竭、EtherCAT DC/WKC稳定、编码器反馈连续和realtime safety有效；UI故障只能造成显示/普通交互降级，不能让运动卡顿、停顿、速度突变或servo overrun。CPU1过载时允许合并/丢弃过期显示帧、降低非安全日志/诊断频率、背压网络或让UI暂时无响应，但禁止降低servo频率、移动运动线程、改变CPU0 IRQ、关闭安全链或让任何非运动任务进入CPU0。若CPU0运动域本身不足，必须优化该运动实时链、预缓冲和周期预算，不能把运动持续性寄托在CPU1调度余量上，也不能把非必要任务塞入CPU0。

CPU1 显示热路径必须遵守“一次采样、一次派生、一次组帧、多方只读消费”。唯一 display projection producer 以 `source_generation + program_generation + native_generation + active_model_generation + rtcp_generation + wcs_generation + view_generation + fit_generation` 组成 scene key；key 未变化时不得重新做旋转轴等效值、坐标格式量、跟随误差、RTCP 姿态、黄色程序点空间变换、world-to-screen 或网络 payload build。RTCP OFF 时 RTCP 变换调用数必须为 0；RTCP ON 时同一 scene generation 最多一次。坐标控件、黄色轨迹、红绿点、active model 标签、跟随误差、本地 framebuffer、缓存和远程 relay 只引用这一份不可变结果，消费者或远程连接增加不得让投影/编码次数线性增加。

固定 30Hz 只定义显示端取得最新完整帧的目标节拍，不代表相同输入每 33ms 强制重算。producer 必须使用单槽 latest-wins 请求：相同 scene key 直接命中已发布 snapshot；生成期间输入前进时丢弃未发布旧结果，只计算当前最新 key；禁止用队列保存过期 source/scene 后逐帧补算。UI 每周期只允许一次 SHM 读取和一次 scene apply，坐标、轨迹、标签、误差和本地/远程显示拿同一个 const snapshot 各取所需；消费者只能产生自己的最小 dirty 绘制，不得反向触发 native 采样、scene build、RTCP transform 或 payload build。完整阶段合同与计数口径由 `REQ-SHM-DISPLAY-PROJECTION` / `功能/0-4共享内存.md` 裁决，黄色轨迹本地作业顺序由 `REQ-RTCP-G53-NATIVE-ACTUAL` / `功能/3-4UI刀路绘图顺序与状态稳定方案.md` 裁决。

已经观察到黄色轨迹 10Hz 与 30Hz 都会把 CPU1 拉满，因此“继续降低轨迹刷新率”不是完成条件，只能作为诊断变量。整改必须用同一程序、同一视角、同一 RTCP 状态、同一远程连接数和同长度采样窗口，对比 producer 的 sample/derive/transform/project/payload-build 计数、UI object rewrite/dirty area、relay capture/send 计数和 CPU1 ticks；只有计数从“按 timer/widget/subscriber 重复”收敛为“按新 generation 一次”，同时功能和 30Hz 动态显示目标仍成立，才能判定 CPU1 热路径关闭。

板端必须以 `/proc/<pid>/task/<tid>/status`、cpuset、调度策略、`/proc/interrupts`、`/proc/irq/*/smp_affinity_list` 为静态准入证据，并在active model匹配的连续golden运动中主动制造UI卡死/CPU1满载窗口，读取servo latency、overrun、EtherCAT DC/WKC、编码器连续性和电机平顺性。只看平均CPU占用、UI帧率或当前从站OP不能证明隔离成功。禁止把CPU1建成第二套状态owner、第二套运动判断、第二套队列或需要跨核文件/SHM拼接的运动链。

EtherCAT DC/servo 稳定性是 CPU 热路径闭环的一部分，不能只用 CPU 占用率低或从站当前 OP 判断通过。BUS 运行配置必须让 `appTimePeriod`、`SERVO_PERIOD`、从站 `sync0Cycle/sync0Shift` 和 lcec PLL/DC 状态在同一周期内稳定。当前5台SV630N已由板端原生SDO只读证明：`0x1C32/0x1C33:01=2`、`:02=1000000ns`、`:04=4`、`:05=125000ns`，错误状态子索引`0x0B/0x0C/0x20`均为0（命令行十进制为`11/12/32`）；这证明输出/输入SM均以DC Sync0运行1ms且设备最小周期为125µs。该机型`0x60C2`返回unsupported access，因此它不是本产品的能力或运行actual，禁止把缺少`0x60C2`误判为1ms失败。当前2ms配置的`refClockSyncCycles=5`表达约10ms reference-clock同步节拍，因此初始1ms迁移固定改为`refClockSyncCycles=10`并固定`sync0Shift=0`，继续保持约10ms同步节拍且不把第二个变量夹入首次周期迁移；两者一起进入原子1ms身份。只有该初始组合独立完成无运动、冷热启动和连续运动窗口后，才允许把`sync0Shift`作为单变量A/B；每次只改shift，失败即回到0。DC-capable 伺服从站使用 `assignActivate="300"` 时，只有在1ms正值同步路径已经独立闭合后，才允许把master-to-reference-clock同步模式（兼容写法为`refClockSyncCycles="-1"`，等价显式意图为`syncToRefClock="true"`）作为另一个后续单变量A/B，禁止与shift同时变化。任何候选都必须在当前板端内核、lcec模块和驱动链路连续运动窗口证明无`Failed to get reference clock time`、无WKC跌落、无`TIMED OUT/UNMATCHED/SKIPPED`；失败必须回到已验证的正值`10`与`sync0Shift=0`组合，不得为了追求表面启动100周期跨度更小而保留会让电机产生沙粒感的配置。验收至少要在同一连续窗口内读回`ethercat master/slaves/domains`、`halcmd show pin/param lcec.*`、`dmesg`和`/run/8ax/v5_linuxcnc.log`，证明每个100ms样本Domain WKC均为`10/10` complete、`dc-phased=TRUE`、`dc-time-valid=TRUE`、age为0、ok-seq推进、error-count无增量，窗口首尾`runtime-period-warning-count`可增但必须报告，`runtime-period-fault-count`、`domain-wc-incomplete-count`、`all-op-false-count`必须零增量；`runtime-phase-step-max/runtime-period-error-max<=100000ns`是目标，任一`>200000ns`才硬失败，并同时满足窗口组合预算。若CPU1已降而电机仍有沙粒感，应优先修正EtherCAT DC shift/PLL/周期配置或从站链路，而不是继续牺牲UI刷新、降低安全轮询、移动LinuxCNC realtime线程或关闭EtherCAT证据。

BUS 冷启动的初始 DC 纪律检查和最终运动准入是两个门。当前产品的 IgH 1.6.9 主站只用一个从站配置状态机串行推进 5 台汇川 SV630N；板端基线已证明 0 号从站的配置期 DC 差值在旧 5000ms 窗口内未达到 10µs 阈值，但超时继续后仍由周期主站完成 DC/WKC 收敛，因此这段等待必须限定为最多 500ms，并在更早达到 10µs 时立即继续。该上限只允许推进 PREOP→SAFEOP 配置，不是 ready 证明，也不得缩短 AL 状态、PDO、watchdog、链路或故障确认超时。Zynq GEM1/eth1 的产品硬件链必须保留 `pclk`、`hclk`、`tx_clk` 三项设备树绑定；板端实测删除 `tx_clk` 后虽消除了 `unable to generate target frequency` 提示，却使主站只有发送、无接收且从站数恒为0，因此非致命提示不得凌驾于真实收发和5台从站读回。

Command Gate 启动 LinuxCNC 前必须在有界窗口内确认 EtherCAT 主站链路为UP、扫描所得从站索引连续且数量与XML配置一致、初始SAFEOP+ERROR已经确认并收敛为每台PREOP且没有ERROR；未完成扫描时不得让lcec执行SDO或激活master/domain。clean restart handoff 在释放 lcec/master 前必须先停止 servo 周期、请求全部从站 PREOP，并严格读回索引连续、数量正确、无 ERROR 且每台均为 PREOP；直接 PREOP 失败时，只允许在仍持有 master 期间执行一次 INIT→PREOP 恢复，任一步失败都必须非零终止 handoff，禁止先卸载 master 再把已带电驱动遗留在 SAFEOP+ERROR。控制器硬断电而驱动不断电时，软件无法在下一次 master attach 之前阻止驱动已经锁存的 0x001A；这类 attach 前旧错误只有在扫描流程已 ACK、全部从站恢复 clean PREOP 后才允许继续。

LinuxCNC 建立 `servo-thread` 后必须先执行唯一一次 `initf lcec.activate servo-thread` 专用启动周期：该周期只激活 lcec master/domain，不运行普通 servo function list；激活完成后由 RTAPI 重置任务 deadline，下一完整 servo 周期才开始 `lcec.read-all`、运动函数和 `lcec.write-all`。lcec 模块加载阶段、普通首读周期、第二入口和运行期均不得激活或重试；系统缺少 `initf` API、激活失败或普通读写在激活前被调用时必须停止启动并 fail-closed。唯一 readiness owner 在有界启动窗口内一次核对resident activation、`all-op`与配置从站数、Domain WKC非零且complete、`dc-phased=TRUE`，并连续两次核对fresh reference clock valid/age=0、序列推进且错误计数不变。owner可在lcec激活完成、HAL结构稳定且非实时reader attach已被原生actual证明安全时先发布同生命周期的`backend_data_ready`；actiond进程可以更早gated启动；Position/WCS/State只在同一owner发布`backend_data_ready`后同批runnable，State等待当前Position lineage。`backend_data_ready`只证明HAL/native数据接口稳定且非实时reader可安全attach，不等于`motion_backend_ready`。

profile和gated relay listener不读取HAL、不输出本地/远程帧、不开放输入，可以与backend收敛并行。当前`v5_lvgl_shell --serve`不是cache-only worker：它启动后立即读取State、创建默认主页面、接管framebuffer并注册touch，因此必须等pre-UI Publisher lineage成立后再启动。开机ready只允许等待主页面对象树、fresh owner字段和主页面首帧cache；其它页面不允许在首屏前创建或缓存。所有非实时init shell必须在执行`/proc`扫描、Python启动、profile/boot barrier、文件操作或服务状态检查之前把自身完整re-exec到CPU1；只给最终daemon设置affinity不算闭合。任何一项未满足都必须保持去使能并fail-closed，不能用旧帧、单次OP、固定sleep或瞬时10/10代替。修改初始等待、激活时序或readiness后先做一次冷启动单变量否决；只有无新增realtime delay、无更严重WKC=0且明显优于基线的候选才进入3次冷启动验收。启动验收不得执行Home/Jog/Start，连续运动稳定性仍按上一段golden motion证据单独关闭。

LinuxCNC supervisor只在`linuxcnc_spawned`后对affinity/priority/IRQ/RPS执行一次设置；唯一native readiness owner随即完成一次实际读回并登记进程身份，consumer不得再次写或重跑这些设置。每个backend generation只允许一个常驻native readiness owner直接读取`/proc`与HAL/EtherCAT actual，维护当前`backend_data_ready`和`motion_backend_ready`，一次完成CPU/IRQ/scheduler、backend PID/start ticks、active INI/model、从站数、OP/WKC/DC和两次fresh DC完整合同。`dmesg` fault baseline/count不进入该合同，只作旁路诊断。owner成功后只按已登记进程身份和HAL/EtherCAT pin持续监测，在进程更换、generation变化或WKC/DC/OP actual失效时立即撤销token；不得把持续监测实现成重复设置、外部`ethercat/halcmd`、全局`/proc/*/task/*`扫描或shell合同。Command Gate、Publisher、actiond、UI boot-ready和relay只做常数级owner alive、PID/start_ticks、generation、SeqLock/CRC和freshness核对，最终事务也只能读取这些结果，不得重跑完整合同。token必须绑定本次`boot_id + backend writer identity + generation`；不得新增持久ready文件、第二owner、wrapper、fallback或轮询旁路。

启动 DAG 只使用真实事件：`linuxcnc_spawned` 后完成一次调度/IRQ设置和lcec activation；同一native readiness owner确认HAL/native数据接口稳定后发布 `backend_data_ready`，Position/WCS/State立即同批runnable，State等待当前Position lineage。profile、actiond进程和gated relay listener可提前准备；当前完整UI进程必须等pre-UI Publisher lineage成立后启动，只完成默认主页面对象树、fresh字段应用和首帧cache，且在最终barrier前不得开放远程输入。owner独立收敛首次完整WKC/OP、两次fresh DC和CPU合同，全部成立才发布 `motion_backend_ready`。最终只有一个低成本barrier，汇合 `motion_backend_ready`、Position到State的writer identity/source generation/source time/SeqLock/CRC/scene generation/progression/freshness lineage、主页面 `main_cache_ready` 和relay生命周期actual，随后同一事务交接远程主页面首帧与输入；设置、刀具、探测、偏置、IO、网络、程序、MDI及以后新增页面不进入启动barrier，用户沿导航路径首次进入目标页时才由唯一UI线程创建、应用fresh resident model、离屏capture并显示，已缓存页面后续直接blit；禁止递归创建未进入路径上的兄弟页面。不得按`S05 -> S06 -> S07 -> S08`人为排队。Position和State是异步30Hz publisher，State必须满足当前Position writer一致、`0 < state.source_generation <= position.current_source_generation`和`0 < state.source_time <= position.current_source_time`，禁止用两份瞬时坐标数组逐元素完全相等证明同源。正常冷启动以继电器完整断电上电到主页面可见可触摸约30秒为目标，验收取3次冷启动逐阶段时间；旧46秒样本曾在backend宣称ready后才出现WKC 10/10并再次跌到0/10，不得作为可恢复基线。

事件DAG的可执行服务边界固定如下：`/etc/init.d/v5-runtime-startup`是运行级别2/3/4/5唯一的产品启动入口并使用`S05`；`v5-linuxcnc-command-gate`、Position、WCS、State、`v5-settings-actiond`和`v5-ui-relay`保留各自init owner供部署、停止和局部维护调用，但必须删除它们各自的boot `S??`软链接，禁止保留shadow启动入口。orchestrator同一时刻启动Command Gate分支、actiond/profile分支和gated UI/relay分支；Publisher分支只等待同一readiness owner的`backend_data_ready`，随后同批启动Position/WCS/State。drive profile resident snapshot在冷启动只允许由`v5-settings-actiond`启动链计算并原子发布一次；开始生成前必须删除上次快照，生成失败必须使actiond分支非零退出，禁止`|| true`或旧快照fallback。UI只等待并消费该快照，必须删除UI内重复构建入口。`v5-ui-relay`先建立gated relay listener，再等待profile、Publisher lineage和`backend_data_ready`的pre-UI barrier；只有该barrier成立后才启动当前完整UI进程完成cache和首次状态应用，随后调用唯一UI ready barrier汇合`motion_backend_ready`。不得在前置阶段开放远程输入、发布正式首帧或用固定sleep代替事件。orchestrator任何分支失败必须停止本次已启动的产品服务并非零退出，不得转回旧S06/S07/S08或顺序fallback。

backend readiness性能定位只允许记录七个无副作用时间点：`linuxcnc_spawned`、`first_full_wkc`、`dc_fresh_pair_ready`、`cpu_contract_ready`、`linuxcncrsh_probe_ready`、`native_gate_socket_ready`、`backend_ready_published`。每项只记录本generation首次成立的monotonic时间，不得为了计时再执行shell检查、HAL/EtherCAT探测、全局`/proc`扫描或额外轮询；这些点只用于定位backend耗时，不是第二套ready owner或新的串行门。UI原有的`ui_service_start/remote_relay_ready/ui_process_spawned/boot_inputs_ready/ui_ready`只读取`/proc/uptime`记录本分支阶段，不得重复探测backend，也不进入运动准入裁决。

State Publisher、WCS publisher 和其它显示投影服务不得提升为 `SCHED_FIFO`，其 affinity/cpuset 必须明确排除 CPU0，所触发的普通 IRQ、workqueue 和 housekeeping 也不得进入 CPU0；不存在板端负载较低或 A/B 结果较好后允许例外进入 CPU0 的口径。它们只能按 30Hz 目标发布显示投影，限制单次采样阻塞，状态源不可用或读取超时时发布 degraded/unavailable/stale 显示状态；CPU1 满载时允许显示降级或 UI 卡死，不得通过实时提权、跨核回退或无界重试抢占微内核/native 运动控制优先级。

CPU 诊断和后续优化必须使用同一时间窗口内的 `/proc/stat`、线程 CPU ticks、`/proc/<pid>/task/<tid>/status` affinity readback 和对应服务 diagnostics 增量；不得用单次 `ps %CPU`、`top` 瞬时排序、SSH/dropbear、verify、one-shot publisher、`halcmd`、测试 sampler 或重启过程中的临时负载判断常驻基线。任何 CPU 优化不得以停止 LinuxCNC servo、降低 native safety 实时性、关闭 EtherCAT 周期、断开远程画面或删除 readback 证据为代价。native safety owner 的空闲轮询可以在满足急停 edge 直通、0.1s 内 native 急停证明和 fail-closed 语义的前提下降到最低必要频率；不得把降低 CPU 占用实现成 UI 轮询替代、安全请求 debounce、普通命令链绕行或安全回读延迟。native safety force/reset latch 在 owner 启动和重启时必须对齐已存在 epoch，旧 reset epoch 不得在新的 `estop_force` ack 后清掉急停锁存；急停验证必须证明 1 秒后仍为 active 且 machine disabled。

- State Publisher 写入 `/dev/shm/v3_status_shm` 时必须保证常驻内存，以 30Hz 为目标周期。采用 SeqLock 奇偶校验、CRC 校验、retry 和 degraded/fail-closed 逻辑。产品路径在应用参数时，为避免同步/flush/seqlock 冲突，如果需要设计 flush 双页实验，必须先更新本地 owner 并在同一个 slice 中提供验证证据。

### P0.1 执行口径

- 参数表里的全部参数都必须走同一条闭环，不按字段另开显示/保存捷径。开机链路是：硬盘参数或原生 owner -> 对应 owner/backend apply 读入微内核 native 运行内存 -> UI/provider 直连显示。参数表不得再通过 SHM 稳定显示。
- UI 参数表稳定显示只能来自 fresh 微内核 native 运行内存或登记只读 API。SHM 投影、Broker/read-only 结果、`/run` JSON、`config_cache`、诊断快照、UI 本地缓存及 dirty 值都不能作为稳定显示兜底；它们只能作为一次请求输入、动作证据或离线诊断。
- 微内核 native 常驻内存是运行态事实和运动安全判断的 owner；除编码器反馈机械坐标和运动学理论坐标外，不得把这些事实投影为 typed SHM 或诊断状态块用于稳定显示。普通产品路径不得通过 SHM、JSON 或上层缓存访问替代下层内存事实，也不得重建自造速度门禁。唯一 owner 的读取必须是启动加载、保存事务、重启/重载、generation/签名变化触发或显式诊断，不能成为每帧同步机制。
- UI 更改链路是：UI dirty/request（保存请求构建，提交瞬间从当前微内核/native 状态 provider 取最新 generation/status，坐标 SHM epoch 不能作为保存准入证明）-> Broker UDS -> 已登记 settings/native apply owner 或受控 `settings_apply` -> 写入唯一硬盘/native 参数 owner -> 源位置回读确认 -> UI 可清该字段 dirty、显示“已保存/待重启”并进入下一步。需要 backend reload 的字段保留 restart pending；普通 `drive_set_parameters` 只消费最终持久轴参数、axis→slave 映射、INI比例和profile/final bit做完整性预检并形成restart pending。右上角 `保存并重启` 不重复调用设置驱动，只创建开发板断电重启等价 clean restart handoff；该 handoff 不重复迁移参数 owner，也不增加 Home/idle/stillness、SHM/runtime freshness 等无关业务 precheck。handoff 必须重新启动并重建 UI/Broker/Action Dispatcher/State Publisher/LinuxCNC/微内核运行服务、HAL/EtherCAT/drive provider 运行闭包，清掉旧 `/run`/socket/SHM/native 状态块/resident snapshot/cache/队列后按开机链路全量入 RAM；新启动链再比较、必要时写驱动并整批fresh readback。重启后 Machine Off、native/runtime generation、从站/bit/比例链/raw limit 和必要 drive readback 作为 fresh 运行态诊断与证明；旧运行态中的 SHM、result JSON 或 resident cache 不能作为完成条件。
- UI 首个可见帧之前，启动脚本或等价 supervisor 必须已经完成参数 owner 闭包装载与 fresh 坐标显示投影就位：self/drive 参数表、drive profile resident snapshot、State Publisher `/dev/shm/v3_status_shm` fresh dynamic 帧必须可读并通过完整性校验后，才允许清屏和启动/显示 UI。首屏不能先显示空坐标、`--` 或 degraded/unavailable 再等待下一帧补齐；若显示投影自身超时，应阻止 UI 出屏并记录启动阻塞，而不是让用户看到不完整坐标界面。active model 与 axis→slave 映射缺失、重复或不一致不是显示投影启动阻塞：Position 显示元数据必须独立来自 active model/SCALE native owner，UI 仍须完成首帧并弹出从站设置提示；该不一致只让使能、Home、Start 和 drive target fail-closed。
- 字段已经写入唯一 owner 并从同一 owner 回读一致后，UI 文案只能表达“已保存/待重启生效”或等价 restart pending；不得继续显示成长期“修改未保存”。软件盘尚未确认的输入态只属于当前控件编辑态，不是参数真源，也不得扩展成 Start/Resume 前置 gate。

### P0.2 核心口径
- 除 `设置驱动` 的驱动私有域外，能由 LinuxCNC runtime INI、`PARAMETER_FILE`、`tool.tbl`、HAL/EtherCAT actual 或已登记 native owner/gate 承载的参数，必须进入对应 native owner。
- 开机入内存是读取 native owner/唯一硬盘 owner 后形成微内核/native 运行内存，不是再造一份可写参数文件，也不是再造一个 SHM 参数投影。
- 常驻脚本/daemon 的职责是将 native/runtime 坐标 actual、commanded/theory、转速、进给速度和倍率显示字段读入内存并按 30Hz 单向投影到 SHM；当前模态文字由微内核/native modal/tool 状态块或等价原生 API 直连提供；它不是新的参数 owner，也不得将 SHM 改造成控制写入通道、诊断总线或通用状态总线。状态源缺失、过期、结构错误或回读失败时必须发送 unavailable/degraded 或让动作 fail-closed，不能用旧缓存、默认值、硬盘文件或 `/run` result 补成有效状态。
- UI dirty 值、一次性 Broker request payload、Broker/result JSON、`/run` 快照、诊断文件和 `config_cache` 只允许作为瞬时请求输入、动作证据或离线诊断，不是稳定显示、保存成功、落盘参数、配置生成输入或最终真源。
- 可持久字段必须先写入对应硬盘/native owner，再刷新 native/runtime 内存并由该 owner 直接回读确认；不得靠 SHM 证明。热 apply 场景也必须证明 owner 持久化结果和微内核 native 内存一致。运行期 actual、解释器瞬态模态、RTCP 运行期开关只进微内核/native 内存，不写回硬盘默认值或 SHM。
- 微内核 native 已经拥有或可拥有的字段，不允许保留任何 V3 自造落盘参数、兼容 JSON、cache、var 或状态文件；历史兼容字段不得作为产品输入、待迁移来源或诊断兜底；同一字段/slice 必须接回 native owner 并删除旧入口。
- V3 自造参数文件按字段迁移和删除；审 owner 卡时只能盘点和同步文档，不能实现写路径、保留兼容路径或声称迁移闭环。

模型切换后的驱动同步不得按模型名写分支。通用算法固定为：在模型提交成功后的受控 reload 中清除旧代 runtime/model/mapping cache 并一次性装入当前 owner -> 读取目标 descriptor 的 active logical axes -> 对每个逻辑轴读取同代 self axis→slave 映射 -> 从该逻辑轴同代 runtime/native owner 读取 pitch、motor_rev/load_rev 或 reducer ratio、SCALE/target precision -> 从映射到的 drive profile 读取最终 encoder bits -> 约分计算电子齿轮 -> 对唯一物理从站写入并 fresh readback。active axes 缺参数、从站映射缺失/重复、profile/encoder bits 缺失、owner reload 不完整、SDO 写入失败或读回不一致时，自动同步失败且保持 Machine Off；不得混用两个 generation，不得套用另一个逻辑轴或上一个模型的比例，也不得因后续新增模型修改这条算法。

参数表不得包含 UI 显示 SHM 字段。`mcs`、`cmd_mcs`、per-axis display metadata、`following_error`、bounded `trajectory/scene`、`spindle_speed_rpm`、`linear_velocity_mm_per_min`、`feedrate_override`、`spindle_override` 只属于 `/dev/shm/v3_status_shm` 显示投影合同，不属于参数表；scene 只保存不可反向恢复控制参数的最终显示结果。历史 `runtime_modal_text` 和 `V5_STATUS_VALID_MODAL` 必须从 SHM ABI 删除，不属于参数表、诊断文本或主 UI 模态真源。VPS 授权、设备 DNA、下载身份链也不属于参数表；它们按 VPS owner 执行。

### P0.3 字段接入要求

1. 每个字段必须有 owner 卡：禁止来源、唯一 owner、唯一 writer、微内核/native 运行内存刷新路径、直接内存 API readback 字段、旧路径删除条件、板端证据。 2. 必须先具备微内核 native 内存直接 readback，再允许接入写入路径。没有 fresh readback 字段时，不得用 UI 的 dirty、显示成功或把 `/run`/`config_cache`/SHM 投影当成兜底。 3. `settings_apply`、已登记 native owner 或受控 apply 只能返回分层结果：已写入 owner/source、等待 runtime refresh、已由微内核/native 内存 fresh 直接回读确认。只有最后一类可触发 UI 稳定成功。 4. 参数表稳定显示只能消费微内核/native 常驻内存或登记只读 API；SHM 只服务同 generation 一次派生的坐标、per-axis metadata、跟随误差、bounded scene、转速、进给速度和倍率显示，当前模态文字直连微内核/native modal/tool 状态块或等价原生 API。action 运行判断必须读微内核/native 内存事实或登记 native gate readback。dirty、本地缓存、Broker result、`/run` JSON 和 `config_cache` 只保留为编辑态、动作证据或离线诊断。 5. 禁止旧入口：启动 `--inject-shm`、`settings_read_result.json` 显示兜底、通用 `config_cache` 运动/安全 fallback、V3 自造字段、测试 allowlist 和文档示例。 6. 本地 gate 只能证明 `local_verified_only`；涉及 UI、Broker、Home、运动、安全、State Publisher 或开机链路时，必须 full current build/deploy 并走完 UI/operator 路径闭环后才能声明板端结果。
## 0. 最终口径：进微内核与废弃文档
最终口径：除 `设置驱动` 的驱动私有域外，机器运动/控制参数能由 LinuxCNC runtime INI、`PARAMETER_FILE`、`tool.tbl`、HAL/EtherCAT 状态或已登记 native owner/gate 承载的，都必须使用微内核/native 真源。V3 不得为这些参数另造或保留任何落盘 JSON、var、cache、状态文件或等价自造参数表。

当前已登记模型的旋转轴减速比真源为 active runtime INI：AC 从 A/C `[AXIS_*]`读取，BC 从 B/C `[AXIS_*]`读取；当前物理实例 A=`50/1`、C=`50/1`，它们是参数实例，不是固定模型算法。BUS和Pulse源INI必须保持一致。C轴以18bit、`PITCH=360`、`MOTOR_REV=50`、`LOAD_REV=1`和`[JOINT_4] SCALE=10000`计算时，`unit_per_count=0.0001 deg`、目标命令计数为`72000 count/motor_rev`、电子齿轮约分为`4096/1125`、负载一圈为`3600000 count`。物理encoder raw一count为`360/(2^18*50)`度，它与经过电子齿轮后的PDO position一count不是同一个量；UI机械坐标换算只认INI `SCALE`。所有值在完整重启后统一读取和回读，不由旧drive-only分子/分母或HAL参数反推。

### 0.1.1 自造参数表的字段级收敛要求

| --- | --- | --- | --- |

轴参数表中的 `max_velocity`、`max_acceleration`、`soft_minus/soft_plus` 均是机械输出侧语义，不是电机侧 RPM、转子速度或电机角加速度；换算和审计必须结合螺距/每转角度、`motor_rev/load_rev` 和减速比所在比例链 owner。UI/readback 速度单位固定为 X/Y/Z 直线轴 `mm/min`、active model descriptor 登记的旋转轴 `deg/min`；runtime INI `MAX_VELOCITY` 使用 LinuxCNC 秒单位，写入时仅由 UI 每分钟单位除以 60，读取显示时乘以 60。当前 AC 模型的默认值为：X/Y/Z `max_velocity=10000`、`max_acceleration=500`、`soft_minus=-500`、`soft_plus=500`；A/C `max_velocity=50000`、`max_acceleration=2000`。BC 和后续模型必须按 descriptor 对应旋转轴使用同类单位和 owner 参数，不得回退固定 A/C；旋转轴软限位默认保持 owner 的 no-soft-limit 语义，不用电机侧默认值兜底。

物理 PDO router 只负责同周期轴/从站、zero anchor 与 wcheckpoint 整数映射，不拥有速度、加速度、追赶或轨迹整形。G-code `RUNNING` 时，LinuxCNC/motmod 已规划完成的 `joint-target-position` 必须在映射后逐周期原值写入对应从站：同一偶数 `diagnostic_seq` 中必须满足 `accepted_raw == desired_raw == slave-target-position`，不得保留 limiter、候选速度、上一输出、追赶模式或第二套轨迹状态。只有 planner 已 `STOPPED/IDLE` 或 Machine Disabled 时，LinuxCNC/native owner 才允许把 commanded/规划器内部状态一次性同步到 actual；Machine Disabled 下该“追赶”只表示内部目标归位，物理 controlword 仍保持安全去使能，绝不驱动电机补跑。重新使能只能在该 commanded/actual 同步成立后开放；若需要真实移动到另一位置，必须重新提交受原 planner 约束的运动。wcheckpoint 换窗必须由 motmod 在同一周期原子平移 `pos_cmd/pos_fb/coarse/free_tp/cubic` 后再由 router 使用同一 base 映射，不能交给 PDO 层追赶。若使能、换窗、零点或重载前无法证明 native commanded 已与 actual/新窗口同步，native motion gate 必须保持去使能并明确拒绝，禁止用 PDO 追赶掩盖 owner 不连续。

字段迁移完成条件：每个字段都审 owner 卡，包含旧入口删除对象、目标 native owner、唯一 writer、微内核/native 内存直接回读字段、旧文件读写入口删除条件和板端证据。迁移测试优先用现有工具；现有工具不能证明字段迁移、旧入口删除、native readback、SHM 仅投影、禁止路径扫描或 board/motion 需求时，先补最小可复用工具并验证工具，再推进产品代码。字段或 slice 先逐个本地验证 `local_verified_only`，多个字段本地稳定后，再由 VM 直接只读挂载完整当前 Windows worktree，在 `VM_BUILD_ROOT` 集中编译、部署并完成 UI/operator/motion 闭环。缺 owner 卡或缺工具证据时，只能做盘点 and 文档同步；不得实现写路径、保留兼容路径或声称迁移闭环。

### 显示/刀路辅助参数
来源：当前 runtime INI 的 `[DISPLAY]` / `[RTCP]` 和微内核/native 内存或原生 API 直接回读；不得同步到 SHM。`settings_runtime.json` 中的旧显示几何字段不得作为 `settings_apply` 的持久配置生成输入、显示长度最终真源、待迁移来源或离线诊断兜底。
UI 状态 provider / 微内核直连读取：
- runtime INI `[DISPLAY]` 下的 `T0_TOOL_HOLDER_LENGTH_MM = 15`，作为 T0 显示刀柄长度直接回读并在启动时进入 native/tool 状态发布内存；若 `tool_display` / `toolpath_display` / `tool` 下的 `t0_tool_holder_length_mm` 必须从产品路径删除，不得作为输入、兜底、过渡来源或离线诊断真源。

- HAL pin/native component、常规 native daemon 或登记 native memory block 读回 RTCP/switchkins actual 状态。RTCP 开机初值的唯一持久 owner 是 active runtime INI `[RTCP].DEFAULT_ACTIVE`，值只允许 `0/1`；HAL/native 初始化时把它装载一次作为 UI latch 的初始输入，再由同一个 switchkins actual 回读确认。默认 seed 为 `0`，但启动代码不得写死 OFF，也不得在 backend ready、Machine On、Home 或 Program Open 后命令式归一化 OFF。UI 状态 provider 从同套 native 内存/API fresh 回读前只能返回 unknown/degraded，不能把持久默认值、按钮文案、旧 SHM 帧、一次性 helper 输出或旧 `/run` 结果当作 actual。

## 2. 后续如何调用

### 设置页显示
设置页字段读取顺序是：
`正在编辑的 UI dirty 值（仅编辑请求态） -> fresh 微内核/native 常驻内存或登记只读 API readback -> 自建参数表中唯一 owner 的轴参数 RAM 快照/readback -> 驱动参数映射表或 drive provider readback -> unavailable/degraded`

对应代码：`board/app/src/v5_settings_page*.c` 负责页面/动作/状态，`board/app/src/v5_settings_axis_table*.c` 负责表格布局、下拉、键盘、commit、resident/runtime readback，`board/app/src/v5_settings_action_ipc.*` 负责 actiond IPC，`board/app/src/v5_settings_parameter_store.c` 只服务自建参数表和 drive-only 持久 owner。dirty 值只表示待提交输入；稳定显示必须回到 fresh native/provider readback。`/run/8ax_v5_product_ui/settings_action_events.jsonl` 和动作响应快照只允许作为一次动作证据或离线诊断，不得成为首屏、SHM fallback 或显示真源。
UI C 不允许直接读 `/opt/8ax/phase0_bus5/settings_runtime.json`、runtime INI、`linuxcnc.var`、`tool.tbl` 或其他硬盘参数文件。内存 readback 缺失时必须显示 unavailable/degraded 或 blocked，不能用静态默认值、旧参数表、旧 `/run` 结果或模型默认值冒充当前生效参数。
### 设置保存

设置页修改后：
- UI 组装 `settings_save_return` / `settings_apply` 请求时，必须在保存请求构建，提交瞬间从当前微内核/native UI 状态 provider 取最新 owner generation/readback token，不得使用进入设置页或开始编辑时缓存的旧 token；SHM `status_epoch` 只用于 UI 显示 freshness 和诊断，不参与保存准入。请求再通过 Command Broker UDS 发送。
- Broker 调用 `board/services/command_gate/v5_settings_apply.c` 总入口及同目录 `v5_settings_apply*.c` 拆分 owner；字段归属由 `v5_parameter_owner_map.*` 决定。settings/native apply owner 可以在一次事务中使用完整内存请求 payload 来生成 runtime INI、校验源输入和执行 readback，但这个 payload 不能从旧 V3 落盘参数表恢复，也不能原样保存成新的 V3 参数表。
- 属于微内核 native 的字段必须写入其唯一硬盘/native owner，再在完整重启或登记受控重载后装入运行内存并直接回读；属于 `设置驱动` 私域的 profile/final bit 可以写入 drive-only `settings_runtime.json` 并完成源位置回读。普通 `设置驱动` 不写物理驱动；电子齿轮/CSP 的物理写入和 UI provider actual 刷新只由新启动链完成，不得依赖 SHM。
- 写入 `settings_runtime.json` 时必须经过 drive-only 提取器，只保留设置驱动私域字段和深度清洗后的驱动证据；运动、坐标、刀具、模态、RTCP/G53 几何、显示默认、安全互锁和倍率字段不得落入该文件，也不得藏在 `zero_model`、`drive_position`、readback 或诊断子对象中。
- UI 本地值或 `immediate_persist` 只能表示编辑态请求态已更新；State Publisher 发布 generic `config_cache` 和 `/run` result 更新不能替代微内核 runtime fresh typed 回读证明。
- 需要 LinuxCNC 生效的字段只保存到唯一持久 owner并标记待重启；字段级“已保存/待重启”以源位置回读为准。右上角 `保存并重启` 不在旧进程自动写驱动或立即回读，而是直接准备开发板断电重启等价的 clean restart handoff。新启动链从最终 INI/model/mapping/profile一次加载、必要时设置驱动并fresh readback；运行态最终生效以重启后的LinuxCNC/native/drive actual为准，不得进入SHM或回退历史值。
### 运行状态和刀路显示
LinuxCNC 运行状态分两条读取链路：
- UI 坐标、per-axis metadata、跟随误差、黄色轨迹/scene、转速、进给速度和倍率的统一显示链路固定为：微内核/native owner + resident program model -> 唯一 display projection producer 按 generation bundle 一次派生 -> State Publisher 一次 commit -> typed SHM -> UI reader 每周期一次 -> `V5UiStatusView/V5DisplaySceneView` -> 全部显示消费者只读。缺 fresh snapshot 只能显示 degraded/unavailable，不能用参数表、文件或本地旧值补有效。
- 当前模态文字、急停/使能、安全、行号、诊断和系统指标仍由对应 UI status provider 直连 native 读取，不进入 SHM。WCS、刀具长度、RTCP/G53、旋转轴 count-domain 的原始 actual 仍不进 SHM，但唯一 display projection producer 可以读取它们并输出不可反向恢复控制参数的最终 bounded scene；`v5_main_page_model_projection.c`、`v5_toolpath_display.*` 等 UI 路径迁移后只保留消费/render，旧 projector 和 fallback 必须删除。
- Broker、后台 action/native dispatcher 和 owner 响应结果不得读取 typed SHM 作为动作依据、准入、保存证明或安全证明；需要状态时必须读微内核/native 内存/API 或登记 native memory block。
### 驱动动作

设置驱动、驱动扫描、驱动读回、驱动复位等 设置驱动 域动作不走 UI 缓存直接推断驱动能力。 - 动作入口只消费启动阶段已加载的 drive-only 轴到从站/profile 绑定、驱动证据和驱动映射内存快照；`settings_runtime.json` 和 profile 映射表只能在启动装载、设置保存后刷新或受控重载时读取。 - EtherCAT 从站扫描结果进入驱动/native provider 内存，动作匹配只消费该内存视图。 - 结果写入 Broker/action 响应和状态文件只能作为动作证据；后续 UI 稳定显示来自驱动/native provider readback，SHM 只显示坐标/轨迹、转速、进给速度和倍率白名单字段；当前模态文字直连微内核/native modal/tool 状态。运行判断和成功证明必须来自微内核 native 内存或驱动 owner 直接回读。总线设零使用这些驱动读回作为证据，但零点/坐标基的长期 owner 应迁入微内核/已登记 native owner/gate 或受控 runtime apply 路径，不再扩展 V3 参数文件真源。
## 3. `settings_runtime.json` drive-only 收口要求

`/opt/8ax/phase0_bus5/settings_runtime.json` 必须收敛为 `schema=re.v3.settings_runtime.drive_only.v1`。该文件只允许持久化设置驱动私域和深度清洗后的最小驱动 count-domain 证据： - 轴到从站/profile 绑定。 - 电子齿轮输入和读回。 - encoder raw/multiturn 参数。 - drive SDO 写入/读回证据。 - active-model 对应旋转轴的 count-domain 校准。 - BUS 设零 `zero_model` 证据。
`zero_model`、`drive_position`、readback 或诊断子对象只能保存驱动对象、电子齿轮、encoder raw/multiturn、count-domain 零点锚点、驱动读回和必要验收时间状态码；不得保存 `pre_status`、`linuxcnc_snapshot_before`、`runtime_scale_check_before`、WCS/G92/TLO/tool/RTCP/modal/status 帧或 runtime INI 回读行。这些运行事实必须留在微内核/native 内存 owner，并由 UI provider 直连显示；离线诊断证据只作诊断；Broker `/run` 结果只能作动作证据，不能落进 `settings_runtime.json`，也不能作为参数表稳定显示或保存成功真源。

`zero_model` 的最终裁决：它只是在设置阶段持久化的 drive-only count-domain 零点锚点/证据，不是当前坐标、Home 目标、homed 状态、运行门禁或 UI 显示真源。开机/clean restart，或Machine Off下成功完成的BUS单轴设0原子live事务，可以由受控 native/runtime count-domain owner 校验并一次性装载该锚点到同一微内核/native 常驻零点内存，生成 generation/readback token；运行期 Home、等效角和坐标投影只能消费该 native resident readback，不得再次读 `settings_runtime.json`。BUS设0live事务还必须同步Router反馈/目标、Home和旋转checkpoint，并以fresh native MCS一count归零闭合；失败全部回滚。LinuxCNC live软限位等较重参数允许由响应明确要求的彻底clean restart装载。当前 BUS Home 的目标仍是微内核/native 定义的真实机器零位/最近等效零并必须真实运动；不得把 `zero_model` 写入、磁盘值等于 0 或 UI 本地归零冒充 Home 完成。若没有可验证的 native 装载/回读，依赖该零点的运行能力必须 fail-closed，不能直接消费硬盘证据。

BUS `zero_model` 以物理 `slave_position` 为唯一 owner 身份，逻辑轴只通过当前唯一 axis→slave 映射消费。映射只有两个合法写入口：运动模型实际切换事务按 descriptor 写入完整默认映射；轴参数 `从站` 下拉由操作员逐轴人工覆盖。任一入口改变映射时，同一物理从站的零点、count-domain 和旧驱动actual证据都随物理从站保持，但旧证据立即标记“待重启”，不得复制给新逻辑轴。脉冲当量/SCALE、软限位距离、速度和加速度归逻辑轴的runtime INI owner；模型事务把最终 `[AXIS_*]` 参数装入目标 `[JOINT_n]`，人工从站覆盖不得迁移它们。映射、SCALE或其它参数持久改变只形成restart pending，设置阶段不做HAL/SDO live apply；彻底clean restart后用同一final model、INI AXIS/JOINT/KINS/TRAJ、HAL mapping、逻辑轴SCALE、物理从站零点和drive actual证明输出跟随最终映射。只有映射与SCALE均未改变的BUS单轴设0允许已登记原子live事务。
该文件禁止持久化： - runtime INI 运动约束：HOME/HOME_OFFSET/HOME_SEQUENCE、软限位、速度、加速度、FERROR、BACKLASH、`[TRAJ]`、`[AXIS_*]`、`[JOINT_*]`。 - `[DISPLAY]` 显示默认、tool/TLO/T0 显示默认。 - WCS、G92、RTCP 运行期开关、G53/kinematics 几何。 - HAL/IO 安全互锁、feed/rapid/spindle override truth。
`settings_apply` 可以继续在内存中使用一次完整请求 payload 来生成 runtime INI 并在同一事务里做源读回，但该 payload 不能从落入 V3 自造参数表恢复，也不能原样写回任何 V3 参数文件。写入 `settings_runtime.json` 的内容必须经过 drive-only 提取器裁剪。UI 参数稳定显示必须来自 native/provider readback；SHM 投影只来自 State Publisher 拆分后的 typed fresh display 白名单字段。真正成功证明来自 owner/native 直接回读。`config_cache` 必须从产品路径删除，不能合并 active runtime INI 和 drive-only 字段后继续充当 UI/SHM 投影或参数真源。
`settings_runtime.json` 不得再由 `machine_settings.json` 回填或恢复。drive-only 提取器必须删除运行时标注型字段（例如 `drive_axis_role`）和非必要派生显示字段；schema、role、truth-boundary 等说明只能作为结果诊断字段或字段卡分类，不得成为新的参数数据模型。
## 4. 设置页字段接口 owner 规则

设置页每个数据必须按“唯一 owner -> 微内核 native 运行内存/API -> UI 直连 provider”显示，UI 修改必须走 “Broker -> settings_apply/native owner -> owner 写入 -> 微内核 native 运行内存刷新 -> native/API fresh
直接回读 -> UI 可清该字段 dirty”闭环。下列字段不得再由 `settings_runtime.json`、`machine_settings.json`、`config_cache`、`/run` result 或 UI 默认值充当真源：
- 轴显示语义元数据：`axis_mode`、目标精度 precision、`mechanical_max_velocity_rpm/max_velocity`、`soft_minus/soft_plus`、`zero`
  可以写入 active runtime INI `[AXIS_*]`/`[JOINT_*]` 及等价 v5 设置元数据，用于 UI 从运行内存读回。`pitch`、`motor_rev`、`load_rev` 属于已由微内核/native 承载的比例链字段，必须写入 active runtime INI/native owner 并从同 owner 运行内存/readback 读取；`slave` 这类微内核/native 和驱动参数映射表都没有承载的设置页轴参数，进入自建参数表并由 UI 自动读取，只作为设置页参数和后续受控 apply 的输入 owner，不得冒充 native 已生效 readback。 - 运动实际真源：`SCALE`、`MAX_VELOCITY`、`MAX_ACCELERATION`、`BACKLASH`、`FERROR`、`MIN_FERROR`、`HOME_SEQUENCE`、Pulse `HOME/HOME_OFFSET/HOME_SEARCH_VEL/HOME_LATCH_VEL`
  必须写入 LinuxCNC runtime INI `[JOINT_*]`/`[AXIS_*]` 原生字段并从 owner 回读。- BUS/Pulse 零位边界：模式差异、切换闭环、Pulse 设零/回零和驱动动作能力边界以 `REQ-BUS-PULSE-MODE-DIFFERENCE` / `功能/2-1总线脉冲区别.md` 为准；本文只维护字段 owner。BUS 设零保留 drive-only `zero_model` 证据并通过 native/runtime 直接回读显示；Pulse 设置 `zero` 显示和保存机械零点到 `HOME_OFFSET`；两者都不能把旧 V3 JSON 当长期 owner。- 驱动私域：`profile_id/slave_index`、电子齿轮、encoder raw/multiturn、SDO 写入/读回证据，active-model 对应旋转轴的 count-domain 校准仍属于 Settings Drive private domain；只允许在 drive-only
  `settings_runtime.json` 或驱动映射 profile owner 中持久化。- `settings_projection` 是历史 SHM 设置页投影债务；设置页必须改为 active runtime INI、drive-only owner 和微内核/native 直连 provider 显示，不得把它作为保存成功、运行安全门禁、board live 或配置生成真源。

## P0 启动预加载与实时常驻内存锁定规则

本章节是顶部最高口径的执行拆解。目标是消除运行时闪存/磁盘 I/O 抖动，确保微内核与上层控制系统的实时响应性能。

### 1. UI 与产品自写代码启动预加载（自写代码常驻）
- **C 语言交互层（C UI）**：
  - 前端进程编译为静态/预链接的单体二进制文件；运行期必须使用启动阶段已加载的页面、字体、图片、布局、按键表、命令表和资源索引。
  - 开机启动后立即调用 `mlockall(MCL_CURRENT | MCL_FUTURE)`，锁定前端交互所有代码页及资源页在 RAM 中，杜绝运行时内存页面置换。
  - UI 运行期不得为了按钮动作、页面刷新、参数显示、图标/字体、状态解释或动作结果再读取硬盘资源兜底；需要可变内容时必须来自微内核/native 常驻内存、SHM 显示白名单或受控程序文件选择。
- **Python 业务层（Command Broker 与 Actions）**：
  - 后台守护进程和 command/action gate 在引导阶段通过固定登记表或等价 resident registry 初始化。
  - 自动遍历并导入（`importlib.import_module`）全部业务动作处理器（包括 `settings_apply`、`settings_axis_zero`、`drive_bus` 等），完成 Python 字节码在 RAM 中的常驻编译，使后续 UDS 指令调用全部在内存中进行，避免运行态临时读取 Python 脚本。
  - Broker、Actions、provider、native gate wrapper 和 action registry 必须在启动时形成固定内存登记表；运行期不得按请求再扫描脚本目录、懒导入模块、按文件名执行未登记脚本或从旧路径 wrapper 兜底。
  - 产品自写守护进程同样必须预加载业务模块、配置 schema、协议表和最小资源索引；运行期只消费内存快照或受控刷新后的 provider。

### 2. 微内核与实时层物理锁定（LCNC / RTAPI 强实时）
- **微内核运行闭包全量入 RAM**：
  - LinuxCNC/微内核启动所需的 executable、共享库、RTAPI/实时模块、HAL component、kinematics component、INI/HAL 配置、`PARAMETER_FILE`、`tool.tbl`、G53/RTCP 几何、WCS/G92 初始 readback、驱动映射和已登记 native gate 表，必须在开机阶段一次性读入 RAM 或由 initramfs/RAM disk 提供。
  - 开机装载完成后，运行期控制、运动准入、状态 readback、UI 显示 provider 和动作证明必须来自 LinuxCNC/微内核/native 常驻内存或原生 API，不得再临时读取硬盘文件作为当前真源。
  - 设置保存允许写回唯一持久 owner；写入后必须刷新或重载对应 native 常驻内存，并用该 owner 的内存/API readback 证明结果。单纯落盘成功不等于运行态已生效。
- **物理内存锁死（Memory Locking）**：
  - 底层实时装载器（`rtapi_app` 及相关的伺服/总线线程）必须无条件启用物理内存锁死。在内核空间与用户空间强制锁定所有实时内存页面（`mlockall`），杜绝页面换出（Page-Out）。
  - LinuxCNC uspace 实时装载器 `/usr/bin/rtapi_app` 必须由 canonical LinuxCNC package 以 `root:root 4755` 交付，canonical bundle installer 在 `chown` 后必须恢复并回读该模式；setuid 特权位只允许 upstream 明确登记的 `rtapi_app` 与 `linuxcnc_module_helper` 两个 helper 持有。LinuxCNC 启动完成前必须回读实际 servo/RTAPI 线程的调度策略，只有已进入平台登记的实时策略才算 ready；若仍为 `SCHED_OTHER`，必须 fail-closed 停止控制栈，不得继续 EtherCAT 状态切换、SDO 写入或运动。UI、Publisher、relay、provider、日志和诊断进程不得继承该特权位或实时调度策略。
  - 使用 `mlockall` 的 UI、provider、State Publisher 和产品自写常驻进程必须有资源预算边界：G-code 只读 resident identity/内存索引/显示模型、轨迹缓冲、页面资源、字体/图片缓存和临时解析结构不得无限增长。具体预算由板端内存、运行闭包大小和 owner 验证确定，不能在基石规则中写死固定 MB 或某个 LVGL 私有 API。超出预算时必须 fail-closed 或 degraded/unavailable，不得用 swap、临时读盘或旧缓存兜底。
  - display projection producer 的 resident G-code parser/program model/scene pool、UI renderer 和其它大块显示内存必须按唯一 owner 进入独立资源域：可使用 Memory Pool、owner 专用 arena、整体生命周期对象池或 `mmap`/`munmap`。设计目标是文件切换、页面销毁和受控重载能整体归还物理页，且 producer/UI/relay 不各存一份等价 scene。
  - 加工文件关闭、`program_close`、程序切换、页面销毁和受控参数 reload 等非实时边界，拥有资源的 program/scene producer 与 UI consumer 必须释放可回收对象、清理 resident 引用，并在空闲期执行 `malloc_trim(0)` 或平台等价 trim；该调用不是成功证明。
  - Start/Resume、实时插补、急停响应、servo 周期、LinuxCNC realtime/HAL/EtherCAT 周期和任何微内核运动热路径中，严禁新增大块动态分配、释放扫描、heap trim、`malloc_trim(0)` 或等价内存整理动作；热路径只能消费已经装载和锁定的 resident 资源。
  - 内存回收验收以板端实际 RSS/Locked 水位为准，不以“代码调用了某个函数”为准。至少通过 `watch -n 1 'grep -E "VmRSS|VmLck|Locked" /proc/$(pgrep v5_lvgl_shell)/status'` 或等价板端自检，在 `N>=10` 次加载/卸载大 G-code 文件、程序切换和页面切换循环后，确认 `VmRSS`/`VmLck`/`Locked` 回落到同一稳定基线；曲线不得出现单调增长的阶梯状泄露趋势。
- **动态链接符号立即绑定（LD_BIND_NOW）**：
  - 守护进程或实时进程的拉起环境变量中，必须显式定义 `LD_BIND_NOW=1`。
  - 指示动态链接器在程序启动的第一时间解析所有关联共享库（`.so`）的动态符号并全量载入 RAM，消除由于懒加载（Lazy Loading）导致运行态突发闪存读取的隐患。
- **物理共享内存限制（tmpfs）**：
  - 所有的进程间同步（SHM，例如 `/dev/shm/v3_status_shm`）必须在 RAM 内存文件系统中创建，完全避免物理闪存写入。
  - SHM 只保留同 generation 一次派生的 UI 显示投影：坐标/per-axis metadata/跟随误差/bounded scene、主轴转速、进给速度和倍率。当前模态文字直连 native modal/tool；原始 RTCP actual、WCS/G92、tool、驱动模式、安全、停稳、动作成功和参数真源不进 SHM，但其不可反向恢复的最终显示 scene 可以由 producer 输出。
- **RAM 盘根文件系统镜像（RAM Disk / initramfs）**：
  - 板端最终运行形态必须在引导阶段将 LinuxCNC/微内核运行闭包和业务运行闭包打包为只读镜像或等价 manifest，并一次性释放到 `initramfs`/RAM disk 或等价 RAM 驻留路径中。
  - 闪存介质在运行期应只读挂载或从热路径隔离；除明确的设置保存、程序文件选择、事件触发且限频的日志/诊断写入和受控重载外，控制与显示热路径不得触发闪存读写。
