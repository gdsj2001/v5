## 1. 边界

UI 刀路层只能做显示：

- 每个动态周期只读取一次同一 generation 的 v5 typed SHM `status/scene view`。
- Program Open/Load 只把原始 G-code 交给唯一 display projection producer 建立 resident 只读程序模型；运行态 UI 不解析或遍历原始程序点。
- renderer 只遍历 producer 已生成的最终 scene 点、segment、颜色/图层和 dirty bounds，把它们交给一个 custom-draw scene；不得在 renderer 内执行 RTCP、active-model 姿态、黄色轨迹、坐标、跟随误差、fit 或全量 world-to-screen 计算。
- 缩放、平移、视角切换和手势只更新 `view_generation` 并请求 producer 生成下一份 scene，一次 UI 事务内不得由多个控件分别投影同一数据。
- 通过正式控制链发送用户命令。

G-code 显示和移动是独立显示链路，和投喂给 LinuxCNC/微内核的执行链路不是同一个 owner。display projection producer 可以在 Program Open/Load 阶段解析用户打开的原始程序，生成 resident 黄色线程序模型、预览点、first-point metadata 和 WCS 标记；运行时再按同一 generation 的 native actual 一次性生成最终 scene。UI 只消费 scene，不再次解析、旋转或投影。这些结果只用于屏幕显示、fit、视角、手势和诊断，不得反向参与 `Set Open` 路径选择、Start/Resume 投喂、G-code 文本生成、A/B/C 等旋转轴 word 改写、M-code 改写、等效角运动判定或微内核运动语义。执行链路必须按 `REQ-GCODE-RUN-HOT-PATH` 使用打开的原始 G-code 文件身份和内容交给 LinuxCNC/微内核；显示链路与执行链路只能通过只读 identity/hash/generation 对齐，不共享写路径。

UI 刀路层禁止做运动真相：

- 不修改 G-code，不为了显示效果改 model-specific `cc-ac.ngc`、`cc-bc.ngc` 或用户程序。
- 不计算 RTCP 真实运动学，不替代 LinuxCNC/HAL/EtherCAT/native owner。
- 不把 G53、回零、等效角、软限位、驱动 preset、rebase 或参数写入逻辑塞进绘图层。
- 不恢复 `/run` JSON、FIFO、临时 response JSON、短生命周期 Python fallback 或 UI 热路径 `fork/exec`。
- 不用绘图修正掩盖 Broker、SHM、LinuxCNC 或 HAL 状态错误。

状态链固定为：

```text
微内核/native 坐标/RTCP/WCS/G53/tool actual + resident program model
-> 唯一 display projection producer 按 generation bundle 一次性生成 immutable status/scene snapshot
-> 常驻 v5 State Publisher 一次组帧/CRC/commit -> /dev/shm/v3_status_shm
-> v5 UI SHM reader 每动态周期一次 -> 坐标/刀路/标签/跟随误差/本地与远程显示只读消费
```

控制链固定为：

```text
UI C -> 已登记 native command gate -> LinuxCNC/HAL/EtherCAT/native owner
```

## 2. 数据来源

刀路显示必须使用同一 generation bundle 的状态：`source_generation`、`program_generation`、`native_generation`、`active_model_generation`、`rtcp_generation`、`wcs_generation`、`view_generation` 和 `fit_generation` 共同组成唯一 scene key。常驻 display projection producer 从微内核/native 常驻内存、登记 API 和 resident program model 取得同一组输入，由唯一 dispatcher 选择恰好一个 active-model branch，并一次性生成坐标、每轴显示 metadata、跟随误差、黄色轨迹、红绿点、WCS/MCS、active-model 几何、标签锚点和 dirty bounds；State Publisher 只把结果写成 immutable typed SHM status/scene snapshot。RTCP OFF 时 producer 不进入 RTCP 变换，RTCP ON 时同一 scene key 只计算一次。坐标面板、fit consumer、renderer、标签、本地 cache 和 remote relay 都只能读取统一 scene，不得再次判断 AC/BC、重读 native owner、解析 G-code、计算姿态或执行 `world -> screen`。AC/`XYZAC_TRT` 与 BC/`XYZBC_TRT` 继续使用互不包含的独立 branch，只共享不可变 scene ABI 和模型无关数学原语；未知 registry、ops 缺失或 generation 不一致时 fail-degraded，不得回退。绿色 actual、红色 commanded、刀柄和其它可见几何的 owner 坐标与刀长关系必须由 producer 在同一 scene 内一次确定；缺少同 generation 输入时隐藏或 degraded，不能用 UI 默认值、文本、JSON、旧缓存或其它图层结果补画。

### 2.1 Active Model 总开关和新增模型规则

`active-model` 是 UI 轨迹图形、微内核运动学和运行生效轴集合共同使用的唯一总开关，不是可以分别选择的开关。一次 clean restart 后，fresh native active-model readback 必须同时对应唯一的 model descriptor、唯一的 LinuxCNC kinematics module、唯一的 UI projector branch 和唯一 active-axis/slot 集合；任一不一致都不得显示或运动。

- UI 调用图固定为 `fresh active model -> registry -> 唯一 dispatcher -> 一个 model branch -> model-neutral scene -> geometry/fit/render`。dispatcher 之外禁止再次判断模型。
- AC branch 只实现 A 摇篮携带 C 的父子关系、旋转顺序、G53 组合和图形投影；BC branch 只实现 B 摇篮携带 C。任一 branch 不得包含、调用或修改另一个 branch 的公式、缓存和状态。
- AC descriptor 只允许 `X/Y/Z/A/C` 进入 scene、坐标面板、黄线姿态、轴线、fit 和跟随误差，B 必须视为 inactive/NAT；BC descriptor 只允许 `X/Y/Z/B/C` 进入上述 UI scene，A 必须视为 inactive/NAT。设置表保留未激活轴参数不代表 UI 可以把该轴画进当前模型。
- UI branch 只能消费 descriptor 给出的 active axes/slots；不能扫描所有非 NAT 行、固定读取 A/C 或 B/C，也不能因 stale/缺失轴值把上一模型的 inactive 轴保留在画面。模型切换时上一 branch 的 axis scene/cache 必须与 geometry/fit cache 一起失效。
- shared 层只允许 registry/descriptor、不可变 scene ABI、模型无关向量/矩阵原语、裁剪以及 `world -> screen`；shared geometry、fit、renderer 和 status 代码不得出现任何具体模型的旋转父子公式。
- 新增一个 active model 时，只允许新增一个 descriptor/registry 条目、一个独立 UI projector branch、一个独立 kinematics branch/module 和各自独立测试；不得修改 AC/BC branch 塞入新模型条件，也不得把新模型公式放进 dispatcher/shared 层。
- 模型切换必须使上一模型 branch 的可变 scene/geometry/fit cache 失效；未知、未登记、branch 缺失或 UI branch 与 kinematics module 不一致时 fail-closed/fail-degraded，禁止回退 AC、BC 或上一模型。

代码结构审查必须能从 registry 项直接追到唯一 UI branch 和唯一 kinematics module，并证明每个模型 branch 的编译/链接闭包不含其它模型公式。该跨域约束的运动学 owner 和 clean-restart 语义由 `功能/微内核.md` 维护。

RTCP/G53/active model 的动作证明和几何真源统一命名为 same-generation native actual，不同步到 SHM。RTCP 的控制因果是：UI 按钮只通过 `native_rtcp_control` 递交 ON/OFF 到微内核 RTCP 控制 latch，微内核 owner 直接写 HAL/switchkins actual 并回填 ack/readback；G-code 显示链不得改写、生成或转义 RTCP 控制文本。modal/tool owner 和 RTCP/switchkins owner 必须暴露 RAM 状态块或等价原生 API，UI/provider 必须直连回读这些 owner 并与 SHM 坐标 generation 收敛。State Publisher 不得把当前模态、当前刀号和对应刀长写入 SHM、硬盘文件或 `/run` result 作为 UI 真源；若存在桥接，只能是登记 RAM/native API pass-through，真源仍是微内核/native owner。WCS、TLO、active model geometry 和 G53 仍必须来自微内核/native 内存或登记 API，不允许 UI 从旧 SHM 字段、多个缓存或 JSON 自行拼帧。`rtcp_actual_enabled` 如保留，只能是从 native RTCP/switchkins actual 派生的显示字段。RTCP request 只能在界面显示 pending；只有后续 native actual 确认 active 后，RTCP 图层才能切 active。

RTCP/switchkins actual 的 native 状态源、HAL pin/native component、native memory block 和无 fork 回读规则由 `功能/微内核.md` 的 `REQ-RTCP-NATIVE-STATUS-SOURCE` 维护。本文只规定 UI/刀路消费 30Hz SHM 显示投影 + 直连微内核状态的显示语义；不得在本文重复维护底层状态源形态。

重建边界固定如下：

总规则：启动 supervisor 先在 UI 不可见状态等待有效 SHM；只有 SeqLock/CRC/size/version、fresh epoch 和 dynamic valid mask 均通过后，UI 才能接管 framebuffer、创建/显示结构对象并用该首个有效帧建立 WCS/MCS/active model 状态层和第一次 fit。打开 G-code 后建黄线，黄色轨迹载入完成后必须把黄色轨迹、WCS/MCS、active model 几何、红点、绿点、刀柄线和标签等全部可见元素纳入同一个 screen fit；RTCP/模态文字、当前刀号和对应刀长必须直连 native modal/tool owner；之后只机械坐标值、native 模态/tool 状态和速度/倍率显示投影驱动动态层，不得重建。

- 开机/页面创建顺序固定为：`v5-ui-relay` 或等价 supervisor 先等待有效 30Hz SHM dynamic 帧；未满足时不得清屏、启动可见 shell 或显示空骨架/degraded 页面。通过后才允许 shell 接管 framebuffer，并以该有效 SHM 帧和直连微内核/native WCS、active model geometry、modal/tool 状态创建同一套 WCS/MCS/active model 对象，再生成第一次 fit。页面在已运行 UI 内重新创建时可以复用 resident last-good scene/cache，但不得用它绕过冷启动 SHM 门槛。
- 打开 G-code 顺序固定为：Program Open/Load 读取原始程序文件并生成只读 UI 程序模型和 resident identity；`runtime_program_path` 始终保存原始 G-code 文件路径，不生成或登记执行用运行副本。成功后只重建黄色程序线，黄色轨迹载入并可见后必须基于当前 scene 的全部可见元素生成一次统一 fit，不允许只按黄色线自身包围盒 fit。
- RTCP 显示状态只能从直连微内核/native RTCP/switchkins actual 派生；产品动作证明必须直连 native actual。运行 G-code 时可以复用上一帧有效 RTCP actual 缓存；RTCP 开关、模态变化、异常、无有效缓存或缓存过期时必须重新读，不能用缓存假装正常。
- 开机结构和黄色线建立完成后，后续只由同一帧内存里的机械坐标值做整张图形的显示投影驱动；模态文字、转速、进给速度和倍率只更新动态显示，不得周期性触发 WCS/MCS、active model 轴线、黄色程序模型或 fit 重建。运行中只允许唯一 producer 使用已经生成的本帧可见候选，对当前冻结安全边界做一次有界越界判定：候选仍在边界内时只更新动态投影；新的可见候选真实越出边界时，才按该越界事件对全部当前可见元素重算一次 fit、推进一次 `fit_generation` 并立即冻结。该次提交后的相同候选、相同 scene generation 或持续越界状态不得再次触发；只有后续 fresh scene 的新候选再次越出新的冻结边界，才形成下一次越界事件。主页面真实前后台可见状态必须作为 `0/1` typed IPC 输入传到 producer；缓存预热/离屏绘制不得冒充前台可见。隐藏期间 producer 可以接收 fresh source 并更新动态 scene，但越界检测和 fit 提交计数必须为 0；隐藏到可见这一显隐事件本身不得推进 UI fit 请求或伪造新候选，但 producer 在重新可见后的首个候选必须对当前冻结边界做一次有界判断：未真实越界时 `fit_generation` 不变，若候选已在隐藏期间真实越界则只提交一次 fit 并立即冻结，相同候选或持续越界后续不得重复。UI、renderer、relay、timer 和隐藏页面不得执行越界判断或 auto-fit。
- fit 在开机/页面创建后首次读到有效 SHM UI 显示投影并直连读到必要 native actual 覆盖状态层时建立一次，在打开 G-code 成功并显示黄色程序线时按全部当前可见元素重建一次；操作员明确点击 XY/XZ/YZ/3D 切换投影平面时，允许在该前台动作边界按新平面计算一次。active model 的 committed 切换必须经过受控重启，新 branch 在冷启动首个有效 scene 上建立自己的 fit；不得沿用上一模型 fit。每次结构边界或真实越界事件完成后 fit-lock 立即冻结。除唯一 producer 确认的新可见候选真实越界事件外，G-code 运行、暂停、恢复、坐标/姿态变化、RTCP/WCS/TLO/模态变化和后台 resident 更新均不得计算、扫描、扩大或重建 fit；页面隐藏时连越界事件也不得触发。未来若提供操作员明确触发的独立“适配视图”动作，它同样只可在前台执行一次 fit，不得由定时器、后台 worker 或运行状态隐式触发。
- WCS/MCS 原点和轴线、active model 旋转轴线和中心点的 UI 对象只在冷启动 SHM 门槛通过后创建为可见对象，并立即用首个有效显示帧及必要 native actual 建立真实投影坐标；不得先显示非真值骨架。后续状态变化只更新同一结构的投影坐标。
- 黄色程序线只在 Program Open/Load 成功时解析、缓存或重建；之后只能由同一帧内存里的机械坐标值驱动已缓存黄线投影，不得重新读文件或重新建黄线。
- rtcp 直接去读内存里的模态；后续只作为内存 actual gate 参与机械坐标投影，不得触发 fit、WCS/MCS、active model 轴线或黄色程序模型重建。

`cmd_mcs` 只能用于显示和诊断，不能替代 `mcs` 通过 Home、Start、Jog、RTCP 或设零门禁。

当前主页面可见坐标必须同时显示当前模态、机械坐标系、加工坐标系和 active model 对应旋转轴坐标；AC/`XYZAC_TRT` 显示 A/C，BC/`XYZBC_TRT` 显示 B/C，后续模型按 active model readback 选择对应轴。当前模态文本必须由微内核/native modal 状态块或等价原生 API 提供，并显示常用 LinuxCNC modal group 的 active G 码，至少覆盖运动 `G0/G1/G2/G3`、平面 `G17/G18/G19`、单位 `G20/G21`、刀补 `G40/G41/G42`、刀长 `G43/G43.1/G49`、当前 active WCS `G54-G59.3`、路径控制 `G61/G61.1/G64`、固定循环 `G80-G89`、距离模式 `G90/G91`、圆弧距离 `G90.1/G91.1`、进给模式 `G93/G94/G95`、主轴速度模式 `G96/G97` 和返回模式 `G98/G99`；显示顺序必须稳定，不能因 LinuxCNC `stat.gcodes` 原始顺序或固定 10 项截断而丢常用模态。当前模态区域必须在轨迹区左侧显示成单列：`当前模态` 标题之后先显示当前刀号和该刀号对应刀长，刀长下一行显示同源 native actual 的 `RTCP ON`、`RTCP OFF` 或 `RTCP --`，随后每个 active G 码独立一行，不得横向长串截断。当前刀号/刀长来源必须同为微内核/native tool owner 回读，RTCP 行来源必须同为 native RTCP actual provider，不得从 SHM modal、按钮文字、请求值、设置页、磁盘 `tool.tbl`、JSON 或旧缓存补齐。

加工坐标数值列必须遵循 `REQ-WCS-NATIVE-OWNER`：使用同源当前 `mcs - active_wcs_offset` 计算，并且始终只与机械坐标相差当前 active WCS offset；不得显示固定 `0,0,0`，不得把 `cmd_mcs`、驱动表、设置页表格、profile、旧缓存、默认 0 或非同源快照当作加工坐标系真源。当前 active WCS 坐标系在轨迹区里的原点必须来自 native 当前 WCS offset 在 G53/MCS 空间中的位置；机械坐标点移动时，轨迹区加工坐标系结构原点不得跟随当前 `mcs` 或刀尖位置移动，只有 active WCS offset native readback 变化才允许改变屏幕投影。

黄色轨迹按微内核/native active model readback 对应的显示旋转方式刷新；AC 模型下随同源 A/C 显示机械值旋转，BC 模型下随同源 B/C 显示机械值旋转。该轨迹变化使用的模型参数必须和微内核运动学使用的模型参数完全相同。

UI 里的当前加工坐标系显示投影必须按 native RTCP actual 分支：

- RTCP OFF 时，当前加工坐标系原点和 X/Y/Z 轴方向不需要跟随 active model 旋转轴旋转。
- RTCP ON 且 native RTCP actual、active model 和 active geometry fresh 有效时，当前加工坐标系原点和 X/Y/Z 轴方向必须使用同一帧 active model 姿态跟随当前模型旋转轴旋转。

两种 RTCP 状态下，X/Y/Z 是否补偿只由微内核/LinuxCNC/native 运动学 owner 实现和证明，不由 UI WCS 图形旋转证明；active WCS offset 数值本身不因 UI 图形投影被改写。`cmd_mcs` 继续只用于红色 commanded 点、理论轨迹和跟随误差诊断。

**跟随误差与坐标统一消费红线**：同一帧内所有机械与理论坐标值（包括 active model 对应旋转轴的等效折算角、每轴显示量子/位数和跟随误差），**必须由 display projection producer 统一进行一次性计算**并写入 immutable scene snapshot。
- **跟随误差** 统一计算并保存为浮点数：
  ```text
  following_error[i] = mcs[i] - cmd_mcs[i]
  ```
- **坐标大字控件、刀路 custom-draw renderer、跟随误差面板、active model 标签和远程画面** 必须同时消费 scene 中已算好的数值或绘制点，严禁任何消费者本地重复计算，也严禁读取 UI 控件文本进行反向解析。消费者增加只增加只读引用或最终像素发送，不得增加 producer 的 sample/transform/project 次数。

操作员可见 actual 坐标、绿色机械点、MCS/WCS 坐标系、active model 轴线、active model 标签、刀柄/动态姿态、黄色刀路姿态以及红色 commanded 刀尖点，都必须先在 display projection producer 收敛为同一份 scene snapshot，再提供给坐标面板和刀路视图。禁止出现“坐标面板用 `rotary_phase`，刀路视图用 `mcs[3]` / `mcs[4]`”的双路径显示；在 BC 或后续模型下，同一规则适用于 active model 选中的旋转轴。若 `rotary_phase`、count-domain、unwrap 或对应有效标志不可用，所有 actual 显示消费者必须同帧进入 degraded/不可用显示或同帧回到受控 raw actual 显示；红色 commanded 点缺少同帧 `cmd_mcs`/刀长/有效 MCS 门禁时也必须隐藏，不得用默认 0 位姿或种子点补画。

### 2.1 状态层与程序层的绘图门禁

UI 刀路视图分为两类图层，不能再用同一个“是否已打开程序”条件一起控制：

- 状态层：MCS 机械坐标系、当前/程序 WCS 加工坐标系、active model 轴线和标签、绿色机械点、刀柄线、红色 commanded 刀尖点、坐标读数和 degraded 文本。
- 程序层：黄色 G-code 进给/切削轨迹。

主页面刀路对象必须由当前 v5 主页面 owner 建立，v3 图层名称只约束可见视觉结果和层级：左侧刀路区先创建透明 `toolpath_clip_layer` 或当前 v5 等价裁剪层，黄色进给线作为裁剪层内的 `lv_line`，红色 commanded/microkernel 点使用 v3 的 `toolpath_microkernel_marker_dot` 视觉样式（7x7、`rgb(255,64,64)`、边框 `rgb(255,230,230)`），绿色机械/holder 点使用 v3 的 `toolpath_holder_marker_line` 视觉样式（7x7、`rgb(68,221,144)`、边框 `rgb(220,255,235)`）。v5 不得在主页面外层另造第二套语义相同的点、说明标签或颜色近似替代层；缺少有效 `cmd_mcs` 或 `mcs` 时对应红/绿点必须隐藏，不得用默认点补画。

v5 主页面轨迹区域必须覆盖 v3 `ReV3ToolpathLayerHandles` 的操作员可见语义，但对象命名、文件拆分和内部 handle 结构以当前 v5 owner 为准：当前模态标签、summary/detail/view 标签、透明裁剪层、黄色 G-code 进给线及分段线、MCS 原点锚和三轴线、当前 WCS 原点锚和三轴线、program WCS 原点/三轴线/标签对象、active model 轴线和标签、active model 中心点、刀柄线、红色 commanded/microkernel 点、绿色 actual/holder 点、MCS/WCS 小标签、XY/XZ/YZ/3D 视图按钮都必须按 v3 的可见层级、颜色、线宽、点大小、标签位置、按钮默认/选中态实现；具体旋转轴线由 active model descriptor 决定，AC 显示 A/C，BC 显示 B/C，未知模型不得回退 AC。加工坐标系可见层只显示当前模态/当前 active WCS；其它非当前 G54-G59.3 program WCS 对象必须保持隐藏，不得在运行态同时铺出 9 组加工坐标系。冷启动首个可见主页面必须已经通过 SHM 门槛并使用该有效帧建立 MCS/当前 WCS/active model 状态对象；不得在 SHM `UNAVAILABLE` 时先显示空页面、非真值骨架、默认坐标或红/绿种子点。黄色 G-code 轨迹在程序未打开时保持空。没有 native tool/geometry/program WCS actual 时，对应动态数值、刀柄线、active model 姿态和非当前 program WCS actual 必须保持隐藏或显示正式 degraded 状态，不能用默认旋转轴 0、设置页表格、驱动 profile、旧缓存或 v3 后端计算补齐。视觉样式必须参考 v3 UI；但 G-code 解析、active model 姿态、WCS/G53/RTCP、投影 fit、状态收敛和运行期刷新逻辑必须继续按本文和对应 `REQ-*` owner 实现，不得照搬 v3 后端逻辑、旧缓存、JSON/result、helper 或 fallback 路径。
轨迹区元素尺寸必须参考 v3 源码常量，不允许自定小号结构：左侧红框完整刀路 viewport 固定为主页面坐标 `(x=0,y=55,width=394,height=386)`，其唯一 canonical 参数 owner 是 `board/app/include/v5_toolpath_viewport.h`；producer request/fit 中心、LVGL `toolpath_clip_layer`、clip 内 custom-draw scene、标签锚点、手势命中区和 dirty 裁剪只能引用该参数，禁止各层复制数字、保留 `388x378` 别名或按自身控件推导另一套画布。黄色刀路、MCS/WCS、active model 轴、红绿点和刀柄等非文字几何必须合并在 clip 内唯一一个铺满 viewport 的 custom-draw scene，不得每类几何各建一张全尺寸画布；producer scene 必须携带改变元素旧位置与新位置的 dirty bounds，UI 只 invalidate 两者并集。未变化 generation 不刷新 custom-draw；文字标签保留为必要的独立 LVGL 对象，但只在文字、位置或显隐实际改变时刷新。普通结构线宽 `1px`，黄色 G-code 轨迹线宽固定为 `1px` 细线，MCS 原点线宽 `2px`，刀柄线宽 `5px`；红/绿锚点为 `7x7`，active model 中心点为 `2x2`；标签与 XY/XZ/YZ/3D 按钮位置和尺寸按 v3 创建。active model 首个有效可见状态和运行态几何方向必须按机械坐标系投影：AC 模型下 A 红线沿 MCS X 方向，C 青线始终平行于机械 YZ 平面，初始沿 MCS Z 方向，MCS 有效后按同帧 A 机械值围绕 A/X 轴旋转后再投影，不能产生 X 方向分量；BC 模型下 B 轴线和 C 轴线按同一 active model projector、B/C G53 几何和同源 B 机械值投影；后续模型必须走对应 model projector。active model 直线总长度固定为 `80.0mm`，即以中心点向两端各投影 `40.0mm`。AC 模型 A/C 中心点必须来自 G53 机床坐标目标位置表的 A/C native readback；BC 模型 B/C 中心点必须来自 B/C native readback；不得用手工屏幕坐标、默认 0、设置页显示文本反推或 v3 后端缓存替代。不得用手工屏幕斜线、短线或与机械坐标系不匹配的替代线。
加工坐标系方向必须来自同一机械坐标投影基向量；active model 对应旋转轴为 0 的姿态下，当前 WCS 的 X/Y/Z 方向必须分别与 MCS X/Y/Z 方向一致，只允许原点位置因当前模态 WCS offset 改变，不允许用另一套手写屏幕坐标造成加工坐标系与机械坐标系方向不一致。黄色程序轨迹投影必须随微内核/native active model readback 对应的显示旋转方式变化：AC 模型随 A/C 显示机械值旋转，BC 模型随 B/C 显示机械值旋转。UI 里的当前加工坐标系显示投影必须按 native RTCP actual 分支：RTCP OFF 时当前加工坐标系的原点和 X/Y/Z 轴方向不需要跟随 active model 旋转轴或其它模型显示旋转方式旋转；RTCP ON 且 native RTCP actual、active model 和 active geometry fresh 有效时当前加工坐标系的原点和 X/Y/Z 轴方向必须跟随同一帧 active model 姿态旋转。两种状态都不得因 RTCP actual 切换而改写 active WCS offset 或加工坐标数字列。非当前加工坐标系保持隐藏。
MCS/WCS 坐标系三轴线的世界长度固定为 v3 `RE_V3_TOOLPATH_POSE_MCS_AXIS_LEN = 40.0mm`；首个有效可见状态、无 G-code 状态和运行态都只能投影这 40mm 轴线，不得为了填满轨迹区域把坐标系线段手工放大、拉长或改成屏幕像素固定长线。屏幕显示长度只能来自统一 projector、fit、手势缩放的结果。

Program Open/Load 成功并返回主页面时，主页面显示缓存必须先用当前页面对象重新渲染并刷新 resident cache，不能直接 blit 开机旧主页面缓存覆盖新黄色轨迹、红点、绿点或程序预览。普通页面切换仍可使用 resident cache，但任何会改变主页面可见内容的 Program Open、MDI 载入或同类动作必须标记主页面 cache dirty 并在返回主页面时回写新缓存。

状态层必须以同一帧 SHM UI 显示投影/v5 typed status view 的有效机械坐标作为实际坐标绘图闸门，并以直连微内核/native 的 WCS、active model geometry、RTCP 动作证明和 tool 状态作为结构参数。只要 native 运行内存/API 能读到 actual 坐标，即使尚未打开 G-code、程序为空或 machine off，State Publisher 也必须把 MCS 和 active model 旋转轴坐标、转速、进给速度和主轴/进给倍率投影到 `/dev/shm`；当前模态、当前刀号和刀长必须由 native modal/tool owner 通过 RAM 状态块或等价 API 提供，State Publisher 不得把它们变成 SHM 或硬盘真源。冷启动 UI 只能在有效 SHM dynamic 帧到达后显示状态层；不得因为尚未打开 G-code、黄色轨迹未解析、程序缓存为空或 Program Open 失败而隐藏 MCS/当前 WCS/active model 坐标读数。该路径禁止读取硬盘文件、设置表、驱动表、profile 或旧缓存补坐标。UI 已经取得 first-good 帧后若运行期短暂 stale，可保留上一帧已确认的当前 WCS/active model 对象并显示 degraded；`status_epoch == 0`、非 finite、默认 0 或不可用哨兵在冷启动阶段不得作为首帧放行，也不得绘制种子图、假坐标数值、默认位姿或假 active model 姿态。
UI init/relay 启动脚本必须在启动可见 `v5_lvgl_shell`、清屏或接管 framebuffer 前阻塞等待 SHM dynamic 帧通过 SeqLock/CRC/size/version、fresh epoch 和 valid mask 校验。未满足时保持 UI 未出屏并由 supervisor 记录明确启动阻塞；不得用空界面、内核企鹅画面超时后强行拉起 shell、非真值骨架或 degraded 首帧绕过。SHM 门槛通过后只启动一次 shell，后续坐标从 fresh 到 stale/degraded 的变化走常规局部刷新，不得反复 stop/start。
UI relay 不得因为“坐标 ready”事件重启已经运行的 `v5_lvgl_shell`。坐标从 degraded 到 fresh 只能通过常规 status/native readback 刷新覆盖当前页面；反复 stop/start shell 会造成主页面坐标在有效值和 `--` 之间闪烁，属于不合格启动/刷新路径。
UI native readback 刷新必须按 owner 独立合并：WCS、modal/tool、RTCP 和安全状态互不作为彼此显示门禁。安全/急停 probe 失败时只能让安全状态 degraded，不得丢弃本轮已经读到的 WCS offset、当前模态、刀号、刀长或 RTCP actual。

黄色程序层只在 Program Open/Load 成功后增加。打开 G-code 前不得画黄色默认轨迹、示意弹簧、假两点路径或旧程序残影；打开 G-code 成功后，黄色轨迹可与同一帧状态一起投影。黄色轨迹解析失败、Broker 返回失败或程序帧过大导致 open 未确认时，只能显示状态层和明确原因，不得用旧黄色轨迹冒充当前程序。

红点不是 MCS ready 门禁的例外，不得为了让界面看起来完整而提前绘制默认 0 位姿、种子点或旧帧红点。程序几何可以在 Program Open/Load 阶段提前解析和缓存，但屏幕实际绘制必须等待 MCS ready 后再把程序几何与同一帧状态投影到同一个 scene。

`empty_status_scene`、假两点背景、初始种子坐标、默认旋转轴 0 或可见的非真值开机骨架都属于退役路径。冷启动缺少有效 MCS/SHM 时 UI 不出屏；运行期已经取得 first-good 帧后短暂丢帧时，只能保留上一帧已确认结构和红/绿点并显示 degraded，同时隐藏无法继续证明的刀柄线、active model 姿态和其它强依赖动态真值，不能用伪造几何让界面看起来“已就绪”。
## 3. 绘图语义

绘图按状态分层推进。所有可见元素必须先在各自真实空间中生成几何点，再通过统一 projector 进入屏幕坐标；某个元素已经投影成屏幕点后，不得再单独按屏幕方向补偿来冒充几何正确。

### 3.1 红色 commanded 刀尖点

红点表示理论 commanded/microkernel 刀尖坐标，按 v3 `re_v3_toolpath_microkernel_kinematic_xyz` 语义由同帧 `cmd_mcs` 和当前显示刀长计算：

```text
red_tip.x = cmd_mcs[0]
red_tip.y = cmd_mcs[1]
red_tip.z = cmd_mcs[2] - tool_len

tool_len = display_tool_length_mm
           if display_tool_length_valid && finite(display_tool_length_mm)
           else 0.0
```

红点要求：

- `cmd_mcs[0..2]` 全部 finite，且本帧非 stale；否则隐藏。
- 不消费 `tool_holder_tip_xyz`、`tool_tip_xyz`、`actual_tip_ac_pose` 或绿色机械点。
- 只用于刀路可视化、诊断和跟随误差，不参与任何控制门禁。

### 3.2 绿色机械点和刀柄线

绿色点/刀柄上点必须直接等于同一帧 `mcs[0..2]` 的 actual 机械反馈点，且与机械坐标大字、轨迹绿色 actual 点使用同一消费端，不得另读参数表、旧缓存、UI 文本或 `cmd_mcs`。刀柄线从该机械坐标点向当前显示刀长方向绘制：同源世界坐标起点为 `mcs[0], mcs[1], mcs[2]`，终点为 `mcs[0], mcs[1], mcs[2] - display_tool_length_mm`，投影和手势 transform 必须与绿色点同帧一致；屏幕 3D 默认视角下表现为从刀柄上点向下的蓝色 5px 粗直线。刀柄线只做显示，不反推 RTCP 真值；缺少 fresh MCS 或有效刀长时隐藏刀柄线但保留绿色点。

### 3.3 黄色主刀路

黄色只表示进给/切削路径。G0/G53 不得进入黄色主刀路，可隐藏、灰线或只做诊断。程序模型是只读 G-code 几何；黄色程序几何只在 Program Open/Load 成功时解析、缓存或重建。之后只由同一帧内存里的机械坐标值驱动整张图的显示投影，不重写源程序、不重新创建黄色程序模型。

RTCP actual 开启或关闭都不改变黄色轨迹的显示旋转数据源：黄色轨迹必须按微内核/native active model readback 选择对应 projector；只要同 generation 的 native active model、对应几何与坐标 SHM 中同源旋转轴显示机械值有效，黄色轨迹就必须按坐标面板同源的旋转轴显示机械值，围绕直连微内核/native 的 active model G53 几何轴做显示投影刷新。AC 模型使用 A/C 显示机械值和 A/C G53 几何，BC 模型使用 B/C 显示机械值和 B/C G53 几何。RTCP OFF 时 active model 旋转轴运动，黄色轨迹也需要跟随同源旋转轴显示机械值旋转；RTCP ON 时使用同一条投影链路，不得切到另一套 raw、多圈、commanded、滤波或种子角度。该旋转只影响 UI 投影输出，不得使黄色程序模型或 fit 失效，不得修改 G-code、HAL、LinuxCNC actual 或运动门禁真值。

黄色主刀路是 Program Open/Load 成功后的增量层，不是 UI 轨迹视图的基础层。主页面进入、回零后、开机后或尚未打开程序时，MCS/WCS、Active Model 对应旋转轴和坐标读数等状态层必须照常显示；黄色层保持空。Program Open/Load 成功并写入当前可见程序元数据后，黄色层才解析并显示当前 G-code。若 Program Open 失败，必须保留状态层、弹出明确失败原因，并清除或拒绝刷新当前黄色层，避免操作员误认为失败程序已经加载。

#### 3.3.1 大程序预览抽样

刀路预览可以为了板端内存和绘制开销对大程序做显示层抽样，但必须满足以下边界：

  - 抽样只影响黄色预览模型和屏幕绘制，不影响 G-code 文件、LinuxCNC 执行、运动门禁、坐标真值、WCS/RTCP/TLO actual 或任何控制链请求。
  - Program Open 的黄色程序层必须使用独立程序预览容量，参考 v3 `RE_V3_TOOLPATH_MAX_POINTS = 512` 或当前 v5 等价预览容量；不得把 `/dev/shm/v3_status_shm` 的 30Hz 动态轨迹点数上限 `V5_STATUS_TRAJECTORY_POINT_COUNT = 16` 当作 G-code 程序预览容量、程序完整性判断或黄色程序 fit 来源。SHM 16 点只属于动态显示投影 ABI，只能用于动态 overlay，不得导致 `cc-ac.ngc`/`cc-bc.ngc` 这类 5 圈弹簧程序被截成 5 点或 16 点方形/折线。
  - 抽样不能完全静默。程序缓存、诊断文本或等价状态必须标出 `preview_decimated=true`、原始候选段数、保留点数、抽样策略、微位移阈值、分段数量和是否触达点数上限。只因 `RE_V3_TOOLPATH_MAX_POINTS` 满而停止追加时，必须显示为预览降级，不得冒充完整路径。
- 抽样策略必须优先保留首尾点、切削段端点、WCS/模态变化点、Active Model 对应旋转轴的大跳边界、明显拐点和 fit 所需包围盒关键点。微位移过滤只允许删除显示上不可分辨的连续小段，不得删除会改变视觉拓扑、WCS 归属、当前模型旋转轴 wrap 边界或黄色/G0/G53 分类的点。
- 大程序必须按显示层 LOD/分段抽样，不能只依赖固定 `RE_V3_TOOLPATH_MAX_POINTS` 数组原地 compact。触达容量上限时，不得把最后一个程序点塞入当前连续折线尾部形成跨大段的终点长拉线；必须在截断/抽样断点处切分 segment，或者显式隐藏不可证明连续性的尾部连接。
- 点数上限不能作为功能正确性的唯一保护。提高 `RE_V3_TOOLPATH_MAX_POINTS` 前，必须先证明同一 scene generation 只投影一次、custom-draw renderer 只遍历最终 bounded scene、且程序点数不会让 UI/relay 建立第二份拷贝；否则放宽点数只会放大 CPU1 和内存占用。
- 如果无法证明抽样后的预览仍保留关键几何和分类边界，UI 必须显示 degraded/preview incomplete，而不是输出看似正常的完整轨迹。

### 3.4 WCS/MCS 与 fit

- 所有图形基础框架是 MCS。
- MCS 原点和轴线可见，但不显示 `MCS`、`MX/MY/MZ` 文字标签。
- MCS、当前 WCS、程序 WCS 的 X/Y/Z 轴线世界长度固定 40mm。
- 当前 model branch 输出的父轴标签必须显示在父轴端点，子轴标签必须显示在子轴端点；AC 输出 A/C，BC 输出 B/C，shared renderer 不固定标签字母。
- A/B/C 中心点用 2px 中心标记，只表示刀路 UI 里的 active model 图形中心，不替代轴线或刀尖点。AC branch 的 A/C marker 必须按同一份 native scene 组合值生成：A 图形中心的 X 使用当前 active WCS 的 X 偏置，A 的 Y/Z 使用 native `G53_A_Y/G53_A_Z`，C 的 X/Y 使用 native `G53_C_X/G53_C_Y`，C 图形中心的 Z 使用当前 active WCS 的 Z 偏置。BC branch 的 B/C marker 对应使用 `G53_B_X/active_wcs_offset_Y/G53_B_Z` 与同一 C 中心组合。设置页 G53 表仍显示原始 A/B/C 几何 readback，顺序为 A 中心、B 中心、C 中心、对刀仪、5 方向检测仪；该设置页表格不得反向覆盖主页面当前 active WCS 参与的图形中心。
- 刀路底部状态区固定显示两行 operator 可读状态。第一行显示 active model 对应旋转轴的 G53 组合显示值：AC 模型显示 `A/C G53 A X... Y... Z...  C X... Y... Z...`，其中 `A.X` 取当前 active WCS 的 X 偏置，`C.Z` 取当前 active WCS 的 Z 偏置，`A.Y/A.Z/C.X/C.Y` 来自同一份 native G53 geometry readback；BC 模型显示 `B/C G53 B X... Y... Z...  C X... Y... Z...`，其中 `B.Y` 取当前 active WCS 的 Y 偏置，`C.Z` 取当前 active WCS 的 Z 偏置，`B.X/B.Z/C.X/C.Y` 来自同一份 native G53 geometry readback。第二行显示当前活动坐标系偏置值，格式按 active model 选择旋转轴：AC 显示 `<当前WCS> 偏置 X... Y... Z... A... C...`，BC 显示 `<当前WCS> 偏置 X... Y... Z... B... C...`，后续模型按 active model 对应轴显示；数据必须来自 fresh native/interpreter WCS offset snapshot/API。两行只能显示同一帧 fresh native readback 值；无有效 native active model、geometry 或 WCS offset 时保留上一帧有效文字或保持空白，不得显示默认 0、设置页表格值、SHM 字段、`geom--`、`wcs-ofs--` 或内部诊断 token 冒充真实值。
- pending WCS 只允许作为主页面 WCS 按钮/命令已发出的短时视觉提示；actual 为 G54 但刀路底部仍显示 `WCS pending: G55` 是错误状态。G54 OFS `X+030.000 Y+020.000 Z-050.000` 时，G54 原点就是 `(30,20,-50)`。
- 开机默认视角必须是 3D，`view_3d` 进入 3D；当前 v5 3D 视角投影 owner 是 `board/app/src/v5_toolpath_display.c`，v3 向量只作为视觉参考。相机方向为 X+ Y- Z+ 的低角度正交视角，正交投影基向量必须与 v3 一致：screen right = `(0.7071067811865476, 0.7071067811865476, 0.0)`，screen up = `(-0.4082482904638631, 0.4082482904638631, 0.8164965809277261)`。在默认 3D 屏幕上 X 轴应向右下，Y 轴应向右上，Z 轴应竖直向上；v5 不得用近似经验公式、按钮选中态、手工屏幕坐标或其它 3D 斜投影替代这组向量，首个有效可见状态也不得使用另一套手写斜率。
- XY/XZ/YZ/3D 的 screen fit 必须让轨迹区域全部元素保持同一世界坐标比例：黄色刀路、MCS/WCS、active model 旋转轴几何、红点、绿点、刀柄线和标签锚点必须共享同一个 projector、同一 fit、同一个缩放系数和同一份 canonical viewport；producer 以该 viewport 的宽高和中心 fit，renderer 不得以控件尺寸二次偏移。屏幕横向和纵向共用该缩放系数，按可用安全区中较小的比例整体居中，剩余方向留白，不得为填满窗口而分别拉伸横纵方向或对任何图层单独缩放。相同世界长度的坐标轴在未叠加手势变换时必须显示为相同屏幕长度，尤其 MCS/WCS 固定 `40.0mm` 的 X/Y/Z 轴在 3D 中三轴等长，在 XZ 中 X/Z 等长，在 YZ 中 Y/Z 等长。
- fit 的常规完整重建入口只有三类：开机/页面创建后首次读到有效 SHM UI 显示投影和必要 native actual 并建立 WCS/MCS、active model 状态层结构时；Program Open/Load 成功并出现黄色 G-code 程序线时；操作员明确点击 XY/XZ/YZ/3D 切换投影平面时。active model committed 切换通过受控重启回到第一类入口。运行期只有一个有界例外：唯一 producer 发现新的可见候选真实越出当前冻结边界时，为该越界事件完整重算一次并立即冻结。运行、暂停、恢复、WCS/RTCP/TLO、active model 姿态或红点/绿点移动本身不得触发 fit；同一越界事件持续存在时不得重复推进 `fit_generation`；页面隐藏后不得由后台 timer/worker 判断越界或预先 fit。
- 最终 fit 使用 `RE_V3_TOOLPATH_WINDOW_FIT_RATIO = 0.80`，候选包含黄色刀路、WCS/MCS、active model 线、红点、绿点、刀柄线和前台标签锚点；G-code 黄色轨迹载入完成后的 fit 必须基于这些全部当前可见元素计算一次，不能沿用开机 fit，也不能只对黄线局部 fit。该 fit 边界与 projector 参数随 program/model/view generation 一起冻结；运行期 producer 只用已经生成的本帧可见候选做一次有界越界判定，未越界时不得再次扫描全部 fit candidates、扩大边界、回中或缩放。真实越界时只允许对全部当前可见元素重算一次，提交后立即冻结；相同候选或持续越界不得连续重算。图形安全区必须排除左侧模态文字、底部状态文字和右侧视角按钮。
- 手势显示态当前 owner 是 `board/app/src/v5_main_page_input.c`，视图 transform 与 fit 分别由 `v5_toolpath_display.*`、`v5_main_page_fit.c` 消费；若手势状态机继续变复杂，必须下沉到新的 `v5_toolpath_*` 或 `v5_main_page_*` 子 owner。`manual_scale`、`manual_screen_rotate_deg`、`manual_wcs_x_rotate_deg`、`manual_wcs_z_rotate_deg` 只影响显示 transform且不得触发 fit。按 XY/XZ/YZ/3D 四个视角按钮时，必须结束当前手势并把旧 `manual_scale` 复位为 `1.0`、旧 screen rotation 和 pan 复位为 `0.0`，恢复该按钮对应的默认视角，并只在这个明确前台动作边界按新投影平面和 canonical viewport 重新居中 fit 一次；不得重建 WCS/MCS、active model 轴线或黄色程序模型。

### 3.5 RTCP 与 Active Model

- active model readback 是 display projection producer 内唯一的模型总开关。每个 fresh scene 只能由 dispatcher 解析一次并冻结 `registry_id/branch/source/native/rtcp/G53/WCS/view/fit generation`；本帧黄色轨迹、模型轴线/中心、RTCP ON WCS-follow、fit、标签和模型状态行全部消费同一个 branch scene。共享 producer 层只允许负责 freshness、不可变 scene ABI、模型无关向量/矩阵、world-to-screen、图层编排和 dirty bounds，不得决定 A/B/C 的具体父子公式；UI renderer 不包含 projector。fresh model id 切换必须先废弃旧 branch scene；同一 model 的短时输入 stale 只保留 last-good scene 且不产生新 pose/fit。
- AC branch 独占 A/C 轴槽、A/C G53 组合、`C-as-child-of-A` 的点姿态、C 中心/方向、黄色轨迹、RTCP-WCS follow、fit 候选和 A/C 标签；BC branch 独占 B/C 对应输入和 `C-as-child-of-B` 的完整输出。AC 与 BC branch 之间禁止 include/call 对方实现，禁止读写对方专属 mutable state；共享缓存只能保存 model-neutral scene/key，切换 registry id 时必须使旧 branch pose/geometry/fit cache 失效。
- 后续新增运动模型必须同时新增唯一 registry entry、独立 branch 文件、该模型的轴/槽/G53 输入、pose composition、geometry builder、fit candidates、标签/状态格式和独立 golden/focused test。新增模型只允许登记 dispatcher 映射并新增该 branch；不得修改 AC/BC branch 来塞入新条件，也不得在 shared dispatcher、geometry、fit 或 renderer 中追加 `if model == ...` 模型公式。删除模型时 entry、branch 和 tests 同 slice 物理删除，不留 fallback。
- RTCP 图层只按微内核/native RTCP/switchkins actual 显示；UI 侧必须通过状态 provider 直连读取 native modal/tool 和 RTCP actual。`rtcp_actual_enabled` 或等价布尔字段只能是 RTCP actual 的派生显示投影。request、dirty、按钮状态、SHM `runtime_modal_text`、设置文件和非同源 helper/cache 只能显示 pending 或诊断，不得提前切 active；State Publisher 内部上一帧有效 RTCP 缓存只允许作为运行期回读优化，开关/异常/无有效缓存时必须重读。
- active model 姿态源必须来自同一 generation 中由 display projection producer 收敛的唯一旋转轴显示投影。AC 模型使用 A/C，BC 模型使用 B/C；producer 以对应旋转轴的编码器反馈机械坐标为动态输入，以微内核/native count-domain/unwrap/degraded actual 为解释输入，并只决策一次。坐标面板、刀路视图和 active model 标签复用同一结果。等效角只在 producer 中计算一次并写入 scene；UI renderer 不再做矩阵姿态计算。runtime `unit_per_count` 继续证明控制精度，所有坐标显示的 `display_digits` 固定为 3；producer 统一用第四位四舍五入为三位，并按每轴 `unit_per_count` 过滤 `<=1 count` 的显示往返抖动。旋转轴四舍五入得到 `360.000` 时归一为 `0.000`，并用一圈 count-domain 环形距离判定 `0/360` 相邻；超出一脉冲窗口的相邻显示桶连续两个 fresh generation 一致后提交。轴未使能时，`64 * unit_per_count` 内的跨多桶往返噪声保持上一显示值；轴使能或超窗立即退出 hold。轴使能 actual 只作 producer 内部显示滤波条件且不得进入 SHM；稳定结果不得改变或反写 native/HAL/drive 控制 actual，UI、刀路 renderer、标签和 relay 不得二次滤波、量化或通过 `lv_label_get_text` 反向驱动图形。
- `rotary_phase` 只有在对应 count-domain/profile/unwrap 有效且错误码为 0 时才能覆盖操作员主显示值；无效时不得把有限默认值 `0.0` 当成真实 active model 旋转轴显示值。
- 刀路视图不得绕过坐标面板显示投影直接读取另一套 active model 旋转轴角度。若刀路需要 raw、多圈或 `tool_tip_pose` 做诊断，必须单独标注为诊断字段，不得混入主 active model 显示链路。
- AC 模型下，A 轴红线表示物理 A 旋转轴，不随 A/C 角度转动；C 轴是安装在 A 摇篮上的刚体子轴。A 轴旋转时，整条 C 轴的中心和方向必须一起围绕物理 A 轴旋转，黄色轨迹必须按同一份 A/C 几何依次完成 C 自转和 A 摇篮旋转。以未旋转模型坐标表达时，点的世界变换固定等价于“先围绕初始 C 轴按 C 值旋转，再把该点和整条 C 轴一起围绕 A 轴按 A 值旋转”；不得只旋转 C 轴方向而把 C 轴中心留在原位，也不得按另一套 C 值、raw 多圈值或 commanded 值摆动 C 轴线。
- BC 模型下，B 轴线表示物理 B 旋转轴，不随 B/C 角度转动；C 轴是安装在 B 摇篮上的刚体子轴。B 轴旋转时，整条 C 轴的中心和方向必须一起围绕物理 B 轴旋转，黄色轨迹必须按同一份 B/C 几何依次完成 C 自转和 B 摇篮旋转。以未旋转模型坐标表达时，点的世界变换固定等价于“先围绕初始 C 轴按 C 值旋转，再把该点和整条 C 轴一起围绕 B 轴按 B 值旋转”；不得只旋转 C 轴方向而把 C 轴中心留在原位，也不得按另一套 C 值、raw 多圈值或 commanded 值摆动 C 轴线。
- 上述 AC/BC 变换也允许采用数学等价实现：先把黄色点、C 轴中心和 C 轴方向一起按 A/B 姿态变换，再让黄色点围绕变换后的 C 轴旋转。无论采用哪种计算顺序，非共心轴和中间角度下的世界坐标必须一致；`C=0`、整圈终点或仅起终点重合不能证明实现正确。
- AC 模型下，C 蓝线中点必须是 `(G53_C_X, G53_C_Y, active_wcs_offset_Z)` 的同帧组合 readback，A 红线中点必须是 `(active_wcs_offset_X, G53_A_Y, G53_A_Z)` 的同帧组合 readback。BC 模型下，C 轴线中点必须是 `(G53_C_X, G53_C_Y, active_wcs_offset_Z)` 的同帧组合 readback，B 轴线中点必须是 `(G53_B_X, active_wcs_offset_Y, G53_B_Z)` 的同帧组合 readback。显示层 active model 投影变换必须使用对应模型的 same-generation native scene 组合中心，不得使用 UI 默认值、自建参数表值、旧缓存、设置页表格或非同源快照替代。
- `G53_A_Y`、`G53_A_Z`、`G53_B_X`、`G53_B_Z`、`G53_C_X`、`G53_C_Y` 必须在开机链路进入微内核参数内存；当前 v5 执行路径由 LinuxCNC HAL 启动的 C `v5_native_hal_owner` 在开机/受控 clean restart 时从唯一 runtime INI `[RTCP]` owner 装载并发布 native memory block，同时按底层 owner 公式驱动 kinematics：共同 `rot-point=(G53_C_X,G53_C_Y,0)`；AC `offset=(0,G53_A_Y-G53_C_Y,G53_A_Z)`；BC `offset=(G53_B_X-G53_C_X,0,G53_B_Z)`。UI 刀路只消费由微内核/native active model 与 geometry owner 发布的原始 G53 fresh readback：AC 模型显示 A/C 几何，BC 模型显示 B/C 几何；不得读取推导后的 rot-point/offset 反算 UI 中心，也不得在运行期直接读取 runtime INI、设置页表格、自建参数表或默认常量来补 active model G53 显示。
- MCS ready 后的首次有效绘图中，active model 旋转轴线空间位置必须使用和底部状态行一致的同帧组合中心。AC 模型中 `A.X = active_wcs_offset_X`、`C.Z = active_wcs_offset_Z`，其它分量来自直连微内核/native 的 A/C geometry；BC 模型中 `B.Y = active_wcs_offset_Y`、`C.Z = active_wcs_offset_Z`，其它分量来自直连微内核/native 的 B/C geometry。C 轴线初始方向与 MCS Z 平行，后续只能按同一条 UI active model 显示投影中的第一旋转轴值围绕第一旋转轴线做空间旋转。冷启动首个可见主页面必须等待有效机械坐标、native active model/geometry 和 active WCS offset，并用同一帧真实 owner readback 建立 active model 姿态、红/绿 actual 点及真实刀柄线；任一必要输入缺失时 UI 不得绘制非真值视觉骨架、默认旋转轴姿态、种子点或声明几何 actual。
- C 机械值只表示绕自身轴的相位/诊断，不得改变 C 轴线方向、端点方向或屏幕方向。XY/XZ/YZ/3D 四个实际绘图视角都只是同一份 MCS world scene 的不同角度快照，不能按视角重新创建或重算 active model 几何关系。
- 屏幕红线和 UI 像素坐标不得作为 active model 旋转轴线显示中心替代。active model 几何只允许来自直连微内核/native 几何 provider，没有有效 native active model/geometry 时不得从设置页表格内存或旧 SHM 字段拼出显示几何；运动中的 active model 姿态变化只由同一 SHM UI 显示帧的对应旋转轴机械坐标驱动。
- XY 视图中只有同一条 UI active model 显示投影中的第一旋转轴值为 0 度等效姿态时，C 轴可显示为点；第一旋转轴显示值非 0 时不得显示这个退化点，C 轴必须按同一份三维 scene 的正常投影线随 active model 姿态变化。XZ/YZ 视图始终按实际 C 轴三维线段显示，不能为 active model 旋转轴线、黄色路径、WCS/MCS 轴线或刀柄/刀尖创建另一套独立绘图逻辑。
- `cmd_mcs[3]` / `cmd_mcs[4]` 只用于跟随误差和诊断。
- 连续旋转解包必须有明确策略或 manifest；unknown 时不猜，不把 `G90 C0` 由 UI/刀路层自动解释成 `C360`。但当 `功能/3-3旋转轴等效角目标与解决方案.md` 的 native rotary owner 已判定 `G90/G53 A0/C0` 属于一圈内相位/回零目标时，执行和显示必须按最近等效零处理，不得退回 raw 机器 0，也不得由黄色线显示层重写程序。
- active model 旋转轴或程序点相邻样本出现接近 360 度整倍数的大跨度、unwrap 边界或等价屏幕投影大跳时，只允许在显示线段上断开连接，形成多个绘制 segment；不得修改程序点、不得重写 active model 旋转轴 actual/commanded、不得用滤波后的姿态替代坐标面板同源旋转轴显示机械值。断线原因应进入诊断或测试可见的 segment metadata，避免后续误判为丢点。

### 3.6 SHM 视觉稳定

SHM 结构错误继续 fail-closed；短暂 seqlock 冲突允许显示层使用上一帧 grace：

| 状态 | UI 行为 | 控制门禁 |
| --- | --- | --- |
| `last_good_age_ms <= 300` | 继续显示上一帧并标 `SHM stale` | 不得复用 stale epoch 放行 |
| `300 < age <= 500` | 隐藏强依赖动态图层，保留静态程序刀路和 degraded | fail-closed |
| `age > 500` | 显示 `SHM unavailable`，动态图层隐藏 | fail-closed |

### 3.7 显示刷新节流

刀路 view 可以做显示层 FPS 自适应节流，但不得影响 SHM 读取、状态应用、命令 freshness、Broker 请求、State Publisher 采样或任何安全/运动门禁。

- Program Open/Load 成功并出现黄色 G-code 轨迹后，screen fit 必须以当前 scene 的全部可见元素为候选集合重新计算，至少包含黄色刀路、WCS/MCS、active model 几何、红点、绿点、刀柄线和标签锚点；不得只按黄色线局部 fit，也不得沿用开机首次 fit。
- XY/XZ/YZ/3D 四个视角按钮属于明确的前台结构边界：点击任一视角按钮必须取消当前手势，清零旧平移/旋转并把旧缩放恢复为 `1.0`，恢复该按钮对应的默认视角投影，并按新投影平面和 canonical viewport 只计算一次居中 fit；不触发状态层结构或黄色程序模型重建，也不得留下后台 fit 任务。

- 刀路热路径分为 producer 的动态 scene generation 和结构缓存。动态显示目标保持 30Hz，包含坐标、当前轨迹姿态、红色 commanded/theory 点、绿色 actual 点、刀柄/刀尖、转速/进给/倍率和跟随误差；UI 只消费完成的 snapshot，不执行这些字段的派生。
- 黄色 G-code 程序层、WCS/MCS、active model 几何和标签与动态点必须由 producer 在同一 scene generation 一次统一投影并给出 dirty bounds。观察到 10Hz 与 30Hz 都拉满 CPU1 后，禁止继续用“最多 10Hz”作为优化结论；scene 生成只能由 generation bundle 变化驱动，同一 key 的 timer tick 必须 cache hit，不能重复旋转、project、对象重写或 payload build。30Hz 是可见动态目标，不是强制每 33ms 重算未变化数据；运行期仍只 flush 实际 dirty 区域。
- 静态 G-code 程序模型只在 Program Open/Load 成功时由 producer 解析/缓存；首个有效 scene、黄色程序模型完成或操作员明确切换 XY/XZ/YZ/3D 视图时计算一次 fit 并冻结当前边界。运行期 producer 只把同一 scene 的新可见候选投影到这份冻结边界做越界判定：候选仍在画面内时必须保持 `fit_generation` 和边界不变；候选真正越出画面时，才为该越界事件重算一次全部可见元素的 fit、推进一次 `fit_generation` 并立即冻结新边界，以后再次越界才允许再次触发。手势、缩放、平移和窗口变化只更新 `view_generation`，不得伪造越界或推进 `fit_generation`；RTCP OFF 不执行 RTCP 黄色轨迹变换，RTCP ON 只按同 generation actual 计算一次。两种状态都不能重新解析 G-code、无越界按帧更新 fit 或改写 active WCS offset。
- 主页面轨迹区域必须支持 v3 操作员视图同类手势：raw evdev 层至少收集两个同帧触点，触点均在轨迹图形区内时由刀路 view 消费；双指距离变化只更新显示缩放，双指中点移动只更新显示平移，双指角度变化只更新屏幕显示旋转或在当前 v5 投影不支持旋转时保持无害忽略。手势 transform 必须同帧作用到黄色 G-code 进给线、红色 commanded 点、绿色 actual/holder 点、MCS/WCS 原点和轴线、active model 几何及标签；不得只缩放黄线或只移动动态点。手势不得读取 G-code 文件、不得重算 screen fit、不得重建状态层结构、不得用设置表/profile/旧缓存补真值，离开轨迹图形区或手指释放时只结束手势状态，不得清空已确认图形。
- 状态层已经由有效 SHM UI 显示投影和必要 native 状态绘出后，短暂 `status_stale`、单帧 `mcs_ready=false`、单帧投影失败或单帧 active model geometry 无效不得调用清空函数把上一帧线对象置空。UI 只能保留上一帧已确认图形并更新正式 degraded 状态，等待下一帧有效状态覆盖；不得在刀路画布底部显示 `diag:`、`rtcp--`、`geom--`、`holder--` 等内部诊断 token。
- 前台状态文字遵循同一保留规则：WCS offset 摘要、active model 几何摘要、runtime modal/RTCP 文本和标签锚点只能由有效 SHM 显示投影或 native 状态 provider 覆盖；冷启动首帧必须先通过 SHM 门槛，不能显示模板值或默认 0。UI 已经取得 last-good 后，单帧缺字段、WCS offset 无效/非有限值、WCS 全表未 ready、RTCP geometry 未 ready 或 runtime modal 未 ready 时，必须保留上一帧有效文字或保持空白，不得显示 `active model geom --` 或内部诊断占位冒充真实数据。
- 清空状态层只允许发生在明确的对象销毁，或运行期长期 SHM 显示投影不可用并进入明确 fail-closed 空场景的路径；冷启动 UI 在 SHM 门槛通过前没有可见状态层。页面重新创建时若仍有 resident last-good scene/cache，可先恢复该已确认帧再局部刷新；没有 last-good 时必须重新等待有效 SHM，不得显示非真值骨架。更换程序只允许清理黄色轨迹，不得清空或重建状态层。清空状态层不得出现在 30Hz 普通刷新路径里。
- G-code 运行时，主页面无遮挡的坐标、红点/绿点、当前刀路位置、黄色程序层、WCS/MCS、active model 几何与标签目标是按 fresh scene 及时显示，动态目标为 30Hz；完整变换只由 producer 对新 generation 执行一次，不再规定 10Hz 全量重投影。唯一 producer 可以用已经生成的本帧可见候选对冻结 fit 做一次有界越界判定，但未越界不得重算 fit，越界后只提交一次新 fit 后立即冻结；刀路 UI 不得触发 G-code 重解析、第二次 fit candidate 扫描、边界扩展、auto-refit、全量 scene 重建、全屏刷新、诊断落盘或慢状态校验。页面隐藏时允许 producer 更新 resident snapshot freshness/generation，但不得为隐藏页执行越界 auto-fit，也不得调用该页 LVGL renderer、fit、timer、cache invalidate 或重复黄色轨迹投影。
- Program Open/Load 后台任务必须把操作员打开的原始 G-code 注册为 resident identity/handle，并由唯一 display projection producer 生成 model-neutral 黄色程序模型；`runtime_program_path`、resident `source_path` 和最终 `Set Open` 继续指向原始文件，不得指向过滤文件、sidecar、运行副本或预览产物。路径存在、metadata 写入、解析任务启动或 loading 都不能冒充“可 Start”；只有 resident program model、全部可见元素组成的首个完整 scene 和对应 screen fit 都完成，且同一 program generation 的 `program_scene_ready=true` 与非零 `scene_generation/fit_generation` 原子发布后，程序才算“可 Start”。Start/Resume 只读取该 resident ready token，不得再读文件、补建 resident handle、重算 scene 或执行 fit。程序点保留自身 G54-G59.3 模态，producer 使用对应 native/interpreter WCS offset 一次性进入机械坐标 scene，缺对应 offset 时 fail-degraded，禁止默认 G54 或 active WCS 二次套用。大文件解析期间 UI 只显示 loading，resident program model 与首次完整 scene/fit 就绪后隐藏。30Hz reader 不得重新解析程序；同一 generation 只由 producer 生成一次最终黄色轨迹和全部可见元素统一 scene。RTCP OFF 不执行 RTCP 变换且 WCS 图形不随 active model 旋转；RTCP ON 按同 generation actual 一次生成 WCS-follow。UI 只通过唯一 custom-draw scene 绘制，旧多 `lv_line` 数组和 `lv_line_set_points()` 路径必须删除。
- active model 旋转轴显示和刀路姿态的 30Hz 输入由同一 SHM scene snapshot 提供；轴字母和 slot 由同一个 descriptor/readback 给出，AC 消费 A/C、BC 消费 B/C，未知模型 fail-degraded。unwrap/count-domain/display phase 原始 actual 仍由 producer 直连 native 使用，不作为独立 SHM 控制字段；UI 只取 producer 输出的同源旋转轴显示值和最终 scene。只更新 X/Y/Z 而不更新同 generation 旋转轴/scene 属于失败。
- 无 G-code 或程序层为空时，WCS offset、holder、active model geometry 等内存字段变化只能作为投影参数或诊断文本输入，不能成为状态层结构或 fit 的重建签名。坐标微抖、对刀点动、单帧 SHM 数值抖动只能更新动态点或后台诊断状态，不得触发全量 scene/signature 重建。
- 机床空闲、无触摸/手势且 generation bundle 未变化时，producer 和 UI 都必须 cache hit：不重算 scene、不重写 LVGL 对象、不产生 remote dirty。fresh source metadata 仍须进入 reader，用于判断 stale/fail-closed。
- 触摸/手势、机械坐标、RTCP actual、红绿点或 degraded 状态变化只推进对应 generation，由 producer 生成一份新 scene；UI 仅按 scene dirty bounds 刷新。普通动态变化未越界时不得推进 `fit_generation`；真正越界时只为该事件推进一次，提交后冻结新边界。XY/XZ/YZ/3D 视角切换取消旧手势并推进 `view_generation/fit_generation`，只在该动作计算一次新 fit。
- 节流只允许跳过重复绘制，不得跳过 fresh source metadata；它不能替代 producer 去重，也不能影响命令、安全或运动 freshness。
- 必须记录同一窗口的 source/scene generation、producer cache hit/miss、sample/RTCP-transform/project 次数、UI scene consume/object rewrite/dirty area、黄线点数和 relay capture/payload-build/send 次数；退役的多 `lv_line` 对象数组和 `lv_line_set_points()` 更新入口必须在迁移到唯一 custom-draw scene 后物理删除，不留 wrapper、disabled 分支或测试入口。
- scene 的现有 `flags` 必须在不改变 `V5StatusDisplayScene` 大小、字段偏移和 v3 版本的前提下声明本 generation 实际改变的显示图层，至少区分黄色程序/WCS 静态投影层、active-model 几何层以及红绿点/刀柄动态层。UI 必须将静态投影层与动态覆盖层拆成引用同一 canonical viewport 和同一 immutable scene 的独立 custom-draw 对象；只变动态层时不得 invalidate 或执行黄色程序层 draw callback。RTCP OFF 下旋转姿态只允许更新 active-model 几何和动态点；RTCP ON 下黄色程序/WCS 屏幕点真实变化时必须标记并正确刷新静态投影层，不能用刀尖小框掩盖整条黄线变化。图层的旧、新位置都必须进入 dirty 并集；首帧、view/fit/结构变化或缺少图层 dirty 证明时必须 fail-safe 刷新全部 scene 图层。
- 黄色程序点的减点只能发生在 producer 已完成 world-to-screen 之后，并必须保留 `break_before`、程序/WCS identity 和屏幕像素等价；第一阶段只允许删除投影到完全相同整数像素且不会跨 segment break 的连续重复点。更广泛的 `0.5px` 曲线简化必须另有最坏误差证明和板端 profile，不能作为 C02 的正确性前提，也不能用潜在 `O(N^2)` 简化替换当前有界 `O(N_visible)` 投影。

### 3.8 与 State Publisher / SHM 热路径的联动

State Publisher、display projection producer、UI renderer 和 relay 必须作为一条显示热路径整体收敛。任何单点 `mmap.flush()`、nice 或降频调整都不能替代同 generation 一次生成、多方只读消费。

#### 3.8.1 黄色轨迹单帧作业顺序

黄色轨迹必须按以下固定顺序完成，步骤不得拆给多个 UI 控件或远程连接重复执行：

1. producer 锁存一次 `source/program/native/active_model/rtcp/wcs/view/fit generation` 组合；任一必需输入无效就产生 degraded/invalid，不得拼接上一代局部字段。
2. producer 只引用一份 resident program model 和一份 immutable native source view。RTCP OFF 直接跳过黄色轨迹 RTCP 变换；RTCP ON 对本 scene key 的全部黄色点只遍历和变换一次。
3. producer 在同一次 scene build 中生成最终屏幕坐标、segment break、黄色轨迹、WCS/MCS、active-model 几何、红绿点、刀柄、标签锚点、跟随误差和 dirty bounds，并提交一个 `scene_generation`；不能先给黄色线生成一份，再让坐标面板、标签或 renderer补算另一份。
4. State Publisher 对该 `scene_generation` 只完成一次 typed ABI 组帧、CRC 和 SeqLock commit；30Hz timer 只发布或复用当前最新完整 scene，不为相同 key 重建。
5. UI shell 每个动态周期只读一次 SHM，把同一个只读 `V5UiStatusView/V5DisplaySceneView` 引用传给坐标面板、custom-draw 刀路、标签、跟随误差和 framebuffer dirty 输出。各消费者只能读取自己需要的字段，不能复制成第二份可变模型，也不能调用投影器。
6. remote relay 只读取 UI 已提交的 dirty framebuffer，并对同一 `frame_id` 准备一份 payload；远程连接数量增加只允许增加必要的 socket send，不得增加 scene transform、project、framebuffer capture 或 payload build 次数。

若 producer 尚在生成旧 key 时收到更新的 generation，只保留最新请求并丢弃未发布旧结果，不建立待补算 scene 队列。验收计数必须满足：`source_sample_count <= source_generation_count`、`scene_build_count <= unique_scene_key_count`、RTCP OFF 的 `rtcp_transform_count = 0`、RTCP ON 的 `rtcp_transform_count <= scene_generation_count`，且 widget、页面或 remote subscriber 数量变化不改变这些 producer 计数。

UI 到 resident scene producer 的 datagram IPC 不得把本地 `sendto()` 成功或 UI 自己记录的 last-request 当作完整程序模型已接收证明。只有 fresh typed scene 回显同一非零 `program_source_identity/program_generation` 后，后续视角/fit/页面可见状态请求才允许缩成不含点列的 `VIEW_UPDATE`；回显尚未匹配时，每次重试都必须继续携带完整 bounded `PROGRAM_MODEL` 点列。初次发送无论成功或失败都必须先记录本轮 attempted key，随后只按 `100ms/200ms/400ms` 三次有限退避重试，不能每个刷新周期重新进入“初次发送”，也不能无限重发。首次完整模型消息被丢弃的 focused test 必须证明下一次重试仍为完整模型、producer 接收后保留全部点列，而 ack 后的 `VIEW_UPDATE` 只更新 view/fit/页面可见状态且不清 resident program model。IPC reader 在合并请求前必须 fail-closed 拒绝：任一点任一轴、viewport/scale/sine/cosine/pan 中的非 finite；超出 G54-G59.3 范围的 `wcs_index` 或 mask；非 `0/1` 的 `break_before/page_visible`；无效消息类型、点数、宽高、plane 或 generation/identity 组合。

Producer/State focused contract 还必须覆盖：完整相同 scene key 连续输入 100 次不推进 scene/project；相同显示值连续 100 个 fresh source generation 不推进 scene/project；同一持续越界输入 100 帧只允许首次真实越界推进一次 fit，隐藏页面同样 100 帧越界的 fit 推进计数必须为 0；未知机型不回退 AC/BC 并输出 degraded/invalid model scene；scene 生成失败只清 `V5_STATUS_VALID_DISPLAY_SCENE/scene_generation`，不得清除同一 Position sample 已发布的 MCS/CMD 坐标 valid 和数值。

- RTCP actual 缓存是 native/provider 内部的同源回读优化，不是 UI 真源。运行 G-code 时可复用上一帧有效 RTCP modal actual；RTCP 开关、状态异常、无有效缓存、缓存过期或读回失败时必须重新采样微内核 modal owner，并返回 invalid/unknown/degraded，不得沿用缓存冒充正常。RTCP native 状态源以 `REQ-RTCP-NATIVE-STATUS-SOURCE` 为准，本文不重复定义 helper/HAL pin/adapter 形态。
- `mmap.flush()` 只能作为 State Publisher 性能问题的 A/B 排查项，不是 UI 刀路修复路线。若板端证据支持关闭 steady-state flush，必须删除旧路径并补当前 canonical 实现，不得保留配置开关、默认关闭分支或周期性 status/perf JSON 写盘。不得删除 seqlock、dual-page header、CRC、retry 或 degraded/fail-closed 逻辑；排查前后必须采集 State Publisher 返回样本的 `timings_ns`、UI stale/degraded 计数、SHM epoch/CRC 异常、坐标刷新和 Broker 急停响应。
- State Publisher、UI 和 remote relay 的 affinity/cpuset 必须始终排除 CPU0，不得提升为实时线程，也不存在通过板端 A/B 证据后允许进入 CPU0 的例外；普通调度参数只能在 CPU1 非运动域内调整。CPU1 满载时应合并或丢弃陈旧显示帧、降低非安全日志/诊断和对网络施加背压，允许 UI 降帧或卡死，但不得让显示链抢占 CPU0、迁移非运动 IRQ/workqueue、降低 servo 周期或使运动停顿。
- 当前根改动就是把坐标/跟随误差/RTCP/黄色轨迹的重复计算收敛到唯一 display projection producer，并通过 typed SHM scene 让 UI 直接消费；不得以“先优化 UI、以后再考虑 publisher/native”推迟该 owner 迁移。迁移完成后旧 UI projector、旧多 `lv_line`、重复状态 provider、重复 scene cache 和 fallback 必须同 slice 物理删除。
- 性能结论必须同时包含 producer sample/derive/RTCP-transform/project、publisher commit timing、UI scene consume/object rewrite/flush area、relay capture/payload-build/send、每核 CPU、任务/IRQ affinity、servo latency/overrun、EtherCAT DC/WKC、编码器连续性、Broker ready/急停响应和最终安全态。必须用相同程序、视角、RTCP 状态、远程连接数和窗口分别跑黄色轨迹 10Hz/30Hz；两者都不再因重复计算拉满 CPU1，且 active-model 连续运动中故意冻结 UI/压满 CPU1 后运动仍连续，才可关闭。

## 6. 实现检查点

实现或审查时按以下顺序检查：

1. 一帧状态输入是否先收敛为 scene，再刷新所有图层。
2. G0/G53 是否被排除出黄色主刀路。
3. WCS 切换是否只更新 display transform，不 reparse G-code。
4. RTCP 按钮是否只通过 `native_rtcp_control` 直通微内核控制 latch，并由 UI/provider 直连微内核/native RTCP actual 状态显示；request/dirty/非同源 helper/cache/旧 SHM 字段是否只显示 pending 或诊断；上一帧有效 RTCP 缓存是否在开关、异常、无有效缓存或缓存过期时强制重读。
5. active model 旋转轴 raw、unwrapped、display phase 是否分离，unknown 策略不自动解包。
6. fit-lock 是否在开机首次有效 SHM 状态层、新程序黄线出现和明确视角切换时完整重建；新程序黄线出现后的 fit 是否纳入黄色刀路、WCS/MCS、active model 几何、红点、绿点、刀柄线和标签锚点全部当前可见元素；运行期可见 scene 未越界时 fit generation 是否保持冻结，真正越界时是否只重算一次并冻结新边界，以后再次越界才再次触发；隐藏页面是否完全没有越界 auto-fit。
7. seqlock 短冲突是否只影响显示 grace，不影响控制 epoch。
8. 30Hz 坐标/轨迹动态路径是否只更新动态层和投影输出；静态黄线程序模型、WCS/MCS、program WCS、active model 轴线和 fit candidate 是否不会因任何运行状态被重建。
9. 无 G-code/空程序状态下，WCS/holder/active model 浮点微抖是否只影响投影/诊断，不会改变结构 signature 并触发全量 scene 或 fit 重建。
10. 大程序预览抽样是否保留关键点、按 LOD/segment 断开不可证明连续的连接、标出 decimated 诊断，并且不修改 G-code 或运动真值。
11. active model 旋转轴 wrap、大角度和屏幕大跳是否只断开显示线段，不改程序点和 actual/commanded。
12. fresh active model 是否在 scene 收敛阶段只解析一次，dispatcher 是否恰好选择一个 branch，后续 geometry/fit/render 是否只消费 model-neutral scene 而没有散落模型判断。
13. AC/BC branch 是否互不调用、互不共享模型专属状态；共享源码是否不存在 A/B/C 轴、G53 中心或 generic first/second 旋转公式；未知模型是否 fail-degraded 且不沿用上一 branch。
14. FPS 节流是否只跳过显示重绘，不跳过 SHM fresh 显示状态进入 UI model；控制 freshness、动作证明和安全门禁必须直连 native/microkernel readback，不由 UI 刀路层或 SHM epoch 提供。
15. 若同时调整 State Publisher `mmap.flush()` 或 nice，是否有 A/B 证据证明 SHM 一致性、Broker 急停和实时线程没有退化。
16. 新逻辑是否落到对应 owner，而不是继续堆到大 view 文件。

只改本文档不执行板端。修改 toolpath 源码后，声明 `fixed`、`done`、`verified` 或 `board_verified` 必须按 `AGENTS.md` 执行板端可见功能规则。

- 产品 UI 构建产物和部署身份。
- 原始 UI/operator 路径截图，截图必须来自 remote relay/LVGL flush framebuffer 或同等级已验证颜色路径。
- 截图中能看到黄色主刀路、红色 commanded 刀尖点、绿色机械点、WCS/MCS、active model 旋转轴或 RTCP 相关显示。
- 本文所有进入 `D:\v5\board` 的 UI 刀路程序修改都消费 `REQ-BOARD-PROGRAM-FULL-BUILD-CLOSURE`：关闭时必须由 VM 直接只读挂载当前完整 Windows `RUNTIME_SOURCE_ROOT`，在 `VM_BUILD_ROOT` 全量编译并部署板端 UI，再走真实程序页打开 G-code -> 返回主页面 -> 视角按钮切换的 operator 路径；黄色线 `1px`、XY/XZ/YZ/3D 四视角切换取消旋转并恢复默认视角、G-code 黄色轨迹载入后全部当前可见元素统一 fit，必须用板端截图/录屏证明同时成立，本地 smoke 只能作为前置门禁，不能替代板端闭环。
- 若涉及性能热路径，必须同时采集 publisher timing、UI perf/flush area、toolpath cache hit/miss、LVGL line rewrite 次数、每核 CPU、Broker ready/急停响应；G-code 运行中不得出现 publisher/刀路互相争抢导致的明显 5Hz 卡顿。
- 若涉及运动，按 `REQ-AUTO-STATE-DRIVEN` 跑 native active model 对应的 `cc-ac.ngc` 或 `cc-bc.ngc` 黄金运动闭环，并保留截图、对齐分析 JSON、黄金测试结果 JSON 和最终安全停机状态。

model-specific `cc-ac.ngc`/`cc-bc.ngc` 采用“三条链独立正确、重合仅作结果验证”的验收口径：

- `cc-ac.ngc`/`cc-bc.ngc` 必须独立满足预定弹簧几何、active model 对应旋转轴语义和 RTCP 开关时序；不得为了追随某次 UI 或板端误差修改程序轨迹。
- UI 黄色轨迹必须独立使用正确的空间刚体旋转顺序、旋转中心、同源 active model 几何和同帧旋转轴显示机械值完成投影；不得读取刀尖误差后反向平移、吸附或修形黄色轨迹。
- 微内核/LinuxCNC/native 运动学必须独立按同一物理 active model、几何参数、刀长和 RTCP actual 正确计算并执行刀尖运动；不得为了追随屏幕黄色轨迹增加显示补偿或专用偏移。
- 红色 commanded 刀尖、绿色 actual 刀尖与黄色弹簧轨迹在运行中的重合，是以上三条链均按预期工作的交叉结果和验收证据，不是任何一条链可以直接追逐的实现目标。出现不重合即表示 `cc` 程序、UI 轨迹变换或微内核运动学中至少一条未按预期工作，问题保持未关闭，必须分别核对三条链并定位根因；不得用一条错误链补偿另一条错误链，也不得只把起点、终点或少量样本调到重合后宣称通过。

最终对齐限额为机器空间误差不超过 `1.0mm`、屏幕投影误差不超过 `5px`。该限额只用于验证静态几何、运动学与投影计算公式是否正确，非运行时重绘的实时像素精度限制；不得通过 UI 吸附、样本中位偏移、手工估计、跨链补偿或修改 G-code 让它看起来重合。

## 9. 一票否决

- 修改 G-code 来修显示。
- G0/G53 进入黄色主刀路。
- UI 自算 RTCP 真值并当 actual。
- RTCP request、dirty 或按钮状态刚发出就切 active 图层。
- active model 旋转轴显示归一化写回 HAL、INI、驱动或 G-code。
- 在 shared dispatcher/geometry/fit/renderer 中保留或新增 A/B/C 模型公式、generic first/second 旋转顺序、散落 `if AC else BC`，或让一个模型 branch 调用/写入另一个 branch。
- 未知模型、ops 缺失、descriptor/ops id 不一致或 branch generation 不新鲜时回退 AC/BC、默认 0、上一模型 pose/geometry/fit cache。
- 新增模型时修改 AC/BC branch 塞条件，而不是新增唯一 registry entry、独立 branch 和独立测试。
- unknown 连续策略下擅自 unwrap。
- RTCP ON/OFF 下黄色轨迹不按微内核/native active model readback 对应的同源几何随坐标面板同源显示机械值旋转，或使用 RTCP 状态切换出另一套 raw、多圈、commanded、滤波、种子角度来驱动黄色轨迹投影。
- RTCP OFF 下当前加工坐标系原点、X/Y/Z 轴方向跟随 active model 显示旋转输入旋转/摆动；RTCP ON 下当前加工坐标系没有随同一帧 active model 姿态旋转；任何模式下 active WCS offset 被 UI 图形投影改写。
- display phase、滤波姿态、raw 多圈值或 commanded 值被当成黄色轨迹主显示几何。
- 将 stale-cache 的 `status_epoch` 用于控制请求；`status_epoch` 只允许表达 SHM 显示投影 freshness，不得进入控制请求、动作证明、安全门禁或 native readback。
- 大程序预览抽样无诊断、无关键点保留证明，或把抽样结果冒充完整轨迹。
- 大程序触达点数上限后把终点塞回当前连续折线，产生跨截断长拉线。
- 30Hz reader 或任何 UI widget/renderer 全量重投影黄色轨迹、WCS/MCS、program WCS、active model 几何，或保留/恢复退役的多 `lv_line` / `lv_line_set_points()` 路径。
- G-code 运行、暂停或恢复期间在候选仍位于冻结画面边界内时重复 fit，单个越界事件连续推进多次 `fit_generation`，在 UI/renderer/relay 侧重复越界判断，或在页面隐藏期间执行越界 auto-fit、后台预先计算下一次 fit。
- 用 `mmap.flush()`/nice 调参结果替代 UI 刀路动态/静态分层修复。
- FPS 节流影响 SHM 读取、显示状态应用，或被误用于命令 freshness、安全/运动门禁。
- UI widget、renderer、本地 cache 或 remote relay 再次计算坐标、跟随误差、RTCP、active-model 姿态、黄色轨迹或 world-to-screen，而不是只读同一 immutable scene。
- 将黄色轨迹从 30Hz 降到 10Hz 后未证明 generation 去重和 CPU1 同窗口改善就宣称完成。
- 坐标、跟随误差、模型或刀路未统一消费唯一 producer 的三位一脉冲及未使能轴64 count有界滤波结果；旋转轴未按环形距离处理 `0/360`，未在使能/超窗时退出 hold，或 UI/renderer/relay另建滤波，造成重复状态或把显示值反写控制 actual。
- 恢复旧 JSON/FIFO/native daemon fallback。
- 恢复或新增历史 v3 刀路源码 owner、兼容壳、wrapper 或 shadow path。
- 在当前 v5 主页面大文件里继续堆几何、投影、fit、诊断或手势状态机，而不是下沉到明确的 `v5_toolpath_*` / `v5_main_page_*` 子 owner。

## 10. 关闭条件

关闭 UI 刀路问题时必须说明：

- 修改范围：文档、源码、板端和 G-code 是否被改动。
- 图层证据：黄色主刀路、红点、绿点、WCS/MCS、active model 旋转轴、RTCP。
- 状态来源：同 generation typed SHM status/scene snapshot、resident program identity、WCS/TLO/runtime modal/RTCP native actual 和 degraded 分支；RTCP request/dirty/非同源 helper/cache 是否只作为 pending 或诊断；publisher/producer 在开关、异常、无效或过期时是否强制重读同一 native owner。
- 精度证据：每轴 runtime/native/HAL `SCALE`、`unit_per_count` 与固定 `display_digits=3` 分离；所有坐标只消费唯一 producer 用第四位四舍五入为三位、按 `unit_per_count` 过滤 `<=1 count` 显示往返抖动、并对未使能轴应用64 count有界hold的结果；旋转四舍五入到 `360.000` 时归一为 `0.000`，`0/360` 环形相邻、未使能有界噪声、使能/超窗退出测试通过，且 native/HAL/drive actual 在滤波前后逐样本一致。
- 性能证据：相同程序、视角、RTCP、remote subscriber 和采样窗口下的 10Hz/30Hz CPU1 ticks，producer cache hit/miss、sample/RTCP-transform/project、UI object rewrite/dirty area、relay capture/payload-build/send；计数必须绑定 generation 而不是 timer/widget/subscriber。
- 验证结果：本地 smoke/contract/截图；若上板，还要有构建产物、部署、截图、日志和最终安全态。
- 未闭合事项只记录真实缺口，不写过程日志或聊天摘录。
