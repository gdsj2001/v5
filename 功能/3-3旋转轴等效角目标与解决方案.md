## 1. 问题和目标

旋转轴同时存在两套语义：

```text
raw_angle_deg   = LinuxCNC/HAL/EtherCAT/驱动/编码器真实累计角度
phase_angle_deg = raw_angle_deg 折算后的显示相位角，360 等效按 0 显示
```

用户希望 A/C 转到 `360`、`720`、`-360` 等等效零位时主界面显示 `0`；但 LinuxCNC、HAL、EtherCAT、驱动和编码器看到的是连续累计 raw 位置。V3 不能为了显示归零静默修改 LinuxCNC 坐标、驱动位置、绝对编码器或 HAL 真值。

目标：

- UI 显示 active model 对应旋转轴的等效角，运动真相仍归 LinuxCNC/HAL/EtherCAT/native。
- 普通定位、First point、UI 等效回零走最近等效 raw 目标。
- 已由 native 判定为非连续累计段的 G90/G53 active model 对应旋转轴一圈内绝对相位目标（`abs(target_deg) < 360`）走最近等效 raw 目标；AC 模型下的 `A0`/`C0` 或其它模型对应轴的零目标走最近 `k*360` 等效零，不得落回 raw 机器 0，不得用 G92 或 UI/Broker 改写源 G-code。连续加工段、显式累计段、`abs(target_deg) >= 360` 多圈、G91 增量、宏表达式、圆弧旋转或语义不明目标不得被上层折叠。
- 连续多圈 G-code 保留连续 raw 语义，不被折叠。
- 微内核/native rotary owner 负责 `rotary_equiv_zero` 和 active model 最近等效零位快速定位；文档不再要求打开额外的旋转包角 mask，也不得把某个 mask 是否存在作为产品需求或通过门禁。实现可以在底层采用等效目标算法，但成功证明必须来自 native rotary owner 的规划、执行和 readback，而不是 UI/Broker 改写目标或检查某个配置开关。
- 最近等效目标不能只改当前一个 `G0` 块。微内核/motmod motion owner 必须为 active model 的每个旋转轴常驻维护 `logical_to_motion_turn_offset_deg = k * 360`：每个带完整 `EmcPose` 目标的后续直线、圆弧和省略该旋转轴 word 的程序段，都先把解释器逻辑目标加上当前整圈偏移；只有已判定为 `phase_nearest` 的 traverse 目标才允许按 planner 当前连续目标选择最近等效角并原子更新该偏移。`continuous_delta`、显式多圈、G91、进给和省略轴 word 的后续块只消费已有偏移，不得再次折叠。例：raw C 已到 `-1800` 时执行逻辑 `G53 G0 C0`，物理目标仍为 `-1800` 且偏移更新为 `-1800`；下一块即使未写 C，也必须继续保持物理 C=`-1800`，不得回卷到 raw 0。该偏移只能由 native motion 生命周期 owner 在成功重新回零或 canonical task/interpreter 坐标同步时清零或重建：LinuxCNC `EMCMOT_ABORT` 若对应 task abort、程序打开/重置、M30/Stop 后的 interpreter sync，就必须在下一个完整 `EmcPose` 目标入队前原子清零旧 offset，避免已同步为物理连续角的目标再被重复加一遍；不触发 interpreter sync 的普通模式切换、Pause/Resume 和 planner idle 不得自行清零。不得由 UI、Broker 或文件改写 offset。offset 未初始化、非整圈、active model 变更未完成 clean restart、目标或生命周期同步不可信时必须 fail-closed，不能用旧 offset 猜测成功。
- BUS/EtherCAT 禁止在 planner idle 时用任何一圈内相位归一化机制重写 joint/motor command、motor offset、驱动位置基准或绝对编码器累计值；否则会在连续段结束时制造大命令跳变、关闭 motion，并使后续 `A0`/`C0` 根本进不了队列。BUS 的 raw/count/drive command 必须连续保留，UI 的 0-360 显示只消费 native/provider `phase_deg` readback。若 Pulse 将来确需无脉冲窗口重整，必须使用独立参数、独立 native Rebase gate 和同周期无脉冲证明，不得复用快速/回零类最近等效目标机制。该 native 一圈内相位目标只用于快速/回零类最近等效目标，不得折叠连续切削段的多圈 G-code。
- active model 对应旋转轴的绝对多圈恢复和告警以整数 count-domain 为第一真相。
- 缺 raw、缺 count-domain、状态 stale、目标不明确、软限位越界、native gate 失败或到达证据不足时 fail-closed。

主界面坐标显示不是运动准入。active model 对应旋转轴坐标显示只能消费允许的 SHM 坐标显示投影，或在需要等效角解释时直连 native/provider readback 取得同源 `phase_deg`；不得把 `rotary_*` count-domain、unwrap、wcheckpoint、range guard、invalid reason 或维护信号新增进 SHM。只要 native/provider 给出有限等效角，就必须直接显示该精确等效角；不得等待回零、不得用 profile、home 或门禁状态把已有坐标隐藏成空白，也不得回退显示机械 raw 角。无用的显示门禁必须删除，不保留“不可用”替代分支。

## 2. 真相边界

| 层 | 可以做 | 禁止 |
| --- | --- | --- |
| UI C | 显示允许的 SHM 坐标投影，必要时直连 native/provider readback 取得等效角解释；发送明确动作请求 | 计算设 0、unwrap、Rebase、运动目标、range guard、invalid reason 或写驱动 |
| Broker/dispatcher 薄入口 | 只做请求封包、身份/权限/参数格式校验、串行转发、结果回传；部署确需 Broker 时也只是 native gate 的薄入口 | 自己规划、自己门禁、自己证明到达、猜 LinuxCNC/驱动真相、硬编码厂家 SDO |
| State Publisher / UI projection | 只读 LinuxCNC/native/drive evidence，并只发布 0-4 允许的 SHM 显示投影 | 写运动参数、写驱动、发布 rotary proof/invalid reason 或把 SHM 投影当证明 |
| LinuxCNC/HAL/EtherCAT/native | 运动和状态最终真相 | 被 V3 JSON/cache/UI 显示值替代 |
| Settings Drive 私域 | 保存驱动 profile、count-domain 校准、电子齿轮和读回证据 | 保存 WCS/G53/RTCP/TLO/G92/runtime-modal 或通用运动参数 |

驱动相关事实的唯一入口是驱动参数映射表/profile 加运行读回证据。对象字典/SDO 索引、寄存器地址、位宽、raw modulus、符号扩展、单位比例、电子齿轮、反馈安装位置、能力声明、命令支持、write/verify/post sequence、故障/状态位解释和厂家差异都不得在业务代码、UI、G-code filter、State Publisher 或测试夹具中写死。代码只能按映射表字段和当前 profile identity 查表；映射表缺失、字段缺证或运行读回不一致时必须 fail-closed 或降级为只读诊断，不得用默认厂家值、ESI 推断值或旧代码常量补齐。

G92 不是永久禁用项，但不能用来绕过 Rebase/RTCP/零点闭环。若某功能选择 G92，必须先补策略文档、gate 白名单、读回字段、fail-closed 条件和运动闭环条件；缺 proof 的路径继续禁止使用。

## 3. Count-Domain 模型

旋转轴等效角必须优先使用整数脉冲/计数域，浮点角度只作为最终显示。

| 字段/概念 | 含义 |
| --- | --- |
| `Ccurr_abs` | 当前可信绝对反馈 count |
| `Czero_basis_persisted` | 用户受控设 0 或维护事务中持久化的物理 count 基准；BUS 设 0 时对应 `zero_model.zero_abs_counts`，旧名 `Czero_init_disk` 不得新增使用 |
| `Czero_mem` | 仅作为等效回零/相位规划可推导的等效零点，不得覆盖或替代 `Czero_basis_persisted` |
| `Crev` | 输出轴/负载轴转 360 度对应的反馈 count，不是电机一圈 count |
| `zero_distance_counts` / `rel_counts` | `Ccurr_abs - Czero_basis_persisted`；允许大于一圈，用于真实累计距离、rebase/wcheckpoint 诊断和 UI 证明 |
| `phase_counts` | `positive_mod(rel_counts, Crev)` |
| `phase_deg` | `phase_counts / Crev * 360`，UI 显示用 |

`positive_mod(value, modulus)` 必须统一定义为返回 `[0, modulus)` 的整数余数；`modulus <= 0` 时 count-domain invalid。实现不得依赖 Python `%`、C `%` 或浮点角度换算的隐式差异。C 实现应使用等价形式：

```text
positive_mod(value, modulus) = ((value % modulus) + modulus) % modulus
```

其中第二次 `%` 必须仍在整数域执行，避免负数余数和溢出行为污染 `phase_counts`。

`Crev` 必须由整数来源验算：

```text
Crev = encoder_counts_per_motor_rev
     * electronic_gear_num / electronic_gear_den
     * reducer_num / reducer_den
     * pulley_or_extra_ratio_num / pulley_or_extra_ratio_den
```

传动比必须保存 numerator/denominator，不得只保存浮点 `ratio`。`Crev` 必须同时与 runtime `SCALE`、电子齿轮读回、驱动 profile 和 native readback 一致。减速机间隙/backlash 不改变 `Crev`，但影响到达判定、方向切换 repeatability 和实际运动允许窗口。缺减速比、电子齿轮读回、runtime `SCALE` 或任一 readback 不一致时，`rotary_count_domain_profile_valid=false`，`rotary_equiv_zero` 必须 fail-closed。

旋转 count-domain 的位宽不得固定为 64 位，也不得在 native、HAL、UI、Provider、测试或字段名中隐含固定 16/32/64 位。命令位置、实际位置和绝对编码器多圈值各自使用的 `raw_count_bits`、`raw_value_type`、`raw_modulus`、符号扩展方式和反馈来源，只能由当前生效的驱动映射表/profile 及运行读回确定；命令与反馈宽度不一致时，运行窗口必须按两者共同可证明的较窄安全数值域约束。ESI/PDO `bitLen` 只能作为映射表生成或核验输入，不能越过当前 profile 成为业务代码默认值。

微内核/native 可以为连续逻辑累计选择比驱动 raw 值更宽的内部整数或多字容器，但容器宽度必须根据当前映射位宽、`Crev`、运行窗口和累计余量计算并登记为 `logical_storage_bits`，不得把某个固定 C 整数类型写成产品语义。当前映射缺少位宽/符号/模数、映射与运行读回不一致，或者 native 容器无法覆盖所需范围时，必须置 `rotary_count_domain_profile_valid=false` 并 fail-closed，不得截断、回绕、饱和或退回固定 64 位。旧 `*_counts64`、`direct64`、`position64`、`candidate64` 命名全部退役。

native/provider readback 至少应能按 active model 对应旋转轴独立返回下列事实，供 UI 显示解释、维护诊断和动作证明使用；这些字段不是 SHM ABI 字段：

```text
rotary_raw_counts
rotary_zero_counts
rotary_rel_counts
rotary_phase_counts
rotary_phase_deg
rotary_raw_count_bits
rotary_raw_value_type
rotary_raw_modulus
rotary_logical_storage_bits
rotary_unwrap_valid
rotary_needs_rebase
rotary_count_domain_profile_valid
rotary_invalid_reason_code
rotary_recovery_source
```

SHM 是显示投影 ABI：只允许保留 0-4 文档列明的坐标/轨迹、转速、进给速度、主轴/进给倍率及必要 frame envelope/epoch/CRC/valid bits。不得把 `rotary_raw_counts`、`rotary_zero_counts`、`rotary_rel_counts`、`rotary_phase_counts`、`rotary_unwrap_valid`、`rotary_needs_rebase`、`rotary_count_domain_profile_valid`、`rotary_invalid_reason_code`、`rotary_recovery_source`、wcheckpoint 或 range guard 扩展进 SHM；需要新增或重排允许显示字段时才 bump version/header size，并同步 C `_Static_assert`、Python `struct.calcsize()`、offset 契约和 writer/reader。init、deploy、supervisor 不得硬编码旧 SHM 容量。

`rotary_zero_counts` 表示设 0 锚点的绝对 count，不表示当前圈等效零点；`rotary_rel_counts` 表示 `Ccurr_abs - Czero_basis_persisted` 的真实累计距离；`rotary_phase_counts` 才是等效相位。上述字段只能由 native/provider readback 提供，不得为了让 UI 显示接近 0 而把 `rotary_zero_counts` 静默平移到当前圈，否则会污染开机 checkpoint 续接、`wcheckpoint` 运行窗口和 rebase 诊断。

最小 invalid reason 枚举必须覆盖：

```text
rotary_count_domain_profile_missing
rotary_raw_count_width_missing
rotary_logical_storage_insufficient
rotary_crev_mismatch
rotary_scale_readback_mismatch
rotary_reducer_ratio_missing
rotary_abs_feedback_untrusted
rotary_phase_tie_ambiguous
rotary_soft_limit_violation
rotary_motion_not_idle
rotary_arrival_readback_failed
rotary_abs_modulus_wrap_ambiguous
```

实现可以保留更细分的等价结构化 reason，但不得把上述缺证、越界、二义和到达失败归并成泛化成功或 UI-only 提示。

## 4. 开机恢复优先级

| 恢复来源 | 使用条件 | 失败处理 |
| --- | --- | --- |
| direct_absolute | 驱动/profile 按映射位宽提供完整多圈绝对 count，读回可信 | 失败转 split |
| split | `drive.read_abs_multi_turn` + `drive.read_abs_single_turn` 可按映射位宽一致合成绝对位置 | 防撕裂失败返回 `DRIVE_ABS_SPLIT_READ_INCOHERENT` |
| bounded checkpoint | 没有 direct_absolute/split，但停机事务、机械约束和 checkpoint 窗口证明断电位移小于安全半窗 | 证明不足时 invalid |
| 重新设 0/维护 | 上述都不可信，或 position basis pending/generation 不匹配 | 禁止旧 checkpoint 恢复，要求受控设 0 或维护 |

驱动原始多圈/位置计数的 `raw_modulus`、位宽、对象字典/SDO 索引和寄存器映射按上述驱动映射表规则取得，不得写死为 16 位、`65536` 或某个厂家固定地址。`raw_modulus` 只是 raw counter modulus，不是硬件物理安全/报警上限；达到已证明 `raw_modulus / 4` 或配置阈值时，microkernel/native owner 应在 native/provider readback 中置位 `rotary_needs_rebase=true` 并给出结构化原因，不得经 SHM 发布。

split/checkpoint 恢复遇到 raw 多圈寄存器 modulus 边界时必须按“证据优先、缺证失效”处理：只有 direct_absolute、split 单圈/多圈一致性、checkpoint generation/hash、低位窗口和断电位移窗口同时证明 wrap 方向与圈数时，才允许按映射位宽重建 `absolute_candidate`。不得仅因为 `raw_turn` 差值接近 `raw_modulus` 就自动补偿；一旦出现半模糊点、窗口越界、zero basis 缺失、generation/hash 不匹配或无法证明断电位移小于安全半窗，必须 fail-closed，置 `rotary_unwrap_valid=false` 并返回 `rotary_abs_modulus_wrap_ambiguous` 或等价结构化 invalid reason，要求重新设 0/维护建基准。

## 5. 运动模式

| 模式 | 场景 | 行为 |
| --- | --- | --- |
| `phase_nearest` | 普通相位定位、First point、UI 等效回零、G-code 非连续累计段的一圈内绝对相位目标（`abs(A/C) < 360`，含 `A0`/`C0`） | 把目标相位映射到最近 raw 目标 |
| `continuous_delta` | G-code 多圈目标、连续加工段、显式累计旋转、持续单方向旋转 | 保留程序要求的连续 raw 增量；未标记的 `G1 C360`、`G1 C720`、`G1 C3600` 等大 C 目标也按连续 delta 处理 |
| `safe_zero_return` | 等效回零 | 在软限位和 native gate 允许时运动到最近等效零 |
| `ambiguous_block` | 目标无法判断是相位还是连续 raw | fail-closed，不猜测 |

G-code 多圈、RTCP 五轴几何变换、旋转轴平铺展开和窗口切换必须交给微内核/native owner。V3 上层不得再维护 Python G-code filter、运行副本重写器或第二套等效角解析器；不得修改用户源程序或 model-specific `cc-ac.ngc`/`cc-bc.ngc` 轨迹语义来绕过失败。`RE_V3_ROTARY_CONTINUOUS_DELTA` 只作为历史显式标记的读取兼容输入，不得作为产品门禁或上层解析分支；大于等于一个输出轴周期的 G-code 旋转目标由微内核/native `logical_abs_counts`、range guard、count-domain、raw 可信、软限位和 native/motion gate 判定。

等效角能力必须保留，但实现位置只能是微内核/native rotary owner、motmod/EMCMOT、HAL/EtherCAT/native gate 或同等底层运动 owner。UI 显示等效角只消费 native/provider readback 或允许的显示投影，UI 对 G-code 的解析、黄色线移动、屏幕 active model 姿态和 first-point 预览均属于显示链路；这些显示结果不得改写源 G-code、不得生成另一个投喂文件、不得替代微内核的 `phase_nearest` / `continuous_delta` / `ambiguous_block` 判定，也不得成为运动到达证明。

G-code 一圈内等效角判定必须在微内核/native parser 或 motion owner 中完成，不得由 UI、Broker、刀路显示层或运行文件重写器猜测。已被 native 判定为非连续累计段的 G90/G53 active model 对应旋转轴绝对 word，且 `abs(target_deg) < 360` 时，按 `phase_nearest` 映射到最近 raw 等效角；AC 模型下的 `A0`/`C0` 或其它模型对应轴的零目标在快速/回零类 normalization 场景下等同最近 `k*360` 等效零。若同一程序段或相邻段已被 native 判定为连续加工段、显式累计旋转、`abs(target_deg) >= 360` 多圈目标、G91 增量、宏表达式、圆弧旋转或语义不明目标，则按 `continuous_delta` 或 `ambiguous_block` 处理，不得为了“一圈内数值”强行折叠连续轨迹。

`phase_nearest` 的半周期中线（例如 360 度周期下的 180 度）是方向二义点，不得让浮点尾数决定顺/逆时针。等效角算法必须在 `period / 2 ± 1.0e-5 deg` 临界区使用固定 tie-break 策略或基于上一稳定方向的滞回。允许的枚举为：

```text
tie_break_policy = prefer_positive | prefer_negative | last_stable_direction | block
```

无历史状态时默认 `prefer_negative`；若选择 `last_stable_direction` 但历史方向缺失或不可信，必须降级到文档化默认策略或 `block`，不得由浮点尾数决定。所有使用 `phase_nearest` 的功能，包括普通相位目标、First point、MCS 旋转单轴回机械 0 和 native G-code parser/motion owner 的一圈内相位目标，都必须使用同一 tie-break 策略。若目标路径会进入干涉区、软限位边缘或当前策略无法确定方向，则按 `ambiguous_block` fail-closed，并返回 `rotary_phase_tie_ambiguous` 或等价结构化 reason。

## 6. 等效回零

主界面“回零”严格服从 `功能/3-2回零按钮开机强制回零与双模式真实回零方案.md` 的选择分支：UI 始终只提交当前选择空间、全轴/单轴和轴标识。选中 MCS active model 对应旋转轴单轴时，登记 native positioning owner 在内部委托 native rotary equivalent-zero owner 计算并执行最近等效机械 0。选择 `机械全轴` 时必须再按 fresh active mode 分流：BUS/EtherCAT 正常 Home 中每个 active-model 旋转轴委托同一个 native rotary equivalent-zero owner；Pulse/StepDir 正常 Home 完整走 LinuxCNC/HAL 原生 homing 链，按 HOME_SEQUENCE、HOME_SEARCH_VEL/HOME_LATCH_VEL、home 输入和 HOME_OFFSET 建立 homed/all_homed，不得委托或插入 `safe_zero_return`。AC 模型对应 A/C，BC 模型对应 B/C，后续模型按 descriptor 对应轴组。MCS 直线单轴和 WCS 单轴由同一 positioning owner 执行对应坐标域的真实到 0 运动。主界面“归零”提交 `work_zero`，不得替代旋转轴等效回零或任何真实到 0 运动。

旋转轴机械零属于 `logical_abs_counts` 与 `zero_model.zero_abs_counts` 定义的物理等效相位，不属于当前软件运行窗口的数值 0。`wcheckpoint` 只平移 `wcheckpoint_base_counts`，所以 Home/positioning owner 必须在同一 native transaction 内读取并锁定 fresh `wcheckpoint_base_counts/generation`，先在逻辑 count-domain 计算最近等效机械零，再映射为当前 `runtime_window_counts` 目标；窗口已平移时，正确的运行窗口目标允许不是 0。

MCS 旋转单轴回机械 0 的正式路径：

```text
UI C selection intent (space=mcs, single axis)
  -> registered native positioning gate
  -> native rotary equivalent-zero owner
  -> microkernel/native count-domain guard and motion owner
  -> LinuxCNC/HAL/EtherCAT/native readback proof
  -> UI top status / event log result
```

执行要求：

- 单次按下 `回零` 就必须产生本次 native positioning transaction；UI 不做二次规划确认、不计算目标角，只提交选择意图。Native/microkernel owner 在执行时重新读取 fresh count-domain、zero_model 与 wcheckpoint generation/readback token；不得携带或依赖 SHM/display `status_epoch` 作为控制准入、执行证明或到达证明。
- 必须通过 count-domain guard、JOINT 软限位、microkernel/native motion gate 和到达证据；V3/Broker 不得自建速度容差或静止门禁。
- UI/Broker 不得提供本地 preview/plan 分支，不得自行规划最近目标、tie-break、软限位路径或到达条件；能力缺失时由 native owner 结构化 fail-closed，不得回退全轴 Home。
- 模式切换失败、软限位越界、count-domain invalid、状态 stale 或到达证据不足时 fail-closed。

`safe_zero_return` 必须使用整数 count-domain 算法规划最近等效零：

```text
rel_counts = logical_abs_counts - zero_model.zero_abs_counts
phase_counts = positive_mod(rel_counts, Crev)

if phase_counts < Crev / 2:
    delta_counts = -phase_counts
elif phase_counts > Crev / 2:
    delta_counts = Crev - phase_counts
else:
    delta_counts = tie_break_policy(...)

logical_target_counts = logical_abs_counts + delta_counts
runtime_target_counts = logical_target_counts - wcheckpoint_base_counts
```

`Crev / 2` 的比较必须在整数域声明奇偶行为：`Crev` 为偶数时 `phase_counts == Crev / 2` 是精确二义点；`Crev` 为奇数时不得制造浮点半点，必须按 `2 * phase_counts < Crev`、`2 * phase_counts > Crev` 比较。最终 joint 目标由 `runtime_target_counts` 映射，不得先转浮点角度再反推 count。若执行期间 native owner 合法推进 wcheckpoint generation，`logical_target_counts` 和 `zero_model.zero_abs_counts` 必须保持不变，owner 必须在同一原子/受控事务内按新 `wcheckpoint_base_counts` 重映射运行窗口目标；UI/Broker 不得自行重试、补偿或把旧 generation 的目标继续下发。

active-model 旋转轴 Home 不登记、传输或判断 target-error 到位容差，也不因停稳后的整数余差启动补偿/重试 leg。`logical_target_counts` 仍由 fresh count-domain 规划并用于生成完整运动命令；`runtime_target_counts = logical_target_counts - wcheckpoint_base_counts` 必须先在整数域固定，再转换成 joint position，且经 CIA402 `position * scale` 后必须无损回到同一个整数 target count。不得用 `joint.pos_fb + delta / scale` 的浮点相对加法把负整数边界截断到相邻 count；运动命令也不得因当前接近零而缩短或跳过。完成只认本次命令已受理、非零命令已有实际运动、planner/motion 已完成、native `in-position` 成立和 actual 连续稳定。若 wcheckpoint generation 合法变化，固定 logical target 不变，只按 fresh `wcheckpoint_base_counts` 重映射 runtime target。

BUS axis router 是物理 zero anchor 的唯一扣除点：反馈由 `physical_actual_counts - zero_model.zero_abs_counts` 转成 zero-relative joint counts，命令只在写向驱动时把同一个 anchor 加回。wcheckpoint 的 `logical_abs_counts/runtime_window_counts` 消费的是该 zero-relative joint 坐标，因此 State Publisher 的等效角投影只能对 zero-relative logical count 做周期归一化，严禁再次减 `zero_model.zero_abs_counts`。原始物理 count 可以等于非零零锚点，但 router target/actual、wcheckpoint logical phase 和 UI MCS/phase 必须同时为精确等效 0；固定一 count 残差也不是可接受的 Home 成功结果。不得用 UI snap、G92、驱动 preset、重写 `zero_model` 或只改显示代替前面的真实完整运动和 native motion-complete/in-position/静止证明。

路径检查不得只检查目标点：

- `runtime_target_counts` 转换后的 joint 目标不得越过当前运行窗口和 JOINT 软限位。
- 当前点到目标点的路径不得穿过禁区、干涉区或 cable wrap 禁区。
- LinuxCNC/microkernel native gate 必须判定当前可安全执行；无 Jog、MDI、program queue 或解释器 look-ahead/运动队列有效执行状态的证明来自 native gate，不由 V3 用速度容差重算。
- 到达结果必须靠 microkernel/native 回读证明，不得靠 action 自己声明或 SHM 显示投影。

`rotary_equiv_zero` 不使用 G92；它是物理运动到最近等效零。若未来采用 G92，必须作为单独登记的 LinuxCNC runtime coordinate policy，不能替代 `safe_zero_return` 或 Rebase。

请求必须明确轴、规划/执行模式和 fresh 状态基准；执行模式必须有显式确认。若部署仍保留 Broker/dispatcher，它只负责封包、串行转发和结果回传，规划和执行语义必须都由 native owner 从同一 count-domain 状态推导，不能让 UI 显示值、旧浮点零位、SHM 显示投影或 action 自身声明替代 native readback。

## 7. Rebase 边界

`rotary_rebase` 是高风险长期能力；没有实时 native gate 证明前应保持未注册或 fail-closed。`rotary_needs_rebase` 只是维护/门禁信号，不等于 Rebase 已执行。

允许的纯内存 Rebase 必须同时满足：

- 微内核/native gate 判定机器处于允许 Rebase 的安全窗口，目标轴无未清故障。
- Broker motion domain 独占。
- native gate 已登记，并证明能在微内核侧单周期或等价原子事务中同步 command、feedback、软件 offset、PID/FF/history 相关状态。
- 不写磁盘、不清驱动多圈、不做 drive-side zero rewrite。
- 完成后由 microkernel/native 回读证明 raw、phase、零偏、跟随误差和 invalid reason 状态一致；SHM 只显示允许的坐标/轨迹、转速、进给速度和主轴/进给倍率投影，当前模态直连 native modal/tool 状态。

运行期硬锁定：LinuxCNC/微内核 native gate 判定不可修改运动基准时，包括 G-code 运行、暂停/hold、MDI 排队、look-ahead 副本仍有效或底层运动安全 owner 拒绝，Broker/socket 入口必须直接拒绝 `rotary_rebase`、position-basis mutation，以及会热修改 rotary scale、HOME、软限位、RTCP/G53、几何中心、kins 或 INI 的设置保存事务。拦截必须在后端命令入口执行，UI 隐藏按钮或前端提示不能作为安全边界。允许的执行窗口只能来自 fresh microkernel/native gate，V3 不得用速度容差、SHM、JSON 或私有状态自行拼出放行条件。

## 7.1 wcheckpoint 运行窗口 checkpoint

`wcheckpoint` 是微内核运行窗口 checkpoint，用于持续单方向旋转时把内部连续绝对位置投影到受控软件窗口，解决 C 轴命令值、LinuxCNC 坐标值或驱动命令窗口持续增大后的软件数值上限。本旋转轴需求明确为无物理圈数限制，目标是在软件层面实现无限圈连续旋转；`wcheckpoint` 不替代开机 `checkpoint`，不替代 `zero_model`，也不建立物理圈数上限模型。

`wcheckpoint` 的运行期 owner 必须是微内核/native 内存常驻状态机，而不是外部“记录脚本 + 同步偏移脚本”。记录脚本、偏移同步脚本、维护命令或离线诊断工具只能生成候选、导出证据或向常驻 owner 发起受控维护请求，不能自己计算偏移后写文件给 G-code 热路径消费，也不能成为 `logical_abs_counts`、`wcheckpoint_base_counts`、`runtime_window_counts`、`generation`、`invalid_reason` 或 motion truth 真源。运行期状态必须常驻在微内核/native 内存域，由 native gate 更新；UI、Broker 和 product Python 必须直接通过 UDS 接口直连消费微内核/native 内存或登记 API readback，发起受控请求，并等待同周期 native/readback 证明。wcheckpoint 状态不得同步到 SHM，SHM 仅承担允许的坐标/速度倍率显示投影。

开机或 LinuxCNC clean restart 后，BUS realtime router 必须在首个 mapping/zero anchor/actual 都可信的伺服周期内建立 wcheckpoint，不能等待 G-code 把连续坐标累积到门禁边界才报错。初始化基准只能取最接近当前 zero-relative actual 的合法完整输出轴周期整数倍；运行中接近窗口边界时继续按同一周期量平移。对减速比 `motor_rev:load_rev`，周期量必须同时满足输出轴整圈和电机侧整数圈；不得把任意角度、任意 raw count 或单个电机圈当作窗口基准。

BUS router 是窗口平移的同周期执行点：反馈进入 LinuxCNC 前执行 `runtime_actual = physical_actual - zero_anchor - wcheckpoint_base`，目标写回驱动前执行 `physical_target = runtime_target + wcheckpoint_base + zero_anchor`。两边必须消费同一 base/generation，并保持驱动实际看到的物理 target 连续不跳变；router 同周期发布这份已经计算好的整数 `runtime_actual`，motmod 直接以 `logical_actual = wcheckpoint_base + runtime_actual` 维护开机前未回零和运行中的 logical/runtime 坐标与规划，不得从 Cartesian pose 再推一份 runtime，也不得重复减 base。初始化、平移或 generation 校验失败时才 fail-closed。

Router 在有效运动中发布新的 `wcheckpoint_base/generation` 时，motmod 必须在读取新 `runtime_actual` 之前，把 joint `pos_cmd/pos_fb/coarse/free_tp/cubic` 等运行窗口状态按同一个完整输出轴周期原子平移；Cartesian logical trajectory、当前剩余行程和物理 drive target 必须保持不变。禁止只接收新 base 后让新窗口 feedback 直接撞上旧窗口 joint command，这会制造整圈级伪跟随误差并中断未完成的连续段。板端验收必须让 active-model C 轴以小于安全半窗的分段目标持续跨过至少一次换窗阈值，证明 generation 递增后当前分段继续运行到原目标、无 `following error`、无提前 M0、WKC/DC 不退化，再反向跨回并执行最近等效机械零。

BUS router 不是轨迹规划器，也不是异常跳变整形器。G-code `RUNNING` 时，上游 LinuxCNC/motmod 已按唯一 `MAX_VELOCITY`、`MAX_ACCELERATION` 和伺服周期生成连续轨迹；router 只在同一周期完成 zero anchor、wcheckpoint base 和逻辑轴到物理从站的整数映射，然后把结果原值写入 PDO。不得保留目标限速器、上一输入/输出、候选速度、追赶模式、`limiter_active/lag_counts` 或其它第二套运动状态。只有 planner 已 `STOPPED/IDLE` 或 Machine Disabled 时，LinuxCNC/native owner 才允许把 commanded/规划器内部状态一次性同步到 actual；Machine Disabled 下只同步内部状态，物理 controlword 继续去使能，不允许电机补跑。重新使能前必须完成 commanded/actual 同步；需要真实移动时必须重新进入原 planner。wcheckpoint 换窗由 motmod 原子平移 `pos_cmd/pos_fb/coarse/free_tp/cubic`，保证 router 看见的目标连续。同步无法证明时由 native motion gate保持去使能并拒绝，不能由 PDO 层平滑追赶。

router 必须为每轴在同一个偶数 `diagnostic_seq` 快照内发布 `desired_raw/accepted_raw/actual_raw`，并保证 `accepted_raw == desired_raw == slave-target-position`；唯一 Position display producer 只从同一快照读取一次 `accepted_raw` 与 `actual_raw`，计算 `following_error = actual_raw - accepted_raw` 并换算单位后随同一 Position generation 写入 SHM。State Publisher 只透传，板屏、设置页、刀路和 WinRemote 只显示，不得异步再减一次，也不得把 SHM 跟随误差反向用于运动控制。五轴验收必须覆盖加速、最高速、减速、反向、RTCP ON/OFF、运动模型切换和换窗，证明 PDO target 与 planner desired count 逐周期完全一致、无 limiter 符号/状态、无下游制造的固定滞后；真实安全边界仍由原 planner、native motion gate、LinuxCNC 限位与驱动安全 owner 闭合。

router 已证明 `accepted_raw == desired_raw` 而 PDO `0x60F4` 仍随速度形成固定位置偏差时，根因属于驱动内部 CSP 位置环/速度前馈 commissioning，不能继续给 router 叠加限速或把误差归咎于显示采样。SV630N 的 2ms CSP 同步周期保持原 2ms DC 配置；当前生效驱动 profile 必须登记并逐物理从站读回 `H05.19(0x2005:14)`、`H08.00/01/02(0x2008:01/02/03)`、`H08.15(0x2008:10)`、`H08.18/19(0x2008:13/14)` 与 `0x60F4`。速度前馈按厂家定义只用于改善位置控制固定速度偏差，参数写入必须在 Machine Off/无运动窗口内按逐轴 drive-only owner 自动完成并复读；不得硬编码统一覆盖机械相关增益。验收以同一 G 代码、同一速度、同一 2ms servo 周期比较 `607A/6064/60F4`：五轴在加速、匀速、减速、反向中均无持续多周期积压、无超调振荡、无新驱动告警，停止后误差回零，并保留修改前后逐轴峰值/P95及真实周期滞后证据。

`wcheckpoint` 的最小状态语义：

```text
logical_abs_counts          // 微内核内部连续绝对位置；存储宽度由映射和累计余量决定
wcheckpoint_base_counts     // 当前运行窗口基准；与 logical 使用同一无损整数域
runtime_window_counts       // 下发/回读窗口值 = logical_abs_counts - wcheckpoint_base_counts
raw_count_bits              // 当前驱动映射表决定的命令/反馈 raw 位宽
raw_value_type              // 当前驱动映射表决定的有符号/无符号解释
raw_modulus                 // 当前驱动映射表决定的 raw 回绕模数
logical_storage_bits        // native 实际登记的逻辑累计容器有效宽度
generation                  // 当前 checkpoint/window 代际
invalid_reason              // 当前恢复、窗口判定或投影失败原因
```

上述字段都是高频/安全相关 readback，必须随 microkernel/native 常驻内存状态一起更新；只有显示所需的坐标/轨迹、转速、进给速度和主轴/进给倍率允许投影到 SHM，当前模态直连 native modal/tool 状态。字段缺失、过期或校验失败时只能由 native/provider 返回 degraded/invalid reason，不得从硬盘 checkpoint、`/run` result、旧快照或默认值补成有效。

窗口平移必须保持 C 输出轴相位不变。平移量只能是减速比后的完整输出轴 360 度周期整数倍；若反馈在电机侧且减速比为 `20:1`，则电机侧平移必须是 20 电机圈的整数倍。count-domain 表达为 `shift_counts = k * output_axis_crev_counts`；如果只能拿到电机侧计数，则必须先按驱动映射表规则证明 `motor_counts_per_rev`、`reduction_ratio`、电子齿轮、反馈来源和 `raw_modulus`，并证明 `output_axis_crev_counts = motor_counts_per_rev * reduction_ratio` 为整数且与 runtime SCALE 一致，否则 fail-closed，不得用任意 raw count、任意电机圈、写死驱动参数或浮点角度平移。

`drive_wrapped_rotary_support=true` 只能表示当前 profile 已证明驱动自身的命令位置与实际位置反馈能够在同一伺服周期按完整输出轴周期进入同一个 periodic window；只证明命令可取模、只证明反馈会回绕、只存在对象位宽或只存在厂家说明均不得置 true。该字段为 false 或缺失时，不得让驱动自身执行回绕；但 BUS realtime router 已按上一段同周期对 command/actual 应用同一软件 base、且驱动物理 target/actual 保持连续时，允许由 router 执行软件 wcheckpoint 平移。两种能力不得混为一项或用 profile false 阻断已经证明的 router 软件窗口。

`wcheckpoint` 平移后 `logical_abs_counts` 和 `zero_model.zero_abs_counts` 不变，UI 零点距离仍按 `logical_abs_counts - zero_model.zero_abs_counts` 计算；只允许 `wcheckpoint_base_counts` 改变，使 `runtime_window_counts` 回到安全窗口。窗口切换必须通过微内核/native gate 或可证明的受控分段执行，并由 microkernel/native 回读证明 logical、base、window、generation、phase、零点距离、映射位宽和 invalid reason 一致。不得在连续插补中间做 UI-only、Python-only 或未证明的静默窗口平移；无法证明安全边界时必须在 Start/运行前门禁中 fail-closed。

禁止：

- Python/HAL 分帧写入替代 Rebase。
- 未登记 G92 替代 Rebase。
- 静默 `drive.preset_position`、`drive.clear_multi_turn` 或厂家位置偏置写入。
- 只收缩 UI 显示而不处理 LinuxCNC/native 运动真相。

## 7.2 无限圈实现说明

“无限圈”不是让驱动、LinuxCNC 坐标或硬件计数器真的没有数值上限，而是在 native/微内核常驻内存里把旋转轴分成三层：

```text
logical_abs_counts          // 逻辑连续绝对位置，可随 G-code 多圈目标持续累加
wcheckpoint_base_counts     // 当前软件运行窗口基准，只按完整输出轴周期平移
runtime_window_counts       // 实际下发/回读窗口值 = logical_abs_counts - wcheckpoint_base_counts
```

G-code 中 `G1 C360`、`G1 C720`、`G1 C3600` 这类连续多圈目标进入 `continuous_delta`，先累加到 `logical_abs_counts`。当 `runtime_window_counts` 接近 LinuxCNC、驱动命令窗口、raw counter 或软件数值域的安全边界时，微内核/native owner 只平移 `wcheckpoint_base_counts`，把窗口值拉回安全范围；`logical_abs_counts` 不回退，`zero_model.zero_abs_counts` 不改变，用户看到的零点距离和连续运动历史仍然连续。

窗口平移只能按完整输出轴 360 度周期整数倍执行：

```text
shift_counts = k * output_axis_crev_counts
wcheckpoint_base_counts += shift_counts
runtime_window_counts = logical_abs_counts - wcheckpoint_base_counts
```

这样平移后相位不变：

```text
phase_counts = positive_mod(logical_abs_counts - zero_model.zero_abs_counts, Crev)
phase_deg = phase_counts / Crev * 360
```

因此 UI 可以一直显示 0-360 的等效相位，G-code/native 内部仍保留多圈累计事实。20:1 减速且反馈在电机侧时，一次窗口平移必须是 20 个电机圈的整数倍；不能按任意 raw count、任意电机圈或浮点角度平移。

实际的合理解决办法（如何做到无感）：
在实际的微内核中，为了避免这种数学差值引发软件保护跳闸，平移绝对不是直接修改发给 PID 的实时位置指令。而是在底层实时适配器（HAL Component，实时硬件抽象层组件）中，让 Command（指令）和 Actual（反馈）在同一个伺服周期内，同步减去相同的偏置量（Offset）。
这样，在 PID 和误差监控器的眼里，差值依然是：
\[(Command - Offset) - (Actual - Offset) = Command - Actual \approx 0\]
从而保持 PID 输入差值不变，避免窗口平移引入伪跟随误差或触发软件保护跳闸。

无限圈闭环的安全边界：

- `logical_abs_counts`、`wcheckpoint_base_counts`、`runtime_window_counts`、映射位宽描述、`generation` 和 `invalid_reason` 必须由微内核/native 常驻内存维护，并由 microkernel/native 回读证明。
- UI、Broker、后台 action/native dispatcher 和 G-code filter 不能自己改窗口基准，也不能用硬盘 checkpoint、`/run` result、旧快照或默认值补成有效。
- `wcheckpoint` 只解决软件窗口和数值域上限，不是物理防缠绕、线缆缠绕、软限位、碰撞区或驱动硬保护的豁免。
- 窗口平移无法证明相位不变、generation 不一致、字段过期、减速比/`Crev`/SCALE/readback 不一致或 native gate 失败时，native/provider 必须返回 degraded/invalid 并 fail-closed，不能继续声称无限圈可用。

## 8. 驱动侧位置基准命令

驱动侧多圈清零、position preset、厂家 offset 写入都属于高风险 position basis mutation，默认不用于普通回零、设 0、故障复位或自动维护。

只有 profile 标准命令声明完整且执行器具备 crash-consistent 事务时才允许：

| 条件 | 要求 |
| --- | --- |
| profile 命令 | `drive.preset_position`、`drive.clear_multi_turn`、`drive.write_position_offset` 等必须 `supported=true` 且有完整 write/verify/post sequence |
| 事务声明 | `mutates_position_basis=true`、`requires_position_basis_transaction=true`、`checkpoint_policy=invalidate_then_rebuild` |
| 前置 | 空闲、停机、显式确认、备份、目标驱动状态满足 profile |

## 8.1 机械传动和反馈边界

等效角算法只解决几何等效和目标规划。backlash compensation 属于 LinuxCNC/HAL/drive/native owner；UI 和后台 action/native dispatcher 不得私自加入反向补偿运动。如果方向反转导致相位到达不可靠，`arrival_ok` 必须基于 microkernel/native 的实际回读误差、跟随误差和允许窗口。

profile 必须声明旋转反馈来源：

```text
feedback_mount = motor_side | load_side
```

若编码器在电机端，负载端真实角度依赖减速机、皮带、联轴器、谐波或蜗轮传动无滑移，profile 必须声明 `transmission_integrity_assumption`。缺负载端一致性证明时只能声明 `motor-side count-domain valid`，不能声明 `load-axis physical zero verified`。需要对外声称负载端物理零位闭环时，必须证明负载端相位和回读一致。

## 11. 功能闭合边界

旋转等效角功能闭合必须满足：

1. active model 对应旋转轴 count-domain 字段由 microkernel/native 维护，UI 只读显示 phase，不参与运动真相。
2. `Crev` 的整数传动比、runtime `SCALE`、电子齿轮读回、驱动 profile 和 native readback 一致；缺证时 `rotary_count_domain_profile_valid=false`。
3. `safe_zero_return` 使用整数 `positive_mod`、`delta_counts` 和固定 `logical_target_counts` 算法生成完整运动命令，整数 runtime target 经 joint/CIA402 转换必须无损到同一驱动 target count，半周期方向按本文 tie-break 策略确定；Home 链不存在 target-error 容差字段/判断、容差失败码或余差补偿循环。
4. `rotary_equiv_zero execute=false` 只请求 native owner 规划，`execute=true` 走注册 native rotary gate、microkernel/native motion gate 和同源 readback proof；最终必须证明本次完整运动、motion-complete、in-position、连续静止、同 generation 的 runtime=logical-base 以及 post-sync MCS/phase 0；Broker/dispatcher 若存在只做薄转发。
5. 软限位目标和路径越界、native rotary generation/readback token 过期、count invalid、profile 缺失、机器非 idle、到达失败全部 fail-closed。
6. 普通相位目标和已由 native 判定为非连续累计段的 G-code 一圈内绝对相位目标（`abs(A/C) < 360`，含 `A0`/`C0`）走 `phase_nearest`；`A0`/`C0` 在 G90/G53 快速/回零类 normalization 场景下走最近等效零；G-code 多圈目标、连续加工段和显式累计目标即使未标记也走 `continuous_delta`；宏表达式、增量模式、圆弧旋转、未知 motion 或其它语义不明目标走 `ambiguous_block` 或等价 fail-closed。
7. `rotary_needs_rebase` 只作为维护信号；没有 native gate proof 时 Rebase action 不注册或 fail-closed。
8. position basis mutation 命令没有完整事务声明时 fail-closed。
9. Start 热路径只做 O(1) resident runtime identity/handle 和 native/motion gate 检查，不做 manifest proof、不重新过滤、hash、全文件扫描或生成 sidecar。
10. 负载端物理零位声明必须有负载端一致性证据；缺证时只能声明 motor-side count-domain 有效。
11. `wcheckpoint` 只能按减速比后的完整输出轴周期平移运行窗口，并必须通过 microkernel/native 回读证明 logical/base/window/phase/zero distance 一致。
