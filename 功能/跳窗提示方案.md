## P0 弹窗蒙版与完整首帧

- 所有全屏弹窗统一使用开机预创建并常驻内存的 modal overlay：全屏蒙版负责在视觉上遮挡/压暗后部页面并吞掉后部输入，弹窗框、标题、三段式正文、倒计时、`确认继续` 和 `关闭` 等控件均属于同一预创建对象树。不得在每次打开时重新创建整棵弹窗对象、重新计算页面级布局或新增第二套 `lv_msgbox`/私有 overlay。
- 所有产品弹窗只允许一种外框尺寸。以当前最大弹窗为 canonical geometry：在 `1024×600` UI 画布内，modal 蒙版固定覆盖全屏 `1024×600 @ (0,0)`，弹窗内容框固定为 `684×424 @ (170,88)`。微内核原生错误、主页面开机回零前置提示、设置页进度/成功/失败结果、驱动/VPS/授权动作、确认继续以及以后新增的所有 popup 必须使用完全相同的内容框位置和尺寸；不得保留 `588×420` 等小弹窗、按文字长短自动缩放外框、为某个功能复制另一套几何或用不同尺寸区分严重程度。
- 统一尺寸必须由一个公共 popup geometry/constructor owner 提供，页面和功能只能传入标题、三段式正文、状态颜色、倒计时和按钮配置，不得在各源码文件重复硬编码 `x/y/w/h`。弹窗内部固定划分标题区、正文区、状态/倒计时区和按钮区；内容较少时保留统一留白，内容较多时只能在固定正文区内换行、裁剪为明确摘要、分页或受控滚动，不能扩大/缩小弹窗外框、挤动按钮或改变关闭/确认继续位置。
- 开机或 canonical reload 只准备蒙版、弹窗框和控件结构，不把空白弹窗作为产品可见帧。实际打开前必须先通过 `REQ-UI-FIRST-FRAME-CACHE` 公共 guard capture 当前可见页面；弹窗保持隐藏期间，把本次初始标题、`提示/原因/下一步`、倒计时初值和按钮启用状态写入常驻对象，并只对弹窗子树完成一次必要布局。
- 弹窗第一次对外可见必须是一个完整事务：蒙版、弹窗框、初始文字和当前可用按钮必须在同一个首帧中出现。禁止先显示只有蒙版/空框的空白弹窗，再在下一帧或后续轮询补文字；这会制造二次跳变并让操作员误判为卡死或内容丢失。
- owner 结果尚未返回时，初始正文必须使用与本次动作相符的明确进行态，例如“正在处理”“正在读取状态”“正在取消”，并继续保持三段式可读语义；不得留空、显示占位破折号或用旧任务文字。结果到达后只替换实际变化的标题、正文、倒计时、颜色或按钮状态，并只 invalidate/flush 对应对象 dirty rect，不重绘蒙版、弹窗框、后部页面或 screen。
- 弹窗蒙版必须拦截触摸、鼠标和远程输入，后部页面普通按钮、表格、键盘入口和手势均不得穿透。主页面急停按钮本体属于独立硬安全层：弹窗显示时只能把该对象提升到蒙版之上继续可用，不得复制第二个急停按钮，也不得让蒙版、弹窗布局或文字更新延迟急停事件。
- 普通关闭必须绑定 `LV_EVENT_RELEASED` 并先吸收当前输入释放；隐藏弹窗对象后立即 blit 恢复打开前 capture 的完整背景帧，再只刷新本次动作真实改变的后部对象。已受理 `保存并重启` 结果窗的 `关闭` 是唯一例外：不得隐藏后再恢复 capture，而要在同一次释放事件内先抑制旧 UI 输出，把本地 framebuffer 与远程 canonical frame 提交为纯黑，再等待系统重启接管。两类关闭都不得等待后部页面重新布局、页面根对象 invalidate、`render_now`/`lv_refr_now` 整屏刷新或下一次 30Hz 状态帧。
- 同时可能存在的原生错误、设置动作结果、开机回零前置提示及后续嵌套确认弹窗必须使用按层级独立的 capture slot/stack；禁止多个存活弹窗共用一个会被后打开弹窗覆盖的背景槽位。关闭内层弹窗必须恢复外层弹窗完整帧，关闭最外层才恢复原页面。
- 验收至少覆盖主页面、设置页和其它任一页面上的弹窗打开/动态文字更新/关闭，覆盖原生错误、开机回零前置、进度、成功、失败、确认继续、结果已知首帧、结果未返回进行态、取消中状态、急停层和两层嵌套；逐项读取 LVGL geometry 或截图像素边界，必须全部为 `684×424 @ (170,88)`，并证明没有第二尺寸、空白弹窗首帧、后部输入穿透、旧文字闪现、整屏刷新或 capture 错层。

## 1. 适用范围

本方案适用于以下三类统一提示来源：

1. 微内核原生错误：由唯一映射 owner 翻译并输出 `popup/top_status/log_only`，UI 不接触原始错误文本。
2. 自写业务功能：设置、驱动、授权、程序、坐标、参数保存等功能返回结构化结果；具体错误语义由各自 indexed feature owner 定义，显示生命周期和三段式格式统一服从本文。
3. 无原生等价的 UI 本地输入错误、owner 通道错误、网络/授权和资源缺失：可以由对应 owner 保留，但必须进入同一集中格式化和显示入口。

现有 `ReV3CommandResult` 后台命令包括但不限于：

- 坐标与偏移：倍率命令。
- 设置页：设置读取、设置保存、零位设0、机械零点保存、驱动扫描/读取/写入/复位、服务器下载、本机码登记、授权下载。

微内核原生错误不按“主页面/设置页”或“普通动作/后台动作”一刀切决定是否弹窗，而由 `REQ-NATIVE-OPERATOR-ERROR-MAP` 的统一 owner 按实际影响输出 `popup`、`top_status` 或 `log_only`。已经阻止当前操作、影响安全/运动/加工结果、涉及限位/驱动/程序执行或需要操作员处理的错误必须显示三段式全屏提示；不影响正常使用、允许继续操作且无需立即人工处理的错误只显示顶部短提示；已经自动恢复、对当前操作和结果无影响且没有操作员处理动作的工程诊断只写日志。需要开机回零的动作在 fresh native `all_homed=false/unknown` 或 command gate 二次校验返回同一条件时仍必须显示统一三段式全屏提示且不得发送、排队或重放原动作；内部 `POWER_ON_HOME_*` 只可用于关联和日志，弹窗标题、原因和下一步必须来自统一映射，不得显示内部码或保留第二套自造原因。

自写功能已有的合法跳窗必须保留并收口到统一框架，不得因为增加微内核错误映射而删除。自写功能是否弹窗由该功能 owner 的实际结果和人工处置需求决定：失败、阻塞、确认继续、需要人工确认的最终结果使用跳窗；不影响正常使用的短提示使用顶部状态；纯诊断写日志。自写功能不得伪装成微内核错误，也不得复制、改写已有微内核错误的可见原因。

本地输入校验类提示也必须写清直接原因，例如“请先点选单个加工坐标轴”，但这类提示可以不显示提示码。只要已经收到 native/owner 返回的 `ReV3CommandResult`，失败跳窗必须按本方案显示。

机械全轴 Home、MCS 单轴回机械零和 WCS 单轴回加工零的终态只负责结束本次 transaction：success、failed、cancelled 任一 fresh terminal 均先清 `active/pending/pressed`，回零按钮立即退出黄色。终态 failed 只有在同一事务同时产生微内核原生错误事件时，才由 State Publisher 的唯一映射 owner按既定分级显示；UI不得根据 Home progress/result 中的 adapter别名、目标、容差、proof、超时或缺字段自行组装 `回零失败` 弹窗。没有原生错误事件时只写工程日志，不猜轴、不沿用旧结果、不保留黄色等待下一事件。急停导致且owner明确返回cancelled时只显示`回零已取消`。

当前没有登记 LVGL `lv_msgbox` 独立消息框产品路径；产品使用 v5 页面内 popup overlay。后续所有跳窗统一按本文格式和 `V5CommandGateResult` / `V5SettingsActionStatus` 结构化结果收敛，不再新增第二套 msgbox、toast 或独立弹窗协议。
全屏 `popup_layer` 属于覆盖层，必须服从“P0 弹窗蒙版与完整首帧”：显示前通过公共首帧 guard 捕获被覆盖页面，隐藏状态下先写入本次初始文字和按钮并完成弹窗子树必要布局，再把蒙版、弹窗框、文字和按钮作为同一首帧显示；关闭时先恢复捕获帧，再只 invalidate 顶部状态、机器码或本次动作实际改变的区域。不得暴露空白弹窗首帧，也不得让关闭动作等待设置页或主页面整页重绘。

设置页 `登记本机码`、`下载授权`、`服务器下载` 三个按钮必须共享同一条 VPS 设备身份链路。`登记本机码` 只负责现读本机 PL DNA 并提交 VPS，由 VPS 返回并保存 canonical `vpsDistributionId`；板端不得用 DNA 本地推导值当作最终 ID。VPS 返回 ID 后，板端本地登记状态只允许保存 `vpsDistributionId`、登记结果码和公钥摘要，不允许保存 raw PL DNA、DNA hash、`pl_dna_hash` 或 id-dna folder；设置页顶部 `本机码` 只显示 VPS 原样返回的 6 位 `vpsDistributionId`，不显示 `已登记` 等登记状态后缀、不显示本机码片段或 hash，也不得从 `device_id`、`vps_distribution_id`、旧状态文件或授权文件回退推断；没有合法 6 位 ID 时，在同一 ID 位置显示 `未登记`。设置页顶部不得再显示 `请求:NAT 实际:NAT` 这类旧状态占位。若 VPS 未返回合法 canonical `vpsDistributionId`、本地写入失败或 ID-DNA 关系不能确认，登记弹窗必须显示失败，不得显示本地猜测 ID。

`下载授权` 和 BUS/EtherCAT 可用的 `服务器下载` 必须先读取本地登记状态中的 `vpsDistributionId`；本地前置顺序固定为先校验 6 位 ID 存在，再现读本机 live DNA 并在内存中计算 DNA hash，之后把 ID 和本次临时 DNA/DNA hash 一起发送给 VPS，由 VPS 一次校验返回。客户端请求路径只能携带 6 位 ID，不能把 DNA hash 或 `id-dna` 放进路径；临时读取的 DNA/DNA hash 不允许落盘保存，不允许写 UI、日志或 SHM。授权文件只能由 VPS 按该 ID 解析到当前设备 private 目录后下载 `device_authorization.json`；私有驱动映射表只能由 VPS 按同一 ID 解析后下载 `driver_profile_map.json`。`pl_dna_hash` 只作为 VPS 侧绑定和路径解析字段；没有已登记 ID 时，这两个按钮必须提示先登记本机码，不得只凭 DNA hash 查找私有目录，也不得从其它设备 private 目录兜底。

## 2. 用户可见格式

失败、阻塞、确认继续、不能继续、安全确认和其它需要操作员处理的结果跳窗，无论来自微内核还是自写功能，操作员可见正文必须统一为 3 个文本项，固定顺序如下；三个文本项之间必须空一行：

```text
提示: <code>

原因: <当前语言的直接原因>

下一步: <操作员下一步应该怎么处理>
```

微内核原生错误是本格式的特例：`提示` 必须显示映射后的中文标题，不显示原生源 ID、原始错误码、adapter 别名或底层实现名称；内部码只进入工程日志。自写功能的 `提示` 使用对应 feature owner 定义的中文动作标题或稳定业务提示码，不得借用微内核源 ID，也不得在页面内复制第二套错误原因。

这 3 个文本项是主格式，不得额外插入 `详情`、`现状`、`风险`、`不能继续`、`继续后` 等第四项或更多项。native/owner 返回的 `detail`、`current_state_cn`、`risk_cn`、`cannot_continue_cn`、`continue_action_cn` 等字段只能用于生成 `原因` 或 `下一步` 的内容；确需保留原始明细时写入事件日志、owner 结果或诊断文件，不直接放进操作员跳窗正文。

`原因` 必须是明确、可判断的直接原因，让操作员知道“到底因为什么被拒绝/失败/等待”。禁止只显示让人无从适从的泛化文案，例如“执行失败”“前置检查失败”“启动失败”“设置保存失败”“状态异常”“条件不满足”“后台返回失败”。如果只拿到这类泛化文案，UI 必须继续从提示码、结构化 detail、第一条 issue/error、轴号/坐标系/驱动从站上下文中提取具体原因；仍无法提取时，应显示“后台未返回具体失败原因”这类明确的 unknown-detail 原因，并在 `下一步` 中要求重新执行原动作或联系维护人员，而不是假装已有具体原因。

设置页 `settings_axis_zero` / 机械零点设0 的进程、成功、失败状态都必须在弹窗顶部标题显示本次具体操作轴，例如 `X 设0`、`A 设0` 或 `X 设零`，不得显示泛化的 `单轴设0` 作为操作标题。正文仍只能保持 `提示 / 原因 / 下一步` 三个文本项，不得新增 `操作轴` 第四项；正文可重复轴号以便读懂原因。该轴号只能从本次 UI 轴行请求、`axis_<轴>_zero` 字段、native/owner 返回的结构化 `axis/detail.axis` 推导，不得用固定默认轴或旧结果伪造。

进程窗口示例：
```text
提示: SETTINGS_ZERO_PENDING

原因: A轴正在设零

下一步: 完成后可继续设置其它轴，最后统一保存并重启
```

失败窗口示例：
```text
提示: SETTINGS_ZERO_ABS_SPLIT_UNAVAILABLE

原因: A轴零位设0失败，驱动绝对位置读回不完整

下一步: 检查轴选择和驱动状态，重新读取驱动后再设0
```

成功结果默认不弹全屏窗口。若成功前存在普通进度弹窗，成功时必须自动关闭进度弹窗，并在顶部状态栏显示约 2 秒简短成功文字，例如“回零完成”“保存完成”。失败时同一窗口更新为失败状态并保留到操作员点击关闭、急停或其它明确动作。

设置页例外：`下载授权`、BUS/EtherCAT 可用的 `服务器下载`、`扫描从站`、`复位驱动`、`读取驱动`、`清除故障`、`设置驱动`、本机码登记，以及轴参数表单轴 `设0/零位` 成功后不得自动关闭弹窗；必须把原进度窗更新为最终成功结果，至少显示 `提示`、成功原因和“确认结果后点关闭”的下一步，并保留到操作员点击关闭。Pulse/StepDir 或现场无 BUS/EtherCAT 目标时，驱动动作按 `功能/2-1总线脉冲区别.md` 返回结构化结果；若后台未进入 job 或判定真实不适用，只更新同一弹窗/顶部状态，不本地伪造成功。后台 job 运行中不存在“只手动关闭弹窗”的状态：点击 `关闭` 必须先取消同一 run_id 的任务并等待 cancelled，再隐藏弹窗；因此取消后的迟到 success 必须按 stale run_id 丢弃，不能重新弹出或覆盖新任务。

`扫描从站` 成功结果弹窗仍只能保持 `提示 / 原因 / 下一步` 三个文本项，不得新增第四项；其中 `原因` 文本项必须显示本次扫描到的驱动/从站列表，固定排成两列四行。每格显示序号、从站编号和驱动名，少于 8 个用 `--` 占位，多于 8 个在同一 `原因` 项内提示剩余数量；该列表只能来自本次 `drive_scan_slaves` 返回的 `slaves`、本次 `ethercat slaves` 行解析出的驱动名，或由本次扫描写入的自建参数表候选，不得用固定默认列表、固定驱动型号、驱动映射表或旧结果伪造。

最终结果弹窗的正文颜色必须跟随结果状态：成功结果使用绿色字体，失败、阻塞、不能继续或安全确认失败使用红色字体。进行中的进度弹窗不按成功/失败染色，保持普通提示色；当进度窗更新为最终成功或失败结果时再切换为绿色或红色。

设置页这几个长耗时弹窗必须在弹窗内容区左下角显示预计时间倒计时：`下载授权`、BUS/EtherCAT 可用的 `服务器下载`、`扫描从站`、`复位驱动`、`读取驱动`、`清除故障`、`设置驱动`。倒计时文案固定为 `预计: MM:SS`，只表示本次后台 job 的 UI 估算剩余时间，不作为成功、失败、超时或运动安全真源；真正结果仍以 native/microkernel/owner 返回和 readback 为准。倒计时只在这些动作的进行中进度弹窗显示，最终成功/失败结果、普通错误弹窗、确认继续弹窗和单轴设0弹窗不得保留倒计时。操作员关闭弹窗、后台 job 完成、取消、超时或进度跟踪结束时，UI 必须清理并隐藏倒计时，避免旧预计时间污染下一次弹窗。

所有后台命令跳窗都必须保留三个操作能力，但急停不得作为弹窗内容区左下角按钮：

```text
急停: 直接复用主页面急停按钮本体，并显示在该按钮当前屏幕位置、尺寸和红色语义
确认继续: 位于弹窗内容区，按 allow_user_continue 启用或禁用
关闭: 位于弹窗内容区
```

`急停` 必须动态引用主页面急停按钮对象，不得复制坐标、尺寸、颜色或重新创建第二个急停按钮。弹窗显示时，UI 只允许把主页面急停按钮本体提升到弹窗遮罩之上并保持可点；弹窗遮罩仍必须拦截其它底层按钮。后续主页面急停按钮位置或尺寸调整时，弹窗急停显示必须自动跟随，无需再改弹窗坐标。

开机回零前置条件在原动作未发送时就是本地阻塞终态；其 `关闭` 只隐藏提示，绝不执行、排队、取消或重放原动作，也不伪造动作成功。内部 `POWER_ON_HOME_REQUIRED` / `POWER_ON_HOME_STATUS_UNAVAILABLE` 只用于事件关联，UI 必须分别映射为原生“动作要求先回零”和 owner 状态不可用中文，不得直接显示这两个内部码。

`关闭` 按弹窗阶段只有两种明确语义。结果已经进入 `success/failed/cancelled/failed_closed` 终态时，`关闭` 只关闭结果弹窗；唯一特例是 `保存并重启` 已取得系统级重启受理终态，此时 `关闭` 不得恢复 capture 的旧设置页，而必须在同一次 `LV_EVENT_RELEASED` 内把物理和远程 canonical frame 原子提交为纯黑并抑制旧 UI 后续发布，直到新开机链路接管。后台动作仍在执行时，`关闭` 的唯一语义是携带本弹窗绑定的 `run_id` 向 action owner 发送正式取消请求，终止对应子进程及其进程组，并在收到同一 run_id 的 `cancelled` 终态后再隐藏弹窗、释放 action lock 和恢复冲突按钮。不得先隐藏弹窗、仅停止 UI 轮询或清本地状态而让后台进程继续。动作 owner 必须声明不可逆提交点；一般动作越过该点后必须移除/禁用无法取消的 `关闭`，但已受理 `保存并重启` 的结果窗必须保留 `关闭` 作为纯黑交接确认，它不再发送取消。取消请求重复发送必须幂等，旧 run_id 的迟到结果不得覆盖新任务。关闭/取消动作必须绑定在 `LV_EVENT_RELEASED` 释放事件上触发，不得在 `LV_EVENT_PRESSED` 按下阶段隐藏弹窗。弹窗隐藏前必须让当前 LVGL 输入设备执行释放吸收；弹窗隐藏后，UI 层必须对底层输入设备施加至少 200ms 的触摸抑制锁定。在该保护窗口内，除主页面急停按钮本体外，所有底层按钮、设置表格单元格、软键盘入口、`保存并重启`、`点动`、`重启` 等 operator 输入都必须在分发前被忽略，防止用户在弹窗上按下、滑动、释放时穿透到刚露出的底层控件。

`保存并重启` 的受理终态还必须保持 UI、remote relay 和 actiond 存活，直到结果窗 `关闭` 用 exact `run_id` 取得 `SETTINGS_SAVE_RESTART_COMMIT_ACK`。UI 只有在收到该 ACK 后才执行纯黑；action owner 必须在 ACK 后延迟至少 0.5 秒才开始停服务。ACK 失败时不得隐藏弹窗、不得纯黑、不得进入 reboot，必须把拒绝原因留在同一结果窗中。

`确认继续` 不是无条件绕过按钮。它必须按 native/owner 返回的结构化字段决定是否启用：

```text
allow_user_continue: true/false
confirm_continue_required: true/false
continue_token: <本次结果绑定的短期 token，可选>
continue_request_patch: <二次请求必须附加的字段，可选>
original_request_id: <产生本弹窗的原请求 id，可选，仅用于绑定弹窗来源>
current_state_cn: <当前现状>
risk_cn: <继续风险>
continue_action_cn: <点击确认继续后会执行什么>
cannot_continue_cn: <不能继续时的明确原因>
```

UI 规则：

- 所有跳窗都显示 `确认继续` 按钮。
- 跳窗不得渲染固定黄色标题文字，例如“提示”；操作员可见内容只保留三个文本项和动作按钮。
- `allow_user_continue=true` 时，按钮启用；二次请求必须带 `confirm_continue=true`，并带回原 `code/request_id` 或 Broker 要求的 `continue_token`，防止旧弹窗误继续。
- 点击 `确认继续` 的瞬间，UI 不得读取、附加或比较 SHM/display `status_epoch`；二次请求只表达操作者确认了当前提示，并带回绑定字段或 `continue_token`。当前安全状态必须由 native/owner 重新直连微内核/native actual 读取并重跑完整门禁。
- `allow_user_continue` 缺失或为 `false` 时，按钮显示为禁用态；`下一步` 必须使用 `cannot_continue_cn` 或本地化后的“当前错误不能继续执行”生成。
- UI 不得自行判断某个硬门禁能否继续；是否可继续只能来自 native/owner 的结构化结果。
- native/owner 收到 `confirm_continue=true` 后，必须在微内核/native 最新 actual 状态上重新执行完整运动门禁校验，包括急停、使能、busy/interpreter 状态、轴状态、限位、程序/刀路/几何/授权等该动作原本需要的全部 precheck。`confirm_continue=true` 或 `continue_token` 只证明操作员确认了风险提示，不得跳过任何门禁，不得信任旧状态快照或 SHM 显示投影强行运动。
- 若后端重跑 native/microkernel 门禁发现新急停、新运动、新限位、新 busy、授权/几何/状态不可用或其它 owner 要求的 actual 证明缺失，必须结构化拒绝并返回新的提示码、原因和下一步。
- 点击 `确认继续` 前，弹窗三个文本项必须让操作员知道当前原因和继续后的处理动作；不允许只显示“是否继续”。
- 点击 `确认继续` 后如果二次请求仍失败，必须显示新的提示码、原因和下一步，不得复用旧弹窗文案。

可继续的跳窗正文必须仍保持三个文本项，并且文本项之间空一行：

```text
提示: <code>

原因: <当前语言的直接原因>

下一步: <continue_action_cn；若存在 current_state_cn/risk_cn，可压缩进这一项>
```

不能继续的跳窗正文也必须仍保持三个文本项，并且文本项之间空一行：

```text
提示: <code>

原因: <当前语言的直接原因>

下一步: <cannot_continue_cn 或硬门禁处理办法>
```

典型不可继续示例：

```text
提示: ROTARY_COUNT_DOMAIN_ENCODER_CALIBRATION_REQUIRED

原因: 旋转轴编码器位数未建立本机可信校准

下一步: 先执行复位驱动，再执行设置驱动；设置驱动会写入并读回电子齿轮，再通过受控微动建立本机校准
```

显示规则：

- `提示` 使用 `ReV3CommandResult.code`；若为空，允许从 `detail` JSON 的 `code` 字段兜底。
- `原因` 使用 UI 本地的 `code -> 本地化原因` 对照表生成，当前默认语言为中文；没有映射时使用 Broker/detail JSON 中的 `direct_reason_cn`、`reason_cn`、`message_cn`、`message`、`reason`、`error` 等字段压缩生成。
- `原因` 不能只描述动作类别，必须包含阻塞事实或失败证据：例如“机器未上使能，不能回零”“程序文件哈希与加载信标不一致”“A轴驱动绝对位置读回不完整”。“回零失败”“启动前置检查失败”“驱动写入失败”只能作为内部 fallback，不能直接给操作员看。
- `下一步` 优先使用 Broker/detail JSON 中的 `handling_cn`、`next_step_cn`、`required_action_cn`、`cannot_continue_cn`、`continue_action_cn`、`operator_action_cn`；缺失时由 UI 按提示码映射生成。下一步必须是给操作员的下一步动作，不得只重复原因。
- C 端解析 Broker/detail JSON 前必须先做有界顶层对象校验：输入必须在固定字节上限内以 NUL 结束，顶层必须是单个 JSON object，闭合 `}` 后只能有空白，关键诊断字段不得出现重复顶层 key。校验失败时按结构化失败处理，不能继续用截断、尾随垃圾或重复字段结果驱动弹窗。
- 找不到提示码映射时，必须显示原始 code，并用 Broker 明细或 fallback 文案兜底生成 `原因` 和 `下一步`，不能吞掉新错误。
- 跳窗正文不得出现“请查看日志”作为操作员提示。若 UI 暂时无法本地化某个返回文本，也必须在窗口里显示原始 code、可见后台返回文本压缩出的原因和下一步，不能用“查看日志”替代原因。
- 已经格式化好的结构化正文也必须符合三个文本项格式；轴号、坐标系、驱动从站等上下文只能压进 `原因` 或 `下一步`，不得新增 `操作轴:`、`详情:` 等额外项。通用英文/泛化过滤不得把这类正文替换掉。
- 跳窗正文面向操作员，绝对不能出现底层控制框架、内部进程、脚本或调试工具名称；若 Broker 明细带出内部名称，UI 必须在显示前替换成“运动控制器”“后台通道”等产品化说法。

示例：

```text
提示: HOME_BUS_ZERO_DELTA_LIMIT

原因: 到设0位置的距离超过安全上限

下一步: 先停止运动并检查当前轴位置，确认安全后重新设0或重新回零
```

```text
提示: BROKER_BUSY

原因: 后台已有命令在执行

下一步: 等当前动作完成，或按急停停止后再重新操作
```

## 3. 本地化对照表

UI 层维护统一对照表，不允许把中文、英文或其他语言原因散落在各个按钮分支里。微内核原生错误、任务层/实时运动层/解释器格式错误和它们的 adapter 别名不在本节重复维护，完整清单及映射唯一 owner 是 `功能/微内核错误提示完整映射表.md`；本节只维护没有原生等价项的 UI 输入、后台 owner、网络、授权、资源和任务生命周期错误。

对照表字段建议：

```c
typedef struct {
    const char *match;
    ReV3ReasonCodeMatch match_type;
    const char *zh_cn;
    const char *en;
    const char *handling_zh_cn;
    const char *handling_en;
} ReV3LocalizedReason;
```

匹配规则：

- 先做精确匹配，再做包含匹配。
- 精确匹配用于稳定提示码，例如 `BROKER_BUSY`、`NO_RESPONSE`、`HOME_BUS_ZERO_DELTA_LIMIT`。
- 包含匹配用于同类前缀或后缀提示码，例如 `STATUS_UNAVAILABLE`、`STALE_EPOCH`、`JOG_PRECONDITION`。
- 当前语言默认 `zh-CN`。后续新增语言时，只扩展对照表字段和语言选择入口，不改按钮业务分支。

常见提示码映射必须覆盖，并且每个映射都必须提供下一步：

| 提示码或片段 | 原因 | 下一步 |
| --- | --- | --- |
| `BROKER_BUSY` / `COMMAND_IN_FLIGHT` | 后台已有命令在执行 | 等当前动作完成；若当前动作疑似卡住，先按急停停止，再重新执行原操作 |
| `STALE_COMMAND` / `STALE_EPOCH` | 状态时间戳已过期 | 等界面状态刷新后重新操作；不要连续快速点击同一按钮 |
| `STATUS_UNAVAILABLE` / `EPOCH_UNAVAILABLE` / `SHM` | 运行状态帧不可用或未刷新 | 等状态恢复；若持续出现，先返回主页观察状态，必要时重启后端和界面 |
| `ESTOP_BROKER_SOCKET_UNAVAILABLE` | 急停后台通道不可用，命令未下发 | 立即使用硬件急停或断使能；确认安全后检查后台通道 |
| `ESTOP_BROKER_DIRECT_FAIL` | 急停后台通道请求失败，命令未确认 | 立即使用硬件急停或断使能；不要继续运行程序 |
| `NO_PROGRAM` / `PROGRAM_OPEN` | 程序文件未打开、不可读或未确认加载 | 返回程序页重新选择并打开程序；确认加载成功后再启动 |
| `VELOCITY_PROOF_MISSING` / `VELOCITY_NOT_SEEN` | 启动已下发但未观察到速度变化 | 不要重复按启动；检查急停、使能、程序加载和解释器状态后重新启动 |
| `HOME_BUS_ZERO_TARGET_MISSING` | BUS 回零设0目标缺失或未证明 | 先完成该轴设0和设置保存/重启确认，再重新执行 BUS 回零 |
| `HOME_BUS_ZERO_COMMAND_START_FAILED` | BUS 回零运动命令未被控制器接受 | 确认急停已取消、机器已使能且解释器空闲后重新回零 |
| `HOME_BUS_ZERO_DRIVE_READBACK_FAILED` | 驱动实际位置读回失败 | 重新读取驱动状态；若仍失败，检查 EtherCAT/驱动通讯后再回零 |
| `HOME_BUS_ZERO_DELTA_LIMIT` | 到设0位置的距离超过安全上限 | 先停止运动并检查当前轴位置，确认安全后重新设0或重新回零 |
| `HOME_BUS_ZERO_TIMEOUT` / `HOME_TIMEOUT` | 回零动作超时或未到位 | 按急停或停止确认机器静止；检查限位、回零方向和速度后重新回零 |
| `SETTINGS_READ` | 设置读取来源不可用、状态帧过期或字段解析失败 | 等状态刷新后重新进入设置页；若持续失败，重启后端/界面后再读取设置 |
| `SETTINGS_APPLY` | 设置保存被具体字段、状态帧、运行态回读或重启 handoff 门禁拒绝 | 按弹窗指出的字段或阶段修正设置；重新保存，不要直接启动机器 |
| `WORK_ZERO` | 加工坐标归零所需的 WCS、轴选择或机器状态不满足 | 先确认已回零、轴选择正确且机器空闲，再重新执行加工坐标归零 |
| `WCS_SWITCH` | 加工坐标系切换失败或未确认 | 先停止程序并确认解释器空闲，再重新切换坐标系 |
| `JOG_NO_POSITION_CHANGE` | 点动下发后未观察到位置变化 | 松开点动按钮，确认轴选择/步长/使能状态后再点动；若仍无变化，检查驱动使能 |
| `DRIVE_READ` / `DRIVE_WRITE` | BUS/EtherCAT 驱动参数读取、写入或读回在具体轴、对象字典、SDO、OP 恢复或读回比对阶段失败 | 按弹窗指出的轴和阶段处理；先确认机器停止并断开运动，再重新读取或设置驱动 |
| `DRIVE_WRITE_JOB_STATUS_CHANNEL_LOST` | 后台任务状态通道连续不可用 | 不要重复点击设置驱动；等待任务状态恢复，必要时重启后端后重新读取驱动 |
| `DEVICE_AUTH` | 授权文件下载、签名验签、DNA hash 比对、时间窗或权限字段校验失败 | 检查授权文件、系统时间和本机码是否匹配；重新下载授权后再上使能 |
| `DNA_REGISTER` | 本机码登记在网络、VPS 返回、厂家 token、本地 DNA 读取或授权文件验签阶段失败 | 检查网络、厂家 token 和本机 DNA 状态；重新登记本机码后再下载授权 |
| `SERVER_DOWNLOAD` | BUS/EtherCAT 服务器下载在网络、授权、公共/私有映射表完整性或 profile 校验阶段失败 | 检查网络和授权状态；重新服务器下载，成功后再扫描/设置驱动 |

同类条件不足必须拆到具体阻塞原因，不允许只显示“条件不足”“未回零”或“前置检查失败”：

| 提示码或片段 | 原因 | 下一步 |
| --- | --- | --- |
| `HOME_STATUS_UNAVAILABLE` | 回零前状态不可用或未刷新 | 等状态刷新；若持续不可用，重启后端和界面后再回零 |
| `HOME_SEQUENCE_PREREQ_UNHOMED` | 前序回零轴未完成，不能单独回零当前轴 | 先按回零顺序完成前序轴回零，再回零当前轴 |
| `HOME_PROFILE_MISSING` / `HOME_SEQUENCE_INVALID` | 回零参数或回零顺序配置缺失、重复或无效 | 进入设置页修正回零参数，保存并重启后重新回零 |
| `HOME_SETTINGS_*` | 回零设置未证明已在运行态生效，或目标轴运行态证明缺失/无效 | 重新保存设置并等待运行态回读确认，再执行回零 |
| `MOTION_GATE_DNA_NOT_VERIFIED` | 本机码或授权未验证，运动命令被阻止 | 先登记本机码、下载授权并完成验签，再取消急停/上使能 |
| `MOTION_GATE_HOME_SEQUENCE_NOT_APPLIED` | 回零顺序策略未在运行态生效 | 保存设置并重启后端，确认回零顺序回读后再运动 |
| `MOTION_GATE_AXIS_ZERO_NOT_APPLIED` | 轴零位设0策略未在运行态证明 | 重新设0并保存/重启，确认运行态证明后再运动 |
| `MOTION_GATE_EGEAR_NOT_VERIFIED` | 电子齿轮参数未完成写入读回验证 | 执行设置驱动并完成读回验证后，再执行运动命令 |
| `MOTION_GATE_DRIVER_MODE_MISMATCH` | 当前驱动模式不支持该运动命令 | 切换到正确驱动模式并完成使能/回读确认后再操作 |
| `ROTARY_EQUIV_ZERO_RAW_UNTRUSTED` / `ROTARY_EQUIV_ZERO_AXIS_ZERO_REQUIRED` | 旋转轴当前位置或零点未建立可信 raw | 重新设0或执行真实回零，确认 raw/unwrap 可信后再操作 |
| `BROKER_ROTARY_EQUIV_ZERO_DISABLED` | 旋转轴等效回零执行门禁已关闭 | 不要继续执行等效回零；由维护人员确认门禁策略后再启用 |

`ESTOP`、`DISABLED`、`INTERPRETER_NOT_IDLE`、`NOT_IDLE`、`JOG_PRECONDITION` 的原生条件分支、`HOME_PRECONDITION_ESTOP`、`HOME_PRECONDITION_DISABLED`、`HOME_PRECONDITION_BUSY`、`POWER_ON_HOME_REQUIRED`、`POWER_ON_HOME_STATUS_UNAVAILABLE`、`WORK_ZERO_NOT_HOMED`、`NOT_HOMED` 和 `UNHOMED` 不再拥有独立可见文案；必要内部码按 `REQ-NATIVE-OPERATOR-ERROR-MAP` 的别名表归并到同一个原生错误或 owner 状态错误映射。

当 `detail` JSON 中包含 `errors`、`profile_errors` 或 `missing_prerequisites` 数组时，三个文本项必须提取第一条现场明细并压缩到 `原因` 或 `下一步`。例如 `missing_prerequisites[0].axis="X"` 时，`原因` 应显示“前序轴 X 未回零”；`profile_errors[0]` 带 `axis/code/message` 时，`原因` 应显示“X轴 <本地化明细>”。这类明细只来自 native/owner 返回的结构化 detail，UI 不得为显示原因再去查询底层控制或 HAL。

## 4. 明细字段优先级

格式化入口按以下顺序取直接明细：

1. `direct_reason_cn`
2. `reason_cn`
3. `message_cn`
4. `message`
5. `reason`
6. `error`
7. fallback 文案

继续相关字段按以下顺序提取和显示：

1. `current_state_cn`
2. `risk_cn`
3. `continue_action_cn`
4. `cannot_continue_cn`
5. `allow_user_continue`
6. `confirm_continue_required`
7. `continue_token`
8. `continue_request_patch`

如果 native/owner 没有返回这些字段，UI 必须按 `allow_user_continue=false` 处理，并显示禁用的 `确认继续` 按钮。

以下内容只能作为泛化 fallback，不得作为唯一原因：

- `执行失败`
- `前置检查失败`
- `启动失败`
- `回零失败`
- `设置保存失败`
- `驱动写入失败`
- `条件不满足`
- `状态异常`
- `后台返回失败`
- `direct failed`
- `broker direct returned guarded status`
- `accepted by broker`

如果明细中包含冒号、逗号、分号或现场变量，例如“服务器下载失败：映射表完整性校验未通过”，应保留为具体详情，不能因为包含“失败”二字就丢弃。

如果过滤泛化 fallback 后仍没有可用明细，三个文本项必须明确暴露“缺少具体原因”这个事实：

```text
提示: <code 或 UNKNOWN_DETAIL>

原因: 后台未返回具体失败原因

下一步: 保持当前状态，不要重复快速点击；重新执行原动作一次，若仍出现此结果则联系维护人员并提供提示码
```

## 5. 架构边界

跳窗格式化属于 UI 显示层：

- UI 只负责展示统一格式化后的失败/阻塞/确认结果，以及成功时的约 2 秒顶部短提示。自写功能可以消费对应 owner 的稳定业务 `code/detail`；微内核错误只能消费集中映射后的中文标题、原因、下一步和显示分级，不得读取或展示原始微内核 `code/detail`。UI 不得为了显示原因直接调用底层控制 API、读写 HAL、执行命令行调试工具，也不得读取 `/opt/8ax/phase0_bus5/settings_runtime.json` 或热路径轮询 JSON。
- 真实运动原因仍由底层运动控制、HAL/EtherCAT 或 native/owner 的结构化结果给出；SHM 只作为 UI 状态显示投影或 degraded 提示，不作为运动原因、门禁或动作证明真源。
- native/owner 必须返回结构化 `code` 和可解析 detail。UI 不得把缺失结果静默包装成成功。
- 事件日志仍保留原始 `detail`，跳窗显示只保留三个文本项本地化摘要。

## 6. 实现入口

实现入口：

- `board/app/src/v5_main_page_action_guard.c`、`v5_main_page_action_execute.c`：开机回零三段式 popup 和 command gate 结果应用；`board/app/src/v5_ui_shell_status.c`、`v5_ui_shell_refresh.c`：统一顶部状态/短提示显示。
- `board/app/src/v5_settings_page_status.c`：设置页 `settings_popup_*` 生命周期、run_id/cancel/final 状态和 first-frame guard。
- `board/app/src/v5_settings_actions.*`、`v5_settings_action_ipc.*`、`board/services/drive_profile/v5_settings_actiond.py`：resident 长动作请求、状态、取消和结构化结果。
- `board/app/src/v5_ui_shell_program_actions.c`、`v5_ui_shell_program_scan.c`、`v5_ui_shell_pages.c` 与 `board/services/program_runtime/`：程序页/MDI 本地提示和 Program Open/Load 结果；`v5_ui_shell.c` 只保留 shell 总入口。
- 原生错误中文映射必须服从 `REQ-NATIVE-OPERATOR-ERROR-MAP` 的唯一 owner；页面代码不得复制第二张原因表。
- 自写功能的跳窗继续由对应 feature owner 提供结构化动作、结果和处置语义，但必须复用统一 popup 生命周期、三段式格式、首帧保护、关闭/取消和急停遮挡规则；不得每个功能另造 popup 协议。
- UI 只消费统一 owner 给出的中文标题、原因、下一步和 `popup/top_status/log_only`，不得读取原始微内核文本后在页面内自行查表、降级或重写显示级别。

设置页轴设0的进程、成功和失败状态必须走共享轴号格式化入口，标题显示具体轴，例如 `X 设0`；正文仍保持 `提示 / 原因 / 下一步` 三个文本项。新增或修改按钮时，只允许把 `V5CommandGateResult` 或 `V5SettingsActionStatus` 失败分支接入当前 v5 popup owner，不得在按钮分支中重新拼接另一套最终文案。

## 7. 验收标准

源码验收：

- 静态检查确认所有后台命令失败分支调用统一格式化入口。
- 覆盖测试确认非原生 owner 对照表存在 `zh_cn` 和后续语言字段；原生错误映射必须对当前源码提取清单 100% 覆盖，并验证所有内部重复别名只指向原生映射、没有第二套可见文案。
- 覆盖测试确认微内核错误统一得到 `popup/top_status/log_only`：影响当前动作、安全、运动、加工结果或需要人工处置时必须弹三段式中文窗口；不影响正常使用且可继续时不得无条件弹窗；纯工程诊断只写日志。
- 覆盖测试确认微内核错误弹窗的 `提示` 是中文标题，不出现原生源 ID、原始错误码、adapter 别名、原始英文或底层实现名称；同一 `generation` 不得重复跳窗。
- 覆盖测试确认现有自写功能跳窗没有因接入微内核映射而丢失，并全部复用统一三段式格式和 popup 生命周期；各功能原因仍来自对应 feature owner，不复制微内核映射文案。
- 覆盖测试确认所有命令跳窗都保留 `急停 / 确认继续 / 关闭` 三个操作能力，其中 `急停` 直接复用主页面急停按钮本体并动态跟随其当前位置，`确认继续 / 关闭` 显示在弹窗内容区。
- 覆盖测试确认普通后台成功结果不会调用全屏弹窗；BUS/EtherCAT 设置页驱动动作、授权/服务器下载和单轴设0成功会保留同一结果弹窗，顶部简短成功提示不能替代弹窗内容。
- 覆盖测试确认 BUS/EtherCAT 单轴设0进程、成功和失败弹窗顶部标题必须显示本次具体轴，例如 `X 设0`，不得显示 `单轴设0`；成功弹窗必须携带本次写盘后的编码器当前位置、硬盘新零位、单位、差值和 `0.1 mm/deg` 容差；失败弹窗也必须携带这些字段和失败原因。只出现 `REQUESTED`、`accepted`、`queued` 或“等待后端”的弹窗只能算进行中，不能算设0成功；拿旧硬盘零位参与最终比较必须判为实现错误。
- 覆盖测试确认 `allow_user_continue=false` 或缺失时，`确认继续` 为禁用态，并显示 `不能继续` 原因。
- 覆盖测试确认 `allow_user_continue=true` 时，二次请求必须带 `confirm_continue=true`、原请求绑定字段或 `continue_token`，不得携带 SHM/display `status_epoch`；旧弹窗、过期 token 或 native/microkernel 门禁重证失败必须由后端结构化拒绝。
- 覆盖测试确认后端收到 `confirm_continue=true` 后仍重新执行该动作完整运动门禁，不能因确认继续跳过急停、使能、busy/interpreter、限位、程序/刀路/几何/授权等 precheck。
- 覆盖测试确认可继续弹窗同时显示 `现状`、`风险`、`继续后`，不可继续弹窗显示 `现状`、`不能继续`。
- `git diff --check` 不报错。

原始路径验收：

- 通过原始 UI/UDS 路径触发至少三类失败：急停/未使能、状态过期或 Broker busy、运动/驱动类失败。
- 微内核错误跳窗必须能直接看出中文标题、具体原因和下一步，不得显示原生提示码；自写功能结果仍按对应 owner 决定是否显示稳定业务提示码。
- 每个跳窗必须可见 `确认继续` 按钮；硬门禁场景按钮禁用且能看出不能继续原因。
- 跳窗正文、顶部提示、设置页状态和程序预览均不得出现底层控制框架、内部进程或调试工具名称；原始 native/detail 只能留在事件日志中供工程诊断。
- 事件日志必须保留原始 Broker detail，便于后续诊断。
- 自写功能的稳定业务提示码未收录时，仍必须按对应 owner 显示经过净化的业务码和具体明细，不能只显示泛化失败。未知微内核错误不得显示原生码或原始明细，只显示中文未知错误、处理下一步和不透明指纹，原始内容仅写工程日志。

## 8. 多语言要求

支持其他语言时必须满足：

1. 增加语言选择来源，例如配置项或 UI 设置项。
2. 在对照表中新增对应语言列。
3. 保持 `code` 稳定，避免因语言变化影响日志、测试和故障定位。
4. Broker 可以继续返回 `message_cn` 等中文明细，但 UI 的首要用户提示应来自本地化对照表；Broker 明细作为 `详情` 兜底。
