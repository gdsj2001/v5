## 主页面组成

当前 v5 主页面 owner 创建以下区域：

1. 左侧刀路视图，由 `v5_toolpath_display.*` 与主页面 owner 组合。
2. 坐标显示和坐标点选控件。
3. 手动输入和程序预览模块。
4. 右侧速度面板：主轴转速、主轴倍率、进给速度、进给倍率。

## v3 视觉布局继承规则

主页面及由主页面按钮进入的本地 UI 页面（程序页、MDI 页、刀具补偿、探测、偏置、IO、网络、设置页）必须保持冻结 v3 视觉参照的屏幕尺寸、区域位置、表格列、键盘、底部按钮、右侧导航按钮、颜色和文案；v5 只能替换底层 owner 和运行语义，不能把这些页面改成自造的标题+单行+返回、说明文字、占位提示或另一套布局。

本文只维护主页面入口、主页面按钮语义、Start/急停 owner 和主页面状态消费边界。设置页具体布局、轴参数表、G53/B 中心、驱动动作行和 `保存并重启` 口径以 `功能/2-2设置页总线模式设置项作用说明.md`、`功能/2-3设置页每个按钮作用和目的.md` 及对应 owner 为准，本文不重复维护设置页坐标细节，避免形成第二套设置页需求。

实现前必须先查对应冻结 v3 视觉参照和当前 v5 owner，按源码坐标、控件层级、文案和颜色落地；不得凭截图印象近似重画。每次改 UI 布局前必须能指出当前 v5 owner 与冻结 v3 参照的源码函数或 `object_tree.txt` 坐标锚点；验收时至少保留当前页面截图和关键控件坐标锚点，未对齐前不能写“v3 一致”。v3 有专用页面的入口（例如程序、MDI、刀具补偿、偏置、网络、设置页）按专用源码恢复；v3 本身只走 `create_aux_shell()` 的入口（例如探测、IO）按 v3 通用辅助页框架恢复，不得新增另一套自造页面。

`打开程序` 页必须保持 v3 程序页格局：左上搜索框、`本机程序文件` 表格、名称/大小/创建日期/修改日期列、右侧 CNC 键盘、顶部 `刷新` 和底部 `本机/删除/打开修改/返回` 按钮。程序行第一次触摸只选中，同一行第二次触摸打开运行；显示可以来自 v5 当前程序目录和 runtime metadata，但前台布局不得偏离 v3。

`手动输入` 页必须保持 v3 MDI 格局：左侧 MDI 编辑区、行号、`MDI>` 输入文本、右侧 CNC 键盘、底部 `清空/发送/保存/关闭` 按钮。主页面 `手动输入` 进入时创建空白 MDI 文本；只有 `发送`/`启动` 路径才按 Start/MDI owner 下发，普通键盘输入只改 UI 内存中的可见文本。

主页面 MDI/程序预览区必须保持 v3 `create_main_program_edit_hit_area()` 的透明命中区语义：坐标 `x=0,y=441,w=560,h=154`，单击只记录双击起点，不发送 MDI、Start、运动或控制命令；550ms 内第二次点击才进入 MDI 编辑页。双击进入前必须把当前可编辑对象载入 MDI 编辑缓冲：当前 runtime 为 MDI 时载入当前 MDI 文本；当前 runtime 为已打开程序时优先使用运行内存中的原始 G-code 文本和源路径作为修改对象；当前 UI 已有未下发 MDI 编辑文本时保持该文本；没有已打开程序或文本、文本超出 MDI 编辑缓冲、路径不是本机 G-code 目录内可写回文件时 fail-closed 并停留当前页，不得默认加载 `cc-ac.ngc`、`cc-bc.ngc` 或其它 fallback 程序。程序页 `打开修改` 与主页面预览区双击必须复用同一条“打开到 MDI 修改区”语义；MDI 页 `保存` 只允许在存在有效原始 G-code 文件元数据时写回该文件，不能替代 `发送`/`启动` 运行路径。

MDI 编辑缓冲必须只有一个字符位置作为插入点/光标 owner。通过主页面预览区双击或程序页 `打开修改` 进入 MDI 页后，左侧编辑区首个可见帧必须显示约 500ms 周期持续闪烁的可见光标；新载入的程序默认把光标放在文本开头，重新进入尚未替换的同一编辑缓冲时保留现有位置。右侧 `<`/`>` 必须逐字符移动，`^`/`v` 必须按当前宽度换行后的可视行上下移动；普通字符、`SP`、`EOB` 和 `BKSP` 必须在该光标位置插入或删除，禁止继续固定追加到文件末尾。左侧编辑区每页固定18个可视行，`UP PAGE`/`PAGE DN` 必须把同一光标向上/向下移动18个可视行并把视口切到光标所在页；方向键、输入或退格跨过页边界时也必须自动换页。左侧行号必须随当前页更新，光标始终位于当前18行范围内，不允许出现文本翻页但光标留在旧页、光标移动但视口不跟随或另建第二套页码位置。MDI页隐藏时停止光标动画，进入时从同一位置恢复，避免隐藏页面持续刷新。

MDI/主页面程序预览高亮行必须跟随 LinuxCNC/微内核 native actual 的当前执行行：MDI 运行态优先使用 native MDI run line/command；暂停态优先显示 native paused line，其次使用 fresh `current_line`，最后才使用 fresh `motion_line`；非暂停程序运行态必须优先使用 fresh `motion_line` 作为实际队列执行行，只有没有 fresh `motion_line` 时才允许退回 fresh `current_line`，不得把解释器预读到后续行的 `current_line` 当成正在执行行。高亮行必须始终显示在主页面 MDI/程序预览区域内：当 native actual 行号不在当前 4 行可见窗口内时，预览窗口必须随行号滚动/换页，让 actual 行落入可见行并高亮；不得停留在文件首行、上次可见窗口或打开程序时的缓存窗口。程序打开但还没开始运行、还没有同一程序 native actual 行历史时，默认高亮第 1 行，并让第 1 行显示在预览区内；操作员在空闲/非执行态手指拖动查看文本时，可高亮当前 4 行可见窗口第一行作为阅读光标，但这只表达查看位置，不得作为 native execution 证明。开始运行后，高亮行视觉位置默认固定在 MDI/程序预览 4 行窗口的第一行：当前 native actual 行号变化时，窗口起始行跟着变为该行号，让当前执行行出现在第一行；运行过程中的每个 fresh native actual 行号变化都必须驱动下一帧预览滚屏，不得只在程序结束、暂停、急停或慢速轮询后跳到最终行；只有执行到 G-code 最后几行、后续行数不足以填满 4 行窗口时，窗口固定到文件最后一屏，高亮才从第一行向下移动，直到最后一行停在窗口最后一行。高亮表达“当前或最近一次 native actual 执行/暂停所在行”：同一已打开程序内一旦收到 fresh native actual 行号，UI 必须保存该 native 行号并持续高亮、持续让该行落在 4 行预览窗口内；运行结束、Stop、Abort、急停或 native 后续行号暂时清零后，必须停留并高亮急停/停止前保存的当前执行行，不得回到第 1 行，也不得只保留滚动位置而清掉高亮。打开新程序、载入新 MDI、当前可见文本无法映射到该 native 行号时，才清掉保存的 native 高亮行。若已存在保存的高亮行或运行/暂停时出现 fresh native actual 行号，则 native actual 行号优先，必须强制滚回当前/最近执行行所在窗口。

## 通用分发规则

1. 主页面可见按钮来自 `re_v3_buttons[]` 中 `page == RE_V3_PAGE_MAIN` 的记录。
2. 按钮事件进入 `handle_button_press()` 后按 `shell_action` 分发。
3. 急停按钮走最高优先级本地路径，不受普通 `command_busy` 拦截。
4. 普通按钮只保留 50ms 重复点击去重；`command_busy` 只能作为显示/诊断字段，不得让非 VPS 普通按钮在 UI 本地拒绝下发。
4a. UI 级交互防误触只能作为视觉提示或真实 owner 返回后的状态表达，不能成为新的产品层 gate。唯一固定前置例外是 `REQ-POWER-ON-HOME-PRECONDITION`：要求开机回零的动作在 fresh native `all_homed` 为 false/unknown 时必须不下发并弹三段式窗口，同时由 command gate 二次校验。除此之外，按钮可根据 native/microkernel actual 做状态弱化、图标变化或提示，但除急停直通、真实不适用、资源确实不存在或 native owner 明确 fail-closed 外，不得用 UI 置灰替代动作打通；强制触发时仍应进入对应 native owner并取得结构化结果。
5. `*_LOCAL` 类动作只改 UI 本地状态或页面，不向后端下发运动命令。
6. 非 VPS/授权/下载链路的普通 `*_COMMAND` 不得再通过 C 端 JSON/Broker 请求构造器发送；已具备 native/LinuxCNC gate 的动作由 UI C 直接调用 canonical native gate，未补回 native gate 的动作保持真实缺口记录，不恢复旧 C JSON 中介。
7. C 端新增或改动的普通控制请求不得手写 `snprintf` JSON、不得恢复 `v3_json_utils`、不得新增 `re_v3_product_command_broker_request_direct` 或等价 Broker direct 胶水；VPS/授权/下载链路如需 payload，只能留在对应身份链路 owner 内。
8. 主页面普通动作失败默认只写顶部短提示与事件日志；Home/positioning 是明确运动结果例外：开机回零前置拒绝，以及已经提交的机械全轴 Home、MCS 单轴回机械零、WCS 单轴回加工零的任何终态失败，都必须使用 `REQ-UI-POPUP-RESULT-POLICY` 的统一全屏三段式提示窗。Home 失败弹窗必须绑定本次 run_id/generation，只能消费 fresh native owner 返回的具体轴、失败阶段、直接原因、actual/target、motion/in-position/静止状态或缺失 readback 和可执行下一步；Home 不得传输或显示 target-error 容差。禁止只写“回零失败”“执行异常”、内部英文码或 adapter 名。操作员主动急停导致且 native 明确回读 cancelled 的事务显示 `回零已取消` 顶部状态，不冒充失败弹窗。不得恢复 `re_v3_ui_show_error_popup()`、`show_command_result_failure_popup()` 等旧中介或新增 Home 私有 popup geometry/formatter；必须复用公共 popup owner。
9. 顶部短提示和事件日志不得出现底层控制内核品牌或项目字眼，只显示面向操作者的中文原因、结果码和可判断的直接原因。

## 主页面显示总规则

主页坐标/轨迹、转速、进给速度和主轴/进给倍率这些允许字段只能来自 State Publisher 发布到 typed SHM 的同一帧运行内存投影；当前模态、WCS、RTCP、tool/TLO、安全、Home、active mode、CPU/网络和动作结果必须直连微内核/native/provider 或诊断 owner。UI 不能直接读取 `/proc`、`/run` JSON、Broker result、settings/runtime 文件、旧本地缓存或按钮 dirty 值作为显示真源。SHM 缺帧、typed group 缺失、版本不匹配或字段降级时，主页对应显示必须进入 `--`/degraded 状态，不得继续显示旧帧数值。

主页面刷新频率必须分层：坐标大字、轨迹当前位置、红/绿点、刀柄/刀尖动态线、转速、进给速度、主轴/进给倍率和 MDI/程序预览 native actual 行号滚屏按 30Hz 目标刷新；普通按钮状态轮询按 10Hz；右下急停按键的显示状态、急停 latch 和 Machine Enable native 回读刷新按 10Hz；CPU/网络、普通诊断文字、后台信息、非运动状态摘要和其它不影响运行连续感的 UI 显示按 5Hz 即可。运行期刷新禁止用周期性整屏刷新或整页重建兜底，必须按坐标数字、按钮、急停、轨迹动态图层、程序预览行、诊断文字等 dirty 对象/dirty 区域分块按需刷新；页面切换首帧 cache/blit 不是稳定刷新循环。急停按下事件本身是 edge 直通 native safety owner，不能等待下一次 10Hz 轮询；刷新节流只影响显示和普通轮询，不能影响命令下发 freshness、急停 0.1s 生效链路、动作证明或安全/运动门禁。

G-code 运行中，主页坐标大字、刀路当前位置、红/绿点和实际刀尖/刀柄反馈点必须继续消费 State Publisher `motion_fast` 的 30Hz 级 fresh native 状态 帧。CPU 优化只能删除/降频非必要诊断、远程转发、重复 LVGL 写入、隐藏分区刷新或慢投影重算；不得把坐标/刀路主刷新降到低频、不得跳过实际位置帧、不得用旧坐标缓存或命令坐标冒充 30Hz actual。

主页普通按钮控制链为 UI C -> 已登记 native command gate -> LinuxCNC/HAL/EtherCAT/native owner；VPS/授权/下载和设置长流程按对应 owner 使用 Command Broker UDS/后台 job。按钮表、动作登记表和 provider 必须来自开机 RAM 闭包；运行期按钮下发不得通过硬盘脚本发现、动态导入 action、读取 wrapper 或读取 UI 资源兜底。按钮成功只允许由 native/对应 owner 触发结果和 microkernel/native 回读确认，不允许用主页本地视觉状态替代后端执行结果。

主页面按钮高亮只允许表达“当前已生效/当前选中/当前实际模态”状态，不能表达“可点击”“已提交请求”或“默认猜测”。WCS 组只能高亮 native actual 当前 WCS；`X1/X10/X100` 只能高亮当前点动步距；`机械全轴` 只能在当前确实选择 MCS 全轴时高亮；RTCP 只能在 fresh native actual 为 ON 时高亮；视图按钮只能高亮当前视图。启动、暂停、回零、归零、点动、首点、打开程序、手动输入等动作按钮没有同源已生效状态时必须保持非高亮底色，即使按钮可点击或命令已提交，也不得提前用高亮冒充已生效。

所有页面、所有可见按钮必须复用同一个中央按钮视觉入口，不得由主页面、设置页、弹窗或局部页面各自保留不同的 pressed 清理实现。黄色只允许表达真实触摸/远程输入按下期间的 LVGL `PRESSED` 瞬时反馈；收到 `RELEASED`、`PRESS_LOST` 或输入取消后，必须在该输入事件事务内立即清除 pressed 状态、立即刷新，并恢复为原底色或对应的非黄色 actual 状态色，不能等待按钮命令、Home、驱动动作、弹窗动作或下一帧 UI 刷新结束。任何按钮都不得把黄色用于“命令已提交”“等待结果”“执行中”“当前选中”“当前生效”或错误提示后的残留状态。

动作按钮在松手后的执行态也统一遵守同一规则：只有对应 command gate/native/provider owner 能证明本次 transaction 已进入执行中，按钮才允许使用统一的非黄色 active 状态色；transaction 成功、失败、取消或失去 fresh owner 证明时必须立即恢复原底色或对应 actual 状态色。`回零` 的 `机械全轴` 对应 Home transaction，MCS/WCS 单轴对应 positioning transaction；执行中可显示非黄色 active 高亮，但 pressed 黄色必须在手指/远程指针松开时先立刻消失，不能一直保持到回零结束。没有可回读执行态的普通命令按钮松手后立即恢复，不得自造执行中高亮；命令已提交但未执行、旧 result、旧 `all_homed=true`、当前位置接近 0、pressed 状态残留或本地缓存都不得维持黄色/高亮。

急停实时下沉硬边界：`estop_force` 的执行 owner 必须下沉到 LinuxCNC/微内核/HAL/EtherCAT realtime/native safety 层，形成常驻 emergency latch/gate；当前 BUS 主线的最终 owner 是 servo-thread 中的 `v5_safety_latch` realtime component，LinuxCNC 启动的 C `v5_native_hal_owner` 只通过窄 UDS 接收 edge、驱动 force/reset pin、维持 heartbeat并等待 realtime actual。heartbeat 超过 0.1s 未变化必须由 realtime latch 自主 fail-closed；Python 进程、5ms sleep loop、可写 control SHM 或 `halcmd` 不得承担 safety owner。UI、remote input、Command Broker 和已登记 urgent/native adapter 只能直通触发该 latch/gate、记录一次动作证据和读取 microkernel/native 结果，不得在 UI/Python/linuxcncrsh 普通命令链里承担急停生效职责。急停按下后不允许任何 V3 产品门禁挡在微内核急停之前：不得等待或检查 `command_busy`、backend-ready、status epoch、状态 fresh、身份/DNA/授权、motion 状态、CPU isolation、当前是否已急停、后台 job cancel、弹窗状态、页面状态或任何普通动作锁。急停直通下发后必须回读微内核/native memory 状态确认是否生效；确认前 UI 只能显示急停待确认并允许再次直通急停。确认生效后 UI 必须立即切换为取消急停入口，并启动同一 `estop_clean_generation` 的开机等价 clean-state 事务；该事务在 safety actual 之后执行，绝不能挡在 0.1s 急停链之前。急停热路径不得依赖 Python handler、普通 TCP/text gate、临时脚本、文件/JSON 轮询或 HAL helper 写 pin 作为最终执行 owner；这些只能在急停直通下发之后作为触发 adapter、异步清理或诊断证据，目标闭环必须由微内核 realtime/servo cycle、HAL realtime component、EtherCAT/drive safety 或等价 native owner 立即禁止运动输出。
C UI 急停实现边界：`execute_estop_priority_command()` 只能直接调用 native safety gate，由该 gate 更新微内核 realtime emergency latch/epoch 并等待同源 native ack/actual；`estop_force` 路径不得打开或复用 linuxcncrsh 普通文本控制链，不得发送 `Set Machine Off`、`Set EStop On`、MDI、HAL helper 写 pin 或其它用户态替代命令。UI 只读取 native safety actual 改变按钮和顶部显示，不写本地急停真值。`execute_estop_reset_gate_command()` 同样先直通 native safety reset/cancel-estop 并回读 latch 已清；恢复上使能是 latch 清除后的独立低频动作，可以由已登记 command gate 发送 `Set Machine On`，但该文本动作不属于急停生效链，也不能反向证明 force 成功。这两条路径不得恢复 Broker/Python/JSON/product action 中介、HAL helper、UI 本地急停真值或 removed-route 占位；请求已发送只作为一次触发证据，按钮语义和最终成功必须由 microkernel/native memory readback 驱动。禁止 `RE_V3_SHELL_FAULT_RESET_COMMAND` 入口；设置页“清除故障”只保留 `drive_fault_reset`/驱动映射表 owner，不走急停 safety 分支。

服务启动/重启只建立显示与控制链，并保持 realtime latch 安全态；不得发送 safety reset 或 Machine On。只有用户点击 `estop_or_reset` 的恢复链在确认 latch 已清后才允许恢复使能，最终结果仍由 native `estop_active` 与 `machine_enabled` actual 裁决。

急停输入与上层状态边界：按键、触摸和远程输入只允许产生 `estop_force requested` 的 edge event，不能成为急停 owner，不能执行门禁、排队、状态证明、上层锁存写入或恢复判断。上层急停状态直接读取微内核内存急停状态；V3 不再维护 `V3 上层急停请求锁存` 或 software emergency latch 作为产品状态源。按下瞬间不得由按键驱动、UI 回调或 remote input 写上层急停状态；UI 最多显示瞬时按压/发送中视觉，最终急停中、取消急停、待确认和恢复为急停都必须由 microkernel/native/microkernel 回读驱动。V3 不得直接改 microkernel/native 状态真值来伪造 `estop=true`、`enabled=false` 或运动输出禁止。

急停记录边界：急停按下必须记录，但记录不得由按键驱动直接写成产品状态，也不得挡在急停直通前面。输入事件只随直通请求携带 request_id、来源和 monotonic 时间给微内核/急停 adapter；上层内存记录由状态同步链直接读取微内核内存急停状态和 request/ack 时间自动生成。持久化日志、结果文件、审计明细、队列清理摘要和耗时统计只能在微内核急停请求已下发后异步写入，或在 microkernel/native 回读确认后补齐；写文件失败、日志失败或审计失败不得影响急停下发、不得影响 0.1s 生效窗口，也不得被当作急停失败或成功证明。

急停时限硬边界：从 UI/remote input 确认急停按下或 Broker 收到 `estop_force` urgent 请求开始计时，0.1s 内必须让 LinuxCNC/微内核/HAL/EtherCAT 急停 owner 完成真实急停生效，并能从 native/microkernel 回读证明 `estop=true` 或 `enabled=false`、运动输出已被禁止且速度进入急停停止边界。UI 本地视觉状态、事件日志、命令已发送、Broker accepted 或结果文件写入都不能单独算作 0.1s 急停成功证明；取得真实急停证明后，UI 必须立即把右下按钮和顶部恢复入口切换为取消急停语义。超过 0.1s 未取得真实急停证明时必须 fail-closed 并保持急停待确认视觉和控制语义。

急停生效后的干净执行态边界：本文所称“恢复到开机上电的干净状态”只指执行链回到无活动动作、无残留命令和无后台事务的干净态，不等于清空操作员上下文。微内核/native 回读确认急停生效后，Command Gate 必须以新的 `estop_clean_generation` 终止当前 Auto/MDI/程序执行，清空未执行的 motion/Start/Jog/Home/positioning 请求与解释器/任务队列，终结本次 Home/程序动作的 active/pending 生命周期，强制 RTCP UI latch 与残留程序输出请求为 OFF，并回读 task/interpreter idle、queue empty、RTCP OFF 和 clean terminal。UI 消费同一 clean generation 后只清旧弹窗/进度、按压态、本地输入队列、旧 transaction poll 和其它短生命周期执行态；必须保留当前已打开/选中的原始 G-code identity、主页面程序名、程序页选择、黄色刀路预览，以及急停前仍有效的 native `homed/all_homed` 证明。取消急停后不要求重新打开程序，也不要求再次 Home；用户若再次按 Start，仍使用同一已登记原始程序身份并从 owner 定义的重新启动入口执行。该事务不得停止或重启 LVGL、remote relay/input、Command Broker、State Publisher、LinuxCNC/HAL/EtherCAT/native owner 等常驻服务，不得删除或重读 G-code 文件，也不得改写 runtime INI、驱动 profile/egear、运动模型、G53/WCS/G92/刀补、授权、故障与急停审计等持久 owner；“重启整套 backend”不能冒充 clean-state。清理失败必须保持 ESTOP/Machine Off 并给出本次结构化失败，后续运动 fail-closed。

急停 clean-state 的性能终态：程序运动或预览产生的高频计算、刀路线重投影、旧 transaction poll 和远端大脏区发布必须随本次易失运行态清除而停止；在状态/显示常驻链继续正常刷新前提下，CPU1 必须在有界采样窗口回到开机空闲基线。不得通过降低产品刷新率、停止 State Publisher/remote relay、冻结 actual 或全屏刷新掩盖残留任务。

急停取消时限硬边界：从 UI/顶部状态条/右下恢复入口确认取消急停点击或 Broker 收到 `estop_or_reset` urgent 请求开始计时，1s 内只执行并确认这条链路：直通微内核 reset/cancel-estop、读取微内核内存急停状态、确认底层 latch 已清、再发起上使能并读取微内核 Machine Enable 状态。右下按钮是否恢复为“急停”必须跟随微内核 `enabled=true` 且急停锁存已清；若 latch 已清但仍 `enabled=false`，按钮继续显示“取消急停”作为上使能/恢复入口。1s 窗口不得等待或检查解释器 idle、底层 motion-ready、5 轴 Operation Enabled、急停源/硬故障、队列/缓存等其它条件；这些状态只在顶部状态/问题区域显示。`BROKER_JOB_ACCEPTED`、后台 job 已提交或 machine-on 命令已发送不能单独算作按钮恢复完成；未读到微内核 latch 已清和 `enabled=true` 时不得把按钮提前恢复为“急停”。

急停按钮颜色边界：右下按钮底色始终保持红色。急停已下发但微内核/native memory 未确认生效时，右下按钮显示“急停中”并保持急停直通语义；微内核/native memory 确认急停生效或确认 `enabled=false` 后，右下按钮显示“取消急停”并发送 `estop_or_reset`；`estop_or_reset` 等待微内核 latch 清除和 Machine Enable 回读期间，用黄色 stop 方块/图标作为统一视觉提醒；读到微内核 latch 已清且 `enabled=true` 后，右下按钮恢复红色“急停”和白色 stop 方块/图标，其它未 ready 状态只在顶部显示。

急停取消后的顶部状态边界：`estop_or_reset` 读取微内核 latch 已清且 Machine Enable 已恢复后，按钮才恢复为“急停”；顶部状态/问题区域再显示其它未 ready 条件，包括解释器 idle、底层 motion-ready、BUS/EtherCAT 5 轴 Operation Enabled、急停源/硬故障、队列、暂停、行号、运行文件或 active MDI 等。顶部问题可以阻止后续运动动作，但不得替代右下按钮的微内核使能状态判断。V3 不得用 SHM 速度、JSON 或私有容差自造 motion-ready。

顶部状态固定按安全与时效优先级选择第一条可用信息，不允许由最后一次写入或控件刷新顺序互相覆盖：P0 是 fresh microkernel/native `estop_active=true`，固定显示 `急停中`，必须覆盖正在关闭 RTCP、Home/单轴定位进度、native 操作错误、回零状态未知和未回零；只有 fresh 同源 actual 明确回到 `estop_active=false` 后才允许显示下一优先级。P1 是绑定本次 run_id/generation 的 Home/positioning 进度或终态；P2 是仍在有效显示窗口内的 fresh native operator error；P3 是回零状态未知、未回零或已经回零后的空状态。低优先级状态不得覆盖高优先级状态，旧急停、旧 Home 结果、按钮文字、按钮颜色、SHM 或本地布尔量不得参与排序。

急停恢复 disabled-only 边界：`enabled=false` 是右下急停按钮的核心显示/分支输入，不是特殊快路径。所有取消急停入口都走同一条最小链路：直通微内核 reset/cancel-estop、读取微内核内存急停状态、上使能、读取微内核 Machine Enable 状态；只要微内核仍回读 `enabled=false`，按钮继续显示“取消急停”并继续走恢复入口。非 idle、速度、队列或驱动 Operation Enabled 等其它状态只在顶部显示。

急停恢复前台响应边界：`estop_or_reset` 按钮/顶部状态条前台不得在 UI 主线程同步等待 native machine-on 完整确认。前台只允许提交 Broker 后台 job 并立即进入 pending 显示，保持界面可刷新、可继续响应急停；后台 job 完成后按微内核急停 latch 与 Machine Enable 回读决定按钮是否从“取消急停”恢复为“急停”。`BROKER_JOB_ACCEPTED` 只表示恢复任务已提交，不能当作取消动作已经完成；取消动作完成必须来自微内核 latch 已清且 `enabled=true` 的回读。fresh native 状态/status 和驱动 Operation Enabled 证明只用于顶部 ready/问题显示和后续运动准入。

急停恢复结果结构边界：`estop_or_reset` 的 `ok`、`code` 和 `RESET_LATCH_CLEARED` 只能表示微内核 latch 已清；上使能请求必须用独立的 `machine_on_requested`、`on_result`、`machine_on_error` 等字段表达。Broker 紧凑响应不得丢弃这些字段，也不得把 machine-on 失败反向改写为取消急停失败；后续 machine-ready、顶部状态和运动准入必须读取这些独立字段和 fresh microkernel/native 回读继续判断。

急停后台清理边界：`estop_force` urgent 请求不得为了身份校验、job ownership 或后台清理而延迟急停直通微内核。Broker 管理的可取消产品后台 job 可以在微内核 emergency latch/gate 已触发之后异步 `cancel_all()`，取消原因记录为 `estop_force:<request_id>`，并在后续结果或诊断中带回取消摘要；确认急停生效时，UI 侧流程/队列清理可以与 UI 切换取消急停同步执行。取消后台 job 和清 UI 队列都不是急停下发门禁，也不能挡在急停生效前。这个清理只覆盖 Broker 管理的可取消产品后台 job，例如服务器下载、授权下载、驱动设置/复位/读写、回零等已接入 `job_token` 的长任务；不得杀掉 LinuxCNC/微内核、State Publisher、Command Broker、remote relay/input、LVGL 主进程等产品常驻安全/状态/显示链路，也不得新建第二套进程表或用 shell `kill` 扫描替代 Broker job 取消。`estop_or_reset` 恢复上使能不触发后台 job 清理。

急停恢复显示边界：右下按钮状态必须与微内核 Machine Enable 状态一致，并同时受微内核急停 latch 约束。`enabled=true` 必须来自 LinuxCNC/HAL/native 实际使能输出的回读，例如同一 native owner 暴露的 Machine Enable actual、`joint.*.amp-enable-out`/等价 HAL native 使能输出或微内核常驻内存中的同源 actual；`Set Machine On` 已发送、`linuxcncrsh Get Machine` 文本返回、UI 事件 `machine_on_status`、命令 accepted、旧缓存或本地按钮状态都不能单独把按钮恢复成“急停”。只要微内核/native actual 回读 `enabled=false` 或 `estop=true`，右下按钮必须显示“取消急停”并点击进入恢复/上使能链路；只有 actual 回读 `enabled=true` 且 `estop=false` 时，按钮才显示“急停”并点击进入急停直通链路。急停已下发但尚未被微内核/native memory 确认生效的 pending 窗口，右下按钮保持急停直通语义；一旦真实 `estop=true` 或 `enabled=false` 停止证明确认，右下按钮必须切换到取消急停。若 `linuxcncrsh` 失联、返回空响应或只读到 `Get Machine` 但实际 HAL/native 使能输出仍为 false，按钮必须按 `enabled=false` 处理，不能沿用上一帧已使能状态。

## Start 分支规则

`REQ-START-RUN-HOT-PATH` 最终口径：启动按钮不是综合准入门禁，也不是程序文件续跑入口。Program Open/Load 必须在 Start 前完成 resident program model、首个完整 scene 和 screen fit，并由唯一 preparation owner 原子发布同一 program generation 的 `program_scene_ready=true`；非暂停程序模式只有该 token fresh 且 generation 匹配才算“可 Start”。按下启动按钮时，产品层只做三个有 owner 的判断：本次开机 fresh native `all_homed` 是否成立、之前停下来是不是因为暂停，以及非暂停程序模式的 `program_scene_ready` 是否成立。`all_homed` 为 false/unknown 时按 `REQ-POWER-ON-HOME-PRECONDITION` 不下发并弹三段式窗口；`program_scene_ready` 未成立时保持 loading/不可 Start，不得点击后补解析、补 scene 或补 fit。除此之外，UI 不得再设置 V3 自写准入门禁，Broker 和 Python product action 不得再作为启动/暂停中间层。启动、暂停、恢复必须由 UI C 端直接向 LinuxCNC/微内核 native command gate 下发。微内核/LinuxCNC/HAL/EtherCAT 的硬安全、实时限位、急停、伺服/驱动保护，以及 VPS/授权/下载身份链不属于本条删除范围，仍由各自 owner 执行。

启动分支：

- 如果当前/最近停下状态是暂停：启动按钮直接下发 native `Set Resume`，由 LinuxCNC/native 从暂停 MDI 行继续执行后续内容。
- 如果不是暂停且处于程序文件模式：启动按钮只使用 Program Open/Load 阶段已登记的原始程序身份/`runtime_program_path`，每次非暂停 Start 都必须在同一事务内按 LinuxCNC 原生顺序下发一次 `Set Task_Plan_Init`、`Set Mode Auto`、且只下发一次 `Set Open <runtime_program_path>`，最后 `Set Run 0`；`Set Open` 路径必须仍指向操作员打开的原始 G-code 文件，不得指向 UI 预览、黄色轨迹、等效角改写、sidecar、临时副本或其它显示链路产物。该序列从原始程序第 1 行开始执行；非暂停路径不得追加 `Set Resume`，不得在 Start/Resume 热路径重新读程序文件、计算 SHA、生成 runtime 文件或读取脚本/配置。
- Start 原生序列不得增加“解释器正在 AUTO waiting/reading/running”产品门禁。序列中的重复 `EMC_TASK_PLAN_INIT` 被正在等待的 AUTO 解释器拒绝、但已接受的 Start/程序执行未中断且运动结果未受影响时，这只是无实质影响的工程诊断：后续原生命令仍按现有事务语义执行，State Publisher 必须按 `REQ-NATIVE-OPERATOR-ERROR-MAP` 精确分类为 `log_only`，不得弹窗、不得把传输已受理改写成启动失败，也不得据此阻止再次 Start。
- 非暂停 Start 不得根据 `Get Program` 返回“当前已打开同一路径”而省略 `Set Open`：程序在 `M30`/停止终态后仍保留相同路径不等于解释器已经 rewind，省略 Open 会使 `Set Run 0` 停留在上一轮末行。UI 在该 Start 事务成功提交后先把主页面 MDI/程序预览恢复到第 1 行；fresh native actual 行出现后立即由 actual 接管。暂停 Resume 不得复位预览行，必须保留暂停行并从原暂停点继续。
- 如果不是暂停且处于可见 MDI 模式：启动按钮直接下发 native `Set Mode MDI`，再把当前可见 MDI 文本的可执行行交给 LinuxCNC/微内核执行。
- 非暂停启动不得根据当前坐标、当前行号、上一次停止行号或 native 当前已加载程序行号决定入口；不得从中间行继续。
- 暂停恢复不得重新从 MDI 首行开始；必须使用 native 暂停恢复语义继续暂停行之后的内容。
- 暂停按钮直接下发 native `Set Pause`；暂停态再次按暂停按钮直接下发 native `Set Resume`。
- `start_run`、`pause_resume_program`、`product_handler_start_run.py`、`product_handler_pause_resume.py`、`actions_program_start.py` 和 `actions_controls.run_pause_resume` 不再是启动/暂停支持路径；同一 slice 必须从 UI/Broker/action registry、能力清单、测试和发布面删除这些自造入口，不能保留为兼容、fallback、诊断或隐藏分支。

除 fresh `all_homed` 和非暂停程序模式的原子 `program_scene_ready` 外，启动按钮禁止新增门禁：

- 不检查当前坐标。
- 不以 CPU isolation beacon、motion ready latch、参数/驱动 proof、runtime INI/native 投影、program manifest、散落的 loaded_epoch/source/runtime SHA256、native 已加载文件回读、status epoch、backend-ready、速度证明、行号推进、encoder delta 或 motion proof 作为是否允许下发启动的前置门禁；程序准备只认唯一 preparation owner 原子发布的 `program_scene_ready + program_generation + scene_generation + fit_generation`。
- 不在 Start 中重新读取 G-code 文件、计算 SHA256、重新过滤 G-code、重写 runtime 程序、补齐后台 proof、解析黄色模型、构建 scene 或执行 fit；Program Open 已准备好且 `program_scene_ready` 成立但 native 当前未加载时，Start 只能由 UI C 端做一次最小 native runtime handoff，不得回到 Broker/Python action。
- 除 `program_scene_ready` 外，上述状态只允许作为诊断字段和后台记录，不能阻止启动按钮按“暂停续跑 / 非暂停首行”的规则下发；不能把诊断结果反向改写成 `START_*_UNAVAILABLE`、`START_*_NOT_SEEN`、`BROKER_START_RUN_*` 或其它产品层失败门禁。

启动结果边界：

- `Set Run`、`Set Pause` 或 `Set Resume` 下发后，UI 只记录 native command gate 返回的最小结构化结果；不得再由 UI/Broker/product action 自写运行证明门禁。
- 原生错误是否可见只按实际影响和自动恢复终态分类。没有中断有效动作、没有影响安全/运动/加工结果、没有操作员处置动作的错误不得弹窗；AUTO waiting 中对 `EMC_TASK_PLAN_INIT` 的重复初始化拒绝固定只写工程日志。真正影响 Start 的事件若存在原生 owner 已登记的安全有界恢复，必须在同一 Start 事务内恢复、fresh actual确认后自动继续且不弹窗；恢复不存在、失败、超时、运动/加工结果不可信或需要操作员选择时才由统一错误 owner 弹窗。UI 和错误映射层不得为了自动继续自行重发或重放 Start/G-code。
- 产品层启动前置只保留 `REQ-POWER-ON-HOME-PRECONDITION` 的 fresh native `all_homed`，以及非暂停程序模式由 Program Open/Load owner 前置完成的原子 `program_scene_ready`。Program Open 已准备好原始程序身份、resident model、首个完整 scene/fit 且 native 当前未加载或加载为 `NONE` 时，Start 在上述两项成立后只提交已准备好的 runtime 程序并继续下发运行，不得增加其它下发前阻断。
- 速度、running、位置、行号、encoder delta、状态 fresh、status epoch 和 `/run` 结果文件只能作为诊断显示或事后记录；不得成为启动是否允许、是否成功返回、是否弹失败窗的 V3 自写判定。
- 真正的运动保护、急停、限位、驱动故障和硬实时联锁由 LinuxCNC/微内核/HAL/EtherCAT owner 负责；VPS/授权/下载身份链仍归 `REQ-DRIVE-PROFILE-AUTH-CHAIN`，不得因本条删除。

## 坐标点选对按钮的影响

坐标列本身不是 `re_v3_buttons[]` 按钮，但它会改变后续按钮目标：

1. 点选机械坐标单轴：设置 `home_select_space="mcs"`、`home_select_axis=X/Y/Z/A/C`。
2. 点选加工坐标单轴：设置 `home_select_space="wcs"`、`home_select_axis=X/Y/Z/A/C`。
3. 点选 WCS 全轴：设置 `home_select_space="wcs"`、`home_select_axis="*"`。
4. 回零按钮严格按当前选择分支：`机械全轴` 提交 BUS/Pulse 正常全轴 Home；MCS 单轴提交该机械轴真实移动到 MCS 0；WCS 单轴提交该轴真实移动到当前 active WCS 0。每次按下都必须有新的 native transaction 和真实运动/到位/readback；不得因坐标接近 0 或旧 homed 空跑成功，单轴分支不得回退全轴 Home。
5. 归零按钮支持从 MCS 或 WCS 坐标列选定的单轴；列别只确定选中位置，写入源始终是该轴 fresh 机械坐标关系，目标始终是当前 active WCS 对应轴偏移。全轴选择不能执行 `work_zero`。
6. 点动按钮当前只支持单个坐标轴选择；全轴选择不能点动。
7. 点选任意 MCS/WCS 单轴后必须启动 3 秒 UI 空闲计时。从最后一次重新点选单轴，或执行使用当前轴选择的点动、归零、回机械 0、回加工 0 等有效操作起重新计时；连续按压操作尚未释放时不得判为空闲。连续 3 秒没有上述操作时，UI 必须自动执行与点击 `机械全轴` 相同的本地选择恢复：清除 MCS/WCS 单轴目标和对应坐标行高亮，恢复为 MCS 全轴选择；超时后再按 `回零` 应进入 BUS/Pulse 正常全轴 Home，后续点动必须重新点选单轴。`REQ-POWER-ON-HOME-PRECONDITION` 三段式模态提示显示期间必须暂停后台选择超时，不能在提示窗后重置选择并覆盖提示；操作员点击 `关闭` 后从 3 秒重新计时，仍无操作才恢复 `机械全轴`。该超时只改变 UI 选择状态，不发送运动、回零、归零或 Jog Stop 命令；`机械全轴` 按钮只按现有已选定状态色显示，不得使用或残留黄色 pressed 状态。
8. 点选、改选或取消 MCS/WCS 单轴选择后，选中色和坐标数码字必须在同一个 UI 事件事务内完成局部重绘；3 秒空闲计时自动取消选择也遵守同一规则。受影响数值单元格必须立即进入 LVGL invalidated rect/remote dirty rect，不能先变成黑块或空白后等待下一次 30Hz 状态帧、坐标值变化、页面重建或取消选择才恢复。仅颜色变化时必须保留原坐标文本和同一 owner 真值，不得依赖整屏刷新兜底。

## 当前限制

- `打开程序` 只进入程序页，不直接打开程序文件。
- `program_open_cc` command 当前不是主页面可见按钮的直接绑定动作。

## Broker 响应边界（非 Start/Pause）

- Start/Pause/Resume 不再经过 Broker/Python product action；不得保留旧的 Start 经 Broker 诊断规则作为启动路径要求。
- VPS/授权/下载、设置保存、设置驱动等仍归 Broker/后台 job 的长流程，UI direct Broker client 必须接受 Broker framed-response 上限内的有界结构化响应；响应缓冲区过小只能作为该长流程客户端实现缺陷处理，不能反向恢复 Start/Pause 的 Broker 中介。
