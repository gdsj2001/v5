### 2.1 当前状态和证据边界

本文只登记稳定owner和核对入口，不冻结“某日板端版本”“某次VM目录盘点”或二进制SHA。当前状态按以下四层重新取得：

| 层 | 当前核对入口 | 不能替代什么 |
| --- | --- | --- |
| Windows source identity | `linux/kernel/v5_linux_source_identity.json`、`linux/realtime/v5_realtime_source_identity.json`、`linuxcnc/v5_linuxcnc_source_identity.json`、`board/petalinux/v5_petalinux_source_identity.json`及当前FPGA owner identity | 不能证明VM投影、ARM产物或板端运行文件一致 |
| VM投影和构建产物 | `/mnt/v5-source`只读挂载、`/root/v5-build/temp_source/current`唯一投影、受影响target的ELF/ABI/hash/manifest | 不能证明已部署或服务已就绪 |
| 板端部署身份 | manifest回读、目标文件SHA、service PID/start_ticks、listener/权限/affinity和runtime readiness | 不能替代原始UI/operator/control结果 |
| 功能与运动结果 | 唯一owner fresh actual、当前画面/事件、模型匹配运动证据和最终ESTOP/Machine Off | 不能由旧截图、旧result、direct backend或版本号推断 |

发现VM或板端仍有唯一源码、硬件输入或不可重建集成输入时，先把它迁入对应Windows canonical owner并证明当前构建消费该owner，再删除旧副本；不得把VM盘点结论长期写在本文或反向覆盖Windows。

## 3. 控制链和状态链

控制链只能是：

```text
UI C -> native command gate UDS framed socket -> LinuxCNC/微内核/HAL/EtherCAT/native owner
```

状态链只能是：

```text
微内核/native actual -> native常驻内存/API/登记状态块 -> UI/gate/自动化按需直连
允许的显示数据 -> 唯一display producer -> typed SHM -> UI/relay只读消费
```

LinuxCNC/HAL/EtherCAT/native owner 已拥有或可拥有的控制语义、运动规则、状态、参数、坐标、工具、RTCP、WCS、G92、IO和安全联锁，V5上层只能做请求、显示、最小适配、诊断和证据，不得新建第二套真源。SHM只承载`REQ-SHM-DISPLAY-PROJECTION`当前白名单中的显示投影；字段、刷新率、freshness和ABI原子部署规则只由`功能/0-4共享内存.md`裁决，本文不复制字段表。

禁止把以下内容作为产品控制或状态真源：

- 已退役的产品控制FIFO/JSON IPC；登记的本地remote framebuffer/input IPC不在此列。
- `v3_status_snapshot.json` 热路径轮询。
- UI 直接读写磁盘参数。
- Broker 结果 JSON、`/run` 快照、诊断快照、UI dirty 值或默认值。
- 上层私有 JSON/cache 持久化微内核已有状态。
- 短生命周期 Python fallback 和产品热路径 popen/fork/exec。

## 6. 开发板访问和失联处理

常规访问顺序：

1. 测 `ping 192.168.1.221`、SSH、`18080` relay 和 `/remote/info`。
2. 若网络失联，使用 `board/tools/v5_board_power_cycle.py` 通过继电器重启开发板。
3. 重启后用 COM console 诊断 boot、网卡、服务和 LinuxCNC/backend 状态。
4. VM/board 操作前按 `AGENTS.md` 获取短时资源锁并实测进程、资源占用、板端忙和运动安全；锁不是状态真源，操作结束立即释放。

失联时不要直接重复全量 acceptance；先单点确认电源、boot、网络、relay、SSH、UI 服务和状态链。板端访问只用于检查、日志、runtime probe、安全恢复和部署 canonical artifact，禁止直接编辑产品文件。

## 7. VM 访问和只读挂载边界

本节只适用于已经按 Windows-first 规则证明 Windows 无法正确执行、确实需要 Linux/PetaLinux/Yocto/ARM/vendor 环境的最小步骤；不得把本节当成所有编译默认进入 VM 的依据。

首选 VM SSH alias 是 `z20-vm`。构建前必须在 VM 内完成以下事实核对：

```sh
findmnt -T /mnt/v5-source
test -r /mnt/v5-source/AGENTS.md
test -r /mnt/v5-source/board/CMakeLists.txt
test ! -e /root/Desktop/v5
```

边界：

- 不安装、不启用、不依赖 WSL。
- `/mnt/v5-source` 必须是 Windows 唯一 canonical 真源的实时只读共享挂载；大量小文件直接在 shared-folder 编译较慢时，使用固定唯一的 `/root/v5-build/temp_source/current` 本地投影。该投影可保留复用，但必须与 Windows identity 一致、禁止人工编辑或反向同步，VM 内不得存在第二套投影、checkout、rsync staging 或持久源码归档。
- VM 只在 `/root/v5-build/linux` 和 `/root/v5-build/board` 写 CMake/Ninja 状态、generated files、中间物和产物，不得在 source mount 内写 build 输出，也不得创建 per-task source/build tree。
- VM 可在 `VM_BUILD_ROOT` 的固定 current 位置保留唯一一套当前镜像集合，供普通更新、重写卡和证据复核；集合可包含 `BOOT.BIN`、`image.ub`、`boot.scr`、rootfs 和完整 SD 镜像，因为始终唯一而无需额外标注“副本”。同目录机器可读清单仍须记录 Windows source identity、build mode、生成时间、产物 SHA-256 和板端验证状态；新集合替换旧集合，不保留第二套版本/日期/任务镜像，来源身份与当前 Windows 真源不一致时不得作为当前产物部署。
- 构建前核对 Windows focused diff、挂载身份、唯一投影 identity、toolchain 和目标架构；identity 一致直接复用投影，不一致只刷新这一处。挂载不可用时不得仅凭旧投影继续构建。
- 板端可见或运行相关改动最终按受影响target/package、部署闭包所需artifact和manifest完成deploy，再走原始UI/operator/control路径；已有最小package/hot-deploy闭包足够时不生成rootfs或整张镜像，只有真实依赖或里程碑需要时才生成唯一current产物。

VM网络只可用于系统维护和用户明确要求的非构建诊断；source import、configure/compile/package/rootfs/image不得借代理补输入。代理不能写入项目、系统持久配置或构建环境，正式构建前必须清除全部大小写proxy变量并通过network-disabled gate；完整规则只看`功能/全局通用配置需求.md`。

## 8. 远程显示、输入和客户端更新

当前产品显示/输入链路：

```text
板端 LVGL UI
  -> /run/8ax_v5_product_ui/remote_framebuffer.bgra + remote_dirty
  -> board/services/ui/v5_remote_ui_relay.py / HTTPS/WSS :18080
  -> WinRemote证书固定 + challenge-response + scoped RAM session
  -> /run/8ax_v5_product_ui/remote_input FIFO local input
```

- UI进程不监听TCP，也不解析TLS/HTTP/WebSocket；独立relay负责网络协议。CIDR仅作第二层收口，不能替代TLS设备身份和应用认证。
- 18080只允许TLS 1.2+的HTTPS/WSS；WinRemote必须固定设备证书fingerprint和6位device ID，经一次性challenge-response取得短时、仅RAM、绑定peer和scope的session。明文、匿名、URL token、重放、过期、缺scope或证书错误全部fail-closed。
- relay/input ready由进程身份、本地mmap/FIFO/socket、TLS listener、认证后`/remote/info`和实际frame/input事件共同证明；端口连通或旧status JSON不能替代。
- 远程画面、CPU显示和系统诊断字段的generation/采样边界只由`功能/0-4共享内存.md`裁决；relay和WinRemote不得另建控制或状态真源。
- WinRemote正式更新使用独立ECDSA P-256分离签名身份，必须先验manifest原始字节再解析，下载后和安装前二次核size/hash，并用`release_sequence`防回退；正式EXE更新必须同次上传VPS并回读主备域三件套。私钥、relay secret和设备私钥互不复用。
- relay及其本地IPC、证书、凭据、端口、UID/GID、capability和CPU affinity必须按`REQ-SERVICE-RUNTIME-SANDBOX`逐项回读；root运行只能是尚未完成降权时的审计状态，不能当最终安全闭环。
- `Test-NetConnection 192.168.1.221 -Port 18080` 和 `/remote/info` 只证明 relay 可达，不替代 UI/operator 结果。
- AI 处理 UI 问题时统一使用 MCP 板端抓屏和鼠标输入，每次先抓屏确认页面、目标和遮挡，只做一个 click/drag，再抓屏确认变化；不得自动操作只供人工使用的 8ax WinRemote，不先跑整条自动化，不用裸坐标连点。

## 9. 参数、设置和驱动 profile

参数表、比例链和启动应用规则以`功能/0-1开机参数入内存.md`为owner。微内核/native参数进入对应常驻内存/API并由UI/provider直连readback；SHM不能作为参数、保存证明或动作gate。驱动比例的持久输入只由active runtime INI、当前model/mapping和profile owner裁决，HAL只消费最终启动值；设置页不得形成第二套运行比例真源。

`settings_runtime.json`只允许drive-only schema/证据，不得持久化WCS/G92/TLO/tool/RTCP/modal/status/runtime INI真值。驱动profile规则看`功能/1驱动命令映射表命令需求.md`和`功能/VPS登录与项目对接说明.md`：新增品牌优先改VPS/profile，不在UI、Command Gate或无owner的通用action增加厂家分支。

| --- | --- |
| 旋转等效角、count-domain、Rebase | `功能/3-3旋转轴等效角目标与解决方案.md` 和索引中的 rotary REQ |
| RTCP/G53 actual | `REQ-RTCP-G53-NATIVE-ACTUAL` owner；不能用上层私有字段冒充actual |
| WCS/G92/tool/TLO | LinuxCNC/native truth；UI和服务只能请求、显示、读回和诊断 |

缺 native gate、缺同 owner fresh native readback、缺单周期原子 proof 或缺板端 motion evidence 时，不能宣称 rotary/rebase/RTCP/WCS 相关行为已闭环。

## 11. 修改流程

默认顺序：

```text
AGENTS.md
  -> 功能/需求真源索引.md
  -> 对应 REQ owner 文档
  -> 当前canonical source + git status/focused diff + 精确文件锁
  -> 需求变化先改唯一owner；实现缺陷直接改canonical source/config/test
  -> Windows focused gate
  -> 仅在目标需要时校验VM只读挂载/唯一投影并构建受影响target/package
  -> 生成最小部署所需artifact/manifest并原子部署
  -> runtime readiness + 受影响原始operator/control路径 + fresh owner readback
  -> 运动任务按active model运行cc-ac.ngc或cc-bc.ngc并恢复ESTOP
```

不创建编辑前备份、第二worktree或任务源码副本；回退只使用根Git历史。`待做工作/`只在用户指定或当前切片已有活动输入时读取，不能插入每个任务的固定流程，也不能替代唯一owner。

规则冲突时，最新用户明确反馈优先于项目旧文档；先修冲突的owner文档，再修源码或配置。owner已经正确而实现未兑现时，不重复改写需求或把错误现状写回owner。

## 12. 验证分级

日常开发默认使用`incremental-current`：Windows本地门禁后只编译受影响target/package；部署闭包变化时才生成必要rootfs/manifest/current artifact，已有最小部署闭包足够时不做整张镜像。VM唯一投影和有效sstate/tmp按输入identity复用，不得无证据清理。空build/downloads/sstate的断网全量构建只用于用户明确要求、发布/灾难恢复里程碑或离线能力重新认证。

| 改动类型 | 最低证据 |
| --- | --- |
| 文档/规则 | `python board/tools/docs/verify_v5_document_routes.py --strict-details`、`git diff --check`、focused owner/index diff review |
| 本地代码 | touched-file compile/py_compile、focused smoke/contract/static check；不恢复通用自写 preflight/audit 门禁 |
| 板端程序/runtime/可见功能 | 本地门禁 -> 最小受影响target/package -> 必要artifact/manifest -> deploy/readiness -> 受影响原始operator/control路径 -> fresh readback |
| 运动相关 | 板端闭环 + fresh native actual + active-model匹配的`cc-ac.ngc`或`cc-bc.ngc`golden motion + 最终ESTOP/Machine Off |
| Git push | 只表示版本恢复点，不表示代码正确、板端已验或 release ready |

没有跑到对应证据时，最终状态只能写`source_only`、`local_verified_only`或`blocked`；板端代码的`local_verified_only`只是进行中的中间状态或明确例外，只有原始板端/operator/control路径达到owner验收条件才能写`board_verified`。

## 13. 快速判断表

| 看到的情况 | 判断 |
| --- | --- |
| V5自写业务owner接近500行且还要增加控制流 | 先按domain/protocol/runtime/verify拆分；tools/tests/config/ABI和上游源码不机械拆 |
| UI 想直接读磁盘配置显示参数 | 不允许；走对应 native/provider 常驻内存 readback，只有显示白名单可走 SHM |
| UI/普通服务想直接调用 LinuxCNC API | 不允许；控制走登记 native command gate，状态走对应 native owner/API |
| 产品热路径想读 HAL pin 或跑 `halcmd` | 不允许；必须注册 native helper 或受控 apply/restart 路径 |
| 新驱动需要支持 | 改 VPS/profile 和 `drive.*` 映射，不改板端厂家分支 |
| 板端网络断了 | 先 relay/SSH/18080 探测；再继电器重启和 COM 诊断 |
| AI UI自动化想直接跑全流程 | 先用项目登记的板端抓屏/鼠标入口确认并执行单个动作，单点闭合后再全量；不得自动操作WinRemote，direct backend或证据脚本不作为交互面 |
| 板端代码本地测试全绿但没部署/没走原始路径 | 只能`local_verified_only`，属于未关闭中间状态，不能称板端已修好 |
