## 3. 控制链和状态链

控制链只能是：

```text
UI C -> native command gate UDS framed socket -> LinuxCNC/微内核/HAL/EtherCAT/native owner
```

状态链只能是：

```text
微内核/native actual -> native 常驻内存/API/登记状态块 -> UI/gate/自动化按需直连
微内核/native 坐标与速度投影 -> State Publisher -> /dev/shm/v3_status_shm -> UI SHM reader
```

LinuxCNC/HAL/EtherCAT/native owner 已拥有或可拥有的控制语义、运动规则、状态、参数、坐标、工具、RTCP、WCS、G92、IO 和安全联锁，V5 上层只能做请求、显示、最小适配、诊断和证据，不得新建第二套真源。SHM 只允许 30Hz UI 显示投影：编码器反馈机械坐标、运动学理论坐标、转速、进给速度和主轴/进给倍率；其它 actual 必须直连 native owner。

禁止把以下内容作为产品控制或状态真源：

- 旧 FIFO/JSON 控制 IPC。
- `v3_status_snapshot.json` 热路径轮询。
- UI 直接读写磁盘参数。
- Broker 结果 JSON、`/run` 快照、诊断快照、UI dirty 值或默认值。
- V3 私有 JSON/cache 持久化微内核已有状态。
- 短生命周期 Python fallback 和产品热路径 popen/fork/exec。

| 层 | 当前 owner/入口 | 说明 |
| --- | --- | --- |
| Linux kernel/PREEMPT_RT | `linux/` | 完整源码、固定版本和身份文件 |

| Program runtime | `board/services/program_runtime/` | 原始 G-code identity、打开/加载和显示预览，不改写执行 G-code |
| State Publisher / SHM ABI | `board/services/state_publisher/`、`board/app/include/v5_status_shm.h` | 只发布 UI 白名单显示投影；其它 actual 直连 native owner |

## 6. 开发板访问和失联处理

常规访问顺序：

1. 测 `ping 192.168.1.221`、SSH、`18080` relay 和 `/remote/info`。
2. 若网络失联，使用 `board/tools/v5_board_power_cycle.py` 通过继电器重启开发板。
3. 重启后用 COM console 诊断 boot、网卡、服务和 LinuxCNC/backend 状态。
4. VM/board 操作前按 `AGENTS.md` 获取短时资源锁并实测进程、资源占用、板端忙和运动安全；锁不是状态真源，操作结束立即释放。

失联时不要直接重复全量 acceptance；先单点确认电源、boot、网络、relay、SSH、UI 服务和状态链。板端访问只用于检查、日志、runtime probe、安全恢复和部署 canonical artifact，禁止直接编辑产品文件。

## 7. VM 访问和只读挂载边界

本节只适用于已经按 Windows-first 规则证明 Windows 无法正确执行、确实需要 Linux/PetaLinux/Yocto/ARM/vendor 环境的最小步骤；不得把本节当成所有编译默认进入 VM 的依据。

首选 VM SSH alias 是 `z20-vm`。构建前必须在 VM 内完成以下事实核对：

```sh
findmnt -T /mnt/v5-source
test -r /mnt/v5-source/AGENTS.md
test -r /mnt/v5-source/board/CMakeLists.txt
test ! -e /root/Desktop/v5
```

边界：

- 不安装、不启用、不依赖 WSL。
- `/mnt/v5-source` 必须是 Windows 唯一 canonical 真源的实时只读共享挂载；大量小文件直接在 shared-folder 编译较慢时，使用固定唯一的 `/root/v5-build/temp_source/current` 本地投影。该投影可保留复用，但必须与 Windows identity 一致、禁止人工编辑或反向同步，VM 内不得存在第二套投影、checkout、rsync staging 或持久源码归档。
- VM 只在 `/root/v5-build/linux` 和 `/root/v5-build/board` 写 CMake/Ninja 状态、generated files、中间物和产物，不得在 source mount 内写 build 输出，也不得创建 per-task source/build tree。
- VM 可在 `VM_BUILD_ROOT` 的固定 current 位置保留唯一一套当前镜像集合，供普通更新、重写卡和证据复核；集合可包含 `BOOT.BIN`、`image.ub`、`boot.scr`、rootfs 和完整 SD 镜像，因为始终唯一而无需额外标注“副本”。同目录机器可读清单仍须记录 Windows source identity、build mode、生成时间、产物 SHA-256 和板端验证状态；新集合替换旧集合，不保留第二套版本/日期/任务镜像，来源身份与当前 Windows 真源不一致时不得作为当前产物部署。
- 构建前核对 Windows focused diff、挂载身份、唯一投影 identity、toolchain 和目标架构；identity 一致直接复用投影，不一致只刷新这一处。挂载不可用时不得仅凭旧投影继续构建。
- 板端可见或运动相关改动最终按受影响 target/package、必要 rootfs/manifest、一次 current 镜像完成 deploy，再走原始 UI/operator path；只有离线恢复认证才清空缓存全量重建。

VM 临时借用 Windows 代理只走以下链路：

```text
Windows 已运行代理（人工决定是否开放受信任专用网络监听）
  -> 运行时取得 Windows 当前地址和 mixed/HTTP/SOCKS 端口
  -> ssh z20-vm 的单条命令以前缀环境变量临时传入
  -> 仅执行明确的系统维护或非构建诊断命令
  -> 命令退出，代理变量随之消失
```

不得在 VM 安装第二套常驻代理客户端，也不得把代理写入 `/etc/environment`、profile、systemd、Git 或 BitBake 配置。host/port 是运行时外部事实，不能硬编码到 V5 源码和正式脚本。源码下载仍只在 Windows 完成；正式 BitBake/PetaLinux/LinuxCNC/kernel configure/compile/package/rootfs/image 前必须清除全部大小写代理变量并保持网络禁用。正在运行的构建与后来建立的 SSH 单命令代理相互隔离，禁止为配置代理中断当前构建。


当前产品显示/输入链路：

```text
板端 LVGL UI
  -> /run/8ax_v5_product_ui/remote_framebuffer.bgra + remote_dirty
  -> board/services/ui/v5_remote_ui_relay.py / configured bind:18080
  -> /run/8ax_v5_product_ui/remote_input FIFO local input
```


- UI 进程不监听 TCP，也不解析 HTTP/WebSocket；独立 relay 负责网络协议、bind 地址和 peer CIDR allow-list。
- relay/input ready 通过 `/remote/info`、本地 socket/进程 health 和实际 frame/input 事件证明，不使用周期性 status JSON 作为 steady-state actual。
- `Test-NetConnection 192.168.1.221 -Port 18080` 和 `/remote/info` 只证明 relay 可达，不替代 UI/operator 结果。
- AI 处理 UI 问题时统一使用 MCP 板端抓屏和鼠标输入，每次先抓屏确认页面、目标和遮挡，只做一个 click/drag，再抓屏确认变化；不得自动操作只供人工使用的 8ax WinRemote，不先跑整条自动化，不用裸坐标连点。

## 9. 参数、设置和驱动 profile

参数表规则以 `功能/0-1开机参数入内存.md` 为最高功能 owner。微内核/native 参数必须进入对应 native 常驻内存/API 并由 UI/provider 直连 readback；SHM 只承载坐标、轨迹、转速、进给速度和倍率显示白名单，不能作为参数、保存证明或动作 gate。保存请求中的诊断 epoch 如仍存在，只能在提交/构造瞬间从当前 provider 取得，不能成为普通动作准入门禁。

`settings_runtime.json` 只允许持久化 Settings Drive 私域和最小 drive/count-domain 证据，不得持久化微内核已有状态。驱动 profile 的规则看 `功能/1驱动命令映射表命令需求.md`：新增品牌只改 VPS/profile，不改板端业务代码；板端只用下载后的 public/private 缓存表，并按真实从站唯一选择。

## 10. 旋转、RTCP、WCS 和工具状态

| 主题 | owner/原则 |
| --- | --- |
| 旋转等效角、count-domain、Rebase | `功能/3-3旋转轴等效角目标与解决方案.md` 和索引中的 rotary REQ |
| RTCP/G53 actual | `REQ-RTCP-G53-NATIVE-ACTUAL` owner；不能用 V3 私有字段冒充 actual |
| WCS/G92/tool/TLO | LinuxCNC/native truth，V3 只能请求、显示、读回和诊断 |
| `current_tool_state.json` 等旧 V3 真源 | 作为迁移债务处理，不能继续扩展 |

缺 native gate、缺同 owner fresh native readback、缺单周期原子 proof 或缺板端 motion evidence 时，不能宣称 rotary/rebase/RTCP/WCS 相关行为已闭环。

## 11. 修改流程

默认顺序：

```text
AGENTS.md
  -> 功能/需求真源索引.md
  -> 对应 REQ owner 文档
  -> git status / git diff
  -> active 改进文档或 遗留.md
-> 不创建编辑前备份；只修改唯一 canonical 文件，回退使用根 Git 历史
  -> 本地针对性修改和测试
  -> 校验 VM 只读挂载与唯一投影 identity
  -> 受影响 target/package -> 必要 rootfs/manifest -> 一次 current 镜像
  -> 必要时 board deploy + 原始 UI/operator path + cc.ngc
```

规则冲突时，最新用户明确反馈优先于项目旧文档；先修冲突的 owner 文档，再修源码或配置。

## 12. 验证分级

日常开发默认使用增量 current 流程：Windows 本地门禁后，只编译受影响 target/package，部署闭包变化时才重做 rootfs/manifest，最后只生成一套 current 镜像。VM 的唯一源码投影和有效 sstate/tmp 可复用，不得无证据清理。空 build/downloads/sstate 的断网全量构建只用于明确的离线灾难恢复认证，不得作为每次功能修复的调试循环。

| 改动类型 | 最低证据 |
| --- | --- |
| 文档/规则 | `git diff --check`、人工 owner/index diff review |
| 本地代码 | touched-file compile/py_compile、focused smoke/contract/static check；不恢复通用自写 preflight/audit 门禁 |
| 板端可见功能 | 本地门禁 -> VM 挂载/唯一投影 identity -> 受影响 target/package -> 必要 rootfs/manifest -> 一次 current 镜像 -> deploy -> 原始 UI/operator path evidence |
| 运动相关 | 板端可见功能证据 + `nc/cc.ngc` golden motion loop |
| Git push | 只表示版本恢复点，不表示代码正确、板端已验或 release ready |

没有跑到对应证据时，最终状态只能写 `source_only`、`local_verified_only` 或 `blocked`；只有原始板端/operator 路径达到 owner 验收条件才能写 `board_verified`。

## 13. 快速判断表

| 看到的情况 | 判断 |
| --- | --- |
| 需要新业务逻辑但目标文件超过 500 行 | 新建 owner/sub-owner 或落到已有小 owner；大文件只做薄转发 |
| UI 想直接读磁盘配置显示参数 | 不允许；走对应 native/provider 常驻内存 readback，只有显示白名单可走 SHM |
| UI/普通服务想直接调用 LinuxCNC API | 不允许；控制走登记 native command gate，状态走对应 native owner/API |
| 产品热路径想读 HAL pin 或跑 `halcmd` | 不允许；必须注册 native helper 或受控 apply/restart 路径 |
| 新驱动需要支持 | 改 VPS/profile 和 `drive.*` 映射，不改板端厂家分支 |
| 板端网络断了 | 先 relay/SSH/18080 探测；再继电器重启和 COM 诊断 |
| AI UI 自动化想直接跑全流程 | 先用 MCP 板端抓屏确认并执行单个鼠标点击/功能，单点闭合后再全量；不得操作 WinRemote，direct backend 或证据脚本不作为交互面 |
| 有本地测试全绿但没上板 | 只能 `local_verified_only`，不能称板端已修好 |
