引用需求真源：`REQ-TOUCH-CALIBRATION-INPUT`。

<a id="touch-hardware"></a>
## 1. 当前硬件和驱动

- 实际出镜设备树是 `board/petalinux/project-spec/meta-user/recipes-bsp/device-tree/files/system-top.dts`。触摸节点位于 PS I2C1、地址 `0x38`，分辨率为 `1024×600`，compatible 为 `focaltech,ft5426`、`edt,edt-ft5406`，使用 `interrupts` 和 `reset-gpios`。
- Linux 实际绑定驱动是 `edt_ft5x06`。同一 I2C 总线上的 `goodix_ts@14` 当前为 `disabled`，不是备用链路或 fallback。
- udev owner 是 `board/petalinux/project-spec/meta-user/recipes-apps/v5-base-overlay/files/udev/99-touchscreen.rules`。它只对 `DRIVERS=="edt_ft5x06"` 的真实 `eventN` 建立 `/dev/input/by-path/z20-touchscreen`。
- `eventN` 由内核动态分配，产品代码、诊断程序和校准工具都不得写死 `event0`。

<a id="touch-runtime"></a>
## 2. 产品运行链路

- Windows canonical 校准配置是 `board/config/safe_ui/re_touch_calibration.json`，由 `board/config/deploy/v5_runtime_deploy_manifest.tsv` 安装为 `/opt/8ax/safe_ui/re_touch_calibration.json`。
- 当前配置格式固定为 `raw-evdev-cal-v2`、`screen-fit-topleft-1024x600-v1`、`topleft-v1`、显示尺寸 `[1024, 600]`；实际六个仿射系数以该 JSON 文件为准，不在本文复制第二份。
- 产品触摸 owner 是 `board/app/src/v5_lvgl_touch_input.c`。`v5_lvgl_touch_input_setup()` 读取 safe_ui 校准文件，打开 `/dev/input/by-path/z20-touchscreen`，解析 raw evdev 多点坐标并注册一个 LVGL pointer device。
- 运行期映射为：

```text
screen_x = raw_x * sx  + raw_y * sxy + ox
screen_y = raw_x * syx + raw_y * sy  + oy
```

- 当前运行期 loader 验证 `raw-evdev-cal-v2` 标记和 `sx/sxy/syx/sy/ox/oy` 六个系数，并把结果限制到 `1024×600`。profile、anchor 的严格验证由校准 apply 工具执行；不得把未实现的运行期检查写成已经存在。
- 退休路径 `/opt/8ax/ui/re_touch_calibration.json` 必须不存在。若该文件存在，产品触摸初始化会明确拒绝启动，而不是读取、迁移或 fallback。
- 物理 UI 启动时，校准读取、evdev 打开或 LVGL input 注册任一步失败，`v5_ui_shell_bootstrap` 都返回失败，不发布可用的 `ui_ready`。
- 可选诊断程序是 `board/services/ui/v5_touch_diagnostics.c`，日志为 `/run/8ax_v5_product_ui/touch_events.jsonl`。诊断日志只提供 raw/screen 证据，不是坐标 owner。

<a id="touch-calibration"></a>
## 3. 当前五点校准实现

- 源码只保留 `board/tools/v5_touch_calibration/` 下的 `v5_touch_calibration.py`、`v5_touch_core.py`、`v5_touch_evdev.py`、`v5_apply_touch_calibration.py`、`v5_run_touch_calibration.sh` 和 `v5_restart_after_touch_calibration.sh`；板端安装目录为 `/opt/8ax/tools/v5_touch_calibration/`。
- `v5_run_touch_calibration.sh` 停止产品 UI、远程显示 relay 和触摸诊断，使校准程序独占 `/dev/fb0` 与 `/dev/input/by-path/z20-touchscreen`。校准界面直接写 framebuffer，不依赖 Qt、GTK、X11、Wayland、鼠标或远程 pointer。
- 五个目标点依次为 `(80,80)`、`(944,80)`、`(944,520)`、`(80,520)`、`(512,300)`；每点等待真实手指最多 30 秒。
- `v5_touch_core.py` 用五点 raw 样本拟合二维仿射矩阵。最大拟合误差上限为 `36 px`，方向无关的边缘误差上限为 `48 px`，因此允许 X/Y 交换以及单轴或双轴反向，但拒绝奇异矩阵和超限结果。
- 只有采样和质量门禁全部通过，工具才原子写入 `/opt/8ax/safe_ui/re_touch_calibration.json`、回读源位置、派生 `/etc/pointercal` 并触发整板重启。超时、质量失败或异常不会提交新矩阵；提交后失败会恢复之前的校准文件。
- `/etc/pointercal` 只是派生兼容输出，不是产品 UI 的校准 owner。
- 当前启动方式是运行已部署的受控入口，然后在物理屏上用真实手指点击五个黄色目标点：

```powershell
ssh re-board "nohup /opt/8ax/tools/v5_touch_calibration/v5_run_touch_calibration.sh >/tmp/v5_touch_calibration.log 2>&1 &"
```

- 现场重校准产生的新矩阵必须回读并更新 Windows canonical 文件 `board/config/safe_ui/re_touch_calibration.json`，再由正常部署链路固化；板端运行文件不能反向成为源码 owner。

<a id="touch-acceptance"></a>
## 4. 验收

1. 本地运行 `board/tools/v5_touch_calibration/test_v5_touch_core.py`，正常轴、X/Y 交换和反向轴用例均通过。
2. 板端确认 `edt_ft5x06` 已绑定，`/dev/input/by-path/z20-touchscreen` 指向真实触摸 `eventN`，且退休校准路径不存在。
3. 用真实手指完成五点校准；回读 safe_ui 文件，确认 mode、profile、anchor、显示尺寸和六个系数与本轮结果一致。
4. 冷启动原始产品 UI，确认启动日志出现触摸注册成功；真实手指点击屏幕四角、中心、右侧按钮、顶部按钮和弹窗关闭按钮，均稳定命中。
5. 需要诊断时启用 marker `/run/8ax_v5_product_ui/enable_touch_diagnostics`，确认 `touch_events.jsonl` 记录同一轮真实 raw 坐标、校准后 screen 坐标和 safe_ui 校准路径。远程输入、模拟点击和手写日志不能替代该证据。
6. 将最终矩阵更新到 Windows canonical 配置后，重新部署并冷启动；clean SD 的 rootfs manifest 与写后回读必须包含同一份 `/opt/8ax/safe_ui/re_touch_calibration.json`，才能形成长期闭环。
