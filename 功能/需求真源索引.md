# 需求真源索引

本文解决 `功能/` 文档重复描述导致需求变化时多处修改的问题。原则是：一个需求只允许一个 owner 文档；非 owner 文档只引用 `REQ-*` 编号和描述本功能如何消费该需求，不再复制可变政策全文。

`功能/需求真源索引.md` 和本文登记的唯一 owner 文档是项目本地最高产品需求。产品、架构和指导 Markdown 全部位于 `功能/`；项目根目录只允许 `AGENTS.md` 作为自动加载执行入口。`AGENTS.md`、未登记的架构规则、`待做工作/`、测试、代码注释、旧聊天和旧实现只能作为流程、实现或记录输入；冲突时先把最新用户反馈落到本文或对应 owner，再同步其它文本和实现。

## AI 阅读入口

<!-- AI_FAST_READ_BEGIN -->
owner_reqs: [REQ-DOC-SINGLE-SOURCE, REQ-FEATURE-DOCS-SUPREMACY, REQ-FILE-CLASSIFIED-PLACEMENT, REQ-DELETE-FIRST-RETIREMENT, REQ-V5-GIT-RECOVERY-SNAPSHOT, REQ-UI-FIRST-FRAME-CACHE, REQ-UI-RUNTIME-REFRESH-RATE, REQ-G92-NATIVE-RUNTIME]
read_when: [需求路由, owner 查询, 文档冲突, 文件放置, 退役删除, UI 全局首帧/刷新, G92]
truth: [本索引唯一登记 REQ 与 owner -> owner 正文裁决 -> owner readback -> consumer]
forbidden: [同一 REQ 多 owner, 非 owner 复制易变规则, 模板卡冒充真实卡, 平行摘要或规则数据库]
readback: [唯一 REQ 行, owner 文件存在, owner 快读卡一致, detail_sections 可达]
impact: [全部功能 owner, AGENTS.md 路由, 系统架构 owner, 文档校验器]
acceptance: [REQ 唯一且 owner 可达；全部 owner 卡字段一致；严格文档路由校验通过]
detail_sections: [AI 快读卡格式, AI 读取深度与停止条件, AI 任务阅读包, 功能目录地图, P0 当前需求 owner 表, REQ-V5-GIT-RECOVERY-SNAPSHOT：V5 Git 范围, REQ-FILE-CLASSIFIED-PLACEMENT, P1 文档一致性]
<!-- AI_FAST_READ_END -->

AI 接手 `功能/` 文档时只按下面的最短路径读，不要先遍历全部 Markdown，也不要把多个文档的重复规则拼成第三套口径。

1. 用 `AI 任务阅读包` 或 `P0 找规则速查` 把用户问题归到一个 `REQ-*`。
2. 用 `P0 当前需求 owner 表` 确认唯一 owner，只读该文件的快读卡。
3. 仅当问题需要字段、公式、阈值、实现或验收细节时，继续读快读卡命中的正文标题；没有命中的章节不读。
4. 回答当前行为时再核对当前源码、配置或运行证据，并把“owner 要求”“当前实现”“已取得证据”分开，不得把目标设计写成已实现。
5. 最新用户反馈改变需求时，先改本文或对应 owner 正文并同步同文件快读卡，再改代码、测试、待办或引用文档。

### AI 快读卡格式

核心 owner 在正文前部使用以下固定边界。快读卡是同一 owner 内的导航层，不是独立需求副本；完整要求仍由该文件正文裁决。

```text
<!-- AI_FAST_READ_BEGIN -->
owner_reqs: [REQ-...]
read_when: [任务关键词]
truth: [唯一 owner/writer -> readback -> consumer]
forbidden: [禁止路径关键词 -> 正文锚点]
readback: [成功前必须取得的原生回读]
impact: [修改时必须一起核对的下游]
acceptance: [最低验收入口和状态]
detail_sections: [需要继续读取的正文标题]
<!-- AI_FAST_READ_END -->
```

使用规则：

1. 先读快读卡，再按 `detail_sections` 精确跳转；不命中的长正文不需要预读。
2. `truth` 只保留唯一 owner 链和决定分支；`forbidden` 只保留会选错实现路径的类别；`readback` 只列成功所需的原生事实；完整字段表、公式、错误码、命令和验收步骤留在正文。
3. `acceptance` 只指出最低证据层级和验收入口，不复制整套测试流程；`detail_sections` 必须是本文件真实存在、能直接定位的标题。
4. 卡片应保持一屏可判断：删除同义词、示例值和正文复述；不能安全压缩的数字或例外改为正文锚点，不得省略或改写。
5. 修改正文后必须在同一文件检查快读卡；发生冲突时正文优先，并立即修复卡片。
6. 禁止把所有快读卡汇总成另一份产品规则文档、JSON 数据库或生成后的第二真源。

### AI 读取深度与停止条件

| 用户问题 | 必读内容 | 何时停止 | 何时继续展开 |
| --- | --- | --- | --- |
| “规则是什么、谁负责、是否允许” | 本索引命中行 + 唯一 owner 快读卡 | `truth/forbidden/readback` 已直接裁决且无冲突 | 需要数字、字段、例外或精确措辞时读 `detail_sections` 命中章节 |
| “文档有没有矛盾、口径是否统一” | 同一 `REQ-*` owner 正文 + 涉嫌冲突的消费文档局部 | 已确认非 owner 只是短引用，或已列出具体冲突位置 | 同一易变规则出现两种值、两条 owner 链或两种验收状态时扩展到索引 owner 行 |
| “当前代码/设备实际怎样” | owner 快读卡 + 当前 canonical source/config/运行回读 | 已能分别陈述 owner 要求、当前实现和证据状态 | 代码路径、常量或 runtime actual 不明确时继续追到唯一 writer/readback |
| “修改需求或实现” | owner 快读卡 + 命中正文 + 当前 source 与 focused diff | 唯一 owner、修改面和最低 focused gate 已明确 | 涉及控制/状态 owner、ABI、生命周期、板端行为时再读架构或闭环 owner |
| “能否声明完成/通过” | 功能 owner `acceptance` + `功能/自动闭环测试方式.md` | 证据只支持的最高状态已经明确 | 命中板端代码修改时默认按 `REQ-BOARD-PROGRAM-FULL-BUILD-CLOSURE` 进入最小构建、部署和受影响 operator/readback；其它范围只有用户目标要求更高状态时再扩展 |

停止规则：找到唯一 `REQ-*`、唯一 owner 且当前问题已由快读卡直接裁决后，立即停止扩展阅读。不能因为“可能相关”预读其它 owner；只有字段/公式、实现核对、冲突定位或验收声明需要时，才增加一层正文或证据读取。

### AI 冲突裁决

| 冲突来源 | 裁决方式 |
| --- | --- |
| 两个 `功能/` 文档写了同一易变规则 | 以本文登记的 `REQ-*` owner 为准；非 owner 重复文字视为引用或待同步债务。 |
| `功能/` owner 与 AGENTS、架构、待办、测试、旧聊天、代码注释冲突 | 以本文和对应 owner 为产品真源；流程文件只提供工作顺序和门禁。 |
| 最新用户反馈与现有 owner 冲突 | 最新反馈是需求变更输入；先落到本文或 owner，再实施。 |
| 旧路径、fallback、兼容、临时、默认值、兜底、镜像路径、AI 自造分支仍存在 | 当前 owner 没明确要求就先物理删除；若仍被真实路径依赖，先迁移到 canonical owner，再同 slice 删除旧路。 |

### 文档类型

| 类型 | 允许内容 | 不允许内容 |
| --- | --- | --- |
| 本文索引 | `REQ-*` 路由、owner、冲突裁决、目录地图、跨功能快变规则 | 具体实现日志、命令输出、板端证据、进度流水 |
| owner 文档 | settled requirement、owner 边界、字段/按钮/协议映射、验收条件；全部位于 `功能/` | 任务进度、旧聊天摘录、重复其它 owner 的可变全文 |
| 非 owner 功能文档 | 本页面/本功能的局部实现细节、消费的 `REQ-*`、本地验收边界 | 自行改写参数/native/SHM/运动/安全等跨功能规则 |
| `待做工作/` | 活动任务输入、未闭合工作卡、人工可读问题描述 | 代替 `功能/` 成为最终产品真源 |

HMI/UI 定位为“人眼视觉辅助与状态保真（数据不欺骗）”。电机实际运动精度和安全是绝对的 P0，UI 刷新率按 `REQ-UI-RUNTIME-REFRESH-RATE` 分层：坐标/轨迹动态层 30Hz，普通按钮轮询 10Hz，急停按键显示/状态刷新 10Hz，其它普通 UI 状态和诊断显示 5Hz；运行期禁止周期性整屏刷新，必须按 dirty 对象、dirty 单元格、dirty 区域或 LVGL invalidated rect 分块按需刷新，避免过度消耗 CPU 影响实时性。文档中 `5px` 投影误差或 `1.0mm` 空间误差仅用于开发阶段的公式正确性校验，不作为运行时重绘开销约束。
