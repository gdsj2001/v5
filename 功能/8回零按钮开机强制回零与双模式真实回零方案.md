## 1. 边界

主页面按钮语义：

| 入口 | 目标 | 是否移动机器 | 是否改变 homed | 能否替代开机回零 |
| --- | --- | --- | --- | --- |
| `回零` | BUS：回到设置页 `设0` 的 count-domain 零点证据对应位置；Pulse：LinuxCNC/HAL home | 是；按下后必须执行回零流程，不用容差判断跳过 | 是，最终通过 LinuxCNC/native 标记 | 是，唯一入口 |
| `归零` / Work Zero | 把选中轴当前机械坐标写入选中 WCS offset，使当前加工坐标为 0 | 否 | 否 | 否 |
| 设置页 BUS `设0` | 记录 BUS 回零目标 count-domain 证据 | 否 | 否 | 否 |
| 设置页 Pulse `HOME_OFFSET` | 写 homing 参数 | 否，apply/restart 后回零才移动 | 否 | 否 |
| `回加工0` / `回机械0` | 已回零后的定位运动 | 是 | 否 | 否 |

禁止抄近路：

- 不修改 G-code 绕过失败。
- `回零` 不调用 Work Zero；Work Zero 不替代机器 Home。
- `归零` 不允许运动。
- UI 坐标、旧诊断、旧 homed 文件或旧结果不能让 `回零` 提前成功。
- 声称通过必须有自动触发、日志/native 回读/截图或测试证据。

## 2. 回零模式

主页面普通 `回零` 不再经 Broker/Python/UI 层的任何位移规划或 jog 适配中介。C UI 始终通过登记的统一 native/linuxcncrsh gate 下发 LinuxCNC/微内核原生 Home 命令（例如 `Set Home -1` 或全轴回零指令）。

**回零的实际操作完全由微内核根据当前的“脉冲/总线”模式在底层直接分支并控制执行，UI 层与 Broker 绝不参与轨迹计算与多轴时序控制：**

| 模式 | 微内核执行语义 | 成功边界 |
| --- | --- | --- |
| **Pulse/StepDir** | 微内核控制 HAL 层的原生 Home 流程：根据 INI 中配置的 `HOME_SEQUENCE` 次序进行排队，控制各轴向 `HOME_SEARCH_VEL` 方向运动寻找限位/home 开关，触碰后触发 latch 减速，最终依据 `HOME_OFFSET` 写入绝对坐标系基准。 | LinuxCNC/HAL 物理 homed 标志置位，UI 通过直连微内核内存/API 回读确认。 |
| **BUS/EtherCAT** | 微内核/native gate 必须先识别当前实际 BUS 模式，再读取 native/编码器同源 MCS 位置块，按当前 MCS 到设置页 `设0` 所定义 count-domain 零点的差值生成一次受控机器零点运动；当前 v5 板端 BUS 运行坐标把 count-domain 零点映射为 native MCS 0，因此 gate 用 native MCS 计算 `G91` 增量回零并恢复绝对模式。`Set Home -1` 只可作为 homed 状态同步，不得替代实际运动；后续专用 EtherCAT home owner 存在时可由同一 native gate 内部替换执行器，但 UI/Broker 不参与轨迹计算、delta 算路或分步 jog。 | 伺服驱动器状态字 OP/到位证明或 LinuxCNC/native 机器坐标到位证明已确认 + 微内核内部 `homed/all_homed` 状态置位，UI 直连回读确认。 |

### 核心硬边界：
1. **彻底取消上层 jog 模拟回零**：物理回零动作必须由微内核在 native 层通过实时周期（realtime cycle）或运动控制器闭环控制，**绝对禁止**在 UI C 代码、Python 脚本或 Broker 中通过计算 delta 距离后用 `jog` 指令模拟回零动作。
2. **微内核统一入参与决策**：不管是 Pulse 还是 BUS 模式，回零运动的最高速度、加速度及超时限制，必须由微内核直接读取 INI 配置或 native 参数执行。UI 及 Broker 发送的 `Home` 指令仅作为触发信号，不携带任何由 UI 算出的速度或目标位移参数。
3. **回零状态单一源回读**：回零的执行进度与最终成功标志（如 `homed/all_homed`）只允许从微内核/native 内存中直连回读确认，严禁依赖任何外部 `/run` 临时 JSON 文件或共享内存（SHM）的第三方状态写入。

## 3. 模式判定

模式判定 owner 不是 UI 下拉框或设置页文字。

BUS/Pulse 模式识别、模式切换闭环和 Pulse 设零/回零总口径以 `REQ-BUS-PULSE-MODE-DIFFERENCE` / `功能/总线脉冲区别.md` 为准；本节只定义主页面 `回零` 如何按 active mode 分支执行。

规则：

- `ethercat` / `bus` 归一为 BUS。
- `pulse` / `stepdir` / `step_dir` / `step/dir` 归一为 Pulse。
- 请求显式指定模式时，必须与 active mode 一致；不一致 fail-closed。
- 缺 active mode、native active mode 尚未确认 或运行模式不可确认时，回零不得运动。
- 旧 `run_mode_bus_pulse` 字段退役，不能作为 UI dirty、显示入口、兼容字段或当前模式真源。

## 4. 执行入口

`回零` 可能持续数十秒，但 UI 不再把普通 Home 投递到 Broker `home_precheck` async job。当前普通 Home 入口只负责直通 native gate；执行进度、最终 homed/all_homed 和失败原因必须由后续 native/microkernel/HAL/EtherCAT 回读和 UI 状态刷新呈现。

执行方式：

```text
C UI Home
  -> native/linuxcncrsh gate
  -> LinuxCNC/HAL/EtherCAT/native Home owner
  -> native/microkernel/HAL/EtherCAT readback
  -> UI 状态刷新显示 homed/all_homed 或失败原因
```

要求：

- BUS 模式下，`Set Home -1` 只用于同步 native/LinuxCNC homed 状态，不能单独算作回零运动完成；同一 Home gate 必须读取 native MCS 并继续触发同源计算出的机器零点运动。
- UI 坐标显示必须实时刷新，不等 Home 完成后才刷新。
- E-stop、Jog stop 等紧急通道不得被 Home 请求或状态刷新阻塞。
- 旧 `home_precheck` async job、`job_status` Home 轮询和 Python product action 结果不得恢复为普通 Home 执行路径。
- 命令已发送只用于触发证据；最终状态以 LinuxCNC/HAL/native 回读为准，SHM 只可作为 UI 坐标/轨迹/模态/速度显示证据。

## 5. 状态 freshness

`回零` 随时可按，不等于可以携带空状态证据发运动。

UI 不得把 SHM `status_epoch`、坐标显示投影、旧状态帧或本地缓存作为 Home 运动准入证明；请求里也不得携带 `status_epoch` 作为 native proof。刚重启、State Publisher 尚未写入或 SHM reader 不可用时，UI 只能影响坐标/模态/速度显示和提示，不能替代 Home product action 对 fresh native/provider 状态的读取。

Product action 可短窗口重试 fresh native/microkernel 状态；最终读不到 fresh native/provider 状态、active mode、homed/all_homed、MCS 或 owner 要求的 BUS/Pulse 证明时 fail-closed，不触发任何 Home 运动。

Home 的 busy/idle 判定必须交给 fresh microkernel/native Home gate，而不是只看裸 `exec_state`，也不得在 V3/Broker 里重建速度容差门禁。急停复位后若微内核/native 判定可进入 Home，即使上层仍看到 `exec_state=EXEC_WAITING_FOR_MOTION_AND_IO` 这类残留执行意图，也不得由 V3 返回 `HOME_PRECONDITION_BUSY`。如果队列、暂停、活动 MDI、行号、运行文件或非 idle 解释器状态确实会阻止 Home，必须由微内核/native gate 返回拒绝原因；上层只显示该结果，不用 SHM、JSON、旧快照或私有阈值自行判定。

## 6. 未回零与其他动作

`回零` 不作为 Start/Jog/Auto 的产品层业务门禁。

- Start/Jog/Auto 只按自身安全、idle、enabled、fresh native 状态、授权、runtime settings 和 LinuxCNC 执行结果校验。
- 不得因为 `all_homed=false`、当前位置离开 MCS 0 或旧 home 失败诊断，在产品层直接拒绝 Start/Jog/Auto。
- LinuxCNC 或 motion gate 因实际状态拒绝时，应返回真实失败原因。

## 7. 归零 / Work Zero

`归零` 是 Work Zero：

- 只能在机器 idle、WCS 和轴选择明确、状态 fresh、RTCP OFF 时执行。
- 把选中轴当前机械坐标写入选中 WCS offset，使当前加工坐标为 0。
- 不运动、不改变 homed、不替代回零。
- 失败码、G10/gate、native 回读和关闭条件以 `功能/2加工坐标系偏移值保存与读取实现.md` 为准。

## 8. 旋转轴等效回零

如果用户选中 MCS A/C 单轴，主页面 `回零` 执行旋转轴等效回零：

- UI 发送强类型 `rotary_equiv_zero`。
- 按当前可信 raw 运动到最近等效 raw 零点。
- 不清驱动多圈、不做 position preset、不写磁盘零偏。
- 缺 fresh native 状态、可信 raw、机器 enabled/idle、软限位或 native gate 证据时 fail-closed。

未选中 MCS A/C 单轴时，主页面 `回零` 执行全轴机器 Home，不能被 WCS 坐标点选改写成回加工 0。

## 9. 失败提示

失败弹窗不得只显示“回零失败”或泛化 guarded status。

UI 必须走全局命令失败弹窗格式化入口，用 `code -> 本地化原因` 对照表显示当前语言原因。Broker/detail JSON 中的 `message_cn`、`message`、`reason`、`error` 作为详情兜底。

至少能区分：

- 急停。
- 未上使能。
- 机器不空闲。
- 状态不可用。
- active mode 不可确认。
- 设0目标缺失。
- 驱动读回失败。
- 到零点距离超过安全上限。
- homed 标记未确认。

## 11. 验收

本地必须覆盖：

- 普通 Home 不再投递 `home_precheck` async job，也不依赖 UI `job_status` 轮询证明执行。
- E-stop/Jog stop 可抢占。
- fresh native/provider 状态缺失时不触发运动；SHM `status_epoch` 不得作为 Home 准入证明。
- BUS 缺 zero evidence、active `SCALE`、驱动 actual position、profile/readback 时 fail-closed。
- Pulse 走 LinuxCNC/HAL homed/all_homed。
- `回零` 不改变 active WCS。
- `归零` 不调用 Home。
- 失败码可本地化显示。

板端声明通过必须有：

- 原始 UI/operator 触发路径。
- UDS request/response 和 job_id/job_status。
- LinuxCNC/HAL 或 BUS drive/readback 证据。
- LinuxCNC/HAL/native `homed/all_homed` 和 MCS 回读；SHM 只可补充 UI 坐标显示截图。
- UI 截图或 remote frame。
- 运动相关闭环按 `AGENTS.md` 跑 `nc/cc.ngc` 黄金运动测试。
- 最终安全停机状态。

## 12. 禁止事项

- 恢复 FIFO、临时 response JSON、旧 native daemon、旧 `v3_linuxcnc_*.py` 裸脚本或短生命周期解释器 fallback。
- 用 Work Zero、G92、设置页设0、rebase、驱动 preset 或 WCS 运动替代机器回零。
- 旧诊断文件、UI 缓存或私有文件恢复已回零。
- Home job 阻塞 UI 刷新、远程输入或紧急通道。
- 未拿到 LinuxCNC/HAL/native 回读就显示“已回零”。

## 13. 关闭条件

关闭回零/归零问题时必须说明：

- 按钮语义：回零、归零、旋转轴等效回零或定位运动。
- active mode 和分支：BUS 还是 Pulse。
- 请求、job、LinuxCNC/HAL/native 读回和 UI 证据；SHM 只记录显示刷新证据。
- 本地测试、板端截图、运动闭环和最终安全态。
- 未闭合事项只写真实缺口，不写过程日志。
