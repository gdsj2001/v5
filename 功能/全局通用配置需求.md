## 目标

本文是 V5 的全局统一需求与通用配置边界 owner。它只规定所有相关页面、服务、动作和配置都必须共同遵守的基线：规则放置、唯一 owner、生命周期、回读、统一 UI 交互和退役方式。

本文不是参数表、运行状态表或“总开关”文件，也不替代 motion、安全、WCS、RTCP、驱动、VPS、单个按钮或单个页面的专项 owner。专项需求的字段、业务语义和验收仍只由各自 `REQ-*` owner 裁决。

<a id="global-classification"></a>
## 1. 适用范围与归类

### 1.1 必须写入本文的全局统一需求

满足以下任一条件的需求，先更新本文，再更新受影响的专项 owner 和实现：

- 每个相关页面、弹窗、服务或动作都必须以同一方式执行的规则；
- 跨页面、跨进程或跨服务共享的配置分类、唯一 writer、加载、生效、回读和迁移规则；
- 用来防止同一能力被每个页面或服务各造一套的公共交互、状态或配置边界；
- 影响所有消费者的 schema、generation、受控 reload、兼容或退役规则。

### 1.2 必须留在专项文档的零散需求

以下内容不得塞进本文：

- 单一按钮、单一页面、单个业务动作、单个错误码、单个驱动 profile 或单个协议字段的具体行为；
- motion、安全、WCS/G92、RTCP、tool/modal、旋转轴、驱动参数、授权、构建配方等已有唯一 native/功能 owner 的真值和字段；
- 某次实现的文件名、命令输出、任务进度、板端证据或临时排查记录。

专项文档必须只描述“该功能如何消费全局规则”和自身差异化业务语义；不得复制或改写全局规则正文，更不得在源码、页面或服务中另建一套私有规则。

## 3. 通用配置生命周期

### 3.1 字段登记与唯一 owner

每个属于全局通用配置的字段或策略，必须至少能定位以下信息：逻辑 key、类型和 schema 版本、适用范围、唯一持久 owner、唯一运行 owner、唯一 writer、消费者、hot/cold 生效方式、generation 和实际 readback。

字段缺少其中任一项时，不能由 UI 或服务补一个默认值继续运行。必须由字段 owner 明确 fail-closed、完成受控迁移，或删除不再支持的消费者；禁止留下 fallback、镜像字段、别名或双写路径。

### 3.2 加载、生效与读取

1. 持久配置只在开机或 canonical controlled reload 由唯一 owner 读入并生成 resident snapshot；消费者读取同一代 snapshot 或 owner API。
2. hot 字段必须被 owner 明确登记，变更使用“请求 → 校验 → 唯一 writer 落盘/更新 → owner readback → generation 发布 → 消费者按代切换”的单一事务；任何阶段失败都不得半生效。
3. cold 字段只能在明确的 clean restart 或 controlled reload 后生效；UI 必须显示 owner 回读的实际生效状态，不能把点击、请求值或写盘成功冒充已生效。
4. 每个 consumer 必须以完整 snapshot/generation 切换，不得把旧字段与新字段拼成混合状态；运行热路径禁止轮询配置文件、目录或临时结果文件。

### 3.3 UI 与服务的共同边界

- UI 只提交请求并显示 owner readback；页面缓存、弹窗对象、顶部状态和 SHM 显示投影均不是配置或控制真源。
- 常驻服务只能消费已登记的 resident 配置或 owner API；不得各自读取不同文件、复制配置、私藏环境变量开关或在失败时退回旧配置。
- 项目路径使用项目相对地址；凭据、私钥、DNA 原文和其他敏感信息不写入普通全局配置、UI、日志或 SHM，按其专项安全/授权 owner 管理。

<a id="product-runtime-closure"></a>
## 4. 产品运行闭包

### 4.1 单一产品闭包入口

- `board/config/deploy/v5_runtime_deploy_manifest.tsv` 是板端 V5 自写运行闭包的唯一部署入口。产品二进制、Python/Shell 入口与模块、启动脚本、配置和资源必须逐项登记；同一运行文件不得由制卡脚本、部署脚本和验收脚本分别维护互相漂移的目标清单。
- CMake 必须提供一个由部署清单中的全部 `binary` 行派生的聚合产品目标。制卡和板端验收只构建该聚合目标，不得在多个脚本再次手写一组二进制名称。
- 每个自写文件必须归类为产品 C/C++ 源、产品脚本/模块/配置、测试/smoke、构建/诊断工具、上游/生成文件或非运行资料。生产目录中的未分类文件必须 fail-closed；核对确认没有运行、测试或工具消费者的死代码必须当场物理删除，不得改名、禁用或保留为备用路径；也不得因为“文件在源码树里”就假定已经编译或部署。

<a id="build-input-closure"></a>
### 4.2 编译、部署和写后回读

Windows-first 总原则：任何 compile、syntax/static check、codegen、package、manifest/hash、schema、unit test 或其它构建步骤，只要 Windows 能以正确工具链和目标语义执行，就必须直接在 Windows 运行。只有 Linux ABI/kernel 行为、PetaLinux/Yocto、Linux-only dependency、ARM cross-runtime 或 vendor tool 等在 Windows 无法正确提供时，才把不可再拆的最小步骤交给 VM；“习惯用 VM”“VM 已开机”“VM 有旧缓存”或“Windows 需要配置工具”都不是理由。混合流程必须把能留在 Windows 的阶段全部留在 Windows，并在调用 VM 前记录具体缺失能力。

<a id="windows-host-gate"></a>
#### 4.2.1 Windows 默认构建与统一 host gate

Windows CMake 目标必须按运行环境在目标定义处分类：`portable_host` 可以在 Windows/Linux host 定义、构建和测试；依赖 POSIX SHM、UDS、`/proc`、`linux/input.h`、LinuxCNC native 或其它 Linux ABI 的 `linux_runtime` 目标在 Windows 不得定义；部署 manifest 的 `arm_product` 聚合目标只在 Linux/ARM 交叉环境定义。不得仅在 `add_test()` 阶段排除 Linux 测试，却继续让对应 `add_library()`、`add_executable()` 进入 Windows 默认 `ALL` 构建；混合实现需要 host smoke 时，只拆出唯一可移植核心并由 Linux 产品目标重新组合，禁止 Windows 假 transport、fallback 或测试影子业务实现。

`board/CMakePresets.json` 是 Windows 空目录 CMake 参数 owner，至少提供 `windows-ci` configure/build/test preset；`board/tools/ci/run_v5_host_gate.ps1` 是本地提交前和 GitHub CI 的唯一主机门禁入口。该脚本从项目根依次执行 diff hygiene、严格文档路由、空 Windows preset configure/default build、`v5_local_regression`/CTest、登记的 Python/Windows 客户端 focused tests与Factory/Dealer构建，逐阶段输出命令、耗时和退出码，任一失败立即以非零退出。测试目标和命令只在该脚本及其直接调用的 canonical 项目入口登记；GitHub workflow 只准备受信环境并调用该脚本，不复制清单。

host gate 不得启动 VM、PetaLinux、kernel、rootfs/image、板端部署/运动、VPS上传、正式客户端发布或读取设备/签名私钥。GitHub workflow 初始只允许 `push` 到 `main` 和人工触发，权限固定为 `contents: read`，有超时与同提交并发取消；不得上传正式 EXE、OTA 或板端产物。唯一 Git staging/commit/push 入口必须在普通 staging 前调用 host gate并对失败 fail-closed；GitHub CI随后对同一 commit在干净 Windows runner复核。门禁脚本自己的 focused smoke 可以用显式测试参数隔离外部客户端阶段，但普通本地提交和CI不得静默降级或跳过失败。

默认构建模式是 `incremental-current`，不是空缓存全量认证。日常功能修复必须使用同一 canonical 构建主线的 fail-fast 梯级：先在 Windows 完成受影响 owner 的文档、diff、语法/静态/单元/身份门禁；需要 Linux/ARM/PetaLinux 目标语义时，再在 VM 只执行最小受影响 recipe/target 的 configure/compile/install/package。真正不进入板端且不依赖板端协议的代码到这里可按实际证据交付 `local_verified_only`；任何会进入板端产物或影响板端运行的修改必须继续按部署闭包变化生成必要 package/manifest/rootfs/current artifact，原子部署后确认 runtime readiness，并走受影响原始 operator/control path 做最小测试和 fresh owner readback。只按直接依赖决定是否生成 rootfs/image，不得把最小上板扩大为无关全量镜像、完整 Golden Motion 或发布认证。任一级失败只修复并重跑该级及其下游，已通过且输入 identity 未变的上游级不得重跑。文档中的 `full-current` 只表示最终产物包含当前 Windows 全部有效源码和配置，不表示清空缓存、重编无关目标或执行灾难恢复认证。

VM 的构建并发必须由 Windows canonical PetaLinux/recipe 配置裁决，禁止只在 VM 临时改生成配置。当前 16 vCPU、约 7.6 GiB RAM 的支持环境采用 `BB_NUMBER_THREADS=4`、全局 `PARALLEL_MAKE=-j4`；单一关键路径 `linuxcnc-prebuilt` 可用 recipe 固定为 `-j8`，不得直接放大到 16×16。每次提高并发后必须从日志或进程命令行读回实际值，并在 compile/package/rootfs 阶段持续检查 available RAM、swap 和 OOM；内存压力出现时只降低当前 canonical 并发配置并复用现有 sstate/tmp/work，不得靠清缓存或 VM 联网规避。构建入口必须始终由同一非 root build user 运行 BitBake；若历史 root 构建在 `linuxcnc-prebuilt` 当前版本的派生 `image` 目录留下 owner 漂移，只允许在严格校验绝对路径仍位于该 recipe work 目录后删除这一个派生 `image`，不得 chown/删除其它 work、tmp、sstate 或 downloads。

项目推进和当前可交付结果是构建流程的目的，规则与门禁只是受影响闭环的证明手段。每个 gate 必须能说明失败会怎样改变当前 owner、target、产物或安全结果；不能证明直接影响的步骤不得进入当前切片。LinuxCNC-only 修改不得因为共用脚本而扫描或验证未变化的 Linux kernel、PREEMPT_RT、完整 PetaLinux overlay/rootfs；kernel-only、UI-only 和文档-only 切片同理。组合入口若强制运行无关子系统，必须提供 target-only 短路或调用已登记的更低层入口，并保留必要缓存与既有有效证据；不得把错误的大入口称为“合规”后反复执行。需求 owner 已经准确而源码未兑现时，只读取 owner 作为 contract 并直接修实现，不为制造流程痕迹重复改文档。

日常 focused 修复采用轻量校验：只检查改动文件、当前 owner 的文件级差量，以及当前 target 在依赖解析、编译和安装时实际读取的直接输入；不得在进入单包前运行完整 source-package inventory、全仓 hash、全部 Linux/kernel/PREEMPT_RT/PetaLinux owner identity、全许可证闭包或离线恢复认证。仓库中与当前 target 无关的源码缺项不阻塞日常推进，统一推迟到 full-current、发布或离线全量构建时扫描，并按“机器可读缺项报告 -> Windows importer 补入 canonical owner -> 只重跑失败层”闭环补齐。若当前 focused target 自己实际请求的输入缺失，只报告该直接缺项并停止当前层，不扩大成全仓检查。

日常切片以实现速度和最小受影响闭环为默认目标：定位 owner 与直接源码后立即编辑并运行最近 focused compile/unit/package；真正非板端代码通过即可交付 `local_verified_only`。任何进入板端产物或影响板端运行的修改必须立即继续使用已有最小 package/artifact、部署和 operator/control 入口完成上板测试与 fresh readback，`local_verified_only` 只能记录进行中的中间状态、用户明确仅本地/保留最终人工测试的边界或真实 blocker。不得为小修自动追加无关 rootfs/image、全量 operator 回归，也不得临时扩建 bundle/包管理器/installer/协议；缺少必要增量部署能力时报告未闭合条件，不得把能力缺口包装成任务完成。

日常快速修复必须拆成可独立关闭的最小切片：默认一个切片只处理一个具体缺陷、一个 owner 链和一个最小可部署 target；用户一次提出多个问题时按依赖顺序逐个关闭并逐个交付，除非源码证明它们属于同一个不可拆原子变更。进入 VM 前必须冻结本切片的缺陷、owner、允许修改的文件集和最小构建 target；进入 VM 后只允许为当前明确失败层修改直接相关的 owner/source/config/test/build tool，新发现的独立问题进入下一切片，不得继续扩成架构整理、顺手重构或更多文档清扫。受影响单包未通过前不得进入 rootfs/image；每个切片完成后立即报告结果并释放其文件锁和 VM/board 资源。

耗时门禁必须绑定“受影响输入 identity + gate 实现 identity + 命令参数”。三者未变时，同一切片内已通过的 source identity/hash、唯一投影、文档路由、runtime policy 和上游构建层直接复用，不得为了等待感或重复确认再次全树扫描；任一项变化时只重跑被影响的门禁和其下游。任务每次跨阶段必须立即更新可见计划。活跃命令连续 120 秒没有新输出时必须读取 PID/CPU/子进程/日志和资源锁：进程仍活跃就报告真实阶段后继续等待，进程已经退出、失去子进程或锁与进程不一致就立即收集失败证据、停止等待并释放锁，不得以“仍在运行”长期占用阶段和资源。

日常模式必须保留有效 `sstate`、`tmp/work`、任务 stamp 和已通过的无关产物；没有指向具体损坏缓存、签名错误或输入 identity 变化的证据时，禁止 `cleanall`、`cleansstate`、kernel clean、删除整个 `tmp/sstate`、重置全部 fetch stamp 或强制重编无关 recipe。LinuxCNC 变化先过 LinuxCNC 单包，内核变化先过内核单包，UI/runtime 变化先过对应产品目标；不得用最终镜像构建充当首轮编译测试。

Windows 是唯一 canonical 源码真值，但 VMware shared-folder 对内核、LinuxCNC、Yocto 这类大量小文件的直接编译可能显著慢于 VM 本地 ext4。VM 允许固定保留唯一源码投影 `${VM_BUILD_ROOT}/temp_source/current`：首次由只读 Windows 挂载按 identity 生成，以后构建先比较 identity，一致就直接复用，不一致只刷新这一处，再在 VM 本地完成“单包 -> rootfs -> current 镜像”。该目录可以跨任务保留，不要求构建后删除，但全 VM 只能有这一套 V5 源码投影；不得再创建 run-id、日期、AI、任务或备用投影。投影不得人工编辑、补丁、提交、作为下载落点或反向覆盖 Windows，所有生成物必须写入独立 build 目录。它是可重建的加速投影，不是第二 canonical owner。

`identity` 变化只表示至少一个输入发生变化，不授权删除并重建整棵投影。日常刷新必须根据已登记文件 identity/内容 hash 计算文件级新增、修改、删除集合，在唯一 `current` 投影内原地、原子地只更新该集合；内容和类型未变的文件必须保持原 inode/mtime，不得用整树 `rmtree`、全量 `checkout-index --force`、全量复制或重新挂载制造虚假过期。Linux/PREEMPT_RT、LinuxCNC 和 PetaLinux 的持久 build 目录必须与投影及任务临时目录分离；设备树、打包或其它 recipe 任务不得清除未受影响的 kernel/LinuxCNC 对象、stamp 或 work。只有投影缺失、机器可读状态损坏且严格校验失败，或用户明确触发空缓存离线认证时，才允许重建整棵投影。小改动的验收必须证明未变文件 mtime 保持、实际重新编译对象集合命中受影响闭包，并且没有无关 kernel/LinuxCNC 全量编译。

LinuxCNC 2.9.7 上游明确拒绝普通 out-of-tree configure，因此合规入口不得把 recipe 的 `B` 指回只读 projection/source，也不得在 source 中执行 `distclean`、`autogen.sh`、`configure` 或 `make`。只允许在固定 canonical LinuxCNC build 目录建立短生命周期 copy-on-write build view：lowerdir 是唯一只读 `${VM_BUILD_ROOT}/temp_source/current/linuxcnc`，upper/work/merged 都位于同一固定外部 build 目录；BitBake `S` 继续指只读 source，`EXTERNALSRC_BUILD/B` 只指 merged build view 的 `src`。构建入口必须分别传递 `V5_LINUXCNC_EXTERNAL_SOURCE=${VM_BUILD_ROOT}/temp_source/current/linuxcnc` 和 `V5_LINUXCNC_EXTERNAL_BUILD=<fixed-build-view>/merged/src`，recipe 只能令 `S=${EXTERNALSRC}`、`B=${EXTERNALSRC_BUILD}`，不得再从 source 变量推导 `B`。该 view 只是为不支持 OOT 的上游工具提供的 writable build 输出视图，不是第二源码投影或 owner；不得人工修改、下载、持久复用或反向同步，任务退出必须卸载，下一轮只允许按输入 identity 重新建立本轮 view。policy gate 必须同时证明两项环境变量分域、source 未写、B 与 S 不同、lower identity 一致和 overlay 已在退出时卸载。

VM 长期具备外网访问能力不属于第二套源码、第二构建主线或第二代理 owner。VM 可以通过正常虚拟网卡直接联网，也可以在直接联网受限时安装一套常驻 Linux 代理客户端并持久保存其服务配置；代理上游可使用 Windows 受信任局域网监听或独立登记的合规出口。该能力只服务 VM 系统维护、非构建诊断和用户明确要求的普通联网操作，不得成为源码、第三方包或正式构建输入真值。VM 全局网络可长期可用，但 Git/source import、BitBake fetch、远程 sstate、configure/compile/package/rootfs/image 和离线认证仍禁止通过外网补齐输入；正式构建入口必须显式清除 `HTTP_PROXY`、`HTTPS_PROXY`、`ALL_PROXY` 及对应小写变量，并通过 network-disabled 门禁后才能继续。不得把代理地址、凭据或订阅写入项目源码、recipe、镜像、普通文档或构建配置；正在运行的构建不得为了设置代理而被中断、重启或改变网络环境。

full-current、发布或离线全量构建发现 Windows canonical 输入缺失或内容 hash 不符时，允许进入受控的“缺项报告 -> Windows 导入 -> 原失败层重试”闭环，但下载动作仍不发生在 VM、BitBake 或构建进程中；日常 focused 构建只有在当前 target 实际请求缺失输入时才生成该直接缺项报告，不预扫完整 inventory。VM 门禁必须写出 `v5-missing-source-inputs-v1` 机器可读报告并以专用状态停止；报告只能引用 `v5_bitbake_source_inventory.json` 已登记、带固定 URL/版本、许可证归属和校验信息的包。Windows importer 读取该报告，只把命中的包下载到 `D:\v5` 对应 canonical source-package owner，校验 SHA-256、重建完整 source manifest 并再次验证；未登记 URL、缺 checksum、缺许可证、路径逃逸、inventory identity 不一致或出现额外文件时必须 `blocked`，不得自动信任或下载。验证通过后 VM 通过只读挂载/唯一 current 投影消费新 identity，并只重跑报告登记的原失败 target/recipe/stage；必须保留有效 `sstate`、`tmp/work` 和 stamp，不得因此清缓存、重编 kernel、扩大到无关包或直接执行最终镜像。只有该最小层通过且部署闭包确实变化时，才继续必要 rootfs/manifest 和最多一次 current 镜像。

1. 所有会被编译器、链接器、代码生成器、补丁任务、配置任务、打包任务或 rootfs 任务读取的源文件，必须在正式 VM 构建开始前完整落盘到 Windows `D:\v5` 唯一源码树；范围包括 Linux/PREEMPT_RT、LinuxCNC、EtherCAT、板端运行代码以及实际进入产品的第三方源码。每份源码在 `v5` 内只能有一个持久 canonical 路径：可维护的展开源码树和作为正式输入的固定源码包分别按其唯一 owner 存放，不得为同一输入再保留备份树、任务树、同步树、重复压缩包或另一份可编辑展开树。每项外部源码输入必须登记固定版本或提交、上游来源、许可证和内容 SHA-256，使本地文件而不是网络分支、浮动标签或远程仓库成为构建输入真值。
2. VM 默认保存工具链、构建输出、可删除的 sstate 编译缓存，以及上述唯一源码投影 `${VM_BUILD_ROOT}/temp_source/current`；VM 不得另存第二套源码树，也不得持久保存 BitBake `downloads` 源码缓存。共享挂载只负责提供 Windows 真源和刷新投影，实际 Linux/PetaLinux/Yocto 编译优先在 identity 一致的 VM 本地投影上执行。若工具强制要求 `DL_DIR`，它只能把 Windows `v5` 内登记的只读 canonical 源码包目录作为本地 mirror，并按需把实际消费的包放入可删除构建状态；不得形成另一个源码 owner。configure/compile/package/rootfs/image 禁止通过 Git、HTTP、HTTPS 或远程 sstate 临时取得任何实际消费的源文件；当前 target 实际输入缺失或 hash 不符时该层 fail-closed，未被当前 target 消费的仓库缺项留到 full-current、发布或离线全量构建再补。生成物例外：VM 允许在 `VM_BUILD_ROOT` 的固定 current 位置保留唯一一套当前镜像集合，供普通重写卡、更新或证据复核使用；集合可同时包含 `BOOT.BIN`、`image.ub`、`boot.scr`、rootfs 和完整 SD 镜像，因为它始终唯一，所以不强制增加 `COPY`/“副本”标签。其机器可读清单仍必须记录 Windows canonical source identity、build mode、生成时间、每个产物 SHA-256 和 `board_verified` 状态。新镜像生成后替换旧集合，不保留按日期、版本或任务区分的第二套镜像；只有来源身份仍与当前 Windows 真源一致且产物 hash 复核通过时才允许复用，否则不得作为当前镜像部署。该唯一镜像集合只是生成物，不是源码真值。
3. sstate mirror 和 VM 本地 sstate 都只能是可选的编译加速手段，不能成为源码真值或编译输入完整性的替代品；是否启用远程 mirror、采用何种超时和重试策略属于构建实现方案，必须结合当前工具链验证后决定。无论是否命中 sstate，都必须能够从已落盘 Windows 的完整源码继续构建，远程缓存不可用不得迫使构建临时联网补源码。允许获取新源码的动作只能由 Windows 主机的独立 importer 执行：可以在正式构建前主动导入，也可以由上述缺项报告闭环触发；两者都必须下载到 `D:\v5` 的唯一 canonical 路径，在 Windows 核对版本、许可证、inventory identity 和 hash 后，VM 才能只读消费。VM 不得承担任何源码下载，也不得把下载结果保留为第二份源码缓存。
4. 产品 C/C++ 源必须出现在 CMake File API 的产品目标及其依赖闭包中；ARM 制卡构建必须验证每个部署二进制是目标架构 ELF 和规定 ABI。
5. Python、Shell 和配置不冒充“已编译”：Python 做语法检查，Shell 做解释器语法检查，结构化配置做格式/schema 检查，随后按部署清单原样进入 rootfs。
6. 测试、smoke 和开发工具可以在本地/构建阶段执行，但除部署清单明确登记且有真实产品消费者的诊断入口外，不得进入产品 rootfs。
7. rootfs staging 完成后必须生成包含每个普通文件和符号链接的类型、权限、大小、内容 hash/链接目标和相对路径的完整清单；正式写卡后重新挂载 rootfs，要求文件集合及各字段逐项一致。只检查 UI、Command Gate 等少量哨兵文件不能证明完整写入。
8. 编译输入清单缺项、Windows 源文件缺失或 hash 不符、正式构建需要联网补源码、远程缓存被当作必要源码输入、新增生产文件未进入产品目标或部署清单、部署目标重复、CMake target 不存在、脚本语法失败、ARM ABI 不符、rootfs 多/少文件或写后内容/权限不一致时，必须在进入板端启动前失败。

<a id="qspi-ssh-recovery"></a>
### 4.3 QSPI 最小 SSH 恢复通道

- QSPI 的唯一职责是在 SD 损坏或产品 rootfs 无法正常使用时，从不依赖待修 SD 的 eMMC 或独立恢复 rootfs 进入最小 Linux，提供网络和 SSH/Dropbear。进入 QSPI 恢复环境后，外部维护端只通过该 SSH 通道识别、挂载、更新和回读 SD 的 boot/rootfs 文件。
- QSPI 恢复验收只要求：独立 rootfs 启动、网络就绪、SSH 可登录、SD 分区可受控挂载，以及按 Windows 唯一源码/VM 正式产物的 manifest 完成文件更新、SHA-256 回读、`sync` 和安全卸载。只更新文件时不得为 QSPI 额外扩展分区修复或产品依赖；仅在 SD 文件系统确实需要修复或调整时才使用 `e2fsck`/`resize2fs`。
- QSPI 不是产品运行环境，不得为它接入或验收 LinuxCNC、EtherCAT、LVGL UI、触摸、页面缓存、产品参数 owner、产品启动时间、物理显示或 login 画面。QSPI 也不得把待修的 SD `rootfs` 当自己的 rootfs，不得把板端手工修改留成唯一真源，不得因为恢复方便变成产品 fallback。
- SD 更新与 hash 回读完成后，必须结束 QSPI 恢复阶段，用继电器切回 SD 完整冷启动。login/console 遮挡、真实触摸、UI 启动时间、LinuxCNC/EtherCAT 和其他产品功能只在 SD 产品系统中验收，不得用 QSPI 恢复系统的状态代替。

<a id="offline-full-sd"></a>
### 4.4 V5 文件夹离线全量生产 SD

- `PROJECT_ROOT` 对应的完整 `v5` 文件夹是 V5 唯一、可搬迁的全量灾难恢复单元。离线验收可把该文件夹传入一台完全断网的受支持电脑，但该传输副本只能是只读、单次验收临时输入，禁止修改或复用并在验收结束后删除。验收不得再依赖 `origin`、GitHub/GitLab、HTTP/HTTPS 源码站、远程 sstate、NAS、原开发电脑、预存 VM build-output、旧 SD/板端文件或人工记忆，必须能够从零生产完整 SD。
- `v5` 文件夹内必须包含或可由其内部离线介质建立全部项目专用输入：Windows 唯一源码、所有固定版本第三方源码、PetaLinux/Yocto/Vivado及交叉编译所需的版本化离线工具链或完整 VM/安装介质、离线许可证边界与校验信息、当前 XSA/bitstream/FSBL/U-Boot/device-tree 输入、recipe/patch/config、制卡与回读工具。任何项目专用输入若仍只存在于已安装 VM、系统目录、在线安装器、远程仓库或个人机器，均不得宣称离线恢复闭合。
- FPGA 有两个不同功能 owner：`vivado_hw_project/` 是当前产品/当前 SD 使用的 FPGA 功能，当前 XSA、bitstream 和 PetaLinux 硬件输入只允许来自这里；`new-vivado/z20_v1_5_hw_project/` 是另一套不同 FPGA 功能，不是前者的副本、升级分支或 fallback，也不得进入当前制卡链。两者各自文件只在自己的 owner 内维护，公共文件不得复制后分别漂移。
- 离线介质和源码输入仍只有 Windows `v5` 内一个 canonical owner。VM 可保留唯一派生投影 `${VM_BUILD_ROOT}/temp_source/current` 加速构建，但不得出现第二个投影目录，不得人工修改、提交或作为灾难恢复输入；空缓存离线认证必须证明即使删除该投影，也能只凭 `v5` 重新生成它和全部 SD 产物。
- 正式离线验收必须从新的空构建输出和空 downloads/sstate 开始，物理断网或由门禁证明无网络访问，完整生成 `BOOT.BIN`、`image.ub`、`boot.scr`、device tree、rootfs 和全量 manifest；不得用历史镜像、旧缓存命中或只重新打包现成产物代替全量再生产。
- 空缓存离线验收只在用户明确要求、发布/灾难恢复里程碑需要重新认证，或需要对外声明 `v5` 离线全量恢复能力时执行，不得自动插入每个日常代码修复。离线源码清单、工具链/VM介质、许可证边界、硬件 handoff、PetaLinux 基线或制卡链变化会使旧离线认证失效并要求后续重新认证，但不妨碍当前功能先按 4.2 的增量梯级完成板端修复与对应功能验收。
- 普通开发期间可在 VM `VM_BUILD_ROOT` 保留唯一一套当前镜像集合以加快重写卡或复核，无需额外标注“副本”，但必须服从 4.2 的唯一性、来源身份和 hash 清单要求。该镜像集合丢失不得影响从完整 `v5` 重建，保留它也不得降低、跳过或冒充本节的空缓存断网验收。
- 生成后必须对一张可擦写的空白 SD 完成分区、格式化、写入、逐文件/hash 回读和安全卸载，再切回开发板进行完整冷启动，证明 boot、rootfs、产品服务、LinuxCNC/EtherCAT、LVGL UI 和真实触摸来自本次离线产物。只有该路径通过才能声明 `board_verified` 的离线全量恢复能力。
- 受许可证或再分发限制的工具仍必须有存放在 `v5` 内、可在断网环境完成安装或启用的合法离线介质、版本、校验值和操作入口；若法律或厂商机制使其无法随 `v5` 离线使用，该最高需求当前状态就是 `blocked`，不得退回“另行联网下载”或“目标电脑预先装好”作为默认前提。

<a id="global-retirement"></a>
## 5. 变更、迁移与退役

1. 先判断变更是全局规则还是专项需求：前者先改本文，后者只改其详细 owner；同时命中时先改本文的全局边界，再改专项细节。
2. 新增全局规则时，必须登记其受影响的专项 `REQ-*`，使页面和服务可从本文定位到唯一细节 owner；不得把完整细节复制进多个功能文档。
3. 迁移字段或公共路径时，先让全部消费者切到新的唯一 owner/readback，再同一 slice 删除旧 writer、旧 reader、旧模板、旧默认值、旧测试和旧文档引用。
4. 不得用兼容 alias、双写、环境变量、仅测试入口、静默默认值或 UI 私有 cache 作为迁移后门。

<a id="global-acceptance"></a>
## 6. 验收

以下条件同时满足，才可称全局通用配置/规则已闭合：

1. 每项全局规则在本文有唯一条目，专项细节只在对应唯一 owner 中存在。
2. 每个全局配置字段可追到唯一 writer、运行 owner、generation 和实际 readback；不存在第二真源或运行期读盘路径。
3. 抽查至少一个主页面、一个设置页、一个其它页面/服务，证明它们复用相同的公共首帧、弹窗、刷新或配置路径，而非各自实现。
4. 对 hot/cold 变更分别验证 owner readback、generation 一致性和规定的生效时机；对退役项验证旧 reader/writer/fallback 已物理删除。
5. 板端可见行为的最终状态仍按其专项 owner 的 operator-path 验收声明；本文自身只定义规则与配置闭合，不替代板端功能验收。
6. 编译输入闭包检查必须证明：每个步骤已先完成 Windows 能力判断，Windows 可执行步骤没有绕到 VM；进入 VM 的步骤记录了不可替代的 Linux/PetaLinux/Yocto/ARM/vendor 能力。所有实际参与构建的源文件在 VM 构建前已落盘 Windows，版本/来源/许可证/hash 已登记，正式构建不需要从远程补充源码；远程或本地 sstate 即使启用也只提供可删除的加速缓存，不影响 Windows 源码闭包的完整性。
7. 离线全量生产检查必须证明：完整 `v5` 文件夹在断网、无历史 VM build-output、空 downloads/sstate 和空白 SD 条件下，独立生成全部启动/rootfs产物，完成写后全量回读并在开发板冷启动产品；缺少工具链、第三方源码、硬件输入、许可证离线介质或任何网络依赖时不得宣称可恢复。
8. 产品运行闭包检查必须证明：部署清单中的全部二进制由同一聚合目标构建、生产目录无未分类文件、脚本/配置检查通过、rootfs 完整文件清单在 SD 写后逐项回读一致；之后仍需按专项 owner 完成真实板端启动和 operator path。
