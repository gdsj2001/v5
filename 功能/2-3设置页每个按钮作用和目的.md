授权/DNA latch 消费边界：本文只记录设置页按钮怎样触发和显示授权/DNA 状态；Device DNA 读取、授权文件验签、VPS challenge/signature、profile private scope 和开机/上使能前 latch 语义只引用 `REQ-DRIVE-PROFILE-AUTH-CHAIN` 及 `功能/VPS登录与项目对接说明.md`。设置页 `登记本机码`、`下载授权`、`服务器下载` 可以发起后台非实时校验、下载和状态刷新，但不得把 raw PL DNA、DNA hash、`pl_dna_hash`、token、私钥或授权原文写入 UI、日志、SHM、本地状态或仓库。`登记本机码` 必须现读本机 PL DNA，并把该 DNA 和设备公钥发给 VPS；VPS 先验证公钥，公钥通过后才登记 DNA、计算并保存 6 位 `vpsDistributionId`，同时在 VPS 数据管理中登记 ID-DNA-公钥关系并创建该设备 private 文件夹；同一 PL DNA 已登记时，VPS 必须直接返回已保存的 canonical `vpsDistributionId` 给板端，不得重新生成 ID。private 文件夹名统一为 `<vpsDistributionId>-<pl_dna_hash>`，但该文件夹名只允许作为 VPS 服务端内部存储路径，不得暴露给板端 URL、UI、日志或下载接口。板端只回读 VPS 响应字段 `vpsDistributionId` 的原样 6 位 ID 和校验状态，不得本地推导最终 ID，也不得从 `device_id`、`vps_distribution_id`、旧状态文件或授权文件回退推断显示 ID；回读成功后本地登记状态只允许保存 `vpsDistributionId`、登记结果码和公钥摘要，设置页顶部 `本机码` 只显示该 6 位 ID，未登记时显示 `未登记`。`DNA_REGISTER_UPLOADED_PENDING_AUTH` 表示公钥已通过 VPS 校验、DNA 已上传、VPS 已算出/返回 ID、VPS 已建立登记关系且板端已保存本地 ID 状态，只是授权文件还等待工厂端生成/上传；设置页不得把它当登记失败，也必须显示已登记 ID。`下载授权` 和 `服务器下载` 必须先从本地登记状态读取 `vpsDistributionId`；本地继续前置顺序固定为先校验 6 位 ID 存在，再现读本机 live DNA 并在内存中计算 DNA hash，之后才允许发起 VPS 请求。下载请求 URL/path 只能携带保存的 6 位 `vpsDistributionId`；本次临时读取的 DNA/DNA hash 必须作为 header 或 body 校验字段随请求发送给 VPS，不得放入 query，并且动作结束后不得落盘保存。VPS 对自带 ID 的访问必须用临时 DNA/DNA hash 校验 ID-DNA 关系，校验通过后在服务端解析到 `<vpsDistributionId>-<pl_dna_hash>` private 文件夹并放行下载；客户端不得直接用 `id-dna`、hash-only 或旧 ID-only private 目录当本地真源。没有已登记 ID 时必须提示先登记本机码，不得只按 DNA hash 访问 VPS。运动、Home、Start、Jog、设0、设置驱动和 `estop_or_reset` 只能消费后台已经锁存的只读授权/DNA 状态；热路径不得每次点击都重新读取硬件 DNA、验签文件或访问 VPS。若 latch 缺失、过期、三次 DNA 读取不一致、授权 hash 不匹配、授权过期或状态不可用，设置页只显示 degraded/失败原因，运动和上使能前置必须 fail-closed，返回 `MOTION_GATE_DNA_NOT_VERIFIED` 或对应授权失败码。

参数域归属：

最新交互边界：设置页每个可编辑字段在软键盘 `OK` 或下拉选择提交时必须确认保存到该字段唯一 owner，并完成本字段源位置回读后才清该字段 dirty；这包括 `motion_model` / `[RTCP].MODEL` 和 `G53_A_Y/G53_A_Z/G53_B_X/G53_B_Z/G53_C_X/G53_C_Y`，它们只写 active runtime INI `[RTCP]` / native model/geometry owner，不写 `settings_runtime.json`。进入设置页后右上角按钮默认显示 `返回主页`，只绑定本地主页导航；本次会话任一字段完成 owner 写入/readback，或设置动作成功并由 owner 回读 `restart_required/restart_deferred` 后，同一按钮立即、单向切换为 `保存并重启`。点击时把已完成 owner 写入/readback 的变更统一进入 canonical clean restart handoff。完成证明必须来自重启后的 resident owner、native/runtime、drive provider 和必要物理总线 readback 一致，不能只靠已发起 handoff、页面切换、machine-on 或驱动 Operation Enabled。

`从站` 和 `bit` 的按钮/单元格交互只负责字段 owner 保存：`从站` 写自建参数表并由 resident self owner 回读，`bit` 写驱动参数表/驱动映射表并由 resident drive owner/drive provider 回读。它们不得在字段提交阶段直接改微内核运行内存、HAL 映射或 EtherCAT live 绑定；物理生效只能在 `保存并重启` 的 canonical clean restart handoff 中由底层去使能、按开机链路重新装载 RAM、重建 HAL/EtherCAT/drive provider 并回读证明，失败保持去使能和 fail-closed。

以下名称只表示 运行内存/UI 显示投影或 UI 分区，不允许落地成 V3 自造参数文件。除 `设置驱动` 的驱动私有域外，可由 LinuxCNC/runtime INI、`PARAMETER_FILE`、`tool.tbl`、HAL/IO 或登记 native gate 承载的字段必须迁入对应 owner；历史上已经造出的 `system_params`、`axis_params`、`tool_params`、`wcs_offset_params`、`io_params` 文件或写入 owner 要删除。

最高参数表闭环：
- 设置页参数表所有字段都遵守 `开机硬盘/native owner -> native/运行内存 -> UI显示；坐标/轨迹、转速、进给速度和主轴/进给倍率再投影到 SHM；当前模态直连 native modal/tool 状态`，以及 `UI dirty -> Broker/settings_apply/native owner -> 写唯一 owner -> 刷新 native/运行内存 -> UI回读`。只有 `功能/0-4共享内存.md` 白名单允许的坐标/轨迹、转速、进给速度和主轴/进给倍率显示字段可以进入 SHM；参数、保存成功、动作结果、active mode、安全状态、Home/homed、WCS/G92/RTCP/tool 和驱动模式不得同步到 SHM。
- UI dirty 值只表示编辑/请求态；按钮弹窗里的“写入参数表”必须理解为写入该字段唯一 owner，不能理解为新增或继续使用 V3 自造参数文件。
- 保存按钮、即时保存、读取驱动后的表格刷新都必须以 fresh native/运行内存回读或受控 read-only 内存回读快照更新 UI；坐标/轨迹、转速、进给速度和主轴/进给倍率可同步刷新允许的 SHM 显示投影，当前模态必须直连 native modal/tool 状态。`运动模型/模型选择` 下拉提交时就是确认保存，必须立即通过 `settings_apply` 单字段写 `motion_model` / active runtime INI `[RTCP].MODEL` owner 并完成源位置回读；设置页下拉文本只是一笔请求输入，不得作为 UI 轨迹、RTCP 投影或微内核运动学真源。`G53_A_Y/G53_A_Z/G53_B_X/G53_B_Z/G53_C_X/G53_C_Y` 在软键盘 `OK` 或下拉/单元格提交时就是确认保存，必须立即通过 `settings_apply` 单字段写 active runtime INI `[RTCP]` owner 并完成源位置回读；它们不得落入 `settings_runtime.json`，也不得把 UI dirty 值当运行态真值。A 中心 X 显示当前模态加工坐标系 X 偏移值，B 中心 Y 显示当前模态加工坐标系 Y 偏移值，C 中心 Z 显示当前模态加工坐标系 Z 偏移值，`G53_A_X` / `G53_B_Y` / `G53_C_Z` 不得进入 dirty 或保存链路；B 中心 X/Z 参考 A/C 中心坐标体系，使用 `G53_B_X/G53_B_Z` 或等价 native geometry owner。旧 `settings_runtime.json`、`machine_settings.json`、默认值、`/run` 结果或 dirty 值不得冒充当前生效参数。

### 2.2 y=236 驱动动作行

BUS/EtherCAT 可用状态下，y=236 这一排驱动动作按钮（`下载授权`、`服务器下载`、`扫描从站`、`复位驱动`、`读取驱动`、`清除故障`、`设置驱动`）按下后必须立即弹出同一格式的进度窗口。窗口格式固定为首行 `进度 n/m: <当前阶段>` 或当前代码等价阶段文案，末行 `耗时 <秒> 秒`；后端返回 `code`、`failed_stage` 或 `message_cn` 时继续在同一个窗口追加显示，不允许只在顶部黄字提示，也不允许长时间停留在固定“正在执行”文案。按当前 v5 代码，`服务器下载` 优先读取 `/run/8ax_v5_auth_download/drive_profile_server_download_progress.json`；驱动读/写类动作的 `/run/8ax_v5_drive/*_result.json` 只允许作为本次弹窗/action evidence、阶段诊断和兼容结果输入，不能作为设置页稳定显示、保存成功或运行态真源。没有专用进度 JSON 的 `下载授权`、`扫描从站`、`读取驱动`、`复位驱动`、`清除故障`、`设置驱动` 必须由 UI progress owner 提供本地阶段兜底，并持续显示耗时，直到 actiond/后台返回终态。执行中弹窗只要显示可点击 `关闭`，该按钮的唯一语义就是携带本次 `run_id` 向 action owner 发送正式取消请求，并终止对应后台子进程/进程组；UI 必须等待 `cancelled` 终态后再隐藏弹窗、释放 action lock 和恢复冲突按钮，不得只隐藏窗口让任务继续。动作越过 owner 声明的不可逆提交点后必须移除/禁用 `关闭`，不能保留无法兑现取消语义的按钮。后台后续返回取消、失败或成功时按同一个 run_id 更新本弹窗/顶部状态，不得由旧任务覆盖新任务。

按 `REQ-BUS-PULSE-MODE-DIFFERENCE`，设置页驱动动作不再把静态禁用/预检拦截组合作为默认闭合方式。代码解决方向是优先跑通真实后台动作：`下载授权`、`服务器下载` 按授权/DNA/VPS owner 执行，不因 Pulse/StepDir 模式静态禁用；`扫描从站`、`读取驱动`、`复位驱动`、`清除故障`、`设置驱动` 由后台按当前硬件能力、目标从站、profile、授权和安全状态执行。若现场没有可用 BUS/EtherCAT 目标、写类动作缺必要 profile/readback/安全条件或目标模式真实不适用，后台返回结构化不适用/失败结果，UI 在同一进度弹窗显示原因和下一步；不得为了掩盖未实现路径而新增静态禁用分支或只让按钮灰掉。

多驱动批处理合同：`复位驱动`、`读取驱动`、`清除故障`、`设置驱动` 都必须把本次实时扫描和轴绑定得到的全部目标作为一个协调批次，不得按单轴完整执行“命令/写入 -> PREOP/OP 或软件重启 -> 读回”后才处理下一轴。EtherCAT SDO/mailbox 报文本身允许按总线安全顺序串行发送，这不等于产品动作逐轴闭合；禁止为追求字面“同时”启动多个线程或进程并发争用同一 EtherCAT mailbox。每个动作必须按全目标阶段屏障执行：先冻结全部目标和 profile，再完成本阶段所有目标的命令/读写，然后只进行一次目标集合协调恢复，最后统一 fresh readback 全部目标。涉及 EtherCAT AL 状态切换时，后台必须先证明本次冻结目标集合与 fresh 主站扫描集合完全一致，再只发一次不带单从站 `-p` 选择的主站级 `ethercat states <STATE>` 请求；禁止循环 `ethercat state -p <position>` 逐从站切换，因为参考时钟从站与其它从站分批切换会破坏同一 DC/SYNC 周期。目标集合不一致时必须 fail-closed，不得退回逐从站状态循环。任一目标失败时整个批次返回 partial/fail-closed，并保留逐目标证据；不得先把已处理的单轴宣布成功、继续用后续轴状态切换扰动前序轴，也不得用末尾清报警掩盖过程中持续产生的同步周期故障。

- `复位驱动`：先对全部目标完成扫描、profile 匹配和一次 native drive-write safety gate，再向全部目标下发 `drive.restore_factory_defaults`；全部复位命令结束后统一恢复 EtherCAT OP/mailbox、统一清除复位产生的可复位报警，并一次性读回全部目标。不得对每轴依次做“恢复出厂 -> PREOP/OP -> 清故障 -> 读回”完整闭环。
- `读取驱动`：先冻结全部目标，再读取全部目标的必读 SDO。仅当确有 mailbox/OP 读失败时，才允许对本次冻结的完整目标集合做一次主站级协调恢复并统一重读；不得每读一个轴就触发 PREOP/OP 状态循环，也不得对失败子集逐从站切换状态、让读取某轴的恢复扰动其它已读目标。
- `清除故障`：先冻结本次实时扫描和轴绑定得到的全部目标。驱动 fault 本身可能使从站停在 `PREOP/SAFEOP`，因此 `OP` 不是发送 `drive.reset_fault` 的前置条件；只要本次 mailbox 与必读 SDO fresh 读回可用，就必须在一个写入阶段向全部目标下发 reset，不得先要求 fault 状态下进入 OP 而形成循环门禁。只有处于 `INIT/BOOT`、mailbox 不可用或必读 SDO 缺失的失败目标集合，才允许先做一次协调的 mailbox-ready 恢复。全部 reset 写入结束后，对整个目标集合只做一次协调 OP 恢复，再统一 fresh readback 全部目标。实际 fault bit/error code 仍非零、OP 恢复失败或读回缺失时，必须保留该目标原始 actual 并返回 partial/fail-closed；不得再按轴循环 PREOP/OP、不得用状态切换代替 reset 写入。
- `设置驱动`：这里只登记批处理边界；完整阶段、activation、读回和失败合同以 `功能/2-4设置页设置驱动按钮作用和目的.md` 为准。必须先完成全部目标参数写入，再协调 activation/总线恢复，最后统一 fresh readback 全部目标。

### 2.3 设置页动作的自动 Machine 状态编排

设置页长动作在 actiond 接受本次 `run_id` 后，必须先由唯一 native 状态编排 owner 读取 fresh 急停和 Machine Enable actual，记录入口态，再按本次 action/profile 的阶段合同执行。状态合同只有以下三种基础值：命令登记 `requires_disabled=true` 表示本阶段必须自动 Machine Off；命令登记 `requires_enabled=true` 表示本阶段必须在急停已由操作员明确解除且 native enable gate 允许时自动 Machine On；两者都未登记表示保持入口态。两者同时为 true、同一阶段的目标合同互相冲突或字段缺失时属于 profile 合同错误，必须在任何写入/运动前 fail-closed。`requires_disabled=false` 绝不能反推为必须上使能。

自动切换按全部目标和阶段屏障只执行一次，不得逐轴反复上下使能。已处于所需状态时直接继续；正常 Machine On/Off 与 CiA402 过渡只更新进度，不弹失败窗。只有 native gate 拒绝、急停仍生效、切换超时、fresh actual 无法确认或真实驱动/总线故障时才显示具体错误。UI 只提交动作并翻译本次结构化结果，不得按按钮名、固定轴顺序或固定从站号自行决定 Machine 状态，也不得提示操作员“请先手动上使能/下使能”来代替后台实现。

| 动作 | 执行期 Machine 合同 | 成功终态 |
| --- | --- | --- |
| `登记本机码`、`下载授权`、`服务器下载`、字段编辑、`返回主页` | 不依赖 Machine 状态，保持入口态 | 保持入口态 |
| `扫描从站`、`读取驱动`、BUS/Pulse 单轴 `设0` | 只读/证据动作，保持入口态；mailbox PREOP/OP recovery 不等于 Machine Enable | 保持入口态 |
| `复位驱动` | 根据 `drive.restore_factory_defaults` 的 profile 合同自动 Machine Off，并等全部目标去使能及无运动 actual | 保持 Machine Off；恢复出厂已经使电子齿轮、设0和 count-domain trust 失效，禁止自动恢复运动使能 |
| `清除故障` | 根据本次各目标 `drive.reset_fault` profile 合同编排；当前命令只要求无运动时不得凭按钮名额外上使能或下使能 | 不自动 Machine On；入口 Off 或故障导致 Off 时继续保持 Off |
| `设置驱动` | 按 `drive.set_egear`、`drive.write_mode`、`drive.save_parameters` 等实际执行命令分阶段聚合合同；当前 SV630N 写入阶段自动 Machine Off | 全部写入/readback 成功且急停/安全 generation 未变化时恢复入口态；入口原为 Off 则保持 Off；失败保持 Off |
| `保存并重启` | canonical clean restart 提交前自动 Machine Off | 重启后保持安全 Off；服务启动/重启不得自动 Machine On |
| `计算bit` / `settings_calc_bit` | 禁止 | 不再执行 `drive_calculate_bit`、微动或样本计算 |

动作开始后的入口态、每次阶段切换、fresh final actual 和是否恢复入口态必须绑定同一 `run_id` 写入结果证据。失败或操作员急停后不得为了恢复入口态自动清急停或 Machine On；急停只能由操作员明确点击 `estop_or_reset` 清除。后续若新增确实需要 Machine On 的设置动作，必须先在唯一 action/profile owner 登记 `requires_enabled=true` 和对应 readback，再复用本节 native 编排；禁止新增 UI 门禁或第二套 enable helper。

上使能本身不是危险确认动作，但只能在无运动意图、无未完成 MDI/程序队列、无冲突后台写类 job、急停/硬故障/限位状态允许，并且 microkernel/native enable gate 通过后由已登记 native/owner apply 自动执行；结果必须以 microkernel/native enable gate 返回和原生回读为准。不得要求操作员手动上使能，也不得把上使能做成危险确认弹窗；若 native gate 拒绝或状态不可用，必须 fail-closed 并显示结构化原因。V3 不再用解释器 idle、速度容差、queued MDI 或 SHM/JSON 状态拼上层门禁。A/C 轴坐标写入、会导致旋转轴实际运动或相位重定的动作必须二次确认；设置页 BUS 单轴 `设0` 不再作为危险确认动作。X/Y/Z 直线轴设零、普通保存、读取、下载和通过 native gate 的上使能本身都不是危险动作。

显式急停按钮不属于上述写驱动安全前置表。急停只能走 native/microkernel 急停 gate 的薄适配，成功以 microkernel/native 回读确认；不得把设置页驱动写入前置、EtherCAT `0x6041` 状态字扫描或 V3 私有 `estop_loop` 逻辑复用成急停成功真源。

`复位驱动` 是长耗时写入动作，等待弹窗不能只显示静态“正在执行”。按钮触发后必须保持弹窗打开，并按批处理阶段刷新进度：提交复位请求、扫描并冻结全部目标/驱动资料、请求一次 native drive-write safety gate、向全部目标恢复厂家默认参数、协调恢复目标集合、统一清除恢复后的报警并读回全部驱动参数、等待后端完成整批校验。复位驱动必须使用专用长流程 job 等待上限，不能和普通驱动写入共用短超时；后台仍在恢复出厂或读回校验时，UI 不得先报 `DRIVE_WRITE_JOB_TIMEOUT`。最后一阶段不能伪装成已经在刷新轴参数表；只有 Broker 返回成功并完成全部目标读回写入后，才允许刷新设置页表格。最终成功或失败弹窗仍以 Broker 返回码和读回校验结果为准。

`服务器下载` 和 `读取驱动` 虽然不是写驱动参数的高风险动作，也必须使用和长流程按钮一致的阶段反馈。按钮触发后弹窗保持打开，先显示请求已提交；`服务器下载` 按当前代码从 `drive_profile_server_download_progress.json` 刷新真实后端阶段；`读取驱动` 必须显示冻结全部目标、批量读取、必要时协调恢复失败目标集合、统一重读和整批完成，当前可用本地阶段/倒计时和 `/run/8ax_v5_drive/drive_read_result.json` 作为本次弹窗/action evidence 与诊断输入，不得写成依赖当前代码不存在的专用 read progress 文件，也不得把 `/run` result 当作设置页稳定刷新或驱动读回真源。阶段文本必须显示 `进度 n/m` 或当前代码等价阶段文案和 `耗时 x 秒`；失败时同一弹窗显示 `code/message_cn/failed_stage`，不得长时间停在固定“正在从服务器下载驱动映射表，等待本机授权校验”或只显示最终“读取驱动完成”。设置页表格刷新只消费 drive/native provider fresh readback 或受控 read-only 内存回读对象。

`设置驱动` 也是长耗时写入动作，等待弹窗必须显示当前代码可取得的批处理阶段进度：提交请求、扫描并冻结全部目标轴、计算全部电子齿轮目标、向全部目标写入电子齿轮和模式、目标集合协调 activation/总线恢复、统一读回全部目标、最终全局健康确认和运行参数刷新。不得再显示或实现为逐轴恢复、逐轴重启、逐轴读回的完整小循环。当前 v5 代码可用 UI 本地阶段/倒计时和 `/run/8ax_v5_drive/drive_set_result.json` 作为本次弹窗/action evidence 与诊断输入，不得写成依赖当前代码不存在的专用 set-drive progress 文件，也不得把 `/run` result 当作设置页稳定刷新、保存成功或驱动已生效真源。`drive_set_parameters` 必须使用专用长流程 job 等待上限，避免后台已返回 `DRIVE_SET_OK` 时 UI 先报 `DRIVE_WRITE_JOB_TIMEOUT`。只有后台返回 `DRIVE_SET_OK`、全部目标的 drive/native provider fresh readback 校验通过、轴参数表按读回对象显示 `已写入` 后，才允许显示成功；成功后右上角按钮必须保持 `保存并重启`。

驱动长任务一旦进入 `BROKER_JOB_ACCEPTED` 异步模式，UI 主线程只允许按短周期提交 `job_status` 快查询并刷新弹窗，不能同步等待最终写入结果。`BROKER_JOB_RUNNING` 表示通道正常并清零连续失败计数；`JOB_STATUS_BROKER_SOCKET_UNAVAILABLE`、`JOB_STATUS_BROKER_DIRECT_FAIL`、handler 返回失败、状态结果无法解析或缺失状态码都计入通信失败。连续 5 次失败后，UI 必须结束当前异步任务并显示“后台通道连续不可用”的结构化失败，不得继续等到长超时才让操作员看到失败。若 `job_status` 返回业务最终失败码，则保持原始业务 `code/message/detail`，不要改写成通道丢失。

Broker 异步长任务必须携带 job ownership 元数据：`job_id`、`job_action`、`state`、`started_ns/submitted_ns/completed_ns`、`deadline_ns`、`cancel_requested`、`cancel_reason` 和 `final_result_owner`。取消只能通过正式 `job_cancel` 请求进入，先把任务置为 `cancelling`，由 worker 短周期检查 cancel token/deadline 后进入 `cancelled` 或 `failed_closed`；UI 不得本地伪造取消完成。取消、超时或异常后的最终 `job_status` 仍必须保留原始 `job_id` 和 `final_result_owner`，防止后台结果丢失或串到其它请求。

## 3. 容易写错的点

1. `读取驱动` 走 `drive_parameter_read`；禁止 `settings_read_only` 产品路径作为设置页按钮、Broker route、UI 刷新兜底或 `/run` 读回入口。
2. `下载授权` 当前不在顶部；它在 y=236 的驱动动作行，且该行动作按钮必须保持 82px 宽的单行文字，不允许再拆成两行。
3. `复位驱动`、`清除故障`、`设置驱动` 当前 UI 发起写类请求时只表达本次操作者动作意图和必要确认；Broker/后台不应把补充 `confirm` 字段或额外全局写门禁当成主要实现。`复位驱动`、`设置驱动` 是长流程，不再把 UI 请求里的旧 `status_epoch` 当成访问驱动前的门禁；安全性由执行时重新读取微内核/native 内存状态、请求身份、目标从站和映射表对象支持情况保证。缺失能力要返回结构化原因，代码修复方向是补齐真实写入/读回路径，而不是继续叠加前置拦截。`设置驱动` 的写类请求边界只引用 `功能/2-4设置页设置驱动按钮作用和目的.md`。
4. `设置驱动` 的按钮功能已剥离出本文；详见 `功能/2-4设置页设置驱动按钮作用和目的.md`。
5. `复位驱动` 就是恢复驱动厂家默认参数，必须走 `drive.restore_factory_defaults`，并在弹窗中写清会清除驱动内部参数、恢复后需要单独重新写电子齿轮和重新设0/验证；不能再把它写成清故障。复位驱动只做驱动复位和回读校验，读回结果只保存为设置驱动私域证据；成功且 owner 回读 restart pending 后，右上角必须切换/保持 `保存并重启`，用于统一触发 canonical clean restart handoff，不得再写成 `保存并返回`。
6. `清除故障` 只是清除驱动报警；它和 `设置驱动` 的边界详见 `功能/2-4设置页设置驱动按钮作用和目的.md`。
7. 轴参数表任意可编辑字段、`运动模型/模型选择` 和 G53 几何字段在软键盘 `OK` 或下拉选择变化后，成功只代表本字段唯一 owner 已按当前 v5 受控路径写入，并只完成本字段源位置回读；运行态生效和允许的 SHM 显示投影刷新必须在右上角 `保存并重启` 触发 canonical clean restart 后，由重启后的 resident owner/native/runtime/drive provider readback 证明。除 `设置驱动` 的驱动私有域外，这不表示 V3 参数表成为长期真源，最终仍以 LinuxCNC/native 内存/API 回读为准。
8. 顶部右上角按钮初始为 `返回主页`，本次会话形成有效参数变更/restart pending 后原位、单向切换为 `保存并重启`。`设置驱动`、UI 黄色轨迹、RTCP ON 当前加工坐标系投影或微内核运动学需要使用模型时，必须消费已写入并在重启/受控 reload 后生效的 active model descriptor/readback，不得从 UI dirty、设置页下拉文本、固定 AC/BC 分支或旧 v3 source 名称推导。
9. `保存并重启` 的完成边界是字段已逐项源回读确认、restart precheck 通过、canonical clean restart handoff 执行，并在重启后证明 resident owner/native/runtime/drive provider 与必要物理总线 readback 一致；恢复主页面只是 UI 展示恢复，不是完成证明。不得把“handoff 已发起”“若干服务已重启”“machine-on 恢复”或驱动 `Operation Enabled` 当成保存重启完成。
10. 如果 clean restart 或后续恢复阶段发现 `status_shm_stale`、`STATE_PUBLISHER_RESTARTING`、machine on、BUS/EtherCAT Operation Enabled、resident owner、native generation 或 drive readback 未恢复，必须保留诊断状态并按对应 owner fail-closed；不能把这些后台状态冒充为已恢复运行态，也不能用轻量返回主页面替代 clean restart 完成证明。后续运行准入仍以独立 native 真值和运动门禁判断。
11. 无有效参数变更且无 restart pending 时不得分发 `settings_save_return`，必须走现有 `nav_main` 本地导航；形成有效变更后只允许分发 `settings_save_return`，不得再走本地返回绕过 clean restart。
12. 从轴、S轴、刀库三个按钮当前都是 blocked，不是待验收的真实功能。
13. `settings_save_return` 的进度弹窗是操作员在等待后端期间的可见反馈，必须持续刷新阶段文本；失败必须在弹窗里显示结果，顶部黄色字不能替代弹窗内容。重启进入不可逆提交前，进行态弹窗 `关闭` 必须取消本次 run_id 对应的 clean-restart 子进程并等待 cancelled；取得 `SETTINGS_SAVE_RESTART_BOARD_REBOOT_SCHEDULED` 或等价“系统级重启已受理”终态后，结果弹窗必须保留 `关闭` 作为操作员确认入口，此时它不再表示取消：点击后必须同帧把物理和远程画面清成纯黑、抑制旧 UI 后续出帧并进入重启，绝不能恢复设置页或留下残影。canonical clean restart 完成前不得只因页面切换就关闭为成功；仅发起 handoff 只能标记为 `handoff_started` / `restart_pending`，不得写成需求闭合。
14. 上述受理终态只能预置 clean-restart 脚本，不能提前停 UI/relay/actiond。结果窗 `关闭` 必须携带本窗 exact `run_id` 发送正式 commit；收到 `SETTINGS_SAVE_RESTART_COMMIT_ACK` 后，同一次释放回调立即提交纯黑，后端至少等待 0.5 秒再停服务和 reboot。commit 失败或 run_id 不一致时保留弹窗并准确报错，禁止黑屏、禁止重启。

## 2026-06-21 轴设0按钮语义修订

设置页每个轴行上的“设0”按钮是单轴按钮，不是全局重启或全轴设零入口。

点击某个轴的“设0”时，UI -> Command Broker UDS/native dispatcher -> 已登记 settings/native apply owner 只处理该轴：
- 只采集该轴驱动当前位置并写入该轴 `zero_model.zero_counts`、绝对/拆分计数和读写证据；不得恢复或写入顶层 `旧浮点零位字段`、`zero_model.旧浮点零位字段`、`zero_model.旧倒数比例字段` 作为产品输入。
- 不扫描全轴、不构建全轴 target、不改其它轴参数。
- 不在按钮动作中自动重启 backend/LinuxCNC、不执行 runtime rebase；BUS 设0本次动作只证明零点证据和 hard/raw limit owner 写入，UI 等效坐标清零只能作为最后统一保存重启后的显示闭环，不作为本次按钮通过门禁。
- 成功后设置页保持可操作，用户可以继续点其它轴设0或修改其它设置。
- 单轴设0成功并形成 restart pending 后，右上角按钮切换/保持为“保存并重启”，最后一次性执行 canonical clean restart handoff 并在重启后用 resident owner/native/runtime/drive readback 证明生效。

设置页 `设0` 的 UI 安全包装必须按驱动模式分流，但不能绕过 `功能/2-5轴设0软件偏置方案.md` 的产品语义：
- BUS/EtherCAT 或绝对总线模式下，X/Y/Z/A/C 点击单轴 `设0` 后直接发送本次单轴 `settings_axis_zero` / `bus_count_domain_zero` 请求，不弹危险确认；BUS 单轴设0不要求自动上使能，也不得把上使能、idle、stillness、SHM ready 或 live raw limit 读回作为本次保存通过门禁。单轴 `设0` 请求只授权本次轴行请求，不得作为全局写门禁复用到其它按钮或后续点击。
- Pulse/StepDir 模式下，点击轴参数表单轴 `设零` 仍打开 `HOME_OFFSET` 软键盘；键盘不得用 SHM/MCS/`rotary_phase` 显示投影自动种子化输入，只能显示当前字段 owner/readback 初值或空值并等待操作员输入。Pulse 轴参数表另有 `回零方向` 列，通过下拉选择回零方向并写 `home_direction` owner；`设零` 键盘本身不得静默启用或提交 `home_direction=current`，X/Y/Z 也不得提交 `current`。
- Pulse 软键盘必须使用 modal blocked mask 或等效全屏输入吞噬机制；键盘关闭时继续使用短输入过渡保护，避免同一次触摸释放落到底层轴表、右上角按钮或 y=236 驱动动作行。

BUS `设0` 成功结果代表当前轴零点证据已落盘、active raw 限位已同步回读；live raw 限位装载和 UI 等效坐标归零要在最后统一保存重启后闭合，不能把 SHM 显示投影写成本次动作成功证明。Pulse `设零` 成功边界按 Pulse owner 的 `HOME_OFFSET` 写入和后续回零/native 证明处理。
