# G-code 兼容实现 AI 作业书（基于当前代码现状）

更新时间：2026-07-15

文档类型：`AI 执行作业书`。

当前功能状态：`source_only`。本文定义实际实施顺序、文件边界、最小验证、上板条件和失败处理；文档通过不代表兼容功能已经写入代码、构建或上板。

兼容翻译输入：[FANUC、三菱、新代、西门子、LinuxCNC 命令功能对照](./FANUC_三菱_新代_西门子_LinuxCNC_命令功能对照表.md)。

执行白名单输入：[G代码执行白名单](./G代码执行白名单.md)。人类可读命令表、放行条件、固定中文报错提示以及AI专用rule/error identity只在该白名单文档维护，本作业书不复制第二套规范表。

当前文档整理任务只允许修改 `功能/g代码兼容/` 内的 Markdown。只有用户后续明确要求进入实现，AI 才能按本作业书修改 Windows 唯一 canonical source、执行最小构建或部署板端；不得把阅读本作业书理解成已经取得代码、VM或板端操作授权。

## 0. AI 强制作业入口

### 0.1 使用规则

1. 一次任务只选择一个最小切片，禁止从作业0一路连续扩到作业8。
2. 用户指定具体缺陷、命令组、profile或阶段时，优先关闭该目标的最小纵向链；用户只说“继续实现”时，选择依赖已经满足的最低未完成切片。
3. 是否完成只能由当前源码和已执行gate判断，不能依据聊天摘要、进度文字、Git dirty数量、旧日志或旧板端结果。
4. owner和源码入口明确后，最多进行两轮focused只读定位；随后必须落最小修改或运行最近的focused gate，不得持续做全仓分析。
5. 每个切片默认交付到 `local_verified_only`。只有用户明确要求上板、当前缺陷必须依靠板端证明或进入批量集成里程碑，才继续VM、部署和operator验证。
6. 正常兼容成功必须少打扰用户；只有无法确定性处理的输入才按白名单错误表拒绝。
7. 本作业书不复制命令白名单。命令语义、四方等效形式、稳定rule identity、成熟度和中文错误提示始终以同目录两份输入文档为准。

### 0.2 每次开工必须先形成的临时作业卡

AI必须在commentary或内部plan中填写下列字段，不得另建任务板或过程Markdown：

```text
slice_id:              阶段号 + 功能组/profile，例如 S2-linuxcnc-wcs
user_goal:             本次唯一可观察结果
source_profile:        精确profile；未知时写 unresolved，禁止写 FANUC-like
rule_group:            本次命中的白名单功能组/rule identity
owner:                 program_runtime / UI / Command Gate / LinuxCNC native
allowed_files:         本切片允许编辑的精确文件集合
retired_paths:         依赖迁移后本切片必须物理删除的旧入口；没有则写 none
nearest_target:        最近compile/unit/smoke/parser target
proof_level:           source_only / local_verified_only / board_verified
board_required:        yes/no + 原因
stop_condition:        本切片取得什么证据后立即交付
```

任一字段不能确定时，只定位该字段的直接owner和输入，不扩成全树清扫。`allowed_files`之外新发现的问题进入下一切片；不得顺手修改。

### 0.3 必读顺序

未来进入代码实施时，每个切片按以下顺序读取：

1. 根目录 `AGENTS.md` 的入口核，只读取当前任务必需规则。
2. 本文A、B、2章和本次切片对应章节。
3. [G代码执行白名单](./G代码执行白名单.md) 的相关功能组、rule schema、成熟度、输出白名单和第13节错误提示。
4. [FANUC、三菱、新代、西门子、LinuxCNC 命令功能对照](./FANUC_三菱_新代_西门子_LinuxCNC_命令功能对照表.md) 中本次精确source profile和等效目标。
5. 当前owner源码、最近focused测试、`git status --short -- <allowed_files>` 和 `git diff -- <allowed_files>`。
6. 只有字段冲突或owner不清时才读取其它需求owner；不得先漫游Linux、LinuxCNC、PetaLinux或整个board树。

若当前用户仍限制“只改 `功能/g代码兼容/` 文档”，执行到第4步后只能修改该目录Markdown并做文档检查，禁止进入第5步以后的源码、VM或板端动作。

### 0.4 不可改变的owner边界

| 内容 | 唯一owner | AI允许做什么 | AI禁止做什么 |
| --- | --- | --- | --- |
| 原文、profile、parser、白名单消费、typed IR、canonical artifact、line map、错误列表 | `program_runtime` | 扩展现有owner并拆分低于500行业务文件 | 新建第二个G-code服务、UI parser或Command Gate parser |
| UI | 现有Program/MDI页面 | 提交请求、显示状态、显示canonical readback、错误跳转 | 自行翻译、保存第二份canonical、用UI文本证明执行成功 |
| Command Gate | 现有native/linuxcncrsh gate | 传输冻结identity、canonical路径/文本并返回同request回执 | 选择profile、改写刀路、绕过READY、保留raw passthrough |
| LinuxCNC/native | LinuxCNC解释器、运动和安全owner | 解析最终RS274/NGC、裁决运动/RTCP/坐标/刀补actual | 接收未白名单外部方言原文、让SHM/UI冒充actual |
| 白名单规范 | `G代码执行白名单.md` | 读取稳定rule和错误映射，代码生成实现投影 | 运行期读取Markdown、在源码另建窄白名单或默认放行 |

任何实现若需要改变这张表，先停止代码切片并只在用户允许的文档范围内提出明确冲突，不能自行增加owner。

### 0.5 单个切片的固定执行流程

每个代码切片严格按以下顺序执行：

1. **冻结目标**：填写0.2作业卡，明确一个用户可观察结果、一个owner链和一个最近target。
2. **核对基线**：确认相关canonical文件存在；读取focused diff，保护用户和其它任务已有修改。canonical文件缺失时先恢复该文件，禁止在缺文件状态跑gate或进入VM。
3. **获取精确文件锁**：只锁实际编辑文件；本地编辑和立即focused检查完成后立即释放。不得用文件锁等待VM、部署或用户反馈。
4. **先固定反例**：优先在现有focused smoke加入一个会暴露当前缺陷的最小失败用例；若是纯新增能力，先固定正例、拒绝例和identity预期。
5. **修改唯一owner**：只实现作业卡中的最小行为，不建立fallback、shadow、wrapper、临时env开关或第二缓存。
6. **删除已迁移旧路**：只有依赖已经迁移并被focused测试覆盖时，在同一切片物理删除对应raw/fallback分支；尚未证明无依赖的路径不得误删。
7. **运行最近gate**：先syntax/compile，再单target smoke，再必要的LinuxCNC parser验证。失败只修当前失败层，不跳到更大入口。
8. **形成状态**：focused gate通过即交付 `local_verified_only`；没有板端必要性就停止，不自动生成rootfs/image。
9. **必要上板**：只有0.2中 `board_required=yes` 才获取单operator资源锁，执行最小ARM target、最小部署和原始operator路径；结束释放资源锁并恢复安全态。
10. **交付并停止**：按0.9格式报告。下一功能组/profile必须作为下一切片重新填写作业卡。

### 0.6 最近gate选择规则

| 改动类型 | 必须先跑 | 通过后允许的状态 | 默认禁止 |
| --- | --- | --- | --- |
| 仅本目录Markdown | Markdown链接/表格/结构检查、文档路由检查、focused `git diff --check` | 文档已更新 | 编译、VM、部署 |
| compat纯C类型/parser/rule/emitter | touched syntax + `v5_gcode_compat_smoke`对应case | `local_verified_only` | UI、rootfs、image |
| `program_runtime`集成 | compat smoke + `v5_program_runtime_smoke` | `local_verified_only` | 全项目CMake |
| Command Gate/Start/MDI合同 | 最近formatter/validator/command smoke + exact LinuxCNC parser最小验证 | `local_verified_only` | direct backend冒充operator通过 |
| UI显示/错误跳转 | UI focused target + MCP 板端抓屏/鼠标的 fresh 页面验证 | `local_verified_only`；真实板端才可提高 | 全量PetaLinux |
| G-code真实运动语义 | 上述全部 + 最小ARM/部署 + 原始UI + fresh native actual + active-model匹配golden | `board_verified` | 只凭文件上板、截图或旧result称通过 |

同一 `受影响输入identity + gate实现identity + 命令参数` 已经通过时复用结果，不重复全树hash或全量构建。

#### 0.6.1 命令模板

命令从 `D:\v5` 执行。尖括号字段必须换成本切片真实路径/target；禁止原样执行占位符，也禁止为了找到target新建第二build目录。

```powershell
# 只看本切片，不用全仓dirty数量评价质量
git status --short -- <allowed_files>
git diff -- <allowed_files>

# 文档切片
python board/tools/docs/verify_v5_document_routes.py --strict-details
git diff --check -- <touched_markdown>

# 定位现有最近target；先rg，不猜target名
rg -n "v5_gcode_compat_smoke|v5_program_runtime_smoke|<affected_source>" board -g "CMakeLists.txt" -g "*.cmake" -g "*.c" -g "*.h"

# target已存在时，只构建/运行该target；build目录取现有canonical目录
cmake --build <existing_windows_build_dir> --target <nearest_target> -j 2
ctest --test-dir <existing_windows_build_dir> -R "^<nearest_test>$" --output-on-failure
```

只有Windows无法提供LinuxCNC解释器/ARM ABI且本切片确实需要时，才在取得单operator资源后进入VM：

```sh
# 先证明Windows只读挂载包含当前改动，再复用唯一现有build目录
grep -n '<expected_contract>' /mnt/v5-source/<affected_file>
cmake --build /root/v5-build/board --target <nearest_target> -j2
file /root/v5-build/board/<artifact>
readelf -h /root/v5-build/board/<artifact>
```

若 `v5_gcode_compat_smoke` 尚不存在，作业1只在现有CMake owner中登记该一个target及其直接源文件；不得顺带登记全部未来模块。文档切片不得执行上述源码/VM命令。

### 0.7 失败和阻塞处理

- **语法/编译失败**：保留证据，在当前target直接输入内修复并重跑同一层；不得先跑rootfs/image。
- **smoke失败**：先判断需求错误、测试旧合同还是实现缺陷；只修改裁决该断言的owner，不用放宽断言掩盖问题。
- **LinuxCNC预检失败**：用canonical行经line map反查source；补准确规则或准确拒绝，禁止删行后继续。
- **profile零匹配/多匹配**：返回固定中文错误和最少缺失字段；不得猜品牌、版本、单位或选件。
- **VM缺包或输入**：只报告当前target实际缺失项；不得让VM联网下载、清缓存或扩成离线全量认证。
- **板端不可达**：按项目恢复入口只恢复访问，不直接改板端产品文件；bounded失败后报告下一条件。
- **命令120秒无新输出**：检查PID、CPU、子进程和日志。仍活跃就报告真实阶段；已退出则收集退出证据并释放锁。
- **范围外缺陷**：在最终回复中列出证据和建议切片，不修改、不顺手修复、不创建遗留Markdown。

只有需要用户提供的真实外部手册、精确profile身份、硬件能力或外部授权才算用户输入条件；代码难、测试失败或gate较慢不算blocked。

### 0.8 第一条纵向链的实际落地顺序

第一次实现不能先做四品牌大全，也不能先做UI。固定顺序如下：

1. `linuxcnc_native_v1` profile、基础类型、source span、normalizer、lexer和错误目录。
2. strict block parser、模态状态、统一白名单引擎，首组只打开核心加工、基础地址、`G53`、`G54-G59.3`。
3. typed IR、canonical emitter、双向line map、输出AST白名单和确定性hash。
4. 接入Program Open事务，形成 `ANALYZING -> CANDIDATE/REJECTED`，失败清空执行资格。
5. exact LinuxCNC预检，同generation成功后才进入 `READY`。
6. Start、Resume、First Point和MDI全部消费同一canonical identity；关闭raw source/raw MDI旁路。
7. 最后接UI自动成功、准确错误、点击确定进入edit-only并跳到source位置。
8. 第一条链 `local_verified_only` 后，再按作业7一次增加一个真实profile或功能组。

前一步的退出条件未满足时，禁止进入下一步；但不得要求未触碰的后续功能组全部完成才关闭当前切片。

### 0.9 每次交付的固定格式

```text
slice_id:
result:                 用户可观察结果
changed_files:          仅列实际修改文件
retired_paths:          本切片实际物理删除项；没有写 none
gates_run:              逐条列真实执行命令/target和PASS/FAIL
state:                  source_only / local_verified_only / board_verified / blocked
board_closure:          未要求写 not requested；已执行写原始operator和fresh actual证据
missing_proof:          仍缺什么；没有写 none
next_slice:             下一最小切片，不自动开始
locks:                  file locks与vm_board.lock均应为released/absent
```

不得使用“已修复、已通过、板端正常”而不列对应gate和operator证据；也不得用长篇过程代替结果。

## A. 用户冻结需求

### A.0 需求总述

系统必须把“打开程序”和“手动输入”统一到同一条自动兼容链。用户输入可以保持FANUC、三菱、新代、西门子或LinuxCNC习惯，但真正进入微内核/LinuxCNC执行链的只能是经过白名单、确定性翻译和目标校验的LinuxCNC canonical G-code。

### A.1 打开程序自动翻译

1. 用户在程序页面打开G-code文件后，系统自动识别精确source profile并自动翻译成LinuxCNC能够直接运行的canonical G-code。
2. 原文件保持不变，用于编辑、追溯和错误定位；LinuxCNC不得直接运行未经确认的外部方言原文。
3. 翻译成功的canonical代码、source hash、canonical hash、profile、ruleset、line map和generation进入program_runtime唯一受控缓存。
4. 缓存提交必须原子完成；解析、翻译或目标预检尚未完成时不得显示可运行。
5. Start、Resume、预览、First Point和微内核/LinuxCNC执行必须使用同一缓存generation和同一canonical内容。

### A.2 手动输入自动翻译

1. 用户在手动输入/MDI页面输入的代码也必须经过相同的profile、normalizer、parser、白名单、typed IR、emitter和目标校验。
2. 可无歧义处理的输入自动翻译并进入MDI唯一受控缓存，不要求用户理解或确认LinuxCNC内部写法。
3. MDI待执行显示区必须显示微内核/Native执行侧当前generation实际接收的canonical代码，不得显示UI自行翻译的另一份文本。
4. UI显示文本、program_runtime缓存文本和微内核/Native待执行文本必须逐字节一致，并具有相同canonical hash和generation。
5. 进入板端 LinuxCNC 执行缓存和 MDI 待执行显示区的文本必须完全采用 LinuxCNC RS274/NGC 原生格式；FANUC、三菱、新代、西门子只表示翻译前 source profile，禁止把 canonical 编辑区标注为任一来源方言。
6. 手动编辑区可以在错误修复模式显示原始程序；错误未修复并重新通过翻译前，该文本不得进入MDI待执行缓存。

### A.3 空MDI和清除合同

1. MDI输入为空、全部删除、取消提交或切换程序时，program_runtime缓存和微内核/Native待执行MDI必须在同一事务中清空。
2. UI中的MDI待执行显示为空时，微内核/Native侧代码长度必须为0、canonical hash为空且不得保留旧generation。
3. 微内核/Native侧没有当前canonical代码时，Command Gate必须拒绝MDI运行请求。
4. UI、program_runtime和微内核/Native任意一处文本、hash或generation不一致时，系统必须fail-closed：清空待执行代码并报一致性错误。
5. 不得用旧缓存、旧SHM、上一次MDI文本或延时兜底恢复已清除的代码，避免误触发旧指令。

### A.4 兼容性和自动化目标

1. 代码白名单应在“能够证明加工语义一致”的前提下尽量扩大，持续覆盖常用格式、命令、profile和确定性转换。
2. 大小写、空白、换行、G/M前导零、成对包装、已登记小数规则、同义代码和确定性方言转换全部自动处理。
3. 唯一profile自动匹配、能力检查、canonical生成、输出白名单和LinuxCNC目标预检全部自动完成。
4. 正常兼容成功不弹逐行确认，不要求用户选择目标LinuxCNC代码。
5. 只有超出白名单、真实格式错误、profile无法唯一匹配、规则歧义、语义无法证明或机床能力缺失时才阻断。
6. 不得为了减少报错而猜测单位、坐标、刀路、刀补、RTCP、宏、IO或持久状态，也不得删除无法处理的代码后继续运行。
7. `cc-ac` 五圈弹簧的 XYZ 加工轨迹必须保持为一个不中断的连续 RTCP 加工段，禁止用密集 `G1` 小线段逼近圆柱螺旋。参考输入和 canonical 输出应使用 G17、增量 I/J 圆心和同步 Z 的 `G2/G3` 螺旋圆弧；为保留 A 轴每圈 `45°→55°→45°→35°→45°` 的姿态摆动，最小表达为每圈四个相切的 90°螺旋圆弧、五圈共20块。20块之间不得插入暂停、快速移动、抬刀、exact-stop或RTCP切换，并必须作为同一连续加工段通过目标解释器预检。

### A.5 错误提示和编辑跳转

1. 报错必须使用白名单文档第13节的固定中文提示，明确显示源程序行号、列号、错误代码、具体原因和修改建议；内部error identity只用于诊断和测试，不得代替中文解释。
2. 禁止只显示“G代码错误”“格式错误”“打开失败”或未经映射的LinuxCNC英文异常。
3. 多个错误应一次汇总；弹窗先显示第一条确定错误和总数。
4. 用户点击错误弹窗“确定”后，系统自动打开手动输入/程序编辑页面，载入原始G-code并跳转到对应行、列和token位置。
5. “确定”只表示进入修复页面，不表示忽略错误、绕过白名单或继续执行。
6. 编辑页面打开错误程序时必须处于edit-only状态，微内核/Native待执行MDI保持为空；用户修改后必须重新提交并重新完成自动翻译。
7. 错误跳转使用source ↔ canonical line map；LinuxCNC返回canonical行时也必须反查原始代码位置。

## B. 实施目标和固定原则

### B.1 最终目标

在 Program Open 或 MDI 提交阶段，系统自动完成：

1. 原文件读取和 identity 固定；
2. 精确 source profile 自动匹配；
3. 无语义格式归一；
4. 来源方言解析和 typed semantic IR；
5. 三层白名单；
6. 确定性 LinuxCNC canonical 生成；
7. line map、translation report 和错误映射；
8. 目标 LinuxCNC 精确版本预检；
9. canonical artifact 原子冻结；
10. Start/Resume 只使用已经冻结的 artifact。

正常可自动处理的程序不要求用户逐行确认。只有超白名单、格式错误、歧义、语义无法证明或目标能力缺失时才拒绝，并返回准确行、列、token、原因和下一步。

### B.2 不采用的做法

- 不在 UI 内实现第二套 G-code 解释器。
- 不在 Command Gate 内实现方言翻译或刀路算法。
- 不把现有预览 parser 直接升级成执行 parser。
- 不把外部方言原文先交给 LinuxCNC，再根据 LinuxCNC 是否报警判断兼容。
- 不允许 raw passthrough、未知 token 跳过、删掉错误行后继续运行。
- 不为每个品牌复制一套 parser。
- 不按品牌、M 码范围或宏目录整体放行。
- 不在运行热路径读 JSON/YAML/TSV 规则文件。
- 不先改 UI 再补 parser、规则和 owner。
- 不为本功能触发全量 Linux/PetaLinux/rootfs/image 构建。

### B.3 最快落地策略

1. 先用 `linuxcnc_native` 和核心加工/WCS建立第一条完整纵向链：Program Open、MDI、中文错误、canonical缓存、Start和清空合同全部贯通。
2. 基础设施只建设一次；此后每次只增加一个白名单功能组或一个精确外部profile，从parser、typed IR、emitter、focused测试到现有runtime入口同一切片闭合。
3. 当前白名单是最终兼容范围输入，不代表所有条目已经实现。运行时只允许已经完成本地验证并登记为可执行的精确规则，未完成条目返回人能看懂的“当前版本不支持执行该命令”。
4. 不等待四个品牌全部完成才接UI，也不为单条规则建立临时旁路。第一条纵向链通过后，新增兼容规则只复用现有链。
5. 每个切片默认停在最近的Windows focused结果；只有LinuxCNC ABI/解释器验证确实需要时才进入VM，只有批量集成或用户明确要求时才上板。
6. 任一切片不得用“完整白名单、完整profile库、全项目CMake或最终镜像”作为当前小功能的前置门禁。

## 2. 目标 owner 和架构边界

### 2.1 唯一 owner

兼容功能应扩展现有 `program_runtime` owner，不新建第二个G-code运行服务：

```text
program_runtime
  ├─ 原文件和source identity
  ├─ source profile
  ├─ lexer/parser/modal state
  ├─ typed IR
  ├─ allowlist/ruleset
  ├─ canonical emitter
  ├─ output verifier
  ├─ line map/error list/report
  └─ canonical artifact identity
```

UI只提交Program Open/MDI并显示结果。Command Gate只传输已冻结路径/文本和identity。LinuxCNC/native owner继续拥有解释、运动、安全、坐标、刀补、RTCP和actual。

### 2.2 目标状态机

`V5ProgramRuntime` 最终需要明确状态：

| 状态        | 含义                                                                | Start是否允许 |
| ----------- | ------------------------------------------------------------------- | ------------- |
| `EMPTY`     | 没有程序                                                            | 否            |
| `ANALYZING` | 正在读取、匹配或解析                                                | 否            |
| `CANDIDATE` | 本地canonical已经完整生成，正在等待同generation目标LinuxCNC预检回执 | 否            |
| `REJECTED`  | 已有本generation结构化错误                                          | 否            |
| `READY`     | source/canonical/profile/ruleset/target identity已原子冻结          | 是            |
| `STALE`     | 任一identity变化                                                    | 否            |

不得继续用单个 `loaded=1` 同时表示“文件读到了”和“程序可以运行”。

### 2.3 目标执行链

```text
Program Open
 -> source read/hash
 -> auto profile match
 -> normalize
 -> strict lexer/parser
 -> source AST/modal
 -> unified allowlist exact rule group
 -> typed IR
 -> canonical emitter
 -> output AST allowlist
 -> LinuxCNC exact preflight
 -> atomic artifact + line map + report
 -> READY

Start
 -> exact READY generation/hash
 -> canonical artifact path
 -> existing Command Gate
 -> LinuxCNC AUTO

MDI
 -> same profile/normalizer/parser/rules/emitter/verifier
 -> one canonical block
 -> existing Command Gate
 -> LinuxCNC MDI
```

6. 首个核心组拒绝active comment、O-call、M100-M199、外部文件、未知remap和M128/M129；后续功能组只能按新通过的精确规则扩大输出AST，不能删除这层验证。
7. 内存中完成artifact候选，不写最终运行路径。

### 7.2 focused测试

- 同一输入+identity得到逐字节相同canonical和hash；
- source数值精度不丢失；
- error能从canonical行反查source span；
- emitter不能制造来源规则未声明的模态；
- 注释/O-call/M100-M199注入全部拒绝；
- artifact候选失败时无部分文件；
- LinuxCNC native profile不做无意义语义改写。

### 7.3 LinuxCNC精确验证

Windows先完成纯C output verifier。只有该层通过后，才进入VM运行最小LinuxCNC parser/Program Open验证；不构建rootfs或镜像。

## 8. 作业4：接入现有 program_runtime

### 8.1 先拆分再接入

`v5_program_runtime.c` 当前437行，不能继续把parser、artifact和错误控制流塞进去。先将现有文件读取/删除、resident查询等按现有语义做最小拆分，再把compat orchestrator接到Open。

### 8.2 新Program Open事务

1. 用户发起新的Program Open时创建新generation并进入 `ANALYZING`，同时撤销旧generation的活动资格；Start/Resume立即拒绝旧artifact。
2. 在同一受控事务中清空程序待执行引用、MDI待执行缓存和微内核/Native待执行代码；旧artifact文件可以留作诊断，但不能继续作为当前READY程序。
3. 读取source和hash并执行compat pipeline。
4. 失败：新generation进入 `REJECTED`，保留结构化错误；active canonical为空，所有执行侧代码保持为空，不允许退回上一个READY程序。
5. 本地成功：原子提交source/canonical/profile/ruleset/line map/report和新的活动generation，进入 `CANDIDATE`；此时仍禁止Start。
6. 作业5取得同generation的目标LinuxCNC精确预检回执后才进入 `READY`；发送成功、旧回执或仅本地parser通过都不能升级。
7. preview和first point从canonical IR生成。
8. 2 MiB限制首阶段保持不变；大文件另立切片。

### 8.3 结果合同

`V5ProgramOpenResult` 至少增加：

- state/generation；
- source profile identity；
- ruleset identity；
- canonical path/hash；
- line map/report identity；
- auto rewrite count；
- error count和first error；
- target preflight状态；
- ready_to_run。

### 8.4 focused测试

扩展现有 `v5_program_runtime_smoke`：

- 旧source hash、preview、删除和2MiB回归不倒退；
- REJECTED不能prepare_start；
- stale不能prepare_start；
- CANDIDATE不能prepare_start，且不能被UI显示为可运行；
- 只有同generation目标预检回执才能把CANDIDATE升级为READY；READY只返回canonical identity；
- 失败不留下半artifact；
- preview/first point与canonical IR一致。

## 9. 作业5：LinuxCNC预检、Start、First Point和MDI

### 9.1 Program Open预检

复用已有 `V5_COMMAND_PROGRAM_OPEN` 和 `Set Open` transport，但只提交canonical path。Command Gate必须返回同一request/generation的LinuxCNC parse receipt；不能仅凭socket发送成功标记READY。

### 9.2 Start

- `v5_program_runtime_prepare_start` 从 `source_path` 改为approved canonical path；
- 同时绑定source hash、canonical hash、profile/ruleset identity和generation；
- Command Gate只接受当前READY identity；
- Start前identity变化立即返回 `GCODE_ARTIFACT_STALE`；
- Command Gate不重新翻译、不自行选择profile。

### 9.3 First Point

`v5_command_first_point.c` 改用canonical IR的首个运动语义，不再从raw preview parser猜测；请求绑定同一artifact generation，保证预览、首点和实际运行同源。

### 9.4 MDI

- `v5_program_runtime_set_mdi_line` 进入同一compat pipeline；
- 只允许单个完整canonical block；
- raw外部方言文本不得直接进入 `Set MDI`；
- 错误映射、profile、白名单和输出验证与程序文件一致；
- 翻译成功后canonical文本进入唯一MDI缓存，并绑定canonical hash和generation；
- MDI待执行显示只能来自微内核/Native同generation readback，不能显示UI本地副本；
- 用户清空MDI时必须同时清空program_runtime和微内核/Native待执行代码，并取得长度0/hash空的fresh readback；
- UI、program_runtime和微内核/Native任一identity不一致即清空并fail-closed；
- MDI翻译成功后仍需用户明确触发执行，禁止因自动翻译完成而自动运动；
- 失败显示准确token，且微内核/Native待执行缓存保持为空。

### 9.5 focused验证

- Program Open preflight失败不得READY；
- source path绝不进入Start transport；
- raw MDI绝不进入transport；
- 空MDI和clear事务后native code length=0、hash为空；
- UI显示、runtime缓存和native readback逐字节/hash/generation一致；
- stale MDI、旧generation和显示为空时的运行请求全部拒绝；
- identity串run/stale/replay反例；
- command gate formatter和validator focused smoke；
- 当前 `v5_program_command_smoke` 从source-path断言迁移为canonical identity断言。

## 17. 完成定义

本功能只有同时满足以下条件才算完成：

1. 用户打开受支持的外部程序时，系统自动匹配profile并自动转换，不要求理解LinuxCNC写法。
2. Program Open成功意味着canonical artifact已经完整生成、输出白名单通过并被目标LinuxCNC确认。
3. Start、Resume、First Point、preview和MDI消费同一identity和同一语义结果。
4. 表外命令和真实格式错误都使用白名单第13节固定中文映射、准确源位置和处理建议，内部错误身份不代替人类解释。
5. 无未知透传、静默删除、猜测修复或部分程序运行。
6. 每个外部profile有精确版本身份和正反golden。
7. UI成功路径少打扰，失败路径一次说明清楚。
8. 旧raw source/MDI旁路和平行白名单物理删除。
9. 日常新增一条规则只触发compat focused target，不触发全量Linux/PetaLinux构建。
10. 涉及真实运动的规则完成原始operator路径和fresh native actual闭环。
11. MDI显示、program_runtime缓存和微内核/Native待执行canonical代码始终逐字节/hash/generation一致。
12. MDI显示为空时微内核/Native没有残留代码，旧generation不能运行。
13. 错误弹窗“确定”只进入手动编辑并定位错误，不会向执行缓存写入代码。
14. 新Program Open、MDI提交或清除一旦开始，旧generation立即失去运行资格；失败时UI、program_runtime和微内核/Native不存在可误触发的旧代码。
