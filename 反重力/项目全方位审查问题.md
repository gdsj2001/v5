# V5 项目全方位审查问题

本文只记录基于当前 Windows 源码、需求 owner 和实际工具链确认的未闭合问题。产品要求仍以 `功能/需求真源索引.md` 及其指向的唯一 owner 为准，本文不复制第二套需求。

## 审查边界

- Git 只作为异地备份/网盘，不作为版本基线、发布门禁或评分标准。
- 微内核、HAL、UIO、native memory 和控制接口直驱所需的高权限不是缺陷；不得为了形式降权增加代理层、包装服务或第二条 IPC 控制路径。
- `AGENTS.md`、需求索引和功能 owner 的详细约束是防止 AI 跨 owner、增加 fallback、绕过 native truth、复制源码或直接修改板端的必要护栏，文档数量和篇幅不作为问题。
- 当前 `vm_board.lock` 由其它任务持有，本次没有进入 VM 或开发板。Windows LinuxCNC、PetaLinux source identity 已通过本地校验；全量构建、部署和板端 operator/motion 尚未在本轮重验。

当前最强状态：`local_verified_only`。

## P0-1：黄金运动验收脚本绕过真实 UI

引用需求：`REQ-AUTO-STATE-DRIVEN`。

### 当前事实

`board/tools/deploy/run_v5_board_acceptance.sh --motion` 最终通过 SSH 直接执行 `v5_linuxcncrsh_golden_run --start`。这能证明后端 LinuxCNC/motion runner，但不能证明程序页选择与双击打开、程序 identity/hash、主页面 Start、急停、取消急停和再次 Home 的真实 UI/operator path。

### 风险

- 后端成功可能掩盖 UI、IPC、按钮状态或 readback 链路回归。
- 该脚本不能单独产生 `board_verified` 结论。

### 关闭条件

1. 黄金测试从主页面真实点击 `打开程序` 开始。
2. 使用 fresh native active model，只选择匹配的原始 `cc-ac.ngc` 或 `cc-bc.ngc`。
3. 通过 UI 完成选中、打开、identity/hash 核对、Home、Start、1 秒急停、取消急停和再次 Home。
4. 最终确认 `estop_active=1`、`machine_enabled=0` 和 Machine Off。
5. 后端 runner 只作为诊断工具，不再充当 operator-path 通过证据。

## P0-2：当前 Windows 源码尚无本轮完整上板闭环证明

引用需求：`REQ-MICROKERNEL-SOURCE-REBUILD`、`REQ-BOARD-PROGRAM-FULL-BUILD-CLOSURE`。

### 当前事实

本轮只完成 Windows source identity 和本地结构审查。由于共享 VM/板资源已有 owner，没有执行当前 Windows 文件内容对应的 VM 干净全量构建、部署和真实板端验证。

### 尚缺证据

- `/mnt/v5-source` 当前确为 Windows 唯一源码的实时只读挂载。
- 构建输出全部位于挂载外 `VM_BUILD_ROOT`，overlay 未被其它任务污染。
- XSA、bit、kernel、LinuxCNC、rootfs、PetaLinux image 和 runtime artifact 与当前 source identity 一致。
- 部署清单已完整落板并重新启动运行闭包。
- 原始 UI/operator/control/motion 路径通过，测试后机器处于安全 ESTOP。

### 关闭条件

独占 `vm_board.lock` 后，从 Windows 唯一源码执行干净完整构建、部署、板端 readback、真实 operator path 和必要 golden motion，并保存同一轮证据。

## P1-1：两个 canonical 构建工具的只读挂载合同不一致

引用需求：`REQ-MICROKERNEL-SOURCE-REBUILD`、`REQ-FILE-CLASSIFIED-PLACEMENT`。

### 当前事实

- `board/tools/petalinux/v5_petalinux_overlay.sh` 接受 `fuse.vmhgfs-fuse`、`9p`、`cifs`、`nfs`、`nfs4` 和 `virtiofs` 等只读共享文件系统。
- `board/tools/linuxcnc/build_v5_linuxcnc_petalinux.sh` 只接受带 `ro` 的 `fuse.vmhgfs-fuse`。

### 风险

更换 VM 共享挂载实现后，可能出现 PetaLinux overlay 校验通过、LinuxCNC 完整构建却拒绝的分裂行为。

### 关闭条件

两个工具复用同一 source-mount validator，统一共享文件系统允许范围、只读判断、Windows 真源身份和路径逃逸检查，并对实际 VM 挂载执行成功与 fail-closed 测试。

## P1-2：完整构建产物身份仍是分散证明

引用需求：`REQ-MICROKERNEL-SOURCE-REBUILD`、`REQ-BOARD-PROGRAM-FULL-BUILD-CLOSURE`。

### 当前事实

Linux kernel、PREEMPT_RT、LinuxCNC 和 PetaLinux 已有各自 source identity，但一次完整构建中的 XSA、bit、kernel、rootfs、LinuxCNC native、runtime 部署清单和板端实际文件仍需跨多个输出分别核对。

### 风险

单项 identity 正确，不自动证明整套板端产物来自同一次 Windows source/build 输入。

### 关闭条件

每次完整构建生成统一 artifact manifest，至少关联四份 source identity、XSA/bit、工具链版本、kernel/rootfs/LinuxCNC/runtime artifact 和部署清单 hash；不依赖 Git commit 或 dirty 状态。板端只读核对结果必须能与该 manifest 对应。

## P1-3：异常链路和长期稳定性证明不足

引用需求：对应功能 owner、`REQ-AUTO-STATE-DRIVEN`、`REQ-NATIVE-OWNER-FIRST`。

### 当前事实

项目已有较多 focused smoke，但本轮没有找到当前整套产物对以下场景的统一板端证据：EtherCAT 断链、驱动 fault、native publisher 卡死、Command Gate 丢失、SHM stale、网络异常、进程崩溃和长时间连续运动/显示。

### 风险

正常路径通过不能证明异常时始终 fail-closed，也不能证明长时间运行中没有状态陈旧、资源泄漏、实时抖动或 WKC/DC 退化。

### 关闭条件

按 BUS/Pulse、AC/BC、RTCP ON/OFF 建立最小故障矩阵，记录安全状态、native readback、CPU/IRQ affinity、servo latency、EtherCAT DC/WKC、恢复过程和最终 ESTOP，并补充长时间 G-code、状态发布和远程显示稳定性测试。

## 处理顺序

1. 修正黄金运动验收，使其只认真实 UI/operator path。
2. VM/板资源可独占后，完成当前 Windows 源码的全量构建、部署和板端闭环。
3. 统一两个 canonical 构建工具的只读挂载合同。
4. 把分散的 source/artifact/board identity 串成同一次构建证明。
5. 完成故障注入和长期稳定性验证。
