# 50MB 小线段 G-code 支持实施操作指导书

## 0. 文档用途与当前结论

本文是 50MB 级密集小线段 G-code 的实施操作指导书，目标是让后续执行者能按固定顺序修改、验证、上板和验收。本文位于 `反重力/`，只保存活动实施输入，不替代 `功能/` 下的唯一需求 owner。

正式实施前必须按以下 owner 收口产品口径：

- Program Open、scene ready、Start 准入：[`../功能/3-1主页面按钮功能.md`](../功能/3-1主页面按钮功能.md)。
- G-code resident、Start 热路径、CPU0/CPU1：[`../功能/0-1开机参数入内存.md`](../功能/0-1开机参数入内存.md)。
- 黄色轨迹、scene generation、fit 和 UI 显示：[`../功能/3-4UI刀路绘图顺序与状态稳定方案.md`](../功能/3-4UI刀路绘图顺序与状态稳定方案.md)。
- 小线段运行连续性和前瞻：[`../功能/微内核.md`](../功能/微内核.md)。
- REQ 路由：[`../功能/需求真源索引.md`](../功能/需求真源索引.md)。

当前功能状态是 `source_only`，不能宣称 50MB 已支持：

- 工作树已把本地 Program Runtime 和远程上传上限从 2MiB 改为 64MiB。
- Program Open 仍同步阻塞 UI，仍把完整文件装入内存并执行多遍扫描。
- 预览仍在 65,536 个候选点或 64 个切削段后提前停止。
- 尚无 50MiB 全文件、真实 UI、ARM、板端内存和 CPU affinity 证据。

因此，单独保留“64MiB 常量修改”不是可交付方案。必须完成本文阶段 A～G；需要通过远程客户端上传 50MB 时，还必须完成阶段 H。

## 1. 最终交付目标

### 1.1 必须实现

1. 产品可从真实 Program 页面直接打开不超过 64MiB 的 G-code。
2. 50MiB 密集小线段必须扫描到 EOF。
3. 黄色轨迹必须覆盖程序开头、中间和末尾的大致形状。
4. Program Open 不阻塞 LVGL 事件循环，打开期间页面和远程输入保持响应。
5. 完整文件不常驻 `V5ProgramRuntime`，内存随文件大小增长的路径必须删除。
6. SHA、行数、首点、modal parse、bounds 和 LOD 在一次顺序读取中完成。
7. worker 固定在 CPU1；CPU0 继续专属 1ms servo、EtherCAT 和 safety。
8. 最终 SHM 仍最多发布 512 点，不改变当前 SHM v4/7136 字节 ABI。
9. Start 在 EOF、preview、scene 和 fit 同 generation ready 前必须拒绝。
10. Start 继续向 Command Gate 投喂原始文件路径，不生成执行副本、过滤文件或 sidecar。

### 1.2 本切片不做

- 不修改 LinuxCNC 原生滚动解释和 trajectory planner。
- 不扩大 SHM。
- 不把 UI 预览 parser 变成第二套运动解释器。
- 不先修改微内核队列、前瞻参数或 CPU0 realtime 链。
- 不让 50MiB 全程序直接执行运动作为 UI 打开功能的首个验收步骤。
- 不以降低黄色轨迹完整性、少扫尾部或提前 ready 换取打开速度。

### 2.1 当前链路

```text
Program页面双击
  -> shell_open_program_row_for_run
  -> v5_main_page_open_program
  -> v5_command_program_open
  -> v5_program_runtime_open_file
       -> 整文件读取
       -> 行数扫描
       -> preview扫描
       -> 首点扫描
       -> SHA扫描
  -> 返回UI
  -> 跳转主页面并生成scene/fit
  -> scene ready后Start携带原始路径
```

当前最大瓶颈在 UI/Program Runtime，不在 LinuxCNC。LinuxCNC只有按下 Start 后才通过原始路径逐行解释。

## 3. 目标链路和状态机

### 3.1 目标链路

```text
Program页面双击
  -> begin_open(path)，生成program_generation
  -> UI立即显示loading并返回LVGL事件循环
  -> CPU1唯一worker以256KiB固定块顺序读取
       -> incremental SHA
       -> 行数和文件进度
       -> 跨块行carry
       -> 顺序modal parser
       -> 首点、bounds、segment统计
       -> 固定容量分桶LOD
  -> EOF后复核文件identity
  -> 原子发布同generation preview snapshot
  -> UI进入主页面并请求唯一scene/fit
  -> scene/fit同generation ready
  -> Start允许，仍携带原始路径
```

### 3.2 唯一状态机

```text
EMPTY
  -> LOADING
       -> PREVIEW_READY
            -> SCENE_PENDING
                 -> READY
       -> CANCELLED
       -> ERROR
```

规则：

- 新请求先推进 generation，再取消旧 generation。
- 同一时刻只允许一个 worker 读取一个程序。
- worker 的 progress、error 和 final snapshot 必须携带 generation。
- UI只接受当前 generation，迟到结果直接丢弃。
- `PREVIEW_READY` 只表示 EOF 和预览完成，不等于允许 Start。
- `READY` 必须同时满足 program、scene 和 fit generation 一致。
- ERROR/CANCELLED 必须清除 Start 准入，不保留上一程序的 ready。

## 4. 文件拆分与代码职责

当前 `v5_program_runtime.c` 已约510行。实施时不得继续把线程、流式I/O、LOD和UI生命周期堆入同一文件。

### 4.1 保留并收缩

`board/services/program_runtime/v5_program_runtime.c/.h`

- 保留公共 runtime 生命周期、MDI、delete、snapshot访问器和Start准入。
- 删除 Program 模式的 `gcode_text`、`gcode_map_size`、`gcode_text_mmap`。
- `has_open_program()` 改为检查当前 generation 的 resident identity 和状态。
- `destroy()` 必须取消并安全回收 worker，不能让线程访问已释放 runtime。

### 4.2 新增流式读取 owner

建议新增：

- `board/services/program_runtime/v5_program_stream.c`
- `board/services/program_runtime/v5_program_stream.h`

职责：

- regular file、0字节和64MiB上限校验；
- 读取开始和EOF时的 size/mtime/inode identity；
- 256KiB固定块读取；
- 跨块行carry；
- 单行最大1MiB，超过返回 `PROGRAM_LINE_TOO_LONG`；
- 每个块和每行边界检查 cancel generation；
- 汇总 `bytes_scanned`、`file_size` 和进度。

禁止：

- 整文件 `malloc/mmap`；
- 无上限行缓冲；
- worker里调用LVGL；
- worker里周期落盘或生成sidecar。

### 4.3 改造 SHA owner

修改：

- `board/services/program_runtime/v5_sha256.c`
- `board/services/program_runtime/v5_sha256.h`

当前 SHA context 和 `init/update/final` 都是文件内 static。需要公开有界增量接口，同时保留 `v5_sha256_hex()` 一次性包装给小输入测试使用：

```c
void v5_sha256_init(V5Sha256 *ctx);
void v5_sha256_update(V5Sha256 *ctx, const unsigned char *data, size_t size);
void v5_sha256_final_hex(V5Sha256 *ctx, char out_hex[65]);
```

Program Open 只调用增量接口，不允许 EOF 后再扫描文件计算 SHA。

### 4.4 新增 streaming preview/LOD owner

建议新增：

- `board/services/program_runtime/v5_program_preview_stream.c`
- `board/services/program_runtime/v5_program_preview_stream.h`

复用当前 `v5_preview_parse_line()` 的 modal 语义，但把输入改成“一行一行推进”，不要复制第二套 parser。

每行一次性完成：

- G90/G91；
- G20/G21；
- G17/G18/G19；
- G53、G54～G59.3；
- G0/G1/G2/G3；
- XYZAC；
- 首个运动点；
- WCS/切削段变化；
- bounds和LOD更新。

内部LOD固定：

- 按文件字节位置划分1024个桶；
- 每桶最多保留8个代表锚点，内部上限8192点；
- 至少考虑桶首/尾、XYZ极值、明显转角、WCS变化和segment break；
- EOF后再从8192点有界集合压缩到最多512点；
- 第一桶、中间桶和最后一桶必须有代表点；
- 被跳过的源区间如果不能证明连续，后一个保留点必须设置 `break_before`；
- anchor超过显示预算时设置 `preview_decimated`，但仍扫描到EOF；
- `preview_truncated` 只表示语义无法完整表达或真实错误，不能再表示“容量满后停止”。

迁移完成后物理删除：

- `V5_PREVIEW_CANDIDATE_LIMIT`；
- `V5_PREVIEW_SEGMENT_LIMIT`；
- 候选池满后 `break`；
- 独立的整文件行数扫描；
- 独立的整文件首点扫描；
- Program Open路径上的一次性完整文本SHA。

### 4.5 新增异步 worker owner

建议新增：

- `board/services/program_runtime/v5_program_open_worker.c`
- `board/services/program_runtime/v5_program_open_worker.h`

使用一个持久 worker 和 latest-wins 请求槽，不为每次点击创建无限线程。

公共API建议收敛为：

```c
int v5_program_runtime_begin_open(V5ProgramRuntime *runtime, const char *path);
int v5_program_runtime_poll_open(V5ProgramRuntime *runtime, V5ProgramOpenSnapshot *out);
void v5_program_runtime_cancel_open(V5ProgramRuntime *runtime);
```

`V5ProgramOpenSnapshot` 必须复制字符串和统计值，禁止继续使用指向可变 runtime 的裸 `const char *` 作为跨线程结果：

- generation和state；
- code；
- source path/display name/SHA；
- file size/bytes scanned/line count；
- preview candidate total/kept/segment total；
- preview decimated/truncated/strategy；
- first point和axis mask；
- 不超过512点的immutable preview；
- WCS index、break flags和identity。

线程规则：

- Linux板端用 `pthread_setaffinity_np()` 将worker固定到CPU1。
- worker使用普通非实时调度；不得设置实时优先级。
- mutex只保护请求槽和snapshot交换，文件I/O/parser期间不得长时间持锁。
- UI线程只poll并复制snapshot。
- 新请求取消旧请求后必须在200ms内被worker观察。
- Windows focused测试允许不执行CPU affinity，但必须测试同一状态机。

8. 部署后核对板端 `/usr/libexec/8ax/v5_lvgl_shell` hash与ARM artifact一致。

#### 退出条件

- ARM focused build通过；
- canonical manifest部署成功；
- UI、State Publisher、Command Gate和relay readiness正常；
- 1ms servo仍为1,000,000ns；
- 最终先保持ESTOP和Machine Off。

### 阶段G：真实Program页面板端验收

#### G1. 无运动准入

1. fresh截图确认处于Program页面、ESTOP有效、Machine Off。
2. 通过产品现有程序导入路径放入50MiB固定SHA输入；不得用SSH直接写产品目录关闭任务。
3. 双击50MiB程序。
4. 100ms内抓取loading画面。
5. loading期间执行一次页面按钮或返回操作并抓后帧，证明UI没有卡死。
6. 执行一次取消，确认200ms内进入CANCELLED且Start不可用。
7. 再次打开同一文件，等待READY。
8. 抓取主页面黄色轨迹，确认能识别头部、中部和尾部几何。
9. 不执行50MiB全程序运动；本阶段只验证打开、scene和Start准入。

#### G2. 必须采集的readback

- `file_size`；
- `bytes_scanned`；
- `line_count`；
- `source_sha256`；
- `program_generation`；
- `scene_generation`；
- `fit_generation`；
- `preview_candidate_total`；
- `preview_kept <= 512`；
- `preview_segment_total > 64`；
- `preview_decimated/truncated/strategy`；
- begin到PREVIEW_READY时间；
- begin到READY端到端时间；
- `v5_lvgl_shell` VmRSS峰值和稳定增量；
- worker TID和 `Cpus_allowed_list`；
- servo overrun；
- EtherCAT 5/5 OP、WKC 10/10 complete、DC fresh。

#### G3. 数值门

| 指标 | 目标/硬边界 |
| --- | --- |
| begin返回UI | 不超过100ms |
| cancel被worker观察 | 不超过200ms |

#### G4. Start链最小证明

使用安全的小程序证明 READY前Start拒绝、READY后Command Gate仍生成：

```text
Set Task_Plan_Init
Set Mode Auto
Set Open <原始路径>
Set Run 0
```

50MiB程序只有在另行确认运动范围、进给、刀具和机床安全后，才进入实际运行性能验收。运行时若出现小线段queue starvation，再另立微内核/LinuxCNC预缓冲切片；不能把UI预览优化混入运动前瞻。

### 阶段H：远程50MB上传闭环

如果商业流程要求WinRemote直接上传50MB，本阶段是必做；如果文件已通过其它受控产品路径进入板端，可在核心Program Open闭环后单独实施。

#### 当前缺口

- `v5_remote_ui_relay_access.py` 一次 `read(content_length)`。
- `ProgramFileService.upload()` 接收完整bytes。
- 64MiB文件会在relay中形成大payload，随后还会写盘和计算SHA。

#### 操作

1. 保留64MiB上传上限和1MiB在线编辑上限。
2. 将HTTP body改成256KiB固定块流式写入同目录临时文件。
3. 写入同时计算SHA和字节数。
4. 内容长度、扩展名、目标路径和SHA全部验证成功后再原子replace。
5. 失败、断连、SHA不一致或超限必须删除临时文件。
6. 不生成长期sidecar或第二执行副本。
7. 上传/覆盖当前已打开程序时，必须使对应program generation失效并要求重新Open。
8. WinRemote显示上传进度，不能一次把50MB复制到多个内存缓冲。

#### focused测试

- 2MiB+1字节仍可上传；
- 64MiB边界接受；
- 64MiB+1由Content-Length门拒绝且不读body；
- 中途断连不留临时文件；
- SHA不一致不替换旧文件；
- overwrite原子；
- 1MiB以上仍不可进入在线文本编辑；
- relay认证/TLS合同不退化。

## 6. CPU0+CPU1双核候选的执行门

普通Program Open默认只使用CPU1。Machine Off不等于CPU0空闲；CPU0此时仍可能运行1ms servo、EtherCAT cyclic和safety。

只有阶段B～G全部完成后，真实50MiB仍持续超过10秒，并且profile证明CPU1 parser/LOD是主要瓶颈，才允许评估离线双核模式。

### 双核A/B前置条件

1. 先修改CPU owner文档，不直接改代码。
2. 建立显式“离线程序准备模式”。
3. 进入模式前ESTOP有效、Machine Off。
4. 停止LinuxCNC/EtherCAT实时闭环。
5. 回读CPU0已无servo、master cyclic和相关实时IRQ。
6. 双核只做有界流水线，例如一核读取/SHA、一核顺序modal parse/LOD。
7. 禁止把有前后modal依赖的G-code任意分块并行解释。
8. 扫描结束后clean restart实时闭环。
9. 重新证明1ms、5/5 OP、WKC10/10、DC fresh和Start readiness。

### 双核采纳条件

必须比较完整端到端时间：

```text
CPU1模式 = Open -> READY
双核模式 = 停实时闭环 -> 双核扫描 -> 重启 -> readiness -> READY
```

只有双核模式端到端显著更快、失败恢复清晰且不降低安全性，才允许进入产品。否则删除双核实验代码，不保留fallback或环境变量选择。

## 7. 失败处理和禁止事项

任何阶段失败时：

- 保持或恢复ESTOP/Machine Off；
- 状态进入ERROR/CANCELLED；
- Start继续拒绝；
- 从失败单点修复后续跑；
- 复用已通过的SHA、构建和验收证据；
- 不重跑无关全树、镜像或运动测试。

禁止：

- 只把2MiB改成64MiB就宣称完成；
- LVGL回调同步读取或解析50MB；
- 完整G-code或全部小线段常驻内存；
- 触达65,536点或64段后停止并冒充全程序；
- 为预览生成执行副本、临时G-code或sidecar；
- 扩大SHM发送全部轨迹；
- UI parser参与实际前瞻、等效角或轨迹规划；
- READY前允许Start；
- 仅凭Machine Off占用CPU0；
- 保留同步open作为异步失败fallback；
- 保留2MiB和64MiB双实现或环境变量切换；
- 手工SCP/SSH改板端产品文件关闭任务。
