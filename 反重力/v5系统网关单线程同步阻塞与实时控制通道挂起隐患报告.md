# v5 系统网关单线程同步阻塞与实时控制通道挂起隐患 AI 作业指导书

## 0. 当前判定

本问题在当前代码中**部分存在**，但原报告的整改口径需要校正。

已确认存在：

1. `board/services/command_gate/v5_command_gate_server.c` 仍是单线程 `accept -> serve_client -> close` 模型；当前请求没有处理完之前，服务端主循环不会回到下一次 `accept`。
2. command gate 服务端 `read_all()` 对已接入 UDS 客户端执行阻塞 `recv(..., 0)`，服务端侧未给这个 accepted client fd 设置读写截止时间；如果客户端连接后不发完整帧，也会卡住主循环。
3. `board/services/command_gate/v5_linuxcncrsh_client.c` 的 LinuxCNC TCP 读路径仍使用阻塞 `recv(..., 0)`，虽然 socket 设置了 `SO_RCVTIMEO/SO_SNDTIMEO`，但一次慢响应仍可能占住当前 command gate 请求，延迟后续 UDS 请求处理。
4. `V5_COMMAND_ESTOP_FORCE`、普通 LinuxCNC 命令和 `SETTINGS_AXIS_COMMIT` 都经过同一个服务端同步执行入口；慢 TCP、慢配置提交或坏客户端都可能拖住后续请求接入。

原报告需要校正的点：

1. `linuxcncrsh` 不能改成多 TCP 连接池或并发抢发。功能真源要求 native command gate 初始化阶段建立**单条常驻 TCP 连接**，后续离散命令按**单 writer/FIFO** 串行确认。
2. 急停 0.1s 生效不能依赖普通 TCP/text gate 作为最终 owner。最终急停 owner 必须下沉到 LinuxCNC/微内核/HAL/EtherCAT realtime/native safety 层；command gate 只能作为产品命令 adapter 或诊断链路。
3. 原文说“强行发送到底层 TCP 并反馈成功”不成立。安全命令不能只以“命令已发送”算成功，必须以 native/microkernel 实际停止或使能关闭回读作为证明。

## 1. 作业目标

把 command gate 从“单个慢请求可阻塞所有新请求接入”的结构，整改为：

- UDS 监听、客户端帧读写和后端执行互不拖死。
- 普通 LinuxCNC text 命令继续保持单条常驻 TCP、单 writer/FIFO。
- `estop_force`、`jog_stop` 等安全边沿不被普通请求、配置保存、慢 TCP 或坏 UDS 客户端挡住。
- 硬实时急停闭环由 native safety owner 证明，不由 product/UI/linuxcncrsh 普通文本链路伪造成功。

## 3. 禁止事项

- 禁止为了降低阻塞而给 `linuxcncrsh` 加多 socket 连接池。
- 禁止按命令短连接、并发抢发、乱序确认或绕开 native command gate。
- 禁止把 `estop_force` 做成普通 FIFO 队列尾部任务。
- 禁止把“命令已发送”“Broker accepted”“UI 显示已切换”当作急停 0.1s 生效证明。
- 禁止新增 Python/Broker/JSON/FIFO/临时脚本第二控制路径。
- 禁止用新弹窗、报错、禁用按钮替代真实链路整改。

## 4. 推荐整改路径

### 4.1 先补最小复现

先做本地 contract/smoke 测试，证明当前问题和修复后边界：

1. 启动 fake `linuxcncrsh`，让普通请求的响应延迟超过配置超时。
2. 在普通请求阻塞期间，发起 `estop_force` 或安全 probe，记录从 UDS 连接到拿到结构化响应的耗时。
3. 构造坏 UDS 客户端：连接 command gate 后只发送半帧，确认后续正常请求不会被它无限阻塞。
4. 构造 `SETTINGS_AXIS_COMMIT` 慢路径，确认它不会阻塞 safety edge 接入。

### 4.2 拆开接入层和执行层

服务端结构应改为以下边界：

- listener 使用非阻塞或 `poll/select/epoll` 驱动，主循环持续接入新客户端。
- accepted client fd 必须有明确读写截止时间；半帧、慢读、断连必须关闭该客户端，不能拖住全局主循环。
- 帧解析完成后再进入请求调度；不要在监听/读帧循环里直接执行慢后端动作。
- 普通 LinuxCNC 请求进入唯一后端 worker，worker 内保持单条常驻 TCP 和 FIFO 顺序。
- `SETTINGS_AXIS_COMMIT` 等慢配置/磁盘动作不得占用 safety edge 的接入与调度通道。

### 4.3 安全边沿单独处理

`estop_force` 和 `jog_stop` 必须在解析出请求类型后进入最高优先级路径：

- 不等待普通 command queue、配置保存、状态刷新或后台清理。
- 不做 UI/Broker 产品层门禁、身份/DNA、`command_busy`、backend-ready、status epoch 等前置检查。
- 如果 native safety owner 提供真实 emergency latch/gate，优先直接触发该 owner。
- 如果当前板端仍只能通过 `linuxcncrsh` adapter 触发一部分 BUS/LinuxCNC 行为，该 adapter 只能作为过渡触发/诊断链路，不能声明为硬实时最终 owner；必须同时给出 native/microkernel 实际停止或使能关闭回读。

### 4.4 LinuxCNC text worker 只做串行命令

`v5_linuxcncrsh_client.c` 允许继续在唯一 worker 内用阻塞读写，但必须满足：

- 每次读写有明确 deadline。
- 超时、协议错误、断线时当前命令明确失败，并由 gate 受控重连。
- 失败不能静默丢命令，不能重排后续命令，不能改走第二路径。
- 如果改为非阻塞 `poll/select`，也只是在同一个 single writer worker 内做 deadline 控制，不能引入并发 TCP writer。

## 5. 验收标准

本问题不能只靠源码阅读关闭。完成代码整改后至少要有这些证据：

1. command gate 目标可编译。
2. fake `linuxcncrsh` 延迟测试通过：普通请求阻塞时，安全 edge 的 UDS 接入和结构化响应不被拖到普通请求完成之后。
3. 坏 UDS 半帧测试通过：单个坏客户端不能阻塞后续正常请求。
4. `estop_force` 路径能在 0.1s 窗口内取得 native/microkernel 停止或使能关闭证明；仅有发送成功不算通过。
5. 若改动进入板端运行产物，必须按项目规则同步到 VM、全量构建、部署到板端，并用原始 UI/operator 路径触发验证。
