# v5 系统网关单线程同步阻塞与实时控制通道挂起隐患 AI 作业指导书

## 0. 当前判定

本问题在当前代码中**依然活跃存在**，原报告的整改口径需要精确定位：

1.  **冲突一（部分存在 - 短连接违例路径）**：
    虽然并发客户端处理没有引入多 socket 连接池，且在 `v5_command_gate_server.c` 中通过 `g_linuxcncrsh_lock` 实现了对所有 `linuxcncrsh` 操作的串行化（普通命令会复用全局连接）。但当前仍有实际的违例路径：
    在 [v5_linuxcncrsh_client.c:L515](file:///d:/v5/board/services/command_gate/v5_linuxcncrsh_client.c#L515) 中，`v5_linuxcncrsh_send_control_command_fresh()` 仍然在执行 `close -> connect -> send -> close` 的 TCP 短连接逻辑。急停 `force` / `reset` (急停强迫/复位) 序列仍然在调用该函数，这严重违反了 [功能/0-3native_helper白名单与验收声明.md:L70](file:///d:/v5/功能/0-3native_helper白名单与验收声明.md#L70) 中“*不按命令短连接、单常驻连接、单 FIFO 串行确认*”的硬性规定。
2.  **冲突二（完全存在 - 急停走普通文本命令链）**：
    在 [v5_command_gate_server.c:L204](file:///d:/v5/board/services/command_gate/v5_command_gate_server.c#L204) 中，`V5_COMMAND_ESTOP_FORCE` 仍然调用了 `linuxcncrsh` 的急停序列，其最终是在 [v5_linuxcncrsh_client.c:L919](file:///d:/v5/board/services/command_gate/v5_linuxcncrsh_client.c#L919) 附近通过 `Set Machine Off` 普通文本命令来执行。
    这与 [系统代码架构硬边界守则.md:L58](file:///d:/v5/系统代码架构硬边界守则.md#L58) 中规定的“*禁止 UI/Python/linuxcncrsh 普通命令链承担急停实时生效职责*”的安全红线直接冲突。

**结论**：先前针对套接字 `select` 和超时机制的修复，仅关闭了“网关客户端阻抗导致通道挂起”的问题，并未真正闭环解决上述两个重大的底层架构红线缺陷。

## 1. 作业目标

1.  **拆除并改造短连接路径**：彻底删除或重构 `v5_linuxcncrsh_send_control_command_fresh()` 短连接接口，确保所有控制和状态查询均运行在唯一的全局常驻 TCP 长连接上。
2.  **急停控制链物理下沉**：将 `V5_COMMAND_ESTOP_FORCE` 从 `linuxcncrsh` 文本命令链中彻底剥离，直接下沉到 native/微内核实时安全门禁（硬件/实时 HAL 侧），UI 与网关仅作状态同步与低频回读。
3.  **单常驻 FIFO 管道收口**：`linuxcncrsh` 通信层只保留普通离散指令（模式切换、程序控制等）的单常驻 FIFO 通道，严格禁止任何插播抢占动作破坏通信时序。

## 4. 推荐整改路径

### 4.1 清理违例短连接路径 (对齐冲突一)
*   **删除 fresh 短连接接口**：在 `v5_linuxcncrsh_client.c` 中彻底废除 `v5_linuxcncrsh_send_control_command_fresh`，并将其在急停 `force` 和 `reset` 中的所有调用改写为复用网关全局唯一的常驻 TCP 连接。
*   **串行 FIFO 队列锁加固**：利用已有的 `g_linuxcncrsh_lock` 确保所有发往 `linuxcncrsh` 的命令发送与 `recv` 读取，在唯一的全局常驻 TCP 管道上严格串行同步，决不允许任何并发 TCP 覆写或短连接。

### 4.2 急停控制链物理下沉 (对齐冲突二)
*   **下沉安全门禁**：重构网关 `V5_COMMAND_ESTOP_FORCE` 分支，停止调用 `linuxcncrsh` 发送任何 `Set Machine Off` 的用户空间文本指令。
*   **微内核/HAL 直接直通**：急停强制信号必须通过专用的 HAL pin (实时硬件管脚) 或者是 realtime (硬实时) 共享内存通道直通底层安全 Latch 模块。网关和 UI 侧仅通过低频回读该 HAL pin 或 SHM 状态以完成界面同步，剔除其控制职责。
*   **网关接入层与执行层非阻塞隔离**：UDS 监听与 accepted fd 读取改用非阻塞多路复用，避免在参数提交写盘或底层网络波动时，由于单线程同步模型导致新控制连接在队列中排队卡死。


## 5. 验收标准

本问题不能只靠源码阅读关闭。完成代码整改后至少要有这些证据：

1. command gate 目标可编译。
2. fake `linuxcncrsh` 延迟测试通过：普通请求阻塞时，安全 edge 的 UDS 接入和结构化响应不被拖到普通请求完成之后。
3. 坏 UDS 半帧测试通过：单个坏客户端不能阻塞后续正常请求。
4. `estop_force` 路径能在 0.1s 窗口内取得 native/microkernel 停止或使能关闭证明；仅有发送成功不算通过。
5. 若改动进入板端运行产物，必须按项目规则同步到 VM、全量构建、部署到板端，并用原始 UI/operator 路径触发验证。
