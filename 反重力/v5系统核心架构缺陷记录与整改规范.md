# v5 工业数控控制系统核心架构缺陷记录与整改规范 (AGENTS 规范对齐版)

本文件由 Antigravity（反重力）团队在深度全面审查 v5 工业数控控制平台 VM 侧 `/root/Desktop/v5/` 核心源码及物理链路，并对照 [AGENTS.md](../AGENTS.md) 最高规范进行自我整改审查后整理产出。主要用于记录当前系统在进程重启、参数校验、软限位重算、内存预算、共享内存物理隔离、例外文件审计、倍率合流以及测试覆盖率这 8 个核心链路上的代码遗留缺陷，并根据 AGENTS 规范确立重构落地规约。

---

## 一、 系统核心架构 8 处代码缺陷记录

### 1. `run_restart_handoff` 缺少总线级物理事务与回读安全保护
*   **物理缺陷**：`/root/Desktop/v5/services/drive_profile/v5_settings_actiond.py` 的 `run_restart_handoff` 逻辑中，仅使用 `systemctl restart` 循环重启了 5 个后台进程。整个重载时序中完全缺失了轴 Disabled 状态切换、HAL/EtherCAT 重置、SDO/PDO 重载初始化、以及从站 evidence 物理回读和校验失败 fail-closed 事务控制。

### 2. `v5_settings_apply.c` 仅起 prepare 过滤作用，未真正收口写入
*   **物理缺陷**：`/root/Desktop/v5/services/command_gate/v5_settings_apply.c` 中仅实现了 `v5_settings_apply_prepare()` 函数。该函数只承担判断 writable 所有权和正有限数校验的工作，参数的物理写入、内存回读、raw limits 同步计算和 restart pending 重启标记逻辑目前依然散落在 UI 和其他模块中。

### 3. 普通比例参数修改路径缺少 limits 软限位同步重算
*   **物理缺陷**：`/root/Desktop/v5/services/drive_profile/v5_drive_bus_action.py` 中只有总线设零动作 `persist_axis_zero_model()` 会计算并修改 INI 中的物理 limits 软限位；当用户常规修改螺距（`pitch`）、精度（`precision`）或减速比时，没有任何代码路径去重新折算修正物理 limits 软限位，会导致当量变化后物理限位漂移失效。

### 4. 程序/G-code 加载内存资源未满足 mlockall 预算与释放要求
*   **物理缺陷**：`/root/Desktop/v5/services/program_runtime/v5_program_runtime.c` 对大体积 G 代码的解析副本和轨迹缓存仍使用常规的 `malloc()` / `free()`。由于没有使用独立的 arena 堆或内存池，在文件关闭或页面销毁等非实时受控边界未执行 `malloc_trim` 迫使物理页退回，在 `mlockall` 锁存全部内存的环境下存在由于堆碎片累积引发 OOM 崩溃的隐患。

### 5. 动作结果确认仍留存文件块影子旁路（未彻底实现 UDS 回读）
*   **物理缺陷**：UI 侧在确认设零/WCS 偏置等动作完成时，仍通过轮询物理文件 `/dev/shm/v5_native_rtcp_status.bin` 影子二进制块来获取确认证据，未彻底从共享内存和文件块的旁路确认中解耦，增加了状态不一致概率。

### 6. `settings_runtime.json` 缺少强 Schema 防污染审计
*   **物理缺陷**：`/root/Desktop/v5/services/drive_profile/v5_drive_bus_action.py` 的 `load_settings_runtime()` 中仅执行了最浅层的 `json.loads` 加载，缺乏深层嵌套 key 的黑名单校验，无法阻止 WCS/G92/TLO/tool/RTCP/modal/status 等核心控制字段被间接嵌套写入，参数归类防线存在被击穿隐患。

### 7. 连续倍率（Override）滑条输入缺乏合流防抖机制
*   **物理缺陷**：`/root/Desktop/v5/app/src/v5_command_override.c` 仅有 100% 重置的 prepare 接口，如果前台开发滑条或连续倍率调整，在 UI 侧和网关侧均无 100ms 级别的合流/防抖机制，会导致高频 Socket 文本命令排队阻塞控制网关。

### 8. 静态红线与单元测试覆盖率为 0%
*   **物理缺陷**：代码库在 `tests/` 或 `app/` 下均无针对“服务/Broker 严禁读主 SHM”、“例外文件强 schema 防污染”、“重载失败 fail-closed 事务验证”、“当量软限位联动重算”以及“mlockall 下堆内存 RSS 稳定性”等核心架构边界的单元测试或静态检查脚本。

## 二、 三项硬性物理重构落地规约 (AGENTS 规范对齐)

在针对上述缺陷执行代码修复时，必须严格遵守以下三项重构生效规约：

### 规约一：内存预算、受控 Trim 与路径相对化规约（对齐 AGENTS.md P0_ENTRY_RULES 1）
*   **独立资源堆管理**：大体积 G 代码解析副本与轨迹缓存数据，代码中必须优先采用静态 pool 或是 `mmap/munmap` 映射管理，实现物理内存的整体申请与彻底释放。
*   **受控 Trim 热路径禁用**：在页面销毁、文件关闭、重载等**非实时边界**，允许并要求调用系统的 `malloc_trim(0)`；**严禁在 Start/Resume、插补、伺服和急停等实时热路径中调用 trim 或任何内存申请/释放动作**。
*   **板端常驻集（RSS）水位验收**：验收不以“是否调用了 trim”为依据，而是通过板端 `status` 监控 UI 进程的 `VmRSS/VmLck/Locked` 水位。要求在经历至少 10 次以上的文件加载/卸载及页面切换循环后，物理常驻集大小必须稳定回落至初始基线，绝对不允许出现单调增长。
*   **绝对路径相对化红线 (SOURCE_RELATIVE_ADDRESS_RULE)**：所有改动涉及的系统路径，在代码中绝对禁止硬编码绝对路径。必须使用基于项目 `PROJECT_ROOT` 的相对定位或通过运行时环境变量动态计算，保证代码库可安全移植。

### 规约二：配置物理生效路径、受控重启与 CPU0 独占（对齐 AGENTS.md CPU0_MICROKERNEL_EXCLUSIVE）
*   **从站与 bit 写入解耦**：从站改变写入自建参数表， bit 改变写入驱动参数表，二者在此阶段均不热改微内核内存，右上角直接进入 `保存并重启`（Restart Pending）状态。
*   **受控 Handoff 重置时序**：点击 `保存并重启` 时，网关必须执行完整的总线事务重载时序：强制将物理从站切换至 Disabled/Safe-Op $\rightarrow$ 重置 HAL / SDO 映射与当量换算 $\rightarrow$ 回读从站状态及从站/bit/scale 物理 evidence 一致。若重载中任一总线事务校验失败，必须触发 fail-closed（报错并维持去使能），严禁上线运行。
*   **CPU0 独占与调度避让 (CPU0_MICROKERNEL_EXCLUSIVE)**：状态发布进程 `v5-state-publisher`、网关进程等所有非实时（NRT）服务，**绝对禁止绑定、affinity 亲和或故意调度到 CPU0**。CPU0 为微内核高精度运动控制专属核心。非实时进程必须显式进行核心避让，且不能通过 `SCHED_FIFO` 抢占微内核主权。

### 规约三：Stale SHM 隔离、UDS Native Readback 与废弃代码物理删除（对齐 AGENTS.md DELETE_FIRST_NO_FALLBACK）
*   **动作器与 Broker 禁用共享内存**：在 C/Python 所有的动作执行器、Broker 服务以及门禁逻辑中，其源程序**绝对禁止直接或间接包含对共享内存 /dev/shm/v3_status_shm 映射的读取**。
*   **同源回读确认路径**：例如坐标设零、WCS/G92 偏置更改成功判定，网关在下发命令后，必须通过强类型 UDS 接口向微内核实时 readback 端口读取实时内存的 offsets 事实作为唯一成功证明，严禁读取 SHM。
*   **坐标值数据保真红线**：机械坐标（MCS）与理论坐标（CMD_MCS）作为底层状态的实时物理投影，**绝对禁止引入任何防抖、合流、平滑滤波或阻尼时延消纳算法**。必须保障 30ms 投影周期的绝对高频传递与物理保真，严禁在显示层隐瞒任何坐标真实的抖动或延迟。
*   **退役字段与影子旁路彻底物理删除 (DELETE_FIRST_NO_FALLBACK)**：所有迁移完毕的旧参数接口、影子二进制文件（如 `/dev/shm/*.bin`）及模态文本字段，**必须从物理代码和共享内存头文件中完全抹除**，严禁以“兼容占位（ABI Padding）”或“诊断保留”为由留在产品 mainline 路径中。
