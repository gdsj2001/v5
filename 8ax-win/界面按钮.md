# 8ax WinRemote 界面按钮

## 范围

本文记录 Windows 远程客户端 `8ax-win` 新增的底部按钮规则。按钮属于 WinRemote 操作入口，但正式板端行为必须走板端 `remote_ui_relay` 的受控 HTTP 接口；Windows 客户端不得 SSH 到板端、不得执行板端 shell、不得直接读写 LinuxCNC/HAL/EtherCAT。

## 界面尺寸

- 默认窗口尺寸为 `1440 x 920`。
- 最小窗口尺寸为 `1280 x 840`。
- 远程画面仍以板端 `1024 x 600` 帧缓冲为源，并在窗口中等比缩放。
- 底部独立按钮行不占用远程画面点击坐标区域。

## 按钮定义

| 按钮 | 动作 | 边界 |
| --- | --- | --- |
| `读取日志` | 调用板端 relay `GET /remote/diagnostics`，读取板端软件运行全貌诊断快照。窗口默认显示中文摘要：诊断时间/编号、总体状态、界面服务、远程显示、远程输入、状态同步、指令通道、程序文件、画面刷新、输入状态、最近错误。`详细信息` 按钮才显示缩进多行 JSON。摘要和详细信息都必须先脱敏 LinuxCNC/底层控制层名称、硬件资源词和直接板端底层路径。 | 只读诊断；不得把诊断数据作为控制真值；不得 SSH、shell out 或从 Windows 直接读取板端 `/run`。 |
| `传G代码` | 本地选择 `.ngc`、`.nc`、`.tap`、`.gcode` 文件，通过 relay `POST /remote/program/upload?filename=<name>` 上传到板端“打开程序”页面扫描目录 `/opt/8ax/v5/gcode/golden`。 | 只负责文件上传和板端读回校验；不得自动打开、运行、预览、选择程序；不得调用 LinuxCNC/HAL。 |
| `OTA升级` | 调用板端 relay `POST /remote/ota/upgrade`，请求板端 OTA client 按当前 DNA private 优先、private 存在则不下载 public 的策略进入 OTA 状态机。当前代码只返回 `OTA_NOT_IMPLEMENTED`，不启动升级任务。 | WinRemote 只发请求和显示状态；不得下载 OTA 包到 Windows，不得 SSH/SFTP/shell 到板端，不得写 rootfs/boot env，不得重启。 |

## 板端接口要求

`GET /remote/diagnostics` 返回 JSON，至少包含：

- relay 基本状态和 `/remote/info` 同源资源指标。
- `/run/8ax_v3_product_ui` 下关键状态文件的存在、大小、mtime、可解析 JSON 摘要和有限尾部内容。
- `/dev/shm/v3_status_shm` 的存在、大小、mtime 和 SHA256 证据。
- 最近 dirty/input/relay 事件的只读诊断信息。

`POST /remote/program/upload?filename=<name>` 要求：

- `filename` 必须是 basename，不允许路径分隔符、盘符、`..` 或空名。
- 允许扩展名仅限 `.ngc`、`.nc`、`.tap`、`.gcode`。
- relay 必须限制上传大小，固定上限 `64 MiB`（`67,108,864 bytes`），与 WinRemote 客户端和板端 program runtime 一致。
- 写入必须先落同目录临时文件，`fsync` 并校验后再原子替换到 `/opt/8ax/v5/gcode/golden/<filename>`。
- 返回 JSON 证据：目标路径、文件名、字节数、SHA256、是否覆盖、时间戳。

`POST /remote/ota/upgrade` 要求：

- 请求必须声明 `dna_private_first_no_public_when_private_present` 选择策略。
- 当前 DNA private 区存在 OTA 包记录时，后续板端 OTA client 只能下载 private 包，不得下载 public 包。
- 当前 DNA private 区明确无包并记录 reason 后，后续板端 OTA client 才可选择 public 包。
- private 包存在但授权、版本、hash、签名、profile 或防降级校验失败时必须 fail-closed，不能改用 public 包。
- 当前代码没有 OTA client、Broker action 和 VPS OTA selector；relay 必须返回 `status=rejected`、`code=OTA_NOT_IMPLEMENTED`、`cancellable=false`，不能调用 deploy/cutover 工具冒充升级。

## 验证要求

本地验证至少覆盖 WinRemote 编译、TLS证书固定与challenge/session/scope负向测试、板端 relay 单元测试和 `git diff --check`。真实交付状态只有在板端部署新 relay、WinRemote 升级到新包，并通过原始 HTTPS/WSS 操作入口点完成诊断读取和 G-code 上传读回后，才能标记为 `board_verified`。

诊断窗口验证必须确认默认视图是中文摘要，不是原始 JSON；`详细信息` 视图也不能以单行横向滚动为主要阅读方式：客户端应先 JSON pretty print，再启用文本自动换行，长字段按窗口宽度折行。窗口正文不得显示 LinuxCNC/底层控制层名称、硬件资源词或直接板端底层路径；这些内容必须替换为 `[redacted]`。

## 打开系统G代码

`打开系统G代码` 是 WinRemote 底部本地目录式入口，用来管理板端“打开程序”页面的同一目录 `/opt/8ax/v5/gcode/golden`。它只做文件管理和文本修改，不执行 G 代码，不发送运动命令。

- 列表：调用 `GET /remote/program/list`，显示板端当前 `.ngc`、`.nc`、`.tap`、`.gcode` 文件；上传完成后该列表和板端程序页都应从同一目录刷新显示。
- 同名确认：上传前调用 `GET /remote/program/file?filename=<name>`；如果板端已有同名文件，必须提示是否覆盖。未确认时不得覆盖；确认后才允许 `POST /remote/program/upload?filename=<name>&overwrite=1`。
- 打开修改：选中文件后调用 `GET /remote/program/file?filename=<name>&content=1`，在本地窗口文本框中打开内容。
- 保存：保存修改等同于确认覆盖写回，走 `POST /remote/program/upload?filename=<name>&overwrite=1`，并校验返回的大小/SHA。
- 删除：选中文件后必须二次确认，再调用 `DELETE /remote/program/file?filename=<name>`。
