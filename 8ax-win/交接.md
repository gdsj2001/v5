# 8ax.WinRemote 交接

更新时间：2026-06-18

## 当前定位

`8ax.WinRemote` 是板端 UI 的远程显示和远程触摸终端，不是第二套控制系统。

正式操作链路必须保持为：

```text
Windows WinRemote
  -> board remote_ui_relay HTTP/WebSocket
  -> 板端 UI 原输入事件
  -> Command Broker UDS
  -> LinuxCNC / HAL / EtherCAT
```

WinRemote 只允许做两件事：

- 显示板端 relay 提供的 full frame / dirty rect 画面。
- 通过 relay WebSocket 发送 pointer event，等价于人在板端屏幕触摸。

WinRemote 不允许：

- 直接实现 Start、Home、Jog、E-stop、WCS、倍率、设零等业务命令。
- 直接写 LinuxCNC、HAL、EtherCAT。
- 直接写 `/run` 下 JSON、FIFO、`tap`、`tap_cmd` 等临时控制文件。
- 把直接 `/dev/fb0` 抓屏路径当远程显示、诊断截图或闭环证据路径。

## 板端需要怎么改才能对接 WinRemote

正式对接只允许当前V5 relay安全链；以下内容不再保留明文或marker兼容入口。

### 1. 板端必须启动正式 relay

板端必须由canonical `v5-ui-relay`服务常驻启动，加载设备TLS证书和`clients.json`；缺失或权限异常时fail-closed。禁止通过marker、环境开关或明文listener恢复旧入口。

涉及板端文件：

- `tools/deploy/install_v5_runtime.sh`
- `app/src/v5_lvgl_remote_display.c`
- `app/src/v5_lvgl_shell.c`

### 2. 板端画面链要走 LVGL flush dirty rect

WinRemote 期望板端提供：

```text
HTTPS GET /remote/info
HTTPS GET /remote/frame/full
WSS   /remote/stream
```

板端实现边界：

- `v5_lvgl_shell.c` 在 LVGL flush 时只做轻量复制。
- UI 主进程不做 zlib、PNG、WebSocket、TLS、磁盘日志或阻塞网络 I/O。
- `v5_ui_relay`常驻服务负责TLS、challenge/session/scope授权、full frame、dirty rect、心跳和客户端连接。
- 直接 `/dev/fb0` dump 抓屏已退役，不能作为诊断截图、远程显示链或闭环证据路径。

当前已有运行文件方向：

```text
/run/8ax_v5_product_ui/remote_framebuffer.bgra
/run/8ax_v5_product_ui/remote_dirty
/run/8ax_v5_product_ui/ui_ready.json
```

这些文件/缓冲只能作为板端 relay 内部实现细节；WinRemote 不能直接读写它们。

### 3. 板端输入链必须注入原 UI 事件

WinRemote 正式输入只会走：

```text
WSS /remote/input
  -> control_request
  -> pointer_event down / move / up
  -> pointer_ack
```

板端 relay 收到 pointer event 后，必须通过板端本地 UDS 输入口注入 UI 原输入事件队列：

```text
remote pointer event
  -> /run/8ax_v5_product_ui/remote_input FIFO framed JSON
  -> LVGL input event
  -> 原页面/按钮 handler
  -> Command Broker UDS
  -> LinuxCNC / HAL / EtherCAT
```

禁止做成：

```text
WinRemote
  -> relay
  -> 直接发 Start/Home/Jog/E-stop/WCS/倍率/设零命令
```

也禁止做成：

```text
WinRemote 或 relay
  -> /run/.../tap 或 tap_cmd
  -> 测试 tap 路径
```

也禁止做成：

```text
WinRemote 或 relay
  -> /run/.../remote_input FIFO
  -> UI 读取全局临时控制 FIFO
```

正式输入只能是连接式 WebSocket 会话加板端本地 UDS framed input。`tap`、`tap_cmd`、SSH tap 和 `/run/.../remote_input` FIFO 只能作为历史/测试痕迹清理对象，不能作为 WinRemote 正式接入链路。

### 4. relay input 的最小协议要求

板端 `/remote/input` 至少要做到：

- 每个客户端独立 session。
- `control_request` 成功后才允许 pointer event。
- pointer event 必须包含 session、seq、phase、x、y、button、client_time。
- phase 只允许 `down`、`move`、`up`。
- x/y 必须校验在 1024 x 600 画面范围内。
- 每个事件返回 `pointer_ack`，包含 accepted、seq、phase、server_time。
- relay 或 UI 不可用时 fail-closed，不吞掉错误。
- 禁止把 pointer event 翻译成业务命令；只能翻译成 UI 输入。

### 5. 板端验收顺序

先做只读显示，再做输入。

显示验收：

```powershell
.\publish\win-x64\8ax.WinRemote.exe --relay-security-profile D:\授权私钥\remote-relay-390529\winremote\relay-security.json --view-only true
```

通过标准：

- `/remote/info` 返回宽高、pixel_format、view_only 等字段。
- `/remote/frame/full` 能返回完整画面。
- authenticated `WSS /remote/stream` 有 dirty rect 更新。
- WinRemote 能稳定显示板端画面，不依赖 `/dev/fb0` 轮询。
- 板端 UI loop 仍接近 30 Hz，relay 不明显拖慢 UI。

输入验收：

```powershell
.\publish\win-x64\8ax.WinRemote.exe
```

通过标准：

- Win 端点击或拖动产生 `pointer_event`。
- 板端 relay 返回 `pointer_ack`。
- 板端 UI 收到和本地触摸一致的事件。
- 若点击按钮，后续仍走原按钮 handler 和 Broker。
- Win 端首个 dirty rect 反馈应在目标时间内返回。
- 失败时 Win 端和板端都记录明确错误，不伪造成功。

### 6. 不需要改 WinRemote 的前提

如果板端按上面协议提供 relay，当前 WinRemote 不需要再增加控制命令。

WinRemote 的默认启动路径已经是：

```text
https://192.168.1.221:18080/
pinned TLS + scoped relay stream/input session
```

后续主要工作应在板端：

- 启动 relay。
- 保证 dirty rect 输出。
- 保证 `/remote/input` 注入 LVGL 原事件。
- 保证 relay ACK 和视觉反馈延迟达标。

## 后续闭环建议

1. 先处理板端 relay 启用和启动链，不要再改 WinRemote 走 SSH tap。
2. 板端认证后的`/remote/info`、`/remote/frame/full`、`WSS /remote/stream`正常后，再用显式设备安全配置启动WinRemote验证画面刷新。
3. relay input 验收要记录：
   - Win 端 click / drag。
   - board relay `pointer_ack`。
   - 板端 UI 收到原输入事件。
   - Broker 命令结果。
   - 首个 dirty rect 反馈时间。
4. WinRemote 操作必须按“板端原始触发链”验收，不能只用直接 UDS 调用替代。
